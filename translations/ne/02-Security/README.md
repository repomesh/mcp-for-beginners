# MCP सुरक्षा: एआई प्रणालीहरूको लागि व्यापक सुरक्षा

[![MCP सुरक्षा उत्कृष्ट अभ्यास](../../../translated_images/ne/03.175aed6dedae133f.webp)](https://youtu.be/88No8pw706o)

_(यस पाठको भिडियो हेर्न माथिको तस्वीरमा क्लिक गर्नुहोस्)_

सुरक्षा एआई प्रणाली डिजाइनको आधारभूत पक्ष हो, त्यसैले हामी यसलाई हाम्रो दोस्रो खण्डको रूपमा प्राथमिकता गर्छौं। यो Microsoft को **Secure by Design** सिद्धान्तसँग मेल खान्छ जुन [Secure Future Initiative](https://www.microsoft.com/security/blog/2025/04/17/microsofts-secure-by-design-journey-one-year-of-success/) बाट आएको हो।

Model Context Protocol (MCP) ले एआई-चालित अनुप्रयोगहरूमा शक्तिशाली नयाँ क्षमताहरू ल्याउँछ, जसले परम्परागत सफ्टवेयर जोखिमहरू भन्दा परे अनौठो सुरक्षा चुनौतीहरू पनि प्रस्तुत गर्दछ। MCP प्रणालीहरूले सुरक्षित कोडिङ, न्यूनतम विशेषाधिकार, आपूर्ति श्रृंखला सुरक्षा जस्ता स्थापित सुरक्षा चिन्ताहरू र Prompt injection, tool poisoning, session hijacking, confused deputy आक्रमणहरू, token passthrough कमजोरीहरू, र dynamic capability modification जस्ता AI-विशिष्ट नयाँ खतरा सामना गर्छन्।

यस पाठले MCP कार्यान्वयनहरूमा सबैभन्दा महत्वपूर्ण सुरक्षा जोखिमहरू अन्वेषण गर्दछ—प्रमाणीकरण, अधिकृतता, अत्यधिक अनुमति, अप्रत्यक्ष prompt injection, सत्र सुरक्षा, confused deputy समस्याहरू, टोकन व्यवस्थापन र आपूर्ति श्रृंखला कमजोरिहरू समावेश गरेर। तपाईंले यी जोखिमहरू कम गर्न कार्यकारी नियन्त्रणहरू र उत्कृष्ट अभ्यासहरू सिक्नुहुनेछ, जसले Microsoft समाधानहरू जस्तै Prompt Shields, Azure Content Safety, र GitHub Advanced Security प्रयोग गरेर तपाईको MCP कार्यान्वयनलाई सुदृढ पार्दछ।

## सिकाइका उद्देश्यहरू

यस पाठको अन्त्यसम्म, तपाईं सक्षम हुनुहुनेछ:

- **MCP-विशिष्ट खतरा पहिचान गर्नुहोस्**: Prompt injection, tool poisoning, अत्यधिक अनुमति, session hijacking, confused deputy समस्याहरू, token passthrough कमजोरीहरू, र आपूर्ति श्रृंखला जोखिमहरू सहित MCP प्रणालीका अनौठा सुरक्षा जोखिमहरू चिन्ह लगाउनुहोस्
- **सुरक्षा नियन्त्रणहरू लागू गर्नुहोस्**: मजबुत प्रमाणीकरण, न्यूनतम विशेषाधिकार पहुँच, सुरक्षित टोकन व्यवस्थापन, सत्र सुरक्षा नियन्त्रणहरू, र आपूर्ति श्रृंखला प्रमाणिकरण सहित प्रभावकारी रोकथामहरू लागू गर्नुहोस्
- **Microsoft सुरक्षा समाधानहरू प्रयोग गर्नुहोस्**: MCP कार्यभार सुरक्षा लागि Microsoft Prompt Shields, Azure Content Safety, र GitHub Advanced Security बुझ्नुहोस् र लागू गर्नुहोस्
- **उपकरण सुरक्षा प्रमाणिकरण गर्नुहोस्**: उपकरण मेटाडाटा प्रमाणिकरणको महत्त्व बुझ्नुहोस्, dynamic परिवर्तनहरूको निगरानी गर्नुहोस्, र अप्रत्यक्ष prompt injection आक्रमणहरू विरुद्ध रक्षा गर्नुहोस्
- **उत्कृष्ट अभ्यासहरू समाहित गर्नुहोस्**: स्थापित सुरक्षा आधारहरू (सुरक्षित कोडिङ, सर्भर कडा पार्ने, शून्य ट्रस्ट) लाई MCP-विशिष्ट नियन्त्रणहरूसँग संयुक्त गरेर व्यापक सुरक्षा सुनिश्चित गर्नुहोस्

# MCP सुरक्षा वास्तुकला र नियन्त्रणहरू

आधुनिक MCP कार्यान्वयनहरूले तहगत सुरक्षा विधिहरू आवश्यक पर्छ जसले परम्परागत सफ्टवेयर सुरक्षा र AI-विशिष्ट खतराहरूलाई सम्बोधन गर्छ। द्रुत रूपमा विकसित हुँदै गरेको MCP विशिष्टता यसको सुरक्षा नियन्त्रणहरू परिपक्व गराउँदैछ, जसले उद्यम सुरक्षा वास्तुकलाहरू र स्थापित उत्कृष्ट अभ्यासहरूसँग राम्रोसँग जोड्न सक्षम बनाउँछ।

[Microsoft Digital Defense Report](https://aka.ms/mddr) को अनुसन्धानले देखाउँछ कि **९८% रिपोर्ट गरिएका सुरक्षा उल्लङ्घनहरू कडा सुरक्षा अभ्यासले रोक्न सकिन्छ**। सबैभन्दा प्रभावकारी सुरक्षा रणनीतिले आधारभूत सुरक्षा अभ्यासहरूलाई MCP-विशिष्ट नियन्त्रणहरूसँग संयोजन गर्छ—परीक्षित आधारभूत सुरक्षा उपायहरू नै समग्र सुरक्षा जोखिम घटाउन सर्वाधिक प्रभावकारी हुन्छन्।

## वर्तमान सुरक्षा परिदृश्य

> **सूचना:** यो जानकारी MCP सुरक्षा मापदण्डहरूलाई **फेब्रुअरी ५, २०२६** सम्मको प्रतिक्रिया गर्दछ, जुन **MCP Specification 2025-11-25** सँग मेल खान्छ। MCP प्रोटोकल द्रुत गतिमा विकास भइरहेको छ, र भविष्यका कार्यान्वयनहरूले नयाँ प्रमाणीकरण ढाँचाहरू र उन्नत नियन्त्रणहरू प्रस्तुत गर्न सक्छ। सधैं हालको [MCP Specification](https://spec.modelcontextprotocol.io/), [MCP GitHub रिपोजिटरी](https://github.com/modelcontextprotocol), र [सुरक्षा उत्कृष्ट अभ्यास कागजात](https://modelcontextprotocol.io/specification/2025-11-25/basic/security_best_practices) लाई परामर्श गर्नुस्।

> **भविष्यको दिशामा:** `2026-07-28` रिलिज क्यान्डिडेटले अधिक कडाइ गर्दैछ — ग्राहकहरूले प्रमाणीकरण जवाफहरूमा `iss` प्यारामिटरको प्रमाणीकरण गर्नुपर्नेछ (RFC 9207), Dynamic Client Registration मा OpenID Connect को `application_type` घोषणा गर्नुपर्नेछ, र दर्ता गरिएको क्रेडेन्सियलहरू जारी गर्ने प्रमाणीकरण सर्भरसँग बाँध्नुपर्नेछ। पूर्ण सूचिका लागि [MCP मा के परिवर्तन हुँदैछ: 2026-07-28 रिलीज क्यान्डिडेट](../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) हेर्नुहोस्।

## 🏔️ MCP सुरक्षा summit कार्यशाला (Sherpa)

**हात-मा-हात सिकाइ प्रशिक्षण**का लागि, हामी अत्यधिक सिफारिस गर्छौं **MCP सुरक्षा summit कार्यशाला** (Sherpa) - Microsoft Azure मा MCP सर्भरहरूलाई सुरक्षित गर्ने व्यावहारिक, मार्गनिर्देशित अभियान।

### कार्यशाला अवलोकन

[MCP सुरक्षा summit कार्यशाला](https://azure-samples.github.io/sherpa/) ले एक प्रमाणित "असुरक्षित → शोषण → समाधान → प्रमाणन" विधिबाट व्यवहारिक र कार्यान्वयन योग्य सुरक्षा प्रशिक्षण प्रदान गर्दछ। तपाईंले:

- **टुटाउन सिक्नुहोस्**: जान-बुझेर असुरक्षित सर्भरहरूमा कमजोरीहरूको अनुभव गर्नुहोस्
- **Azure-आधारित सुरक्षा प्रयोग गर्नुहोस्**: Azure Entra ID, Key Vault, API Management, र AI Content Safety प्रयोग गर्नुहोस्
- **Defence-in-Depth पालना गर्नुहोस्**: चरणबद्ध क्याम्पहरू हुँदै व्यापक सुरक्षा तहहरू निर्माण गर्नुहोस्
- **OWASP मापदण्डहरू पालना गर्नुहोस्**: प्रत्येक प्रविधिले [OWASP MCP Azure सुरक्षा गाइड](https://microsoft.github.io/mcp-azure-security-guide/) मा नक्सांकन गर्दछ
- **उत्पादन कोड प्राप्त गर्नुहोस्**: काम गर्ने, परीक्षण गरिएको कार्यान्वयन लिएर जानुहोस्

### MCP प्रमाणीकरणको विकास

MCP विशिष्टताले प्रमाणीकरण र अधिकृततामा महत्वपूर्ण रूपान्तरणहरू गरेको छ:

- **मूल दृष्टिकोण**: प्रारम्भिक विशिष्टताले विकासकर्ताहरूलाई कस्टम प्रमाणीकरण सर्भरहरू कार्यान्वयन गर्न आवश्यक पार्थ्यो, MCP सर्भरहरू सीधा OAuth 2.0 प्रमाणीकरण सर्भरको रूपमा प्रयोग गर्दै
- **भर्खरको मापदण्ड (2025-11-25)**: अद्यावधिक विशिष्टताले MCP सर्भरलाई बाह्य पहिचान प्रदायकहरू (जस्तै Microsoft Entra ID) लाई प्रमाणीकरण जिम्मेवारी सुम्पिने अनुमति दिन्छ, जसले सुरक्षा अवस्थालाई सुदृढ पार्दछ र कार्यान्वयन जटिलता कम गर्दछ
- **परिवहन तह सुरक्षा**: दुवै स्थानीय (STDIO) र रिमोट (Streamable HTTP) जडानहरूका लागि उपयुक्त प्रमाणीकरण ढाँचासहित सुरक्षित भौतिक उपकरण समर्थन सुधारिएको छ

## प्रमाणीकरण र अधिकृतता सुरक्षा

### वर्तमान सुरक्षा चुनौतीहरू

आधुनिक MCP कार्यान्वयनले प्रमाणीकरण र अधिकृततासम्बन्धी विभिन्न चुनौतीहरू सामना गर्छ:

### जोखिम र खतरा स्रोतहरू

- **गलत कन्फिगर गरिएको अधिकृतता तर्क**: MCP सर्भरमा गलत अधिकृतता कार्यान्वयनले संवेदनशील डाटा खुलाउन सक्छ र पहुँच नियन्त्रणहरू गलत लागू हुन सक्छन्
- **OAuth टोकन चोरी**: स्थानीय MCP सर्भर टोकन चोरीले आक्रमणकारीहरूलाई सर्भरहरूको नक्कली पहिचान र डाउनस्ट्रीम सेवाहरूमा पहुँच दिन सक्षम बनाउँछ
- **टोकन पास-थ्रु कमजोरीहरू**: गलत टोकन व्यवस्थापनले सुरक्षा नियन्त्रणहरू छलेर जिम्मेवारी अभाव सिर्जना गर्दछ
- **अत्यधिक अनुमति**: अधिक विशेषाधिकार प्राप्त MCP सर्भरहरूले न्यूनतम विशेषाधिकार सिद्धान्त उल्लङ्घन गर्छ र आक्रमण क्षेत्र बढाउँछ

#### टोकन पास-थ्रु: एक महत्वपूर्ण निगेटिभ ढाँचा

वर्तमान MCP अधिकृतता विशिष्टतामा **टोकन पास-थ्रु स्पष्ट रूपमा निषेध गरिएको छ** किनभने यसले गम्भीर सुरक्षा नतिजा निम्त्याउँछ:

##### सुरक्षा नियन्त्रण बाइपास
- MCP सर्भरहरू र डाउनस्ट्रीम API हरूले महत्वपूर्ण सुरक्षा नियन्त्रणहरू (दर सीमितीकरण, अनुरोध प्रमाणीकरण, ट्राफिक निगरानी) लागू गर्दछ, जुन उचित टोकन प्रमाणीकरणमा निर्भर छन्
- ग्राहकदेखि API सिधा टोकन उपयोगले यी आवश्यक सुरक्षा संरक्षणहरूलाई छ्लेका छन्, जसले सुरक्षा वास्तुकलालाई कमजोर बनाउँछ

##### जिम्मेवारी र अडिट चुनौतीहरू  
- MCP सर्भरहरूले उप-प्रदाता द्वारा जारी गरिएका टोकनहरू प्रयोग गर्ने ग्राहकहरूलाई फरक छुट्याउन सक्दैनन्, जसले अडिट ट्रेलहरू तोड्छ
- डाउनस्ट्रीम स्रोत सर्भरका लगहरूले वास्तविक MCP सर्भर मध्यस्थकर्ताको सट्टा भ्रमपूर्ण अनुरोध उत्पत्तिहरू देखाउँछन्
- घटना अनुसन्धान र अनुपालन अडिटिंग अत्यन्त जटिल हुन्छ

##### डाटा चोरी जोखिमहरू
- अप्रमाणित टोकन दावीहरूले चोरी गरिएका टोकनहरूसँग खराब लक्ष्यहरूलाई MCP सर्भरहरूलाई प्रोक्सीको रूपमा प्रयोग गर्न सक्षम पार्दछ
- विश्वास सिमा उल्लङ्घनले अनधिकृत पहुँच ढाँचा निम्त्याउँछ जसले उद्देश्यित सुरक्षा नियन्त्रणहरू छलेको छ

##### बहु-सेवा आक्रमण भेक्टरहरू
- कम्प्रोमाइज गरिएको टोकनहरू बहु सेवाले स्वीकार गर्दा प्रणालीहरू बीच तिरछो आन्दोलन हुन्छ
- सेवा बीचको विश्वास अनुमानहरू टोकन उत्पत्ति प्रमाणिकरण गर्न नसक्दा उल्लङ्घन हुन सक्छ

### सुरक्षा नियन्त्रणहरू र रोकथामहरू

**महत्वपूर्ण सुरक्षा आवश्यकताहरू:**

> **अनिवार्य:** MCP सर्भरहरूले **कुनै पनि टोकन स्वीकार गर्नु हुँदैन जुन स्पष्ट रूपमा त्यस MCP सर्भरका लागि जारी गरिएको छैन**

#### प्रमाणीकरण र अधिकृतता नियन्त्रणहरू

- **गम्भीर अधिकृतता समीक्षा**: MCP सर्भर अधिकृतता तर्कको व्यापक अडिट गर्नुहोस् कि केवल लक्षित प्रयोगकर्ता र ग्राहकले मात्र संवेदनशील स्रोतहरू पहुँच गर्न सकून्
  - **कार्यान्वयन मार्गदर्शन**: [Azure API Management as Authentication Gateway for MCP Servers](https://techcommunity.microsoft.com/blog/integrationsonazureblog/azure-api-management-your-auth-gateway-for-mcp-servers/4402690)
  - **पहिचान एकीकरण**: [Microsoft Entra ID प्रयोग गर्दै MCP सर्भर प्रमाणीकरण](https://den.dev/blog/mcp-server-auth-entra-id-session/)

- **सुरक्षित टोकन व्यवस्थापन**: [Microsoft को टोकन प्रमाणीकरण र जीवनचक्र उत्कृष्ट अभ्यासहरू](https://learn.microsoft.com/en-us/entra/identity-platform/access-tokens) लागू गर्नुहोस्
  - टोकन दर्शक दावीहरू MCP सर्भर पहिचानसँग मेल खान्छ भनेर प्रमाणिकरण गर्नुहोस्
  - उचित टोकन घुमाउने र समाप्ति नीतिहरू लागू गर्नुहोस्
  - टोकन पुन: प्ले आक्रमण र अप्रमाणित प्रयोग रोक्नुहोस्

- **सुरक्षित टोकन भण्डारण**: टोकनलाई विश्राम र ट्रान्सिट दुवैमा सङ्केतांकित गरेर सुरक्षित भण्डारण गर्नुहोस्
  - **उत्कृष्ट अभ्यासहरू**: [Secure Token Storage and Encryption Guidelines](https://youtu.be/uRdX37EcCwg?si=6fSChs1G4glwXRy2)

#### पहुँच नियन्त्रण कार्यान्वयन

- **न्यूनतम विशेषाधिकारको सिद्धान्त**: MCP सर्भरलाई चाहिने न्यूनतम अनुमति मात्र दिनुहोस्
  - नियमित अनुमति समीक्षा र अद्यावधिक गरी विशेषाधिकार क्रिपलाई रोक्नुहोस्
  - **Microsoft कागजात**: [Secure Least-Privileged Access](https://learn.microsoft.com/entra/identity-platform/secure-least-privileged-access)

- **भूमिका-आधारित पहुँच नियन्त्रण (RBAC)**: सूक्ष्म भूमिकाहरू लागू गर्नुहोस्
  - भूमिकाहरूलाई विशिष्ट स्रोत र क्रियाहरूमा कडाइका साथ सीमित गर्नुहोस्
  - अनावश्यक वा व्यापक अनुमति जसले आक्रमण क्षेत्र बढाउँछ त्यस्ता अनुमति टार्नुहोस्

- **लगातार अनुमति निगरानी**: पहुँच अडिट र निगरानी निरन्तर गर्नुहोस्
  - अनियमितताहरूको लागि अनुमति उपयोग ढाँचाहरूको निगरानी गर्नुहोस्
  - अत्यधिक वा प्रयोग नभएका विशेषाधिकार तुरुन्तै जाँच र सुधार गर्नुहोस्

## AI-विशिष्ट सुरक्षा खतरा

### Prompt Injection र उपकरण हेरफेर आक्रमणहरू

आधुनिक MCP कार्यान्वयनहरूले जटिल AI-विशिष्ट आक्रमण भेक्टरहरूको सामना गर्छ जसलाई परम्परागत सुरक्षा उपायहरूले पूर्ण रूपमा सम्बोधन गर्न सक्दैनन्:

#### **अप्रत्यक्ष Prompt Injection (क्रस-डोमेन Prompt Injection)**

**अप्रत्यक्ष Prompt Injection** MCP सक्षम AI प्रणालीहरूमा सबैभन्दा गम्भीर कमजोरीहरू मध्ये एक हो। आक्रमणकारीहरूले बाह्य सामग्रीमा प्रमादपूर्ण निर्देशनहरू समावेश गर्छन्—कागजातहरू, वेब पृष्ठहरू, इमेलहरू, वा डाटास्रोतहरू—जसलाई AI प्रणालीहरूले वैध कमाण्डको रूपमा प्रशोधन गर्छन्।

**आक्रमण परिदृश्यहरू:**
- **कागजात-आधारित इंजेक्शन**: प्रशोधित कागजातहरूमा लुकेको प्रमादपूर्ण निर्देशनले अवाञ्छित AI क्रियाकलाप चालु गर्दछ
- **वेब सामग्री शोषण**: संक्रमित वेब पृष्ठहरूमा सम्मिलित prompt हरू जुन स्क्र्यापिङ गर्दा AI व्यवहारलाई करेसा गर्छन्
- **इमेल-आधारित आक्रमणहरू**: इमेलहरूमा लुकेका प्रमादपूर्ण prompt ले AI सहायकहरूलाई जानकारी चुहाउन वा अनधिकृत क्रियाकलाप गर्न बाध्य पार्छ
- **डाटा स्रोत प्रदूषण**: संक्रमित डेटाबेस वा API हरू जसले AI प्रणालीहरूलाई दूषित सामग्री सेवा गर्छन्

**वास्तविक विश्व प्रभाव**: यी आक्रमणहरूले डाटा चोरी, गोपनीयता उल्लङ्घन, हानिकारक सामग्री निर्माण, र प्रयोगकर्ता अन्तरक्रियाको मनिपुलेसन ल्याउन सक्छ। विस्तृत विश्लेषणका लागि हेर्नुहोस् [Prompt Injection in MCP (Simon Willison)](https://simonwillison.net/2025/Apr/9/mcp-prompt-injection/)।

![Prompt Injection आक्रमणको चित्रण](../../../translated_images/ne/prompt-injection.ed9fbfde297ca877.webp)

#### **उपकरण विषाक्तता आक्रमणहरू**

**उपकरण विषाक्तता** MCP उपकरणहरूलाई परिभाषित गर्ने मेटाडाटा लक्षित गर्दछ, जसले LLMs ले उपकरण विवरण र प्यारामिटरहरू कसरी व्याख्या गर्छन् र कार्यान्वयन निर्णय गर्छन्, त्यसलाई दुरुपयोग गर्छ।

**आक्रमण विधिहरू:**
- **मेटाडाटा हेरफेर**: आक्रमणकारीहरूले उपकरण विवरण, प्यारामिटर परिभाषा, वा उपयोग उदाहरणहरूमा प्रमादपूर्ण निर्देशनहरू समावेश गर्छन्
- **अदृश्य निर्देशनहरू**: उपकरण मेटाडाटामा लुकेका प्रमादपूर्ण prompt जसलाई AI मोडेलहरूले प्रशोधन गर्छन् तर मानिस प्रयोगकर्तालाई देखिदैन
- **डाइनामिक उपकरण परिवर्तन ("रग पुल्स")**: प्रयोगकर्ताहरूले स्वीकृत गरेका उपकरणहरू पछि परिवर्तन गरिन्छ जसले प्रयोगकर्ता अनजानमा प्रमादपूर्ण कामहरू गर्दछ
- **प्यारामिटर इंजेक्शन**: उपकरण प्यारामिटर स्किमामा समाहित प्रमादपूर्ण सामग्रीले मोडेल व्यवहारलाई प्रभावित पार्दछ


**होस्ट गरिएको सर्भर जोखिमहरू**: रिमोट MCP सर्भरहरूले उच्च जोखिमहरू प्रस्तुत गर्छन् किनभने उपकरण परिभाषाहरू आरम्भिक प्रयोगकर्ता स्वीकृतिपछि अपडेट गर्न सकिन्छ, जसले पहिले सुरक्षित उपकरणहरू दुष्ट प्रवृत्तिमा परिणत हुने स्थिति सिर्जना गर्दछ। व्यापक विश्लेषणको लागि हेर्नुहोस् [Tool Poisoning Attacks (Invariant Labs)](https://invariantlabs.ai/blog/mcp-security-notification-tool-poisoning-attacks)।

![Tool Injection Attack Diagram](../../../translated_images/ne/tool-injection.3b0b4a6b24de6bef.webp)

#### **अतिरिक्त AI आक्रमण मार्गहरू**

- **क्रस-डोमेन प्रॉम्प्ट इन्जेक्शन (XPIA)**: सुरक्षा नियन्त्रण पार गर्न विभिन्न डोमेनहरूबाट सामग्रीलाई उपयोग गर्ने-जटिल आक्रमणहरू
- **डायनामिक क्षमता संशोधन**: उपकरण क्षमताहरूमा वास्तविक-समय परिवर्तनहरू जसले प्रारम्भिक सुरक्षा मूल्याङ्कनबाट बच्छ
- **सन्दर्भ विन्डो विषाक्तता**: ठूलो सन्दर्भ विन्डोहरूलाई दुष्ट निर्देशनहरू लुकाउन मैनिपुलेट गर्ने आक्रमणहरू
- **मोडेल भ्रम आक्रमणहरू**: मोडेल सीमितताहरूलाई दुरुपयोग गरी अप्रत्याशित वा असुरक्षित व्यवहार सिर्जना गर्ने


### AI सुरक्षा जोखिम प्रभाव

**उच्च-प्रभाव परिणामहरू:**
- **डेटा चोरी**: संवेदनशील उद्यम वा व्यक्तिगत डेटामा अनधिकृत पहुँच र चोरी
- **गोपनीयता उल्लङ्घनहरू**: व्यक्तिगत पहिचानयोग्य जानकारी (PII) र गोप्य व्यवसायिक डाटाको खुलासा  
- **प्रणाली मैनिपुलेसन**: महत्वपूर्ण प्रणालीहरू र कार्यप्रवाहहरूमा अनपेक्षित संशोधनहरू
- **प्रमाणीकरण चोरी**: प्रमाणीकरण टोकन र सेवा प्रमाणहरूमा कम्पोमाइज
- **पार्श्वगत गति**: व्यापक नेटवर्क आक्रमणको लागि कम्प्रोमाइज्ड AI प्रणालीहरूको रूपमा प्रयोग

### माइक्रोसफ्ट AI सुरक्षा समाधानहरू

#### **AI प्रॉम्प्ट शिल्डहरू: इन्जेक्शन आक्रमणहरू विरुद्ध उन्नत सुरक्षा**

माइक्रोसफ्ट **AI प्रॉम्प्ट शिल्डहरू** विभिन्न सुरक्षा तहहरूमार्फत प्रत्यक्ष र अप्रत्यक्ष प्रॉम्प्ट इन्जेक्शन आक्रमणहरूबाट व्यापक सुरक्षा प्रदान गर्छन्:

##### **मूल सुरक्षा संयन्त्रहरू:**

1. **उन्नत पत्ता लगाउने र फिल्टर गर्ने**
   - मेसिन लर्निङ एल्गोरिदमहरू र NLP प्राविधिहरूले बाह्य सामग्रीमा दुष्ट निर्देशनहरू पत्ता लगाउँछन्
   - कागजातहरू, वेब पेजहरू, इमेलहरू, र डेटा स्रोतहरूको वास्तविक-समय विश्लेषणमा सम्भावित खतरा 
   - वैध र दुष्ट प्रॉम्प्ट ढाँचाहरूको सन्दर्भगत समझ

2. **स्पटलाइटिङ प्रविधिहरू**  
   - विश्वास गरिएको प्रणाली निर्देशन र सम्भावित कम्प्रोमाइज बाह्य इनपुटहरू बीच फरक गर्छ
   - पाठ रूपान्तरण विधिहरूले मोडल प्रासंगिकता बढाउँछन् र दुष्ट सामग्री अलग गर्छन्
   - AI प्रणालीहरूलाई उचित निर्देशन पदानुक्रम कायम राख्न र इन्जेक्ट गरिएको आदेशलाई बेवास्ता गर्न मद्दत गर्छ

3. **डिलिमिटर र डाटामार्किङ प्रणालीहरू**
   - विश्वास गरिएको प्रणाली सन्देश र बाह्य इनपुट पाठ बीच स्पष्ट सीमाना परिभाषित गर्दछ
   - विशेष मार्करहरूले विश्वास गरिएको र अविश्वसनीय डेटा स्रोतहरू बीच सीमाना हाइलाइट गर्छ
   - स्पष्ट अलगावले निर्देशन भ्रम र अनधिकृत आदेश कार्यान्वयन रोक्छ

4. **निरन्तर खतरा बुद्धिमत्ता**
   - माइक्रोसफ्टले नयाँ आक्रमण ढाँचाहरू निगरानी र सुरक्षा सुधार गर्दछ
   - नयाँ इन्जेक्शन प्रविधि र आक्रमण मार्गहरू खोजीका लागि सक्रिय खतरा शिकार
   - विकासशील खतरा विरुद्ध प्रभावकारिता कायम राख्न नियमित सुरक्षा मोडल अपडेटहरू

5. **एजर कन्टेन्ट सेफ्टी इंटीग्रेशन**
   - समग्र Azure AI कन्टेन्ट सेफ्टी सूटको भाग
   - जेलब्रेक प्रयास, हानिकारक कन्टेन्ट, र सुरक्षा नीति उल्लङ्घनहरूको लागि अतिरिक्त पत्ता लगाउने
   - AI अनुप्रयोग घटकहरूमा एकीकृत सुरक्षा नियन्त्रण

## उन्नत MCP सुरक्षा खतरा

### सत्र अपहरण कमजोरीहरू

**सत्र अपहरण** राज्यपूर्ण MCP कार्यान्वयनहरूमा एक गम्भीर आक्रमण मार्ग हो जहाँ अनधिकृत पक्षहरूले वैध सत्र पहिचानकर्ता प्राप्त गरी दुरुपयोग गर्छन् ताकि क्लाइन्टको नक्कल गरी अनधिकृत कार्यहरू गर्न सकून्।

#### **आक्रमण परिदृश्य र जोखिमहरू**

- **सत्र अपहरण प्रॉम्प्ट इन्जेक्शन**: चोरी गरिएका सत्र ID भएका आक्रमणकारीहरूले सर्भरमा दुष्ट घटनाहरू इन्जेक्ट गर्छन् जुन सत्र स्थिति साझा गर्छन्, सम्भावित रूपमा हानिकारक कार्यहरू ट्रिगर गर्न वा संवेदनशील डाटा पहुँच गर्न सक्छन्
- **प्रत्यक्ष नक्कली बनावट**: चोरी गरिएका सत्र ID ले प्रमाणीकरण बाइपास गर्ने प्रत्यक्ष MCP सर्भर कल सक्षम पार्छ, आक्रमणकारीलाई वैध प्रयोगकर्ताको रूपमा व्यवहार गरिन्छ
- **कम्प्रोमाइज्ड पुनः सुरु हुने प्रवाहहरू**: आक्रमणकारीहरूले अनुरोधहरू अग्रिम समाप्त गर्न सक्छन् जसले वैध क्लाइन्टहरूलाई सम्भावित दुष्ट सामग्रीसहित पुनः सुरु गर्न बाध्य पार्छ

#### **सत्र व्यवस्थापनका लागि सुरक्षा नियन्त्रणहरू**

**महत्त्वपूर्ण आवश्यकता:**
- **प्राधिकरण प्रमाणीकरण**: MCP सर्भरहरूले सबै इनबाउन्ड अनुरोधहरूको प्रमाणीकरण गर्नुपर्छ र सत्रमा आधारित प्रमाणीकरणमा निर्भर हुनु हुँदैन
- **सुरक्षित सत्र उत्पादन**: क्रिप्टोग्राफिक रूपमा सुरक्षित, गैर-निर्धारित सत्र ID प्रयोग गर्नुपर्छ जुन सुरक्षा यादृच्छिक सङ्ख्या जनरेटरबाट उत्पन्न हुन्छ
- **प्रयोगकर्ता-विशिष्ट बाइन्डिङ**: सत्र ID लाई `<user_id>:<session_id>` जस्तै फारम्याटमा प्रयोगकर्ता-विशिष्ट जानकारीसँग बाँध्ने, ताकि क्रस-प्रयोगकर्ता सत्र दुरुपयोग रोकियोस्
- **सत्र जीवनचक्र व्यवस्थापन**: उपयुक्त म्याद समाप्ति, रोटेसन, र अमान्यकरण कार्यान्वयन गरेर जोखिम सिमित गर्ने
- **ट्रान्सपोर्ट सुरक्षा**: सबै सञ्चारका लागि अनिवार्य HTTPS ताकि सत्र ID कैद हुन नपाओस्

### भ्रमित प्रतिनिधि समस्या

**भ्रमित प्रतिनिधि समस्या** तब हुन्छ जब MCP सर्भरहरूले क्लाइन्ट र तेस्रो पक्ष सेवाहरू बीच प्रमाणीकरण प्रोक्सीको रूपमा कार्य गर्छन्, जसले स्थिर क्लाइन्ट ID दुरुपयोग मार्फत प्राधिकरण बाइपासका अवसरहरू सिर्जना गर्दछ।

#### **आक्रमण तंत्रहरू र जोखिमहरू**

- **कुकी-आधारित सहमति बाइपास**: अघिल्लो प्रयोगकर्ता प्रमाणीकरणले सहमति कुकीहरू सिर्जना गर्छ जसलाई आक्रमणकारीहरूले रिडाइरेक्ट URI संग बनाइएको दुष्ट प्राधिकरण अनुरोधमार्फत दुरुपयोग गर्छन्
- **प्राधिकरण कोड चोरी**: विद्यमान सहमति कुकीहरूले प्राधिकरण सर्भरलाई सहमति स्क्रिनहरू स्किप गरी कोड आक्रमणकारीको नियन्त्रण अन्तिम पोइन्टमा पठाउन सक्छ  
- **अनधिकृत API पहुँच**: चोरी गरिएका प्राधिकरण कोडहरूले बिना स्पष्ट स्वीकृतिको टोकन विनिमय र प्रयोगकर्ता नक्कली बनावट सम्भव बनाउँछ

#### **निवारण रणनीतिहरू**

**अनिवार्य नियन्त्रणहरू:**
- **स्पष्ट सहमति आवश्यकताहरू**: स्थिर क्लाइन्ट ID प्रयोग गर्ने MCP प्रोक्सी सर्भरहरूले हरेक गतिशील रूपमा दर्ता गरिएको क्लाइन्टको लागि प्रयोगकर्ता सहमति लिनुपर्छ
- **OAuth 2.1 सुरक्षा कार्यान्वयन**: सबै प्राधिकरण अनुरोधका लागि PKCE (Proof Key for Code Exchange) सहित आधुनिक OAuth सुरक्षा अभ्यासहरू पालना गर्नुपर्छ
- **कडा क्लाइन्ट प्रमाणीकरण**: दुरुपयोग रोक्न रिडाइरेक्ट URI र क्लाइन्ट पहिचानकर्ताहरूको कडाइका साथ प्रमाणीकरण कार्यान्वयन गर्नुपर्छ

### टोकन पासथ्रू कमजोरीहरू  

**टोकन पासथ्रू** एक स्पष्ट विरोधी-नमूना हो जहाँ MCP सर्भरहरूले क्लाइन्ट टोकनहरू उचित प्रमाणीकरण बिना स्वीकार्छन् र तिनीहरूलाई डाउनस्ट्रीम API हरूसम्म अग्रेषित गर्छन्, जसले MCP प्राधिकरण मापदण्डहरूको उल्लङ्घन गर्दछ।

#### **सुरक्षा निहितार्थहरू**

- **नियन्त्रण बाइपास**: प्रत्यक्ष क्लाइन्ट-देखि-API टोकन प्रयोगले महत्वपूर्ण दर सीमित, प्रमाणीकरण, र मोनिटरिङ नियन्त्रणहरू पार गर्छ
- **परीक्षण ट्रेल भ्रष्टाचार**: माथिल्लो स्तरमा जारी गरिएका टोकनहरूले क्लाइन्ट पहिचान असम्भव बनाउँछ, जसले घटना अनुसन्धान क्षमता भंग गर्छ
- **प्रोक्सी-आधारित डेटा चोरी**: अप्रमाणीकृत टोकनहरूलाई दुष्ट पात्रहरूले अनधिकृत डेटा पहुँचको लागि सर्भरहरूलाई प्रोक्सीको रूपमा प्रयोग गर्न सक्षम पार्दछ
- **विश्वास सीमा उल्लङ्घन**: डाउनस्ट्रीम सेवाहरूको विश्वास अनुमानी टोकन उत्पत्तिको प्रमाणीकरण नगर्दा उल्लङ्घन हुन सक्छ
- **बहु-सेवा आक्रमण विस्तार**: कम्प्रोमाइज्ड टोकनहरू विभिन्न सेवाहरूमा स्वीकार गरिएमा पार्श्वगत गतिलाई अनुमति दिन्छ

#### **आवश्यक सुरक्षा नियन्त्रणहरू**

**बातचीत योग्य छैन:**
- **टोकन प्रमाणीकरण**: MCP सर्भरहरूले स्पष्ट रूपमा MCP सर्भरका लागि जारी गरिएको नभएको टोकनहरू स्वीकार गर्नु हुँदैन
- **दर्शक प्रमाणीकरण**: सधैं टोकनको दर्शक दाबीहरू MCP सर्भरको पहिचानसँग मेल खान्छ कि भनी प्रमाणीकरण गर्नुपर्छ
- **उचित टोकन जीवनचक्र**: छोटो-आयु भएका पहुँच टोकनहरू सुरक्षित रोटेसन अभ्यासहरूसहित कार्यान्वयन गर्नुपर्छ


## AI प्रणालीहरूको आपूर्ति श्रृंखला सुरक्षा

आपूर्ति श्रृंखला सुरक्षा परम्परागत सफ्टवेयर निर्भरता भन्दा बाहिर विकसित भई सम्पूर्ण AI पारिस्थितिकी तन्त्र समेट्न पुगेको छ। आधुनिक MCP कार्यान्वयनहरूले सबै AI सम्बन्धित घटकहरूलाई कडाइका साथ प्रमाणीकरण गर्ने र निगरानी गर्नुपर्छ, किनभने प्रत्येकले प्रणाली अखण्डता जोखिममा पार्न सक्ने सम्भावित कमजोरीहरू प्रस्तुत गर्दछ।

### विस्तार गरिएको AI आपूर्ति श्रृंखला घटकहरू

**परम्परागत सफ्टवेयर निर्भरता:**
- खुला स्रोत पुस्तकालयहरू र फ्रेमवर्कहरू
- कन्टेनर इमेजहरू र आधार प्रणालीहरू  
- विकास उपकरणहरू र निर्माण पाइपलाइनहरू
- पूर्वाधार घटकहरू र सेवाहरू

**AI-विशिष्ट आपूर्ति श्रृंखला तत्वहरू:**
- **फाउन्डेसन मोडेलहरू**: विभिन्न प्रदायकहरूबाट पूर्व-प्रशिक्षित मोडेलहरू जसको उत्पत्तिको प्रमाणीकरण आवश्यक छ
- **एम्बेडिङ सेवाहरू**: बाह्य भेक्टराइजेसन र सेम्यान्टिक खोज सेवाहरू
- **सन्दर्भ प्रदायकहरू**: डेटा स्रोतहरू, ज्ञान आधारहरू, र कागजात रिपोजिटरीहरू  
- **तेस्रो पक्ष API हरू**: बाह्य AI सेवा, ML पाइपलाइनहरू, र डेटा प्रशोधन अन्तिम पोइन्टहरू
- **मोडेल अवशेषहरू**: तौलहरू, कन्फिगरेसनहरू, र सूक्ष्म-समायोजित मोडेल भेरियन्टहरू
- **प्रशिक्षण डेटा स्रोतहरू**: मोडेल प्रशिक्षण र सूक्ष्म-समायोजनका लागि प्रयोग गरिएका डाटासेटहरू

### समग्र आपूर्ति श्रृंखला सुरक्षा रणनीति

#### **घटक प्रमाणीकरण र विश्वास**
- **उत्पत्ति प्रमाणीकरण**: सबै AI घटकहरूको उत्पत्ति, लाइसेन्सिङ्ग, र अखण्डता एकीकरणअघि प्रमाणीकरण गर्ने
- **सुरक्षा मूल्यांकन**: मोडेल, डेटा स्रोतहरू, र AI सेवाहरूको लागि कमजोरी स्क्यान र सुरक्षा समीक्षा सञ्चालन गर्ने
- **प्रतिष्ठा विश्लेषण**: AI सेवा प्रदायकहरूको सुरक्षा इतिहास र अभ्यास मूल्याङ्कन गर्ने
- **पालना प्रमाणीकरण**: सबै घटकहरू संस्थागत सुरक्षा र नियम अनुपालन आवश्यकताहरू पूरा गर्छन् भन्ने सुनिश्चित गर्ने

#### **सुरक्षित परिनियोजन पाइपलाइनहरू**  
- **स्वचालित CI/CD सुरक्षा**: स्वचालित परिनियोजन पाइपलाइनहरूभरि सुरक्षा स्क्यानिङ्ग एकीकृत गर्ने
- **अवशेष अखण्डता**: सबै परिनियोजित अवशेषहरूको क्रिप्टोग्राफिक प्रमाणीकरण कार्यान्वयन गर्ने (कोड, मोडेल, कन्फिगरेसनहरू)
- **स्टेज्ड परिनियोजन**: प्रत्येक चरणमा सुरक्षा प्रमाणीकरणसहित प्रगतिशील परिनियोजन रणनीतिहरू प्रयोग गर्ने
- **विश्वसनीय अवशेष रिपोजिटरीहरू**: प्रमाणीकरण गरिएका, सुरक्षित अवशेष रजिस्ट्रीहरू र रिपोजिटरीहरूबाट मात्र परिनियोजन गर्ने

#### **निरन्तर निगरानी र प्रतिक्रिया**
- **निर्भरता स्क्यानिङ्ग**: सबै सफ्टवेयर र AI कम्पोनेन्ट निर्भरता लागि कमजोरी निगरानी निरन्तर चलाउने
- **मोडेल निगरानी**: मोडेल व्यवहार, प्रदर्शन विचलन, र सुरक्षा अनियमितताहरूको निरन्तर मूल्याङ्कन
- **सेवा स्वास्थ्य ट्र्याकिङ्ग**: बाह्य AI सेवाहरूको उपलब्धता, सुरक्षा घटना, र नीति परिवर्तनहरूको निगरानी गर्ने
- **खतरा बुद्धिमत्ता एकीकरण**: AI र ML सुरक्षा जोखिमहरूको लागि खतरा फिडहरू समावेश गर्ने

#### **प्रवेश नियन्त्रण र न्यूनतम सशक्तता**
- **घटक-स्तरीय अनुमतिहरू**: व्यवसायलाई आवश्यक पर्ने आधारमा मोडेल, डेटा, र सेवाहरूमा पहुँच सीमित गर्ने
- **सेवा खाता व्यवस्थापन**: न्यूनतम आवश्यक अनुमतिहरू सहित समर्पित सेवा खाता कार्यान्वयन गर्ने
- **नेटवर्क खण्डीकरण**: AI घटकहरू अलग गर्ने र सेवाहरूबीच नेटवर्क पहुँच सीमित गर्ने
- **API गेटवे नियन्त्रणहरू**: बाह्य AI सेवाहरूको पहुँच नियन्त्रण र निगरानीका लागि केन्द्रित API गेटवेहरू प्रयोग गर्ने

#### **घटना प्रतिक्रिया र पुनर्प्राप्ति**
- **छिटो प्रतिक्रिया प्रक्रिया**: कम्प्रोमाइज्ड AI घटकहरूको प्याचिङ्ग वा प्रतिस्थापनका लागि स्थापित प्रक्रिया
- **प्रमाणीपत्र घुमाउने**: गोप्य, API कुञ्जीहरू, र सेवा प्रमाणपत्रहरू स्वचालित रूपमा घुमाउने प्रणाली
- **रोलब्याक क्षमता**: AI घटकहरूको पहिलेको ज्ञात-सही संस्करणमा छिटो फिर्ता जान सक्ने क्षमता
- **आपूर्ति श्रृंखला उल्लङ्घन पुनर्प्राप्ति**: माथिल्लो स्तरीय AI सेवा कम्प्रोमाइजमा प्रतिक्रिया जनाउने विशेष प्रक्रिया

- **निर्भरता स्क्यानिङ्ग**: खुला स्रोत निर्भरताहरू र पुस्तकालयहरूको कमजोरी मूल्याङ्कन
- **CodeQL विश्लेषण**: सुरक्षा कमजोरी र कोडिंग समस्याहरूको लागि स्थिर कोड विश्लेषण
- **आपूर्ति श्रृंखला अन्तर्दृष्टि**: निर्भरता स्वास्थ्य र सुरक्षा स्थिति स्पष्ट पार्ने

**Azure DevOps & Azure Repos एकीकृत:**
- माइक्रोसफ्ट विकास प्लेटफर्महरूमा सहज सुरक्षा स्क्यानिङ्ग एकीकरण
- AI कार्यभारहरूको लागि Azure पाइपलाइनहरूमा स्वचालित सुरक्षा जाँचहरू
- सुरक्षित AI घटक परिनियोजनका लागि नीति कार्यान्वयन

**माइक्रोसफ्ट आन्तरिक अभ्यासहरू:**
माइक्रोसफ्टले सबै उत्पादनहरूमा व्यापक आपूर्ति श्रृंखला सुरक्षा अभ्यासहरू कार्यान्वयन गर्दछ। [The Journey to Secure the Software Supply Chain at Microsoft](https://devblogs.microsoft.com/engineering-at-microsoft/the-journey-to-secure-the-software-supply-chain-at-microsoft/) मा प्रमाणित दृष्टिकोणहरू जान्नुहोस्।


## आधारभूत सुरक्षा उत्तम अभ्यासहरू

MCP कार्यान्वयनहरूले तपाईंको संगठनको विद्यमान सुरक्षा अवस्थाबाट अनुवांशित र निर्माण गर्छ। आधारभूत सुरक्षा अभ्यासहरूलाई सुदृढ पार्दा AI प्रणालीहरू र MCP परिनियोजनहरूको समग्र सुरक्षा महत्वपूर्ण रूपमा सुधारिन्छ।

### मूल सुरक्षा आधारहरू

#### **सुरक्षित विकास अभ्यासहरू**
- **OWASP अनुपालन**: [OWASP Top 10](https://owasp.org/www-project-top-ten/) वेब अनुप्रयोग कमजोरीहरू विरुद्ध सुरक्षा
- **AI-विशेष सुरक्षा**: [OWASP Top 10 for LLMs](https://genai.owasp.org/download/43299/?tmstv=1731900559) का लागि नियन्त्रण कार्यान्वयन गर्ने
- **सुरक्षित गोप्य व्यवस्थापन**: टोकन, API कुञ्जीहरू, र संवेदनशील कन्फिगरेसन डेटा लागि समर्पित भल्ट प्रयोग गर्ने
- **अंत-देखि-अंत एन्क्रिप्शन**: सबै अनुप्रयोग घटकहरू र डेटा प्रवाहहरूमा सुरक्षित सञ्चार लागू गर्ने
- **इनपुट प्रमाणीकरण**: सबै प्रयोगकर्ता इनपुट, API प्यारामिटरहरू, र डेटा स्रोतहरू कडाइका साथ प्रमाणीकरण गर्ने

#### **पूर्वाधार कडा बनाउने**
- **बहु-कारक प्रमाणीकरण**: सबै प्रशासनिक र सेवा खाताहरूका लागि अनिवार्य MFA
- **प्याच व्यवस्थापन**: अपरेटिङ प्रणाली, फ्रेमवर्क, र निर्भरता लागि स्वचालित, तत्काल प्याचिङ्ग
- **पहिचान प्रदायक एकीकरण**: उद्यम पहिचान प्रदायकहरू (Microsoft Entra ID, Active Directory) मार्फत केन्द्रित पहिचान व्यवस्थापन
- **नेटवर्क खण्डीकरण**: पार्श्वगत गति सम्भावना सीमित गर्न MCP कम्पोनेन्टहरूको तार्किक पृथकीकरण
- **न्यूनतम सशक्तता सिद्धान्त**: सबै प्रणाली कम्पोनेन्टहरू र खाताहरूका लागि न्यूनतम आवश्यक अनुमतिहरू मात्र दिनुहोस्

#### **सुरक्षा निगरानी र पत्ता लगाउने**
- **सम्पूर्ण लिगिङ्ग**: MCP क्लाइन्ट-सर्भर अन्तरक्रियाहरू सहित AI अनुप्रयोग गतिविधिहरूको विस्तृत लिगिङ्ग गर्ने
- **SIEM एकीकरण**: असामान्य पत्ता लगाउन केन्द्रित सुरक्षा सूचना र घटना व्यवस्थापन
- **व्यवहार विश्लेषण**: प्रणाली र प्रयोगकर्ता व्यवहारमा असामान्य ढाँचाहरू पत्ता लगाउन AI-समर्थित निगरानी
- **खतरा बुद्धिमत्ता**: बाह्य खतरा फीड र सम्झौता सूचकांकहरू (IOC) को एकीकरण
- **घटना प्रतिक्रिया**: सुरक्षा घटना पत्ता लगाउन, प्रतिक्रिया दिन, र पुनर्प्राप्ति गर्न सुव्यवस्थित प्रक्रिया

#### **जीरो ट्रस्ट आर्किटेक्चर**
- **कहिल्यै विश्वास नगर, सधैं प्रमाणीकरण**: प्रयोगकर्ता, उपकरणहरू, र नेटवर्क कनेक्शनहरूको निरन्तर प्रमाणीकरण
- **माइक्रो-सेग्मेन्टेसन**: व्यक्तिगत कार्यभारहरू र सेवाहरूलाई अलग गर्ने सूक्ष्म नेटवर्क नियन्त्रणहरू
- **पहिचान-कन्द्रित सुरक्षा**: नेटवर्क स्थान भन्दा प्रमाणीक्रित पहिचानहरूमा आधारित सुरक्षा नीति
- **निरन्तर जोखिम मूल्याङ्कन**: वर्तमान सन्दर्भ र व्यवहारका आधारमा गतिशील सुरक्षा मूल्याङ्कन
- **सशर्त पहुँच**: जोखिम कारक, स्थान, र उपकरण विश्वास आधारमा अनुकूलित पहुँच नियन्त्रण

### उद्यम एकीकरण ढाँचाहरू

#### **माइक्रोसफ्ट सुरक्षा पारिस्थितिकी तन्त्र एकीकरण**
- **Microsoft Defender for Cloud**: समग्र क्लाउड सुरक्षा अवस्थाको व्यवस्थापन
- **Azure Sentinel**: AI कार्यभार सुरक्षाका लागि क्लाउड-नेटिभ SIEM र SOAR क्षमता
- **Microsoft Entra ID**: सशर्त पहुँच नीतिहरू सहित उद्यम पहिचान र पहुँच व्यवस्थापन
- **Azure Key Vault**: हार्डवेयर सुरक्षा मोड्युल (HSM) समर्थन सहित केन्द्रित गोप्य व्यवस्थापन
- **Microsoft Purview**: AI डेटा स्रोतहरू र कार्यप्रवाहहरूको लागि डेटा शासन र अनुपालन

#### **अनुपालन र शासन**
- **नियमकीय मेल**: MCP कार्यान्वयनहरूले उद्योग-विशिष्ट अनुपालन आवश्यकताहरू पूरा गर्छन् (GDPR, HIPAA, SOC 2)

- **डेटा वर्गीकरण**: एआई प्रणालीहरूले प्रशोधन गर्ने संवेदनशील डेटा को उचित वर्गीकरण र हातमैलो
- **अडिट ट्रेलहरू**: नियामक पालन र फरेन्सिक अनुसन्धानका लागि व्यापक लगिङ
- **गोपनीयता नियन्त्रणहरू**: एआई प्रणालीको वास्तुकलामा प्राइभेसी-बाइ-डिजाइन सिद्घान्तहरूको कार्यान्वयन
- **परिवर्तन व्यवस्थापन**: एआई प्रणालीको संशोधनहरूको सुरक्षा समीक्षा गर्न औपचारिक प्रक्रिया

यी आधारभूत अभ्यासहरूले एक शक्तिशाली सुरक्षा आधार सिर्जना गर्छ जसले MCP-विशिष्ट सुरक्षा नियन्त्रणहरूको प्रभावकारिता वृद्धि गर्दछ र एआई-चालित अनुप्रयोगहरूको व्यापक संरक्षण प्रदान गर्दछ।

## प्रमुख सुरक्षा सन्देशहरू

- **स्तरीय सुरक्षा दृष्टिकोण**: आधारभूत सुरक्षा अभ्यासहरू (सुरक्षित कोडिङ, न्यूनतम अधिकार, आपूर्ति श्रृंखला प्रमाणीकरण, निरन्तर निगरानी) लाई एआई-विशिष्ट नियन्त्रणहरूसँग मिलाएर व्यापक सुरक्षा प्रदान गर्नुहोस्

- **एआई-विशिष्ट खतरा परिदृश्य**: MCP प्रणालीहरूलाई अद्वितीय जोखिमहरू, जस्तै प्रॉम्प्ट इन्जेक्सन, उपकरण विषाक्तता, सत्र अपहरण, भ्रमित डिप्युटी समस्या, टोकन पास-थ्रु कमजोरीहरू, र अत्यधिक अनुमतिहरूको सामना गर्न विशेष समाधानहरू आवश्यक छ

- **प्रमाणीकरण र अनुमति उत्कृष्टता**: बाह्य पहिचान प्रदायकहरू (Microsoft Entra ID) प्रयोग गरेर मजबूत प्रमाणीकरण कार्यान्वयन गर्नुहोस्, उचित टोकन मान्यकरण लागू गर्नुहोस्, र कहिल्यै त्यस्ता टोकनहरू स्वीकार नगर्नुहोस् जुन तपाईंको MCP सर्भरका लागि स्पष्ट रूपमा जारी गरिएको छैनन्

- **एआई आक्रमण रोकथाम**: Microsoft Prompt Shields र Azure Content Safety प्रयोग गरेर अप्रत्यक्ष प्रॉम्प्ट इन्जेक्सन र उपकरण विषाक्तता आक्रमणबाट सुरक्षा गर्नुहोस्, उपकरण मेटाडेटा मान्य गर्नुहोस् र गतिशील परिवर्तनहरूको निगरानी गर्नुहोस्

- **सत्र र ट्रान्सपोर्ट सुरक्षा**: प्रयोगकर्ता पहिचानसँग बाँधिएका क्रिप्टोग्राफिक रूपमा सुरक्षित, गैर-निर्धारित सत्र ID प्रयोग गर्नुहोस्, उचित सत्र जीवनचक्र व्यवस्थापन लागू गर्नुहोस्, र प्रमाणीकरणका लागि कहिल्यै सत्रहरू प्रयोग नगर्नुहोस्

- **OAuth सुरक्षा उत्तम अभ्यासहरू**: गतिशील रूपमा दर्ता गरिएका क्लाइंटहरूका लागि स्पष्ट प्रयोगकर्ता सहमति मार्फत भ्रमित डिप्युटी आक्रमणहरू रोक्नुहोस्, PKCE सहित सटीक OAuth 2.1 कार्यान्वयन गर्नुहोस्, र कडा रिडिरेक्ट URI मान्यकरण गर्नुहोस्  

- **टोकन सुरक्षा सिद्घान्तहरू**: टोकन पास-थ्रु एन्टी-पाटर्नहरूबाट बच्नुपर्छ, टोकन दर्शक दावीहरू मान्य गर्नुपर्छ, छोटो-अवधि टोकनहरू सुरक्षित रोटेसनसहित कार्यान्वयन गर्नुपर्छ, र स्पष्ट विश्वास सीमा कायम राख्नुपर्छ

- **व्यापक आपूर्ति श्रृंखला सुरक्षा**: सबै एआई पारिस्थितिकी तन्त्रका घटकहरू (मोडलहरू, एम्बेडिंगहरू, सन्दर्भ प्रदायकहरू, बाह्य API हरू) लाई परम्परागत सफ्टवेयर निर्भरताहरू जस्तै सुरक्षा गम्भीरताका साथ व्यवहार गर्नुपर्छ

- **निरन्तर विकास**: द्रुत विकसित हुँदै गरेका MCP विनिर्देशहरूसँग अद्यावधिक रहनुहोस्, सुरक्षा समुदायका मापदण्डहरूमा योगदान दिनुहोस्, र प्रोटोकल परिपक्वतासँग सुरक्षा अवस्थाहरूलाई अनुकूल राख्नुहोस्

- **Microsoft सुरक्षा एकीकरण**: Microsoft को व्यापक सुरक्षा पारिस्थितिकी तन्त्र (Prompt Shields, Azure Content Safety, GitHub Advanced Security, Entra ID) लाई प्रयोग गरी MCP परिनियोजन संरक्षण बढाउनुहोस्
