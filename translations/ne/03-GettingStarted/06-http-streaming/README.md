# मोडेल सन्दर्भ प्रोटोकल (MCP) संग HTTPS स्ट्रिमिङ

यस अध्यायले मोडेल सन्दर्भ प्रोटोकल (MCP) प्रयोग गरेर HTTPS सँग सुरक्षित, विस्तारयोग्य, र वास्तविक-समय स्ट्रिमिङ कार्यान्वयन गर्ने व्यापक मार्गदर्शन प्रदान गर्दछ। यसले स्ट्रिमिङको प्रेरणा, उपलब्ध यातायात संयन्त्रहरू, MCP मा स्ट्रिमयोग्य HTTP कसरी कार्यान्वयन गर्ने, सुरक्षा उत्तम अभ्यासहरू, SSE बाट माइग्रेशन, र आफ्नै स्ट्रिमिङ MCP एप्लिकेसनहरू निर्माण गर्ने व्यावहारिक मार्गदर्शन समेट्छ।

> **आगामी उपलब्धीहरू:** यो पाठ **MCP विनिर्देशन 2025-11-25** अन्तर्गत स्ट्रिमयोग्य HTTP वर्णन गर्दछ, जहाँ सत्र `initialize` समयमा स्थापना हुन्छ र `Mcp-Session-Id` हेडरले जडान गरिन्छ। `2026-07-28` रिलिज क्यान्डिडेटले हस्तमिलन र सत्र ID पूर्ण रूपमा हटाउनेछ, जसले हरेक अनुरोधलाई स्व-सम्पूर्ण र कुनै पनि सर्भर उदाहरणमा स्टिकी सत्र बिना रूट गर्न योग्य बनाउँछ। विवरणका लागि [MCP मा के के परिवर्तन हुँदैछ: The 2026-07-28 Release Candidate](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) हेर्नुहोस्।

## MCP मा यातायात संयन्त्रहरू र स्ट्रिमिङ

यस खण्डले MCP मा उपलब्ध विभिन्न यातायात संयन्त्रहरू र तिनीहरूले क्लाइन्ट र सर्भरबीच वास्तविक-समय सञ्चार सक्षम बनाउनको लागि स्ट्रिमिङ क्षमतामा खेलेको भूमिकाको अन्वेषण गर्दछ।

### यातायात संयन्त्र भनेको के हो?

यातायात संयन्त्रले क्लाइन्ट र सर्भरबीच डाटा कसरी आदानप्रदान गरिन्छ भनेर परिभाषित गर्छ। MCP ले विभिन्न वातावरण र आवश्यकताहरूलाई मेल खानका लागि धेरै यातायात प्रकारहरू समर्थन गर्दछ:

- **stdio**: मानक इनपुट/आउटपुट, स्थानीय र CLI आधारित उपकरणहरूका लागि उपयुक्त। सरल तर वेब वा क्लाउडका लागि उपयुक्त छैन।
- **SSE (सर्भर-सेन्ट इभेन्ट्स)**: सर्भरहरूले HTTP मार्फत क्लाइन्टलाई वास्तविक-समय अपडेटहरू पठाउन अनुमति दिन्छ। वेब UI हरूका लागि राम्रो, तर विस्तारयोग्यता र लचिलोपनमा सीमित। MCP विनिर्देशन 2025-06-18 अनुसार, स्वतन्त्र SSE यातायात अब डिप्रिकेट गरिएको छ र "स्ट्रिमयोग्य HTTP" यातायातले प्रतिस्थापित गरेको छ।
- **स्ट्रिमयोग्य HTTP**: आधुनिक HTTP आधारित स्ट्रिमिङ यातायात, सूचना र राम्रो विस्तारयोग्यता समर्थन गर्दछ। धेरै उत्पादन र क्लाउड परिदृश्यहरूको लागि सिफारिश गरिएको।

### तुलना तालिका

यी यातायात संयन्त्रहरूबीचको फरक बुझ्न तलको तुलना तालिकामा हेर्नुहोस्:

| यातायात          | वास्तविक-समय अपडेट | स्ट्रिमिङ | विस्तारयोग्यता | प्रयोग मामला                  |
|-------------------|------------------|-----------|-------------|-------------------------|
| stdio             | होइन             | होइन      | कम          | स्थानीय CLI उपकरणहरू       |
| SSE               | हो               | हो         | मध्यम        | वेब, वास्तविक-समय अपडेटहरू |
| स्ट्रिमयोग्य HTTP | हो               | हो         | उच्च        | क्लाउड, बहु-क्लाइन्ट      |

> **सुझाव:** सही यातायात चयनले प्रदर्शन, विस्तारयोग्यता, र प्रयोगकर्ता अनुभवमा प्रभाव पार्छ। **स्ट्रिमयोग्य HTTP** आधुनिक, विस्तारयोग्य, र क्लाउड-तयार एप्लिकेसनहरूको लागि सिफारिश गरिएको हो।

पहिलेका अध्यायहरूमा तपाईलाई देखाइएको stdio र SSE यातायातहरूलाई ध्यान दिनुहोस् र यस अध्यायमा समेटिएको स्ट्रिमयोग्य HTTP यातायात कस्तो छ।

## स्ट्रिमिङ: अवधारणाहरू र प्रेरणा

स्ट्रिमिङको मूल अवधारणाहरू र प्रेरणाहरू बुझ्नु प्रभावकारी वास्तविक-समय सञ्चार प्रणालीहरू कार्यान्वयन गर्न आवश्यक छ।

**स्ट्रिमिङ** नेटवर्क प्रोग्रामिङमा एउटा प्रविधि हो जसले डाटालाई सानो, व्यवस्थापनयोग्य भागहरूमा वा घटनाहरूको शृंखलाको रूपमा पठाउन र प्राप्त गर्न अनुमति दिन्छ, सम्पूर्ण प्रतिक्रिया तयार हुनको लागि पर्खनुको सट्टा। यो विशेष गरी उपयोगी छ:

- ठूलो फाइलहरू वा डेटासेटहरू।
- वास्तविक-समय अपडेटहरू (जस्तै, च्याट, प्रगति बारहरू)।
- लामो समय चल्ने गणनाहरू जहाँ तपाईं प्रयोगकर्तालाई जानकारीमूलक राख्न चाहनुहुन्छ।

स्ट्रिमिङका उच्च स्तरका केहि कुरा यहाँ छन्:

- डाटा क्रमिक रूपमा हामीलाई प्राप्त हुन्छ, सबै एकपटकमा होइन।
- क्लाइन्टले डाटा आउने बित्तिकै प्रक्रिया गर्न सक्छ।
- अनुमानित विलम्ब घटाउँछ र प्रयोगकर्ता अनुभव सुधार गर्दछ।

### किन स्ट्रिमिङ प्रयोग गर्ने?

स्ट्रिमिङ प्रयोग गर्नका कारणहरू निम्न छन्:

- प्रयोगकर्ताहरूलाई तुरुन्त प्रतिक्रिया प्राप्त हुन्छ, केवल अन्त्यमा मात्र होइन।
- वास्तविक-समय एप्लिकेसनहरू र प्रतिक्रियाशील UI सक्षम बनाउँछ।
- नेटवर्क र गणना स्रोतहरूको बढी प्रभावकारी प्रयोग।

### सरल उदाहरण: HTTP स्ट्रिमिङ सर्भर र क्लाइन्ट

यहाँ कसरी स्ट्रिमिङ कार्यान्वयन गर्न सकिन्छ भन्ने सरल उदाहरण छ:

#### Python

**सर्भर (Python, FastAPI र StreamingResponse प्रयोग गर्दै):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**क्लाइन्ट (Python, requests प्रयोग गर्दै):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

यो उदाहरणले सर्भर कसरी क्लाइन्टलाई सन्देशहरूको श्रृंखला उपलब्ध हुँदा नै पठाउँछ देखाउँछ, सम्पूर्ण सन्देशहरू तयार हुनेसम्म पर्खनुको सट्टा।

**यो कसरी काम गर्छ:**

- सर्भर प्रत्येक सन्देश तयार हुनासाथ पठाउँछ।
- क्लाइन्टले प्रत्येक भाग प्राप्त गर्दा छपाइ गर्छ।

**आवश्यकताहरू:**

- सर्भरले स्ट्रिमिङ प्रतिक्रिया प्रयोग गर्नुपर्छ (जस्तै, FastAPI मा `StreamingResponse`)।
- क्लाइन्टले प्रतिक्रिया स्ट्रिमको रूपमा प्रक्रिया गर्नुपर्छ (`stream=True` requests मा)।
- सामग्री प्रकार सामान्यतः `text/event-stream` वा `application/octet-stream` हुन्छ।

#### Java

**सर्भर (Java, Spring Boot र Server-Sent Events प्रयोग गर्दै):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**क्लाइन्ट (Java, Spring WebFlux WebClient प्रयोग गर्दै):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**Java कार्यान्वयन नोटहरू:**

- Spring Boot को प्रतिक्रिया स्ट्याक `Flux` मार्फत स्ट्रिमिङ प्रयोग गर्दछ
- `ServerSentEvent` ले संरचित घटना स्ट्रिमिङ घटना प्रकारहरू सहित प्रदान गर्दछ
- `WebClient` संग `bodyToFlux()` प्रतिक्रियाशील स्ट्रिमिङ उपभोग सक्षम पार्दछ
- `delayElements()` घटनाहरू बिच प्रशोधन समय अनुकरण गर्दछ
- घटनाहरूले प्रकारहरू (जस्तै, `info`, `result`) हुन सक्छ क्लाइन्ट ह्यान्डलिंगका लागि

### तुलना: क्लासिक स्ट्रिमिङ बनाम MCP स्ट्रिमिङ

"क्लासिक" तरिकाले स्ट्रिमिङ कसरी काम गर्छ र MCP मा कसरी काम गर्छ भनी फरक यसरी देखाउन सकिन्छ:

| सुविधा                | क्लासिक HTTP स्ट्रिमिङ      | MCP स्ट्रिमिङ (सूचनाहरू)          |
|------------------------|-----------------------------|---------------------------------|
| मुख्य प्रतिक्रिया       | खण्डित                      | एकल, अन्त्यमा                   |
| प्रगति अपडेटहरू        | डाटा खण्डहरूका रूपमा पठाइन्छ    | सूचनाहरूका रूपमा पठाइन्छ         |
| क्लाइन्ट आवश्यकताहरू  | स्ट्रिम प्रक्रिया गर्नैपर्छ          | सन्देश ह्यान्डर कार्यान्वयन गर्नैपर्छ       |
| प्रयोग मामला          | ठूलो फाइलहरू, AI टोकन स्ट्रिमहरू | प्रगति, लगहरू, वास्तविक-समय प्रतिक्रिया   |

### देखिएका प्रमुख फरकहरू

थप रूपमा, यहाँ केहि मुख्य फरकहरू छन्:

- **सञ्चार ढाँचा:**
  - क्लासिक HTTP स्ट्रिमिङ: सरल खण्डित ट्रान्सफर इन्कोडिंग प्रयोग गरेर डाटा खण्डहरू पठाउँछ
  - MCP स्ट्रिमिङ: JSON-RPC प्रोटोकलसहित संरचित सूचना प्रणाली प्रयोग गर्दछ

- **सन्देश ढाँचा:**
  - क्लासिक HTTP: नयाँ लाइनहरूसहितको प्लेन टेक्स्ट खण्डहरू
  - MCP: मेटाडाटा सहित संरचित LoggingMessageNotification वस्तुहरू

- **क्लाइन्ट कार्यान्वयन:**
  - क्लासिक HTTP: स्ट्रिमिंग प्रतिक्रियाहरू प्रक्रिया गर्ने सरल क्लाइन्ट
  - MCP: फरक-फरक प्रकारका सन्देशहरू प्रक्रिया गर्न सन्देश ह्यान्डर सहित थप परिष्कृत क्लाइन्ट

- **प्रगति अपडेटहरू:**
  - क्लासिक HTTP: प्रगति मुख्य प्रतिक्रिया स्ट्रिमको हिस्सा हो
  - MCP: प्रगति छुट्टै सूचना सन्देशहरू मार्फत पठाइन्छ जबकि मुख्य प्रतिक्रिया अन्त्यमा आउँछ

### सिफारिशहरू

क्लासिक स्ट्रिमिङ (जस्तै `/stream` अन्त बिन्दु) कार्यान्वयन गर्ने वा MCP मार्फत स्ट्रिमिङ चयन गर्ने विषयमा हामीले केही सिफारिशहरू गरेका छौं।

- **सजिलो स्ट्रिमिङ आवश्यकताहरूका लागि:** क्लासिक HTTP स्ट्रिमिङ कार्यान्वयन गर्न सजिलो र आधारभूत स्ट्रिमिङ आवश्यकताहरूका लागि पर्याप्त छ।

- **जटिल, अन्तरक्रियात्मक एप्लिकेसनहरूको लागि:** MCP स्ट्रिमिङले बढी संरचित दृष्टिकोण प्रदान गर्दछ, संपन्न मेटाडाटा र सूचनाहरू तथा अन्तिम परिणामबीच पृथक्करण सहित।

- **AI एप्लिकेसनहरूको लागि:** MCP को सूचना प्रणाली लामो समय चल्ने AI कार्यहरूका लागि विशेष उपयोगी छ जहाँ प्रयोगकर्ताहरूलाई प्रगतिको जानकारी दिनुपर्छ।

## MCP मा स्ट्रिमिङ

अब, तपाईले क्लासिक स्ट्रिमिङ र MCP स्ट्रिमिङबीचको तुलना र सिफारिसहरू देख्नुभयो। अब MCP मा कसरी स्ट्रिमिङ गर्न सकिन्छ विस्तारमा बुझौं।

MCP संरचनाभित्र स्ट्रिमिङ कसरी काम गर्छ बुझ्नु प्रतिक्रियाशील एप्लिकेसनहरू निर्माण गर्न आवश्यक छ जुन लामो समय चल्ने कार्यहरूमा प्रयोगकर्तालाई वास्तविक-समय प्रतिक्रिया प्रदान गर्छ।

MCP मा, स्ट्रिमिङ भनेको मुख्य प्रतिक्रिया खण्डहरूमा पठाउने कुरा होइन, तर उपकरणले अनुरोध प्रक्रिया गर्दा क्लाइन्टलाई **सूचनाहरू** पठाउने कुरा हो। यी सूचनाहरूमा प्रगति अपडेटहरू, लगहरू, वा अन्य घटनाहरू समावेश हुन सक्छन्।

### यो कसरी काम गर्छ

मुख्य नतिजा अझै एकल प्रतिक्रियाको रूपमा पठाइन्छ। तर प्रक्रिया भइरहेका बेला छुट्टै सन्देशहरूको रूपमा सूचनाहरू पठाउन सकिन्छ र यसले क्लाइन्टलाई वास्तविक-समय अद्यावधिक गर्न मद्दत गर्दछ। क्लाइन्टले यी सूचनाहरूलाई सम्हाल्ने र प्रदर्शन गर्ने सक्षम हुनुपर्छ।

## सूचना के हो?

हामीले "सूचना" भन्नुभयो, MCP सन्दर्भमा यसको अर्थ के हो?

सूचना एउटा सन्देश हो जुन सर्भरले क्लाइन्टलाई लामो समय चल्ने कार्यको प्रगति, स्थिति, वा अन्य घटनाहरूको बारेमा जानकारी दिन पठाउँछ। सूचनाले पारदर्शिता र प्रयोगकर्ता अनुभव सुधार गर्दछ।

उदाहरणका लागि, क्लाइन्टले सर्भरसँग प्रारम्भिक हस्तमिलन भएपछि सूचना पठाउनुपर्छ।

सूचना JSON सन्देशको रूपमा यसरी देखिन्छ:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

सूचना MCP मा ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) शीर्षक अन्तर्गत एक विषयमा पर्छ।

लगिङ काम गर्नको लागि, सर्भरले यसलाई सुविधा/क्षमता रूपमा सक्रिय पार्नुपर्छ जस्तै:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> प्रयोग गरिएको SDK अनुसार, लगिङ डिफल्ट रूपमा सक्रिय हुन सक्छ, वा तपाईंले आफ्नो सर्भर कन्फिगरेसनमा यसलाई स्पष्ट रूपमा सक्षम पार्न आवश्यक पर्न सक्छ।

विभिन्न प्रकारका सूचनाहरू छन्:

| स्तर       | विवरण                          | उदाहरण प्रयोग मामला             |
|------------|--------------------------------|-------------------------------|
| debug      | विस्तृत डिबगिङ जानकारी        | फङ्क्शन प्रवेश/निर्गमन बिन्दुहरू |
| info       | सामान्य सूचना सन्देशहरू          | अपरेशन प्रगति अपडेटहरू          |
| notice     | सामान्य तर महत्वपूर्ण घटनाहरू    | कन्फिगरेसन परिवर्तनहरू          |
| warning    | चेतावनी अवस्था                  | अवप्रचलित सुविधा प्रयोग        |
| error      | त्रुटि अवस्था                   | अपरेशन असफलता                  |
| critical   | गंभीर अवस्था                   | प्रणाली कम्पोनेन्ट विफलता       |
| alert      | तुरुन्त कारबाही आवश्यक           | डाटा भ्रष्टाचार पत्ता लागेको    |
| emergency  | प्रणाली प्रयोगयोग्य छैन          | पूर्ण प्रणाली विफलता            |

## MCP मा सूचनाहरू कार्यान्वयन गर्दै

MCP मा सूचनाहरू कार्यान्वयन गर्न, तपाईंले सर्भर र क्लाइन्ट दुवै पक्षहरू वास्तविक-समय अद्यावधिकहरू ह्यान्डल गर्ने गरी सेटअप गर्नु पर्छ। यसले तपाईंको एप्लिकेसनलाई लामो समय चल्ने अपरेशनहरूमा प्रयोगकर्ताहरूलाई तुरुन्त प्रतिक्रिया प्रदान गर्न अनुमति दिन्छ।

### सर्भर-पक्ष: सूचनाहरू पठाउने

सर्भर पक्षबाट सुरु गरौं। MCP मा, तपाईंले उपकरणहरू परिभाषित गर्नुहुन्छ जसले अनुरोधहरू प्रक्रिया गर्दा सूचनाहरू पठाउन सक्छ। सर्भर सन्देशहरू क्लाइन्टलाई पठाउन सन्दर्भ वस्तु (साधारणतया `ctx`) प्रयोग गर्छ।

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

अघिल्लो उदाहरणमा, `process_files` उपकरणले प्रत्येक फाइल प्रक्रिया गर्दा क्लाइन्टलाई तीनवटा सूचनाहरू पठाउँछ। `ctx.info()` विधि सूचना सन्देशहरू पठाउन प्रयोग गरिन्छ।

थप रूपमा, सूचनाहरू सक्षम गर्न, सुनिश्चित गर्नुहोस् तपाईंको सर्भरले स्ट्रिमिङ यातायात (जस्तै, `streamable-http`) प्रयोग गर्छ र क्लाइन्टले सूचनाहरू प्रक्रिया गर्न सन्देश ह्यान्डर कार्यान्वयन गर्छ। यहाँ कसरी सर्भरलाई `streamable-http` यातायात प्रयोग गर्न सेटअप गर्ने:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

यस .NET उदाहरणमा, `ProcessFiles` उपकरण `Tool` विशेषतासहित डेकरेट गरिएको छ र प्रत्येक फाइल प्रक्रिया गर्दा क्लाइन्टलाई तीन सूचनाहरू पठाउँछ। `ctx.Info()` विधि सूचना सन्देशहरू पठाउन प्रयोग गरिन्छ।

तपाईंको .NET MCP सर्भरमा सूचनाहरू सक्षम गर्न, सुनिश्चित गर्नुहोस् कि तपाईं स्ट्रिमिङ यातायात प्रयोग गर्दै हुनुहुन्छ:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### क्लाइन्ट-पक्ष: सूचनाहरू प्राप्त गर्ने

क्लाइन्टले सूचनाहरू आउँदै गर्दा तिनीहरूलाई प्रक्रिया र प्रदर्शन गर्न सन्देश ह्यान्डर कार्यान्वयन गर्नुपर्छ।

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

अघिल्लो कोडमा, `message_handler` फङ्क्शनले आउने सन्देश सूचना हो कि होइन जाँच गर्छ। यदि हो भने, सूचना छाप्छ; नत्र सामान्य सर्भर सन्देश जस्तै प्रक्रिया गर्छ। साथै `ClientSession` लाई `message_handler` को साथ सुरु गरिएको छ जसले आउने सूचनाहरू ह्यान्डल गर्न सक्षम बनाउँछ।

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

यस .NET उदाहरणमा, `MessageHandler` फङ्क्शनले आउने सन्देश सूचना हो कि होइन जाँच गर्छ। यदि हो भने, सूचना छाप्छ; नत्र सामान्य सर्भर सन्देश जस्तै प्रक्रिया गर्छ। `ClientSession` लाई `ClientSessionOptions` मार्फत सन्देश ह्यान्डर सहित सुरु गरिएको छ।

सूचनाहरू सक्षम गर्न, सुनिश्चित गर्नुहोस् तपाईंको सर्भरले स्ट्रिमिङ यातायात (जस्तै, `streamable-http`) प्रयोग गर्छ र क्लाइन्टले सूचनाहरू प्रक्रिया गर्न सन्देश ह्यान्डर कार्यान्वयन गर्छ।

## प्रगति सूचनाहरू र परिदृश्यहरू

यस खण्डले MCP मा प्रगति सूचनाको अवधारणा, तिनीहरू किन महत्वपूर्ण छन्, र त्यसलाई स्ट्रिमयोग्य HTTP प्रयोग गरेर कसरी कार्यान्वयन गर्ने भन्ने व्याख्या गर्दछ। तपाईंले आफ्नो बुझाइलाई बलियो बनाउन व्यावहारिक कार्य पनि पाउनुहुनेछ।

प्रगति सूचनाहरू लामो अवधि चल्ने अपरेशनहरूमा सर्भरबाट क्लाइन्टलाई वास्तविक-समय सन्देशहरू हुन्। सम्पूर्ण प्रक्रिया समाप्त हुन नपर्खेरै सर्भरले क्लाइन्टलाई वर्तमान स्थितिको जानकारी दिन्छ। यसले पारदर्शिता, प्रयोगकर्ता अनुभव सुधार गर्दछ, र डिबगिङ सजिलो बनाउँछ।

**उदाहरण:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### किन प्रगति सूचनाहरू प्रयोग गर्ने?

प्रगति सूचनाहरू निम्न कारणहरूले आवश्यक छन्:

- **बेहतर प्रयोगकर्ता अनुभव:** प्रयोगकर्ताहरूले कामको प्रगतिको रूपमा अपडेटहरू हेर्छन्, केवल अन्त्यमा होइन।
- **वास्तविक-समय प्रतिक्रिया:** क्लाइन्टहरूले प्रगति बारहरू वा लगहरू देखाउन सक्दछन्, जसले एप्लिकेसनलाई प्रतिक्रियाशील बनाउँछ।
- **डिबगिङ र अनुगमन सजिलो:** विकासकर्ता र प्रयोगकर्ताहरूले प्रक्रिया कहाँ ढिलो वा अड्किएको छ देख्न सक्छन्।

### प्रगति सूचनाहरू कसरी कार्यान्वयन गर्ने

यहाँ तपाईं MCP मा प्रगति सूचनाहरू कसरी कार्यान्वयन गर्न सक्नुहुन्छ:

- **सर्भरमा:** प्रत्येक आइटम प्रक्रिया हुँदाचाहिँ `ctx.info()` वा `ctx.log()` प्रयोग गरी सूचना पठाउनुहोस्। यसले मुख्य नतिजा तयार हुनअघि क्लाइन्टलाई सन्देश पठाउँछ।
- **क्लाइन्टमा:** एउटा सन्देश ह्यान्डर कार्यान्वयन गर्नुहोस् जसले आउने सूचनाहरू सुन्छ र प्रदर्शन गर्छ। यो ह्यान्डर सूचनाहरू र अन्तिम परिणामबीच फरक छुट्याउँछ।

**सर्भर उदाहरण:**

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**क्लाइन्ट उदाहरण:**

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## सुरक्षा विचारहरू

HTTP आधारित यातायातहरूसँग MCP सर्भरहरू कार्यान्वयन गर्दा, सुरक्षा धेरै महत्वपूर्ण विषय हो जसले धेरै आक्रमण मार्ग र संरक्षण संयन्त्रहरूको विचार आवश्यक पार्दछ।

### अवलोकन

MCP सर्भरहरू HTTP मार्फत सार्वजनिक गर्दा सुरक्षा अत्यावश्यक हुन्छ। स्ट्रिमयोग्य HTTP ले नयाँ आक्रमण सतहहरू प्रस्तुत गर्दछ र सावधानीपूर्वक कन्फिगरेसन आवश्यक पर्छ।

### मुख्य बुँदाहरू


- **अରिजिन हेडर प्रमाणीकरण**: DNS रिबाइन्डिङ आक्रमणबाट जोगिन सधैं `Origin` हेडरलाई प्रमाणित गरेर मात्र स्वीकार गर्नुहोस्।
- **लोकलहोष्ट बाइन्डिङ**: स्थानीय विकासको लागि, सर्भरहरूलाई `localhost` मा बाँध्नुहोस् ताकि तिनीहरू सार्वजनिक इन्टरनेटमा उजागर नगरियोस।
- **प्रमाणीकरण**: उत्पादन तैनातीहरूको लागि प्रमाणीकरण (जस्तै, API कुञ्जीहरू, OAuth) कार्यान्वयन गर्नुहोस्।
- **CORS**: पहुँच सीमित गर्न क्रस-ओरिजिन रिसोर्स शेयरिंग (CORS) नीतिहरू संयोजन गर्नुहोस्।
- **HTTPS**: यातायातलाई इन्क्रिप्ट गर्न उत्पादनमा HTTPS प्रयोग गर्नुहोस्।

### उत्कृष्ट अभ्यासहरू

- प्रमाणीकरणबिना आउने अनुरोधहरू कहिल्यै विश्वास नगर्नुहोस्।
- सबै पहुँच र त्रुटिहरूको लग राख्नुहोस् र निगरानी गर्नुहोस्।
- सुरक्षा कमीकमजोरीहरू पत्तालागाउन नियमित रूपमा निर्भरता अपडेट गर्नुहोस्।

### चुनौतीहरू

- विकासको सहजता र सुरक्षा सन्तुलन गर्नु
- विभिन्न ग्राहक वातावरणहरूसँग मेल खाने सुनिश्चित गर्नु

## SSE बाट Streamable HTTP मा उन्नयन

हाल Server-Sent Events (SSE) प्रयोग गरिरहेका अनुप्रयोगहरूको लागि, Streamable HTTP मा माइग्रेट गर्नुले थप क्षमताहरू र दीर्घकालीन दिगोपना प्रदान गर्दछ तपाईंको MCP कार्यान्वयनहरूका लागि।

### किन उन्नयन गर्ने?

SSE बाट Streamable HTTP मा उन्नयन गर्नुका दुई मुख्य कारणहरू छन्:

- Streamable HTTP ले SSE भन्दा उत्तम स्केलेबिलिटी, अनुकूलता, र धनी सूचना समर्थन प्रदान गर्दछ।
- यो नयाँ MCP अनुप्रयोगहरूको लागि सिफारिस गरिएको यातायात हो।

### माइग्रेशन चरणहरू

तपाईंको MCP अनुप्रयोगहरूमा SSE बाट Streamable HTTP मा कसरी माइग्रेट गर्ने:

- `mcp.run()` मा `transport="streamable-http"` प्रयोग गर्ने गरी सर्वर कोड अपडेट गर्नुहोस्।
- SSE क्लाइन्टको सट्टा `streamablehttp_client` प्रयोग गर्ने गरी क्लाइन्ट कोड अपडेट गर्नुहोस्।
- सूचना प्रक्रिया गर्न क्लाइन्टमा मेसेज ह्यान्डलर कार्यान्वयन गर्नुहोस्।
- अवस्थित उपकरणहरू र कार्यप्रवाहहरू सँग अनुकूलता परीक्षण गर्नुहोस्।

### अनुकूलता कायम राख्ने

माइग्रेशन प्रक्रियाको क्रममा अवस्थित SSE क्लाइन्टहरूसँग अनुकूलता कायम राख्न सिफारिस गरिएको छ। यहाँ केही रणनीतिहरू छन्:

- दुबै SSE र Streamable HTTP समर्थन गर्न अलग-अगल endpoints मा दुबै ट्रान्सपोर्टहरू चलाउन सक्नुहुन्छ।
- क्लाइन्टहरूलाई नयाँ ट्रान्सपोर्टमा क्रमशः माइग्रेट गर्नुहोस्।

### चुनौतीहरू

माइग्रेशनका क्रममा निम्न चुनौतीहरू समाधान गर्न सुनिश्चित गर्नुहोस्:

- सबै क्लाइन्टहरू अपडेट गरिएको सुनिश्चित गर्नु
- सूचना वितरणमा फरकपनको व्यवस्थापन गर्नु

## सुरक्षा विचारहरू

MCP कार्यान्वयन गर्दा सुरक्षा उच्च प्राथमिकता हुनुपर्छ, विशेष गरी Streamable HTTP जस्ता HTTP-आधारित ट्रान्सपोर्ट प्रयोग गर्दा।

HTTP-आधारित ट्रान्सपोर्टहरूसँग MCP सर्भर कार्यान्वयन गर्दा सुरक्षा धेरै महत्वपूर्ण हुन्छ र विभिन्न आक्रमण मार्गहरू र सुरक्षा उपायहरूमा ध्यान दिनुपर्छ।

### अवलोकन

HTTP मार्फत MCP सर्भरहरू उजागर गर्दा सुरक्षा अत्यावश्यक छ। Streamable HTTP नयाँ आक्रमण सतहहरू ल्याउँछ र सावधानीपूर्वक संयोजन आवश्यक हुन्छ।

यहाँ केहि प्रमुख सुरक्षा विचारहरू छन्:

- **अरिजिन हेडर प्रमाणीकरण**: DNS रिबाइन्डिङ आक्रमणबाट जोगिन सधैं `Origin` हेडरलाई प्रमाणित गर्नुहोस्।
- **लोकलहोष्ट बाइन्डिङ**: स्थानीय विकासको लागि सर्भरहरूलाई `localhost` मा बाँध्नुहोस् ताकि तिनीहरू सार्वजनिक इन्टरनेटमा उजागर नगरियोस।
- **प्रमाणीकरण**: उत्पादन तैनातीहरूको लागि प्रमाणीकरण (जस्तै, API कुञ्जीहरू, OAuth) कार्यान्वयन गर्नुहोस्।
- **CORS**: पहुँच सीमित गर्न क्रस-ओरिजिन रिसोर्स शेयरिंग (CORS) नीतिहरू संयोजन गर्नुहोस्।
- **HTTPS**: यातायातलाई इन्क्रिप्ट गर्न उत्पादनमा HTTPS प्रयोग गर्नुहोस्।

### उत्कृष्ट अभ्यासहरू

तपाईंको MCP स्ट्रिमिङ सर्भरमा सुरक्षा कार्यान्वयन गर्दा यहाँ केहि उत्कृष्ट अभ्यासहरू छन्:

- प्रमाणीकरणबिना आउने अनुरोधहरू कहिल्यै विश्वास नगर्नुहोस्।
- सबै पहुँच र त्रुटिहरूको लग राख्नुहोस् र निगरानी गर्नुहोस्।
- सुरक्षा कमीकमजोरीहरू पत्तालागाउन नियमित रूपमा निर्भरता अपडेट गर्नुहोस्।

### चुनौतीहरू

MCP स्ट्रिमिङ सर्भरहरूमा सुरक्षा कार्यान्वयन गर्दा तपाईंले केही चुनौतीहरूको सामना गर्नु पर्नेछ:

- विकासको सहजता र सुरक्षा सन्तुलन गर्नु
- विभिन्न ग्राहक वातावरणहरूसँग मेल खाने सुनिश्चित गर्नु

### कार्य: आफ्नै स्ट्रिमिङ MCP अनुप्रयोग बनाउनुहोस्

**परिदृश्य:**
एउटा MCP सर्भर र क्लाइन्ट बनाउनुहोस् जहाँ सर्भरले वस्तुहरूको सूची (जस्तै, फाइलहरू वा कागजातहरू) प्रक्रिया गर्छ र प्रत्येक वस्तु प्रक्रिया गरिरहँदा सूचना पठाउँछ। क्लाइन्टले प्राप्त हुने प्रत्येक सूचना प्रदर्शन गर्नुपर्छ।

**चरणहरू:**

1. सूची प्रक्रिया गरी प्रत्येक वस्तुको सूचना पठाउने सर्भर उपकरण कार्यान्वयन गर्नुहोस्।
2. सूचना वास्तविक समयमा प्रदर्शन गर्न मेसेज ह्यान्डलर सहितको क्लाइन्ट कार्यान्वयन गर्नुहोस्।
3. आफ्नो कार्यान्वयनलाई दुवै सर्भर र क्लाइन्ट चलाएर परीक्षण गर्नुहोस् र सूचनाहरू अवलोकन गर्नुहोस्।
