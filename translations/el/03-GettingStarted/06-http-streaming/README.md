> **Κοιτώντας μπροστά:** αυτό το μάθημα περιγράφει το Streamable HTTP σύμφωνα με την **Προδιαγραφή MCP 2025-11-25**, όπου μια συνεδρία δημιουργείται κατά την `initialize` και ακινητοποιείται με την κεφαλίδα `Mcp-Session-Id`. ## Μηχανισμοί Μεταφοράς και Ροή στο MCP

Αυτή η ενότητα εξερευνά τους διαφορετικούς μηχανισμούς μεταφοράς που είναι διαθέσιμοι στο MCP και τον ρόλο τους στην ενεργοποίηση λειτουργιών ροής για επικοινωνία σε πραγματικό χρόνο μεταξύ πελατών και διακομιστών.

### Τι είναι ένας Μηχανισμός Μεταφοράς;

Ένας μηχανισμός μεταφοράς ορίζει πώς ανταλλάσσονται τα δεδομένα μεταξύ του πελάτη και του διακομιστή. Το MCP υποστηρίζει πολλούς τύπους μεταφοράς για να ταιριάζει σε διάφορα περιβάλλοντα και απαιτήσεις:

- **stdio**: Τυπική είσοδος/έξοδος, κατάλληλη για τοπικά και εργαλεία γραμμής εντολών. Απλό αλλά μη κατάλληλο για web ή cloud.
- **SSE (Server-Sent Events)**: Επιτρέπει στους διακομιστές να στέλνουν ενημερώσεις σε πραγματικό χρόνο στους πελάτες μέσω HTTP. Κατάλληλο για περιβάλλοντα ιστού, αλλά περιορισμένο σε επεκτασιμότητα και ευελιξία. Σύμφωνα με την Προδιαγραφή MCP 2025-06-18, η αυτόνομη μεταφορά SSE έχει καταργηθεί και αντικατασταθεί από το "Streamable HTTP".
- **Streamable HTTP**: Σύγχρονη μεταφορά ροής βάσει HTTP, που υποστηρίζει ειδοποιήσεις και καλύτερη επεκτασιμότητα. Συνιστάται για τις περισσότερες παραγωγικές και cloud περιπτώσεις.

### Πίνακας Σύγκρισης

Ρίξτε μια ματιά στον παρακάτω πίνακα σύγκρισης για να κατανοήσετε τις διαφορές μεταξύ αυτών των μηχανισμών μεταφοράς:

| Μεταφορά        | Ενημερώσεις σε Πραγματικό Χρόνο | Ροή     | Επεκτασιμότητα | Περίπτωση Χρήσης          |
|-----------------|-------------------------------|---------|----------------|--------------------------|
| stdio           | Όχι                           | Όχι     | Χαμηλή        | Τοπικά εργαλεία CLI      |
| SSE             | Ναι                           | Ναι     | Μεσαία        | Web, ενημερώσεις πραγματικού χρόνου |
| Streamable HTTP | Ναι                           | Ναι     | Υψηλή         | Cloud, πολλαπλοί πελάτες |

> **Συμβουλή:** Η επιλογή σωστής μεταφοράς επηρεάζει την απόδοση, την επεκτασιμότητα και την εμπειρία χρήστη. Το **Streamable HTTP** συνιστάται για σύγχρονες, επεκτάσιμες και έτοιμες για το cloud εφαρμογές.

Σημειώστε τις μεταφορές stdio και SSE που παρουσιάστηκαν στα προηγούμενα κεφάλαια και πώς το Streamable HTTP είναι η μεταφορά που καλύπτεται σε αυτό το κεφάλαιο.

## Ροή: Έννοιες και Κίνητρα

Η κατανόηση των θεμελιωδών εννοιών και των κινήτρων πίσω από τη ροή είναι απαραίτητη για την υλοποίηση αποτελεσματικών συστημάτων επικοινωνίας σε πραγματικό χρόνο.

**Η ροή** είναι μια τεχνική στον προγραμματισμό δικτύων που επιτρέπει την αποστολή και λήψη δεδομένων σε μικρά, διαχειρίσιμα κομμάτια ή ως ακολουθία γεγονότων, αντί να περιμένει για ολόκληρη την απάντηση. Αυτό είναι ιδιαίτερα χρήσιμο για:

- Μεγάλα αρχεία ή σύνολα δεδομένων.
- Ενημερώσεις σε πραγματικό χρόνο (π.χ. συνομιλία, μπάρες προόδου).
- Μακροχρόνιους υπολογισμούς όπου θέλετε να κρατάτε τον χρήστη ενήμερο.

Ακολουθεί τι πρέπει να γνωρίζετε για τη ροή σε υψηλό επίπεδο:

- Τα δεδομένα παραδίδονται σταδιακά, όχι όλα μαζί.
- Ο πελάτης μπορεί να επεξεργαστεί τα δεδομένα καθώς φτάνουν.
- Μειώνει την αντιληπτή καθυστέρηση και βελτιώνει την εμπειρία χρήστη.

### Γιατί να χρησιμοποιήσετε ροή;

Οι λόγοι για τη χρήση της ροής είναι οι εξής:

- Οι χρήστες λαμβάνουν άμεσα ανατροφοδότηση, όχι μόνο στο τέλος.
- Υποστηρίζει εφαρμογές σε πραγματικό χρόνο και ανταποκρινόμενα UI.
- Αποδοτικότερη χρήση πόρων δικτύου και υπολογιστών.

### Απλό Παράδειγμα: HTTP Streaming Server & Client

Ακολουθεί ένα απλό παράδειγμα για το πώς μπορεί να υλοποιηθεί η ροή:

#### Python

**Διακομιστής (Python, χρησιμοποιώντας FastAPI και StreamingResponse):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**Πελάτης (Python, χρησιμοποιώντας requests):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

Αυτό το παράδειγμα δείχνει έναν διακομιστή που στέλνει μια σειρά μηνυμάτων στον πελάτη καθώς γίνονται διαθέσιμα, αντί να περιμένει όλα τα μηνύματα να είναι έτοιμα.

**Πώς λειτουργεί:**

- Ο διακομιστής αποδίδει κάθε μήνυμα καθώς είναι έτοιμο.
- Ο πελάτης λαμβάνει και εκτυπώνει κάθε κομμάτι όταν φτάνει.

**Απαιτήσεις:**

- Ο διακομιστής πρέπει να χρησιμοποιεί απάντηση ροής (π.χ. `StreamingResponse` στο FastAPI).
- Ο πελάτης πρέπει να επεξεργάζεται την απάντηση ως ροή (`stream=True` στα requests).
- Το Content-Type είναι συνήθως `text/event-stream` ή `application/octet-stream`.

#### Java

**Διακομιστής (Java, χρησιμοποιώντας Spring Boot και Server-Sent Events):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**Πελάτης (Java, χρησιμοποιώντας Spring WebFlux WebClient):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**Σημειώσεις Υλοποίησης Java:**

- Χρησιμοποιεί το αντιδραστικό στοίβαγμα του Spring Boot με `Flux` για ροή
- `ServerSentEvent` παρέχει δομημένη ροή συμβάντων με τύπους συμβάντων
- `WebClient` με `bodyToFlux()` επιτρέπει την κατανάλωση ροής
- `delayElements()` προσομοιώνει χρόνο επεξεργασίας μεταξύ των συμβάντων
- Τα συμβάντα μπορούν να έχουν τύπους (`info`, `result`) για καλύτερο χειρισμό από τον πελάτη

### Σύγκριση: Κλασική Ροή εναντίον MCP Ροής

Οι διαφορές μεταξύ του πώς λειτουργεί η ροή με «κλασικό» τρόπο και πώς λειτουργεί στο MCP μπορούν να απεικονιστούν ως εξής:

| Χαρακτηριστικό           | Κλασική Ροή HTTP              | MCP Ροή (Ειδοποιήσεις)             |
|-------------------------|-------------------------------|-----------------------------------|
| Κύρια απάντηση           | Τμηματοποιημένη                | Ενιαία, στο τέλος                 |
| Ενημερώσεις προόδου      | Αποστέλλονται ως κομμάτια     | Αποστέλλονται ως ειδοποιήσεις    |
| Απαιτήσεις πελάτη        | Πρέπει να επεξεργαστεί ροή    | Πρέπει να υλοποιήσει χειριστή μηνυμάτων |
| Περίπτωση χρήσης         | Μεγάλα αρχεία, ροές token AI  | Πρόοδος, καταγραφές, ανατροφοδότηση σε πραγματικό χρόνο |

### Βασικές Διαφορές που Παρατηρήθηκαν

Επιπλέον, εδώ είναι κάποιες βασικές διαφορές:

- **Πρότυπο Επικοινωνίας:**
  - Κλασική ροή HTTP: Χρησιμοποιεί απλή κωδικοποίηση μεταφοράς σε τμήματα για να στείλει δεδομένα σε κομμάτια
  - MCP ροή: Χρησιμοποιεί δομημένο σύστημα ειδοποιήσεων με πρωτόκολλο JSON-RPC

- **Μορφή Μηνύματος:**
  - Κλασική HTTP: Απλά κείμενα με αλλαγές γραμμής
  - MCP: Δομημένα αντικείμενα LoggingMessageNotification με μεταδεδομένα

- **Υλοποίηση Πελάτη:**
  - Κλασική HTTP: Απλός πελάτης που επεξεργάζεται απαντήσεις ροής
  - MCP: Πιο σύνθετος πελάτης με χειριστή μηνυμάτων για να επεξεργάζεται διαφορετικούς τύπους μηνυμάτων

- **Ενημερώσεις Προόδου:**
  - Κλασική HTTP: Η πρόοδος είναι μέρος του κύριου ρεύματος απάντησης
  - MCP: Η πρόοδος αποστέλλεται μέσω ξεχωριστών μηνυμάτων ειδοποιήσεων ενώ η κύρια απάντηση έρχεται στο τέλος

### Συστάσεις

Υπάρχουν μερικά πράγματα που συνιστούμε όσον αφορά την επιλογή μεταξύ υλοποίησης κλασικής ροής (ως ένα endpoint που δείξαμε παραπάνω με χρήση του `/stream`) έναντι της ροής μέσω MCP.

- **Για απλές ανάγκες ροής:** Η κλασική ροή HTTP είναι πιο απλή στην υλοποίηση και επαρκής για βασικές ανάγκες ροής.

- **Για σύνθετες, διαδραστικές εφαρμογές:** Η ροή MCP παρέχει μια πιο δομημένη προσέγγιση με πλουσιότερα μεταδεδομένα και διαχωρισμό μεταξύ ειδοποιήσεων και τελικών αποτελεσμάτων.

- **Για εφαρμογές ΤΝ:** Το σύστημα ειδοποιήσεων του MCP είναι ιδιαίτερα χρήσιμο για μακροχρόνιες εργασίες AI όπου θέλετε να κρατάτε τους χρήστες ενήμερους για την πρόοδο.

## Ροή στο MCP

Εντάξει, έχετε δει μερικές συστάσεις και συγκρίσεις μέχρι τώρα για τη διαφορά μεταξύ κλασικής ροής και ροής στο MCP. Ας δούμε λεπτομερώς πώς ακριβώς μπορείτε να αξιοποιήσετε τη ροή στο MCP.

Η κατανόηση του πώς λειτουργεί η ροή εντός του πλαισίου MCP είναι απαραίτητη για την κατασκευή ανταποκρινόμενων εφαρμογών που παρέχουν ανατροφοδότηση σε πραγματικό χρόνο στους χρήστες κατά τη διάρκεια μακροχρόνιων λειτουργιών.

Στο MCP, η ροή δεν αφορά την αποστολή της κύριας απάντησης σε τμήματα, αλλά την αποστολή **ειδοποιήσεων** προς τον πελάτη ενώ ένα εργαλείο επεξεργάζεται ένα αίτημα. Αυτές οι ειδοποιήσεις μπορούν να περιλαμβάνουν ενημερώσεις προόδου, καταγραφές ή άλλα γεγονότα.

### Πώς λειτουργεί

Το κύριο αποτέλεσμα αποστέλλεται ακόμη ως μία ενιαία απάντηση. Ωστόσο, οι ειδοποιήσεις μπορούν να αποστέλλονται ως ξεχωριστά μηνύματα κατά την επεξεργασία και έτσι ενημερώνουν τον πελάτη σε πραγματικό χρόνο. Ο πελάτης πρέπει να μπορεί να χειρίζεται και να εμφανίζει αυτές τις ειδοποιήσεις.

## Τι είναι μια Ειδοποίηση;

Είπαμε "Ειδοποίηση", τι σημαίνει αυτό στο πλαίσιο του MCP;

Μια ειδοποίηση είναι ένα μήνυμα που αποστέλλεται από τον διακομιστή προς τον πελάτη για να ενημερώσει για πρόοδο, κατάσταση ή άλλα γεγονότα κατά τη διάρκεια μιας μακροχρόνιας λειτουργίας. Οι ειδοποιήσεις βελτιώνουν τη διαφάνεια και την εμπειρία χρήστη.

Για παράδειγμα, ένας πελάτης πρέπει να στείλει μια ειδοποίηση μόλις ολοκληρωθεί η αρχική χειραψία με τον διακομιστή.

Μια ειδοποίηση μοιάζει ως εξής ως μήνυμα JSON:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

Οι ειδοποιήσεις ανήκουν σε ένα θέμα στο MCP που αναφέρεται ως ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging).

Για να λειτουργήσει η καταγραφή, ο διακομιστής πρέπει να την ενεργοποιήσει ως χαρακτηριστικό/δυνατότητα ως εξής:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> Ανάλογα με το SDK που χρησιμοποιείται, η καταγραφή μπορεί να είναι ενεργοποιημένη από προεπιλογή, ή να χρειάζεται να την ενεργοποιήσετε ρητά στη διαμόρφωση του διακομιστή.

Υπάρχουν διαφορετικοί τύποι ειδοποιήσεων:

| Επίπεδο    | Περιγραφή                     | Παράδειγμα Χρήσης               |
|-----------|-------------------------------|---------------------------------|
| debug     | Λεπτομερείς πληροφορίες αποσφαλμάτωσης | Σημεία εισόδου/εξόδου λειτουργίας |
| info      | Γενικά ενημερωτικά μηνύματα    | Ενημερώσεις προόδου λειτουργίας  |
| notice    | Κανονικά αλλά σημαντικά γεγονότα | Αλλαγές ρυθμίσεων               |
| warning   | Συνθήκες προειδοποίησης        | Χρήση αποσυρθέντων λειτουργιών  |
| error     | Συνθήκες σφάλματος             | Αποτυχίες λειτουργίας            |
| critical  | Κρίσιμες συνθήκες              | Αποτυχίες συστατικών συστήματος  |
| alert     | Άμεση ενέργεια απαιτείται      | Ανίχνευση διαφθοράς δεδομένων    |
| emergency | Το σύστημα είναι μη χρησιμοποποιήσιμο | Πλήρης αποτυχία συστήματος         |

## Υλοποίηση Ειδοποιήσεων στο MCP

Για να υλοποιήσετε ειδοποιήσεις στο MCP, πρέπει να ρυθμίσετε και τις πλευρές του διακομιστή και του πελάτη ώστε να χειρίζονται ενημερώσεις σε πραγματικό χρόνο. Αυτό επιτρέπει στην εφαρμογή σας να παρέχει άμεση ανατροφοδότηση στους χρήστες κατά τη διάρκεια μακροχρόνιων λειτουργιών.

### Πλευρά Διακομιστή: Αποστολή Ειδοποιήσεων

Ας ξεκινήσουμε με την πλευρά του διακομιστή. Στο MCP, ορίζετε εργαλεία που μπορούν να στέλνουν ειδοποιήσεις κατά την επεξεργασία αιτημάτων. Ο διακομιστής χρησιμοποιεί το αντικείμενο context (συνήθως `ctx`) για να στείλει μηνύματα στον πελάτη.

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

Στο προηγούμενο παράδειγμα, το εργαλείο `process_files` στέλνει τρεις ειδοποιήσεις στον πελάτη καθώς επεξεργάζεται κάθε αρχείο. Η μέθοδος `ctx.info()` χρησιμοποιείται για αποστολή ενημερωτικών μηνυμάτων.

Επιπλέον, για να ενεργοποιηθούν οι ειδοποιήσεις, βεβαιωθείτε ότι ο διακομιστής σας χρησιμοποιεί μεταφορά ροής (όπως `streamable-http`) και ο πελάτης σας υλοποιεί χειριστή μηνυμάτων για να επεξεργάζεται τις ειδοποιήσεις. Δείτε πώς να ρυθμίσετε τον διακομιστή να χρησιμοποιεί τη μεταφορά `streamable-http`:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

Σε αυτό το παράδειγμα .NET, το εργαλείο `ProcessFiles` έχει το χαρακτηριστικό `Tool` και στέλνει τρεις ειδοποιήσεις στον πελάτη καθώς επεξεργάζεται κάθε αρχείο. Η μέθοδος `ctx.Info()` χρησιμοποιείται για αποστολή ενημερωτικών μηνυμάτων.

Για να ενεργοποιήσετε τις ειδοποιήσεις στον διακομιστή MCP .NET σας, βεβαιωθείτε ότι χρησιμοποιείτε μεταφορά ροής:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### Πλευρά Πελάτη: Λήψη Ειδοποιήσεων

Ο πελάτης πρέπει να υλοποιήσει χειριστή μηνυμάτων για να επεξεργάζεται και να εμφανίζει τις ειδοποιήσεις καθώς φτάνουν.

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

Στον παραπάνω κώδικα, η συνάρτηση `message_handler` ελέγχει αν το εισερχόμενο μήνυμα είναι ειδοποίηση. Αν ναι, την εκτυπώνει, διαφορετικά την επεξεργάζεται ως κανονικό μήνυμα διακομιστή. Παρατηρήστε επίσης πώς η `ClientSession` αρχικοποιείται με τον `message_handler` για να χειρίζεται τις εισερχόμενες ειδοποιήσεις.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

Σε αυτό το παράδειγμα .NET, η συνάρτηση `MessageHandler` ελέγχει αν το εισερχόμενο μήνυμα είναι ειδοποίηση. Αν ναι, την εκτυπώνει, διαφορετικά την επεξεργάζεται ως κανονικό μήνυμα διακομιστή. Η `ClientSession` αρχικοποιείται με τον χειριστή μηνυμάτων μέσω των `ClientSessionOptions`.

Για να ενεργοποιήσετε τις ειδοποιήσεις, βεβαιωθείτε ότι ο διακομιστής σας χρησιμοποιεί μεταφορά ροής (όπως `streamable-http`) και ο πελάτης σας υλοποιεί χειριστή μηνυμάτων για να επεξεργάζεται τις ειδοποιήσεις.

## Ειδοποιήσεις Προόδου & Σενάρια

Αυτή η ενότητα εξηγεί την έννοια των ειδοποιήσεων προόδου στο MCP, γιατί έχει σημασία και πώς να τις υλοποιήσετε χρησιμοποιώντας το Streamable HTTP. Επίσης θα βρείτε μια πρακτική άσκηση για να ενισχύσετε την κατανόησή σας.

Οι ειδοποιήσεις προόδου είναι μηνύματα σε πραγματικό χρόνο που αποστέλλονται από τον διακομιστή προς τον πελάτη κατά τη διάρκεια μακροχρόνιων λειτουργιών. Αντί να περιμένει να τελειώσει όλη η διαδικασία, ο διακομιστής κρατά τον πελάτη ενημερωμένο για την τρέχουσα κατάσταση. Αυτό βελτιώνει τη διαφάνεια, την εμπειρία χρήστη και διευκολύνει τον εντοπισμό σφαλμάτων.

**Παράδειγμα:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### Γιατί να χρησιμοποιήσετε τις Ειδοποιήσεις Προόδου;

Οι ειδοποιήσεις προόδου είναι ουσιώδεις για πολλούς λόγους:

- **Καλύτερη εμπειρία χρήστη:** Οι χρήστες βλέπουν ενημερώσεις καθώς προχωράει η εργασία, όχι μόνο στο τέλος.
- **Ανατροφοδότηση σε πραγματικό χρόνο:** Οι πελάτες μπορούν να εμφανίζουν μπάρες προόδου ή καταγραφές, καθιστώντας την εφαρμογή πιο ανταποκρινόμενη.
- **Εύκολος εντοπισμός σφαλμάτων και παρακολούθηση:** Οι προγραμματιστές και οι χρήστες μπορούν να δουν πού πιθανώς είναι αργή ή κολλημένη μια διαδικασία.

### Πώς να υλοποιήσετε τις Ειδοποιήσεις Προόδου

Δείτε πώς να υλοποιήσετε τις ειδοποιήσεις προόδου στο MCP:

- **Στον διακομιστή:** Χρησιμοποιήστε `ctx.info()` ή `ctx.log()` για να στείλετε ειδοποιήσεις καθώς κάθε στοιχείο επεξεργάζεται. Αυτό στέλνει μήνυμα στον πελάτη πριν ετοιμαστεί το κύριο αποτέλεσμα.
- **Στον πελάτη:** Υλοποιήστε χειριστή μηνυμάτων που ακούει και εμφανίζει ειδοποιήσεις καθώς φτάνουν. Αυτός ο χειριστής διαχωρίζει τις ειδοποιήσεις από το τελικό αποτέλεσμα.

**Παράδειγμα Διακομιστή:**

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**Παράδειγμα Πελάτη:**

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## Ζητήματα Ασφαλείας

Κατά την υλοποίηση MCP διακομιστών με μεταφορές βάσει HTTP, η ασφάλεια γίνεται κορυφαία ανησυχία που απαιτεί προσεκτική προσοχή σε πολλούς διαύλους επίθεσης και μηχανισμούς προστασίας.

### Επισκόπηση

Η ασφάλεια είναι κρίσιμη όταν εκτίθενται MCP διακομιστές μέσω HTTP. Το Streamable HTTP εισάγει νέες επιφάνειες επίθεσης και απαιτεί προσεκτική διαμόρφωση.

### Βασικά Σημεία


- **Επικύρωση Επικεφαλίδας Origin**: Πάντα επικυρώνετε την επικεφαλίδα `Origin` για να αποτρέψετε επιθέσεις DNS rebinding.
- **Δέσμευση σε localhost**: Για τοπική ανάπτυξη, δεσμεύστε τους διακομιστές σε `localhost` για να αποφύγετε την έκθεσή τους στο δημόσιο διαδίκτυο.
- **Πιστοποίηση**: Υλοποιήστε πιστοποίηση (π.χ., κλειδιά API, OAuth) για παραγωγικές αναπτύξεις.
- **CORS**: Διαμορφώστε τις πολιτικές Cross-Origin Resource Sharing (CORS) για να περιορίσετε την πρόσβαση.
- **HTTPS**: Χρησιμοποιήστε HTTPS στην παραγωγή για κρυπτογράφηση της κυκλοφορίας.

### Καλύτερες Πρακτικές

- Ποτέ μην εμπιστεύεστε εισερχόμενα αιτήματα χωρίς επικύρωση.
- Καταγράψτε και παρακολουθήστε όλη την πρόσβαση και τα σφάλματα.
- Ενημερώνετε τακτικά τις εξαρτήσεις για να επιδιορθώνετε τρωτά σημεία ασφαλείας.

### Προκλήσεις

- Ισορροπία ανάμεσα στην ασφάλεια και την ευκολία ανάπτυξης
- Εξασφάλιση συμβατότητας με διάφορα περιβάλλοντα πελατών

## Αναβάθμιση από SSE σε Streamable HTTP

Για εφαρμογές που χρησιμοποιούν Server-Sent Events (SSE), η μετάβαση σε Streamable HTTP προσφέρει βελτιωμένες δυνατότητες και καλύτερη μακροπρόθεσμη βιωσιμότητα για τις υλοποιήσεις MCP σας.

### Γιατί να Αναβαθμίσετε;

Υπάρχουν δύο ισχυροί λόγοι για να αναβαθμίσετε από SSE σε Streamable HTTP:

- Το Streamable HTTP προσφέρει καλύτερη επεκτασιμότητα, συμβατότητα και πλουσιότερη υποστήριξη ειδοποιήσεων από το SSE.
- Είναι ο συνιστώμενος μεταφορικός μηχανισμός για νέες εφαρμογές MCP.

### Βήματα Μεταφοράς

Δείτε πώς μπορείτε να μεταβείτε από SSE σε Streamable HTTP στις εφαρμογές MCP σας:

- **Ενημερώστε τον κώδικα διακομιστή** για να χρησιμοποιεί `transport="streamable-http"` στο `mcp.run()`.
- **Ενημερώστε τον κώδικα πελάτη** για να χρησιμοποιεί `streamablehttp_client` αντί για τον SSE client.
- **Υλοποιήστε έναν χειριστή μηνυμάτων** στον πελάτη για να επεξεργάζεται τις ειδοποιήσεις.
- **Δοκιμάστε για συμβατότητα** με υπάρχοντα εργαλεία και ροές εργασίας.

### Διατήρηση της Συμβατότητας

Συνιστάται να διατηρείτε συμβατότητα με τους υπάρχοντες πελάτες SSE κατά τη διαδικασία μετανάστευσης. Εδώ είναι μερικές στρατηγικές:

- Μπορείτε να υποστηρίξετε και τις δύο επιλογές, SSE και Streamable HTTP, τρέχοντας και τα δύο πρωτόκολλα σε διαφορετικά endpoints.
- Μεταφέρετε τους πελάτες σταδιακά στον νέο μεταφορικό μηχανισμό.

### Προκλήσεις

Βεβαιωθείτε ότι αντιμετωπίζετε τις παρακάτω προκλήσεις κατά τη μετανάστευση:

- Ενημέρωση όλων των πελατών
- Διαχείριση διαφορών στην παράδοση των ειδοποιήσεων
