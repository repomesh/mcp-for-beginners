# मॉडल कंटेक्स्ट प्रोटोकॉल (MCP) के साथ HTTPS स्ट्रीमिंग

यह अध्याय मॉडल कंटेक्स्ट प्रोटोकॉल (MCP) का उपयोग करके HTTPS के साथ सुरक्षित, स्केलेबल और रीयल-टाइम स्ट्रीमिंग को लागू करने के लिए एक व्यापक गाइड प्रदान करता है। इसमें स्ट्रीमिंग के लिए प्रेरणा, उपलब्ध ट्रांसपोर्ट तंत्र, MCP में स्ट्रीमेबल HTTP को कैसे लागू करें, सुरक्षा सर्वोत्तम प्रथाएँ, SSE से माइग्रेशन, और अपने स्वयं के स्ट्रीमिंग MCP एप्लिकेशन बनाने के लिए व्यावहारिक मार्गदर्शन शामिल हैं।

> **आगे देखते हुए:** यह पाठ MCP स्पेसिफिकेशन 2025-11-25 के तहत स्ट्रीमेबल HTTP का वर्णन करता है, जहां एक सेशन `initialize` के दौरान स्थापित होता है और `Mcp-Session-Id` हेडर के साथ पिन किया जाता है। `2026-07-28` रिलीज़ उम्मीदवार पूरी तरह से हैंडशेक और सेशन ID हटा देता है, जिससे हर अनुरोध आत्मनिर्भर होता है और किसी भी सर्वर इंस्टेंस को स्टिकी सेशन्स के बिना रूट किया जा सकता है। विवरण के लिए देखें [MCP में क्या बदल रहा है: 2026-07-28 रिलीज़ उम्मीदवार](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md)।

## MCP में ट्रांसपोर्ट तंत्र और स्ट्रीमिंग

यह अनुभाग MCP में उपलब्ध विभिन्न ट्रांसपोर्ट तंत्रों और क्लाइंट और सर्वर के बीच रीयल-टाइम संचार के लिए स्ट्रीमिंग क्षमताओं को सक्षम करने में उनकी भूमिका का अन्वेषण करता है।

### ट्रांसपोर्ट तंत्र क्या है?

एक ट्रांसपोर्ट तंत्र यह परिभाषित करता है कि क्लाइंट और सर्वर के बीच डेटा कैसे आदान-प्रदान किया जाता है। MCP विभिन्न वातावरणों और आवश्यकताओं के लिए कई ट्रांसपोर्ट प्रकारों का समर्थन करता है:

- **stdio**: मानक इनपुट/आउटपुट, स्थानीय और CLI-आधारित उपकरणों के लिए उपयुक्त। सरल लेकिन वेब या क्लाउड के लिए उपयुक्त नहीं।
- **SSE (सर्वर-सेंट इवेंट्स)**: HTTP पर क्लाइंटों को रीयल-टाइम अपडेट पुश करने की अनुमति देता है। वेब UI के लिए अच्छा, लेकिन स्केलेबिलिटी और लचीलापन में सीमित। MCP स्पेसिफिकेशन 2025-06-18 के अनुसार, स्टैंडअलोन SSE ट्रांसपोर्ट अब बंद कर दिया गया है और इसे "स्ट्रीमेबल HTTP" ट्रांसपोर्ट द्वारा प्रतिस्थापित किया गया है।
- **Streamable HTTP**: आधुनिक HTTP-आधारित स्ट्रीमिंग ट्रांसपोर्ट, अधिसूचनाओं का समर्थन करता है और बेहतर स्केलेबिलिटी प्रदान करता है। अधिकांश प्रोडक्शन और क्लाउड परिदृश्यों के लिए अनुशंसित।

### तुलना सारणी

इन ट्रांसपोर्ट तंत्रों के बीच अंतर को समझने के लिए नीचे दी गई तुलना सारणी देखें:

| ट्रांसपोर्ट        | रीयल-टाइम अपडेट्स | स्ट्रीमिंग | स्केलेबिलिटी | उपयोग मामला              |
|-------------------|--------------------|-----------|---------------|-------------------------|
| stdio             | नहीं               | नहीं      | कम            | स्थानीय CLI उपकरण       |
| SSE               | हाँ                | हाँ       | मध्यम         | वेब, रीयल-टाइम अपडेट्स|
| Streamable HTTP   | हाँ                | हाँ       | उच्च          | क्लाउड, मल्टी-क्लाइंट   |

> **टिप:** सही ट्रांसपोर्ट चुनना प्रदर्शन, स्केलेबिलिटी और उपयोगकर्ता अनुभव को प्रभावित करता है। **Streamable HTTP** आधुनिक, स्केलेबल, और क्लाउड-तैयार एप्लिकेशन के लिए अनुशंसित है।

पिछले अध्यायों में आपको जो stdio और SSE ट्रांसपोर्ट दिखाए गए थे, और यह अध्याय जिसमें स्ट्रीमेबल HTTP ट्रांसपोर्ट शामिल है, कृपया ध्यान दें।

## स्ट्रीमिंग: अवधारणाएँ और प्रेरणा

स्ट्रीमिंग के मौलिक सिद्धांतों और प्रेरणाओं को समझना प्रभावी रीयल-टाइम संचार प्रणाली लागू करने के लिए आवश्यक है।

**स्ट्रीमिंग** नेटवर्क प्रोग्रामिंग में एक तकनीक है जो डेटा को छोटे, प्रबंधनीय टुकड़ों या घटनाओं की श्रंखला के रूप में भेजने और प्राप्त करने की अनुमति देती है, बजाय इसके कि पूर्ण प्रतिक्रिया के तैयार होने तक इंतजार किया जाए। यह विशेष रूप से उपयोगी है:

- बड़े फाइलों या डेटा सेट के लिए।
- रीयल-टाइम अपडेट (जैसे, चैट, प्रोग्रेस बार) के लिए।
- लंबे समय तक चलने वाले कम्प्यूटेशन जहां आप उपयोगकर्ता को सूचित रखना चाहते हैं।

उच्च स्तर पर स्ट्रीमिंग के बारे में आपको जो जानना चाहिए वह यह है:

- डेटा लगातार वितरित किया जाता है, एक ही बार में नहीं।
- क्लाइंट प्राप्त होते ही डेटा को प्रोसेस कर सकता है।
- प्रतीत होने वाली विलंबता को कम करता है और उपयोगकर्ता अनुभव को बेहतर बनाता है।

### स्ट्रीमिंग क्यों उपयोग करें?

स्ट्रीमिंग के उपयोग के कारण निम्नलिखित हैं:

- उपयोगकर्ताओं को तुरंत प्रतिक्रिया मिलती है, केवल अंत में नहीं।
- रीयल-टाइम एप्लिकेशन और प्रतिक्रियाशील UI को सक्षम करता है।
- नेटवर्क और कंप्यूटिंग संसाधनों का अधिक कुशल उपयोग।

### सरल उदाहरण: HTTP स्ट्रीमिंग सर्वर और क्लाइंट

यहां एक सरल उदाहरण है कि स्ट्रीमिंग को कैसे लागू किया जा सकता है:

#### पायथन

**सर्वर (पायथन, FastAPI और StreamingResponse का उपयोग करते हुए):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**क्लाइंट (पायथन, requests का उपयोग करते हुए):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

यह उदाहरण यह दिखाता है कि सर्वर क्लाइंट को उपलब्ध होते ही संदेशों की एक श्रृंखला भेज रहा है, बजाय इसके कि सभी संदेश तैयार होने तक इंतजार करे।

**यह कैसे काम करता है:**

- सर्वर प्रत्येक संदेश को जैसे ही वह तैयार होता है, यील्ड करता है।
- क्लाइंट प्रत्येक प्राप्त हिस्से को प्राप्त करता है और प्रिंट करता है।

**आवश्यकताएँ:**

- सर्वर को स्ट्रीमिंग प्रतिक्रिया का उपयोग करना चाहिए (जैसे FastAPI में `StreamingResponse`)।
- क्लाइंट को प्रतिक्रिया को स्ट्रीम के रूप में प्रोसेस करना चाहिए (`stream=True` requests में)।
- Content-Type आमतौर पर `text/event-stream` या `application/octet-stream` होता है।

#### जावा

**सर्वर (जावा, Spring Boot और Server-Sent Events का उपयोग करते हुए):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**क्लाइंट (जावा, Spring WebFlux WebClient का उपयोग करते हुए):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**जावा कार्यान्वयन नोट्स:**

- स्ट्रीमिंग के लिए Spring Boot का रिएक्टिव स्टैक `Flux` का उपयोग करता है
- `ServerSentEvent` ईवेंट प्रकारों के साथ संरचित स्ट्रीमिंग प्रदान करता है
- `WebClient` `bodyToFlux()` के साथ रिएक्टिव स्ट्रीमिंग सेवन सक्षम करता है
- `delayElements()` घटनाओं के बीच प्रोसेसिंग समय का अनुकरण करता है
- घटनाओं के लिए प्रकार हो सकते हैं (`info`, `result`) बेहतर क्लाइंट हैंडलिंग के लिए

### तुलना: क्लासिक स्ट्रीमिंग बनाम MCP स्ट्रीमिंग

"क्लासिक" तरीके से स्ट्रीमिंग कैसे काम करती है और MCP में कैसे काम करती है, इसके बीच के भेद को इस प्रकार दिखाया जा सकता है:

| फीचर                  | क्लासिक HTTP स्ट्रीमिंग        | MCP स्ट्रीमिंग (अधिसूचनाएं)     |
|------------------------|------------------------------|---------------------------------|
| मुख्य प्रतिक्रिया       | टुकड़ेबद्ध                     | एकल, अंत में                    |
| प्रगति अपडेट्स        | डेटा खंड के रूप में भेजा जाता है | अधिसूचनाओं के रूप में भेजा जाता है|
| क्लाइंट आवश्यकताएँ     | स्ट्रीम को प्रोसेस करना आवश्यक है | संदेश हैंडलर को लागू करना आवश्यक|
| उपयोग मामला            | बड़े फाइल, AI टोकन स्ट्रीम    | प्रगति, लॉग, रीयल-टाइम प्रतिक्रिया|

### देखे गए प्रमुख अंतर

इसके अतिरिक्त, यहां कुछ मुख्य अंतर हैं:

- **संचार पैटर्न:**
  - क्लासिक HTTP स्ट्रीमिंग: सरल टुकड़ेबद्ध ट्रांसफर एन्कोडिंग का उपयोग कर डेटा हिस्सों में भेजता है
  - MCP स्ट्रीमिंग: JSON-RPC प्रोटोकॉल के साथ संरचित अधिसूचना प्रणाली का उपयोग करता है

- **संदेश प्रारूप:**
  - क्लासिक HTTP: नई पंक्तियों के साथ प्लेन टेक्स्ट टुकड़े
  - MCP: मेटाडेटा के साथ संरचित LoggingMessageNotification ऑब्जेक्ट्स

- **क्लाइंट कार्यान्वयन:**
  - क्लासिक HTTP: स्ट्रीमिंग प्रतिक्रियाओं को प्रोसेस करने वाला सरल क्लाइंट
  - MCP: विभिन्न प्रकार के संदेशों को प्रोसेस करने वाला संदेश हैंडलर के साथ अधिक परिष्कृत क्लाइंट

- **प्रगति अपडेट्स:**
  - क्लासिक HTTP: प्रगति मुख्य प्रतिक्रिया स्ट्रीम का हिस्सा होती है
  - MCP: प्रगति को अलग अधिसूचना संदेशों के माध्यम से भेजा जाता है जबकि मुख्य प्रतिक्रिया अंत में आती है

### अनुशंसाएँ

क्लासिक स्ट्रीमिंग (जैसे हमने ऊपर `/stream` उपयोग करते हुए दिखाया) बनाम MCP स्ट्रीमिंग के बीच चुनते समय कुछ चीजें हम अनुशंसित करते हैं।

- **सरल स्ट्रीमिंग आवश्यकताओं के लिए:** क्लासिक HTTP स्ट्रीमिंग लागू करने में सरल है और बुनियादी स्ट्रीमिंग आवश्यकताओं के लिए पर्याप्त है।

- **जटिल, इंटरैक्टिव एप्लिकेशन के लिए:** MCP स्ट्रीमिंग अधिक संरचित दृष्टिकोण प्रदान करता है, जिसमें समृद्ध मेटाडेटा और अधिसूचनाओं व अंतिम परिणामों के बीच पृथक्करण होता है।

- **AI एप्लिकेशन के लिए:** MCP की अधिसूचना प्रणाली लंबे समय तक चलने वाले AI कार्यों के लिए विशेष रूप से उपयोगी है जहाँ आप उपयोगकर्ताओं को प्रगति से सूचित रखना चाहते हैं।

## MCP में स्ट्रीमिंग

ठीक है, अब तक आपने क्लासिक स्ट्रीमिंग और MCP स्ट्रीमिंग के बीच अंतर और अनुशंसाएँ देखी हैं। आइए विस्तार से देखें कि आप MCP में स्ट्रीमिंग का लाभ कैसे उठा सकते हैं।

MCP फ्रेमवर्क के भीतर स्ट्रीमिंग कैसे काम करती है यह समझना प्रतिक्रिया क्षमता वाले एप्लिकेशन बनाने के लिए आवश्यक है जो लंबे समय तक चलने वाले ऑपरेशनों के दौरान उपयोगकर्ताओं को रीयल-टाइम फीडबैक प्रदान करते हैं।

MCP में, स्ट्रीमिंग का मतलब मुख्य प्रतिक्रिया को टुकड़ों में भेजना नहीं है, बल्कि एक उपकरण अनुरोध को प्रोसेस करते समय क्लाइंट को **अधिसूचनाएं** भेजने का है। ये अधिसूचनाएं प्रगति अपडेट, लॉग, या अन्य घटनाएं हो सकती हैं।

### यह कैसे काम करता है

मुख्य परिणाम अभी भी एक ही प्रतिक्रिया के रूप में भेजा जाता है। हालांकि, प्रोसेसिंग के दौरान अलग-अलग संदेश के रूप में अधिसूचनाएं भेजी जा सकती हैं और इस प्रकार क्लाइंट को रीयल टाइम में अपडेट किया जाता है। क्लाइंट को इन अधिसूचनाओं को संभालने और प्रदर्शित करने में सक्षम होना चाहिए।

## अधिसूचना क्या है?

हमने "अधिसूचना" कहा, तो MCP के संदर्भ में इसका क्या मतलब है?

अधिसूचना सर्वर से क्लाइंट को भेजा गया एक संदेश है जो लंबे समय तक चलने वाले ऑपरेशन के दौरान प्रगति, स्थिति, या अन्य घटनाओं के बारे में जानकारी देता है। अधिसूचनाएं पारदर्शिता और उपयोगकर्ता अनुभव में सुधार करती हैं।

उदाहरण के लिए, एक क्लाइंट को अधिसूचना भेजनी चाहिए जब सर्वर के साथ प्रारंभिक हैंडशेक हो जाए।

अधिसूचना इस प्रकार एक JSON संदेश की तरह दिखती है:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

अधिसूचनाएं MCP में ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) विषय के अंतर्गत आती हैं।

लॉगिंग को कार्यान्वित करने के लिए, सर्वर को इसे इस प्रकार सुविधा/क्षमता के रूप में सक्षम करना होगा:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> उपयोग किए गए SDK के आधार पर, लॉगिंग डिफ़ॉल्ट रूप से सक्षम हो सकता है, या आपको अपने सर्वर कॉन्फ़िगरेशन में इसे स्पष्ट रूप से सक्षम करना पड़ सकता है।

विभिन्न प्रकार की अधिसूचनाएं होती हैं:

| स्तर          | विवरण                       | उदाहरण उपयोग मामला             |
|--------------|-----------------------------|---------------------------------|
| debug        | विस्तृत डिबग जानकारी        | फ़ंक्शन प्रवेश/निकास बिंदु      |
| info         | सामान्य सूचना संदेश          | ऑपरेशन प्रगति अपडेट्स           |
| notice       | सामान्य लेकिन महत्वपूर्ण घटनाएं | कॉन्फ़िगरेशन परिवर्तन           |
| warning      | चेतावनी की स्थिति           | अप्रचलित फीचर उपयोग            |
| error        | त्रुटि की स्थिति            | ऑपरेशन विफलताएं                |
| critical     | गंभीर स्थिति               | सिस्टम घटक विफलताएं            |
| alert        | तत्काल कार्रवाई आवश्यक      | डेटा भ्रष्टाचार का पता चला    |
| emergency    | सिस्टम अनुपयोगी है          | पूर्ण सिस्टम विफलता            |

## MCP में अधिसूचनाएं लागू करना

MCP में अधिसूचनाओं को लागू करने के लिए, आपको रीयल-टाइम अपडेट्स को संभालने के लिए सर्वर और क्लाइंट दोनों पक्षों को सेट अप करना होगा। यह आपके एप्लिकेशन को लंबे समय तक चलने वाले ऑपरेशनों के दौरान उपयोगकर्ताओं को तुरंत फीडबैक प्रदान करने में सक्षम बनाता है।

### सर्वर-पक्ष: अधिसूचनाएं भेजना

चलिए सर्वर पक्ष से शुरू करते हैं। MCP में, आप ऐसे उपकरण परिभाषित करते हैं जो अनुरोधों को प्रोसेस करते समय अधिसूचनाएं भेज सकते हैं। सर्वर संदर्भ ऑब्जेक्ट (आमतौर पर `ctx`) का उपयोग क्लाइंट को संदेश भेजने के लिए करता है।

#### पायथन

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

पिछले उदाहरण में, `process_files` टूल तीन अधिसूचनाएं क्लाइंट को भेजता है क्योंकि वह प्रत्येक फ़ाइल को प्रोसेस करता है। `ctx.info()` मेथड का उपयोग सूचना संदेश भेजने के लिए किया जाता है।

इसके अतिरिक्त, अधिसूचनाओं को सक्षम करने के लिए, सुनिश्चित करें कि आपका सर्वर स्ट्रीमिंग ट्रांसपोर्ट (जैसे `streamable-http`) का उपयोग करता है और आपका क्लाइंट अधिसूचनाओं को प्रोसेस करने के लिए एक संदेश हैंडलर लागू करता है। सर्वर को `streamable-http` ट्रांसपोर्ट का उपयोग करने के लिए कैसे सेट करें, यह इस प्रकार है:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

इस .NET उदाहरण में, `ProcessFiles` टूल `Tool` एट्रिब्यूट के साथ सजाया गया है और जैसे ही वह प्रत्येक फ़ाइल प्रोसेस करता है क्लाइंट को तीन अधिसूचनाएं भेजता है। `ctx.Info()` मेथड सूचना संदेश भेजने के लिए उपयोग किया जाता है।

अपने .NET MCP सर्वर में अधिसूचनाएं सक्षम करने के लिए, सुनिश्चित करें कि आप स्ट्रीमिंग ट्रांसपोर्ट का उपयोग कर रहे हैं:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### क्लाइंट-पक्ष: अधिसूचनाएं प्राप्त करना

क्लाइंट को अधिसूचनाओं को प्रोसेस और प्रदर्शित करने के लिए एक संदेश हैंडलर लागू करना चाहिए जब वे आते हैं।

#### पायथन

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

पिछले कोड में, `message_handler` फ़ंक्शन जांचता है कि क्या आने वाला संदेश एक अधिसूचना है। यदि हाँ, तो यह अधिसूचना प्रिंट करता है; अन्यथा, इसे एक सामान्य सर्वर संदेश के रूप में प्रोसेस करता है। इसके अलावा ध्यान दें कि `ClientSession` `message_handler` के साथ इनिशियलाइज़ किया गया है ताकि आने वाली अधिसूचनाओं को हैंडल किया जा सके।

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

इस .NET उदाहरण में, `MessageHandler` फ़ंक्शन जांचता है कि क्या आने वाला संदेश अधिसूचना है। यदि हाँ, तो यह अधिसूचना प्रिंट करता है; अन्यथा, इसे सामान्य सर्वर संदेश के रूप में प्रोसेस करता है। `ClientSession` को `ClientSessionOptions` के माध्यम से संदेश हैंडलर के साथ इनिशियलाइज़ किया गया है।

अधिसूचनाओं को सक्षम करने के लिए, सुनिश्चित करें कि आपका सर्वर स्ट्रीमिंग ट्रांसपोर्ट (जैसे `streamable-http`) का उपयोग करता है और आपका क्लाइंट अधिसूचनाओं को प्रोसेस करने के लिए एक संदेश हैंडलर लागू करता है।

## प्रगति अधिसूचनाएं और उपयोग के परिदृश्य

यह अनुभाग MCP में प्रगति अधिसूचनाओं की अवधारणा, वे क्यों महत्वपूर्ण हैं, और Streamable HTTP का उपयोग करके उन्हें कैसे लागू करें, यह समझाता है। आप एक व्यावहारिक असाइनमेंट भी पाएंगे जो आपकी समझ को मजबूत करता है।

प्रगति अधिसूचनाएं लंबे समय तक चलने वाले ऑपरेशनों के दौरान सर्वर से क्लाइंट को भेजे जाने वाले रीयल-टाइम संदेश होते हैं। पूरी प्रक्रिया समाप्त होने के बजाय, सर्वर क्लाइंट को वर्तमान स्थिति के बारे में अपडेट रखता है। इससे पारदर्शिता, उपयोगकर्ता अनुभव बेहतर होता है और डिबगिंग आसान होती है।

**उदाहरण:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### प्रगति अधिसूचनाओं का उपयोग क्यों करें?

प्रगति अधिसूचनाएं कई कारणों से आवश्यक हैं:

- **बेहतर उपयोगकर्ता अनुभव:** उपयोगकर्ता कार्य प्रगति के रूप में अपडेट देखते हैं, केवल अंत में नहीं।
- **रीयल-टाइम फीडबैक:** क्लाइंट प्रगति बार या लॉग दिखा सकते हैं, जिससे ऐप प्रतिक्रियाशील लगता है।
- **डिबगिंग और मॉनिटरिंग आसान:** डेवलपर्स और उपयोगकर्ता देख सकते हैं कि प्रक्रिया कहाँ धीमी या अटकी हुई है।

### प्रगति अधिसूचनाओं को कैसे लागू करें

यहां बताया गया है कि आप MCP में प्रगति अधिसूचनाओं को कैसे लागू कर सकते हैं:

- **सर्वर पर:** `ctx.info()` या `ctx.log()` का उपयोग करें ताकि जैसे ही प्रत्येक आइटम प्रोसेस हो को यह अधिसूचना भेजी जाए। यह मुख्य परिणाम तैयार होने से पहले क्लाइंट को संदेश भेजता है।
- **क्लाइंट पर:** एक संदेश हैंडलर लागू करें जो अधिसूचनाओं के लिए सुनता है और उन्हें आते ही प्रदर्शित करता है। यह हैंडलर अधिसूचनाओं और अंतिम परिणाम के बीच भेद करता है।

**सर्वर उदाहरण:**

#### पायथन

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**क्लाइंट उदाहरण:**

#### पायथन

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## सुरक्षा विचार

जब HTTP-आधारित ट्रांसपोर्ट के साथ MCP सर्वर लागू करते हैं, तो सुरक्षा एक प्रमुख चिंता बन जाती है जिसे कई हमले के वेक्टरों और सुरक्षा तंत्रों पर सावधानी से ध्यान देने की आवश्यकता होती है।

### अवलोकन

HTTP पर MCP सर्वर खोलते समय सुरक्षा अत्यंत महत्वपूर्ण है। स्ट्रीमेबल HTTP नए हमले के सतह क्षेत्र प्रस्तुत करता है और सावधानीपूर्वक कॉन्फ़िगरेशन की आवश्यकता होती है।

### मुख्य बिंदु


- **Origin Header Validation**: DNS रीबाइंडिंग हमलों को रोकने के लिए हमेशा `Origin` हेडर का सत्यापन करें।
- **Localhost Binding**: स्थानीय विकास के लिए, सर्वरों को `localhost` से बाँधें ताकि उन्हें सार्वजनिक इंटरनेट पर उजागर होने से बचाया जा सके।
- **Authentication**: उत्पादन परिनियोजन के लिए प्रमाणीकरण (जैसे, API कुंजी, OAuth) लागू करें।
- **CORS**: पहुँच को प्रतिबंधित करने के लिए Cross-Origin Resource Sharing (CORS) नीतियाँ कॉन्फ़िगर करें।
- **HTTPS**: ट्रैफ़िक को एन्क्रिप्ट करने के लिए उत्पादन में HTTPS का उपयोग करें।

### सर्वोत्तम प्रथाएँ

- सत्यापन के बिना आने वाले अनुरोधों पर कभी विश्वास न करें।
- सभी पहुँच और त्रुटियों को लॉग और मॉनिटर करें।
- सुरक्षा खामियों को पैच करने के लिए निर्भरताओं को नियमित रूप से अपडेट करें।

### चुनौतियाँ

- विकास की सहजता के साथ सुरक्षा का संतुलन बनाना
- विभिन्न क्लाइंट वातावरणों के साथ संगतता सुनिश्चित करना

## SSE से Streamable HTTP में अपग्रेड करना

उन अनुप्रयोगों के लिए जो वर्तमान में Server-Sent Events (SSE) का उपयोग कर रहे हैं, Streamable HTTP पर माइग्रेट करना आपके MCP कार्यान्वयन के लिए उन्नत क्षमताएँ और बेहतर दीर्घकालिक स्थिरता प्रदान करता है।

### क्यों अपग्रेड करें?

SSE से Streamable HTTP में अपग्रेड करने के दो प्रमुख कारण हैं:

- Streamable HTTP बेहतर स्केलेबिलिटी, संगतता, और SCC से अधिक समृद्ध अधिसूचना समर्थन प्रदान करता है।
- यह नए MCP अनुप्रयोगों के लिए अनुशंसित ट्रांसपोर्ट है।

### माइग्रेशन चरण

यहाँ बताया गया है कि आप अपने MCP अनुप्रयोगों में SSE से Streamable HTTP पर कैसे माइग्रेट कर सकते हैं:

- सर्वर कोड को अपडेट करें ताकि `mcp.run()` में `transport="streamable-http"` का उपयोग हो।
- क्लाइंट कोड को SSE क्लाइंट के बजाय `streamablehttp_client` का उपयोग करने के लिए अपडेट करें।
- अधिसूचनाओं को संसाधित करने के लिए क्लाइंट में एक संदेश हैंडलर लागू करें।
- मौजूदा टूल और वर्कफ़्लो के साथ संगतता के लिए परीक्षण करें।

### संगतता बनाए रखना

माइग्रेशन प्रक्रिया के दौरान मौजूदा SSE क्लाइंट के साथ संगतता बनाए रखना सलाह योग्य है। यहाँ कुछ रणनीतियाँ हैं:

- आप विभिन्न एंडपॉइंट्स पर दोनों ट्रांसपोर्ट्स को चलाकर SSE और Streamable HTTP दोनों का समर्थन कर सकते हैं।
- क्लाइंट्स को धीरे-धीरे नए ट्रांसपोर्ट पर माइग्रेट करें।

### चुनौतियाँ

माइग्रेशन के दौरान निम्न चुनौतियों को संबोधित करना सुनिश्चित करें:

- सभी क्लाइंट्स को अपडेट करना सुनिश्चित करना
- अधिसूचना वितरण में भिन्नताओं को संभालना

## सुरक्षा विचार

किसी भी सर्वर को लागू करते समय सुरक्षा सर्वोच्च प्राथमिकता होनी चाहिए, विशेषकर MCP में Streamable HTTP जैसे HTTP-आधारित ट्रांसपोर्ट का उपयोग करते समय।

MCP सर्वर HTTP-आधारित ट्रांसपोर्ट के साथ लागू करते समय, सुरक्षा एक अत्यंत महत्वपूर्ण विषय हो जाता है जिसमें कई हमले के वेक्टर और सुरक्षा उपायों पर सावधानीपूर्वक ध्यान देना आवश्यक है।

### अवलोकन

MCP सर्वरों को HTTP पर उजागर करते समय सुरक्षा महत्वपूर्ण होती है। Streamable HTTP नए हमले के सतहें पेश करता है और सावधानीपूर्वक कॉन्फ़िगरेशन की आवश्यकता है।

यहाँ कुछ प्रमुख सुरक्षा विचार हैं:

- **Origin Header Validation**: DNS रीबाइंडिंग हमलों को रोकने के लिए हमेशा `Origin` हेडर का सत्यापन करें।
- **Localhost Binding**: स्थानीय विकास के लिए, सर्वरों को `localhost` से बाँधें ताकि उन्हें सार्वजनिक इंटरनेट पर उजागर होने से बचाया जा सके।
- **Authentication**: उत्पादन परिनियोजन के लिए प्रमाणीकरण (जैसे, API कुंजी, OAuth) लागू करें।
- **CORS**: पहुँच को प्रतिबंधित करने के लिए Cross-Origin Resource Sharing (CORS) नीतियाँ कॉन्फ़िगर करें।
- **HTTPS**: ट्रैफ़िक को एन्क्रिप्ट करने के लिए उत्पादन में HTTPS का उपयोग करें।

### सर्वोत्तम प्रथाएँ

इसके अतिरिक्त, अपने MCP स्ट्रीमिंग सर्वर में सुरक्षा लागू करते समय यहां कुछ सर्वोत्तम प्रथाओं का पालन करें:

- सत्यापन के बिना आने वाले अनुरोधों पर कभी विश्वास न करें।
- सभी पहुँच और त्रुटियों को लॉग और मॉनिटर करें।
- सुरक्षा खामियों को पैच करने के लिए निर्भरताओं को नियमित रूप से अपडेट करें।

### चुनौतियाँ

MCP स्ट्रीमिंग सर्वरों में सुरक्षा लागू करते समय आपको कुछ चुनौतियाँ आएंगी:

- विकास की सहजता के साथ सुरक्षा का संतुलन बनाना
- विभिन्न क्लाइंट वातावरणों के साथ संगतता सुनिश्चित करना

### असाइनमेंट: अपना खुद का स्ट्रीमिंग MCP ऐप बनाएं

**परिदृश्य:**
एक MCP सर्वर और क्लाइंट बनाएं जहाँ सर्वर आइटमों की एक सूची (जैसे, फ़ाइलें या दस्तावेज़) संसाधित करता है और प्रत्येक संसाधित आइटम के लिए एक अधिसूचना भेजता है। क्लाइंट को प्रत्येक अधिसूचना को आते ही प्रदर्शित करना चाहिए।

**चरण:**

1. एक सर्वर टूल लागू करें जो एक सूची संसाधित करें और प्रत्येक आइटम के लिए अधिसूचनाएँ भेजे।
2. एक क्लाइंट लागू करें जिसमें अधिसूचनाओं को वास्तविक समय में प्रदर्शित करने के लिए एक संदेश हैंडलर हो।
3. अपने कार्यान्वयन का परीक्षण करने के लिए सर्वर और क्लाइंट दोनों चलाएं, और अधिसूचनाओं को देखें।
