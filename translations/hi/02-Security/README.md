# MCP सुरक्षा: AI सिस्टम के लिए व्यापक संरक्षण

[![MCP सुरक्षा सर्वोत्तम प्रथाएँ](../../../translated_images/hi/03.175aed6dedae133f.webp)](https://youtu.be/88No8pw706o)

_(इस पाठ का वीडियो देखने के लिए ऊपर की छवि पर क्लिक करें)_

सुरक्षा AI सिस्टम डिज़ाइन का मूलभूत हिस्सा है, इसलिए हम इसे हमारे दूसरे भाग के रूप में प्राथमिकता देते हैं। यह Microsoft के [Secure Future Initiative](https://www.microsoft.com/security/blog/2025/04/17/microsofts-secure-by-design-journey-one-year-of-success/) के **डिज़ाइन द्वारा सुरक्षित** सिद्धांत के अनुरूप है।

मॉडल संदर्भ प्रोटोकॉल (MCP) AI-संचालित अनुप्रयोगों में शक्तिशाली नई क्षमताएँ लाता है, साथ ही ऐसे अनूठे सुरक्षा चुनौतियाँ प्रस्तुत करता है जो पारंपरिक सॉफ़्टवेयर जोखिमों से परे हैं। MCP सिस्टम को स्थापित सुरक्षा चिंताएँ (सुरक्षित कोडिंग, न्यूनतम अधिकार, आपूर्ति श्रृंखला सुरक्षा) और नए AI-विशिष्ट खतरों का सामना करना पड़ता है, जिनमें प्रॉम्प्ट इंजेक्शन, उपकरण विषाक्तता, सत्र हैकिंग, भ्रमित डिप्टी हमले, टोकन पासथ्रू कमजोरियां, और गतिशील क्षमता संशोधन शामिल हैं।

यह पाठ MCP कार्यान्वयन में सबसे महत्वपूर्ण सुरक्षा खतरों का अन्वेषण करता है—जिसमें प्रमाणीकरण, प्राधिकरण, अतिरेक अनुमति, अप्रत्यक्ष प्रॉम्प्ट इंजेक्शन, सत्र सुरक्षा, भ्रमित डिप्टी समस्याएँ, टोकन प्रबंधन, और आपूर्ति श्रृंखला कमजोरियाँ शामिल हैं। आप इन खतरों को कम करने के लिए कार्यक्षम नियंत्रण और सर्वोत्तम प्रथाएँ सीखेंगे, साथ ही Microsoft समाधानों जैसे प्रॉम्प्ट शील्ड्स, Azure कंटेंट सेफ़्टी, और GitHub एडवांस्ड सुरक्षा का उपयोग करके अपने MCP परिनियोजन को मजबूत करेंगे।

## सीखने के उद्देश्य

इस पाठ के अंत तक, आप सक्षम होंगे:

- **MCP-विशिष्ट खतरों की पहचान करें**: MCP सिस्टम के विशिष्ट सुरक्षा खतरों को पहचानें, जिनमें प्रॉम्प्ट इंजेक्शन, उपकरण विषाक्तता, अतिरेक अनुमति, सत्र हैकिंग, भ्रमित डिप्टी समस्याएँ, टोकन पासथ्रू कमजोरियाँ, और आपूर्ति श्रृंखला खतरें शामिल हैं
- **सुरक्षा नियंत्रण लागू करें**: मजबूत प्रमाणीकरण, न्यूनतम अधिकार पहुँच, सुरक्षित टोकन प्रबंधन, सत्र सुरक्षा नियंत्रण, और आपूर्ति श्रृंखला सत्यापन सहित प्रभावी रोकथाम क्रियान्वित करें
- **Microsoft सुरक्षा समाधान का लाभ उठाएँ**: MCP कार्यभार संरक्षण के लिए Microsoft प्रॉम्प्ट शील्ड्स, Azure कंटेंट सेफ़्टी, और GitHub एडवांस्ड सुरक्षा को समझें और लागू करें
- **उपकरण सुरक्षा मान्य करें**: उपकरण मेटाडेटा सत्यापन, गतिशील परिवर्तनों की निगरानी, और अप्रत्यक्ष प्रॉम्प्ट इंजेक्शन हमलों के खिलाफ सुरक्षा के महत्व को समझें
- **सर्वश्रेष्ठ प्रथाएं एकीकृत करें**: स्थापित सुरक्षा मूल सिद्धांतों (सुरक्षित कोडिंग, सर्वर हार्डनिंग, ज़ीरो ट्रस्ट) को MCP-विशिष्ट नियंत्रणों के साथ संयोजित करके व्यापक सुरक्षा सुनिश्चित करें

# MCP सुरक्षा वास्तुकला और नियंत्रण

आधुनिक MCP कार्यान्वयन परतदार सुरक्षा दृष्टिकोण की मांग करते हैं जो पारंपरिक सॉफ़्टवेयर सुरक्षा और AI-विशिष्ट खतरों दोनों को संबोधित करें। तेजी से विकसित हो रहे MCP विनिर्देश सुरक्षा नियंत्रणों को परिपक्व कर रहा है, जिससे उद्यम सुरक्षा वास्तुकला और स्थापित सर्वोत्तम प्रथाओं के साथ बेहतर समाकलन संभव हो रहा है।

[Microsoft Digital Defense Report](https://aka.ms/mddr) का शोध दर्शाता है कि **98% रिपोर्टेड उल्लंघनों को मजबूत सुरक्षा स्वच्छता द्वारा रोका जा सकता है**। सबसे प्रभावी संरक्षण रणनीति बुनियादी सुरक्षा प्रथाओं को MCP-विशिष्ट नियंत्रणों के साथ संयोजित करती है—प्रमाणित आधारभूत सुरक्षा उपाय कुल सुरक्षा जोखिम को कम करने में सबसे प्रभावशाली हैं।

## वर्तमान सुरक्षा परिदृश्य

> **नोट:** इस जानकारी में MCP सुरक्षा मानक **5 फरवरी 2026** तक के अनुसार दर्शाए गए हैं, जो **MCP विनिर्देश 2025-11-25** के अनुरूप हैं। MCP प्रोटोकॉल तेजी से विकसित हो रहा है, और भविष्य के कार्यान्वयन में नए प्रमाणीकरण पैटर्न और उन्नत नियंत्रण हो सकते हैं। नवीनतम मार्गदर्शन के लिए हमेशा वर्तमान [MCP विनिर्देश](https://spec.modelcontextprotocol.io/), [MCP GitHub रिपॉजिटरी](https://github.com/modelcontextprotocol), और [सुरक्षा सर्वोत्तम प्रथाएं दस्तावेज़](https://modelcontextprotocol.io/specification/2025-11-25/basic/security_best_practices) देखें।

> **आगे देखते हुए:** `2026-07-28` रिलीज़ उम्मीदवार प्राधिकरण को और मजबूत करता है — क्लाइंट को प्राधिकरण प्रतिक्रियाओं में `iss` पैरामीटर (RFC 9207) का सत्यापन करना होगा, डायनामिक क्लाइंट पंजीकरण के दौरान OpenID Connect `application_type` घोषित करना होगा, और पंजीकृत प्रमाणपत्रों को जारी करने वाले प्राधिकरण सर्वर से लिंक करना होगा। पूरी सूची के लिए देखें [MCP में क्या बदल रहा है: 2026-07-28 रिलीज़ उम्मीदवार](../01-CoreConcepts/mcp-2026-07-28-release-candidate.md)।

## 🏔️ MCP सुरक्षा शिखर सम्मेलन कार्यशाला (शेरपा)

**हैंड्स-ऑन सुरक्षा प्रशिक्षण** के लिए, हम अत्यधिक सिफारिश करते हैं कि आप **MCP सुरक्षा शिखर सम्मेलन कार्यशाला** (शेरपा) लें - एक व्यापक निर्देशित अभियान जो Microsoft Azure में MCP सर्वरों को सुरक्षित करने के लिए है।

### कार्यशाला अवलोकन

[MCP सुरक्षा शिखर सम्मेलन कार्यशाला](https://azure-samples.github.io/sherpa/) व्यावहारिक, लागू करने योग्य सुरक्षा प्रशिक्षण प्रदान करती है, जो एक सिद्ध "असुरक्षित → शोषण → सुधार → सत्यापन" कार्यप्रणाली के माध्यम से होती है। आप:

- **टूट-फूटकर सीखें**: जानबूझकर असुरक्षित सर्वरों का प्रयोग करके कमजोरियों का अनुभव करें
- **Azure-नेटिव सुरक्षा का उपयोग करें**: Azure Entra ID, Key Vault, API Management, और AI कंटेंट सेफ़्टी का लाभ उठाएं
- **डिफेंस-इन-डेप्थ का पालन करें**: शिविरों के माध्यम से व्यापक सुरक्षा परतें बनाएं
- **OWASP मानकों को लागू करें**: हर तकनीक [OWASP MCP Azure सुरक्षा गाइड](https://microsoft.github.io/mcp-azure-security-guide/) के अनुरूप है
- **उत्पादन कोड प्राप्त करें**: कार्यशील, परीक्षण किए गए कार्यान्वयन के साथ निकलें

### MCP प्रमाणीकरण का विकास

MCP विनिर्देश ने प्रमाणीकरण और प्राधिकरण के दृष्टिकोण में महत्वपूर्ण विकास किया है:

- **मूल दृष्टिकोण**: प्रारंभिक विनिर्देशों ने डेवलपर्स को कस्टम प्रमाणीकरण सर्वर लागू करने की आवश्यकता बताई, जहां MCP सर्वर OAuth 2.0 प्राधिकरण सर्वर के रूप में उपयोगकर्ता प्रमाणीकरण सीधे प्रबंधित करते थे
- **वर्तमान मानक (2025-11-25)**: अपडेटेड विनिर्देश MCP सर्वरों को बाहरी पहचान प्रदाता (जैसे Microsoft Entra ID) को प्रमाणीकरण सौंपने की अनुमति देता है, सुरक्षा स्थिति में सुधार करता है और कार्यान्वयन जटिलता को कम करता है
- **ट्रांसपोर्ट लेयर सुरक्षा**: स्थानीय (STDIO) और दूरस्थ (Streamable HTTP) कनेक्शनों दोनों के लिए सुरक्षित ट्रांसपोर्ट तंत्रों के लिए बेहतर समर्थन और उचित प्रमाणीकरण पैटर्न

## प्रमाणीकरण एवं प्राधिकरण सुरक्षा

### वर्तमान सुरक्षा चुनौतियाँ

आधुनिक MCP कार्यान्वयन कई प्रमाणीकरण और प्राधिकरण चुनौतियों का सामना करते हैं:

### खतरे और हमलावर वेक्टर

- **गलत कॉन्फ़िगर किए गए प्राधिकरण तर्क**: MCP सर्वरों में त्रुटिपूर्ण प्राधिकरण कार्यान्वयन संवेदनशील डेटा को उजागर कर सकता है और अधिग्रहण नियंत्रणों को गलत तरीके से लागू कर सकता है
- **OAuth टोकन समझौता**: स्थानीय MCP सर्वर टोकन चोरी हमलावरों को सर्वर का नकल करने और डाउनस्ट्रीम सेवाओं तक पहुँचने की अनुमति देती है
- **टोकन पासथ्रू कमजोरियाँ**: अनुचित टोकन हैंडलिंग सुरक्षा नियंत्रणों को दरकिनार करती है और जवाबदेही में दोष उत्पन्न करती है
- **अधिकृत अनुमतियाँ**: बहुत अधिक अधिकार वाले MCP सर्वर न्यूनतम अधिकार सिद्धांत का उल्लंघन करते हैं और हमले की सतह बढ़ाते हैं

#### टोकन पासथ्रू: एक गंभीर विरोधाभास

वर्तमान MCP प्राधिकरण विनिर्देश में **टोकन पासथ्रू स्पष्ट रूप से प्रतिबंधित है** क्योकि इसके गंभीर सुरक्षा निहितार्थ हैं:

##### सुरक्षा नियंत्रण परिपथ शुरूआत
- MCP सर्वर और डाउनस्ट्रीम API महत्वपूर्ण सुरक्षा नियंत्रण (रेट लिमिटिंग, अनुरोध सत्यापन, ट्रैफ़िक निगरानी) लागू करते हैं जो उचित टोकन सत्यापन पर निर्भर करते हैं
- सीधे क्लाइंट-से-API टोकन उपयोग इन अनिवार्य सुरक्षा उपायों को दरकिनार करता है, जिससे सुरक्षा वास्तुकला कमजोर होती है

##### जवाबदेही और ऑडिट चुनौतियाँ  
- MCP सर्वर क्लाइंट-आधारित टोकन का区分 नहीं कर सकते, जिससे ऑडिट ट्रेल टूट जाते हैं
- डाउनस्ट्रीम संसाधन सर्वर लॉग्स असली MCP सर्वर मध्यस्थों की बजाय गलत अनुरोध स्रोत दिखाते हैं
- घटना जांच और अनुपालन ऑडिटिंग बहुत कठिन हो जाती है

##### डेटा चोरी के जोखिम
- अप्रमाणित टोकन दावों से हमलावर चोरी किए गए टोकन के साथ MCP सर्वरों का उपयोग डेटा चोरी के लिए कर सकते हैं
- ट्रस्ट सीमा उल्लंघन अनधिकृत पहुँच पैटर्न की अनुमति देते हैं जो निर्धारित सुरक्षा नियंत्रण को दरकिनार करते हैं

##### बहु-सेवा हमला वेक्टर
- हमलावर टोकन कई सेवाओं द्वारा स्वीकार किए जाने पर जुड़े सिस्टम में पार्श्व आंदोलन कर सकते हैं
- टोकन के स्रोत सत्यापन में विफलता सेवाओं के बीच ट्रस्ट अनुमानों का उल्लंघन करती है

### सुरक्षा नियंत्रण एवं रोकथाम

**महत्वपूर्ण सुरक्षा आवश्यकताएँ:**

> **अनिवार्य**: MCP सर्वरों को **किसी भी ऐसे टोकन को स्वीकार नहीं करना चाहिए जो स्पष्ट रूप से MCP सर्वर के लिए जारी नहीं किए गए हों**

#### प्रमाणीकरण एवं प्राधिकरण नियंत्रण

- **कठोर प्राधिकरण समीक्षा**: MCP सर्वर प्राधिकरण तर्क के व्यापक ऑडिट करें ताकि केवल इच्छित उपयोगकर्ता और क्लाइंट संवेदनशील संसाधनों तक पहुँच सकें
  - **कार्यान्वयन मार्गदर्शिका**: [MCP सर्वरों के लिए Azure API प्रबंधन प्रमाणीकरण गेटवे](https://techcommunity.microsoft.com/blog/integrationsonazureblog/azure-api-management-your-auth-gateway-for-mcp-servers/4402690)
  - **पहचान समाकलन**: [MCP सर्वर प्रमाणीकरण के लिए Microsoft Entra ID का उपयोग](https://den.dev/blog/mcp-server-auth-entra-id-session/)

- **सुरक्षित टोकन प्रबंधन**: [Microsoft के टोकन सत्यापन और जीवनचक्र सर्वोत्तम अभ्यास](https://learn.microsoft.com/en-us/entra/identity-platform/access-tokens) लागू करें
  - टोकन दर्शक दावों की MCP सर्वर पहचान से मिलान करें
  - उचित टोकन घुमाव और समाप्ति नीतियाँ लागू करें
  - टोकन पुन:प्रयास हमलों और अनधिकृत उपयोग को रोकें

- **सुरक्षित टोकन भंडारण**: डेटा स्टोर और ट्रांज़िट दोनों में टोकन एन्क्रिप्टेड संग्रह
  - **सर्वश्रेष्ठ प्रथाएँ**: [सुरक्षित टोकन भंडारण और एन्क्रिप्शन निर्देश](https://youtu.be/uRdX37EcCwg?si=6fSChs1G4glwXRy2)

#### पहुँच नियंत्रण कार्यान्वयन

- **न्यूनतम अधिकार सिद्धांत**: MCP सर्वरों को केवल आवश्यक न्यूनतम अनुमतियाँ दें
  - नियमित अनुमति समीक्षा और अद्यतन से विशेषाधिकार वृद्धि रोकें
  - **Microsoft दस्तावेज़**: [सुरक्षित न्यूनतम अधिकार पहुँच](https://learn.microsoft.com/entra/identity-platform/secure-least-privileged-access)

- **भूमिका-आधारित पहुँच नियंत्रण (RBAC)**: सूक्ष्म भूमिका असाइनमेंट लागू करें
  - भूमिकाएँ विशिष्ट संसाधनों और क्रियाओं तक सीमित करें
  - व्यापक या अनावश्यक अनुमतियाँ जो हमले की सतह बढ़ाती हों, उनसे बचें

- **लगातार अनुमति निगरानी**: निरंतर पहुँच ऑडिटिंग और निगरानी लागू करें
  - अनुमत उपयोग पैटर्न को असामान्यताओं के लिए जांचें
  - अतिरेक या अप्रयुक्त अनुमतियों को तुरंत सुधारें

## AI-विशिष्ट सुरक्षा खतरें

### प्रॉम्प्ट इंजेक्शन और उपकरण हेरफेर हमले

आधुनिक MCP कार्यान्वयन उन्नत AI-विशिष्ट हमला वेक्टरों का सामना करते हैं जिन्हें पारंपरिक सुरक्षा उपाय पूरी तरह से संबोधित नहीं कर पाते:

#### **अप्रत्यक्ष प्रॉम्प्ट इंजेक्शन (Cross-Domain Prompt Injection)**

**अप्रत्यक्ष प्रॉम्प्ट इंजेक्शन** MCP-सक्षम AI सिस्टम में सबसे गंभीर कमजोरियों में से एक है। हमलावर बाहरी सामग्री—दस्तावेज़, वेब पेज, ईमेल, या डेटा स्रोतों में दुर्भावनापूर्ण निर्देश छुपा देते हैं जिन्हें AI सिस्टम वैध आदेश के रूप में संसाधित करता है।

**हमला परिदृश्य:**
- **दस्तावेज-आधारित इंजेक्शन**: संसाधित दस्तावेजों में छुपे दुर्भावनापूर्ण निर्देश जो आकस्मिक AI क्रियाएँ उत्पन्न करते हैं
- **वेब सामग्री शोषण**: संलग्न प्रॉम्प्ट वाले समझौता किए गए वेब पेज जो स्क्रैपिंग पर AI व्यवहार को प्रभावित करते हैं
- **ईमेल-आधारित हमले**: ईमेलों में छुपे प्रॉम्प्ट जो AI सहायकों को जानकारी लीक करने या अनधिकृत क्रियाएँ करने के लिए प्रेरित करते हैं
- **डेटा स्रोत संदूषण**: दूषित डाटाबेस या API जो AI सिस्टम को दूषित सामग्री प्रदान करते हैं

**वास्तविक प्रभाव**: ये हमले डेटा चोरी, गोपनीयता उल्लंघन, हानिकारक सामग्री उत्पादन, और उपयोगकर्ता इंटरैक्शन के मनिपुलेशन जैसी समस्याएँ कर सकते हैं। विस्तृत विश्लेषण के लिए देखें [MCP में प्रॉम्प्ट इंजेक्शन (Simon Willison)](https://simonwillison.net/2025/Apr/9/mcp-prompt-injection/)।

![प्रॉम्प्ट इंजेक्शन हमला चित्र](../../../translated_images/hi/prompt-injection.ed9fbfde297ca877.webp)

#### **उपकरण विषाक्तता हमले**

**उपकरण विषाक्तता** MCP उपकरण को परिभाषित करने वाले मेटाडेटा पर लक्षित होती है, यह समझकर कि LLM उपकरण विवरण और पैरामीटर की व्याख्या कैसे करते हैं ताकि निष्पादन निर्णय लें।

**हमला यांत्रिकी:**
- **मेटाडेटा हेरफेर**: हमलावर उपकरण विवरण, पैरामीटर परिभाषा, या उपयोग के उदाहरणों में दुर्भावनापूर्ण निर्देश इंजेक्ट करते हैं
- **अदृश्य निर्देश**: उपकरण मेटाडेटा में छुपे प्रॉम्प्ट जिन्हें AI मॉडल संसाधित करते हैं लेकिन मानव उपयोगकर्ताओं को दिखाई नहीं देते
- **गतिशील उपकरण संशोधन ("रग पुल्स")**: उपयोगकर्ताओं द्वारा स्वीकृत उपकरण बाद में अनजाने में दुर्भावनापूर्ण क्रियाएँ करने के लिए संशोधित किए जाते हैं
- **पैरामीटर इंजेक्शन**: उपकरण पैरामीटर स्कीमाओं में छुपी दुर्भावनापूर्ण सामग्री जो मॉडल व्यवहार को प्रभावित करती है


**होस्टेड सर्वर जोखिम**: रिमोट MCP सर्वर उन्नत जोखिम प्रस्तुत करते हैं क्योंकि टूल परिभाषाओं को प्रारंभिक उपयोगकर्ता स्वीकृति के बाद अपडेट किया जा सकता है, जिससे ऐसे परिदृश्यों का निर्माण होता है जहां पहले सुरक्षित टूल दुर्भावनापूर्ण हो जाते हैं। व्यापक विश्लेषण के लिए, देखें [Tool Poisoning Attacks (Invariant Labs)](https://invariantlabs.ai/blog/mcp-security-notification-tool-poisoning-attacks)।

![Tool Injection Attack Diagram](../../../translated_images/hi/tool-injection.3b0b4a6b24de6bef.webp)

#### **अतिरिक्त AI हमले के मार्ग**

- **क्रॉस-डोमेन प्रॉम्प्ट इंजेक्शन (XPIA)**: परिष्कृत हमले जो सुरक्षा नियंत्रणों को बायपास करने के लिए कई डोमेन की सामग्री का उपयोग करते हैं
- **डायनामिक क्षमता संशोधन**: टूल क्षमताओं में रियल-टाइम परिवर्तन जो प्रारंभिक सुरक्षा आकलनों से बच निकलते हैं
- **कंटेक्स्ट विंडो पॉयज़निंग**: बड़े कंटेक्स्ट विंडोज़ को नियंत्रित करने वाले हमले जो दुर्भावनापूर्ण निर्देशों को छिपाते हैं
- **मॉडल भ्रम हमले**: मॉडल की सीमाओं का फायदा उठाकर अप्रत्याशित या असुरक्षित व्यवहार उत्पन्न करना


### AI सुरक्षा जोखिम प्रभाव

**उच्च-प्रभाव परिणाम:**
- **डेटा चोरी**: संवेदनशील उद्यम या व्यक्तिगत डेटा की अनधिकृत पहुँच और चोरी
- **गोपनीयता उल्लंघन**: व्यक्तिगत पहचान योग्य जानकारी (PII) और गोपनीय व्यावसायिक डेटा का खुलासा  
- **सिस्टम हेरफेर**: महत्वपूर्ण प्रणालियों और वर्कफ़्लोज़ में अनियंत्रित परिवर्तन
- **प्रमाणीकरण टोकन चोरी**: प्रमाणीकरण टोकन और सेवा क्रेडेंशियल्स का समझौता
- **लैटरल मूवमेंट**: व्यापक नेटवर्क हमलों के लिए कम्प्रोमाइज्ड AI सिस्टम का उपयोग

## उन्नत MCP सुरक्षा खतरे

### सत्र अपहरण कमजोरियां

**सत्र अपहरण** एक महत्वपूर्ण हमला मार्ग है जहाँ असत्यापित पक्ष वैध सत्र पहचानकर्ताओं को प्राप्त करके क्लाइंट्स की नकल करते हैं और अनधिकृत क्रियाएँ करते हैं।

#### **हमले के परिदृश्य और जोखिम**

- **सत्र अपहरण प्रॉम्प्ट इंजेक्शन**: चुराए गए सत्र IDs वाले हमलावर सर्वरों में दुर्भावनापूर्ण घटनाएँ इंजेक्ट करते हैं, जो हानिकारक क्रियाएं या संवेदनशील डेटा तक पहुँच सकते हैं
- **प्रत्यक्ष नकल**: चोरी हुए सत्र IDs सीधे MCP सर्वर कॉल की अनुमति देते हैं जो प्रमाणीकरण को बायपास करते हैं और हमलावरों को वैध उपयोगकर्ता बनाते हैं
- **समझौता किए गए पुनः आरंभशील स्ट्रीम**: हमलावर अनुरोधों को समय से पहले समाप्त कर सकते हैं, जिससे वैध क्लाइंट संभावित दुर्भावनापूर्ण सामग्री के साथ पुनः आरंभ करते हैं

#### **सत्र प्रबंधन के लिए सुरक्षा नियंत्रण**

**महत्वपूर्ण आवश्यकताएँ:**
- **अधिकार प्रमाणन सत्यापन**: MCP सर्वर सभी इनबाउंड अनुरोधों का सत्यापन करें और प्रमाणीकरण के लिए सत्र पर भरोसा न करें
- **सुरक्षित सत्र जनरेशन**: क्रिप्टोग्राफ़िक सुरक्षा के साथ यादृच्छिक रूप से सत्र IDs बनाएं
- **उपयोगकर्ता-विशिष्ट बाइंडिंग**: सत्र IDs को उपयोगकर्ता-विशिष्ट जानकारी से बाँधें जैसे `<user_id>:<session_id>` ताकि क्रॉस-यूज़र सत्र दुरुपयोग रोका जा सके
- **सत्र जीवनचक्र प्रबंधन**: उपयुक्त समाप्ति, घुमाव, और अमान्यता लागू करें ताकि जोखिम सीमित हो
- **ट्रांसपोर्ट सुरक्षा**: सभी संचार के लिए HTTPS अनिवार्य करें ताकि सत्र ID अवरोधन से बचा जा सके

### कंफ्यूज्ड डेप्टी समस्या

**कंफ्यूज्ड डेप्टी समस्या** तब होती है जब MCP सर्वर क्लाइंट्स और थर्ड-पार्टी सेवाओं के बीच प्रमाणीकरण प्रॉक्सी के रूप में कार्य करते हैं, जिससे स्थिर क्लाइंट ID के दुरुपयोग से अधिकार बायपास के अवसर बनते हैं।

#### **हमले की प्रक्रिया और जोखिम**

- **कुकि-आधारित सहमति बायपास**: पूर्व उपयोगकर्ता प्रमाणीकरण सहमति कुकि बनाता है जिसे हमलावर दुर्भावनापूर्ण प्राधिकरण अनुरोधों के साथ हानि पहुंचाते हैं, जिसमें निर्मित रिडायरेक्ट URI शामिल हैं
- **प्राधिकरण कोड चोरी**: मौजूदा सहमति कुकि प्राधिकरण सर्वर को सहमति स्क्रीन छोड़ने और हमलावर नियंत्रित एंडपॉइंट पर कोड भेजने के लिए प्रेरित कर सकते हैं  
- **अनधिकृत API पहुँच**: चोरी हुए प्राधिकरण कोड बिना स्पष्ट अनुमोदन के टोकन एक्सचेंज और उपयोगकर्ता नकल की अनुमति देते हैं

#### **निवारण रणनीतियाँ**

**अनिवार्य नियंत्रण:**
- **स्पष्ट सहमति आवश्यकताएं**: स्थिर क्लाइंट ID वाले MCP प्रॉक्सी सर्वर प्रत्येक गतिशील पंजीकृत क्लाइंट के लिए उपयोगकर्ता सहमति प्राप्त करें
- **OAuth 2.1 सुरक्षा पालन**: सभी प्राधिकरण अनुरोधों के लिए PKCE सहित वर्तमान OAuth सुरक्षा सर्वोत्तम प्रथाओं का पालन करें
- **क्लाइंट सत्यापन सख्ती से करें**: रिडायरेक्ट URI और क्लाइंट पहचानकर्ताओं का कठोर सत्यापन लागू करें ताकि दुरुपयोग से बचा जा सके

### टोकन पासथ्रू कमजोरियां  

**टोकन पासथ्रू** एक स्पष्ट एंटी-पैटर्न है जहाँ MCP सर्वर बिना उचित सत्यापन के क्लाइंट टोकन स्वीकार करते हैं और उन्हें डाउनस्ट्रीम API को अग्रेषित करते हैं, जो MCP प्राधिकरण विनिर्देशों का उल्लंघन है।

#### **सुरक्षा प्रभाव**

- **नियंत्रण परिहार**: सीधे क्लाइंट-से-API टोकन उपयोग महत्वपूर्ण दर सीमाएं, सत्यापन, और निगरानी नियंत्रणों को बायपास करता है
- **ऑडिट ट्रेल भ्रष्टाचार**: अपस्ट्रीम जारी किए गए टोकन के कारण क्लाइंट पहचान संभव नहीं, जिससे घटना जांच बाधित होती है
- **प्रॉक्सी-आधारित डेटा चोरी**: अप्रमाणित टोकन हमलावरों को अनधिकृत डेटा एक्सेस के लिए सर्वरों का उपयोग प्रॉक्सी के रूप में करने देते हैं
- **ट्रस्ट सीमा उल्लंघन**: टोकन के स्रोत सत्यापन न होने पर डाउनस्ट्रीम सेवाओं की ट्रस्ट मान्यताएँ उल्लंघित हो सकती हैं
- **मल्टी-सर्विस हमला विस्तार**: कई सेवाओं में स्वीकार किए गए समझौता किए टोकन लैटरल मूवमेंट को सक्षम करते हैं

#### **आवश्यक सुरक्षा नियंत्रण**

**असम्मत आवश्यकताएँ:**
- **टोकन सत्यापन**: MCP सर्वर केवल टोकन स्वीकार करें जो स्पष्ट रूप से MCP सर्वर के लिए जारी किए गए हों
- **दर्शक सत्यापन**: टोकन दर्शक दावों का हमेशा MCP सर्वर की पहचान से मिलान करें
- **सही टोकन जीवनचक्र**: सुरक्षित घुमाव प्रथाओं के साथ अल्पकालिक एक्सेस टोकन लागू करें


## AI सिस्टम के लिए सप्लाई चेन सुरक्षा

सप्लाई चेन सुरक्षा पारंपरिक सॉफ़्टवेयर निर्भरताओं से परे जाकर पूरे AI इकोसिस्टम को शामिल करती है। आधुनिक MCP कार्यान्वयन को सभी AI-संबंधित घटकों का कड़े निरीक्षण और सत्यापन करना चाहिए क्योंकि प्रत्येक संभावित कमजोरियाँ लाता है जो प्रणाली की अखंडता को खतरे में डाल सकता है।

### विस्तृत AI सप्लाई चेन घटक

**पारंपरिक सॉफ्टवेयर निर्भरताएँ:**
- ओपन-सोर्स पुस्तकालय और फ्रेमवर्क
- कंटेनर इमेज और बेस सिस्टम  
- विकास उपकरण और बिल्ड पाइपलाइंस
- बुनियादी ढांचा घटक और सेवाएं

**AI-विशिष्ट सप्लाई चेन तत्व:**
- **फाउंडेशन मॉडल**: विभिन्न प्रदाताओं के प्री-ट्रेन मॉडल जिनके लिए प्रामाणिकता सत्यापन आवश्यक है
- **एम्बेडिंग सेवाएं**: बाहरी वेक्टराइजेशन और सेमांटिक सर्च सेवाएं
- **संदर्भ प्रदाता**: डेटा स्रोत, ज्ञान आधार, और दस्तावेज़ भंडार  
- **थर्ड-पार्टी API**: बाहरी AI सेवाएं, ML पाइपलाइंस, और डेटा प्रोसेसिंग एंडपॉइंट्स
- **मॉडल आर्टिफैक्ट्स**: वेट्स, कॉन्फ़िगरेशन, और फाइन-ट्यून मॉडल वेरिएंट्स
- **ट्रेनिंग डेटा स्रोत**: मॉडल प्रशिक्षण और फाइन-ट्यूनिंग के लिए उपयोग किए गए डेटासेट्स

### व्यापक सप्लाई चेन सुरक्षा रणनीति

#### **घटक सत्यापन और विश्वास**
- **प्रामाणिकता सत्यापन**: सभी AI घटकों के स्रोत, लाइसेंस, और अखंडता का एकीकरण से पहले सत्यापन करें
- **सुरक्षा आकलन**: मॉडलों, डेटा स्रोतों, और AI सेवाओं के लिए कमजोरियों के स्कैन और सुरक्षा समीक्षाएँ करें
- **प्रतिष्ठा विश्लेषण**: AI सेवा प्रदाताओं के सुरक्षा रिकॉर्ड और प्रथाओं का मूल्यांकन करें
- **अनुपालन सत्यापन**: सुनिश्चित करें कि सभी घटक संगठनात्मक सुरक्षा और नियामक आवश्यकताओं को पूरा करते हैं

#### **सुरक्षित डिप्लॉयमेंट पाइपलाइंस**  
- **स्वचालित CI/CD सुरक्षा**: स्वचालित डिप्लॉयमेंट पाइपलाइंस में सुरक्षा स्कैनिंग को शामिल करें
- **आर्टिफैक्ट अखंडता**: तैनात सभी आर्टिफैक्ट्स (कोड, मॉडल, कॉन्फ़िगरेशन) के लिए क्रिप्टोग्राफिक सत्यापन लागू करें
- **चरणबद्ध डिप्लॉयमेंट**: प्रत्येक चरण में सुरक्षा मान्यता के साथ प्रगतिशील परिनियोजन रणनीतियाँ अपनाएं
- **विश्वसनीय आर्टिफैक्ट रिपोजिटरीज़**: केवल प्रमाणित और सुरक्षित रिपोजिटरीज़ से ही डिप्लॉय करें

#### **लगातार निगरानी और प्रतिक्रिया**
- **निर्भरता स्कैनिंग**: सभी सॉफ़्टवेयर और AI घटक निर्भरताओं के लिए लगातार कमजोरियों की निगरानी
- **मॉडल निगरानी**: मॉडल व्यवहार, प्रदर्शन में बदलाव, और सुरक्षा असामान्यताओं का निरंतर मूल्यांकन
- **सेवा स्वास्थ्य ट्रैकिंग**: बाहरी AI सेवाओं की उपलब्धता, सुरक्षा घटनाओं, और नीति परिवर्तनों की निगरानी
- **खतरा खुफिया एकीकरण**: AI और ML सुरक्षा खतरों के लिए खतरा फीड एकीकृत करें

#### **पहुँच नियंत्रण और न्यूनतम विशेषाधिकार**
- **घटक-स्तर अनुमतियाँ**: व्यवसायिक आवश्यकता के आधार पर मॉडलों, डेटा, और सेवाओं की पहुँच सीमित करें
- **सेवा खाता प्रबंधन**: न्यूनतम आवश्यक अनुमतियों के साथ समर्पित सेवा खाते लागू करें
- **नेटवर्क सेगमेंटेशन**: AI घटकों को अलग करें और सेवाओं के बीच नेटवर्क पहुँच सीमित करें
- **API गेटवे नियंत्रण**: बाहरी AI सेवाओं की पहुँच को नियंत्रित करने और निगरानी हेतु केंद्रीकृत API गेटवे का उपयोग करें

#### **घटना प्रतिक्रिया और पुनर्प्राप्ति**
- **त्वरित प्रतिक्रिया प्रक्रियाएं**: समझौता किए गए AI घटकों को पैच करने या बदलने के लिए स्थापित प्रक्रियाएं
- **प्रमाणपत्र घुमाव**: रहस्यों, API कुंजियों, और सेवा क्रेडेंशियल्स को घुमाने के लिए स्वचालित प्रणालियाँ
- **रोलबैक क्षमताएं**: AI घटकों के पहले से ज्ञात-अच्छे संस्करणों में शीघ्र पुनःवापसी की क्षमता
- **सप्लाई चेन उल्लंघन पुनर्प्राप्ति**: अपस्ट्रीम AI सेवा समझौतों के लिए विशिष्ट प्रतिक्रिया प्रक्रियाएं

### Microsoft सुरक्षा उपकरण और एकीकरण

**GitHub Advanced Security** व्यापक सप्लाई चेन सुरक्षा प्रदान करता है जिसमें शामिल हैं:
- **सीक्रेट स्कैनिंग**: रिपोजिटरीज़ में क्रेडेंशियल्स, API कुंजियों, और टोकन का स्वचालित पता लगाना
- **निर्भरता स्कैनिंग**: ओपन-सोर्स निर्भरताओं और पुस्तकालयों के लिए कमजोरियों का आकलन
- **CodeQL विश्लेषण**: सुरक्षा कमजोरियों और कोडिंग मुद्दों के लिए स्थैतिक कोड विश्लेषण
- **सप्लाई चेन इनसाइट्स**: निर्भरता स्वास्थ्य और सुरक्षा स्थिति की दृश्यता

**Azure DevOps & Azure Repos एकीकरण:**
- Microsoft विकास प्लेटफॉर्म्स पर निर्बाध सुरक्षा स्कैनिंग एकीकरण
- AI वर्कलोड के लिए Azure पाइपलाइंस में स्वचालित सुरक्षा जांच
- सुरक्षित AI घटक परिनियोजन के लिए नीति प्रवर्तन

**Microsoft आंतरिक प्रथाएं:**
Microsoft सभी उत्पादों में व्यापक सप्लाई चेन सुरक्षा प्रथाएं लागू करता है। प्रमाणित दृष्टिकोणों के बारे में जानें [The Journey to Secure the Software Supply Chain at Microsoft](https://devblogs.microsoft.com/engineering-at-microsoft/the-journey-to-secure-the-software-supply-chain-at-microsoft/)।


## फाउंडेशन सुरक्षा सर्वोत्तम प्रथाएं

MCP कार्यान्वयन आपकी संस्था के मौजूदा सुरक्षा स्थिति को अपनाते और उस पर निर्माण करते हैं। मौलिक सुरक्षा प्रथाओं को मजबूत करने से AI सिस्टम और MCP डिप्लॉयमेंट की कुल सुरक्षा काफी बढ़ती है।

### मूल सुरक्षा आधारभूत सिद्धांत

#### **सुरक्षित विकास प्रथाएं**
- **OWASP अनुपालन**: [OWASP टॉप 10](https://owasp.org/www-project-top-ten/) वेब अनुप्रयोग कमजोरियों से सुरक्षा करें
- **AI-विशिष्ट सुरक्षा**: [OWASP टॉप 10 फॉर LLMs](https://genai.owasp.org/download/43299/?tmstv=1731900559) के लिए नियंत्रण लागू करें
- **सुरक्षित सीक्रेट प्रबंधन**: टोकन, API कुंजी, और संवेदनशील कॉन्फ़िगरेशन डेटा के लिए समर्पित वॉल्ट्स का उपयोग करें
- **एंड-टू-एंड एन्क्रिप्शन**: सभी एप्लिकेशन घटकों और डेटा फ्लो में सुरक्षित संचार लागू करें
- **इनपुट सत्यापन**: सभी उपयोगकर्ता इनपुट, API पैरामीटर, और डेटा स्रोतों का कठोर सत्यापन

#### **बुनियादी ढांचा कड़ा करना**
- **मल्टी-फैक्टर प्रमाणीकरण**: सभी प्रशासनिक और सेवा खातों के लिए अनिवार्य MFA
- **पैच प्रबंधन**: ऑपरेटिंग सिस्टम, फ्रेमवर्क, और निर्भरताओं के लिए स्वचालित, समय पर पैचिंग  
- **पहचान प्रदाता एकीकरण**: एंटरप्राइज़ पहचान प्रदाताओं (Microsoft Entra ID, Active Directory) के माध्यम से केंद्रीकृत पहचान प्रबंधन
- **नेटवर्क सेगमेंटेशन**: MCP घटकों का तार्किक पृथक्करण ताकि लैटरल मूवमेंट सीमित हो
- **न्यूनतम विशेषाधिकार सिद्धांत**: सभी सिस्टम घटकों और खातों के लिए न्यूनतम आवश्यक अनुमतियाँ

#### **सुरक्षा निगरानी और पहचान**
- **व्यापक लॉगिंग**: AI एप्लिकेशन गतिविधियों का विवरण, जिसमें MCP क्लाइंट-सर्वर इंटरैक्शन शामिल हैं
- **SIEM एकीकरण**: असामान्यताओं की पहचान के लिए केंद्रीकृत सुरक्षा जानकारी और घटना प्रबंधन
- **व्यवहारात्मक विश्लेषण**: प्रणाली और उपयोगकर्ता व्यवहार में असामान्य पैटर्न पहचानने के लिए AI-समर्थित निगरानी
- **खतरा खुफिया**: बाहरी खतरा फ़ीड और समझौता संकेतक (IOCs) का एकीकरण
- **घटना प्रतिक्रिया**: सुरक्षा घटना की पहचान, प्रतिक्रिया, और पुनर्प्राप्ति के लिए सुव्यवस्थित प्रक्रियाएं

#### **ज़ीरो ट्रस्ट आर्किटेक्चर**
- **कभी भरोसा न करें, हमेशा सत्यापित करें**: उपयोगकर्ताओं, उपकरणों, और नेटवर्क कनेक्शनों का सतत सत्यापन
- **माइक्रो-सेगमेंटेशन**: व्यक्तिगत वर्कलोड और सेवाओं को अलग करने वाले सूक्ष्म नेटवर्क नियंत्रण
- **पहचान-केंद्रित सुरक्षा**: नेटवर्क स्थान के बजाय सत्यापित पहचान पर आधारित सुरक्षा नीतियां
- **सतत जोखिम मूल्यांकन**: वर्तमान संदर्भ और व्यवहार के आधार पर गतिशील सुरक्षा स्थिति मूल्यांकन
- **सशर्त पहुँच**: जोखिम कारकों, स्थान, और डिवाइस विश्वास के आधार पर अनुकूलित पहुँच नियंत्रण

### उद्यम एकीकरण पैटर्न

#### **Microsoft सुरक्षा पारिस्थितिकी तंत्र एकीकरण**
- **Microsoft Defender for Cloud**: व्यापक क्लाउड सुरक्षा स्थिति प्रबंधन
- **Azure Sentinel**: AI कार्यभार सुरक्षा के लिए क्लाउड-नेटिव SIEM और SOAR क्षमताएं
- **Microsoft Entra ID**: सशर्त पहुँच नीतियों के साथ एंटरप्राइज़ पहचान और पहुँच प्रबंधन
- **Azure Key Vault**: हार्डवेयर सुरक्षा मॉड्यूल (HSM) समर्थन के साथ केंद्रीकृत रहस्य प्रबंधन
- **Microsoft Purview**: AI डेटा स्रोतों और कार्यप्रवाहों के लिए डेटा शासन और अनुपालन

#### **अनुपालन और शासन**
- **नियामक समन्वय**: सुनिश्चित करें कि MCP कार्यान्वयन उद्योग-विशिष्ट अनुपालन आवश्यकताओं (GDPR, HIPAA, SOC 2) को पूरा करते हैं

- **डेटा वर्गीकरण**: AI प्रणालियों द्वारा संसाधित संवेदनशील डेटा का उचित वर्गीकरण और प्रबंधन
- **ऑडिट ट्रेल्स**: नियामक अनुपालन और फोरेंसिक जांच के लिए व्यापक लॉगिंग
- **प्राइवेसी नियंत्रण**: AI सिस्टम वास्तुकला में प्राइवेसी-बाय-डिज़ाइन सिद्धांतों का कार्यान्वयन
- **परिवर्तन प्रबंधन**: AI सिस्टम परिवर्तनों की सुरक्षा समीक्षाओं के लिए औपचारिक प्रक्रियाएं

ये बुनियादी प्रथाएँ एक मजबूत सुरक्षा बेसलाइन बनाती हैं जो MCP-विशिष्ट सुरक्षा नियंत्रणों की प्रभावशीलता को बढ़ाती हैं और AI-चालित अनुप्रयोगों के लिए व्यापक सुरक्षा प्रदान करती हैं।

## प्रमुख सुरक्षा निष्कर्ष

- **स्तरीय सुरक्षा दृष्टिकोण**: व्यापक सुरक्षा के लिए बुनियादी सुरक्षा प्रथाओं (सुरक्षित कोडिंग, न्यूनतम विशेषाधिकार, आपूर्ति श्रृंखला सत्यापन, निरंतर निगरानी) को AI-विशिष्ट नियंत्रणों के साथ संयोजित करें

- **AI-विशिष्ट खतरे का परिदृश्य**: MCP प्रणालियों को अनूठे जोखिमों का सामना करना पड़ता है जैसे कि प्रॉम्प्ट इंजेक्शन, टूल पॉइज़निंग, सेशन हाईजैकिंग, भ्रमित डेप्यूटी समस्याएं, टोकन पासथ्रू कमजोरियां, और अत्यधिक अनुमतियाँ जिन्हें विशेष मुआवज़ों की आवश्यकता होती है

- **प्रमाणीकरण और प्राधिकरण में उत्कृष्टता**: बाहरी पहचान प्रदाताओं (Microsoft Entra ID) का उपयोग करके मजबूत प्रमाणीकरण लागू करें, उचित टोकन सत्यापन लागू करें, और कभी भी ऐसे टोकन स्वीकार न करें जो स्पष्ट रूप से आपके MCP सर्वर के लिए जारी नहीं किए गए हों

- **AI हमलों की रोकथाम**: अप्रत्यक्ष प्रॉम्प्ट इंजेक्शन और टूल पॉइज़निंग हमलों से बचाव के लिए Microsoft Prompt Shields और Azure Content Safety का उपयोग करें, साथ ही टूल मेटाडेटा का सत्यापन करें और गतिशील परिवर्तनों की निगरानी करें

- **सेशन और ट्रांसपोर्ट सुरक्षा**: क्रिप्टोग्राफ़िक सुरक्षित, गैर-नियतात्मक सेशन ID जो उपयोगकर्ता पहचानों से बंधी हों, उचित सेशन जीवनचक्र प्रबंधन लागू करें, और प्रमाणीकरण के लिए सेशन का उपयोग कभी न करें

- **OAuth सुरक्षा सर्वोत्तम प्रथाएँ**: डायनामिक रूप से पंजीकृत क्लाइंट्स के लिए स्पष्ट उपयोगकर्ता सहमति के माध्यम से भ्रमित डेप्यूटी हमलों को रोकें, PKCE के साथ उचित OAuth 2.1 कार्यान्वयन करें, और कड़ी रीडायरेक्ट URI सत्यापन लागू करें  

- **टोकन सुरक्षा सिद्धांत**: टोकन पासथ्रू एंटी-पैटर्न से बचें, टोकन ऑडियंस दावे सत्यापित करें, सुरक्षित रोटेशन के साथ अल्पकालिक टोकन लागू करें, और स्पष्ट विश्वास सीमाएँ बनाए रखें

- **व्यापक आपूर्ति श्रृंखला सुरक्षा**: सभी AI पारिस्थितिकी तंत्र घटकों (मॉडल, एम्बेडिंग, संदर्भ प्रदाता, बाहरी API) को पारंपरिक सॉफ़्टवेयर निर्भरता की तरह ही कड़ी सुरक्षा के साथ मानें

- **निरंतर विकास**: तेजी से विकसित हो रहे MCP विनिर्देशों के साथ अद्यतन रहें, सुरक्षा समुदाय मानकों में योगदान दें, और जैसे-जैसे प्रोटोकॉल परिपक्व होता है अनुकूलनशील सुरक्षा रुख बनाए रखें

- **Microsoft सुरक्षा एकीकरण**: MCP परिनियोजन सुरक्षा के लिए Microsoft के व्यापक सुरक्षा इकोसिस्टम (Prompt Shields, Azure Content Safety, GitHub Advanced Security, Entra ID) का लाभ उठाएं
