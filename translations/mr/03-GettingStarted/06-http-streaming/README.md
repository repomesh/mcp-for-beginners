# मॉडेल कॉन्टेक्स्ट प्रोटोकॉल (MCP) सह HTTPS स्ट्रीमिंग

हे अध्याय मॉडेल कॉन्टेक्स्ट प्रोटोकॉल (MCP) वापरून HTTPS सह सुरक्षित, स्केलेबल आणि रिअल-टाइम स्ट्रीमिंग कसे राबवायचे याबाबत सविस्तर मार्गदर्शन प्रदान करतो. यात स्ट्रीमिंगसाठी प्रोत्साहन, उपलब्ध ट्रान्सपोर्ट मेकॅनिझम, MCP मध्ये स्ट्रीम करू शकणारे HTTP कसे लागू करायचे, सुरक्षा उत्तम प्रॅक्टिसेस, SSE मधून स्थलांतर आणि तुमची स्वतःची स्ट्रीमिंग MCP ॲप्लिकेशन्स तयार करण्यासाठी व्यावहारिक मार्गदर्शन यांचा समावेश आहे.

> **पुढे पाहत:** ही धडा **MCP स्पेसिफिकेशन 2025-11-25** अंतर्गत स्ट्रीम करू शकणारे HTTP वर्णन करतो, जिथे `initialize` दरम्यान एक सत्र तयार केले जाते आणि `Mcp-Session-Id` हेडरने पिन केले जाते. `2026-07-28` चा रिलीझ कॅंडिडेट हातमिळवणी आणि सत्र क्रमांक पूर्णपणे काढून टाकतो, ज्यामुळे प्रत्येक विनंती स्वयंपूर्ण होते आणि कोणत्याही सर्व्हर उदाहरणाकडे राउट केली जाऊ शकते, स्टिकी सत्रांशिवाय. तपशीलांसाठी [MCP मध्ये काय बदलत आहे: 2026-07-28 रिलीझ कॅंडिडेट](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) पहा.

## MCP मधील ट्रान्सपोर्ट मेकॅनिझम आणि स्ट्रीमिंग

हा विभाग MCP मध्ये उपलब्ध वेगवेगळ्या ट्रान्सपोर्ट मेकॅनिझमचे अन्वेषण करतो आणि क्लायंट आणि सर्व्हरमधील वास्तविक-वेळ संवादासाठी स्ट्रीमिंग क्षमता सक्षम करण्यामध्ये त्यांची भूमिका पाहतो.

### ट्रान्सपोर्ट मेकॅनिझम म्हणजे काय?

ट्रान्सपोर्ट मेकॅनिझम म्हणजे क्लायंट आणि सर्व्हर दरम्यान डेटा कसा देवाणघेवाण होतो हे परिभाषित करते. MCP विविध वातावरणे आणि गरजांनुसार खालील प्रकारचे ट्रान्सपोर्ट सपोर्ट करतो:

- **stdio**: स्थानिक आणि CLI-आधारित साधनांसाठी योग्य असलेले स्टँडर्ड इनपुट/आउटपुट. सोपे पण वेब किंवा क्लाउडकडे योग्य नाही.
- **SSE (सर्व्हर-सेंट इव्हेंट्स)**: सर्व्हर्सना HTTP वर क्लायंटवर रियल-टाइम अपडेट पाठविण्याची परवानगी देते. वेब यूआयसाठी चांगले, पण स्केलेबिलिटी आणि लवचीकतेसाठी मर्यादित. MCP स्पेसिफिकेशन 2025-06-18 नुसार स्वतंत्र SSE ट्रान्सपोर्ट बंद करण्यात आला आहे आणि त्याऐवजी "स्ट्रीम करू शकणारे HTTP" ट्रान्सपोर्ट आले आहे.
- **स्ट्रीम करू शकणारे HTTP**: आधुनिक HTTP-आधारित स्ट्रीमिंग ट्रान्सपोर्ट, सूचना व उत्तम स्केलेबिलिटीसाठी. बहुतेक उत्पादन आणि क्लाउड परिस्थितीसाठी शिफारस केली जाते.

### तुलना सारणी

खालील तुलना सारणीत या ट्रान्सपोर्ट मेकॅनिझममधील फरक समजून घ्या:

| ट्रान्सपोर्ट         | रिअल-टाइम अपडेट्स | स्ट्रीमिंग | स्केलेबिलिटी | उपयोग प्रकरण                |
|-------------------|------------------|-----------|-------------|-------------------------|
| stdio             | नाही               | नाही        | कमी         | स्थानिक CLI साधने         |
| SSE               | होय               | होय       | मध्यम      | वेब, रिअल-टाइम अपडेट्स  |
| स्ट्रीम करू शकणारे HTTP   | होय               | होय       | उच्च        | क्लाउड, मल्टी-क्लायंट     |

> **टीप:** योग्य ट्रान्सपोर्ट निवडणे कार्यक्षमता, स्केलेबिलिटी आणि वापरकर्ता अनुभवावर परिणाम करते. **स्ट्रीम करू शकणारे HTTP** आधुनिक, स्केलेबल आणि क्लाउड-तयार ऍप्लिकेशन्ससाठी शिफारसीय आहे.

मागील अध्यायांमध्ये आपण पाहिलेल्या stdio आणि SSE या ट्रान्सपोर्टला नोंद घ्या आणि या अध्यायात स्ट्रीम करू शकणारे HTTP हे ट्रान्सपोर्ट कव्हर केले आहे.

## स्ट्रीमिंग: कल्पना आणि प्रोत्साहन

स्ट्रीमिंगच्या मूलभूत संकल्पना आणि प्रोत्साहन समजणे प्रभावी रिअल-टाइम संवाद प्रणाली राबवण्यासाठी आवश्यक आहे.

**स्ट्रीमिंग** ही नेटवर्क प्रोग्रामिंगमधील एक तंत्र आहे जी संपूर्ण प्रतिसाद तयार होण्याची वाट न पाहता डेटा लहान, हाताळण्याजोग्या तुकड्यांमध्ये किंवा घटनांच्या मालिकेमध्ये पाठविण्यास आणि प्राप्त करण्यास अनुमती देते. हे विशेषतः उपयुक्त आहे:

- मोठ्या फाइल्स किंवा डेटासेटसाठी.
- रिअल-टाइम अपडेट्ससाठी (उदा., चॅट, प्रोग्रेस बार).
- लांबणाऱ्या गणनांसाठी जिथे वापरकर्त्याला सतत माहिती देणे आवश्यक आहे.

स्ट्रीमिंगबद्दल उच्चस्तरीय माहिती खालीलप्रमाणे:

- डेटा प्रगतिशीलपणे वितरित केला जातो, सर्व एकाच वेळी नाही.
- क्लायंट डेटा येताच प्रक्रिया करू शकतो.
- संभाव्य विलंब कमी करतो आणि वापरकर्ता अनुभव सुधारतो.

### स्ट्रीमिंग का वापरावे?

स्ट्रीमिंग वापरण्याची कारणे पुढीलप्रमाणे आहेत:

- वापरकर्त्यांना लगेच प्रतिसाद मिळतो, केवळ शेवटी नव्हे.
- रिअल-टाइम अनुप्रयोग आणि प्रतिसाद देणारी UI सक्षम करतो.
- नेटवर्क आणि संगणकीय संसाधनांचा अधिक कार्यक्षम वापर.

### सोपे उदाहरण: HTTP स्ट्रीमिंग सर्व्हर आणि क्लायंट

खाली स्ट्रीमिंग कसे राबवू शकतो याचे एक सोपे उदाहरण आहे:

#### पायथन

**सर्व्हर (पायथन, FastAPI आणि StreamingResponse वापरून):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**क्लायंट (पायथन, requests वापरून):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

हा उदाहरण दाखवते कसा सर्व्हर एकाच वेळी सर्व संदेशांच्या प्रतीक्षेत न राहता क्लायंटला संदेशांची मालिका पाठवतो.

**हे कसे कार्य करते:**

- सर्व्हर प्रत्येक संदेश तयार झाल्यावर त्याला यील्ड करतो.
- क्लायंट प्रत्येक येत असलेल्या तुकड्याला प्राप्त करून प्रिंट करतो.

**आवश्यकता:**

- सर्व्हरला स्ट्रीमिंग प्रतिसाद वापरावा लागतो (उदाहरणार्थ, FastAPI मधील `StreamingResponse`).
- क्लायंटला प्रतिसाद स्ट्रीम म्हणून प्रक्रिया करणे आवश्यक आहे (`stream=True` requests मध्ये).
- Content-Type सहसा `text/event-stream` किंवा `application/octet-stream` असतो.

#### जावा

**सर्व्हर (जावा, Spring Boot आणि Server-Sent Events वापरून):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**क्लायंट (जावा, Spring WebFlux WebClient वापरून):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**जावा अंमलबजावणी नोट्स:**

- स्ट्रीमिंगसाठी Spring Boot च्या reactive stack सह `Flux` वापरतो
- `ServerSentEvent` संरचित इव्हेंट स्ट्रीमिंगसाठी इव्हेंट प्रकारांसह पुरवतो
- `WebClient` च्या `bodyToFlux()` द्वारे प्रतिक्रियात्मक स्ट्रीमिंग खप समर्थन करते
- `delayElements()` इव्हेंट्स दरम्यान प्रक्रिया करण्याचा कालावधी सादर करतो
- इव्हेंट्सना चांगल्या क्लायंट हाताळणीसाठी प्रकार (उदा. `info`, `result`) असू शकतात

### तुलना: क्लासिक स्ट्रीमिंग vs MCP स्ट्रीमिंग

"क्लासिक" पद्धतीने स्ट्रीमिंग कसे कार्य करते आणि MCP मध्ये कसे कार्य करते यातील फरक अशा प्रकारे दिसू शकतो:

| वैशिष्ट्य                 | क्लासिक HTTP स्ट्रीमिंग          | MCP स्ट्रीमिंग (सूचना)         |
|--------------------------|---------------------------------|--------------------------------|
| मुख्य प्रतिसाद           | तुकड्यांमध्ये                   | एकदाच, शेवटी                  |
| प्रगती अपडेट्स          | डेटा तुकड्यांप्रमाणे पाठवले जातात | सूचना स्वरूपात पाठवले जातात     |
| क्लायंट गरजा           | स्ट्रीमला प्रक्रिया करणे आवश्यक  | मेसेज हँडलरची अंमलबजावणी करणे आवश्यक |
| वापर प्रकरण             | मोठ्या फाइल्स, AI टोकन स्ट्रीम्स | प्रगती, लॉग्ज, रिअल-टाइम प्रतिसाद |

### लक्षवेधी फरक

शिवाय, काही महत्त्वाचे फरकः

- **संवाद नमुना:**
  - क्लासिक HTTP स्ट्रीमिंग: साधी chunked ट्रान्सफर एन्कोडिंग वापरून डेटा पाठवतो
  - MCP स्ट्रीमिंग: JSON-RPC प्रोटोकॉलसह संरचित सूचना प्रणाली वापरतो

- **मेसेज स्वरूप:**
  - क्लासिक HTTP: नवीन ओळींसह प्लेन टेक्स्ट तुकडे
  - MCP: मेटाडेटासहित संरचित LoggingMessageNotification ऑब्जेक्ट्स

- **क्लायंट अंमलबजावणी:**
  - क्लासिक HTTP: साधा क्लायंट जो स्ट्रीमिंग प्रतिसाद प्रक्रिया करतो
  - MCP: अधिक प्रगतिशील क्लायंट जो विविध प्रकारच्या मेसेजेससाठी मेसेज हँडलर वापरतो

- **प्रगती अपडेट्स:**
  - क्लासिक HTTP: प्रगती मुख्य प्रतिसाद स्ट्रीमचाच भाग असते
  - MCP: प्रगती वेगळ्या सूचना संदेशांना पाठवली जाते आणि मुख्य प्रतिसाद शेवटी येतो

### शिफारसी

क्लासिक स्ट्रीमिंग (`/stream` वापरून केसेसमध्ये दाखवल्या प्रमाणे) आणि MCP स्ट्रीमिंग यामध्ये निवड करताना काही गोष्टी आपण सुचवितो.

- **सोप्या स्ट्रीमिंगसाठी:** क्लासिक HTTP स्ट्रीमिंग राबवणे सोपे आणि मूलभूत गरजांसाठी पुरेसे आहे.

- **संकुल, संवादात्मक अनुप्रयोगांसाठी:** MCP स्ट्रीमिंग अधिक संरचित दृष्टिकोन देतो ज्यामध्ये समृद्ध मेटाडेटा आणि सूचना व अंतिम निकाल यामध्ये वेगळेपणा आहे.

- **AI अनुप्रयोगांसाठी:** MCP ची सूचना प्रणाली लांबच्या AI कामांसाठी उपयुक्त आहे, जिथे प्रगती बद्दल वापरकर्त्यांना माहितीसह ठेवायचे असते.

## MCP मध्ये स्ट्रीमिंग

बरं, आपण क्लासिक स्ट्रीमिंग आणि MCP मधील स्ट्रीमिंग यातील फरकावर काही शिफारसी आणि तुलना पाहिली आहेत. आता तपशीलवार पाहूया की तुम्ही MCP मध्ये स्ट्रीमिंग कसे वापरू शकता.

MCP फ्रेमवर्कमध्ये स्ट्रीमिंग कसे कार्य करते हे समजून घेणे हे लांबच्या ऑपरेशन्स दरम्यान वापरकर्त्यांना रिअल-टाइम प्रतिसाद देणाऱ्या प्रतिसादक्षम ॲप्लिकेशन्स तयार करण्यासाठी आवश्यक आहे.

MCP मध्ये, स्ट्रीमिंगचा अर्थ मुख्य प्रतिसाद तुकड्यांमध्ये पाठवणे नाही, तर प्रत्येक वेळी एक टूल विनंती प्रक्रिया करत असताना **सूचना** क्लायंटला पाठवणे आहे. या सूचनांमध्ये प्रगती अपडेट्स, लॉग्ज किंवा अन्य घटना असू शकतात.

### हे कसे कार्य करते

मुख्य निकाल अजूनही एकाच प्रतिसादात पाठवला जातो. तथापि, प्रक्रिया दरम्यान सूचनांचे स्वतंत्र मेसेजेस पाठवून ते क्लायंटला रिअल-टाइम मध्ये अद्यतनित केले जाते. क्लायंटने या सूचनांचा हाताळणी आणि प्रदर्शन करण्यास सक्षम असणे आवश्यक आहे.

## सूचना म्हणजे काय?

आपण "सूचना" हा शब्द वापरला, तर MCP च्या संदर्भात त्याचा अर्थ काय आहे?

सूचना म्हणजे सर्व्हरकडून क्लायंटकडे पाठवलेला मेसेज, जो लांबच्या ऑपरेशन दरम्यान प्रगती, स्थिती किंवा इतर घटनांबद्दल माहिती देतो. सूचना पारदर्शकता आणि वापरकर्ता अनुभव सुधारतात.

उदाहरणार्थ, जर क्लायंटने सर्व्हरशी प्राथमिक हातमिळवणी केल्यानंतर सूचना पाठविणीची अपेक्षा असेल.

सूचना JSON मेसेज म्हणून अशा प्रकारे दिसू शकते:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

सूचना MCP मध्ये ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) या विषयाशी संबंधित असतात.

लॉगिंग कार्यान्वित करण्यासाठी, सर्व्हरने ते फिचर/क्षमता म्हणून सक्षम करणे आवश्यक आहे, उदाहरणार्थ:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> वापरल्या जाणाऱ्या SDK नुसार, लॉगिंग आपोआप सक्षम असू शकते, किंवा तुम्हाला ते स्पष्टपणे तुमच्या सर्व्हर कॉन्फिगरेशनमध्ये सक्षम करावे लागू शकते.

सूचनांच्या विविध प्रकार आहेत:

| पातळी    | वर्णन                       | उदाहरण वापर प्रकरण               |
|----------|----------------------------|---------------------------------|
| debug    | सविस्तर डिबगिंग माहिती    | फंक्शन एंट्री/एग्झिट पॉइंट्स   |
| info     | सामान्य माहितीपर मेसेजेस   | ऑपरेशन प्रगती अपडेट्स          |
| notice   | सामान्य पर महत्त्वपूर्ण घटना| कॉन्फिगरेशन बदल                  |
| warning  | इशारा स्थिती               | जुनी फीचर वापर                   |
| error    | त्रुटी स्थिती               | ऑपरेशन फेल्युअर                  |
| critical | गंभीर स्थिती               | सिस्टीम कॉम्पोनेंट फेल्युअर     |
| alert    | त्वरित कृती आवश्यक          | डेटाच्या भ्रष्टाचाराचा शोध लागला |
| emergency| सिस्टीम वापरायोग्य नाही    | संपूर्ण सिस्टीम फेल्युअर        |

## MCP मध्ये सूचनांची अंमलबजावणी

MCP मध्ये सूचनांची अंमलबजावणी करण्यासाठी, तुम्हाला सर्व्हर आणि क्लायंट दोन्ही बाजू तयार कराव्या लागतात जेणेकरून रिअल-टाइम अपडेट्स हाताळता येतील. यामुळे तुमचे ॲप्लिकेशन लांबच्या ऑपरेशन्स दरम्यान त्वरित प्रतिक्रिया देऊ शकेल.

### सर्व्हर बाजू: सूचना पाठवणे

सर्व्हर बाजूपासून सुरू करूया. MCP मध्ये तुम्ही असे टूल्स परिभाषित करता जे विनंत्या प्रक्रिया करताना सूचना पाठवू शकतात. सर्व्हर संदर्भ ऑब्जेक्ट (`ctx` सहसा) वापरून क्लायंटला मेसेजेस पाठवतो.

#### पायथन

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

मागील उदाहरणात, `process_files` टूल प्रत्येक फाईल प्रक्रिया करताना क्लायंटला तीन सूचना पाठवतो. `ctx.info()` मेथड माहितीपर मेसेजेस पाठवण्यासाठी वापरली जाते.

याव्यतिरिक्त, सूचनांना सक्षम करण्यासाठी, तुमच्या सर्व्हरने स्ट्रीमिंग ट्रान्सपोर्ट (जसे की `streamable-http`) वापरले पाहिजे आणि क्लायंटने सूचना प्रक्रिया करण्यासाठी मेसेज हँडलरची अंमलबजावणी केली पाहिजे. खाली सर्व्हरमध्ये `streamable-http` ट्रान्सपोर्ट वापरण्यासाठी कसे सेट करायचे ते आहे:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

या .NET उदाहरणात, `ProcessFiles` टूलला `Tool` अ‍ॅट्रिब्युटने सजवले गेले आहे आणि प्रत्येक फाइल प्रक्रिया करताना क्लायंटला तीन सूचना पाठवतो. `ctx.Info()` मेथड माहितीपर संदेश पाठवण्यासाठी वापरली जाते.

तुमच्या .NET MCP सर्व्हरमध्ये सूचनांना सक्षम करण्यासाठी, याची खात्री करा की तुम्ही स्ट्रीमिंग ट्रान्सपोर्ट वापरत आहात:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### क्लायंट बाजू: सूचना प्राप्त करणे

क्लायंटने मेसेज हँडलरची अंमलबजावणी करणे आवश्यक आहे जेणेकरून येणाऱ्या सूचना प्रक्रिया आणि प्रदर्शन करू शकेल.

#### पायथन

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

मागील कोडमध्ये, `message_handler` फंक्शन तपासतंय की येणारा मेसेज सूचना आहे का. असल्यास, ती सूचना प्रिंट करतो; नाही तर सामान्य सर्व्हर मेसेज म्हणून प्रक्रिया करतो. तसेच `ClientSession` `message_handler` सह प्रारंभ केलेले आहे ज्यामुळे सूचनांना हाताळले जाते.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

या .NET उदाहरणात, `MessageHandler` फंक्शन तपासतंय की येणारा मेसेज सूचना आहे का. असल्यास, ती सूचना प्रिंट करतो; नाही तर सामान्य सर्व्हर मेसेज म्हणून प्रक्रिया करतो. `ClientSession` ला मेसेज हँडलर `ClientSessionOptions` द्वारे जोडले गेले आहे.

सूचनांना सक्षम करण्यासाठी, याची खात्री करा की तुमच्या सर्व्हरने स्ट्रीमिंग ट्रान्सपोर्ट (जसे की `streamable-http`) वापर केला आहे आणि तुमच्या क्लायंटने मेसेज हँडलर ची अंमलबजावणी केली आहे.

## प्रगती सूचना आणि परिस्थिती

हा विभाग MCP मधील प्रगती सूचनांच्या संकल्पना, का महत्वाच्या आहेत आणि Streamable HTTP वापरून त्यांची अंमलबजावणी कशी करायची हे स्पष्ट करतो. यासोबतच तुमचा समज अधिक दृढ करण्यासाठी व्यावहारिक कार्यसुद्धा दिले आहे.

प्रगती सूचना म्हणजे लांबच्या ऑपरेशन दरम्यान सर्व्हरकडून क्लायंटला पाठवले जाणारे रिअल-टाइम मेसेजेस. संपूर्ण प्रक्रिया पूर्ण होण्याची वाट पाहण्याऐवजी, सर्व्हर क्लायंटला चालू स्थितीबद्दल अद्ययावत ठेवतो. हे पारदर्शकता, वापरकर्ता अनुभव सुधारते आणि डीबगिंग सुलभ करते.

**उदाहरण:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### प्रगती सूचना का वापराव्यात?

प्रगती सूचना महत्त्वाच्या आहेत कारण:

- **उत्तम वापरकर्ता अनुभव:** काम चालू असतानाच अपडेट दिसतात, केवळ शेवटी नाही.
- **रिअल-टाइम फीडबॅक:** क्लायंट प्रगती बार किंवा लॉग्स दर्शवू शकतात, ज्यामुळे ऍप प्रतिसादक्षम वाटते.
- **सोपे डीबगिंग व मॉनिटरिंग:** विकसक आणि वापरकर्ते पाहू शकतात की कुठे प्रक्रिया मंदावते किंवा अडकते.

### प्रगती सूचनांची अंमलबजावणी कशी करावी

खालीलप्रमाणे तुम्ही MCP मध्ये प्रगती सूचना अंमलात आणू शकता:

- **सर्व्हरवर:** `ctx.info()` किंवा `ctx.log()` वापरून प्रत्येक आयटम प्रक्रिया होताना सूचना पाठवा. हे मुख्य निकाल येण्याआधी क्लायंटला मेसेज पाठवते.
- **क्लायंटवर:** मेसेज हँडलर तयार करा जो येणाऱ्या सूचनांना ऐकतो आणि दाखवतो. हा हँडलर सूचना व अंतिम निकाल यांमध्ये फरक करतो.

**सर्व्हर उदाहरण:**

#### पायथन

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**क्लायंट उदाहरण:**

#### पायथन

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## सुरक्षा विचार

HTTP-आधारित ट्रान्सपोर्टसह MCP सर्व्हर तयार करताना सुरक्षा अत्यंत महत्त्वाची बाब आहे जी अनेक हल्ल्यांच्या मार्गदर्शक तत्त्वे आणि संरक्षण यंत्रणा आवश्यक करतो.

### आढावा

HTTP वर MCP सर्व्हर प्रदर्शित करताना सुरक्षा महत्त्वाची आहे. स्ट्रीम करू शकणारे HTTP नवीन हल्ल्यांसाठी शक्य तट निर्माण करते आणि तत्पर कॉन्फिगरेशनची गरज आहे.

### मुख्य मुद्दे


- **मूळ हेडर प्रमाणीकरण**: DNS रीबाइंडिंग हल्ल्यांपासून संरक्षण करण्यासाठी `Origin` हेडर नेहमीच प्रमाणीकरा.
- **लोकलहोस्ट बाइंडिंग**: स्थानिक विकासासाठी, सर्व्हर `localhost` ला बाइंड करा जेणेकरून ते सार्वजनिक इंटरनेटवर उघडे पडणार नाहीत.
- **प्रमाणीकरण**: उत्पादनासाठी तैनाती करताना प्रमाणीकरण (उदा., API कीज, OAuth) अंमलात आणा.
- **CORS**: प्रवेश मर्यादीसाठी Cross-Origin Resource Sharing (CORS) धोरणे सेट करा.
- **HTTPS**: ट्राफिक एन्क्रिप्ट करण्यासाठी उत्पादनात HTTPS वापरा.

### सर्वोत्तम पद्धती

- प्रमाणीकरणाशिवाय येणाऱ्या विनंत्या कधीही विश्वासार्ह समजू नका.
- सर्व प्रवेश आणि त्रुटीचे लॉग करा आणि निरीक्षण करा.
- सुरक्षा त्रुटी दुरुस्त करण्यासाठी नियमितपणे निर्भरता अपडेट करा.

### आव्हाने

- सुरक्षितता आणि विकासाची सुलभता यामध्ये संतुलन साधणे
- विविध क्लायंट वातावरणांसह सुसंगतता सुनिश्चित करणे

## SSE मधून Streamable HTTP मध्ये अपग्रेड करणे

सध्या Server-Sent Events (SSE) वापरणाऱ्या अ‍ॅप्लिकेशन्ससाठी, Streamable HTTP मध्ये स्थलांतरण केल्याने सुधारित क्षमता आणि तुमच्या MCP अंमलबजावणीत दीर्घकालीन टिकाव मिळतो.

### का अपग्रेड करावे?

SSE पासून Streamable HTTP कडे अपग्रेड करण्यासाठी दोन महत्त्वाचे कारणे आहेत:

- Streamable HTTP हा SSE पेक्षा चांगला स्केलिबिलिटी, सुसंगतता आणि समृद्ध अधिसूचना समर्थन ऑफर करतो.
- नवीन MCP अ‍ॅप्लिकेशन्ससाठी हा शिफारस केलेला प्रवाह आहे.

### स्थलांतर चरण

तुमच्या MCP अ‍ॅप्लिकेशन्समध्ये SSE वरून Streamable HTTP मध्ये स्थलांतर करण्याचा मार्ग खालीलप्रमाणे आहे:

- **सर्व्हर कोड अपडेट करा** `mcp.run()` मध्ये `transport="streamable-http"` वापरण्यासाठी.
- **क्लायंट कोड अपडेट करा** SSE क्लायंटच्या ऐवजी `streamablehttp_client` वापरायला.
- **क्लायंटमध्ये संदेश हँडलर लागू करा** जेणेकरून अधिसूचना प्रक्रिया करता येतील.
- **सध्याच्या टूल्स आणि वर्कफ्लोजशी सुसंगततेची चाचणी करा**.

### सुसंगतता राखणे

स्थलांतर प्रक्रियेदरम्यान सध्याच्या SSE क्लायंट्सशी सुसंगतता राखणे शिफारस केले जाते. काही धोरणे याप्रमाणे आहेत:

- तुम्ही वेगवेगळ्या एंडपॉइंट्सवर दोन्ही SSE आणि Streamable HTTP चालवून दोन्ही प्रकारांना समर्थन देऊ शकता.
- हळूहळू क्लायंट्सना नवीन प्रवाहावर स्थलांतरित करा.

### आव्हाने

स्थलांतर करताना खालील आव्हानांवर लक्ष देणे आवश्यक आहे:

- सर्व क्लायंट्स अपडेट करणे सुनिश्चित करणे
- अधिसूचना वितरणातील फरक हाताळणे

## सुरक्षा बाबी

कोणताही सर्व्हर अंमलात आणताना, विशेषतः MCP मध्ये Streamable HTTP सारख्या HTTP आधारित प्रवाहांचा वापर करताना सुरक्षा सर्वोच्च गरज आहे.

HTTP आधारित प्रवाहांसह MCP सर्व्हर अंमलबजावणी करताना सुरक्षा ही एक महत्त्वाची बाब असते ज्यासाठी विविध हल्ल्यांच्या मार्गांवर व संरक्षण यंत्रणांवर काळजीपूर्वक लक्ष देणे आवश्यक आहे.

### आढावा

MCP सर्व्हर HTTP वर थेट प्रदर्शित करतांना सुरक्षा अत्यंत महत्त्वाची आहे. Streamable HTTP नवीन हल्ल्यांचे मार्ग उघडतो आणि नीट कॉन्फिगरेशनची गरज असते.

येथे काही महत्त्वाच्या सुरक्षा बाबी आहेत:

- **मूळ हेडर प्रमाणीकरण**: DNS रीबाइंडिंग हल्ल्यांपासून संरक्षणासाठी `Origin` हेडर नेहमीच प्रमाणीकरा.
- **लोकलहोस्ट बाइंडिंग**: स्थानिक विकासासाठी, सर्व्हर `localhost` ला बाइंड करा जेणेकरून ते सार्वजनिक इंटरनेटवर उघडे पडणार नाहीत.
- **प्रमाणीकरण**: उत्पादनासाठी तैनाती करताना प्रमाणीकरण (उदा., API कीज, OAuth) अंमलात आणा.
- **CORS**: प्रवेश मर्यादीसाठी Cross-Origin Resource Sharing (CORS) धोरणे सेट करा.
- **HTTPS**: ट्राफिक एन्क्रिप्ट करण्यासाठी उत्पादनात HTTPS वापरा.

### सर्वोत्तम पद्धती

तसेच, तुमच्या MCP स्ट्रीमिंग सर्व्हरमध्ये सुरक्षा अंमलबजावणी करताना खालील सर्वोत्तम पद्धती पाळा:

- प्रमाणीकरणाशिवाय येणाऱ्या विनंत्या कधीही विश्वासार्ह समजू नका.
- सर्व प्रवेश आणि त्रुटीचे लॉग करा आणि निरीक्षण करा.
- सुरक्षा त्रुटी दुरुस्त करण्यासाठी नियमितपणे निर्भरता अपडेट करा.

### आव्हाने

MCP स्ट्रीमिंग सर्व्हरमध्ये सुरक्षा अंमलबजावणी करताना काही आव्हाने येतील:

- सुरक्षितता आणि विकासाची सुलभता यामध्ये संतुलन साधणे
- विविध क्लायंट वातावरणांसह सुसंगतता सुनिश्चित करणे

### असाइनमेंट: स्वतःची स्ट्रीमिंग MCP अ‍ॅप तयार करा

**परिस्थिती:**
असा MCP सर्व्हर आणि क्लायंट तयार करा जिथे सर्व्हर आयटम्सची यादी (उदा., फाइल्स किंवा दस्तऐवज) प्रक्रिया करतो आणि प्रत्येक आयटम प्रक्रियेनंतर एक अधिसूचना पाठवतो. क्लायंट ट्रिगर झालेली प्रत्येक अधिसूचना दाखवायला हवी.

**चरणे:**

1. एक सर्व्हर टूल तयार करा जे यादी प्रक्रिया करून प्रत्येक आयटमसाठी अधिसूचना पाठवते.
2. एक क्लायंट तयार करा ज्यामध्ये संदेश हँडलर असेल आणि जो अधिसूचना रिअल-टाइममध्ये दाखवेल.
3. तुमची अंमलबजावणी सर्व्हर आणि क्लायंट दोन्ही चालवून तपासा आणि अधिसूचना पाहा.
