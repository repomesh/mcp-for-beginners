# മോഡൽ കോൺടെക്സ് പ്രോട്ടോക്കോൾ (MCP) ഉപയോഗിച്ചുള്ള HTTPS സ്റ്റ്രീമിംഗ്

ഈ അധ്യായം HTTPS ഉപയോഗിച്ച് മോഡൽ കോൺടെക്സ് പ്രോട്ടോക്കോൾ (MCP) വഴി സുരക്ഷിതവും സ്കെയിലബിളുമായ റിയൽ-ടൈം സ്റ്റ്രീമിംഗ് നടപ്പാക്കാനുള്ള വിശദമായ മാർഗ്ഗനിർദ്ദേശം നൽകുന്നു. സ്റ്റ്രീമിംഗിന്റെ പ്രേരണ, ലഭ്യമായ ട്രാൻസ്പോർട്ട് മെക്കാനിസങ്ങൾ, MCPയിൽ സ്റ്റ്രീമബിൾ HTTP എങ്ങനെ നടപ്പാക്കാം, സുരക്ഷാ മികച്ച രീതികൾ, SSE-യിൽനിന്നുള്ള മൈഗ്രേഷൻ, കൂടാതെ നിങ്ങളുടെ സ്വന്തം സ്റ്റ്രീമിംഗ് MCP ആപ്ലിക്കേഷനുകൾ നിർമ്മിക്കുന്നതിന് പ്രായോഗിക മാർഗ്ഗനിർദ്ദേശങ്ങൾ ഉൾപ്പെടുത്തിയിട്ടുണ്ട്.

> **മുന്നോട്ട് നോക്കുമ്പോൾ:** ഈ പാഠം **MCP Specification 2025-11-25** അനുസരിച്ച് സ്റ്റ്രീമബിൾ HTTPയെ വിവരിക്കുന്നു, ഇവിടെ `initialize` സമയത്ത് ഒരു സെഷൻ സ്ഥാപിക്കുകയും `Mcp-Session-Id` ഹെഡർ ഉപയോഗിച്ച് പിൻ ചെയ്യപ്പെടുകയും ചെയ്യുന്നു. `2026-07-28` റിലീസ് കാൻഡിഡേറ്റ് ഹാൻഡ്‌شേക്കും സെഷൻ ഐഡിയും പൂർണ്ണമായും ഒഴിവാക്കി ഓരോ അഭ്യർത്ഥനയും സ്വയംകൃതവും ഏതൊരു സെർവർ ഇൻസ്റ്റാൻസിലേക്കും റൂട്ടുചെയ്യാവുന്നതുമായതായി മാറ്റുന്നു. വിശദാംശങ്ങൾക്ക് [What's Changing in MCP: The 2026-07-28 Release Candidate](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) കാണുക.

## MCP-യിലെ ട്രാൻസ്പോർട്ട് മെക്കാനിസങ്ങളும் സ്റ്റ്രീമിംഗും

MCP-യിൽ ലഭ്യമായ വ്യത്യസ്ത ട്രാൻസ്പോർട്ട് മെക്കാനിസങ്ങളെപ്പറ്റിയും ക്ലയന്റും സെർവറും തമ്മിലുള്ള റിയൽ-ടൈം ആശയവിനിമയത്തിനായി ഇവയുടെ സ്റ്റ്രീമിംഗ് കഴിവുകൾ സജ്ജമാക്കുന്നതിലെ പങ്ക് എന്നതും ഈ വിഭാഗം പര്യവേക്ഷിക്കുന്നു.

### ട്രാൻസ്പോർട്ട് മെക്കാനിസം എന്താണ്?

ഒരു ട്രാൻസ്പോർട്ട് മെക്കാനിസം ക്ലയന്റും സെർവറും തമ്മിൽ ഡാറ്റ എങ്ങനെ കൈമാറ്റം ചെയ്യപ്പെടുന്നു എന്നതിന് നിർവ്വചനം നൽകുന്നു. MCP വിവിധ പരിസ്ഥിതികളിലും ആവശ്യകതകളിലും അനുയോജ്യമായ പല ട്രാൻസ്പോർട്ട് തരങ്ങൾ പിന്തുണക്കുന്നു:

- **stdio**: സ്റ്റാൻഡേർഡ് ഇൻപുട്ട്/ഔട്ട്‌പുട്ട്, പ്രാദേശികവും CLI അടിസ്ഥാനമാക്കിയുള്ള ടൂളുകൾക്കും അനുയോജ്യം. ലളിതമായുള്ളതെങ്കിലും വെബിനും ക്ലൗഡിനും അനുയോജ്യമല്ല.
- **SSE (Server-Sent Events)**: സെർവർ HTTP വഴി റിയൽ-ടൈം അപ്‌ഡേറ്റുകൾ ക്ലയന്റുകളിൽ പുഷ് ചെയ്യാൻ അനുവദിക്കുന്നു. വെബ് UI-കളിക്കായി നല്ലത്, പക്ഷേ സ്കെയിലബിലിറ്റി, ഫോംപ്രകാരം പരിമിതമായതാണ്. MCP Specification 2025-06-18 പ്രകാരം സ്റ്റാൻഡലോൺ SSE (Server-Sent Events) ട്രാൻസ്പോർട്ട് ഡിജേർകൃതമാക്കി "Streamable HTTP" ട്രാൻസ്പോർട്ടുക് മാറ്റി.
- **Streamable HTTP**: ആധുനിക HTTP-അധിഷ്ഠിത സ്റ്റ്രീമിംഗ് ട്രാൻസ്പോർട്ട്, അറിയിപ്പുകളും മെച്ചപ്പെട്ട സ്കെയിലബിലിറ്റിയും പിന്തുണയ്‌ക്കുന്നു. മിക്ക ഉത്പാദനവും ക്ലൗഡ് സാഹചര്യതകൾക്കും ശുപാർശ ചെയ്യുന്നു.

### താരതമ്യ പട്ടിക

ഈ ട്രാൻസ്പോർട്ട് മെക്കാനിസങ്ങൾ തമ്മിലുള്ള വ്യത്യാസങ്ങൾ മനസിലാക്കാൻ താഴെ കാണിക്കുന്ന താരതമ്യ പട്ടിക കാണുക:

| ട്രാൻസ്പോർട്ട്        | റിയൽ-ടൈം അപ്‌ഡേറ്റുകൾ | സ്റ്റ്രീമിംഗ് | സ്കെയിലബിലിറ്റി | ഉപയോഗം                |
|---------------------|---------------------|-----------|---------------|------------------------|
| stdio               | ഇല്ല                 | ഇല്ല       | കുറഞ്ഞത്       | പ്രാദേശിക CLI ടൂൾസ്    |
| SSE                 | ഉണ്ട്                | ഉണ്ട്      | മധ്യമം        | വെബ്, റിയൽ-ടൈം അപ്‌ഡേറ്റുകൾ |
| Streamable HTTP     | ഉണ്ട്                | ഉണ്ട്      | ഉയർന്നത്       | ക്ലൗഡ്, മൾട്ടി-ക്ലയന്റ് |

> **ടിപ്പ്:** ശരിയായ ട്രാൻസ്പോർട്ട് തിരഞ്ഞെടുക്കുക പെർഫോർമൻസ്, സ്കെയിലബിലിറ്റി, ഉപയോക്തൃ അനുഭവത്തിൽ സ്വാധീനം ചെലുത്തുന്നു. ആധുനികവും സ്കെയിലബിളുമായ ക്ലൗഡ്-സജ്ജവുമായ ആപ്ലിക്കേഷനുകൾക്കായി **Streamable HTTP** ശുപാർശ ചെയ്യുന്നു.

മുമ്പ് കാണിച്ച stdio, SSE ട്രാൻസ്പോർട്ടുകളെയും ഈ അധ്യായത്തിൽ ചർച്ചചെയ്ത Streamable HTTP ട്രാൻസ്പോർട്ടിനെയും ശ്രദ്ധിക്കുക.

## സ്റ്റ്രീമിംഗ്: ആശയങ്ങളും പ്രേരണകളും

സ്റ്റ്രീമിംഗിന്റെ അടിസ്ഥാന ആശയങ്ങളും പ്രേരണകളും മനസ്സിലാക്കുന്നത് ഫലപ്രദമായ റിയൽ-ടൈം ആശയവിനിമയം നടപ്പിലാക്കാൻ അനിവാര്യമാണ്.

**സ്റ്റ്രീമിംഗ്** ഒരു നെറ്റ്‌വർക്ക് പ്രോഗ്രാമിംഗ് സാങ്കേതിക വിദ്യയാണ്, ഇത് മുഴുവൻ പ്രതികരണം തയാറാകുന്നത് കാത്തിരിക്കാതെ ഡാറ്റയിൽ ചെറുതും നിയന്ത്രിക്കാവുന്നതുമായ ഭാഗങ്ങളായി അല്ലെങ്കിൽ സീരിസായി ഇവന്റ്‌സ് ആയി അയയ്ക്കാനും സ്വീകരിക്കാനുമുള്ള സൗകര്യം നൽകുന്നു. ഇത് പ്രത്യേകിച്ച് ഉപകാരപ്രദമാണ്:

- വലുതായ ഫയലുകളിലോ ഡാറ്റാസെറ്റുകളിലോ.
- റിയൽ-ടൈം അപ്‌ഡേറ്റുകൾ (ഉദാ., ചാറ്റ്, പ്രോഗ്രസ്സ് ബാറുകൾ).
- ദൈർഘ്യമേറിയ കണക്കുകൂട്ടലുകൾ, ഉപയോക്താവിനെ അറിയിച്ച് വയ്ക്കേണ്ടത് ആവശ്യമായപ്പോൾ.

സ്റ്റ്രീമിംഗിനെക്കുറിച്ച് നിങ്ങൾ അറിയേണ്ടതുളള പ്രധാന बातें:

- ഡാറ്റ ഉൾക്കൂട്ടികളായി അല്ലെങ്കിൽ പാതിവയായി വിഭജിച്ച് ക്രമം പാലിച്ച് നൽകുന്നു.
- ക്ലയന്റ് വരുന്നതുപോലെ ഡാറ്റ പ്രോസസ് ചെയ്യാം.
- അറിയപ്പെടുന്ന വൈകീർ‌തമുറപ്പ് കുറയ്ക്കുകയും ഉപയോക്തൃ അനുഭവം മെച്ചപ്പെടുത്തുകയും ചെയ്യുന്നു.

### സ്റ്റ്രീമിംഗ് എതുകൊണ്ട് ഉപയോഗിക്കണം?

സ്റ്റ്രീമിംഗ് ഉപയോഗിക്കാനുള്ള കാരണങ്ങൾ:

- ഉപയോക്താക്കൾക്ക് ഉടനെ പ്രതികരണം കിട്ടുകയും, ഒടുവിൽ മാത്രമല്ല.
- റിയൽ-ടൈം ആപ്ലിക്കേഷനുകളും പ്രതികരണീയമായ UI-കളും സജ്ജമാക്കുന്നു.
- നെറ്റ്‌വർക്ക്, കമ്പ്യൂട്ട് റിസോഴ്സുകൾ കൂടുതൽ കാര്യക്ഷമമായി ഉപയോഗിക്കുന്നു.

### ലളിതമായ ഉദാഹരണം: HTTP സ്റ്റ്രീമിംഗ് സെർവർ & ക്ലയന്റ്

സ്റ്റ്രീമിംഗ് എങ്ങനെ നടപ്പാക്കാമെന്ന് കാണിക്കുന്ന ലളിതമായ ഉദാഹരണമാണ് ഇതിൽ:

#### പൈതൺ

**സർവർ (പൈതൺ, FastAPI, StreamingResponse ഉപയോഗിച്ച്):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**ക്ലയന്റ് (പൈതൺ, requests ഉപയോഗിച്ച്):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

ഈ ഉദാഹരണം എല്ലാം തയ്യാറാകുന്നതിന് കാത്തിരുന്നുവല്ല, സർവർ ലഭ്യമായ സന്ദേശങ്ങൾ സീരിയൽ ആയി ക്ലയന്റിലേക്ക് അയയ്ക്കുന്ന രീതി കാണിക്കുന്നു.

**എങ്ങനെ പ്രവർത്തിക്കുന്നു:**

- മസേഴ്‌ർ ഓരോ സന്ദേശവും തയ്യാറാകുമ്പോൾ അയയ്ക്കുന്നു.
- ക്ലയന്റ് വരുന്നതുപോലെ ഓരോ ചങ്കും സ്വീകരിക്കുകയും പ്രിന്റ് ചെയ്യുകയും ചെയ്യുന്നു.

**ആവശ്യകതകൾ:**

- മസേഴ്‌ർ ഒരു സ്റ്റ്രീമിംഗ് പ്രതികരണം ഉപയോഗിക്കണം (ഉദാ., FastAPI-യിലെ `StreamingResponse`).
- ക്ലയന്റ് പ്രതികരണത്തെ ഒരു സ്റ്റ്രീമായി പ്രോസസ് ചെയ്യണം (`stream=True` requests-ൽ).
- Content-Type സാധാരണയായി `text/event-stream` അല്ലെങ്കിൽ `application/octet-stream` ആണ്.

#### ജാവ

**സർവർ (ജാവ, Spring Boot, Server-Sent Events ഉപയോഗിച്ച്):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**ക്ലയന്റ് (ജാവ, Spring WebFlux WebClient ഉപയോഗിച്ച്):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**ജാവ നടപ്പാക്കൽ കുറിപ്പുകൾ:**

- Reactive ഫ്ലക്സ് സ്റ്റാക്ക് ഉപയോഗിച്ച് Spring Boot സ്ട്രീമിംഗ്.
- `ServerSentEvent` ഇവന്റ് സ്റ്റ്രീമിംഗിൽ ഘടനാപരമായ ഇനങ്ങൾ നൽകുന്നു.
- `WebClient` ന്റെ `bodyToFlux()` reactive streaming ഉപഭോഗം അനുവദിക്കുന്നു.
- `delayElements()` ഇവന്റുകൾക്കിടയിലെ പ്രോസസിംഗ് സമയം അനുകരിക്കുന്നു.
- ഇവന്റുകൾക്കായി ടൈപ്പുകൾ (ഉദാ., `info`, `result`) ക്ലയന്റ് കൈകാര്യം മെച്ചപ്പെടുത്താൻ.

### താരതമ്യം: ക്ലാസിക്കൽ സ്റ്റ്രീമിംഗ് vs MCP സ്റ്റ്രീമിംഗ്

സ്റ്റ്രീമിംഗ് ക്ലാസിക്കൽ രീതിയിലും MCP-യിലെ സ്റ്റ്രീമിംഗിലും പ്രവർത്തന വ്യത്യാസങ്ങൾ താഴെപ്പറയുന്ന പോലെ:

| സവിശേഷത              | ക്ലാസിക്കൽ HTTP സ്റ്റ്രീമിംഗ്       | MCP സ്റ്റ്രീമിംഗ് (അറിയിപ്പുകൾ)      |
|-----------------------|--------------------------------|------------------------------------|
| മുഖ്യ പ്രതികരണം        | ചെരിഞ്ഞു വന്ന ചുങ്കുകൾ              | ഒറ്റ, അവസാനത്തിൽ                     |
| പുരോഗതി അപ്‌ഡേറ്റുകൾ   | ഡാറ്റ ചുങ്കുകളായി അയങ്ങുന്നു          | അറിയിപ്പുകളായി അയങ്ങുന്നു             |
| ക്ലയന്റ് ആവശ്യകതകൾ     | സ്റ്റ്രീമാണ് പ്രോസസ് ചെയ്യണം            | മെസ്സേജ് കൈകാര്യം ചെയ്യുന്ന ഹാൻഡ്ലർ വേണം |
| ഉപയോഗം               | വലുതൻ ഫയലുകൾ, AI ടോക്കൺ സ്ട്രീമുകൾ | പുരോഗതി, ലോഗുകൾ, റിയൽ-ടൈം ഫീഡ്ബാക്ക് |

### പ്രധാന വ്യത്യാസങ്ങൾ

കൂടാതെ, ഇവിടെ ചില പ്രധാന വ്യത്യാസങ്ങൾ:

- **ആശയവിനിമയ പാറ്റേൺ:**
  - ക്ലാസിക്കൽ HTTP സ്റ്റ്രീമിംഗ്: ലളിതമായ ചെറുചുങ്കുകളും ട്രാൻസ്ഫർ എൻക്കോഡിങ്ങും ഉപയോഗിക്കുന്നത്
  - MCP സ്റ്റ്രീമിംഗ്: JSON-RPC പ്രോട്ടോക്കോൾ ഉപയോഗിച്ച ഘടനാപരമായ അറിയിപ്പ് സമ്പ്രദായം

- **മെസ്സേജ് ഫോർമാറ്റ്:**
  - ക്ലാസിക്കൽ HTTP: പൂർണ്ണമായ ടെക്സ്റ്റ് ചങ്‌കുകളും ന്യൂലൈൻ രേഖകൾ
  - MCP: ധാരാളം മെറ്റാഡേറ്റ ഉള്ള ഘടനാപരമായ LoggingMessageNotification ഒബ്‌ജക്റ്റുകൾ

- **ക്ലയന്റ് നടപ്പാക്കൽ:**
  - ക്ലാസിക്കൽ HTTP: ലളിതമായ സ്റ്റ്രീം പ്രോസസർ ക്ലയന്റ്
  - MCP: വ്യത്യസ്ത ഇനങ്ങളായ മെസ്സേജ് കൈകാര്യംക്ക് മെസ്സേജ് ഹാൻഡ്ലർ ഉള്ള ക്ലയന്റ്

- **പുരോഗതി അപ്‌ഡേറ്റുകൾ:**
  - ക്ലാസിക്കൽ HTTP: മുഖ്യ പ്രതികരണത്തിന്റെ ഭാഗമാണ്
  - MCP: പ്രധാന പ്രതികരണം അവസാനത്തേക്ക് മാത്രമാണ്; പുരോഗതി പ്രത്യേകം അറിയിപ്പുകൾ വഴി അയയ്ക്കുന്നു

### ശുപാർശകൾ

നിങ്ങളുടെ ആവശ്യകതകൾ പ്രകാരം ക്ലാസിക്കൽ സ്റ്റ്രീമിംഗ് (ഉദാ., `/stream` എന്ന എന്റ്പോയിന്റ്) തിരഞ്ഞെടുക്കണമെന്നോ MCP ഉപയോഗിച്ച സ്റ്റ്രീമിംഗ് തിരക്കണമെന്നോ നടത്തുമ്പോൾ നിർദേശങ്ങൾ ഉണ്ട്.

- **പരമാർശമില്ലാത്ത ലളിത സ്റ്റ്രീമിംഗിനായി:** ക്ലാസിക്കൽ HTTP സ്റ്റ്രീമിംഗ് ലളിതവും എളുപ്പത്തിൽ നടപ്പാക്കാവുന്നതും ആണ്.

- **ജടിലമായ, ഇന്ററാക്ടീവ് ആപ്ലിക്കേഷനുകൾക്കായി:** MCP സ്റ്റ്രീമിംഗ് മെറ്റാഡാറ്റയും അറിയിപ്പുകളും വ്യത്യസ്തവും ഘടനാപരമായ സമീപനമാണ് നൽകുന്നത്.

- **AI ആപ്ലിക്കേഷനുകൾക്കായി:** MCP-യുടെ അറിയിപ്പ് സംവിധാനം ദൈർഘ്യമേറിയ AI ടാസ്‌ക്കുകളിൽ പുരോഗതി പിന്തുടരണം ഉറപ്പാക്കാൻ ഉപകരിക്കുന്നു.

## MCP-യിൽ സ്റ്റ്രീമിംഗ്

ഇപ്പോൾ ക്ലാസിക്കൽ സ്റ്റ്രീമിംഗും MCP സ്റ്റ്രീമിംഗും തമ്മിലുള്ള വ്യത്യാസങ്ങളും ശുപാർശകളും നിങ്ങൾ കണ്ടിട്ടുണ്ട്. ഇനി MCP-യിൽ സ്റ്റ്രീമിംഗ് എങ്ങനെ ഉപയോഗിക്കാമെന്ന് വിശദമായി നോക്കാം.

MCP ഫ്രെയിമ്വർക്കിൽ സ്റ്റ്രീമിംഗ് എങ്ങനെ പ്രവർത്തിക്കുന്നു എന്നത് ബോധ്യമാക്കുന്നത് ദൈർഘ്യമേറിയ പ്രവർത്തനങ്ങളുടെ ഇടയിൽ ഉപയോക്താക്കൾക്ക് റിയൽ-ടൈം ഫീഡ്ബാക്ക് നൽകുന്ന പ്രതികരണക്ഷമ ആപ്ലിക്കേഷനുകൾ നിർമ്മിക്കുന്നതിനും ആവശ്യമാണ്.

MCP-യിൽ സ്റ്റ്രീമിംഗ് പ്രധാനമായി മുഖ്യ പ്രതികരണം ചെരിഞ്ഞു അയയ്ക്കാനുള്ളതു അല്ല, പക്ഷെ ഒരു ടൂൾ അഭ്യർത്ഥന പ്രോസസ്സ് ചെയ്യുമ്പോൾ **അറിയിപ്പുകൾ** ക്ലയന്റിലേക്ക് അയയ്ക്കുന്നതാണ്. പുരോഗതി അപ്‌ഡേറ്റുകളും ലോഗുകളും മറ്റ് ഇവന്റുകളും അറിയിപ്പുകളിൽ ഉൾപ്പെടുത്തിയിരിക്കുന്നു.

### എങ്ങനെ പ്രവർത്തിക്കുന്നു

മുഖ്യ ഫലം ഒരു ഒറ്റ പ്രതികരണമായി അയയ്ക്കുന്നു. എന്നാൽ പ്രോസസ്സ് നടത്തുമ്പോൾ അറിയിപ്പ് മെസ്സേജുകൾ വേർതിരിച്ച് ക്ലയന്റിനെ实时മായി അപ്‌ഡേറ്റ് ചെയ്യാം. ക്ലയന്റ് ഈ അറിയിപ്പുകൾ കൈകാര്യം ചെയ്ത് പ്രദർശിപ്പിക്കാൻ সক্ষমമായിരിക്കണം.

## അറിയിപ്പ് എന്താണ്?

"അറിയിപ്പ്" എന്നത് MCP കൺ‌ടെക്സ്റ്റിൽ എന്താണെന്ന് നോക്കാം.

ഒരു അറിയിപ്പ് സെർവറിൽ നിന്നു ക്ലയന്റിലേക്ക് അയക്കുന്ന ഒരു സന്ദേശമാണ്, ദൈർഘ്യമേറിയ പ്രവർത്തനത്തിന് പുരോഗതി, നില, അല്ലെങ്കിൽ മറ്റ് ഇവന്റുകൾ അറിയിക്കാൻ. അറിയിപ്പുകൾ പരദർശിത്വവും ഉപയോക്തൃ അനുഭവവും മെച്ചപ്പെടുത്തുന്നു.

ഉദാഹരണത്തിന്, സെർവറുമായുള്ള പ്രാരംഭ ഹാൻഡ്‌ഷേക്ക് പൂർത്തിയാകുമ്പോൾ ക്ലയന്റ് ഒരു അറിയിപ്പ് അയക്കേണ്ടതാണ്.

ഒരു അറിയിപ്പ് JSON സന്ദേശമായി ഇങ്ങനെ പ്രത്യക്ഷപ്പെടും:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

MCP-യിൽ അറിയിപ്പുകൾ ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) എന്ന ടോപ്പിക്കിൽ属ിച്ചവയാണ്.

ലോഗിംഗ് പ്രവർത്തനക്ഷമമാക്കാൻ, സെർവർ ഈ സവിശേഷത/ക്ഷമത ഇങ്ങനെ സജ്ജമാക്കണം:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> ഉപയോഗിക്കുന്ന SDK അനുസരിച്ച്, ഡിഫാൾട്ടായി ലോഗിംഗ് പ്രവർത്തനക്ഷമമാണോ, അല്ലെങ്കിൽ സെർവർ കോൺ‌ഫിഗറേഷനിൽ ഇത് വ്യക്തമായി പ്രവർത്തനക്ഷമമാക്കണമെന്നും തീർച്ച ചെയ്യണം.

വിവിധ തരത്തിലുള്ള അറിയിപ്പുകൾ ഉണ്ട്:

| തലത്തിൽ    | വിവരണം                      | ഉദാഹരണ ഉപയോഗം                  |
|------------|-----------------------------|---------------------------------|
| debug      | വിശദമായ ഡീബഗിങ് വിവരങ്ങൾ   | ഫംഗ്ഷൻ പ്രവേശന/പുറത്തേയ്ക്കു പോയിട്ടില്ലാത്ത പോയിന്റുകൾ |
| info       | പൊതുവായ വിവര സന്ദേശങ്ങൾ      | പ്രവർത്തന പുരോഗതി അപ്‌ഡേറ്റുകൾ    |
| notice     | സാധാരണ എന്നാൽ സൂചിപ്പിക്കപ്പെട്ട ഇവന്റുകൾ | കോൺഫിഗറേഷൻ മാറ്റങ്ങൾ            |
| warning    | മുന്നറിയിപ്പുകൾ              | മടങ്ങലായത് ഫീച്ചർ ഉപയോഗം       |
| error      | പിശക് അവസ്ഥകൾ              | പ്രവർത്തന പരാജയങ്ങൾ              |
| critical   | നിർണായക അവസ്ഥകൾ           | സിസ്റ്റം ഘടക പരാജയങ്ങൾ           |
| alert      | തక్షണമേൽപ്പെട്ട നടപടി വേണം | ഡാറ്റ തകർച്ച കണ്ടു               |
| emergency  | സിസ്റ്റം ഉപയോക്താവിന് അപ്രയോജ്യം | സമ്പൂർണ്ണ സിസ്റ്റം പരാജയം          |

## MCP-യിൽ അറിയിപ്പുകൾ നടപ്പാക്കൽ

MCP-യിൽ അറിയിപ്പുകൾ നടപ്പാക്കാനായി, സഹായകാലിക അപ്‌ഡേറ്റുകൾ കൈകാര്യം ചെയ്യുന്നതിന് സെർവർ, ക്ലയന്റ് ഇരുവശവും സജ്ജീകരിക്കണം. ഇതിലൂടെ നിങ്ങളുടെ ആപ്ലിക്കേഷൻ ദൈർഘ്യമേറിയ പ്രവർത്തനങ്ങൾക്കിടയിൽ ഉപയോക്താക്കൾക്ക് ഉടൻ പ്രതികരണം നൽകാൻ കഴിയും.

### സെർവർ വീതി: അറിയിപ്പുകൾ അയയ്ക്കുന്നത്

സെർവർ വീതി മുതലായിട്ട് തുടങ്ങാം. MCP-യിൽ, അഭ്യർത്ഥനകൾ പ്രോസസ്സ് ചെയ്യുമ്പോൾ അറിയിപ്പുകൾ അയയ്ക്കുന്ന ടൂളുകൾ നിർവ്വചിക്കുന്നു. സെർവർ കോൺടെക്സ്オബ്‌ജക്റ്റ് (സാധാരണയായി `ctx`) ഉപയോഗിച്ച് മെസ്സേജുകൾ ക്ലയന്റിലേക്ക് അയയ്ക്കുന്നു.

#### പൈതൺ

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

മുൻകൂട്ടി നൽകിയ ഉദാഹരണത്തിൽ, `process_files` ടൂൾ ഓരോ ഫയൽ പ്രോസസ് ചെയ്യുമ്പോൾ ക്ലയന്റിലേക്ക് മൂന്ന് അറിയിപ്പുകൾ അയയ്ക്കുന്നു. `ctx.info()` മെതഡ് വിവര സന്ദേശങ്ങൾ അയയ്ക്കാൻ ഉപയോഗിക്കുന്നു.

കൂടാതെ, അറിയിപ്പുകൾ പ്രവർത്തനക്ഷമമാക്കാൻ, നിങ്ങളുടെ സെർവർ streamable-http പോലുള്ള ഒരു സ്റ്റ്രീമിംഗ് ട്രാൻസ്പോർട്ട് ഉപയോഗിക്കണം, നിങ്ങളുടെ ക്ലയന്റ് അറിയിപ്പുകൾ കൈകാര്യംചെയ്‌ത ഒരു മെസ്സേജ് ഹാൻഡ്ലർ നടപ്പാക്കണം. സെർവറിൽ streamable-http ട്രാൻസ്പോർട്ട് ഉപയോഗിക്കാൻ ഇത് പോലെ സജ്ജീകരിക്കാം:

```python
mcp.run(transport="streamable-http")
```

#### .NET


നിങ്ങളുടെ .NET MCP സെർവറിൽ അറിയിപ്പുകൾ പ്രവർത്തനക്ഷമമാക്കാൻ, സ്റ്റ്രീമിംഗ് ട്രാൻസ്പോർട്ട് ഉപയോഗിക്കേണ്ടതുണ്ട്:

### ക്ലയന്റ് വീതി: അറിയിപ്പുകൾ സ്വീകരിക്കൽ

ക്ലയന്റ് നൽകിയ അറിയിപ്പുകൾ കൈകാര്യം ചെയ്ത് പ്രദർശിപ്പിക്കാൻ മെസ്സേജ് ഹാൻഡ്ലർ നടപ്പാക്കണം.

#### പൈതൺ

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

മുൻകൂട്ടിയ കോഡിൽ, `message_handler` ഫംഗ്ഷൻ വരുന്ന മെസ്സേജ് ഒരു അറിയിപ്പാണോ എന്ന് പരിശോധിക്കുന്നു. ശരിയെങ്കിൽ അത് പ്രിന്റ് ചെയ്യുന്നു, അല്ലെങ്കിൽ സാധാരണ സെർവർ മെസ്സേജായി പ്രോസസ് ചെയ്യുന്നു. കൂടാതെ `ClientSession` ഇൻഷ്യലൈസ് ചെയ്യുമ്പോൾ `message_handler` ഉപയോഗിക്കുന്നു.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

ഈ .NET ഉദാഹരണത്തിൽ, `MessageHandler` ഫംഗ്ഷൻ വരുന്ന മെസ്സേജ് ഒരു അറിയിപ്പാണോ എന്ന് പരിശോധിക്കുന്നു. ശരിയെങ്കിൽ അയവ് പ്രിന്റ് ചെയ്യുന്നു, അല്ലെങ്കിൽ സാധാരണ സെർവർ മെസ്സേജായി പ്രോസസ് ചെയ്യുന്നു. `ClientSession` മെസ്സേജ് ഹാൻഡ്ലർ `ClientSessionOptions` വഴിയാണ് ഇൻഷ്യലൈസ് ചെയ്യുന്നത്.

അറിയിപ്പുകൾ പ്രവർത്തനക്ഷമമാക്കാൻ, സെർവർ streamable-http പോലൊരു സ്റ്റ്രീമിംഗ് ട്രാൻസ്പോർട്ട് ഉപയോഗിക്കണം, ക്ലയന്റ് മെസ്സേജ് ഹാൻഡ്ലർ実装ചെയ്തിരിക്കണം.

## പുരോഗതി അറിയിപ്പുകളും സാഹചര്യങ്ങളും

ഈ വിഭാഗം MCP-യിലെ പുരോഗതി അറിയിപ്പുകളുടെ ആശയം, അവയുടെ പ്രധാന്യം, Streamable HTTP ഉപയോഗിച്ച് അവ എങ്ങനെ നടപ്പാക്കാമെന്ന് വിശദീകരിക്കുന്നു. നിങ്ങളുടെ മനസ്സിലാക്കലിനെ ശക്തിപ്പെടുത്താൻ പ്രായോഗിക ഒരു അസൈൻമെന്റ് ഇതിനൊപ്പം ലഭ്യമാണ്.

പുരോഗതി അറിയിപ്പുകൾ ദൈർഘ്യമേറിയ പ്രവർത്തനത്തിനിടെ സെർവറിൽ നിന്നുള്ള ക്ലയന്റിലേക്ക് റിയൽ-ടൈം സന്ദേശങ്ങളാണ്. മുഴുവൻ പ്രക്രിയ പൂർത്തിയാകുന്നതിനു കാത്തിരിക്കുന്നതിന് പകരം സെർവർ നിലവിലെ നില നിരന്തരം അപ്‌ഡേറ്റ് ചെയ്യുന്നു. ഇത് പരദർശിത്വം, ഉപയോക്തൃ അനുഭവം മെച്ചപ്പെടുത്തുകയും ഡീബഗിംഗ് എളുപ്പമാക്കുകയും ചെയ്യുന്നു.

**ഉദാഹരണം:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### പുരോഗതി അറിയിപ്പുകൾ എതുകൊണ്ട് ഉപയോഗിക്കണം?

പുരോഗതി അറിയിപ്പുകൾ ആവശ്യമാണ് പല കാരണങ്ങൾക്കായി:

- **മെച്ചപ്പെട്ട ഉപയോക്തൃ അനുഭവം:** പ്രവർത്തനം പുരോഗമിക്കുന്നതുപോലെ അപ്‌ഡേറ്റുകൾ കാണുന്നു, ഒടുവിൽ മാത്രമല്ല.
- **റിയൽ-ടൈം ഫീഡ്ബാക്ക്:** ക്ളയന്റുകൾ പ്രോഗ്രസ് ബാറുകളും ലോ​ഗുകളും പ്രദർശിപ്പിക്കാൻ കഴിയും, ആപ്പ് പ്രതികരണീയമാക്കുന്നു.
- **ഡീബഗിംഗ്, നിരീക്ഷണം എളുപ്പം:** വികസിപ്പിച്ചെടുക്കുന്നോ ഉപയോക്താവോ പ്രക്രിയ എവിടെ മന്ദഗതിയിലാണ് എന്നും കണ്ടുപിടിക്കാം.

### പുരോഗതി അറിയിപ്പുകൾ എങ്ങനെ നടപ്പാക്കാം

MCP-ൽ പുരോഗതി അറിയിപ്പുകൾ ഇങ്ങനെ നടപ്പാക്കാം:

- **സെർവറിൽ:** ഓരോ ഇനവും പ്രോസസ്സ് ചെയ്യുമ്പോൾ `ctx.info()` അല്ലെങ്കിൽ `ctx.log()` ഉപയോഗിച്ച് അറിയിപ്പ് അയയ്‌ക്കുക. ഇത് മുഖ്യ ഫലം തയ്യാറാവുന്നതിനു മുമ്പ് ക്ലയന്റിനെ സന്ദേശത്തോടെ അറിയിക്കുന്നു.
- **ക്ലയന്റിൽ:** വന്നറിയിപ്പുകൾ കേൾക്കുകയും പ്രദർശിപ്പിക്കുകയും ചെയ്യുന്ന മെസ്സേജ് ഹാൻഡ്ലർ നടപ്പാക്കുക. ഇത് അറിയിപ്പുകളും അവസാന ഫലവും വേർതിരിക്കുന്നു.

**സെർവർ ഉദാഹരണം:**

#### പൈതൺ

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**ക്ലയന്റ് ഉദാഹരണം:**

#### പൈതൺ

```python

    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## സുരക്ഷാരംഗം ശ്രദ്ധിക്കേണ്ട കാര്യങ്ങൾ

HTTP അടിസ്ഥാനത്തിലുള്ള ട്രാൻസ്പോർട്ടുകളിലൂടെ MCP സെർവറുകൾ നടപ്പാക്കുമ്പോൾ, ബഹുഭാദ് ആറ്റാക്കുകൾക്കും സംരക്ഷണ സംവിധാനങ്ങൾക്കും ഗണ്യമായ ശ്രദ്ധ നൽകേണ്ടത് അനിവാര്യമാകും.

### അവലോകനം

MCP സെർവറുകൾ HTTP വഴി പുറത്തു തുറക്കുമ്പോൾ സുരക്ഷ നിർണായകമാണ്. Streamable HTTP പുതിയ ആറ്റാക്ക് സാന്ദ്രതകൾ പരിചയപ്പെടുത്തുന്നു, അതിനാൽ കൃത്യമായ കോൺഫിഗറേഷൻ ആവശ്യമാണ്.

### പ്രധാനപ്പെട്ട കാര്യങ്ങൾ


- **ഒറിജിൻ ഹെഡർ വാലിഡേഷൻ**: DNS റിബൈന്റിംഗ് ആക്രമണങ്ങൾ തടയാൻ `Origin` ഹെഡർ എല്ലായ്പോഴും വാലിഡേറ്റ് ചെയ്യുക.
- **ലോകൽഹോസ്റ്റിനെ ബൈൻഡ് ചെയ്യൽ**: ലൊക്കൽ ഡെവലപ്പ്മെന്റ് కోసం, സെർവറുകൾ പൊതു ഇന്റർനെറ്റിന് പുറത്താക്കി `localhost` എന്നിടത്ത് ബൈൻഡ് ചെയ്യുക.
- **പ്രാമാണീകരണം**: പ്രൊഡക്ഷൻ ഡിപ്പ്ലോയ്മെന്റുകൾക്കായി (ഉദാ: API കീസുകൾ, OAuth) പ്രാമാണീകരണം നടപ്പിലാക്കുക.
- **CORS**: ആക്‌സസ് നിയന്ത്രിക്കാനുള്ള ക്രോസ്-ഒറിജിൻ റിസോഴ്‌സ് ഷെയറിംഗ് (CORS) നയങ്ങൾ ക്രമീകരിക്കുക.
- **HTTPS**: ട്രാഫിക്ക് എൻക്രിപ്റ്റ് ചെയ്യാൻ പ്രൊഡക്ഷനിൽ HTTPS ഉപയോഗിക്കുക.

### മികച്ച രീതികൾ

- വാലിഡേഷൻ ഇല്ലാത്ത വരവു കൊണ്ടുവരൽ അഭ്യർത്ഥനകളിൽ ഒരിക്കലും വിശ്വാസം വെക്കരുത്.
- എല്ലാ ആക്‌സസുകളും പിശകുകളും ලോഗ് ചെയ്ത് മോണിറ്റർ ചെയ്യുക.
- സുരക്ഷാ ഇടിവുകൾ പൂരിപ്പിക്കാൻ നിത്യവും ഡിപ്പെൻഡൻസികൾ അപ്ഡേറ്റ് ചെയ്യുക.

### വെല്ലുവിളികൾ

- വികസന സുഗമതയോടെയുള്ള സുരക്ഷയുടെ സമന്വയം
- വിവിധ ക്ലയന്റ് പരിതസ്ഥിതികളോടുള്ള പൊരുത്തം ഉറപ്പാക്കൽ

## SSE നിന്ന് Streamable HTTP-യിലേക്ക് അപ്ഡേറ്റ് ചെയ്യൽ

നിലവിൽ Server-Sent Events (SSE) ഉപയോഗിക്കുന്ന ആപ്ലിക്കേഷനുകൾക്ക് Streamable HTTP-യിലേക്ക് മാറ്റം നൽകുന്നതുമായി MCP നടപ്പാക്കലുകൾക്കായി മെച്ചപ്പെട്ട പ്രവർത്തനക്ഷമതയും ദീർഘകാല കാര്യക്ഷമതയും നൽകുന്നു.

### എന്തിനാണ് അപ്‌ഗ്രേഡ് ചെയ്യേണ്ടത്?

SSE-ൽനിന്നും Streamable HTTP-യിലേക്ക് അപ്ഗ്രേഡ് ചെയ്യാനുള്ള രണ്ടു പ്രധാന കാരണങ്ങൾ ഉണ്ടാക്കുന്നു:

- Streamable HTTP SSE-ശേഷം മെച്ചപ്പെട്ട സ്കെയിലബിലിറ്റി, പൊരുത്തം, സമ്പന്നമായ നോട്ടിഫിക്കേഷൻ പിന്തുണ നൽകുന്നു.
- പുതിയ MCP ആപ്ലിക്കേഷനുകൾക്കുള്ള ശുപാർശ ചെയ്ത ട്രാൻസ്പോർട്ടാണ് ഇത്.

### മൈഗ്രേഷൻ ചുവടുകൾ

നിങ്ങളുടെ MCP ആപ്ലിക്കേഷനുകളിൽ SSE-ൽനിന്ന് Streamable HTTP-യിലേക്ക് മാറ്റം നടത്തുന്നത് ഇങ്ങനെ ചെയ്യാം:

- **സർവർ കോഡ് അപ്ഡേറ്റ് ചെയ്യുക** `mcp.run()`-ൽ `transport="streamable-http"` ഉപയോഗിക്കാൻ.
- **ക്ലയന്റ് കോഡ് അപ്ഡേറ്റ് ചെയ്യുക** SSE ക്ലയന്റ് പകരം `streamablehttp_client` ഉപയോഗിക്കാൻ.
- **ക്ലയന്റിൽ ഒരു മെസേജ് ഹാൻഡ്ലർ നടപ്പിലാക്കുക** നോട്ടിഫിക്കേഷനുകൾ പ്രോസസ് ചെയ്യാൻ.
- **ഉഭയോക്തൃ കാണികൾ ഉപയോഗിച്ചുള്ള പൊരുത്തം പരിശോദിക്കുക**.

### പൊരുത്തം നിലനിർത്തൽ

മാറ്റത്തിനിടെ നിലവിലുള്ള SSE ക്ലയന്റുകളോടൊപ്പം പൊരുത്തം നിലനിർത്തുന്നത് ശുപാർശ ചെയ്യുന്നു. ചില ռազմനയങ്ങൾ ഇവിടെ ഉണ്ട്:

- വ്യത്യസ്ത എൻഡ്‌പോയിൻറുകളിൽ രണ്ട് തരത്തിലുള്ള ട്രാൻസ്പോർട്ടുകളും (SSE, Streamable HTTP) പരിപാലിക്കുക.
- ക്രമാതീതമായി ക്ലയന്റുകൾ പുതിയ ട്രാൻസ്പോർട്ടിലേക്ക് മാറ്റുക.

### വെല്ലുവിളികൾ

മൈഗ്രേഷൻ സമയത്ത് ഈ വെല്ലുവിളികൾ പരിഹരിക്കുക:

- എല്ലാ ക്ലയന്റുകളും അപ്ഡേറ്റ് ചെയ്യൽ ഉറപ്പാക്കൽ
- നോട്ടിഫിക്കേഷൻ ഡെലിവറിയിലെ വ്യത്യാസങ്ങൾ കൈകാര്യം ചെയ്യൽ

## സുരക്ഷാ പരിഗണനകൾ

MCP സーバറുകൾ നടപ്പിലാക്കുമ്പോൾ പ്രത്യേകിച്ച് Streamable HTTP പോലുള്ള HTTP അടിസ്ഥാനത്തിലുള്ള ട്രാൻസ്പോർട്ടുകൾ ഉപയോഗിക്കുമ്പോൾ സുരക്ഷയാണ് മുൻനിര പ്രാധാന്യം.

HTTP അടിസ്ഥാനമാക്കിയുള്ള ട്രാൻസ്പോർട്ടുകളോടെയുള്ള MCP സーバറുകൾ നടപ്പിലാക്കുമ്പോൾ ആക്രമണ മാർഗ്ഗങ്ങളും സംരക്ഷണ സംവിധാനങ്ങളും ശ്രദ്ധാപൂർവ്വം പരിശോധിക്കേണ്ടത് പ്രസക്തമാണ്.

### അവലോകനം

MCP സーバറുകൾ HTTP വഴി പുറപ്പടയ്ക്കുമ്പോൾ സുരക്ഷ അത്യന്താപേക്ഷിതമാണ്. Streamable HTTP പുതിയ ആക്രമണ കാഴ്ചപ്പാടുകൾ സൃഷ്ടിക്കുന്നു, ഇത് സൂക്ഷ്മ ക്രമീകരണം ആവശ്യമാണ്.

ഇവിടെ ചില മുഖ്യ സുരക്ഷാ പരിഗണനകൾ കൊടുക്കുന്നു:

- **ഒറിജിൻ ഹെഡർ വാലിഡേഷൻ**: DNS റിബൈന്റിംഗ് ആക്രമണംവയ്‌ക്കല്ലാതെ `Origin` ഹെഡർ എല്ലായ്പോഴും വാലിഡേറ്റ് ചെയ്യുക.
- **ലോകൽഹോസ്റ്റ് ബൈൻഡിംഗ്**: ലൊക്കൽ വികസനത്തിനായി പൊതു ഇന്റർനെറ്റിൽ തുറക്കാതെ സെർവറുകൾ `localhost`-ൽ ബൈൻഡ് ചെയ്യുക.
- **പ്രാമാണീകരണം**: പ്രൊഡക്ഷൻ പ്രയോഗങ്ങൾക്ക് (ഉദാ: API കീകൾ, OAuth) പ്രാമാണീകരണം നടപ്പിലാക്കുക.
- **CORS**: ആക്‌സസ് നിയന്ത്രിക്കാൻ ക്രോസ്-ഒറിജിൻ റിസോഴ്‌സ് ഷെയറിംഗ് (CORS) നയങ്ങൾ ക്രമീകരിക്കുക.
- **HTTPS**: ട്രാഫിക് എൻക്രിപ്റ്റ് ചെയ്യാൻ HTTPS ഉപയോഗിക്കുക.

### മികച്ച രീതികൾ

MCP സ്ട്രീമിംഗ് സർവറിൽ സുരക്ഷ നടപ്പിലാക്കുമ്പോൾ പാലിക്കേണ്ട ചില മികച്ച രീതികൾ ഇവിടെ കൊടുക്കുന്നു:

- വാലിഡേഷൻ ഇല്ലാത്ത വരവു കൊണ്ടുവരൽ അഭ്യർത്ഥനകളിൽ ഒരിക്കലും വിശ്വാസം വെക്കരുത്.
- എല്ലാ ആക്‌സസുകളും പിശകുകളും ലോഗ് ചെയ്ത് മോണിറ്റർ ചെയ്യുക.
- സുരക്ഷാ ഇടിവുകൾ പൂരിപ്പിക്കാൻ നിത്യവും ഡിപ്പെൻഡൻസികൾ അപ്ഡേറ്റ് ചെയ്യുക.

### വെല്ലുവിളികൾ

MCP സ്ട്രീമിംഗ് സർവറുകളിൽ സുരക്ഷ നടപ്പിലാക്കുമ്പോൾ ചില വെല്ലുവിളികൾ നേരിടേണ്ടി വരും:

- വികസന സുഗമതയോടെയുള്ള സുരക്ഷയുടെ സമന്വയം
- വിവിധ ക്ലയന്റ് പരിതസ്ഥിതികളോടുള്ള പൊരുത്തം ഉറപ്പാക്കൽ

### നിരീക്ഷണം: നിങ്ങളുടെ സ്വന്തം സ്ട്രീമിംഗ് MCP ആപ്പ് നിർമ്മിക്കുക

**സിഗ്‌നാരിൻ:**
ഒരു MCP സർവർ-ക്ലയന്റ് സംവിധാനം നിർമ്മിക്കുക, സർവർ സാധനങ്ങളുടെ (ഉദാ ഡോക്യുമെന്റുകൾ, ഫയലുകൾ) പട്ടിക പ്രോസസ്സ് ചെയ്ത് ഓരോ സാധനത്തിനും ഒരു നോട്ടിഫിക്കേഷൻ അയയ്ക്കും. ക്ലയന്റ് ഓരോ നോട്ടിഫിക്കേഷനും എപ്പോൾ എത്തിച്ചേരുകയാണോ അതു പ്രദർശിപ്പിക്കും.

**പടി:**

1. സാധനങ്ങളുടെ പട്ടിക പ്രോസസ്സ് ചെയ്ത് ഓരോ സാധനത്തിനും നോട്ടിഫിക്കേഷൻ അയയ്ക്കുന്ന സർവർ ടൂൾ നടപ്പിലാക്കുക.
2. നോട്ടിഫിക്കേഷനുകൾ റിയൽ ടൈമിൽ പ്രദർശിപ്പിക്കാൻ മെസേജ് ഹാൻഡ്ലർ അടങ്ങിയ ക്ലയന്റ് നടപ്പിലാക്കുക.
3. സർവർ, ക്ലയന്റ് ഇരുവരും പ്രവർത്തിപ്പിച്ച് നോട്ടിഫിക്കേഷനുകൾ നിരീക്ഷിച്ച് നിങ്ങളുടെ നടപ്പിലാക്കലിന്റെ പരിശോധന നടത്തുക.
