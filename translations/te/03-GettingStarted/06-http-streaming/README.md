# HTTPS స్ట్రీమింగ్ తో మోడల్ కాంటెక్స్ట్ ప్రోటోకాల్ (MCP)

ఈ అధ్యాయం MCP ఉపయోగించి HTTPS ద్వారా సురక్షిత, స్కేలబుల్ మరియు రియల్‌టైం స్ట్రీమింగ్‌ను అమలు చేయడానికి సమగ్ర గైడ్‌ను అందిస్తుంది. ఇది స్ట్రీమింగ్ కోసం ప్రేరణ, అందుబాటులో ఉన్న ట్రాన్స్పోర్ట్ మెకానిజమ్‌లు, MCPలో స్ట్రీమబుల్ HTTPని ఎలా అమలు చేయాలి, సెక్యూరిటీ ఉత్తమ ఆచారాలు, SSE నుండి మైగ్రేషన్ మరియు మీ స్వంత స్ట్రీమింగ్ MCP ఆప్లికేషన్‌లను నిర్మించడానికి ప్రాక్టికల్ మార్గదర్శకాలను కవర్ చేస్తుంది.

> **భవిష్యత్ దృష్టిలో:** ఈ పాఠం **MCP స్పెసిఫికేషన్ 2025-11-25** క్రింద స్ట్రీమబుల్ HTTPని వివరిస్తుంది, ఇక్కడ సెషన్ `initialize` సమయంలో ఏర్పాటు అవుతుంది మరియు `Mcp-Session-Id` హెడర్‌తో పిన్ చేయబడుతుంది. `2026-07-28` విడుదల అభ్యర్థనకి హ్యాండ్షేక్ మరియు సెషన్ ID పూర్తిగా తొలగించబడతాయి, దాంతో ప్రతి అభ్యర్ధన స్వయంపూరితంగా ఉంటుంది మరియు సటీక ప్రాంత చేర్పుల లేని ఏ సర్వర్ సూచికకు కూడా రౌటు చేయగలదు. వివరాలకు [MCPలో మార్పులు: 2026-07-28 రిలీజ్ అభ్యర్థి](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) చూడండి.

## MCPలో ట్రాన్స్పోర్ట్ మెకానిజమ్‌లు మరియు స్ట్రీమింగ్

ఈ విభాగం MCPలో అందుబాటులో ఉన్న వివిధ ట్రాన్స్పోర్ట్ మెకానిజమ్‌లను మరియు క్లయింట్లు మరియు సర్వర్ల మధ్య రియల్‌టైం కమ్యూనికేషన్ కోసం స్ట్రీమింగ్ సామర్థ్యాలను సాధించడంలో వాటి పాత్రను వివరంగా పరిశీలిస్తుంది.

### ట్రాన్స్పోర్ట్ మెకానిజమ్ అంటే ఏమిటి?

ట్రాన్స్పోర్ట్ మెకానిజమ్ ఒక క్లయింట్ మరియు సర్వర్ మధ్య డేటా ఎటువంటి పద్ధతిలో మార్పిడీ అవుతుందో నిర్వచిస్తుంది. MCP వేర్వేరు వాతావరణాలు మరియు అవసరాలకు అనుగుణంగా అనేక రకాల ట్రాన్స్పోర్ట్‌లను మద్దతు ఇస్తుంది:

- **stdio**: స్టాండర్డ్ ఇన్పుట్/ఆట్పుట్, స్థానిక మరియు CLI ఆధారిత సాధనాలకు అనుకూలంగా ఉంటుంది. సులభం కానీ వెబ్ లేదా క్లౌడ్‌కి అనుకూలం కాదు.
- **SSE (సర్వర్-సెంట్ ఈవెంట్స్)**: HTTP వేదికపై సర్వర్లు క్లయింట్లకు రియల్‌టైమ్ అప్డేట్‌లను పుష్ చేయడానికి అనుమతిస్తుంది. వెబ్ UIల కోసం మంచి, కాని స్కేలబిలిటీ మరియు ఫ్లెక్సిబిలిటీ పరిమితి కలిగినది. MCP స్పెసిఫికేషన్ 2025-06-18 నుంచి స్టాండఅలోన్ SSE (సర్వర్-సెంట్ ఈవెంట్స్) ట్రాన్స్పోర్ట్ డిప్రికేటెడ్ అయి "స్ట్రీమబుల్ HTTP" ట్రాన్స్పోర్ట్‌తో మార్చబడింది.
- **Streamable HTTP**: ఆధునిక HTTP ఆధారిత స్ట్రీమింగ్ ట్రాన్స్పోర్ట్, నోటిఫికేషన్లు మరియు మెరుగైన స్కేలబిలిటీకి మద్దతు ఇస్తుంది. ఎక్కువ భాగం ప్రొడక్షన్ మరియు క్లౌడ్ పరిప్రేక్షలకు సూచించబడింది.

### సరాసరి పట్టిక

ఈ ట్రాన్స్పోర్ట్ మెకానిజమ్‌ల మధ్య తేడాలు అర్థం చేసుకోవడానికి దిగువ సరాసరి పట్టికను పరిశీలించండి:

| ట్రాన్స్పోర్ట్      | రియల్‌టైం అప్‌డేట్స్   | స్ట్రీమింగ్ | స్కేలబిలిటీ | ఉపయోగ సందర్భం          |
|-------------------|------------------|-----------|-------------|-------------------------|
| stdio             | లేదు               | లేదు        | తక్కువ         | స్థానిక CLI సాధనాలు        |
| SSE               | అవును              | అవును       | మధ్యస్థం     | వెబ్, రియల్-టైం అప్‌డేట్స్  |
| Streamable HTTP   | అవును              | అవును       | ఎక్కువ        | క్లౌడ్, బహు-క్లయింట్       |

> **సూచన:** సరైన ట్రాన్స్పోర్ట్ ఎంపిక ప్రదర్శన, స్కేలబిలిటీ, మరియు వినియోగదారు అనుభవాన్ని ప్రభావితం చేస్తుంది. ఆధునిక, స్కేలబుల్ మరియు క్లౌడ్-సన్నిహిత అప్లికేషన్ల కోసం **Streamable HTTP** ని సూచించబడుతుంది.

ముందు అధ్యాయాలలో మీరు చూశిన stdio మరియు SSE ట్రాన్స్పోర్ట్లు గమనించండి మరియు ఈ అధ్యాయం స్ట్రీమబుల్ HTTP ట్రాన్స్పోర్టును వివరించడం గమనించండి.

## స్ట్రీమింగ్: సూత్రాలు మరియు ప్రేరణ

స్ట్రీమింగ్ యొక్క ప్రాథమిక సూత్రాలను మరియు ప్రేరణలను అర్థం చేసుకోవడం సమర్థవంతమైన రియల్‌టైం కమ్యూనికేషన్ సిస్టమ్‌లను అమలు చేయడానికి అవసరం.

**స్ట్రీమింగ్** అనేది నెట్‌వర్క్ ప్రోగ్రామింగ్‌లో ఒక టెక్నిక్, ఇది డేటాను ఒక సమగ్ర ప్రతిస్పందన సిద్ధమయ్యే వరకు ఆలస్యం కాకుండా చిన్ని, నిర్వహించదగిన భాగాలుగా లేదా ఈవెంట్‌ల శ్రేణిగా పంపడం మరియు స్వీకరించడం అనుమతిస్తుంది. ఇది ప్రత్యేకంగా ఉపయోగపడుతుంది:

- పెద్ద ఫైళ్ళు లేదా డేటాసెట్స్.
- రియల్‌టైం అప్‌డేట్స్ (ఉదా: చాట్, ప్రోగ్రెస్ బార్లు).
- యూజర్‌ను అప్డేట్ చేయాలనుకునే దీర్ఘకాలిక గణనలలో.

స్ట్రీమింగ్ గురించి మీకు తెలుసుకోవలసిన విషయాలు ఇక్కడ ఉన్నాయి:

- డేటా క్రమంగా అందించబడుతుంది, ఒకేసారి కాదు.
- క్లయింట్ డేటా అందగానే ప్రాసెస్ చేయవచ్చు.
- గ్రహించబడే ఆలస్యం తగ్గి వినియోగదారు అనుభవం మెరుగుపడుతుంది.

### స్ట్రీమింగ్ ఎందుకు ఉపయోగించాలి?

స్ట్రీమింగ్ వాడేందుకు ఈ కారణాలు ఉన్నాయి:

- వినియోగదారులు తక్షణమే ఫీడ్బ్యాక్ పొందుతారు, కేవలం చివరకి మాత్రమే కాదు.
- రియల్‌టైం అప్లికేషన్లు మరియు స్పందనాత్మక UIల కోసం అనుమతిస్తుంది.
- నెట్‌వర్క్ మరియు కంప్యూట్ వనరుల మరింత సమర్థవంతమైన వినియోగం.

### సులభ ఉదాహరణ: HTTP స్ట్రీమింగ్ సర్వర్ & క్లయింట్

స్ట్రీమింగ్ ఎలా అమలవుతుందో ఒక సులభ ఉదాహరణ ఇక్కడ ఉంది:

#### Python

**సర్వర్ (Python, FastAPI మరియు StreamingResponse ఉపయోగించి):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**క్లయింట్ (Python, requests ఉపయోగించి):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

ఈ ఉదాహరణలో సర్వర్ అందుబాటులో ఉన్న సందేశాలను క్లయింట్‌కు సిరిగా పంపిస్తుంది, అంతేకాక అన్ని సందేశాలు సిద్ధం కావడానికి వేచి ఉండదు.

**ఇది ఎలా పనిచేస్తుంది:**

- సర్వర్ ప్రతి సందేశాన్ని సిద్ధమైనప్పుడు ఇవ్వగా..
- క్లయింట్ అందుకున్న ప్రతీ భాగాన్ని పదించుకుంటుంది.

**అవసరాలు:**

- సర్వర్ స్ట్రీమింగ్ రెస్పాన్స్ (ఉదా: `StreamingResponse` FastAPIలో) ఉపయోగించాలి.
- క్లయింట్ రెస్పాన్స్‌ను స్ట్రీమ్ గా (`stream=True` requestsలో) ప్రాసెస్ చేయాలి.
- Content-Type సాధారణంగా `text/event-stream` లేదా `application/octet-stream` ఉంటుంది.

#### Java

**సర్వర్ (Java, Spring Boot మరియు Server-Sent Events ఉపయోగించి):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**క్లయింట్ (Java, Spring WebFlux WebClient ఉపయోగించి):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**Java అమలులో ముఖ్యాంశాలు:**

- స్ట్రీమింగ్ కోసం Spring Boot యొక్క రియాక్టివ్ స్టాక్ `Flux` ఉపయోగిస్తుంది
- `ServerSentEvent` ఈవెంట్ల ప్ర‌కార సంస్థ‌ల‌నూ ఐవెంట్ల‌ని ప్రదర్శిస్తుంది
- `WebClient` `bodyToFlux()` తో రియాక్టివ్ స్ట్రీమింగ్ వినియోగాన్ని సౌకర్యం చేస్తుంది
- `delayElements()` ఈవెంట్స్ మధ్య ప్రాసెస్ సమయంలో తాళం వేస్తుంది
- ఈవెంట్స్ కి మెరుగైన క్లయింట్ హ్యాండ్లింగ్ కోసం రకాలు (ఉదా: `info`, `result`) ఉంటాయి

### సరాసరి: క్లాసిక్ స్ట్రీమింగ్ vs MCP స్ట్రీమింగ్

"క్లాసిక్" విధానంలో స్ట్రీమింగ్ ఎలా పనిచేస్తుందో MCPలో ఎలా పనిచేస్తుందో తేడాలను ఇలాగే చూడవచ్చు:

| లక్షణం              | క్లాసిక్ HTTP స్ట్రీమింగ్         | MCP స్ట్రీమింగ్ (నోటిఫికేషన్లు)      |
|------------------------|-------------------------------|-------------------------------------|
| ప్రధాన ప్రతిస్పందన      | చంక్‌లుగా (భాగాలుగా)          | ఒక్కటి, చివరలో                      |
| ప్రగతి అప్‌డేట్స్       | డేటా భాగాలుగా పంపబడును       | నోటిఫికేషన్లుగా పంపబడును           |
| క్లయింట్ అవసరాలు      | స్ట్రీమ్‌ను ప్రాసెస్ చేయాలి       | సందేశ హ్యాండర్ అమలు చేసుకోవాలి         |
| ఉపయోగ సందర్భం         | పెద్ద ఫైళ్ళు, AI టోకెన్ స్ట్రీమ్స్ | ప్రగతి, లాగ్స్, రియల్-టైం ఫీడ్బ్యాక్  |

### ముఖ్య తేడాలు గమనించబడ్డాయి

అదనంగా, కొన్ని ప్రధాన తేడాలు ఇవే:

- **కమ్యూనికేషన్ ప్యాటర్న్:**
  - క్లాసిక్ HTTP స్ట్రీమింగ్: సాదాసీద్దా చంక్ ట్రాన్స్ఫర్ ఎంకోడింగ్ డేటాను భాగాలలో పంపుతుంది
  - MCP స్ట్రీమింగ్: JSON-RPC ప్రోటోకాల్‌తో ఒక నిర్మాణమైన నోటిఫికేషన్ సిస్టమ్‌ను ఉపయోగిస్తుంది

- **సందేశ ఫార్మాట్:**
  - క్లాసిక్ HTTP: సాదా టెక్స్ట్ చంక్‌లు, కొత్త లైన్లతో
  - MCP: మెటాడేటాతో నిర్మాణమైన LoggingMessageNotification ఆబ్జెక్టులు

- **క్లయింట్ అమలు:**
  - క్లాసిక్ HTTP: స్ట్రీమింగ్ ప్రతిస్పందనల్ని ప్రాసెస్ చేసే సాదా క్లయింట్
  - MCP: వివిధ రకాల సందేశాలను ప్రాసెస్ చేసే సందేశ హ్యాండర్‌తో మరింత సంక్లిష్టమైన క్లయింట్

- **ప్రగతి అప్‌డేట్స్:**
  - క్లాసిక్ HTTP: ప్రగతి ప్రధాన రెస్పాన్స్ స్ట్రీమ్ భాగం
  - MCP: ప్రగతి వేరే నోటిఫికేషన్ సందేశాలుగా పంపబడుతుంది, ప్రధాన రెస్పాన్స్ చివర వస్తుంది

### సిఫార్సులు

క్లాసిక్ స్ట్రీమింగ్ (మనం పైగా /stream ఉపయోగించి చూపించినట్లు) మరియు MCP సమర్పణ మధ్య ఎంపిక చేయటానికి కొన్ని సిఫార్సులు ఉన్నాయి.

- **సులభమైన స్ట్రీమింగ్ అవసరాలకు:** క్లాసిక్ HTTP స్ట్రీమింగ్ అమలు చేయడంలో సులభం మరియు ప్రాథమిక అవసరాలకు సరిపోతుంది.

- **సంక్లిష్ట, ఇన్‌టరాక్టివ్ అప్లికేషన్ల కోసం:** MCP స్ట్రీమింగ్ అధిక మెటాడేటాతో నిర్మాణాత్మక విధానాన్ని అందిస్తుంది, నోటిఫికేషన్లు మరియు తుది ఫలితాల మధ్య వేరు చెయ్యడాన్ని కలిగి ఉంటుంది.

- **AI అప్లికేషన్ల కోసం:** MCP యొక్క నోటిఫికేషన్ సిస్టమ్ దీర్ఘకాలిక AI పనులలో వాడుకోవడానికి ప్రత్యేకంగా ఉపయోగకరం, ఇక్కడ యూజర్లకు ప్రగతి తెలియజేయాలి.

## MCPలో స్ట్రీమింగ్

మీరు ఇప్పుడు వరకు క్లాసిక్ స్ట్రీమింగ్ మరియు MCP స్ట్రీమింగ్ మధ్య తేడాల గురించి కొన్ని సూచనలు మరియు సరాసరీ గురించి చూశారు. ఇప్పుడు MCPలో మీరు స్ట్రీమింగ్ ను ఎలా ఉపయోగించగలరో వివరంగా తెలుసుకుందాం.

MCP ఫ్రేమ్‌వర్క్‌లో స్ట్రీమింగ్ ఎలా పనిచేస్తుందో అర్థం చేసుకోవడం దీర్ఘకాలిక ఆపరేషన్లలో వినియోగదారులకు రియల్‌టైం ఫీడ్బ్యాక్ అందించడానికి స్పందనాత్మక అప్లికేషన్లు నిర్మించడానికి అవసరం.

MCPలో, స్ట్రీమింగ్ అంటే ప్రధాన ప్రతిస్పందనను భాగాలుగా పంపించడంను కాదు, కానీ టూల్ అభ్యర్థనను ప్రాసెస్ చేస్తుండగా క్లయింట్‌కు **నోటిఫికేషన్లు** పంపడాన్ని సూచిస్తుంది. ఈ నోటిఫికేషన్లు ప్రగతి అప్‌డేట్స్, లాగ్స్ లేదా ఇతర ఈవెంట్‌లను కలిగి ఉండవచ్చు.

### ఇది ఎలా పనిచేస్తుంది

ప్రధాన ఫలితం ఇంకా ఒకే రెస్పాన్స్‌గా పంపబడుతుంది. అయితే ప్రాసెసింగ్ సమయంలో వేరే సందేశాలుగా నోటిఫికేషన్లు పంపండి, తద్వారా క్లయింట్‌కి రియల్‌టైం లో అప్డేట్‌లు ఇవ్వబడతాయి. క్లయింట్ ఈ నోటిఫికేషన్లను హ్యాండిల్ చేసి ప్రదర్శించగలగాలి.

## నోటిఫికేషన్ అంటే ఏమిటి?

"నోటిఫికేషన్" అనగా MCP పరిధిలో ఏమిటి?

నోటిఫికేషన్ అనేది సర్వర్ నుండి క్లయింట్‌కు పంపించే సందేశం, దీర్ఘకాలిక ఆపరేషన్‌లో ప్రగతి, స్థితి లేదా ఇతర ఈవెంట్‌ల గురించి తెలియజేస్తుంది. నోటిఫికేషన్ల ద్వారా పారదర్శకత మరియు వినియోగదారు అనుభవం మెరుగుపడుతుంది.

ఉదాహరణకు, క్లయింట్ సర్వర్‌తో ప్రాథమిక హ్యాండ్షేక్ పూర్తయిన తర్వాత ఒక నోటిఫికేషన్ పంపాలి.

నోటిఫికేషన్ JSON సందేశంగా ఇలా ఉంటుంది:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

నోటిఫికేషన్లు MCPలో ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) అనే టాపిక్‌కు చెందుతాయి.

లాగింగ్ అమలు చేయడానికి, సర్వర్ వద్ద ఫీచర్/సామర్థ్యం రూపంలో దీన్ని యాక్టివేట్ చేయాల్సి ఉంటుంది, ఉదా:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> ఉపయోగించే SDKపై ఆధారపడి, లాగింగ్ డిఫాల్ట్‌గా ఎనేబుల్ అయి ఉండవచ్చు, లేకపోతే సర్వర్ కాన్ఫిగరేషన్‌లో స్పష్టం చేసి ఎనేబుల్ చేయాలి.

వివిధ రకాల నోటిఫికేషన్లు ఉన్నాయి:

| స్థాయి      | వివరణ                         | ఉదాహరణ ఉపయోగం                 |
|-----------|-------------------------------|---------------------------------|
| debug     | వివరమైన డీబగ్ సమాచారం       | ఫంక్షన్ ఎంట్రీ/ఎగ్జిట్ పాయింట్లు |
| info      | సాధారణ ఆపరేషన్ సమాచారం       | ఆపరేషన్ ప్రగతి అప్‌డేట్స్         |
| notice    | సాధారణ కాని ముఖ్యమైన ఈవెంట్లు  | కాన్ఫిగరేషన్ మార్పులు             |
| warning   | హెచ్చరిక పరిస్థితులు             | డిప్రికేటెడ్ ఫీచర్ వినియోగం     |
| error     | తప్పిద పరిస్థితులు               | ఆపరేషన్ విఫలములు               |
| critical  | తీవ్రమైన పరిస్థితులు           | సిస్టమ్ భాగాల విఫలములు          |
| alert     | తక్షణ చర్య తప్పనిసరి           | డేటా పాడుచేయబడినట్లు గుర్తింపు  |
| emergency | సిస్టమ్ ఉపయోగించలేము           | సంపూర్ణ సిస్టమ్ వైఫల్యం          |

## MCPలో నోటిఫికేషన్లను అమలు చేయడం

MCPలో నోటిఫికేషన్లను అమలు చేయడానికి, సర్వర్ మరియు క్లయింట్ రెండు వైపులా రియల్‌టైం అప్‌డేట్స్ నిర్వహించేందుకు ఏర్పాట్లు చేయాలి. దీని వల్ల మీ అప్లికేషన్ దీర్ఘకాలిక ఆపరేషన్లలో తక్షణ ఫీడ్బ్యాక్ ఇవ్వగలదు.

### సర్వర్-వైపు: నోటిఫికేషన్ల పంపడం

సర్వర్ వైపు నుండి ప్రారంభిద్దాం. MCPలో, అభ్యర్థనలు ప్రాసెస్ చేయడానికి బృందాలు నోటిఫికేషన్లు పంపగలవు. సర్వర్ సాందర్భ విషయాన్ని (అధికంగా `ctx`) ఉపయోగించి క్లయింట్‌కు సందేశాలు పంపుతుంది.

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

క్రింద ఉదాహరణలో, `process_files` టూల్ ప్రతి ఫైల్ ప్రాసెస్ సమయంలో మూడు నోటిఫికేషన్లు పంపుతుంది. `ctx.info()` పద్ధతిని సమాచార సందేశాలు పంపడానికి ఉపయోగిస్తారు.

అదనంగా, నోటిఫికేషన్లను ఎనేబుల్ చేయడానికి, సర్వర్ స్ట్రీమింగ్ ట్రాన్స్పోర్ట్ (అలా `streamable-http`) ఉపయోగించాలి మరియు క్లయింట్ సందేశ హ్యాండర్ అమలు చేయాలి. ఇలా సర్వర్‌ను `streamable-http` ట్రాన్స్పోర్ట్ ఉపయోగించడానికి ఎలా సెట్ చెయ్యాలో చూడండి:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

ఈ .NET ఉదాహరణలో, `ProcessFiles` టూల్ `Tool` అట్రిబ్యూట్‌తో గుర్తించబడింది మరియు ప్రతి ఫైల్ ప్రాసెస్‌చేస్తూ మూడు నోటిఫికేషన్లు పంపుతుంది. `ctx.Info()` పద్ధతి సమాచార సందేశాలను పంపడానికి ఉపయోగించబడుతుంది.

.NET MCP సర్వర్‌లో నోటిఫికేషన్లను ఎనేబుల్ చేయడానికి, స్ట్రీమింగ్ ట్రాన్స్పోర్ట్‌ను వాడుతున్నదని నిర్ధారించుకోండి:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### క్లయింట్-వైపు: నోటిఫికేషన్లు స్వీకరించడం

క్లయింట్ ఒక సందేశ హ్యాండర్ అమలు చేయాలి, అది నోటిఫికేషన్లు వచ్చినప్పుడు వాటిని ప్రాసెస్ చేసి ప్రదర్శిస్తుంది.

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

పైన కఱ్ఱ కోడ్‌లో, `message_handler` ఫంక్షన్ వచ్చే సందేశం నోటిఫికేషన్ అయితే అది ప్రింట్ చేస్తుంది; కాకపోతే సాధారణ సర్వర్ సందేశం అని ప్రాసెస్ చేస్తుంది. అలాగే `ClientSession`ని `message_handler`తో ఇనిషియలైజ్ చేయడం గమనించండి.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

ఈ .NET ఉదాహరణలో, `MessageHandler` ఫంక్షన్ వచ్చిన సందేశం నోటిఫికేషన్ అయితే ప్రింట్ చేస్తుంది; లేకపోతే సాధారణ సర్వర్ సందేశంగా ప్రాసెస్ చేస్తుంది. `ClientSession`కు `ClientSessionOptions` ద్వారా సందేశ హ్యాండర్ ఇస్తారు.

నోటిఫికేషన్లను ఎనేబుల్ చేయడానికి, సర్వర్ స్ట్రీమింగ్ ట్రాన్స్పోర్ట్ (ఉదా: `streamable-http`) వాడుతున్నదని మరియు క్లయింట్ సందేశ హ్యాండర్ కలిగి ఉన్నదని నిర్ధారించండి.

## ప్రగతి నోటిఫికేషన్లు & పరిస్థితులు

ఈ విభాగం MCPలో ప్రగతి నోటిఫికేషన్ల సూత్రాన్ని, వాటి ప్రాముఖ్యతను మరియు Streamable HTTP ఉపయోగించి వాటిని ఎలా అమలు చేయాలో వివరిస్తుంది. మీరు మీ అర్థాన్ని బలోపేతం చేసుకోవడానికి ఒక ప్రాక్టికల్ అసైన్‌మెంట్ కూడా ఉంటుంది.

ప్రగతి నోటిఫికేషన్లు సర్వర్ నుండి క్లయింట్‌కు రియల్‌టైమ్ సందేశాలు, దీర్ఘకాలిక ఆపరేషన్ల సమయంలో పంపబడతాయి. మొత్తం ప్రక్రియ పూర్తయ్యేవరకు వేచి ఉండకుండానే సర్వర్ ప్రస్తుత పరిస్థితి గురించి క్లయింట్‌ను అప్డేట్ చేస్తుంది. ఇది పారదర్శకత, వినియోగదారు అనుభవం మెరుగుపడుతుంది మరియు డీబగింగ్ సులభం చేస్తుంది.

**ఉదాహరణ:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### ప్రగతి నోటిఫికేషన్లు ఎందుకు ఉపయోగించాలి?

ప్రగతి నోటిఫికేషన్లు కొన్ని ముఖ్యమైన కారణాల కోసం అవసరం:

- **మంచి వినియోగదారు అనుభవం:** పనులు సాగుతున్నన్ని అప్డేట్లు యూజర్లు చూడగలరు, కేవలం చివరికి కాదు.
- **రియల్‌టైమ్ ఫీడ్బ్యాక్:** క్లయింట్లు ప్రగతి బార్లు లేదా లాగ్స్ ప్రదర్శించగలరు, దాంతో అప్లికేషన్ స్పందనాత్మకం అనిపిస్తుంది.
- **డీబగింగ్ మరియు మానిటరింగ్ సులభత:** డెవలపర్లు మరియు యూజర్లు ఎక్కడ ప్రక్రియ నిద్రిస్తోందో గమనించవచ్చు.

### ప్రగతి నోటిఫికేషన్లను ఎలా అమలు చేయాలి

MCPలో ప్రగతి నోటిఫికేషన్లను ఇలా అమలు చేయవచ్చు:

- **సర్వర్ మీద:** ప్రతీ ఐటెం ప్రాసెస్ అయినప్పుడు `ctx.info()` లేదా `ctx.log()` ద్వారా నోటిఫికేషన్లు పంపండి. ఇది ప్రధాన ఫలితం సిద్ధం కాక ముందే క్లయింట్‌కు సందేశం పంపుతుంది.
- **క్లయింట్ మీద:** నోటిఫికేషన్లు వచ్చినప్పుడు వాటిని వినడానికి మరియు ప్రదర్శించడానికి సందేశ హ్యాండర్‌ను అమలు చేయండి. ఈ హ్యాండర్ నోటిఫికేషన్లు మరియు తుది ఫలితాన్ని తేడా చేయగలగాలి.

**సర్వర్ ఉదాహరణ:**

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**క్లయింట్ ఉదాహరణ:**

#### Python

```python
async def message_handler(message):

    else:
        print("SERVER MESSAGE:", message)
```

## భద్రతా పరిగణనలు

HTTP ఆధారిత ట్రాన్స్పోర్ట్లతో MCP సర్వర్లను అమలు చేయడంలో భద్రత అత్యంత ముఖ్యమైన విషయం, ఇది అనేక దాడి మార్గాలు మరియు రక్షణ విధానాలపై జాగ్రత్త తీసుకోవాలి.

### అవలోకనం

MCP సర్వర్లను HTTP ద్వారా ఎక్స్‌పోజ్ చేయడంలో భద్రత అత్యవసరమే. స్ట్రీమబుల్ HTTP కొత్త దాడి పరోక్షాలను పరిచయం చేస్తుంది మరియు జాగ్రత్తగా కాన్ఫిగర్ చేయాల్సి ఉంటుంది.

### ముఖ్య పాయింట్లు


- **Origin Header నిర్ధారణ**: DNS రీబైన్డింగ్ దాడులని నివారించడానికి ఎప్పుడూ `Origin` హెడర్‌ను నిర్ధారించండి.
- **లొకల్‌హోస్ట్ బైండింగ్**: లోకల్ డెవలప్‌మెంట్ కోసం, సర్వర్లను ప్రజా ఇంటర్నెట్‌కి ప exposição కాకుండా `localhost`కు బైండ్ చేయండి.
- **ప్రమాణీకరణ**: ఉత్పత్తి వినియోగాల కోసం ప్రమాణీకరణ (ఉదా: API కీలు, OAuth) ను అమలు చేయండి.
- **CORS**: యాక్సెస్ పరిమితం చేయడానికి Cross-Origin Resource Sharing (CORS) విధానాలను కాన్ఫిగర్ చేయండి.
- **HTTPS**: ట్రాఫిక్ కుడగించడానికి ఉత్పత్తిలో HTTPS ఉపయోగించండి.

### ఉత్తమ ఆచారాలు

- ధ్రువీకరించకుండా వచ్చిన అభ్యర్థనలపై ఎప్పుడూ నమ్మకం పెట్టకండి.
- అన్ని యాక్సెస్ మరియు పొరపాట్లను లాగ్ చేసి పర్యవేక్షించండి.
- పునరావృతంగా వంశాన్ని సురక్షితంగా ఉంచడానికి ఆధారాలను నవీకరించండి.

### సవాళ్లు

- అభివృద్ధి సౌలభ్యంతో భద్రత సమతౌల్యం చేయడం
- వివిధ క్లయింట్ పరిసరాలకు అనుగుణంగా ఉండటం నిర్ధారించడం

## SSE నుండి Streamable HTTP కి అప్‌గ్రేడ్ చేయడం

ప్రస్తుతం Server-Sent Events (SSE) ఉపయోగిస్తున్న అనువర్తనాల కోసం, Streamable HTTPకి మారడం మీ MCP అమలులకు మెరుగైన సామర్థ్యం మరియు దీర్ఘకాలిక స్థిరత్వాన్ని అందిస్తుంది.

### ఎందుకు అప్‌గ్రేడ్ చేయాలి?

SSE నుండి Streamable HTTP కి అప్‌గ్రేడ్ చేయడానికి రెండు ప్రబలమైన కారణాలు ఉన్నాయి:

- Streamable HTTP SSE కంటే మెరుగైన స్కేలబిలిటీ, అనుగుణత, మరియు సమృద్ధి నోటిఫికేషన్ మద్దతును అందిస్తుంది.
- ఇది కొత్త MCP అనువర్తనాల కోసం సిఫార్సు చేసిన రవాణా మాధ్యమం.

### మార్పిడి దశలు

మీ MCP అనువర్తనాలలో SSE నుండి Streamable HTTP కి ఎలా మార్పిడి చేయాలో ఇక్కడ ఉంది:

- **సర్వర్ కోడ్‌ను నవీకరించండి** `mcp.run()`లో `transport="streamable-http"` ఉపయోగించడానికి.
- **క్లయింట్ కోడ్‌ను నవీకరించండి** SSE క్లయింట్ స్థానంలో `streamablehttp_client` ఉపయోగించడానికి.
- **క్లయింట్‌లో ఒక సందేశ హ్యాండ్లర్‌ను అమలు చేయండి** నోటిఫికేషన్లను ప్రాసెస్ చేయడానికి.
- **ఉపయోగంలో ఉన్న సాధనాలు మరియు వర్క్‌ఫ్లోలతో అనుగుణత కోసం పరీక్షించండి**.

### అనుగుణతను నిర్వహించడం

మార్పిడి సమయంలో ప్రస్తుత SSE క్లయింట్లతో అనుగుణతను నిలుపుకొనడం సిఫార్సు చేయబడుతుంది. కొన్ని వ్యూహాలు ఇక్కడ ఉన్నాయి:

- మీరు రెండు రవాణాలు విడి ఎండ్‌పాయింట్లలో నడుపుతూ SSE మరియు Streamable HTTP రెండింటినీ మద్దతు ఇవ్వవచ్చు.
- క్లయింట్లను నెమ్మదిగా కొత్త రవాణాకు మార్చండి.

### సవాళ్లు

మార్పిడి సమయంలో ఈ క్రింది సవాళ్లను పరిష్కరించాలి:

- అన్ని క్లయింట్లు నవీకరించబడటం నిర్ధారించడం
- నోటిఫికేషన్ డెలివరీలో తేడాలను నిర్వహించడం

## భద్రతా పరిగణనల

ఏ సర్వర్‌ను అమలు చేసినా భద్రత మొదటి ప్రాధాన్యతగా ఉండాలి, ముఖ్యంగా MCPలో Streamable HTTP వంటి HTTP-ఆధారిత రవాణాలను ఉపయోగించినప్పుడు.

MCP సర్వర్లను HTTP-ఆధారిత రవాణాలతో అమలు చేసినప్పుడు, భద్రత అనేది అనేక దాడి వధుకు మరియు రక్షణ యంత్రాంగాలకు జాగ్రత్తపూర్వక శ్రద్ధ అవసరమైన ప్రధాన సంబంధాంశం అవుతుంది.

### అవలోకనం

MCP సర్వర్లను HTTP ద్వారా ప్రదర్శిస్తున్నప్పుడు భద్రత అత్యంత ముఖ్యమైనది. Streamable HTTP కొత్త దాడి ఉపరితలాలను పరిచయం చేస్తుంది మరియు జాగ్రత్తగా కాన్ఫిగర్ చేయాల్సి ఉంటుంది.

కొన్ని ముఖ్యమైన భద్రతా పరిగణనలు ఇక్కడ ఉన్నాయి:

- **Origin Header నిర్ధారణ**: DNS రీబైన్డింగ్ దాడులని నివారించడానికి ఎప్పుడూ `Origin` హెడర్‌ను నిర్ధారించండి.
- **లొకల్‌హోస్ట్ బైండింగ్**: లోకల్ డెవలప్‌మెంట్ కోసం, సర్వర్లను ప్రజా ఇంటర్నెట్‌కి ప exposición కాకుండా `localhost`కు బైండ్ చేయండి.
- **ప్రమాణీకరణ**: ఉత్పత్తి వినియోగాల కోసం ప్రమాణీకరణ (ఉదా: API కీలు, OAuth) ను అమలు చేయండి.
- **CORS**: యాక్సెస్ పరిమితం చేయడానికి Cross-Origin Resource Sharing (CORS) విధానాలను కాన్ఫిగర్ చేయండి.
- **HTTPS**: ట్రాఫిక్ కుడగించడానికి ఉత్పత్తిలో HTTPS ఉపయోగించండి.

### ఉత్తమ ఆచారాలు

అదనంగా, మీ MCP స్ట్రీమింగ్ సర్వర్‌లో భద్రతను అమలు చేసే సమయంలో పాటించవలసిన కొన్ని ఉత్తమ ఆచారాలు ఈ క్రింది విధంగా ఉన్నాయి:

- ధ్రువీకరించకుండా వచ్చిన అభ్యర్థనలపై ఎప్పుడూ నమ్మకం పెట్టకండి.
- అన్ని యాక్సెస్ మరియు పొరపాట్లను లాగ్ చేసి పర్యవేక్షించండి.
- పునరావృతంగా వంశాన్ని సురక్షితంగా ఉంచడానికి ఆధారాలను నవీకరించండి.

### సవాళ్లు

MCP స్ట్రీమింగ్ సర్వర్‌లలో భద్రతను అమలు చేసినప్పుడు మీరు కొన్ని సవాళ్లను ఎదుర్కొంటారు:

- అభివృద్ధి సౌలభ్యంతో భద్రత సమతౌల్యం చేయడం
- వివిధ క్లయింట్ పరిసరాలకు అనుగుణంగా ఉండటం నిర్ధారించడం

### పనిది: మీ స్వంత స్ట్రీమింగ్ MCP యాప్ నిర్మించండి

**దృశ్యం:**
ఒక MCP సర్వర్ మరియు క్లయింట్‌ను నిర్మించండి, అందులో సర్వర్ ఒక అంశాలకు (ఉదా: ఫైళ్ల లేదా పత్రాల) జాబితాను ప్రాసెస్ చేస్తుంది మరియు ప్రతి అంశం ప్రాసెస్ అయినప్పుడు ఒక నోటిఫికేషన్ పంపుతుంది. క్లయింట్ వచ్చినప్పుడు ప్రతి నోటిఫికేషన్‌ను ప్రదర్శించాలి.

**దశలు:**

1. ఒక సర్వర్ సాధనాన్ని అమలు చేయండి, ఇది జాబితాను ప్రాసెస్ చేసి ప్రతి అంశానికి నోటిఫికేషన్లను పంపుతుంది.
2. ఒక క్లయింట్‌ను అమలు చేయండి, సందేశ హ్యాండ్లర్‌తో నోటిఫికేషన్లను ప్రత్యక్షంగా ప్రదర్శించడానికి.
3. రెండు సర్వర్ మరియు క్లయింట్ నడిపించి, నోటిఫికేషన్లను గమనించి మీ అమలును పరీక్షించండి.
