# HTTPS стримовање са протоколом контекста модела (MCP)

Ово поглавље пружа свеобухватан водич за имплементацију безбедног, скалабилног и приказа у реалном времену помоћу протокола контекста модела (MCP) користећи HTTPS. Обухвата мотивацију за стримовање, расположиве механизме транспорта, како имплементирати стримабилни HTTP у MCP, најбоље безбедносне праксе, миграцију са SSE и практичне смернице за изградњу сопствених стриминг MCP апликација.

> **Унапред:** ова лекција описује стримабилни HTTP под **MCP спецификацијом 2025-11-25**, где се сесија успоставља током инициализације и фиксира заглављем `Mcp-Session-Id`. Кандидат за издање из `2026-07-28` у потпуности уклања руковање и ID сесије, чинећи сваки захтев самоодрживим и усмерљивим на било који сервер без потребе за лепљивим сесијама. Погледајте [Шта се мења у MCP: Кандидат за издање 2026-07-28](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) за детаље.

## Механизми транспорта и стримовање у MCP

Овај одељак истражује различите механизме транспорта доступне у MCP и њихову улогу у омогућавању функција стримовања за комуникацију у реалном времену између клијената и сервера.

### Шта је механизам транспорта?

Механизам транспорта дефинише како се подаци размењују између клијента и сервера. MCP подржава више типова транспорта прилагођених различитим окружењима и захтевима:

- **stdio**: Стандардни улаз/излаз, погодан за локалне и алате засноване на командној линији. Једноставан али није прикладан за веб или облак.
- **SSE (Server-Sent Events)**: Омогућава серверима да шаљу ажурирања у реалном времену клијентима преко HTTP. Добро за веб интерфејсе, али ограничено у скалабилности и флексибилности. Од MCP спецификације 2025-06-18, самостални SSE транспорт је деприкативан и замењен "Streamable HTTP" транспортом.
- **Streamable HTTP**: Модеран HTTP заснован транспорт за стримовање, који подржава нотификације и бољу скалабилност. Препоручен за већину производних и облачних сценарија.

### Табела поређења

Погледајте табелу поређења испод да бисте разумели разлике између ових механизама транспорта:

| Транспорт        | Ажурирања у реалном времену | Стримовање | Скалабилност | Случај употребе         |
|------------------|------------------------------|------------|--------------|------------------------|
| stdio            | Не                           | Не         | Ниска        | Локални CLI алати      |
| SSE              | Да                           | Да         | Средња       | Веб, ажурирања уживо   |
| Streamable HTTP  | Да                           | Да         | Висока       | Облак, мулти-клијент   |

> **Савет:** Избор правог транспорта утиче на перформансе, скалабилност и корисничко искуство. **Streamable HTTP** се препоручује за модерне, скалабилне и облачно спремне апликације.

Обратите пажњу на транспортне механизме stdio и SSE које сте видели у претходним поглављима и како је стримабилни HTTP транспорт покривен у овом поглављу.

## Стримовање: концепти и мотивација

Разумевање основних концепата и мотивација иза стримовања је кључно за имплементацију ефикасних система комуникације у реалном времену.

**Стримовање** је техника у мрежном програмирању која омогућава слање и пријем података у малим, управљивим деловима или као низ догађаја, уместо да се чека цео одговор да буде спреман. Ово је посебно корисно за:

- Велике датотеке или скупове података.
- Ажурирања у реалном времену (нпр. чет, траке напретка).
- Дуготрајне израчуне где желите да корисник буде информисан.

Ево шта треба да знате о стримовању на високом нивоу:

- Подаци се испоручују постепено, не одједном.
- Клијент може да обрађује податке како стижу.
- Смањује перципирану латенцију и побољшава корисничко искуство.

### Зашто користити стримовање?

Разлози за коришћење стримовања су следећи:

- Корисници добијају повратну информацију одмах, не само на крају.
- Омогућава апликације у реалном времену и одзивне корисничке интерфејсе.
- Ефикаснија употреба мрежних и рачунарских ресурса.

### Једноставан пример: HTTP стриминг сервер и клијент

Ево једноставног примера како се стримовање може имплементирати:

#### Python

**Сервер (Python, користећи FastAPI и StreamingResponse):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**Клијент (Python, користећи requests):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

Овај пример демонстрира сервер који шаље низ порука клијенту како постану доступне, уместо да чека све поруке да буду спремне.

**Како ради:**

- Сервер шаље сваку поруку како постане спремна.
- Клијент прима и исписује сваки део како пристиже.

**Захтеви:**

- Сервер мора користити стриминг одговор (нпр. `StreamingResponse` у FastAPI).
- Клијент мора обрађивати одговор као стрим (`stream=True` у requests).
- Content-Type је обично `text/event-stream` или `application/octet-stream`.

#### Java

**Сервер (Java, користећи Spring Boot и Server-Sent Events):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**Клијент (Java, користећи Spring WebFlux WebClient):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**Напомене о Јава имплементацији:**

- Користи реактивни стек Spring Boot-а са `Flux` за стримовање
- `ServerSentEvent` пружа структурирано стримовање догађаја са типовима догађаја
- `WebClient` са `bodyToFlux()` омогућава реактивну потрошњу стримова
- `delayElements()` симулира време обраде између догађаја
- Догађаји могу имати типове (`info`, `result`) за бољу обраду на клијенту

### Поређење: Класично стримовање против MCP стримовања

Разлике између класичног начина стримовања и како то функционише у MCP могу се приказати овако:

| Карактеристика            | Класично HTTP стримовање       | MCP стримовање (нотификације)   |
|--------------------------|-------------------------------|---------------------------------|
| Главни одговор           | Подељен у делове               | Један, на крају                 |
| Ажурирања о напретку     | Шаљу се као делови података   | Шаљу се као нотификације        |
| Захтеви клијента          | Клијент мора обрадити стрим    | Клијент мора имати обрађивач порука |
| Случај употребе          | Велике датотеке, AI токени     | Напредак, дневници, повратне информације у реалном времену |

### Кључне уочене разлике

Поред тога, ево неких кључних разлика:

- **Обрасци комуникације:**
  - Класично HTTP стримовање: Користи једноставно преслање у деловима за слање података у комадима
  - MCP стримовање: Користи структурирани систем нотификација са JSON-RPC протоколом

- **Формат поруке:**
  - Класично HTTP: Плаин текст делови са новим редовима
  - MCP: Структуриране LoggingMessageNotification објекте са метаподацима

- **Имплементација клијента:**
  - Класично HTTP: Једноставан клијент који обрађује стриминг одговоре
  - MCP: Сложенији клијент са обрађивачем порука за обраду различитих типова порука

- **Ажурирања напретка:**
  - Класично HTTP: Напредак је део главног одговора струје
  - MCP: Напредак се шаље као посебне нотификационе поруке док главни резултат долази на крају

### Препоруке

Постоје неке ствари које препоручујемо приликом избора између класичног стримовања (као крајње тачке приказане горе коришћењем `/stream`) и стримовања преко MCP.

- **За једноставне потребе стримовања:** Класично HTTP стримовање је лакше за имплементацију и довољно за основне потребе стримовања.

- **За сложене, интерактивне апликације:** MCP стримовање пружа структуриранији приступ са богатијим метаподацима и раздвајањем нотификација и коначних резултата.

- **За AI апликације:** MCP систем нотификација је посебно користан за дуготрајне AI задатке где желите да корисници буду информисани о напретку.

## Стримовање у MCP

Добро, дакле видели сте неке препоруке и поређења до сада о разликама између класичног стримовања и стримовања у MCP. Хајде да се детаљно позабавимо како можете искористити стримовање у MCP.

Разумевање како стримовање ради у оквиру MCP оквира је неопходно за изградњу одзивних апликација које пружају повратне информације у реалном времену током дуготрајних операција.

У MCP, стримовање није слање главног одговора у деловима, већ слање **нотификација** клијенту док алат обрађује захтев. Ове нотификације могу укључивати ажурирања напретка, дневнике или друге догађаје.

### Како то функционише

Главни резултат се и даље шаље као један одговор. Међутим, нотификације се могу слати као посебне поруке током обраде и тиме ажурирати клијента у реалном времену. Клијент мора бити у стању да рукује и приказује те нотификације.

## Шта је нотификација?

Рекли смо „нотификација“, шта то значи у контексту MCP?

Нотификација је порука послата са сервера ка клијенту да информише о напретку, статусу или другим догађајима током дуготрајне операције. Нотификације побољшавају транспарентност и корисничко искуство.

На пример, клијент треба да пошаље нотификацију када је иницијални руковање са сервером обављено.

Нотификација изгледа овако као JSON порука:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

Нотификације припадају теми у MCP познатој као ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging).

Да би логовање радило, сервер треба да га омогући као функцију/способност на следећи начин:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> У зависности од коришћеног SDK-а, логовање може бити подразумевано омогућено, или ћете можда морати експлицитно да га укључите у конфигурацији сервера.

Постоје различити типови нотификација:

| Ниво      | Опис                         | Пример употребе              |
|-----------|------------------------------|-----------------------------|
| debug     | Детаљне информације за дебаговање  | Тачке уласка/изласка функције |
| info      | Опште информативне поруке    | Ажурирања напретка операције|
| notice    | Нормални али значајни догађаји | Промене конфигурације        |
| warning   | Упозорења                   | Коришћење застареле функције |
| error     | Услови грешке               | Неуспеси операција           |
| critical  | Критични услови             | Неуспеси система             |
| alert     | Мере морају бити предузете одмах | Откривена корупција података |
| emergency | Систем је неупотребљив     | Потпуни отказ система        |

## Имплементација нотификација у MCP

Да бисте имплементирали нотификације у MCP, потребно је да подесите како серверску тако и клијентску страну да рукују ажурирањима у реалном времену. Ово омогућава вашој апликацији да пружа тренутне повратне информације корисницима током дуготрајних операција.

### Серверска страна: Слање нотификација

Почнимо са серверском страном. У MCP, дефинишете алате који могу слати нотификације током обраде захтева. Сервер користи контекст објекат (обично `ctx`) за слање порука ка клијенту.

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

У претходном примеру, алат `process_files` шаље три нотификације клијенту док обрађује сваки фајл. Метод `ctx.info()` се користи за слање информативних порука.

Поред тога, да бисте омогућили нотификације, уверите се да ваш сервер користи транспорт за стримовање (као `streamable-http`) и да ваш клијент имплементира обрађивач порука за обраду нотификација. Ево како можете подесити сервер да користи `streamable-http` транспорт:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

У овом .NET примеру, алат `ProcessFiles` је означен атрибутом `Tool` и шаље три нотификације клијенту док обрађује сваки фајл. Метод `ctx.Info()` се користи за слање информативних порука.

Да бисте омогућили нотификације у вашем .NET MCP серверу, уверите се да користите транспорт за стримовање:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### Клијентска страна: Примaње нотификација

Клијент мора имплементирати обрађивач порука за обраду и приказивање нотификација како пристижу.

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

У претходном коду, функција `message_handler` проверава да ли је долазна порука нотификација. Ако јесте, исписује нотификацију; у супротном, обрађује је као редовну поруку сервера. Такође обратите пажњу како је `ClientSession` иницијализована са `message_handler` за руковање долазним нотификацијама.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

У овом .NET примеру, функција `MessageHandler` проверава да ли је долазна порука нотификација. Ако јесте, исписује нотификацију; у супротном, обрађује је као редовну поруку сервера. `ClientSession` је иницијализована са обрађивачем порука преко `ClientSessionOptions`.

Да бисте омогућили нотификације, уверите се да ваш сервер користи транспорт за стримовање (као `streamable-http`) и да ваш клијент имплементира обрађивач порука за обраду нотификација.

## Нотификације напретка & сценарији

Овај одељак објашњава концепт нотификација напретка у MCP, зашто су битне и како их имплементирати користећи Streamable HTTP. Такође ћете пронаћи практичну вежбу за учвршћивање разумевања.

Нотификације напретка су поруке у реалном времену послате са сервера ка клијенту током дуготрајних операција. Уместо да се чека да цео процес заврши, сервер стално ажурира клијента о тренутном стању. Ово побољшава транспарентност, корисничко искуство и олакшава отклањање грешака.

**Пример:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### Зашто користити нотификације напретка?

Нотификације напретка су кључне из више разлога:

- **Боље корисничко искуство:** Корисници виде ажурирања како рад напредује, а не само на крају.
- **Повратне информације у реалном времену:** Клијенти могу приказивати траке напретка или дневнике, чинећи апликацију реактивнијом.
- **Лакоћа отклањања грешака и праћење:** Развојни инжењери и корисници могу видети где процес можда успорава или је заглављен.

### Како имплементирати нотификације напретка

Ево како можете имплементирати нотификације напретка у MCP:

- **На серверу:** Користите `ctx.info()` или `ctx.log()` да пошаљете нотификације како свака ставка буде обрађена. Ово шаље поруку клијенту пре него што главни резултат буде спреман.
- **На клијенту:** Имплементирајте обрађивач порука који слуша и приказује нотификације како пристижу. Овај обрађивач разликује нотификације од коначног резултата.

**Пример сервера:**

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**Пример клијента:**

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## Безбедносне разматрања

При имплементацији MCP сервера са HTTP базираним транспортима, безбедност постаје пресудна тема која захтева пажљиво бављење разним нападима и механизмима заштите.

### Преглед

Безбедност је критична када се MCP сервери излажу преко HTTP. Streamable HTTP уводи нове површине напада и захтева пажљиву конфигурацију.

### Кључне тачке


- **Валидација Origin заглавља**: Увек верификујте `Origin` заглавље да бисте спречили DNS ребајндинг нападе.
- **Везивање за localhost**: За локални развој, повежите сервере на `localhost` како бисте избегли излагање јавном интернету.
- **Аутентикација**: Имплементирајте аутентикацију (нпр. API кључеве, OAuth) за продукцијске имплементације.
- **CORS**: Конфигуришите политике Cross-Origin Resource Sharing (CORS) да бисте ограничили приступ.
- **HTTPS**: Користите HTTPS у продукцији ради шифровања саобраћаја.

### Најбоље праксе

- Никада не верујте долазним захтевима без валидације.
- Логовање и праћење свих приступа и грешака.
- Редовно ажурирајте зависности да бисте исправили безбедносне рањивости.

### Изазови

- Балансирање безбедности са лакоћом развоја
- Обезбеђивање компатибилности са разним клијентским окружењима

## Надоградња са SSE на Streamable HTTP

За апликације које тренутно користе Server-Sent Events (SSE), миграција на Streamable HTTP пружа побољшане могућности и бољу дугорочну одрживост за ваше MCP имплементације.

### Зашто надоградити?

Постоје два убедљива разлога за надоградњу са SSE на Streamable HTTP:

- Streamable HTTP нуди бољу скалабилност, компатибилност и богатију подршку за обавештења у односу на SSE.
- То је препоручени транспорт за нове MCP апликације.

### Кораци миграције

Ево како можете мигрирати са SSE на Streamable HTTP у вашим MCP апликацијама:

- **Ажурирајте серверски код** да користи `transport="streamable-http"` у `mcp.run()`.
- **Ажурирајте клијентски код** да користи `streamablehttp_client` уместо SSE клијента.
- **Имплементирајте обраду порука** у клијенту за процесирање обавештења.
- **Тестирајте компатибилност** са постојећим алатима и токовима рада.

### Одржавање компатибилности

Препоручује се одржавање компатибилности са постојећим SSE клијентима током процеса миграције. Ево неких стратегија:

- Можете подржати и SSE и Streamable HTTP тако што ћете покренути оба транспорта на различитим крајњим тачкама.
- Постепено мигрирајте клијенте на нови транспорт.

### Изазови

Обратите пажњу на следеће изазове током миграције:

- Осигуравање да су сви клијенти ажурирани
- Разлике у испоруци обавештења

## Безбедносне препоруке

Безбедност треба да буде врхунски приоритет при имплементацији било ког сервера, посебно када користите HTTP базиране протоколе као што је Streamable HTTP у MCP-у.

При имплементацији MCP сервера са HTTP транспортима, безбедност постаје критично питање које захтева пажњу према различитим видовима напада и механизмима заштите.

### Преглед

Безбедност је кључна када се MCP сервери излажу преко HTTP-а. Streamable HTTP уводи нове ризике напада и захтева пажљиву конфигурацију.

Ево неких кључних безбедносних разматрања:

- **Валидација Origin заглавља**: Увек верификујте `Origin` заглавље да бисте спречили DNS ребајндинг нападе.
- **Везивање за localhost**: За локални развој, повежите сервере на `localhost` како бисте избегли излагање јавном интернету.
- **Аутентикација**: Имплементирајте аутентикацију (нпр. API кључеве, OAuth) за продукцијске имплементације.
- **CORS**: Конфигуришите политике Cross-Origin Resource Sharing (CORS) да бисте ограничили приступ.
- **HTTPS**: Користите HTTPS у продукцији ради шифровања саобраћаја.

### Најбоље праксе

Поред тога, ево неких најбољих пракси које треба пратити приликом имплементације безбедности у вашем MCP стриминг серверу:

- Никада не верујте долазним захтевима без валидације.
- Логовање и праћење свих приступа и грешака.
- Редовно ажурирајте зависности да бисте исправили безбедносне рањивости.

### Изазови

Суочићете се са неким изазовима приликом имплементације безбедности у MCP стриминг серверима:

- Балансирање безбедности са лакоћом развоја
- Обезбеђивање компатибилности са разним клијентским окружењима

### Задатак: Направите своју MCP апликацију за стримовање

**Сценарио:**
Направите MCP сервер и клијента где сервер обрађује листу ставки (нпр. фајлове или документе) и шаље обавештење за сваку обрађену ставку. Клијент треба да приказује сваки долазни сигнал.

**Кораци:**

1. Имплементирајте серверски алат који обрађује листу и шаље обавештења за сваку ставку.
2. Имплементирајте клијента са обрадом порука за приказивање обавештења у реалном времену.
