# پخش جریانی HTTPS با پروتکل زمینه مدل (MCP)

این فصل راهنمای جامعی برای پیاده‌سازی پخش جریانی امن، مقیاس‌پذیر و بلادرنگ با استفاده از پروتکل زمینه مدل (MCP) با HTTPS ارائه می‌دهد. این فصل انگیزه پخش جریانی، مکانیزم‌های انتقال موجود، نحوه پیاده‌سازی HTTP قابل پخش در MCP، بهترین شیوه‌های امنیتی، مهاجرت از SSE و راهنمای عملی برای ساخت برنامه‌های MCP پخش جریانی خودتان را پوشش می‌دهد.

> **نگاه به آینده:** این درس HTTP قابل پخش را تحت **مشخصات MCP 2025-11-25** توصیف می‌کند، جایی که جلسه‌ای در هنگام `initialize` ایجاد شده و با هدر `Mcp-Session-Id` ثابت می‌شود. کاندید انتشار `2026-07-28` کل فرآیند دست‌دهی و شناسه جلسه را حذف می‌کند و هر درخواست را مستقل و قابل مسیریابی به هر نمونه سرور بدون جلسات چسبنده می‌سازد. برای جزئیات بیشتر رجوع کنید به [چه چیزهایی در MCP تغییر می‌کند: کاندید انتشار 2026-07-28](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md).

## مکانیزم‌های انتقال و پخش جریانی در MCP

این بخش به بررسی مکانیزم‌های انتقال مختلف موجود در MCP و نقش آن‌ها در فعال‌سازی قابلیت‌های پخش جریانی برای ارتباط بلادرنگ بین کلاینت‌ها و سرورها می‌پردازد.

### مکانیزم انتقال چیست؟

مکانیزم انتقال تعیین می‌کند چگونه داده بین کلاینت و سرور مبادله می‌شود. MCP از چندین نوع انتقال برای تطبیق با محیط‌ها و نیازهای مختلف پشتیبانی می‌کند:

- **stdio**: ورودی/خروجی استاندارد، مناسب برای ابزارهای محلی و خط فرمان. ساده ولی برای وب یا فضای ابری مناسب نیست.
- **SSE (رویدادهای ارسال شده توسط سرور)**: به سرورها اجازه می‌دهد به‌روزرسانی‌های بلادرنگ را از طریق HTTP به کلاینت ارسال کنند. برای رابط‌های وب خوب است، اما از نظر مقیاس‌پذیری و انعطاف‌پذیری محدود است. از مشخصات MCP 2025-06-18، انتقال SSE مستقل منسوخ شده و جایگزین آن "HTTP قابل پخش" شده است.
- **HTTP قابل پخش**: انتقال جدید مبتنی بر HTTP برای پخش جریانی، با پشتیبانی از اعلان‌ها و مقیاس‌پذیری بهتر. برای اکثر سناریوهای تولید و ابری توصیه می‌شود.

### جدول مقایسه

برای درک تفاوت‌های این مکانیزم‌های انتقال، جدول زیر را ببینید:

| مکانیزم انتقال       | به‌روزرسانی‌های بلادرنگ | پخش جریانی | مقیاس‌پذیری | مورد استفاده              |
|---------------------|--------------------------|-------------|-------------|---------------------------|
| stdio               | خیر                      | خیر         | پایین       | ابزارهای محلی خط فرمان   |
| SSE                 | بله                      | بله         | متوسط       | وِب، به‌روزرسانی‌های بلادرنگ |
| HTTP قابل پخش       | بله                      | بله         | بالا        | ابر، چند کلاینت           |

> **نکته:** انتخاب مکانیزم انتقال مناسب عملکرد، مقیاس‌پذیری و تجربه کاربری را تحت تأثیر قرار می‌دهد. **HTTP قابل پخش** برای برنامه‌های مدرن، مقیاس‌پذیر و آماده ابر توصیه می‌شود.

توجه کنید به مکانیزم‌های انتقال stdio و SSE که در فصول قبلی نشان داده شدند و HTTP قابل پخش که موضوع این فصل است.

## پخش جریانی: مفاهیم و انگیزه

درک مفاهیم بنیادی و انگیزه‌های پشت پخش جریانی برای پیاده‌سازی سیستم‌های موثر ارتباط بلادرنگ ضروری است.

**پخش جریانی** تکنیکی در برنامه‌نویسی شبکه است که امکان ارسال و دریافت داده‌ها را به صورت قطعات کوچک و قابل مدیریت یا به شکل دنباله‌ای از رویدادها فراهم می‌کند و نیازی نیست منتظر آماده شدن کل پاسخ باشید. این مخصوصاً برای موارد زیر مفید است:

- فایل‌ها یا داده‌های حجیم.
- به‌روزرسانی‌های بلادرنگ (مثلاً چت، نوار پیشرفت).
- محاسبات طولانی که می‌خواهید کاربر را در جریان نگه دارید.

نکات کلیدی درباره پخش جریانی به صورت خلاصه:

- داده به صورت تدریجی ارسال می‌شود، نه همه یکجا.
- کلاینت می‌تواند داده را به محض دریافت پردازش کند.
- تأخیر محسوس را کاهش داده و تجربه کاربری را بهبود می‌بخشد.

### چرا از پخش جریانی استفاده کنیم؟

دلایل استفاده از پخش جریانی عبارتند از:

- کاربران بلافاصله بازخورد دریافت می‌کنند، نه فقط در پایان.
- امکان‌پذیری برنامه‌های بلادرنگ و رابط‌های کاربری پاسخگو.
- استفاده بهینه‌تر از منابع شبکه و محاسبات.

### مثال ساده: سرور و کلاینت پخش جریانی HTTP

در اینجا یک مثال ساده از پیاده‌سازی پخش جریانی آورده شده است:

#### پایتون

**سرور (پایتون، با استفاده از FastAPI و StreamingResponse):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**کلاینت (پایتون، با استفاده از requests):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

این مثال نشان می‌دهد چگونه سرور پیام‌هایی را به محض آماده شدن برای کلاینت ارسال می‌کند، به جای آنکه برای آماده شدن همه پیام‌ها منتظر بماند.

**نحوه عملکرد:**

- سرور هر پیام را به محض آماده شدن ارسال می‌کند.
- کلاینت هر قسمت را هنگام دریافت دریافت و چاپ می‌کند.

**نیازمندی‌ها:**

- سرور باید از پاسخ پخش جریانی استفاده کند (مثلاً `StreamingResponse` در FastAPI).
- کلاینت باید پاسخ را به صورت جریان پردازش کند (`stream=True` در requests).
- نوع محتوا معمولا `text/event-stream` یا `application/octet-stream` است.

#### جاوا

**سرور (جاوا، با استفاده از Spring Boot و رویدادهای ارسال شده توسط سرور):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**کلاینت (جاوا، با استفاده از Spring WebFlux WebClient):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**نکات پیاده‌سازی جاوا:**

- استفاده از استک واکنشی Spring Boot با `Flux` برای پخش جریانی
- `ServerSentEvent` پخش رویدادهای ساختارمند با انواع رویداد را ارائه می‌دهد
- `WebClient` با `bodyToFlux()` مصرف پخش جریانی واکنشی را فراهم می‌کند
- `delayElements()` شبیه‌سازی زمان پردازش بین رویدادها
- رویدادها می‌توانند انواعی مثل (`info`, `result`) برای مدیریت بهتر کلاینت داشته باشند

### مقایسه: پخش جریانی کلاسیک در مقابل پخش جریانی MCP

تفاوت‌های نحوه عملکرد پخش جریانی به صورت "کلاسیک" در برابر پخش جریانی MCP را می‌توان به شرح زیر نشان داد:

| ویژگی                  | پخش جریانی HTTP کلاسیک       | پخش جریانی MCP (اعلان‌ها)      |
|------------------------|------------------------------|--------------------------------|
| پاسخ اصلی              | قطعه‌قطعه                    | تک‌پارچه، در پایان             |
| به‌روزرسانی پیشرفت     | به صورت قطعات داده ارسال می‌شود | به صورت اعلان ارسال می‌شود     |
| نیازمندی‌های کلاینت    | باید جریان را پردازش کند    | باید هندلر پیام پیاده‌سازی کند  |
| مورد استفاده           | فایل‌های بزرگ، جریان توکن‌های هوش مصنوعی | پیشرفت، لاگ‌ها، بازخورد بلادرنگ |

### تفاوت‌های کلیدی مشاهده شده

علاوه بر این، تفاوت‌های کلیدی زیر را داریم:

- **الگوی ارتباط:**
  - پخش جریانی HTTP کلاسیک: از کدینگ انتقال قطعه‌قطعه ساده برای ارسال داده‌ها استفاده می‌کند
  - پخش جریانی MCP: از سیستم اعلان ساختارمند با پروتکل JSON-RPC استفاده می‌کند

- **فرمت پیام:**
  - HTTP کلاسیک: قطعات متن ساده با خطوط جدید
  - MCP: اشیاء LoggingMessageNotification ساختارمند با فراداده

- **پیاده‌سازی کلاینت:**
  - HTTP کلاسیک: کلاینت ساده که پاسخ‌های پخش جریانی را پردازش می‌کند
  - MCP: کلاینت پیشرفته‌تر با هندلر پیام برای پردازش انواع مختلف پیام‌ها

- **به‌روزرسانی‌های پیشرفت:**
  - HTTP کلاسیک: پیشرفت بخشی از جریان پاسخ اصلی است
  - MCP: پیشرفت از طریق پیام‌های اعلان جداگانه ارسال می‌شود در حالی که پاسخ اصلی در پایان می‌آید

### توصیه‌ها

هنگام انتخاب بین پیاده‌سازی پخش جریانی کلاسیک (به عنوان نقطه پایانی که بالاتر با `/stream` نشان داده‌ایم) و پخش جریانی از طریق MCP، موارد زیر توصیه می‌شود:

- **برای نیازهای ساده پخش جریانی:** پخش جریانی HTTP کلاسیک ساده‌تر برای پیاده‌سازی است و برای نیازهای پایه‌ای کافی است.

- **برای برنامه‌های پیچیده و تعاملی:** پخش جریانی MCP رویکردی ساختارمندتر با فراداده غنی‌تر و جداسازی بین اعلان‌ها و نتایج نهایی ارائه می‌دهد.

- **برای برنامه‌های هوش مصنوعی:** سیستم اعلان MCP برای وظایف طولانی هوش مصنوعی که می‌خواهید کاربران را از پیشرفت کار مطلع نگه دارید به‌ویژه مفید است.

## پخش جریانی در MCP

خوب، اکنون برخی توصیه‌ها و مقایسه‌ها تا اینجا در مورد تفاوت پخش جریانی کلاسیک و MCP دیدید. حالا به جزئیات دقیق‌تر اینکه چگونه می‌توانید پخش جریانی را در MCP به کار ببرید می‌پردازیم.

فهمیدن اینکه پخش جریانی در چارچوب MCP چگونه کار می‌کند برای ساخت برنامه‌های پاسخگو که بازخورد بلادرنگ به کاربران در طول عملیات طولانی ارائه می‌دهند ضروری است.

در MCP، پخش جریانی به معنای ارسال پاسخ اصلی به صورت قطعات نیست، بلکه ارسال **اعلان‌ها** به کلاینت است در حالی که ابزاری درخواست را پردازش می‌کند. این اعلان‌ها می‌توانند شامل به‌روزرسانی‌های پیشرفت، لاگ‌ها یا سایر رویدادها باشند.

### نحوه کارکرد

نتیجه اصلی هنوز به صورت یک پاسخ واحد ارسال می‌شود. با این حال، اعلان‌ها می‌توانند به صورت پیام‌های جداگانه در طول پردازش ارسال شوند و در نتیجه کلاینت را به صورت بلادرنگ به‌روزرسانی کنند. کلاینت باید قادر به پردازش و نمایش این اعلان‌ها باشد.

## اعلان چیست؟

ما گفتیم "اعلان"، در زمینه MCP چه معنایی دارد؟

اعلان پیام ارسالی از سرور به کلاینت است که درباره پیشرفت، وضعیت یا دیگر رویدادهای یک عملیات طولانی اطلاع می‌دهد. اعلان‌ها شفافیت و تجربه کاربری را بهبود می‌بخشند.

برای مثال، کلاینت باید پس از ایجاد اولیه دست‌دهی با سرور اعلان ارسال کند.

اعلان به شکل زیر به صورت پیام JSON است:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

اعلان‌ها به موضوعی در MCP به نام ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) تعلق دارند.

برای فعال کردن لاگگیری، سرور باید آن را به عنوان قابلیت/ویژگی فعال کند بدین شکل:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> بسته به SDK استفاده شده، لاگگیری ممکن است به طور پیش‌فرض فعال باشد یا نیاز باشد به طور صریح در پیکربندی سرور فعال شود.

انواع مختلفی از اعلان‌ها وجود دارد:

| سطح       | توضیح                         | مثال مورد استفاده            |
|-----------|------------------------------|-----------------------------|
| debug     | اطلاعات دقیق اشکال‌زدایی     | نقاط ورود/خروج توابع         |
| info      | پیام‌های عمومی اطلاع‌رسانی  | به‌روزرسانی پیشرفت عملیات    |
| notice    | رویدادهای عادی اما مهم       | تغییرات پیکربندی             |
| warning   | شرایط هشدار                  | استفاده از ویژگی منسوخ شده   |
| error     | شرایط خطا                    | شکست عملیات                  |
| critical  | شرایط بحرانی                | خرابی اجزای سیستم           |
| alert     | اقدام فوری لازم است          | کشف خرابی داده               |
| emergency | سیستم غیرقابل استفاده است   | خرابی کامل سیستم            |

## پیاده‌سازی اعلان‌ها در MCP

برای پیاده‌سازی اعلان‌ها در MCP، نیاز است هم سمت سرور و هم سمت کلاینت برای دریافت به‌روزرسانی‌های بلادرنگ تنظیم شوند. این به برنامه شما اجازه می‌دهد تا هنگام انجام عملیات طولانی، بازخورد فوری به کاربران ارائه دهد.

### سمت سرور: ارسال اعلان‌ها

ابتدا با سمت سرور شروع کنیم. در MCP، شما ابزارهایی تعریف می‌کنید که در هنگام پردازش درخواست‌ها می‌توانند اعلان ارسال کنند. سرور از شیء بافت (معمولاً `ctx`) برای ارسال پیام به کلاینت استفاده می‌کند.

#### پایتون

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

در مثال قبلی، ابزار `process_files` هنگام پردازش هر فایل سه اعلان به کلاینت ارسال می‌کند. متد `ctx.info()` برای ارسال پیام‌های اطلاع‌رسانی استفاده شده است.

علاوه بر این، برای فعال کردن اعلان‌ها اطمینان حاصل کنید که سرور شما از انتقال پخش جریانی (مانند `streamable-http`) استفاده می‌کند و کلاینت شما یک هندلر پیام برای پردازش اعلان‌ها پیاده‌سازی کرده است. در اینجا نحوه تنظیم سرور برای استفاده از انتقال `streamable-http` آورده شده است:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

در این مثال .NET، ابزار `ProcessFiles` با ویژگی `Tool` تزئین شده و هنگام پردازش هر فایل سه اعلان به کلاینت ارسال می‌کند. متد `ctx.Info()` برای ارسال پیام‌های اطلاع‌رسانی استفاده شده است.

برای فعال کردن اعلان‌ها در سرور MCP .NET خود، اطمینان حاصل کنید که از انتقال پخش جریانی استفاده می‌کنید:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### سمت کلاینت: دریافت اعلان‌ها

کلاینت باید یک هندلر پیام پیاده‌سازی کند که اعلان‌ها را هنگام دریافت پردازش و نمایش دهد.

#### پایتون

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

در کد قبلی، تابع `message_handler` بررسی می‌کند آیا پیام ورودی اعلان است یا خیر. اگر بود، اعلان را چاپ می‌کند؛ در غیر این صورت آن را به عنوان پیام عادی سرور پردازش می‌کند. همچنین مشاهده می‌کنید که `ClientSession` با استفاده از `message_handler` راه‌اندازی می‌شود تا اعلان‌های دریافتی مدیریت شوند.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

در این مثال .NET، تابع `MessageHandler` بررسی می‌کند آیا پیام ورودی اعلان است یا خیر. اگر بود، اعلان را چاپ می‌کند؛ در غیر این صورت به عنوان پیام عادی سرور پردازش می‌نماید. `ClientSession` با هندلر پیام از طریق `ClientSessionOptions` راه‌اندازی می‌شود.

برای فعال کردن اعلان‌ها، اطمینان حاصل کنید که سرور شما از انتقال پخش جریانی (مانند `streamable-http`) استفاده می‌کند و کلاینت شما هندلر پیام برای پردازش اعلان‌ها پیاده‌سازی کرده است.

## اعلان‌های پیشرفت و سناریوها

این بخش مفهوم اعلان‌های پیشرفت در MCP، اهمیت آن‌ها و نحوه پیاده‌سازی آن‌ها با استفاده از HTTP قابل پخش را توضیح می‌دهد. همچنین یک تمرین عملی برای تقویت درک شما ارائه شده است.

اعلان‌های پیشرفت پیام‌های بلادرنگی هستند که از سرور به کلاینت در طول عملیات طولانی ارسال می‌شوند. به جای اینکه منتظر پایان کامل فرآیند باشید، سرور کلاینت را درباره وضعیت فعلی به‌روزرسانی می‌کند. این باعث بهبود شفافیت، تجربه کاربری و آسان‌تر شدن اشکال‌زدایی می‌شود.

**مثال:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### چرا از اعلان‌های پیشرفت استفاده کنیم؟

اعلان‌های پیشرفت به دلایل متعددی ضروری هستند:

- **تجربه بهتر کاربر:** کاربران به‌روزرسانی‌ها را در حین پیشرفت کار می‌بینند، نه فقط در پایان.
- **بازخورد بلادرنگ:** کلاینت‌ها می‌توانند نوار پیشرفت یا لاگ‌ها را نمایش دهند و برنامه را پاسخگوتر جلوه دهند.
- **اشکال‌زدایی و نظارت آسان‌تر:** توسعه‌دهندگان و کاربران می‌توانند ببینند کجا ممکن است فرایند کند یا متوقف شده باشد.

### چگونه اعلان‌های پیشرفت را پیاده‌سازی کنیم

در اینجا نحوه پیاده‌سازی اعلان‌های پیشرفت در MCP آورده شده است:

- **در سرور:** برای ارسال اعلان‌ها هنگام پردازش هر مورد از `ctx.info()` یا `ctx.log()` استفاده کنید. این پیام قبل از آماده شدن نتیجه اصلی به کلاینت ارسال می‌شود.
- **در کلاینت:** هندلر پیام پیاده‌سازی کنید که به اعلان‌ها گوش فرا داده و آن‌ها را نمایش دهد. این هندلر بین اعلان‌ها و نتیجه نهایی تمایز قائل می‌شود.

**مثال سرور:**

#### پایتون

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**مثال کلاینت:**

#### پایتون

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## ملاحظات امنیتی

هنگام پیاده‌سازی سرورهای MCP با انتقال‌های مبتنی بر HTTP، امنیت به موضوعی حیاتی تبدیل می‌شود که نیازمند توجه دقیق به بردارهای حمله متعدد و مکانیزم‌های حفاظتی است.

### مروری کلی

امنیت هنگام ارائه سرورهای MCP از طریق HTTP اهمیت بالایی دارد. HTTP قابل پخش سطوح جدیدی از حمله را معرفی می‌کند و نیازمند پیکربندی دقیق است.

### نکات کلیدی


- **اعتبارسنجی هدر Origin**: همیشه هدر `Origin` را برای جلوگیری از حملات بازبینی DNS معتبرسازی کنید.
- **بست کردن به localhost**: برای توسعه محلی، سرورها را به `localhost` متصل کنید تا از در معرض قرار گرفتن در اینترنت عمومی جلوگیری شود.
- **احراز هویت**: برای استقرارهای تولیدی احراز هویت (مانند کلیدهای API، OAuth) را پیاده‌سازی کنید.
- **CORS**: سیاست‌های به اشتراک‌گذاری منابع میان‌منشأ (CORS) را برای محدود کردن دسترسی تنظیم کنید.
- **HTTPS**: در تولید از HTTPS برای رمزنگاری ترافیک استفاده کنید.

### بهترین روش‌ها

- هرگز به درخواست‌های ورودی بدون اعتبارسنجی اعتماد نکنید.
- تمام دسترسی‌ها و خطاها را ثبت و مانیتور کنید.
- به صورت منظم وابستگی‌ها را به‌روزرسانی کنید تا آسیب‌پذیری‌های امنیتی برطرف شوند.

### چالش‌ها

- متعادل‌سازی امنیت با سهولت توسعه
- اطمینان از سازگاری با محیط‌های مختلف مشتری

## ارتقا از SSE به Streamable HTTP

برای برنامه‌هایی که هم‌اکنون از Server-Sent Events (SSE) استفاده می‌کنند، مهاجرت به Streamable HTTP قابلیت‌های پیشرفته‌تر و پایداری بلندمدت بهتری برای پیاده‌سازی‌های MCP شما فراهم می‌کند.

### چرا ارتقا بدهیم؟

دو دلیل قانع‌کننده برای ارتقا از SSE به Streamable HTTP وجود دارد:

- Streamable HTTP مقیاس‌پذیری، سازگاری، و پشتیبانی غنی‌تری از اعلان‌ها نسبت به SSE ارائه می‌دهد.
- این روش حمل‌ونقل توصیه‌شده برای برنامه‌های جدید MCP است.

### مراحل مهاجرت

در اینجا نحوه مهاجرت از SSE به Streamable HTTP در برنامه‌های MCP شما آمده است:

- **کد سرور را به‌روزرسانی کنید** تا از `transport="streamable-http"` در `mcp.run()` استفاده کند.
- **کد کلاینت را به‌روزرسانی کنید** تا به جای کلاینت SSE از `streamablehttp_client` استفاده کند.
- **یک هندلر پیام پیاده‌سازی کنید** در کلاینت برای پردازش اعلان‌ها.
- **سازگاری را تست کنید** با ابزارها و جریان‌های کاری موجود.

### حفظ سازگاری

توصیه می‌شود در طول فرایند مهاجرت با کلاینت‌های SSE موجود سازگاری را حفظ کنید. در اینجا برخی استراتژی‌ها آمده است:

- می‌توانید هر دو روش SSE و Streamable HTTP را با اجرای هر دو حمل‌ونقل در نقاط انتهایی مختلف حمایت کنید.
- کلاینت‌ها را به تدریج به حمل‌ونقل جدید مهاجرت دهید.

### چالش‌ها

باید چالش‌های زیر را در طول مهاجرت مدنظر داشته باشید:

- اطمینان از به‌روزرسانی همه کلاینت‌ها
- مدیریت تفاوت‌ها در تحویل اعلان‌ها
