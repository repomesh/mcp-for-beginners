# ماڈل کانٹیکسٹ پروٹوکول (MCP) کے ساتھ HTTPS اسٹریمنگ

اس باب میں ماڈل کانٹیکسٹ پروٹوکول (MCP) کا استعمال کرتے ہوئے HTTPS کے ذریعے محفوظ، اسکیل ایبل، اور حقیقی وقت کی اسٹریمنگ کو نافذ کرنے کے لئے ایک جامع رہنما فراہم کیا گیا ہے۔ یہ اسٹریمنگ کی تحریک، دستیاب ٹرانسپورٹ میکانزم، MCP میں اسٹریمنگ ایبل HTTP کو نافذ کرنے کا طریقہ، سیکیورٹی کی بہترین مشقیں، SSE سے منتقلی، اور اپنے ذاتی اسٹریمنگ MCP ایپلیکیشنز بنانے کے لیے عملی رہنمائی کا احاطہ کرتا ہے۔

> **آگے کی نظر:** یہ سبق Streamable HTTP کو **MCP Specification 2025-11-25** کے تحت بیان کرتا ہے، جہاں ایک سیشن `initialize` کے دوران قائم کیا جاتا ہے اور اسے `Mcp-Session-Id` ہیڈر کے ساتھ پن کیا جاتا ہے۔ `2026-07-28` ریلیز کینڈیڈیٹ ہینڈشیک اور سیشن آئی ڈی کو مکمل طور پر ہٹا دیتا ہے، جس سے ہر درخواست خود مختار اور کسی بھی سرور انسٹینس کو بھیجی جا سکتی ہے بغیر اسٹکی سیشنز کے۔ تفصیلات کے لیے دیکھیں [MCP میں کیا تبدیلی آ رہی ہے: 2026-07-28 ریلیز کینڈیڈیٹ](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md)۔

## MCP میں ٹرانسپورٹ میکانزم اور اسٹریمنگ

یہ سیکشن MCP میں دستیاب مختلف ٹرانسپورٹ میکانزم کا جائزہ لیتا ہے اور ان کا کلائنٹس اور سرورز کے درمیان حقیقی وقت کی مواصلات کے لیے اسٹریمنگ کی صلاحیتوں کو فعال بنانے میں کردار۔

### ٹرانسپورٹ میکانزم کیا ہے؟

ایک ٹرانسپورٹ میکانزم یہ متعین کرتا ہے کہ کس طرح کلائنٹ اور سرور کے درمیان ڈیٹا کا تبادلہ ہوتا ہے۔ MCP مختلف ماحول اور ضروریات کے مطابق متعدد ٹرانسپورٹ اقسام کی حمایت کرتا ہے:

- **stdio**: معیاری ان پٹ/آؤٹ پٹ، مقامی اور CLI پر مبنی اوزاروں کے لیے موزوں۔ آسان لیکن ویب یا کلاؤڈ کے لیے موزوں نہیں۔
- **SSE (سرور- بھیجے گئے ایونٹس)**: سرورز کو HTTP کے ذریعے کلائنٹس کو حقیقی وقت کی اپ ڈیٹس بھیجنے کی اجازت دیتا ہے۔ ویب UIs کے لیے اچھا، لیکن اسکیل ایبلٹی اور لچک میں محدود۔ MCP Specification 2025-06-18 کے تحت، اسٹینڈ اکیلان SSE (سرور- بھیجے گئے ایونٹس) ٹرانسپورٹ کو ختم کر کے "Streamable HTTP" ٹرانسپورٹ سے تبدیل کیا گیا ہے۔
- **Streamable HTTP**: جدید HTTP پر مبنی اسٹریمنگ ٹرانسپورٹ، نوٹیفیکیشنز اور بہتر اسکیل ایبلٹی کو سپورٹ کرتا ہے۔ زیادہ تر پروڈکشن اور کلاؤڈ کے منظرناموں کے لیے تجویز کردہ۔

### موازنہ جدول

نیچے دیے گئے موازنہ جدول کو دیکھیں تاکہ ان ٹرانسپورٹ میکانزم میں فرق کو سمجھ سکیں:

| ٹرانسپورٹ      | حقیقی وقت کی اپ ڈیٹس | اسٹریمنگ | اسکیل ایبلٹی | استعمال کی مثال          |
|----------------|--------------------|----------|--------------|-------------------------|
| stdio          | نہیں                | نہیں      | کم           | مقامی CLI آلات          |
| SSE            | جی ہاں              | جی ہاں    | درمیانہ      | ویب، حقیقی وقت کی اپ ڈیٹس|
| Streamable HTTP| جی ہاں              | جی ہاں    | زیادہ         | کلاؤڈ، متعدد کلائنٹس    |

> **مشورہ:** صحیح ٹرانسپورٹ کا انتخاب کارکردگی، اسکیل ایبلٹی، اور صارف کے تجربے کو متاثر کرتا ہے۔ **Streamable HTTP** جدید، اسکیل ایبل اور کلاؤڈ-ریڈی ایپلیکیشنز کے لیے تجویز کردہ ہے۔

نوٹ کریں کہ پچھلے ابواب میں آپ کو جو stdio اور SSE دکھایا گیا تھا، اور کیسے اس باب میں پھر Streamable HTTP ٹرانسپورٹ کا احاطہ کیا گیا ہے۔

## اسٹریمنگ: تصورات اور محرکات

اسٹریمنگ کے بنیادی تصورات اور محرکات کو سمجھنا مؤثر حقیقی وقت کے مواصلاتی نظام کو نافذ کرنے کے لیے ضروری ہے۔

**اسٹریمنگ** نیٹ ورک پروگرامنگ میں ایک تکنیک ہے جو ڈیٹا کو چھوٹے، قابل انتظام حصّوں یا واقعات کی تسلسل کے طور پر بھیجے اور وصول کرنے کی اجازت دیتی ہے، بجائے اس کے کہ پورا ردعمل تیار ہونے کا انتظار کیا جائے۔ یہ خاص طور پر مفید ہے:

- بڑے فائلوں یا ڈیٹا سیٹس کے لئے۔
- حقیقی وقت کی اپ ڈیٹس (مثلاً چیٹ، پروگریس بارز)۔
- طویل مدتی حساب کتاب جہاں آپ صارف کو مطلع رکھنا چاہتے ہیں۔

یہاں اسٹریمنگ کے بارے میں آپ کو اعلی سطح پر جاننے کی ضرورت ہے:

- ڈیٹا تدریجی طور پر فراہم کیا جاتا ہے، ایک ساتھ نہیں۔
- کلائنٹ ڈیٹا کو جیسے جیسے آتا ہے پروسیس کر سکتا ہے۔
- محسوس کی جانے والی تاخیر کو کم کرتا ہے اور صارف کے تجربے کو بہتر بناتا ہے۔

### اسٹریمنگ کیوں استعمال کریں؟

اسٹریمنگ کے استعمال کی وجوہات درج ذیل ہیں:

- صارفین فوری طور پر فیڈبیک حاصل کرتے ہیں، صرف آخر میں نہیں۔
- حقیقی وقت کی ایپلیکیشنز اور جوابی UI کو قابل بناتا ہے۔
- نیٹ ورک اور کمپیوٹ کے وسائل کا زیادہ مؤثر استعمال۔

### سادہ مثال: HTTP اسٹریمنگ سرور اور کلائنٹ

یہاں ایک سادہ مثال دی گئی ہے کہ اسٹریمنگ کیسے نافذ کی جا سکتی ہے:

#### پائتھون

**سرور (پائتھون، FastAPI اور StreamingResponse کے استعمال کے ساتھ):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**کلائنٹ (پائتھون، requests کا استعمال کرتے ہوئے):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

یہ مثال دکھاتی ہے کہ سرور کیسے دستیاب ہوتے ہی پیغامات کی ایک سلسلہ کلائنٹ کو بھیجتا ہے، بجائے اس کے کہ تمام پیغامات تیار ہونے کا انتظار کرے۔

**یہ کیسے کام کرتا ہے:**

- سرور ہر پیغام کو جیسے ہی وہ تیار ہو نکلتا ہے۔
- کلائنٹ ہر حصّے کو وصول کرتا ہے اور جیسے ہی آتا ہے پرنٹ کرتا ہے۔

**ضروریات:**

- سرور کو اسٹریمنگ ریسپانس استعمال کرنا چاہیے (جیسے FastAPI میں `StreamingResponse`)۔
- کلائنٹ کو ریسپانس کو سٹریم کے طور پر پروسیس کرنا چاہیے (`stream=True` requests میں)۔
- مواد کی قسم عموماً `text/event-stream` یا `application/octet-stream` ہوتی ہے۔

#### جاوا

**سرور (جاوا، Spring Boot اور Server-Sent Events کے استعمال کے ساتھ):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**کلائنٹ (جاوا، Spring WebFlux WebClient کا استعمال کرتے ہوئے):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**جاوا کے نفاذ کے نوٹس:**

- Spring Boot کا ریئیکٹو اسٹیک `Flux` کے ساتھ اسٹریمنگ کے لیے استعمال کرتا ہے
- `ServerSentEvent` واقعات کی اقسام کے ساتھ ساختہ ایونٹ اسٹریمنگ فراہم کرتا ہے
- `WebClient` `bodyToFlux()` کے ساتھ ریئیکٹو اسٹریمنگ استعمال کو فعال کرتا ہے
- `delayElements()` واقعات کے درمیان پروسیسنگ کے وقت کی نقل کرتا ہے
- واقعات میں اقسام (`info`, `result`) ہو سکتی ہیں بہتر کلائنٹ ہینڈلنگ کے لیے

### موازنہ: کلاسیکی اسٹریمنگ بمقابلہ MCP اسٹریمنگ

یہ فرق کہ کلاسیکی انداز میں اسٹریمنگ کیسے کام کرتی ہے اور MCP میں کیسے کام کرتی ہے، کچھ یوں بیان کیا جا سکتا ہے:

| خصوصیت                | کلاسیکی HTTP اسٹریمنگ         | MCP اسٹریمنگ (نوٹیفیکیشنز)      |
|------------------------|-------------------------------|-------------------------------------|
| مرکزی ریسپانس          | چنکس میں                     | ایک ہی، آخر میں                      |
| پروگریس اپ ڈیٹس       | ڈیٹا چنکس کی صورت میں بھیجے جاتے ہیں | نوٹیفیکیشنز کی صورت میں بھیجے جاتے ہیں   |
| کلائنٹ کی ضروریات     | سٹریم کو پروسیس کرنا ضروری ہے   | میسج ہینڈلر کو نافذ کرنا ضروری ہے    |
| استعمال کی مثال        | بڑی فائلیں، AI ٹوکن سٹریمز   | پروگریس، لاگز، حقیقی وقت کا فیڈبیک     |

### کلیدی فرق جو دیکھے گئے

اس کے علاوہ یہاں کچھ کلیدی فرق درج ہیں:

- **مواصلاتی پیٹرن:**
  - کلاسیکی HTTP اسٹریمنگ: سادہ چنکڈ ٹرانسفر انکوڈنگ استعمال کرتا ہے ڈیٹا کی چنکس بھیجنے کے لیے
  - MCP اسٹریمنگ: ساختہ نوٹیفیکیشن سسٹم استعمال کرتا ہے JSON-RPC پروٹوکول کے ساتھ

- **میسج فارمیٹ:**
  - کلاسیکی HTTP: نئے لائن کے ساتھ سادہ ٹیکسٹ چنکس
  - MCP: ساختہ LoggingMessageNotification آبجیکٹس میٹا ڈیٹا کے ساتھ

- **کلائنٹ نفاذ:**
  - کلاسیکی HTTP: ایسطرنگ ریسپانسز کو پروسیس کرنے والا سادہ کلائنٹ
  - MCP: زیادہ پیچیدہ کلائنٹ جس میں مختلف قسم کے میسجز کو پروسیس کرنے کے لیے میسج ہینڈلر ہوتا ہے

- **پروگریس اپ ڈیٹس:**
  - کلاسیکی HTTP: پروگریس مرکزی ریسپانس سٹریم کا حصہ ہوتا ہے
  - MCP: پروگریس علیحدہ نوٹیفیکیشن میسجز کے ذریعے بھیجی جاتی ہے جبکہ مرکزی ریسپانس آخر میں آتا ہے

### سفارشات

کلاسیکی اسٹریمنگ (جیسا کہ آپ کو اوپر `/stream` کے ذریعے دکھایا گیا ہے) اور MCP کے ذریعے اسٹریمنگ اپنانے کے درمیان انتخاب کے لیے ہم کچھ سفارشات پیش کرتے ہیں۔

- **سادہ اسٹریمنگ ضروریات کے لیے:** کلاسیکی HTTP اسٹریمنگ نفاذ میں سادہ ہے اور بنیادی اسٹریمنگ ضروریات کے لیے کافی ہے۔

- **پیچیدہ، انٹرایکٹو ایپلیکیشنز کے لیے:** MCP اسٹریمنگ زیادہ ساختہ طریقہ فراہم کرتا ہے، زیادہ میٹا ڈیٹا کے ساتھ نوٹیفیکیشنز اور حتمی نتائج میں تفریق۔

- **AI ایپلیکیشنز کے لیے:** MCP کا نوٹیفیکیشن سسٹم طویل مدتی AI ٹاسک کے لیے خاص طور پر مفید ہے جہاں آپ صارفین کو پروگریس سے آگاہ رکھنا چاہتے ہیں۔

## MCP میں اسٹریمنگ

تو، آپ نے ابھی تک کلاسیکی اسٹریمنگ اور MCP میں اسٹریمنگ کے درمیان بعض سفارشات اور موازنہ دیکھے ہیں۔ آئیے تفصیل میں جاتے ہیں کہ آپ MCP میں اسٹریمنگ کو کیسے استعمال کر سکتے ہیں۔

MCP فریم ورک کے اندر اسٹریمنگ کیسے کام کرتی ہے یہ سمجھنا ضروری ہے تاکہ آپ جواب دینے والی ایپلیکیشنز بنا سکیں جو طویل مدتی آپریشنز کے دوران صارفین کو حقیقی وقت کی فیڈبیک فراہم کریں۔

MCP میں، اسٹریمنگ کا مطلب مرکزی ریسپانس کو چنکس میں بھیجنا نہیں ہے، بلکہ درخواست پر عمل کے دوران کلائنٹ کو **نوٹیفیکیشنز** بھیجنا ہے۔ یہ نوٹیفیکیشنز پروگریس اپ ڈیٹس، لاگز، یا دیگر واقعات ہو سکتے ہیں۔

### یہ کیسے کام کرتا ہے

مرکزی نتیجہ اب بھی ایک واحد ریسپانس کے طور پر بھیجا جاتا ہے۔ تاہم، پروسیسنگ کے دوران نوٹیفیکیشنز علیحدہ پیغامات کے طور پر بھیجی جا سکتی ہیں، اور اس طرح کلائنٹ کو حقیقی وقت میں اپ ڈیٹ کرتی ہیں۔ کلائنٹ کو ان نوٹیفیکیشنز کو ہینڈل اور دکھانے کے قابل ہونا چاہیے۔

## نوٹیفیکیشن کیا ہے؟

ہم نے کہا "نوٹیفیکیشن"، MCP کے سیاق و سباق میں اس کا کیا مطلب ہے؟

ایک نوٹیفیکیشن ایک پیغام ہوتا ہے جو سرور سے کلائنٹ کو بھیجا جاتا ہے تاکہ طویل مدتی آپریشن کے دوران پروگریس، حیثیت، یا دیگر واقعات کے بارے میں معلومات فراہم کی جا سکیں۔ نوٹیفیکیشنز شفافیت اور صارف کے تجربے کو بہتر بناتی ہیں۔

مثال کے طور پر، کلائنٹ کو سرور کے ساتھ ابتدائی ہینڈشیک مکمل ہونے کے بعد نوٹیفیکیشن بھیجنی چاہیے۔

ایک نوٹیفیکیشن JSON پیغام کے طور پر کچھ یوں نظر آتی ہے:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

نوٹیفیکیشنز MCP میں ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) کے موضوع سے تعلق رکھتی ہیں۔

لاگنگ کو کام میں لانے کے لیے، سرور کو اسے فیچر/صلاحیت کے طور پر اس طرح فعال کرنا ہوتا ہے:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> استعمال ہونے والے SDK پر منحصر ہے، لاگنگ ڈیفالٹ طور پر فعال ہو سکتی ہے، یا آپ کو سرور کنفیگریشن میں اسے واضح طور پر فعال کرنا پڑ سکتا ہے۔

مختلف قسم کی نوٹیفیکیشنز ہیں:

| سطح       | وضاحت                         | استعمال کی مثال               |
|-----------|------------------------------|------------------------------|
| debug     | تفصیلی ڈیبگ معلومات          | فنکشن کے داخلے/اخراج کے نقاط|
| info      | عمومی معلوماتی پیغامات       | آپریشن کے پروگریس اپ ڈیٹس   |
| notice    | معمول لیکن اہم واقعات          | کنفیگریشن میں تبدیلیاں       |
| warning   | انتباہی حالات               | متروک فیچر کا استعمال        |
| error     | ایرر کے حالات               | آپریشن کی ناکامیاں          |
| critical  | سنگین حالات                | سسٹم کے جزو کی ناکامیاں      |
| alert     | فوری ایکشن کی ضرورت           | ڈیٹا کرپشن معلوم ہوئی        |
| emergency | نظام غیر قابل استعمال      | مکمل نظام کی ناکامی          |

## MCP میں نوٹیفیکیشنز کا نفاذ

MCP میں نوٹیفیکیشنز کو نافذ کرنے کے لیے، آپ کو سرور اور کلائنٹ دونوں کو حقیقی وقت کی اپ ڈیٹس سنبھالنے کے قابل بنانا ہوگا۔ اس سے آپ کی ایپلیکیشن طویل مدتی آپریشنز کے دوران صارفین کو فوری فیڈبیک فراہم کر سکتی ہے۔

### سرور سائیڈ: نوٹیفیکیشنز بھیجنا

آئیے سرور سائیڈ سے شروع کرتے ہیں۔ MCP میں، آپ ایسے ٹولز متعین کرتے ہیں جو درخواستوں پر عمل کرتے وقت نوٹیفیکیشنز بھیج سکتے ہیں۔ سرور کلائنٹ کو پیغامات بھیجنے کے لیے کانٹیکسٹ آبجیکٹ (عام طور پر `ctx`) استعمال کرتا ہے۔

#### پائتھون

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

پچھلی مثال میں، `process_files` ٹول فائلوں پر عمل کرتے ہوئے کلائنٹ کو تین نوٹیفیکیشنز بھیجتا ہے۔ `ctx.info()` میتھڈ معلوماتی پیغامات بھیجنے کے لیے استعمال ہوتی ہے۔

مزید برآں، نوٹیفیکیشنز فعال کرنے کے لیے، یقینی بنائیں کہ آپ کا سرور ایک اسٹریمنگ ٹرانسپورٹ (جیسے `streamable-http`) استعمال کرتا ہے اور آپ کا کلائنٹ نوٹیفیکیشنز کو پروسیس کرنے کے لیے میسج ہینڈلر نافذ کرتا ہے۔ یہاں دکھایا گیا ہے کہ سرور کو `streamable-http` ٹرانسپورٹ کے استعمال کے لیے کیسے سیٹ اپ کریں:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

اس .NET مثال میں، `ProcessFiles` ٹول کو `Tool` اٹریبیوٹ کے ساتھ نشان زد کیا گیا ہے اور یہ ہر فائل پر عمل کرتے ہوئے کلائنٹ کو تین نوٹیفیکیشنز بھیجتا ہے۔ `ctx.Info()` میتھڈ معلوماتی پیغامات بھیجنے کے لیے استعمال ہوتی ہے۔

اپنے .NET MCP سرور میں نوٹیفیکیشنز فعال کرنے کے لیے، اس بات کو یقینی بنائیں کہ آپ ایک اسٹریمنگ ٹرانسپورٹ استعمال کر رہے ہیں:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### کلائنٹ سائیڈ: نوٹیفیکیشنز وصول کرنا

کلائنٹ کو ایک میسج ہینڈلر نافذ کرنا چاہیے جو نوٹیفیکیشنز کو وصول کرے اور دکھائے جب وہ آتے ہیں۔

#### پائتھون

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

پچھلے کوڈ میں، `message_handler` فنکشن چیک کرتا ہے کہ آیا آنے والا پیغام نوٹیفیکیشن ہے۔ اگر ہے تو، وہ نوٹیفیکیشن پرنٹ کرتا ہے؛ بصورت دیگر اسے عام سرور میسج کے طور پر پروسیس کرتا ہے۔ ساتھ ہی نوٹ کریں کہ `ClientSession` کو `message_handler` کے ساتھ نوٹیفیکیشنز سنبھالنے کے لیے شروع کیا گیا ہے۔

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

اس .NET مثال میں، `MessageHandler` فنکشن چیک کرتا ہے کہ آیا آنے والا پیغام نوٹیفیکیشن ہے۔ اگر ہے تو، وہ نوٹیفیکیشن پرنٹ کرتا ہے؛ بصورت دیگر اسے عام سرور میسج کے طور پر پروسیس کرتا ہے۔ `ClientSession` کو `ClientSessionOptions` کے ذریعے میسج ہینڈلر کے ساتھ شروع کیا گیا ہے۔

نوٹیفیکیشنز فعال کرنے کے لیے، یقینی بنائیں کہ آپ کا سرور اسٹریمنگ ٹرانسپورٹ (جیسے `streamable-http`) استعمال کرتا ہے اور آپ کا کلائنٹ نوٹیفیکیشنز کو پروسیس کرنے والا میسج ہینڈلر نافذ کرتا ہے۔

## پروگریس نوٹیفیکیشنز اور منظرنامے

یہ سیکشن MCP میں پروگریس نوٹیفیکیشن کے تصور، ان کی اہمیت، اور انہیں Streamable HTTP کے استعمال سے نافذ کرنے کے طریقے کی وضاحت کرتا ہے۔ آپ کی سمجھ کو مضبوط بنانے کے لیے ایک عملی اسائنمنٹ بھی شامل ہے۔

پروگریس نوٹیفیکیشنز وہ حقیقی وقت کے پیغامات ہیں جو سرور سے کلائنٹ کو طویل مدتی آپریشنز کے دوران بھیجے جاتے ہیں۔ پورے عمل کے ختم ہونے کا انتظار کرنے کے بجائے، سرور کلائنٹ کو موجودہ حیثیت کے بارے میں اپ ڈیٹ کرتا رہتا ہے۔ یہ شفافیت، صارف کے تجربے کو بہتر بناتا ہے، اور ڈیبگنگ آسان بناتا ہے۔

**مثال:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### پروگریس نوٹیفیکیشنز کیوں استعمال کریں؟

پروگریس نوٹیفیکیشنز کئی وجوہات کی بنا پر ضروری ہیں:

- **بہتر صارف کا تجربہ:** صارفین کام کے جاری رہنے پر اپ ڈیٹس دیکھتے ہیں، صرف آخر میں نہیں۔
- **حقیقی وقت کا فیڈبیک:** کلائنٹس پروگریس بارز یا لاگز دکھا سکتے ہیں، جس سے ایپ زیادہ جوابدہ محسوس ہوتی ہے۔
- **آسان ڈیبگنگ اور مانیٹرنگ:** ڈیولپرز اور صارفین دیکھ سکتے ہیں کہ عمل کہاں سست یا رکا ہوا ہے۔

### پروگریس نوٹیفیکیشنز کیسے نافذ کریں

یہاں MCP میں پروگریس نوٹیفیکیشنز نافذ کرنے کا طریقہ ہے:

- **سرور پر:** `ctx.info()` یا `ctx.log()` استعمال کریں تاکہ ہر چیز پروسیس ہوتے ہوئے نوٹیفیکیشنز بھیجیں۔ یہ مرکزی نتیجہ کے تیار ہونے سے پہلے کلائنٹ کو پیغام بھیجتا ہے۔
- **کلائنٹ پر:** ایک میسج ہینڈلر نافذ کریں جو نوٹیفیکیشنز وصول کرے اور دکھائے جب وہ آئیں۔ یہ ہینڈلر نوٹیفیکیشنز اور حتمی نتیجہ میں فرق کرتا ہے۔

**سرور کی مثال:**

#### پائتھون

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**کلائنٹ کی مثال:**

#### پائتھون

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## سیکیورٹی کے پہلو

جب MCP سرورز کو HTTP پر مبنی ٹرانسپورٹس کے ساتھ نافذ کیا جاتا ہے، تو سیکیورٹی ایک اہم مسئلہ بن جاتی ہے جس کو مختلف حملوں کے راستوں اور حفاظتی تدابیر پر غور کرنے کی ضرورت ہوتی ہے۔

### جائزہ

MCP سرورز کو HTTP پر ظاہر کرتے وقت سیکیورٹی انتہائی اہم ہوتی ہے۔ Streamable HTTP نئے حملوں کے مواقع پیش کرتا ہے اور احتیاطی ترتیبات کا محتاج ہوتا ہے۔

### اہم نکات


- **Origin ہیڈر کی تصدیق**: DNS ری بانڈنگ حملوں سے بچنے کے لیے ہمیشہ `Origin` ہیڈر کی تصدیق کریں۔
- **لوکل ہوسٹ بائنڈنگ**: مقامی ترقی کے لیے، سرورز کو `localhost` پر بائنڈ کریں تاکہ انہیں پبلک انٹرنیٹ پر ظاہر ہونے سے بچایا جا سکے۔
- **تصدیق**: پروڈکشن ڈپلائمنٹس کے لیے تصدیق نافذ کریں (مثلاً API keys، OAuth)۔
- **CORS**: رسائی کو محدود کرنے کے لیے کراس-اورجن ریسورس شیئرنگ (CORS) پالیسیاں ترتیب دیں۔
- **HTTPS**: ٹریفک کو انکرپٹ کرنے کے لیے پروڈکشن میں HTTPS استعمال کریں۔

### بہترین طریقے

- بغیر تصدیق کے آنے والی درخواستوں پر کبھی اعتماد نہ کریں۔
- تمام رسائی اور غلطیوں کو لاگ کریں اور مانیٹر کریں۔
- سیکیورٹی کمزوریوں کو دور کرنے کے لیے باقاعدگی سے ڈیپنڈینسز کو اپ ڈیٹ کریں۔

### چیلنجز

- ترقی کی آسانی اور سیکیورٹی کے درمیان توازن قائم کرنا
- مختلف کلائنٹ ماحولیات کے ساتھ مطابقت کو یقینی بنانا

## SSE سے Streamable HTTP پر اپ گریڈ کرنا

ان ایپلیکیشنز کے لیے جو اس وقت Server-Sent Events (SSE) استعمال کر رہی ہیں، Streamable HTTP پر منتقل ہونا آپ کے MCP اطلاق میں بہتر صلاحیتیں اور طویل مدتی پائیداری فراہم کرتا ہے۔

### اپ گریڈ کیوں کریں؟

SSE سے Streamable HTTP پر اپ گریڈ کرنے کی دو مضبوط وجوہات ہیں:

- Streamable HTTP SSE سے بہتر اسکیل ایبلیٹی، مطابقت اور زیادہ جامع نوٹیفیکیشن سپورٹ فراہم کرتا ہے۔
- یہ نئے MCP ایپلیکیشنز کے لیے تجویز کردہ ٹرانسپورٹ ہے۔

### منتقلی کے مراحل

آپ اپنی MCP ایپلیکیشنز میں SSE سے Streamable HTTP پر منتقلی اس طرح کر سکتے ہیں:

- **سرور کوڈ کو اپ ڈیٹ کریں** تاکہ `mcp.run()` میں `transport="streamable-http"` استعمال ہو۔
- **کلائنٹ کوڈ کو اپ ڈیٹ کریں** تاکہ SSE کلائنٹ کے بجائے `streamablehttp_client` استعمال ہو۔
- **کلائنٹ میں پیغام ہینڈلر نافذ کریں** جو نوٹیفیکیشنز کو پراسیس کرے۔
- **موجودہ ٹولز اور ورک فلو کے ساتھ مطابقت کے لیے ٹیسٹ کریں**۔

### مطابقت برقرار رکھنا

منتقلی کے دوران موجودہ SSE کلائنٹس کے ساتھ مطابقت برقرار رکھنا تجویز کیا جاتا ہے۔ یہاں کچھ حکمت عملیاں ہیں:

- آپ دونوں SSE اور Streamable HTTP کو مختلف اینڈپوائنٹس پر چلا کر سپورٹ کر سکتے ہیں۔
- کلائنٹس کو نئے ٹرانسپورٹ کی طرف بتدریج منتقل کریں۔

### چیلنجز

منتقلی کے دوران درج ذیل چیلنجز کو حل کرنا ضروری ہے:

- یہ یقینی بنانا کہ تمام کلائنٹس اپ ڈیٹ ہو گئے ہیں
- نوٹیفیکیشن ڈیلیوری میں فرق کو سنبھالنا

## سیکیورٹی کے پہلو

سیکیورٹی کسی بھی سرور کو نافذ کرتے وقت اولین ترجیح ہونی چاہیے، خاص طور پر جب MCP میں HTTP بیسڈ ٹرانسپورٹس جیسے Streamable HTTP استعمال کی جا رہی ہوں۔

MCP سرورز کو HTTP بیسڈ ٹرانسپورٹس کے ساتھ نافذ کرتے ہوئے سیکیورٹی ایک اہم مسئلہ ہے جس میں متعدد حملوں کے زاویوں اور حفاظتی طریقوں پر باریک بینی سے توجہ دینی ہوتی ہے۔

### جائزہ

MCP سرورز کو HTTP پر ایکسپوز کرتے وقت سیکیورٹی انتہائی اہم ہے۔ Streamable HTTP نئے حملوں کے مواقع فراہم کرتا ہے اور محتاط ترتیب کی ضرورت ہوتی ہے۔

یہاں کچھ اہم سیکیورٹی پہلو ہیں:

- **Origin ہیڈر کی تصدیق**: DNS ری بانڈنگ حملوں سے بچنے کے لیے ہمیشہ `Origin` ہیڈر کی تصدیق کریں۔
- **لوکل ہوسٹ بائنڈنگ**: مقامی ترقی کے لیے، سرورز کو `localhost` پر بائنڈ کریں تاکہ انہیں پبلک انٹرنیٹ پر ظاہر ہونے سے بچایا جا سکے۔
- **تصدیق**: پروڈکشن ڈپلائمنٹس کے لیے تصدیق نافذ کریں (مثلاً API keys، OAuth)۔
- **CORS**: رسائی کو محدود کرنے کے لیے کراس-اورجن ریسورس شیئرنگ (CORS) پالیسیاں ترتیب دیں۔
- **HTTPS**: ٹریفک کو انکرپٹ کرنے کے لیے پروڈکشن میں HTTPS استعمال کریں۔

### بہترین طریقے

مزید یہ کہ، جب آپ اپنے MCP اسٹریمنگ سرور میں سیکیورٹی نافذ کر رہے ہوں تو درج ذیل بہترین طریقے اپنائیں:

- بغیر تصدیق کے آنے والی درخواستوں پر کبھی اعتماد نہ کریں۔
- تمام رسائی اور غلطیوں کو لاگ کریں اور مانیٹر کریں۔
- سیکیورٹی کمزوریوں کو دور کرنے کے لیے باقاعدگی سے ڈیپنڈینسز کو اپ ڈیٹ کریں۔

### چیلنجز

MCP اسٹریمنگ سرورز میں سیکیورٹی نافذ کرتے وقت آپ کو کچھ چیلنجز کا سامنا ہوگا:

- ترقی کی آسانی اور سیکیورٹی کے درمیان توازن قائم کرنا
- مختلف کلائنٹ ماحولیات کے ساتھ مطابقت کو یقینی بنانا

### اسائنمنٹ: اپنا خود کا اسٹریمنگ MCP ایپ بنائیں

**صورتحال:**
ایک MCP سرور اور کلائنٹ بنائیں جہاں سرور اشیاء کی فہرست (مثلاً فائلیں یا دستاویزات) کو پروسیس کرے اور ہر پروسیس کی ہوئی شے کے لیے نوٹیفیکیشن بھیجے۔ کلائنٹ کو چاہیے کہ وہ ہر آنے والی نوٹیفیکیشن کو فوراً دکھائے۔

**مراحل:**

1. ایک سرور ٹول نافذ کریں جو فہرست کو پروسیس کرے اور ہر شے کے لیے نوٹیفیکیشن بھیجے۔
2. ایک کلائنٹ نافذ کریں جس میں نوٹیفیکیشنز کو حقیقی وقت میں دکھانے کے لیے پیغام ہینڈلر ہو۔
3. اپنا نفاذ ٹیسٹ کریں، سرور اور کلائنٹ دونوں چلائیں اور نوٹیفیکیشنز کو ملاحظہ کریں۔
