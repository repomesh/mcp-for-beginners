# សុវត្ថិភាព MCP៖ ការការពារយ៉ាងទូលំទូលាយសម្រាប់ប្រព័ន្ធ AI

[![ការអនុវត្តការសុវត្ថិភាពនៅ MCP](../../../translated_images/km/03.175aed6dedae133f.webp)](https://youtu.be/88No8pw706o)

_(ចុចរូបភាពខាងលើដើម្បីមើលវីដេអូមេរៀននេះ)_

សុវត្ថិភាពគឺជាគ្រឹះមួយសម្រាប់ការរចនាប្រព័ន្ធ AI ដែលជាហេតុផលដែលយើងផ្តល់អាទិភាពជាផ្នែកទីពីររបស់យើង។ វាស្របនឹងគោលការណ៍ **Secure by Design** របស់ Microsoft ពី [កិច្ចប្រឹងប្រែងអនាគតសុវត្ថិភាព](https://www.microsoft.com/security/blog/2025/04/17/microsofts-secure-by-design-journey-one-year-of-success/)។

ពិធីការណ៍ Model Context Protocol (MCP) ផ្តល់នូវសមត្ថភាពថ្មីទាក់ទាញសម្រាប់កម្មវិធីដែលមានកម្មវិធី AI ដំណើរការ ខណៈដែលនាំមកនូវបញ្ហាសុវត្ថិភាពចម្រុះដែលលើសពីហានិភ័យកម្មវិធីធម្មតា។ ប្រព័ន្ធ MCP ប្រឈមមុខនឹងបញ្ហាសុវត្ថិភាពដែលបានបង្កើតជាមុន (កូដសុវត្ថិភាព, បញ្ជីប្រើប្រាស់កំណត់អតិថិជន, សុវត្ថិភាពខ្សែភ្លើងផ្គត់ផ្គង់) និងគំរាមកំហែងពិសេស AI ថ្មីៗរួមទាំងការចាក់បញ្ចូលបញ្ជា (prompt injection), ការបំពុលឧបករណ៍, ការចាប់យកសម័យ, ការវាយប្រហារប្រភេទ confused deputy, ចំណុចខ្សោយបញ្ជូន​ token, និងការផ្លាស់ប្តូរសមត្ថភាពមានបន្ទាន់។

មេរៀននេះស្វែងយល់អំពីហានិភ័យសុវត្ថិភាពសំខាន់ៗក្នុងការអនុវត្ត MCP — រាប់បញ្ចូលការផ្ទៀងផ្ទាត់អត្តសញ្ញាណ, អាជ្ញាបណ្ណ, សិទ្ធិលើកលែង, ការចាក់បញ្ចូលបញ្ជាមិនផ្ទាល់, សុវត្ថិភាពសម័យ, បញ្ហា confused deputy, ការគ្រប់គ្រង token និងហានិភ័យខ្សែភ្លើងផ្គត់ផ្គង់។ អ្នកនឹងរៀនគ្រប់គ្រងហានិភ័យនិងអនុវត្តការអនុវត្តល្អបំផុតដើម្បីបន្ថយហានិភ័យនៅពេលប្រើប្រាស់ដំណោះស្រាយ Microsoft ដូចជា Prompt Shields, Azure Content Safety និង GitHub Advanced Security ដើម្បីពង្រឹងការចាត់ចែង MCP របស់អ្នក។

## គោលបំណងរៀន

នៅចុងបញ្ចប់មេរៀននេះ អ្នកនឹងអាច៖

- **កំណត់គំរាមកំហែងពិសេស MCP**៖ ស្គាល់ហានិភ័យសុវត្ថិភាពផ្ទាល់ខ្លួននៅប្រព័ន្ធ MCP រួមទាំងការចាក់បញ្ចូលបញ្ជា, ការបំពុលឧបករណ៍, សិទិ្ធលើកលែង, ការចាប់យកសម័យ, បញ្ហា confused deputy, ចំណុចខ្សោយបញ្ជូន token និងហានិភាពខ្សែភ្លើងផ្គត់ផ្គង់
- **អនុវត្តការគ្រប់គ្រងសុវត្ថិភាព**៖ ប្រើវិធានការការពារដែលមានប្រសិទ្ធភាព រួមមានការផ្ទៀងផ្ទាត់អត្តសញ្ញាណខ្លាំង, ការចូលប្រើក្រោមកម្រិត, ការគ្រប់គ្រង token សុវត្ថិភាព, ការគ្រប់គ្រងសុវត្ថិភាពសម័យ និងការពិនិត្យហេតុការណ៍ខ្សែភ្លើងផ្គត់ផ្គង់
- **ប្រើប្រាស់ដំណោះស្រាយសុវត្ថិភាព Microsoft**៖ យល់និងដំឡើង Microsoft Prompt Shields, Azure Content Safety និង GitHub Advanced Security សម្រាប់ការពារការងារ MCP
- **ផ្ទៀងផ្ទាត់សុវត្ថិភាពឧបករណ៍**៖ ស្គាល់សារៈសំខាន់នៃការផ្ទៀងផ្ទាត់មេតាដាតាឧបករណ៍, ត្រួតពិនិត្យការផ្លាស់ប្តូរមានបន្ទាន់ និងការពារការចាក់បញ្ចូលបញ្ជា មិនផ្ទាល់
- **បញ្ចូលអនុវត្តល្អបំផុត**៖ បញ្ចូលមូលដ្ឋានសុវត្ថិភាពដែលបានបង្កើតជាមុន (កូដសុវត្ថិភាព, ការពារផ្ទុកម៉ាស៊ីនបម្រើ, ការជឿទុកចិត្តសូន្យ) ជាមួយការគ្រប់គ្រង MCP ដើម្បីការពារយ៉ាងទូលំទូលាយ

# ស្ថាបត្យកម្ម និងការគ្រប់គ្រងសុវត្ថិភាព MCP

ការអនុវត្ត MCP ទំនើបទាមទារវិធានការសុវត្ថិភាពជាស្រទាប់ដែលគ្របដណ្តប់ទាំងការពារ​កម្មវិធី​ទំនើប និង​គំរាម​កំហែង​ពិសេស AI។ បញ្ជាក់ MCP កំពុងអភិវឌ្ឍ​យ៉ាងលឿន ដើម្បីបង្កើតការគ្រប់គ្រងសុវត្ថិភាពល្អជាងមុន ដែលអាចបញ្ចូលជាមួយស្ថាបត្យកម្មសុវត្ថិភាពសហគ្រាស និងការអនុវត្តល្អបំផុតដែលមានរួចរាល់។

ការស្រាវជ្រាវពី [របាយការណ៍ការពារឌីជីថល Microsoft](https://aka.ms/mddr) បង្ហាញថា **៩៨% នៃការបង្ហាញពីការបំពានអាចត្រូវបានទប់ស្កាត់ដោយការថែទាំសុវត្តិភាពយ៉ាងរឹងមាំ**។ វិធីសាស្រ្តការពារមានប្រសិទ្ធភាពបំផុតគឺការរួមបញ្ចូលអនុវត្តល្អបំផុតសុវត្ថិភាពជារចនាសម្ព័ន្ធជាមួយការគ្រប់គ្រង MCP — វិធានការសុវត្ថិភាពមូលដ្ឋាននៅបើកញាតិគឺមានឥទ្ធិពលបំផុតក្នុងការកាត់បន្ថយហានិភ័យសុវត្ថិភាពច្រើនទៀត។

## ទិដ្ឋភាពសុវត្ថិភាពបច្ចុប្បន្ន

> **ចំណាំ៖** ព័ត៌មាននេះបង្ហាញពីស្តង់ដារសុវត្ថិភាព MCP ដល់ថ្ងៃទី **៥ កុម្ភៈ ២០២៦**, ស្របតាម **MCP Specification 2025-11-25**។ ពិធីការណ៍ MCP កំពុងអភិវឌ្ឍយ៉ាងលឿន ហើយការអនុវត្តន៍អនាគតអាចនាំមកនូវទម្រង់ផ្ទៀងផ្ទាត់អត្តសញ្ញាណថ្មី និងការគ្រប់គ្រងកំណត់ត្រាលំដាប់ខ្ពស់។ តែងតែយោងទៅកាន់ [MCP Specification](https://spec.modelcontextprotocol.io/), [សម្រាប់ MCP GitHub repository](https://github.com/modelcontextprotocol), និង [ឯកសារអនុវត្តល្អបំផុតក្នុងការសុវត្ថិភាព](https://modelcontextprotocol.io/specification/2025-11-25/basic/security_best_practices) សម្រាប់ការណែនាំថ្មីៗ។

> **មើលទៅខាងមុខ៖** ប្រភព candidate លក់ចេញ `2026-07-28` បានបង្កប់ការកំចាត់អោយបានត្រឹមត្រូវទៀត — អតិថិជនត្រូវបញ្ជាក់ប៉ារ៉ាម៉ែត្រ `iss` នៅលើការឆ្លើយតប authorization (RFC 9207), បញ្ចាក់ `application_type` OpenID Connect នៅពេល Dynamic Client Registration, និងភ្ជាប់លិខិតសម្គាល់ដែលបានចុះបញ្ជាក់ទៅម៉ាស៊ីនបម្រើ authorization ដែលបានចេញ។ សូមមើល [អ្វីដែលកំពុងផ្លាស់ប្តូរនៅ MCP: រូបមន្តចេញ candidate 2026-07-28](../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) សំរាប់បញ្ជីពេញលេញនៃ SEPs authorization។

## 🏔️ សិក្ខាសាលា MCP Security Summit (Sherpa)

សម្រាប់ **ការបណ្ដុះបណ្ដាលសុវត្ថិភាពដោយផ្ទាល់**, យើងសូមណែនាំយ៉ាងខ្លាំងសម្រាប់ **សិក្ខាសាលា MCP Security Summit** (Sherpa) - ដែលជាការផ្លាស់ទីមានមគ្គុទេសក៍ពេញលេញ ដើម្បីការពារម៉ាស៊ីនបម្រើ MCP ក្នុង Microsoft Azure។

### ទិដ្ឋភាពសិក្ខាសាលា

[សិក្ខាសាលា MCP Security Summit](https://azure-samples.github.io/sherpa/) ផ្តល់ការបណ្ដុះបណ្ដាលសុវត្ថិភាពជាការអនុវត្តន៍តាមវិធី "ខូច → ប្រើកំណត់ → ដោះស្រាយ → ផ្ទៀងផ្ទាត់" ដែលបានបញ្ជាក់។ អ្នកនឹង៖

- **រៀនដោយការបំបែករឿង**៖ បទពិសោធន៍ហានិភ័យដោយផ្ទាល់ដោយសារម៉ាស៊ីនបម្រើដែលមានឯកសារកំហុស
- **ប្រើប្រាស់សុវត្ថិភាពធម្មជាតិ Azure**៖ ប្រើប្រាស់ Azure Entra ID, Key Vault, API Management និង AI Content Safety
- **អនុវត្តការពាររួមបញ្ចូលជាស្រទាប់**៖ តាមដានតាមជំរឿនដំណាក់កាលដែលបង្កើតស្រទាប់សុវត្ថិភាពទូលំទូលាយ
- **អនុវត្តវិធានការ OWASP**៖ វិធីសាស្រ្តនីមួយៗផ្គូរផ្គងទៅ [OWASP MCP Azure Security Guide](https://microsoft.github.io/mcp-azure-security-guide/)
- **ទទួលបានកូដផលិតកម្ម**៖ ទៅជាមួយកូដដែលដំណើរការ និងបានគេចាត់ទុករួច

### ការ Evolution នៃការផ្ទៀងផ្ទាត់ MCP

បញ្ជាក់ MCP បានអភិវឌ្ឍយ៉ាងច្រើនក្នុងវិធីសាស្រ្តផ្ទៀងផ្ទាត់អត្តសញ្ញាណ និងអនុញ្ញាត៖

- **វិធីដើម**៖ បញ្ជាក់ដំបូងត្រូវការអ្នកអភិវឌ្ឍអនុវត្តម៉ាស៊ីនបម្រើផ្ទៀងផ្ទាត់ផ្ទាល់ខ្លួន នៅពេលម៉ាស៊ីនបម្រើ MCP ធ្វើជាម៉ាស៊ីនបម្រើ OAuth 2.0 Authorization Server ដើម្បីគ្រប់គ្រងការផ្ទៀងផ្ទាត់អ្នកប្រើដោយផ្ទាល់
- **ស្តង់ដារបច្ចុប្បន្ន (2025-11-25)**៖ បញ្ជាក់បានធ្វើបច្ចុប្បន្នភាពអនុញ្ញាតឲ្យម៉ាស៊ីនបម្រើ MCP ផ្គាប់ការផ្ទៀងផ្ទាត់ទៅអ្នកផ្តល់អត្តសញ្ញាណខាងក្រៅ (ដូចជា Microsoft Entra ID), បង្កើនភាពសុវត្ថិភាព និងកាត់បន្ថយភាពស្មុគស្មាញនៃការអនុវត្ត
- **សុវត្ថិភាពស្រទាប់ដឹកជញ្ជូន**៖ បង្កើនការគាំទ្រសម្រាប់មេកានិកដឹកជញ្ជូនសុវត្ថិភាពជាមួយនឹងបែបបទផ្ទៀងផ្ទាត់ឲ្យបានត្រឹមត្រូវ រួមទាំងភ្ជាប់ STDIO លើកន្លែង និង Streamable HTTP ចេញក្រៅ

## សុវត្ថិភាពផ្ទៀងផ្ទាត់អត្តសញ្ញាណ និងអនុញ្ញាត

### បញ្ហាសុវត្ថិភាពបច្ចុប្បន្ន

ការអនុវត្ត MCP ទំនើបប្រឈមមុខបញ្ហាច្រើនទាក់ទងនឹងការផ្ទៀងផ្ទាត់អត្តសញ្ញាណ និងអនុញ្ញាត៖

### ហានិភ័យ និង វ៉ិចទ័រ​គំរាមកំហែង

- **ការគ្រប់គ្រងអនុញ្ញាតមិនត្រឹមត្រូវ**៖ ការអនុវត្តអនុញ្ញាតខូចនៅម៉ាស៊ីនបម្រើ MCP អាចបង្ហាញនូវទិន្នន័យឯកជន និងអនុវត្តការត្រួតពិនិត្យការចូលដំណើរការមិនត្រឹមត្រូវ
- **OAuth Token ត្រូវបានបន្លំ**៖ ការលួច token នៅម៉ាស៊ីនបម្រើ MCP ផ្គូរផ្គងឱ្យអ្នកវាយប្រហារអាចបង្ហាញជាម៉ាស៊ីនបម្រើ និងចូលប្រើសេវាកម្មខាងក្រោម
- **ចំណុចខ្សោយ Token Passthrough**៖ ការគ្រប់គ្រង token ខ្វះខាតបង្កើតផ្លូវរត់លុបបំបាត់ការគ្រប់គ្រងសុវត្ថិភាព និងចន្លោះបញ្ចាក់
- **សិទ្ធិលើកលែងច្រើនពេក**៖ ម៉ាស៊ីនបម្រើ MCP ដែលមានសិទ្ធិលើកលែងអតិបរមាជ្រៀតជ្រែកនូវគោលការណ៍សិទ្ធិអប្បបរមា និងពង្រីកផ្ទៃសកម្មភាព

#### Token Passthrough៖ គំរូរំលងសុវត្ថិភាពសំខាន់

**Token passthrough ត្រូវបានហាមឃាត់យ៉ាងច្បាស់** នៅក្នុងបញ្ជាក់ MCP authorization បច្ចុប្បន្នដោយសារវាប៉ះពាល់សុវត្ថិភាពធ្ងន់ធ្ងរ៖

##### ការលុបបំបាត់វិនិយោគសុវត្ថិភាព
- ម៉ាស៊ីនបម្រើ MCP និង APIs ខាងក្រោមអនុវត្តការគ្រប់គ្រងសុវត្ថិភាពសំខាន់ៗ (កំណត់កម្រិតអត្រា, ផ្ទៀងផ្ទាត់សំណើ, ត្រួតពិនិត្យចរន្ត) ដែលពឹងផ្អែកលើការផ្ទៀងផ្ទាត់ token យ៉ាងត្រឹមត្រូវ
- ការប្រើ token ពីអ្នកអតិថិជនទៅ API តាមផ្លូវបញ្ច្រាសដល់ការការពារ, បំផ្លាញស្ថាបត្យកម្មសុវត្ថិភាព

##### បញ្ហាការទទួលខុសត្រូវ និងការត្រួតពិនិត្យ
- ម៉ាស៊ីនបម្រើ MCP មិនអាចបំបែករវាងអតិថិជនប្រើ token ផ្តល់ពី upstream បាន, បំបែកស្នាម audit
- កំណត់ត្រាម៉ាស៊ីនបម្រើធនធានខាងក្រោមបង្ហាញកំណត់ត្រាមិនត្រឹមត្រូវទាំងពីរគោលដៅប្ដូរជាជាងម៉ាស៊ីនបម្រើ MCP
- ការស៊ើបអង្កេតហេតុការណ៍ និង audit ស្របច្បាប់ក្លាយជាមានឥរិយាបថពិបាក

##### ហានិភ័យដកប្រាក់ទិន្នន័យ
- ការទាមទារតែមិនត្រឹមត្រូវនៃ token អនុញ្ញាតឱ្យអ្នកវាយប្រហារទាមទារប្រើ token លួចដើម្បីប្រើម៉ាស៊ីនបម្រើ MCP ដើម្បីដកប្រាក់ទិន្នន័យ
- ការបំពានកំណត់ដែនទំនុកចិត្តអនុញ្ញាតឱ្យមានការចូលដំណើរការមិនអាជ្ញាបណ្ណដែលលើកលែងការគ្រប់គ្រងសុវត្ថិភាពបានគ្រោងទុក

##### វិធីសាស្រ្តវាយប្រហារច្រើនសេវាកម្ម
- token ដែលបំពុលអាចត្រូវបានទទួលយកដោយសេវាកម្មច្រើនពីរបាន អនុញ្ញាតឱ្យចលនា ទៅមុខចន្លោះប្រព័ន្ធតភ្ជាប់
- ការសន្មត់ទុកចិត្តរវាងសេវាកម្មអាចត្រូវបានបំពាន ប្រសិនបើមិនអាចផ្ទៀងផ្ទាត់ដើម token បាន

#### ការអនុវត្តការត្រួតពិនិត្យការចូលប្រើ

  - ការត្រួតពិនិត្យនិងធ្វើបច្ចុប្បន្នក្នុងសិទ្ធិជារឿយៗ ដើម្បីទប់ស្កាត់ការលេចចេញសិទ្ធិ

  - ជៀសវាងសិទ្ធិទូលំទូលាយ ឬមិនចាំបាច់ដែលបង្កើនផ្ទៃហាតដំណើរការ

  - ដោះស្រាយបញ្ហាសិទ្ធិលើកលែង ឬមិនប្រើប្រាស់ភ្លាមភ្លារពេល

## គំរាមកំហែងសុវត្ថិភាពពិសេស AI

### ការចាក់បញ្ចូលបញ្ជា និងការវាយប្រហារតាមឧបករណ៍

ការអនុវត្ត MCP ទំនើបប្រឈមមុខជាមួយវិថីសញ្ញាប្រយុទ្ធពិសេស AI ដែលការពារទម្រង់សុវត្ថិភាពបែបលើកមុខមិនអាចដោះស្រាយបានទាំងមូល៖

#### **ការចាក់បញ្ចូលបញ្ជាមិនផ្ទាល់ (Indirect Prompt Injection / Cross-Domain Prompt Injection)**

**ការចាក់បញ្ចូលបញ្ជាមិនផ្ទាល់** តំណាងឱ្យហានិភ័យមួយក្នុងប្រព័ន្ធ AI ដែលគាំទ្រ MCP។ អ្នកវាយប្រហារបញ្ចូលបញ្ជារប្រយុទ្ធនៅក្នុងមាតិកាប្រភពខាងក្រៅ — ឯកសារ, ទំព័របណ្ដាញ, អ៊ីមែល ឬប្រភពទិន្នន័យ — ដែលប្រព័ន្ធ AI បន្ទាប់មកប្រើឱ្យជាការណែនាំត្រឹមត្រូវ។

**សถานการณ์វាយប្រហារ៖**
- **ការចាក់បញ្ចូលនៅឯកសារ**៖ បញ្ជារប្រយុទ្ធសុំពីកន្លែងមួយដាក់លាក់នៅក្នុងឯកសារដែលបានដំណើរការនាំឱ្យសកម្មភាព AI មិនបានគ្រោងទុក
- **ការបំពុលមាតិកាវេបសាយ**៖ ទំព័របណ្ដាញដែលត្រូវបានគេចោទថាបញ្ចូលបញ្ជារប្រយុទ្ធដែលគ្រប់គ្រងអាកប្បកិរិយា AI នៅពេល scraping
- **ការវាយប្រហារពីអ៊ីមែល**៖ បញ្ជារប្រយុទ្ធនៅអ៊ីមែលដែលធ្វើឲ្យជំនួយ AI រអិលព័តមានឬអនុវត្តសកម្មភាពមិនអនុញ្ញាត
- **ការបំពុលប្រភពទិន្នន័យ**៖ មូលដ្ឋានទិន្នន័យ ឬ API ដែលផ្គត់ផ្គង់មាតិកាបំពុលទៅប្រព័ន្ធ AI

**ផលប៉ះពាល់ក្នុងពិភពពិត**៖ ការវាយប្រហារទាំងនេះអាចបណ្តាលឲ្យមានការដកស្រង់ទិន្នន័យ, ការបំពានអត្តសញ្ញាណ, ការបង្កើតមាតិកាអាក្រក់ និងការគ្រប់គ្រងអន្តរកម្មអ្នកប្រើ។ សម្រាប់វិភាគលំអិត សូមមើល [ការចាក់បញ្ចូលបញ្ជាក្នុង MCP (Simon Willison)](https://simonwillison.net/2025/Apr/9/mcp-prompt-injection/)។

#### **ការវាយប្រហារបំពុលឧបករណ៍ (Tool Poisoning Attacks)**

**ការបំពុលឧបករណ៍** ទៅលើមេតាដាតាដែលកំណត់ឧបករណ៍ MCP ប្រើប្រាស់ ដោយអាស្រ័យលើរបៀបដែល LLM អានពិពណ៌នាឧបករណ៍ និងប៉ារ៉ាម៉ែត្រដើម្បីធ្វើសេចក្ដីសម្រេចអនុវត្ត។

**មេកានិចវាយប្រហារ៖**
- **ការបំភ្លោះមេតាដាតា**៖ អ្នកវាយប្រហារចាក់បញ្ចូលបញ្ជារប្រយុទ្ធនៅក្នុងពិពណ៌នាឧបករណ៍, កំណត់ប៉ារ៉ាម៉ែត្រ ឬឧទាហរណ៍ប្រើប្រាស់
- **បញ្ជារមានលក្ខណៈមិនច្បាស់**៖ បញ្ជារប្រយុទ្ធនៅក្នុងមេតាដាតាឧបករណ៍ ដែលត្រូវបានដំណើរការដោយម៉ូដែល AI ប៉ុន្តែមិនមែនមនុស្សមើលឃើញ
- **ការផ្លាស់ប្តូរឧបករណ៍មានបន្ទាន់ ("Rug Pulls")**៖ ឧបករណ៍ដែលបានអនុម័តដោយអ្នកប្រើ ក្រោយពីនេះត្រូវបានបំផ្លាញឲ្យអនុវត្តសកម្មភាពប្រយុទ្ធដោយគ្មានការយល់ដឹងរបស់អ្នកប្រើ
- **ការចាក់បញ្ជាពិពណ៌នា Parameter**៖ មាតិកាប្រយុទ្ធបានបញ្ចូលក្នុងស្ឈេម៉ាប៉ារ៉ាម៉ែត្របស់ឧបករណ៍ ដែលមានឥទ្ធិពលលើអាកប្បកិរិយាម៉ូដែល


**ហានិភ័យម៉ាស៊ីនមេផ្អែកលើម៉ាស៊ីនមេ**៖ ម៉ាស៊ីនមេ MCP ចម្ងាយបង្ហាញពីហានិភ័យកើនឡើង ព្រោះពណ៌នាឧបករណ៍អាចត្រូវបានធ្វើបច្ចុប្បន្នភាពបន្ទាប់ពីការអនុម័តដំបូងពីអ្នកប្រើ បង្កើតឱកាសដែលឧបករណ៍ដែលមានសុវត្ថិភាពពីមុនក្លាយជាឧបករណ៍អាក្រក់។ សម្រាប់ការវិភាគយ៉ាងទូលំទូលាយ សូមមើល [ការវាយប្រហារបំពុលឧបករណ៍ (Invariant Labs)](https://invariantlabs.ai/blog/mcp-security-notification-tool-poisoning-attacks)។

#### **វិធីវាយប្រហារបន្ថែម AI**

- **ការបញ្ចូលដើម្បីបង្កកខុសតាមដែនកំណត់ (XPIA)**៖ វិធីវាយប្រហារដែលមានជំនាញខ្ពស់ដែលប្រើអត្ថបទពីដែនជាច្រើនដើម្បីលោភលន់ការត្រួតពិនិត្យសុវត្ថិភាព
- **ការផ្លាស់ប្តូរ​សមត្ថភាព​ដោយឡែក (Dynamic Capability Modification)**៖ ការផ្លាស់ប្តូរពេលវេលា​ពិត​នៃសមត្ថភាពឧបករណ៍ដែលគេចខ្លួនការវាយតម្លៃសុវត្ថិភាពដំបូង
- **ការបំពុលបង្ហាញបរិបទ (Context Window Poisoning)**៖ ការវាយប្រហារដែលបំលែងបង្ហាញបរិបទធំដើម្បីលាក់បញ្ជាពីផ្សេងគ្នាដែលមិនល្អ
- **ការវាយប្រហារអោយម៉ូដែលស្រឡាំង (Model Confusion Attacks)**៖ 利用កំណត់កម្រិតម៉ូដែលដើម្បីបង្កើតអត្តចរណ៍មិនអាចគួរឱ្យទុកចិត្តឬមិនសុវត្ថិភាព


### ផលប៉ះពាល់ហានិភ័យសុវត្ថិភាព AI

**ផលគ្រោះថ្នាក់ខ្ពស់៖**
- **ការចោលទិន្នន័យ (Data Exfiltration)**៖ ការចូលប្រើ និងលួចទិន្នន័យសំខាន់អ្នកជំនួញឬផ្ទាល់ខ្លួនដែលមិនបានអនុញ្ញាត
- **បំលែងឯកជនភាព (Privacy Breaches)**៖ ការបង្ហាញព័ត៌មានអត្តសញ្ញាណផ្ទាល់ខ្លួន (PII) និងទិន្នន័យអាជីវកម្មសម្ងាត់
- **ការបំផ្លាស់ប្ដូរប្រព័ន្ធ (System Manipulation)**៖ ការផ្លាស់ប្ដូរឥតបំណុលទៅលើប្រព័ន្ធនិងដំណើរការសំខាន់
- **ការលួចគ្រឿងស្គាល់ (Credential Theft)**៖ ការបំផ្លាញសញ្ញាប័ណ្ណផ្ទៀងផ្ទាត់ និងគ្រឿងស្គាល់សេវាកម្ម
- **ចលនាប្រព័ន្ធបណ្តាញអេឡិចត្រូនិច (Lateral Movement)**៖ ការប្រើប្រាស់ប្រព័ន្ធ AI ដែលត្រូវបានបំផ្លាញជាគន្លងសម្រាប់ការវាយប្រហារបណ្តាញទូលំទូលាយ

### ដំណោះស្រាយសុវត្ថិភាព Microsoft AI

#### **សញ្ឈឹមការចូលបញ្ចូល AI: ការការពារដ៏ខ្ពស់ប្រឆាំងការវាយប្រហារបញ្ចូល**

Microsoft **សញ្ឈឹមការចូលបញ្ចូល AI** ផ្តល់ការពារយ៉ាងទូលំទូលាយប្រឆាំងការវាយប្រហារបញ្ចូលផ្ទាល់ និងអូសកម្ម តាមរយៈស្រទាប់សុវត្ថិភាពជាច្រើន៖

##### **យន្តការការពារចម្បង៖**

1. **ការរកឃើញ និងតម្រងខ្ពស់**
   - 알고리즘ຮຽນម៉ាស៊ីន និងបច្ចេកទេស NLP ស្វែងរកបញ្ជាសង្ស័យក្នុងមាតិកាផ្ទៃក្រៅ
   - ការវិភាគពេលវេលា​ពិត នៃឯកសារ ទំព័របណ្ដាញ អ៊ីមែល និងប្រភពទិន្នន័យសម្រាប់គ្រោះថ្នាក់
   - ការយល់ដឹងរបៀបបញ្ជាបង្កើតសុវត្ថិភាព និងបញ្ជាប្រកបដោយហានិភ័យ

2. **វិធីសាស្រ្តបង្ហាញចំណុចសំខាន់**  
   - សម្គាល់ចន្លោះបញ្ជាសន្ធិសញ្ញាទៅនឹងការបញ្ចូលពីខាងក្រៅដែលអាចមានការបំផ្លាញ
   - វិធីសាស្រ្តបំលែងអត្ថបទ ដែលបង្កើនសមត្ថភាពម៉ូដែល ខណៈបំបែកមាតិកាអាក្រក់
   - ជួយប្រព័ន្ធ AI រក្សាសន្ទស្សន៍បញ្ជាទាន់តែត្រឹមត្រូវ និងមិនបញ្ជាទៅកាន់បញ្ជា​ដែលបានបញ្ចូល

3. **ប្រព័ន្ធចំណុចកំណត់ និងសញ្ញាកំណត់ទិន្នន័យ**
   - កំណត់ដែនកំណត់ច្បាស់លាស់រវាងសារប្រព័ន្ធដែលទុកចិត្ត និងអត្ថបទបញ្ចូលខាងក្រៅ
   - សញ្ញាពិសេសបង្ហាញចន្លោះរវាងប្រភពទិន្នន័យដែលទុកចិត្ត និងមិនទុកចិត្ត
   - ការបំបែកច្បាស់លាស់ បាញ់ការច្រឡំការបញ្ជា និងការអនុវត្តបញ្ជាដែលមិនឱ្យអនុញ្ញាត

4. **បញ្ហាបញ្ចេញបញ្ហាសង្គ្រោះជាបន្តបន្ទាប់**
   - Microsoft តែងតែរំពឹងមើលលំនាំវាយប្រហារថ្មីៗ និងធ្វើបច្ចុប្បន្នភាពការពារ
   - បើកប្រមាញ់គ្រោះថ្នាក់មុនគេ សម្រាប់វិធីវាយប្រហារបញ្ចូលានិងវិធីវាយប្រហារថ្មីៗ
   - ធ្វើបច្ចុប្បន្នភាពម៉ូដែលសុវត្ថិភាពជាប្រចាំ ដើម្បីរក្សាភាពមានប្រសិទ្ធិភាពប្រឆាំងនឹងហានិភ័យច្នៃប្រឌិត

5. **ការរួមបញ្ចូល Azure Content Safety**
   - ជាផ្នែកមួយនៃកញ្ចប់ Azure AI Content Safety ដ៏ទូលំទូលាយ
   - ការស្វែងរកបន្ថែមសម្រាប់ការក្លាយចេញពីការព្យាយាម jailbreak ការមានមាតិការអាក្រក់ និងការបំពាននយោបាយសុវត្ថិភាព
   - ការត្រួតពិនិត្យសុវត្ថិភាពរួមគ្នាក្នុងគ្រប់គ្រងជាភាសាដំណើរការអភិវឌ្ឍន៍ AI

## ហានិភ័យសុវត្ថិភាព MCP ជាន់ខ្ពស់

### ហានិភ័យការចូលអំណាចប្រើសម័យសម័យ

**ការចូលអំណាចប្រើសម័យសម័យ** ជាវិធីវាយប្រហារសំខាន់ក្នុងការអនុវត្ត MCP ប្រភេទមានរដ្ឋភាពដែលអ្នកមិនអនុញ្ញាតយកលេខសម្គាល់សម័យសម័យដែលត្រឹមត្រូវ ហើយប្រើវាទទួលបានអត្តសញ្ញាណដើម្បីធ្វើសកម្មភាពដែលមិនបានហាមឃាត់។

#### **ទិដ្ឋភាពវាយប្រហារ និងហានិភ័យ**

- **ការចូលបញ្ចូលបន្លំសម័យសម័យ**៖ អ្នកវាយប្រហារដែលមានលេខសម្គាល់សម័យសម័យដែលបានលួចបញ្ចូលព្រឹត្តិការណ៍អាក្រក់ទៅម៉ាស៊ីនមេចែករំលែកសភាពសម័យ, អាចបង្ករអោយមានសកម្មភាពបោកប្រាស់ ឬចូលដំណើរការទិន្នន័យសំងាត់
- **ការបន្លំផ្ទាល់ខ្លួន**៖ លេខសម្គាល់សម័យសម័យដែលបានលួចអាចធ្វើអោយហៅម៉ាស៊ីនមេ MCP ដោយផ្ទាល់ដែលហួសសុវត្ថិភាព ហើយព្យួរឱ្យអ្នកវាយប្រហារជាអ្នកប្រើត្រឹមត្រូវ
- **ប្រព័ន្ធច្រកចូលបន្តដែលបានបំពុល**៖ អ្នកវាយប្រហារអាចបញ្ចប់សំណើមុនពេលវេលា ធ្វើឲ្យអតិថិជនត្រឹមត្រូវបន្តប្រើមេរោគដែលអាចមានការបំផ្លាញ

#### **ការគ្រប់គ្រងសុវត្ថិភាពសម្រាប់ការគ្រប់គ្រងសម័យសម័យ**

**ការទាមទារសំខាន់ៗ៖**
- **ការត្រួតពិនិត្យអាជ្ញាបណ្ណ**៖ ម៉ាស៊ីនមេ MCP ដែលអនុវត្តអាជ្ញាបណ្ណ **ត្រូវតែ** ត្រួតពិនិត្យសំណើរទាំងអស់ជាប់ចូល ហើយមិនត្រូវពឹងផ្អែកលើសម័យសម័យសម្រាប់ការផ្ទៀងផ្ទាត់
- **ការបង្កើតសម័យសម័យយ៉ាងសុវត្ថិភាព**៖ ប្រើលេខសម្គាល់សម័យសម័យដែលមានសុវត្ថិភាពគ្រាប់លេខហត្ថលេខា និងកូដចៃដន្យ
- **ការចងក្រងប្រវត្តិអ្នកប្រើជាក់លាក់**៖ ចងក្រងលេខសម្គាល់សម័យជាមួយព័ត៌មានអ្នកប្រើ ដូចជា `<user_id>:<session_id>` ដើម្បីទប់ស្កាត់ការប្រើប្រាស់សម័យសម័យរវាងអ្នកប្រើផ្សេងគ្នា
- **ការគ្រប់គ្រងជីវិតសម័យសម័យ**៖ អនុវត្តពេលវេលាបញ្ចប់ ការប្ដូររៀងរាល់ និងការលុបចោលអោយបានត្រឹមត្រូវ ដើម្បីកំណត់ជំហានកំណត់ហានិភ័យ
- **សុវត្ថិភាពការបញ្ជូនទិន្នន័យ**៖ បំពេញតម្រូវការការប្រើ HTTPS សម្រាប់ទំនាក់ទំនងគ្រប់យ៉ាង ដើម្បីទប់ស្កាត់ការចាប់យកលេខសម្គាល់សម័យសម័យ

### បញ្ហា Deputy Confused

បញ្ហា **Deputy Confused** កើតឡើងនៅពេលម៉ាស៊ីនមេ MCP ធ្វើជារន្ធតំណាងផ្ទាល់ខ្លួនរវាងអតិថិជន និងសេវាកម្មភាគីទីបី បង្កើតឱកាសសម្រាប់ច្រឡំការអនុញ្ញាតតាមរយៈការប្រើពួក Client ID រឹងមាំ

#### **យន្តការវាយប្រហារ និងហានិភ័យ**

- **ការលោភលន់ដោយរាល់កូគ៊ីឯកភាព**៖ ការផ្ទៀងផ្ទាត់អ្នកប្រើមុនផលិតកូគ៊ីឯកភាពដែលអ្នកវាយប្រហារប្រើលំអៀងតាមការស្នើសុំឥណទានដែលច្រឡំជាមួយ URI បញ្ជូនឡើងវិញ
- **ការលួចកូដអនុញ្ញាត**៖ កូគ៊ីឯកភាពដែលមានរួចអាចធ្វើអោយម៉ាស៊ីនមេអនុញ្ញាតលើសកម្រិត, បញ្ជូនកូដទៅចុងផ្លូវដែលគ្រប់គ្រងដោយអ្នកវាយប្រហារ
- **ការចូលប្រើ API ឥតអាជ្ញា**៖ កូដអនុញ្ញាតដែលបានលួចធ្វើអោយអាចប្តូរលេខសម្ងាត់ និងបន្លំដូចជា​អ្នកប្រើដោយគ្មានការអនុញ្ញាតច្បាស់លាស់

#### **យុទ្ធសាស្រ្តការពារ**

**តម្រូវការត្រូវតែបំពេញ៖**
- **ការទាមទារសម្រេចចិត្តច្បាស់លាស់**៖ ម៉ាស៊ីនមេ MCP ប្រើ Client ID រឹងមាំ **ត្រូវតែ** ទទួលបានការយល់ព្រមពីអ្នកប្រើសម្រាប់ Client ដែលចុះបញ្ជី δυναμικά
- **ការអនុវត្តសុវត្ថិភាព OAuth 2.1**៖ អនុវត្តបទប្បញ្ញាតិដ៏ល្អបំផុតរបស់ OAuth រួមទាំង PKCE សម្រាប់សំណើការអនុញ្ញាតទាំងអស់
- **ការផ្ទៀងផ្ទាត់ Client មានភាពតឹងរ៉ឹង**៖ អនុវត្តការផ្ទៀងផ្ទាត់ URI បញ្ជូនឡើងវិញ និងលេខសម្គាល់ Client ដើម្បីទប់ស្កាត់ការលោភលន់

### ហានិភ័យ Token Passthrough  

**Token passthrough** ជារបៀបប្រព្រឹត្តិមិនត្រឹមត្រូវដែលម៉ាស៊ីនមេ MCP ទទួលយកតូកិន Client ដោយគ្មានការផ្ទៀងផ្ទាត់គ្រប់គ្រាន់ ហើយបញ្ជូនទៅ API ខាងក្រោមបំផ្លាញការកំណត់ MCP អនុញ្ញាត។

#### **ផលប៉ះពាល់សុវត្ថិភាព**

- **ការលោភលន់ការត្រួតពិនិត្យ**៖ ការប្រើប្រាស់​តូកិន​តាមផ្លូវ​តូចទៅ API ផ្ទាល់លុបបំបាត់ការកំណត់កម្រិតអត្រា ការត្រួតពិនិត្យ និងការតាមដាន
- **ការលបបំបែកប្រវត្តិ**៖ តូកិនដែលចេញពី upstream ធ្វើអោយមិនអាចកំណត់អត្តសញ្ញាណ Client នឹងបំបែកសមត្ថភាពស្រាវជ្រាវហេតុការណ៍
- **ការបង្ហាញទិន្នន័យតាម Proxy**៖ តូកិនមិនត្រឹមត្រូវអនុញ្ញាតឱ្យអ្នកវាយប្រហារ​ប្រើម៉ាស៊ីនមេជាគន្លងចូលដំណើរការ​ទិន្នន័យ​ឥតអនុញ្ញាត
- **ការលុបបំបែកព្រំដែនទំនុកចិត្ត**៖ សេវាកម្មខាងក្រោមអាចហួសផ្អែកលើការជឿទុកចិត្តបាន ពេលដែលមេរោគមូលដ្ឋានអត់អាចផ្ទៀងផ្ទាត់
- **ការពង្រីកវាយប្រហារជារៀងរាល់សេវាកម្ម**៖ តូកិនត្រូវបានបំពុលដែលទទួលយកក្នុងសេវាកម្មជាច្រើនអាចអនុញ្ញាតចលនាប្រព័ន្ធទូលំទូលាយ

#### **ការគ្រប់គ្រងសុវត្ថិភាពចាំបាច់**

**តម្រូវការមិនអាចចរចា៖**
- **ការផ្ទៀងផ្ទាត់តូកិន**៖ ម៉ាស៊ីនមេ MCP **មិនត្រូវ** ទទួលយកតូកិនដែលមិនត្រូវបានចេញសម្រាប់ម៉ាស៊ីនមេ MCP
- **ការផ្ទៀងផ្ទាត់អ្នកទស្សនា**៖ តែងតែផ្ទៀងផ្ទាត់អំណួតបែបអ្នកទស្សនាតូកិនឲ្យត្រូវគ្នាជាមួយអត្តសញ្ញាណម៉ាស៊ីនមេ MCP
- **ជីវិតតូកិនត្រឹមត្រូវ**៖ អនុវត្តតូកិនសុវត្ថិភាពមានអាយុកាលខ្លី ជាមួយនឹងការប្ដូរយ៉ាងសុវត្ថិភាព


## សុវត្ថិភាពខ្សែផ្គត់ផ្គង់សម្រាប់ប្រព័ន្ធ AI

សុវត្ថិភាពខ្សែផ្គត់ផ្គង់ បានរីកចម្រើនពីការពឹងផ្អែកលើកម្មវិធីទន់បែបប្រពៃណី ដល់ការលំដាប់ដោយរួមបញ្ចូលប្រព័ន្ធ AI ទាំងមូល។ ការ​អនុវត្ត MCP ទំនើប ត្រូវតែផ្ទៀងផ្ទាត់ និងត្រួតពិនិត្យគ្រប់ផ្នែកដែលទាក់ទងនឹង AI ដោយយ៉ាងតឹងរឹង ព្រោះនីមួយៗត្រូវបានផ្ដល់ឱកាសហានិភ័យដែលអាចបាក់បែកភាពរឹងមាំប្រព័ន្ធ។

### ផ្នែកខ្សែផ្គត់ផ្គង់ AI ពង្រីក

**ការពឹងផ្អែកកម្មវិធីទន់បែបប្រពៃណី៖**
- បណ្ណាល័យ និងស៊ុមផ្សេងៗដែលមានប្រភពលើកទឹកចិត្តសង្គម
- រូបតំណាងកុងតឺន័រ និងប្រព័ន្ធមូលដ្ឋាន
- ឧបករណ៍អភិវឌ្ឍន៍ និងបំពង់ចាក់បន្ថែម
- សមាសធាតុហេដ្ឋារចនាសម្ព័ន្ធ និងសេវាកម្ម

**ធាតុខ្សែផ្គត់ផ្គង់ជាក់លាក់សម្រាប់ AI៖**
- **ម៉ូដែលផ្នែកមូលដ្ឋាន**៖ ម៉ូដែលដែលបានបណ្តុះបណ្តាលមុនពីអ្នកផ្គត់ផ្គង់ផ្សេងៗ ត្រូវតែផ្ទៀងផ្ទាត់ប្រភព
- **សេវាកម្មបញ្ចូលទិន្នន័យ**៖ សេវាកម្មវែកវាយវេសិប្បកម្ម និងស្វែងរកអត្ថន័យពីក្រៅ
- **អ្នកផ្តល់បរិបទ**៖ ប្រភពទិន្នន័យ, មូលដ្ឋានចំណេះដឹង និងទីតាំងឯកសារ
- **API ភាគីទីបី**៖ សេវាកម្ម AI ខាងក្រៅ, បំពង់ ML និងចុងផ្លូវដំណើរការទិន្នន័យ
- **ឯកសារម៉ូដែល**៖ ទម្ងន់ ការកំណត់គម្រោង និងម៉ូដែលដែលបានបំណិល
- **ប្រភពទិន្នន័យបណ្តុះបណ្តាល**៖ សំណុំទិន្នន័យដែលបានប្រើសម្រាប់បណ្តុះបណ្តាល និងការបំណិលម៉ូដែល

### ផែនការសុវត្ថិភាពខ្សែផ្គត់ផ្គង់ទូលំទូលាយ

#### **ការផ្ទៀងផ្ទាត់ផ្នែក និងទុកចិត្ត**
- **ការផ្ទៀងផ្ទាត់ប្រភព**៖ ផ្ទៀងផ្ទាត់ប្រភព អាជ្ញាបណ្ណ និងភាពត្រឹមត្រូវនៃផ្នែក AI ទាំងអស់ មុនចូលបញ្ចូលប្រព័ន្ធ
- **ការវាយតម្លៃសុវត្ថិភាព**៖ បញ្ជាក់ការស្គាល់ហានិភ័យ និងពិនិត្យសុវត្ថិភាពសម្រាប់ម៉ូដែល ទិន្នន័យ និងសេវាកម្ម AI
- **វិភាគមតិយោបល់**៖ វាយតម្លៃកំណត់ត្រាសុវត្ថិភាព និងការអនុវត្តន៍របស់អ្នកផ្គត់ផ្គង់សេវា AI
- **ការផ្ទៀងផ្ទាត់ការគោរព**៖ ធានាថាផ្នែកទាំងអស់បានបំពេញតម្រូវការគោរព និងការគ្រប់គ្រងតាមច្បាប់របស់អង្គការនាក់

#### **បំពង់ចាក់បន្ថែមធ្វើឲ្យមានសុវត្ថិភាព**  
- **ការត្រួតពិនិត្យ CI/CD ដោយស្វ័យប្រវត្តិ**៖ រួមបញ្ចូលការស្កេនសុវត្ថិភាពជាសកលក្នុងបំពង់ចាក់បន្ថែមដោយស្វ័យប្រវត្តិ
- **ភាពត្រឹមត្រូវឯកសារ**៖ អនុវត្តការផ្ទៀងផ្ទាត់គ្រឿងចក្រ​ឌីជីថល សម្រាប់ឯកសារដែលបានចែកចាយទាំងអស់ (កូដ, ម៉ូដែល, ការកំណត់)
- **ការចាក់បន្ថែមជាស្ទេជ**៖ ប្រើយុទ្ធសាស្រ្តចាក់បន្ថែមដំណាក់កាល ជាមួយការផ្ទៀងផ្ទាត់សុវត្ថិភាពក្នុងគ្រប់ជំហាន
- **ឃ្លាំងឯកសារដែលទុកចិត្ត**៖ ចាក់បន្ថែមពីឃ្លាំងឯកសារដែលបានផ្ទៀងផ្ទាត់ និងសុវត្ថិភាពតែប៉ុណ្ណោះ

#### **ការត្រួតពិនិត្យនិងប្រតិបត្តិការ យ៉ាងបន្តបន្ទាប់**
- **ស្កេនការពឹងផ្អែកនិងហានិភ័យ**៖ ការត្រួតពិនិត្យហានិភ័យនៃកម្មវិធីទន់ និងធាតុ AI សម្រាប់ទាំងអស់
- **ត្រួតពិនិត្យម៉ូដែល**៖ វិភាគសេចក្ដីប្រព្រឹត្តិធម៌ម៉ូដែល ប្រែប្រួលការងារ និងចលនាសុវត្ថិភាព
- **ត្រួតពិនិត្យសុខភាពសេវាកម្ម**៖ តាមដានសេវាកម្ម AI ខាងក្រៅ សម្រាប់ការចូលដំណើរការ ការរីករាលដាលហេតុការណ៍ និងការផ្លាស់ប្ដូរពីគោលការណ៍
- **រួមបញ្ចូលព័ត៌មានគ្រោះថ្នាក់**៖ រួមបញ្ចូលព័ត៌មានគ្រោះថ្នាក់អំពី AI និងហានិភ័យ ML

#### **ការត្រួតពិនិត្យចូល និងតិចតួចវាចាត់ចេញ**
- **សិទ្ធិផ្នែក និងមូលដ្ឋាន**៖ កំណត់ការចូលប្រើម៉ូដែល ទិន្នន័យ និងសេវាកម្មផ្អែកលើតម្រូវការអាជីវកម្ម
- **ការគ្រប់គ្រងគណនីសេវា**៖ អនុវត្តគណនីសេវាផ្ដោតលើតិចតួចសិទ្ធិចាំបាច់បំផុត
- **ការបំបែកបណ្តាញ**៖ បំបែកសមាសធាតុ AI ហើយកំណត់ការចូលប្រើបណ្តាញរវាងសេវាកម្ម

## លទ្ធផលសុវត្ថិភាពមូលដ្ឋាន

ការអនុវត្ត MCP ស្នាក់នៅលើ និងកសាងលើការតម្រង់ដៃសុវត្ថិភាពដែលមានរបស់អង្គការរបស់អ្នក។ ការពង្រឹងទ្រឹស្តីសុវត្ថិភាពមូលដ្ឋានបានបង្កើនសុវត្ថិភាពទូទៅសម្រាប់ប្រព័ន្ធ AI និងការចាក់បន្ថែម MCP។

### គោលការណ៍សុវត្ថិភាពមូលដ្ឋាន

#### **ទ្រឹស្តីអភិវឌ្ឍសុវត្ថិភាព**
- **ការគោរព OWASP**៖ ការការពារពី [OWASP Top 10](https://owasp.org/www-project-top-ten/) ការលលំបាកវេលបណ្តាញដូចជាពេលវេលា
- **ការការពារបញ្ជាក់អត្តសញ្ញាណ AI**៖ អនុវត្តន៍ការគ្រប់គ្រងសម្រាប់ [OWASP Top 10 សម្រាប់ LLMs](https://genai.owasp.org/download/43299/?tmstv=1731900559)
- **ការគ្រប់គ្រងសម្ងាត់យ៉ាងសុវត្ថិភាព**៖ ប្រើឃ្លាំងពិសេសសម្រាប់តូកិន កូនសោ API និងទិន្នន័យកំណត់រចនាសម្ព័ន្ធសំងាត់
- **ការអ៊ិនគ្រីបបញ្ចប់ទាំងមូល**៖ អនុវត្តការទំនាក់ទំនងយ៉ាងសុវត្ថិភាពរវាងភាគីទាំងអស់ក្នុងកម្មវិធីនិងចរន្តទិន្នន័យ
- **ការផ្ទៀងផ្ទាត់ការបញ្ចូលទិន្នន័យ**៖ ការផ្ទៀងផ្ទាត់យ៉ាងតឹងរឹងលើការបញ្ចូលអ្នកប្រើ ទិន្នន័យ API និងប្រភពទិន្នន័យទាំងអស់

#### **ការបង្កើតមូលដ្ឋានរឹងមាំ**
- **ការផ្ទៀងផ្ទាត់ច្រើនជំហាន**៖ តម្រូវការផ្ទៀងផ្ទាត់ MFA សម្រាប់គណនីរដ្ឋបាល និងសេវាកម្មទាំងអស់
- **ការគ្រប់គ្រងការកែលម្អ**៖ ការគ្រប់គ្រងការប្ដូរដោយស្វ័យប្រវត្តិ នូវប្រព័ន្ធប្រតិបត្តិការ ស៊ុមដំណើរការ និងការពឹងផ្អែក
- **ការរួមបញ្ចូលអ្នកផ្តល់អត្តសញ្ញាណ**៖ ការគ្រប់គ្រងអត្តសញ្ញាណមួយកន្លែងតាមរយៈអ្នកផ្តល់អត្តសញ្ញាណសហគ្រាស (Microsoft Entra ID, Active Directory)
- **ការបំបែកបណ្តាញ**៖ ការផាត់ខ្លាំងនៃសមាសធាតុ MCP ដើម្បីកំណត់ចលនាដ៏ទូលំទូលាយ
- **គោលការណ៍តិចតួចសិទ្ធិ**៖ សិទ្ធិចាំបាច់តិចបំផុតសម្រាប់គ្រប់មុខងារពហុប្រព័ន្ធ និងគណនីទាំងអស់

#### **ការត្រួតពិនិត្យ និងការរកឃើញសុវត្ថិភាព**
- **កំណត់ត្រាឱ្យបានបរិមាណ**៖ កំណត់ត្រាដោយលម្អិតសកម្មភាពកម្មវិធី AI រួមទាំងអន្តរកម្ម MCP រវាងអតិថិជននិងម៉ាស៊ីនមេ
- **ការរួមបញ្ចូល SIEM**៖ ការគ្រប់គ្រងព័ត៌មានសុវត្ថិភាព និងព្រឹត្តិការណ៍មួយកណ្ដាលសម្រាប់ការរកឃើញសញ្ញាផ្សេងពីធម្មតា
- **វិភាគអាកប្បកិរិយា**៖ ការត្រួតពិនិត្យការគ្រប់គ្រងដោយ AI សម្រាប់រកឲ្យឃើញលំនាំចម្លែកក្នុងប្រព័ន្ធ និងអ្នកប្រើ
- **ព័ត៌មានគ្រោះថ្នាក់**៖ ផ្ដល់ស្រទាប់ព័ត៌មានគ្រោះថ្នាក់ខាងក្រៅ និងសញ្ញាបន្ថែមនៃការបំពុល (IOC)
- **ការឆ្លើយតបហេតុការណ៍**៖ នីតិវិធីដាក់ច្បាស់សម្រាប់ការរកឃើញហេតុការណ៍សុវត្ថិភាព ការឆ្លើយតប និងការស្ដារឡើងវិញ

#### **ស្ថាបត្យកម្ម Zero Trust**
- **មិនទុកចុក តែត្រូវផ្ទៀងផ្ទាត់ជាបន្តបន្ទាប់**៖ ផ្ទៀងផ្ទាត់ជាប់លាប់លើអ្នកប្រើ ឧបករណ៍ និងការតភ្ជាប់បណ្តាញ
- **ការបំបែកតូចតាចៃ**៖ ការត្រួតពិនិត្យបណ្តាញលំអិតដែលបំបែកលំនៅដ្ឋាន និងសេវាកម្មនីមួយៗ
- **សុវត្ថិភាពផ្អែកលើអត្តសញ្ញាណ**៖ គោលនយោបាយសុវត្ថិភាពផ្អែកលើគណនីបានផ្ទៀងផ្ទាត់ មិនមែនទីតាំងបណ្តាញ
- **ការវាយតម្លៃហានិភ័យបន្តបន្ទាប់**៖ ការវាយតម្លៃស្ថានភាពសុវត្ថិភាពបែបសកម្មរួមបញ្ចូលបរិបទនិងអាកប្បកិរិយាបច្ចុប្បន្ន
- **ការត្រួតពិនិត្យចូលជាក់លាក់**៖ ការត្រួតពិនិត្យចូលដែលបត់បែនបើយោងតាមហានិភ័យ តំបន់ និងការជឿទុកចិត្តឧបករណ៍

- **Microsoft Purview**៖ ការគ្រប់គ្រងទិន្នន័យ និងការគោរពច្បាប់សម្រាប់ចំណុចទិន្នន័យ AI និងដំណើរការ

#### **ការគោរពនិងគ្រប់គ្រង**
- **ការតម្រង់ធម្មតា**៖ ធានារីកចម្រើន MCP បានបំពេញតាមច្បាប់ និងតម្រូវការ​ឧស្សាហកម្មជាក់លាក់ (GDPR, HIPAA, SOC 2)

- **ការបែងចែកទិន្នន័យ**: ការបែងចែកប្រភេទនិងការដោះស្រាយយ៉ាងត្រឹមត្រូវសម្រាប់ទិន្នន័យសំសៃដែលបានដំណើរការដោយប្រព័ន្ធ AI
- **គន្លងតាមដាន**: កំណត់ត្រាពេញលេញសម្រាប់ការអនុវត្តតាមច្បាប់និងការស៊ើបអង្កេត forensic
- **ការត្រួតពិនិត្យឯកជនភាព**: ការអនុវត្តន៍គោលការណ៍ឯកជនភាពតាមរចនាសម្ព័ន្ធប្រព័ន្ធ AI
- **ការគ្រប់គ្រងការផ្លាស់ប្តូរ**: ដំណើរការផ្លូវការសម្រាប់ការត្រួតពិនិត្យសុវត្ថិភាពនៃការផ្លាស់ប្តូរប្រព័ន្ធ AI

ការអនុវត្តមូលដ្ឋានទាំងនេះបង្កើតគន្លងសុវត្ថិភាពដ៏រឹងមាំដែលបង្កើនប្រសិទ្ធភាពនៃការគ្រប់គ្រងសុវត្ថិភាពជាក់លាក់សម្រាប់ MCP ហើយផ្តល់ការពារយ៉ាងពេញលេញសម្រាប់កម្មវិធីដែលបើកចំហដោយ AI។

## លទ្ធផលសុវត្ថិភាពសាកសម

- **វិធីសាស្រ្តសុវត្ថិភាពប្លង់ស្រទាប់**: ផ្ទៀងផ្ទាត់ការអនុវត្តផ្នែកសុវត្ថិភាពមូលដ្ឋាន (ការសរសេរកូដរឹងមាំ, សិទ្ធិដើមតិចបំផុត, ពិនិត្យខ្សែផ្គត់ផ្គង់, ការត្រួតពិនិត្យបន្ត) ជាមួយការគ្រប់គ្រងជាក់លាក់សម្រាប់ AI ដើម្បីការពារយ៉ាងពេញលេញ

- **ផ្ទៃកាសារ៉ាប់អារោមជាក់លាក់នៃគ្រោះថ្នាក់ AI**: ប្រព័ន្ធ MCP ប្រឈមមុខនឹងហានិភ័យពិសេសរួមមានការវាយប្រហារបញ្ចូលសារ prompt injection, ការបង្ហូរឧបករណ៍ tool poisoning, ការចាប់យកវគ្គសម័យ session hijacking, បញ្ហា confused deputy, ចំណុចខ្សែបញ្ជា token passthrough, និងសិទ្ធិច្រើនពេកដែលត្រូវការ វិធានការពិសេស

- **ភាពឯកទេសក្នុងការផ្ទៀងផ្ទាត់ និងការអនុញ្ញាត**: អនុវត្តិការផ្ទៀងផ្ទាត់រឹងមាំប្រើអ្នកផ្គត់ផ្គង់អត្តសញ្ញាណខាងក្រៅ (Microsoft Entra ID), អនុវត្តការត្រួតពិនិត្យ token ត្រឹមត្រូវ, ហើយមិនទទួលយក token ដែលមិនបានចេញត្រឹមត្រូវសម្រាប់ម៉ាស៊ីនមេ MCP របស់អ្នកទេ

- **ការការពារការវាយប្រហារប្រព័ន្ធ AI**: ចាត់វិធានការពារដោយប្រើ Microsoft Prompt Shields និង Azure Content Safety ដើម្បីការពារពីការវាយប្រហារចូលរួមដោយប្រើ prompt injection និង tool poisoning ខណៈពេលធានាការត្រួតពិនិត្យមេតាទិន្នន័យឧបករណ៍ និងត្រួតពិនិត្យការផ្លាស់ប្តូរដែលមានលក្ខណៈឌីណាមិច

- **សុវត្ថិភាពវគ្គសម័យ និងការដឹកជញ្ជូន**: ប្រើលេខសម្គាល់វគ្គសម័យ session ដែលមានសុវត្ថិភាព cryptographically និងមិនត្រូវវាយតម្លៃជាការប្រក់លាន់ ហើយភ្ជាប់ជាមួយអត្តសញ្ញាណអ្នកប្រើ, អនុវត្តការគ្រប់គ្រងរយៈពេលវគ្គសម័យត្រឹមត្រូវ, និងមិនប្រើវគ្គសម័យសម្រាប់ការផ្ទៀងផ្ទាត់អត្តសញ្ញាណទេ

- **អនុវត្តិ OAuth ជាលក្ខណៈល្អបំផុត**: ការការពារការវាយប្រហារប្រភេទ confused deputy តាមរយៈការយល់ព្រមយ៉ាងច្បាស់ពីអ្នកប្រើសម្រាប់កម្មវិធីដែលបានចុះបញ្ជីឌីណាមិច, អនុវត្ត OAuth 2.1 ដោយមាន PKCE, និងការត្រួតពិនិត្យ URI ផ្លាស់ទីយ៉ាងតឹងរឹង  

- **គោលការណ៍សុវត្ថិភាព token**: ជៀសវាងការអនុវត្ត token passthrough ដែលមិនល្អ, ត្រួតពិនិត្យសំណើ token audience, អនុវត្ត token រយៈពេលខ្លីជាមួយការបង្វិលយ៉ាងសុវត្ថិភាព, ហើយរក្សាបន្ទាប់ស្មោះត្រង់ក្នុងការទុកចិត្ត

- **សុវត្ថិភាពខ្សែផ្គត់ផ្គង់ពេញលេញ**: ដំណើរការភាគីទាំងអស់ក្នុងបរិស្ថាន AI (ម៉ូដែល, embeddings, អ្នកផ្គត់ផ្គង់ context, API ខាងក្រៅ) ដូច្នេះមានសុវត្ថិភាពដូចជាសម្ភារៈកម្មវិធីធម្មតា

- **កំណត់បច្ចុប្បន្នភាពបន្ត**: មានភាពទាន់សម័យជាមួយកំណែ MCP ដែលកំពុងអភិវឌ្ឍយ៉ាងលឿន, ធ្វើឲ្យមានការចូលរួមក្នុងស្តង់ដារសហគមន៍សុវត្ថិភាព, ហើយរក្សាទម្រង់សុវត្ថិភាពដែលអាចខំញឹកញយពេលច្បាប់ត្រូវបានរីកចម្រើន

- **ការរួមបញ្ចូលសុវត្ថិភាព Microsoft**: ប្រើប្រព័ន្ធសុវត្ថិភាពទូលំទូលាយរបស់ Microsoft (Prompt Shields, Azure Content Safety, GitHub Advanced Security, Entra ID) សម្រាប់ការការពារក្នុងការបញ្ចូន MCP
