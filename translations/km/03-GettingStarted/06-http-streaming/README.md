# ការផ្ទេរទិន្នន័យតាម HTTPS ជាមួយនឹង Model Context Protocol (MCP)

អធ្យាស្រ័យនេះផ្តល់នូវមគ្គុទេសក៍ពេញលេញសម្រាប់ការអនុវត្តន៍ការផ្ទេរទិន្នន័យដែលមានសុវត្ថិភាព អាចពង្រីកបាន និងមានម៉ោងពិតជាមួយ Model Context Protocol (MCP) តាមរយៈ HTTPS។ វាគ្របដណ្តប់ពីមូលហេតុនៃការផ្ទេរទិន្នន័យ មេកានិចដឹកជញ្ជូនដែលមាន ស្វីតការអនុវត្តន៍ HTTP ដែលអាចផ្ទេរទិន្នន័យបាននៅក្នុង MCP វិធីសាស្រ្តល្អបំផុតសម្រាប់សុវត្ថិភាព ការផ្ទេរពី SSE និងមគ្គុទេសក៍ប្រតិបត្តិបាក់សម្រាប់ការបង្កើតកម្មវិធី MCP ផ្ទេរទិន្នន័យរបស់អ្នកដោយខ្លួនឯង។

> **មើលទៅមុខ៖** មេរៀននេះពិពណ៌នាអំពី Streamable HTTP ក្រោម **MCP Specification 2025-11-25** ដែលក្នុងនោះសម័យត្រូវបានបង្កើតឡើងពេល `initialize` ហើយភ្ជាប់ជាមួយក្បៀសក្បាល `Mcp-Session-Id`។ កំណែជាលក្ខណៈលក់កម្មវិធី `2026-07-28` យកការចាប់ដៃ និងអត្តសញ្ញាណសម័យចេញទាំងស្រុង ធ្វើឲ្យការស្នើរសុំរាល់មួយគឺមានខ្លួនឯង ហើយអាចផ្ញើទៅម៉ាស៊ីនមេណាមួយបានដោយមិនចាំបាច់មានសម័យតាមដាន។ ចូលទៅរក [មានអ្វីផ្លាស់ប្តូរនៅក្នុង MCP: កំណែជាលក្ខណៈលក់កម្មវិធី 2026-07-28](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) សម្រាប់ព័ត៌មានលម្អិត។

## មេកានិចដឹកជញ្ជូន និងការផ្ទេរទិន្នន័យនៅក្នុង MCP

ផ្នែកនេះពិនិត្យមើលមេកានិចដឹកជញ្ជូនផ្សេងៗក្នុង MCP ហើយតួនាទីរបស់វាក្នុងការផ្ដល់សមត្ថភាពផ្ទេរទិន្នន័យសម្រាប់ការទំនាក់ទំនងម៉ោងពិតរវាងភ្ញៀវប្រើនៃកម្មវិធី និងម៉ាស៊ីនមេ។

### មេកានិចដឹកជញ្ជូនគឺជាអ្វី?

មេកានិចដឹកជញ្ជូនកំណត់ពីរបៀបដែលទិន្នន័យត្រូវបានប្ដូរជារវាងភ្ញៀវប្រើនិងម៉ាស៊ីនមេ។ MCP គាំទ្រអ្នកដឹកជញ្ជូនច្រើនប្រភេទ ដើម្បីឆបគ្នានឹងបរិបទ និងតម្រូវការផ្សេងៗ៖

- **stdio**: បញ្ចូល/បញ្ចេញស្តង់ដារ សមនឹងឧបករណ៍ក្នុងស្រុក និង CLI។ ងាយស្រួល ប៉ុន្តែមិនសមសម្រាប់វែបឬពពកទេ។
- **SSE (Server-Sent Events)**: អនុញ្ញាតឲ្យម៉ាស៊ីនមេបង្កើតបច្ចុប្បន្នភាពម៉ោងពិតទៅកាន់ភ្ញៀវប្រើតាម HTTP។ ល្អសម្រាប់ផ្ទាំង UI វែប ប៉ុន្តែមានកំណត់នៅក្នុងភាពផ្ដំឡើងនិងភាពបត់បែន។ ចាប់តាំងពី MCP Specification 2025-06-18 ការដឹកជញ្ជូន SSE ផ្ទាល់ខ្លួនត្រូវបានបដិសេធ ហើយបានជំនួសដោយ "Streamable HTTP"។
- **Streamable HTTP**: ដឹកជញ្ជូនផ្ទេរទិន្នន័យថ្មីមូលដ្ឋានលើ HTTP ដែលគាំទ្រការជូនដំណឹង និងភាពអាចផ្ដំឡើងបានល្អ។ គួរអោយប្រើសម្រាប់លក្ខខណៈផលិតកម្ម និងពពកជាច្រើន។

### តារាងប្រៀបធៀប

មើលតារាងប្រៀបធៀបខាងក្រោម ដើម្បីយល់ពីភាពខុសគ្នារវាងមេកានិចដឹកជញ្ជូនទាំងនេះ៖

| ដឹកជញ្ជូន        | បច្ចុប្បន្នភាពម៉ោងពិត | ផ្ទេរទិន្នន័យ | ភាពអាចពង្រីកបាន | ករណីប្រើ              |
|-------------------|------------------|-----------|-------------|-------------------------|
| stdio             | ទេ               | ទេ        | ទាប        | ឧបករណ៍ CLI ក្នុងស្រុក    |
| SSE               | បាទ              | បាទ       | មធ្យម       | វែប, បច្ចុប្បន្នភាពម៉ោងពិត  |
| Streamable HTTP   | បាទ              | បាទ       | ខ្ពស់       | ពពក, អតិថិជនច្រើន        |

> **ដំណឹង៖** ការជ្រើសរើសដឹកជញ្ជូនត្រឹមត្រូវប៉ះពាល់ដល់ការសម្រួល សមត្ថភាពពង្រីក និងបទពិសោធន៍ប្រើប្រាស់។ **Streamable HTTP** ត្រូវបានផ្ដល់អនុសាសន៍សម្រាប់កម្មវិធីទំនើប អាចពង្រីកបាន និងសមរម្យសម្រាប់ពពក។

សូមចំណាំពីឧបករណ៍ stdio និង SSE ដែលបានបង្ហាញក្នុងអធ្យាស្រ័យមុនៗ ហើយបើកវែងម៉ាស៊ីន HTTP ដែលអាចផ្ទេរទិន្នន័យបានគឺជាអ្នកដឹកជញ្ជូននៅក្នុងមេរៀននេះ។

## ការផ្ទេរទិន្នន័យ៖ ព្រឹត្តិប័ត្រ និងមូលហេតុ

ការយល់ដឹងពីមូលដ្ឋានទ្រឹស្តី និងមូលហេតុខាងក្រោយការផ្ទេរទិន្នន័យគឺមានសារៈសំខាន់សម្រាប់ការអនុវត្តប្រព័ន្ធទំនាក់ទំនងម៉ោងពិតមានប្រសិទ្ធភាព។

**ការផ្ទេរទិន្នន័យ** គឺជាវិធីសាស្រ្តក្នុងកម្មវិធីបណ្តាញដែលអនុញ្ញាតឲ្យទិន្នន័យត្រូវបានបញ្ជូន និងទទួលបានជាខ្ទង់តូចៗ ឬជាលំដាប់ព្រឹត្តិការណ៍ជាន់ៗ មិនមែនរង់ចាំឲ្យចម្លើយចុងក្រោយទាំងមូល។

- ឯកសារ ឬទិន្នន័យរាបសារ។
- បច្ចុប្បន្នភាពម៉ោងពិត (ឧ. សន្ទនា, របារកំណត់ការរីកចម្រើន)។
- ការគណនាជាយូរអង្វែង ដែលអ្នកចង់ឲ្យអ្នកប្រើប្រាស់ទទួលបានព័ត៌មានបន្តិចបន្តួច។

នេះគឺជាព័ត៌មានដែលអ្នកត្រូវបានដឹងអំពីការផ្ទេរទិន្នន័យនៅកម្រិតខ្ពស់៖

- ទិន្នន័យត្រូវបានផ្ដល់ជាបន្តៗ មិនមែនទាំងមូលជារួម។
- ភ្ញៀវអាចដំណើរការទិន្នន័យពេលវាមកដល់។
- បន្ថយពេលវេលាដែលដឹងបាន និងបង្កើនបទពិសោធន៍អ្នកប្រើ។

### ហេតុអ្វីបានជាគួរប្រើការផ្ទេរទិន្នន័យ?

មូលហេតុសម្រាប់ការប្រើការផ្ទេរទិន្នន័យមានដូចជា៖

- អ្នកប្រើទទួលបានមតិយោបល់ភ្លាម ភ្លឺ មិនមែនត្រឹមតែនៅចុងក្រោយទេ
- អនុញ្ញាតកម្មវិធីម៉ោងពិត និង UI តបត្រូវ
- ប្រើប្រាស់ធនធានបណ្តាញ និងកុំព្យូទ័រប្រសើរឡើង

### ឧទាហរណ៍សាមញ្ញ៖ ម៉ាស៊ីនបម្រើនិងភ្ញៀវប្រែផ្ទេរទិន្នន័យ HTTP

នេះគឺជាឧទាហរណ៍សាមញ្ញពីរបៀបការផ្ទេរទិន្នន័យអាចត្រូវបានអនុវត្តបាន៖

#### Python

**ម៉ាស៊ីនបម្រើ (Python ប្រើ FastAPI និង StreamingResponse):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**ភ្ញៀវ (Python ប្រើ requests):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

ឧទាហរណ៍នេះបង្ហាញពីម៉ាស៊ីនបម្រើផ្ញើសារជាច្រើនទៅភ្ញៀវបានពេលវាផ្ទៀងផ្ទាត់បាន មិនមែនរង់ចាំសារទាំងអស់រួចទេ។

**របៀបវាដំណើរការ៖**

- ម៉ាស៊ីនបម្រើចេញសារក្នុងមួយៗពេលវាចំនេញ។
- ភ្ញៀវទទួលបាន និងបង្ហាញខ្ទង់នីមួយៗពេលវា​មកដល់។

**តម្រូវការ៖**

- ម៉ាស៊ីនបម្រើត្រូវប្រើការឆ្លើយតបផ្ទេរទិន្នន័យ (ឧ. `StreamingResponse` នៅក្នុង FastAPI)។
- ភ្ញៀវត្រូវដំណើរការឆ្លើយតបជាតិចចរទិន្នន័យ (`stream=True` នៅក្នុង requests)។
- ប្រភេទខ្លឹមសារ ជាទូទៅគឺជា `text/event-stream` ឬ `application/octet-stream`។

#### Java

**ម៉ាស៊ីនបម្រើ (Java ប្រើ Spring Boot និង Server-Sent Events):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**ភ្ញៀវ (Java ប្រើ Spring WebFlux WebClient):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**កំណត់សម្គាល់អំពី Java:**

- ប្រើ Spring Boot's reactive stack ជាមួយ `Flux` សម្រាប់ការផ្ទេរទិន្នន័យ
- `ServerSentEvent` ផ្ដល់នូវការផ្ទេរព្រឹត្តិការណ៍ដែលមានរចនាសម្ព័ន្ធមួយជាមួយប្រភេទព្រឹត្តិការណ៍
- `WebClient` ជាមួយ `bodyToFlux()` អនុញ្ញាតឲ្យប្រើប្រាស់ការផ្ទេរទិន្នន័យប្រតិកម្ម
- `delayElements()` ចាក់សោរ​ពេលព្យួរពីរវាងព្រឹត្តិការណ៍
- ព្រឹត្តិការណ៍អាចមានប្រភេទ (`info`, `result`) សម្រាប់ការដំណើរការល្អរបស់ភ្ញៀវ

### ផ្ទៀងផ្ទាត់៖ ការផ្ទេរទិន្នន័យផ្នែកចាស់ជាមួយការផ្ទេរទិន្នន័យ MCP

ភាពខុសគ្នារវាងរបៀបការផ្ទេរទិន្នន័យជារបៀប "ផ្ទៃ" ជាមួយរបៀបរបស់ MCP អាចបង្ហាញដូចខាងក្រោម៖

| លក្ខណៈពិសេស           | ការផ្ទេរទិន្នន័យ HTTP ផ្នែកចាស់ | ការផ្ទេរទិន្នន័យ MCP (ការជូនដំណឹង)    |
|-------------------------|-------------------------------|-------------------------------------|
| ចម្លើយចម្បង            | និចខ្ទង់ចែក                     | មួយតែម្ដង នៅចុងបំផុត                |
| ការជូនដំណឹងដំណើរការជាបន្ត  | ផ្ញើជា ខ្ទង់ទិន្នន័យ            | ផ្ញើជា ជូនដំណឹង                    |
| តម្រូវការពាណិជ្ជកម្ម      | ត្រូវដំណើរការផ្ទេរទិន្នន័យ      | ត្រូវអនុវត្តអ្នកដំណើរការសារជា ។      |
| ករណីប្រើ               | ឯកសារធំៗ បច្ចុប្បន្នភាព AI  | ការរីកចម្រើន កំណត់ហេតុ បញ្ចេញមតិយោបល់ម៉ោងពិត |

### ភាពខុសគ្នាចម្បងដែលបានសង្កេតឃើញ

ដោយផ្តល់អប្បបរមារ, នេះជាភាពខុសគ្នាចម្បង៖

- **លំនាំទំនាក់ទំនង៖**
  - ការផ្ទេរទិន្នន័យ HTTP ផ្នែកចាស់៖ ប្រើកូដផ្ទេរខ្ទង់ស្រាលសម្រាប់ផ្ញើទិន្នន័យជាខ្ទង់
  - ការផ្ទេរទិន្នន័យ MCP៖ ប្រើប្រព័ន្ធជូនដំណឹងដែលមានរចនាសម្ព័ន្ធជាមួយពិធីការ JSON-RPC

- **ទ្រង់ទ្រាយសារ៖**
  - HTTP ចាស់៖ ខ្ទង់អក្សរធម្មតាពីរបីបន្ទាត់
  - MCP៖ LoggingMessageNotification ដែលមានរចនាសម្ព័ន្ធ និងព័ត៌មានបន្ថែម

- **អនុវត្តភ្ញៀវ៖**
  - HTTP ចាស់៖ ភ្ញៀវធម្មតាដែលដំណើរការឆ្លើយតបផ្ទេរទិន្នន័យ
  - MCP៖ ភ្ញៀវខ្ពស់ជាងដែលមានអ្នកដំណើរការសារដើម្បីដំណើរការប្រភេទសារផ្សេងៗ

- **ការជូនដំណឹងដំណើរការ៖**
  - HTTP ចាស់៖ ការជូនដំណឹងដំណើរការជាផ្នែកចម្បងដំណើរការឆ្លើយតប
  - MCP៖ ការជូនដំណឹងដំណើរការផ្ញើដោយសារជូនដំណឹងដែលផ្សេងពីចម្លើយចម្បងនៅចុងបំផុត

### អនុសាសន៍

មានចំនុចខ្លះដែលយើងផ្តល់អនុសាសន៍ពេលជ្រើសរើសរវាងអនុវត្តការផ្ទេរទិន្នន័យផ្នែកចាស់ (ជា endpoint ដែលយើងបានបង្ហាញក្នុង `/stream`) ឬជ្រើសរើសការផ្ទេរទិន្នន័យតាម MCP។

- **សម្រាប់តម្រូវការផ្ទេរទិន្នន័យសាមញ្ញ៖** ការផ្ទេរទិន្នន័យ HTTP ផ្នែកចាស់សាមញ្ញក្នុងការអនុវត្ត ហើយគ្រប់គ្រាន់សម្រាប់តម្រូវការសាមញ្ញ។

- **សម្រាប់កម្មវិធីស្មុគស្មាញ និងមានអន្តរាគមន៍៖** ការផ្ទេរទិន្នន័យ MCP ផ្ដល់នូវវិធីសាស្រ្តមានរចនាសម្ព័ន្ធជាមួយព័ត៌មានបន្ថែម និងការបែកបាក់ចេញរវាងការជូនដំណឹងនិងលទ្ធផលចុងក្រោយ។

- **សម្រាប់កម្មវិធី AI៖** ប្រព័ន្ធជូនដំណឹងរបស់ MCP មានប្រយោជន៍ពិសេសសម្រាប់ភារកិច្ច AI ដែលរយៈពេលវែង ដែលអ្នកចង់រក្សាអ្នកប្រើឲ្យបានដឹងអំពីដំណើរការរីកចម្រើន។

## ការផ្ទេរទិន្នន័យនៅក្នុង MCP

យើងបានឃើញអនុសាសន៍ និងតារាងប្រៀបធៀបជាលក្ខណៈទូទៅ រវាងការផ្ទេរទិន្នន័យផ្នែកចាស់ និង MCP។ ឥឡូវនេះដំណើរការផ្ទាល់ អំពីរបៀបដែលអ្នកអាចប្រើការផ្ទេរទិន្នន័យនៅក្នុង MCP។

ការយល់ដឹងពីរបៀបដែលការផ្ទេរទិន្នន័យដំណើរការនៅក្នុងស៊ុមការដំណើរការ MCP គឺសំខាន់សម្រាប់បង្កើតកម្មវិធីតបត្រូវបានលឿនដែលផ្ដល់មតិយោបល់ម៉ោងពិតឲ្យអ្នកប្រើពេលកំពុងដំណើរការពេលវែង។

ក្នុង MCP, ការផ្ទេរទិន្នន័យមិនមែនជាការផ្ញើចម្លើយចម្បងជាច្រើនខ្ទង់ទេ ប៉ុន្តែជាការផ្ញើ **ការជូនដំណឹង** ទៅភ្ញៀវខណៈឧបករណ៍កំពុងដំណើរការស្នើរសុំ។ ការជូនដំណឹងទាំងនេះអាចមានការជូនដំណឹងពីរីកចម្រើន កំណត់ហេតុ ឬព្រឹត្តិការណ៍ផ្សេងៗ។

### របៀបវាដំណើរការ

លទ្ធផលចម្បងនៅតែត្រូវបានផ្ញើជាចម្លើយតែមួយ។ ទោះយ៉ាងណាក៏ដោយ, ការជូនដំណឹងអាចផ្ញើបានជាសារតែមួយៗក្នុងពេលដំណើរការ ហើយបន្ទាប់មកធ្វើឲ្យភ្ញៀវប្រែប្រួលនៅម៉ោងពិត។ ភ្ញៀវត្រូវអាចដំណើរការនិងបង្ហាញការជូនដំណឹងទាំងនេះ។

## ការជូនដំណឹងគឺជាអ្វី?

យើងបាននិយាយពី "ការជូនដំណឹង", តើវាមានន័យអ្វីនៅក្នុងបរិបទ MCP?

ការជូនដំណឹងគឺជាសារដែលផ្ញើពីម៉ាស៊ីនមេចូលទៅភ្ញៀវដើម្បីជូនដំណឹងអំពីការរីកចម្រើន ស្ថានភាព ឬព្រឹត្តិការណ៍ផ្សេងៗពេលកំពុងដំណើរការការងាររយៈពេលវែង។ ការជូនដំណឹងបង្កើនភាពច្បាស់លាស់និងបទពិសោធន៍អ្នកប្រើ។

ឧទាហរណ៍, ភ្ញៀវត្រូវផ្ញើការជូនដំណឹងមួយពេលការចាប់ដៃដាលជាមួយម៉ាស៊ីនមេបានសម្រេច។

ការជូនដំណឹងមើលទៅដូចជា សារជា JSON:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

ការជូនដំណឹងជារួមបង្កប់នៅក្នុងប្រធានបទក្នុង MCP ដែលហៅថា ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging)។

ដើម្បីអោយ logging ធ្វើការ, ម៉ាស៊ីនមេត្រូវតែបើកការកំណត់នេះជាគ្រាារឹក/សមត្ថភាព ដូចខាងក្រោម:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> គិតតាម SDK ដែលប្រើ, logging អាចត្រូវបានបើកដោយខ្លួនឯង ឬអ្នកប្រហែលជាត្រូវបើកវាផ្ទាល់ក្នុងការកំណត់ម៉ាស៊ីនមេរបស់អ្នក។

មានប្រភេទនៃការជូនដំណឹងខុសៗគ្នា៖

| កម្រិត       | ពិពណ៌នា                  | ករណីប្រើជាក់លាក់             |
|-------------|---------------------------|-------------------------------|
| debug       | ព័ត៌មានលម្អិតសម្រាប់ការដោះស្រាយបញ្ហា| ចំណុចចូល/ចេញសកម្មភាព      |
| info        | សារព័ត៌មានទូទៅ             | ការឆ្លើយតបរីកចម្រើន          |
| notice      | ព្រឹត្តិការណ៍ធម្មតាដែលមានសារៈសំខាន់ | ការផ្លាស់ប្តូរការកំណត់       |
| warning     | ស្ថានភាពព្រមាន              | ការប្រើប្រាស់លក្ខណៈដែលបានចាស់បំផុត |
| error       | ស្ថានភាពកំហុស              | ការបរាជ័យក្នុងការងារ            |
| critical    | ស្ថានភាពសំខាន់             | ការបរាជ័យផ្នែករបៀបប្រព័ន្ធ   |
| alert       | ត្រូវអនុវត្តភារកិច្ចភ្លាម      | ការរងគ្រោះនៃទិន្នន័យ         |
| emergency   | ប្រព័ន្ធមិនអាចប្រើបាន     | ការធ្លាស់ប្តូរប្រព័ន្ធទាំងមូល |

## ការអនុវត្តការជូនដំណឹងនៅក្នុង MCP

ដើម្បីអនុវត្តការជូនដំណឹងក្នុង MCP អ្នកត្រូវតែតំឡើងទាំងម៉ាស៊ីនបម្រើ និងភ្ញៀវឲ្យដំណើរការការជូនដំណឹងម៉ោងពិត។ វាអនុញ្ញាតឲ្យកម្មវិធីរបស់អ្នកផ្តល់មតិយោបល់ភ្លាមៗដល់អ្នកប្រើក្នុងពេលដំណើរការប្រតិបត្តិការរយៈពេលវែង។

### មុខងារផ្នែកម៉ាស៊ីនបម្រើ៖ ផ្ញើការជូនដំណឹង

ចាប់ផ្តើមពីផ្នែកម៉ាស៊ីនបម្រើ។ ក្នុង MCP អ្នកកំណត់ឧបករណ៍ដែលអាចផ្ញើការជូនដំណឹងខណៈដែលកំពុងដំណើរការស្នើរសុំ។ ម៉ាស៊ីនមេប្រើវត្ថុ context (ភាគច្រើនជា `ctx`) ដើម្បីផ្ញើសារទៅភ្ញៀវ។

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

ក្នុងឧទាហរណ៍មុននេះ ឧបករណ៍ `process_files` ផ្ញើការជូនដំណឹងបីទៅភ្ញៀវខណៈពេលវាដំណើរការឯកសារ។ វិធីសាស្រ្ត `ctx.info()` ត្រូវបានប្រើសម្រាប់ផ្ញើសារព័ត៌មាន។

បន្ថែមពីនេះ ដើម្បីបើកការជូនដំណឹង សូមប្រាកដថាម៉ាស៊ីនមេប្រើអ្នកដឹកជញ្ជូនផ្ទេរទិន្នន័យ (ដូចជា `streamable-http`) ហើយភ្ញៀវបានអនុវត្តអ្នកដំណើរការសារសម្រាប់ស្គាល់ការជូនដំណឹង។ វិធីនេះបង្ហាញពីរបៀបចាប់ផ្តើមម៉ាស៊ីនមេលើអ្នកដឹកជញ្ជូន `streamable-http`៖

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

ក្នុងឧទាហរណ៍ .NET នេះ ឧបករណ៍ `ProcessFiles` ត្រូវបានសម្គាល់ជាមួយអត្រា `Tool` ហើយផ្ញើការជូនដំណឹងបីទៅភ្ញៀវខណៈពេលដំណើរការឯកសារ។ វិធីសាស្រ្ត `ctx.Info()` ត្រូវបានប្រើសម្រាប់ផ្ញើសារព័ត៌មាន។

ដើម្បីបើកការជូនដំណឹងនៅម៉ាស៊ីនមេ MCP .NET របស់អ្នក សូមប្រាកដថាអ្នកកំពុងប្រើអ្នកដឹកជញ្ជូនផ្ទេរទិន្នន័យ៖

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### មុខងារផ្នែកភ្ញៀវ៖ ទទួលការជូនដំណឹង

ភ្ញៀវត្រូវតែអនុវត្តអ្នកដំណើរការសារដើម្បីដំណើរការនិងបង្ហាញការជូនដំណឹងពេលវាអាចមកដល់។

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

ក្នុងកូដមុននេះ មុខងារ `message_handler` ពិនិត្យមើលថាសារចូលមកជាការជូនដំណឹងទេឬទេ។ ប្រសិនបើខ្លះ វាបង្ហាញការជូនដំណឹង មិនដូច្នោះវាធ្វើដំណើរការពីរបៀបធម្មតារបស់ម៉ាស៊ីនមេ។ សូមចំណាំថា `ClientSession` ត្រូវបានចាប់ផ្តើមជាមួយ `message_handler` ដើម្បីដំណើរការការជូនដំណឹងចូល។

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

ក្នុងឧទាហរណ៍ .NET នេះ មុខងារ `MessageHandler` ពិនិត្យថាសារចូលមកជាការជូនដំណឹង ឬមិនមែន។ ប្រសិនបើមែនវាបង្ហាញការជូនដំណឹង មានការប្រតិបត្តិការធម្មតាជាសារម៉ាស៊ីនមេផ្សេងៗ។ `ClientSession` ត្រូវបានចាប់ផ្តើម្ជជាមួយអ្នកដំណើរការសារតាមរយៈ `ClientSessionOptions`។

ដើម្បីបើកការជូនដំណឹង សូមប្រាកដថាម៉ាស៊ីនមេប្រើអ្នកដឹកជញ្ជូនផ្ទេរទិន្នន័យ (ដូចជា `streamable-http`) ហើយភ្ញៀវបានអនុវត្តអ្នកដំណើរការសារដើម្បីដំណើរការការជូនដំណឹង។

## ការជូនដំណឹងដំណើរការ និងស្ពឺស៊ី

ផ្នែកនេះពន្យល់ពីមូលដ្ឋាននៃការជូនដំណឹងដំណើរការ នៅក្នុង MCP អំពីហេតុផលដែលវាមានសារៈសំខាន់ និងរបៀបអនុវត្តតាម Streamable HTTP។ អ្នកនឹងឃើញភារកម្មសម្រាប់បង្កើតនូវសមត្ថភាពរបស់អ្នក។

ការជូនដំណឹងដំណើរការ គឺជាសារម៉ោងពិតដែលផ្ញើពីម៉ាស៊ីនមេចូលទៅភ្ញៀវក្នុងពេលបញ្ចប់ផ្សេងៗ។ មិនរង់ចាំឲ្យដំណើរការសរុបបញ្ចប់ទេ មុខងារ ម៉ាស៊ីនមេអោយពាក់ព័ន្ធទៅភ្ញៀវក្នុងស្ថានភាពបច្ចុប្បន្ន។ វាបង្កើនភាពច្បាស់លាស់ បទពិសោធន៍អ្នកប្រើ និងបង្កើនភាពងាយស្រួលក្នុងការជួសជុលកំហុស។

**ឧទាហរណ៍៖**

```text

"Processing document 2/10"
...
"Processing complete!"

```

### ហេតុអ្វីបានជាត្រូវប្រើការជូនដំណឹងដំណើរការ?

ការជូនដំណឹងដំណើរការមានសារៈសំខាន់សម្រាប់ហេតុផលជាច្រើន៖

- **បទពិសោធន៍ល្អប្រសើរ**៖ អ្នកប្រើក៏មើលឃើញការអាប់ដេតនៃការងារជាបន្តៗ មិនមែនត្រឹមតែចុងក្រោយតែប៉ុណ្ណោះ។
- **មតិយោបល់ម៉ោងពិត**៖ ភ្ញៀវអាចបង្ហាញរបារកំណត់ប្រសិទ្ធភាព ឬកំណត់ហេតុ ដែលធ្វើឲ្យកម្មវិធីមានមហិច្ឆិតាលឿន។
- **ងាយស្រួលក្នុងការជួសជុលកំហុស និងត្រួតពិនិត្យ**៖ អ្នកអភិវឌ្ឍន៍ និងអ្នកប្រើអាចមើលឃើញថាតើកម្រិតណាចំណុចណាដែលយឺត ឬឈប់ឈរ។

### របៀបអនុវត្តការជូនដំណឹងដំណើរការ

នេះជារបៀបដែលអ្នកអាចអនុវត្តការជូនដំណឹងដំណើរការនៅក្នុង MCP៖

- **នៅផ្នែកម៉ាស៊ីនបម្រើ:** ប្រើ `ctx.info()` ឬ `ctx.log()` ដើម្បីផ្ញើការជូនដំណឹងនៅពេលមុខងារមួយបានដំណើរការ។ វាផ្ញើសារទៅភ្ញៀវមុនពេលលទ្ធផលចម្បងរួចរាល់។
- **នៅផ្នែកភ្ញៀវ:** អនុវត្តអ្នកដំណើរការសារដែលស្ដាប់ និងបង្ហាញការជូនដំណឹងពេលវាទៅដល់។ អ្នកដំណើរការនេះមានមុខងារបែងចែកចម្លើយចុងក្រោយពីការជូនដំណឹង។

**ឧទាហរណ៍ម៉ាស៊ីនបម្រើ៖**

#### Python

```python

    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

## ការពិចារណាសុវត្ថិភាព

នៅពេលអនុវត្តម៉ាស៊ីនមេ MCP ជាមួយអ្នកដឹកជញ្ជូនផ្អែកលើ HTTP សុវត្ថិភាពក្លាយជាប្រធានបទសំខាន់ដែលត្រូវការយកចិត្តទុកដាក់យ៉ាងហ្មត់ចត់ចំពោះវិធីវាយប្រហារ និងមុខងារការពារ។

### សង្ខេប

សុវត្ថិភាពមានសារៈសំខាន់សម្រាប់ការបង្ហាញម៉ាស៊ីនមេ MCP តាម HTTP។ Streamable HTTP បង្ហាញពីចំណុចប្រទះការវាយប្រហារថ្មីៗ ហើយតម្រូវការការកំណត់យ៉ាងហ្មត់ចត់។

### ចំណុចសំខាន់


- **ការផ្ទៀងផ្ទាត់ Origin Header**: ត្រូវតែផ្ទៀងផ្ទាត់ក្បាល `Origin` ជានិច្ច ដើម្បីទប់ស្កាត់ការវាយប្រហាររបស់ DNS rebinding។
- **ការចងភ្ជាប់ Localhost**: សម្រាប់ការអភិវឌ្ឍន៍ក្នុងស្រុក ចងភ្ជាប់ម៉ាស៊ីនបម្រើទៅ `localhost` ដើម្បីរក្សាការមិនឲ្យវាបង្ហាញទៅសាធារណៈអ៊ីនធឺណិត។
- **ការផ្ទៀងផ្ទាត់អត្តសញ្ញាណ**: ទទួលយកការផ្ទៀងផ្ទាត់អត្តសញ្ញាណ (ឧ. កូនសោ API, OAuth) សម្រាប់ការដាក់បញ្ចូលផលិតផល។
- **CORS**: កំណត់គោលការណ៍ការចែករំលែកប្រភពឆ្លង (CORS) ដើម្បីដាក់កំណត់ការចូលប្រើ។
- **HTTPS**: ប្រើ HTTPS ក្នុងការផលិតដើម្បីសម្ងាត់ចរាចរណ៍ទិន្នន័យ។

### អនុវត្តល្អបំផុត

- កុំទុកចិត្តសំណើដែលមកដល់ដោយគ្មានការផ្ទៀងផ្ទាត់។
- កត់ត្រា និងត្រួតពិនិត្យការចូលប្រើ និងកំហុសទាំងអស់។
- ធ្វើបច្ចុប្បន្នភាពអាស្រ័យភាពជាប្រចាំ ដើម្បីជាអ្នកជួសជុលកំហុសសុវត្ថិភាព។

### បញ្ហា

- តុល្យភាពរវាងសុវត្ថិភាព និងភាពងាយស្រួលក្នុងការអភិវឌ្ឍ
- ប្រាកដភាពសមរម្យជាមួយបរិយាកាសអតិថិជននានា

## ការវិវត្តពី SSE ទៅ Streamable HTTP

សម្រាប់កម្មវិធីដែលកំពុងប្រើ Server-Sent Events (SSE) ការផ្លាស់ប្តូរទៅ Streamable HTTP ផ្តល់នូវសមត្ថភាពពង្រីក និងការរីកចម្រើនល្អប្រសើរជាងមុនសម្រាប់ការអនុវត្ត MCP របស់អ្នក។

### ហេតុផលចាំបាច់ត្រូវធ្វើអាប់ដេត?

មានពីរហេតុផលសំខាន់សម្រាប់ការវិវត្តចេញពី SSE ទៅ Streamable HTTP:

- Streamable HTTP ផ្តល់នូវសមត្ថភាពពង្រីក, សមរម្យភាព និងការគាំទ្រជូនដំណឹងដែលសម្បូរបែបជាង SSE។
- វាជាស្ទង់ផ្លូវផ្គូផ្សារដើម្បីអភិវឌ្ឍកម្មវិធី MCP ថ្មីៗ។

### ជំហានការផ្លាស់ប្តូរ

វិធីដែលអ្នកអាចផ្លាស់ប្តូរពី SSE ទៅ Streamable HTTP ក្នុងកម្មវិធី MCP របស់អ្នក៖

- **ធ្វើបច្ចុប្បន្នកូដម៉ាស៊ីនបម្រើ** ដើម្បីប្រើ `transport="streamable-http"` នៅក្នុង `mcp.run()`។
- **ធ្វើបច្ចុប្បន្នកូដអតិថិជន** ដើម្បីប្រើ `streamablehttp_client` ជំនួស SSE client។
- **អនុវត្តអ្នកចាប់យកសារមួយ** នៅក្នុងអតិថិជនដើម្បីដំណើរការជូនដំណឹង។
- **សាកល្បងសមរម្យភាព** ជាមួយឧបករណ៍ និងដំណើរការបច្ចុប្បន្ន។

### ការរក្សាសមរម្យភាព

អ្នកគួរតែរក្សាសមរម្យភាពជាមួយ SSE clients អតិថិជនបច្ចុប្បន្នក្នុងពេលប្តូរនេះ។ នេះជាយុទ្ធសាស្រ្តខ្លះៗ៖

- អ្នកអាចគាំទ្ររួមទាំង SSE និង Streamable HTTP ដោយរត់ម៉ាស៊ីនចែកចាយទាំងពីរនៅចំណុចចូលផ្សេងៗ។
- ធ្វើការផ្លាស់ប្តូរអតិថិជនទៅកាន់ផ្លូវផ្គូផ្សារថ្មីយ៉ាងជាក្រោមក្រាស់។

### បញ្ហា

ត្រូវប្រាកដថាអ្នកដោះស្រាយបញ្ហានៅពេលប្តូរនេះ៖

- ប្រាកដភាពថាអតិថិជនទាំងអស់បានធ្វើអាប់ដេត
- ការដោះស្រាយនូវភាពខុសគ្នានៃការផ្តល់ជូនការជូនដំណឹង

## ការពិចារណាអំពីសុវត្ថិភាព

សុវត្ថិភាពគួរតែជាអាទិភាពខ្ពស់បំផុតនៅពេលអនុវត្តម៉ាស៊ីនបម្រើណាមួយ ជាពិសេសនៅពេលប្រើប្រាស់ការដឹកជញ្ជូនផ្អែកលើ HTTP ដូចជា Streamable HTTP នៅក្នុង MCP។

ពេលដែលអនុវត្តម៉ាស៊ីនបម្រើ MCP ជាមួយដឹកជញ្ជូនផ្អែកលើ HTTP សុវត្ថិភាពក្លាយជាបញ្ហាសំខាន់ដែលត្រូវការយកចិត្តទុកដាក់យ៉ាងម៉ត់ចត់ចំពោះចំនុចក្មេងក្នុងការវាយប្រហារ និងយានការពារ។

### ទិដ្ឋភាពទូទៅ

សុវត្ថិភាពមានសារៈសំខាន់ខ្លាំងពេលបើកច្រកម៉ាស៊ីនបម្រើ MCP តាម HTTP។ Streamable HTTP បង្កើតផ្ទៃប្រលោមបញ្ជារថ្មី និងត្រូវការការកំណត់សម្របសម្រួលយ៉ាងម៉ត់ចត់។

នេះជាការពិចារណាសុវត្ថិភាពសំខាន់ៗ៖

- **ការផ្ទៀងផ្ទាត់ Origin Header**: ត្រូវតែផ្ទៀងផ្ទាត់ក្បាល `Origin` ជានិច្ច ដើម្បីទប់ស្កាត់ការវាយប្រហាររបស់ DNS rebinding។
- **ការចងភ្ជាប់ Localhost**: សម្រាប់ការអភិវឌ្ឍន៍ក្នុងស្រុក ចងភ្ជាប់ម៉ាស៊ីនបម្រើទៅ `localhost` ដើម្បីរក្សាការមិនឲ្យវាបង្ហាញទៅសាធារណៈអ៊ីនធឺណិត។
- **ការផ្ទៀងផ្ទាត់អត្តសញ្ញាណ**: ទទួលយកការផ្ទៀងផ្ទាត់អត្តសញ្ញាណ (ឧ. កូនសោ API, OAuth) សម្រាប់ការដាក់បញ្ចូលផលិតផល។
- **CORS**: កំណត់គោលការណ៍ការចែករំលែកប្រភពឆ្លង (CORS) ដើម្បីដាក់កំណត់ការចូលប្រើ។
- **HTTPS**: ប្រើ HTTPS ក្នុងការផលិតដើម្បីសម្ងាត់ចរាចរណ៍ទិន្នន័យ។

### អនុវត្តល្អបំផុត

រួចរាល់នេះ គឺជាអនុវត្តល្អបំផុតខ្លះៗដែលត្រូវធ្វើនៅពេលអនុវត្តសុវត្ថិភាពនៅម៉ាស៊ីនបម្រើ MCP របស់អ្នក៖

- កុំទុកចិត្តសំណើដែលមកដល់ដោយគ្មានការផ្ទៀងផ្ទាត់។
- កត់ត្រា និងត្រួតពិនិត្យការចូលប្រើ និងកំហុសទាំងអស់។
- ធ្វើបច្ចុប្បន្នភាពអាស្រ័យភាពជាប្រចាំ ដើម្បីជាអ្នកជួសជុលកំហុសសុវត្ថិភាព។

### បញ្ហា

អ្នកនឹងជួបបញ្ហាខ្លះៗនៅពេលអនុវត្តសុវត្ថិភាពនៅម៉ាស៊ីនបម្រើ MCP streaming៖

- តុល្យភាពរវាងសុវត្ថិភាព និងភាពងាយស្រួលក្នុងការអភិវឌ្ឍ
- ប្រាកដភាពសមរម្យជាមួយបរិយាកាសអតិថិជននានា

### បេសកកម្ម៖ បង្កើតកម្មវិធី MCP Streaming របស់អ្នកផ្ទាល់

**ស្ថានការណ៍៖**
បង្កើតម៉ាស៊ីនបម្រើ និងអតិថិជន MCP ដែលម៉ាស៊ីនបម្រើដំណើរការបញ្ជីវត្ថុមួយ (ឧ. ឯកសារ ឬឯកសារយោង) ហើយផ្ញើជូនដំណឹងសម្រាប់វត្ថុដែលបានដំណើរការ។ អតិថិជនគួរតែបង្ហាញជូនដំណឹងនីមួយៗនៅពេលវាចូល។

**ជំហាន៖**

1. អនុវត្តឧបករណ៍ម៉ាស៊ីនបម្រើមួយដែលដំណើរការបញ្ជី ហើយផ្ញើជូនដំណឹងសម្រាប់វត្ថុនីមួយៗ។
2. អនុវត្តអតិថិជនមួយដែលមានអ្នកដំណើរការសាររួមដើម្បីបង្ហាញជូនដំណឹងពេលវាពិតប្រាកដ។
3. សាកល្បងការអនុវត្តរបស់អ្នកដោយហូបដំណើរការម៉ាស៊ីនបម្រើ និងអតិថិជន ទាំងពីរ ហើយមើលការជូនដំណឹង។
