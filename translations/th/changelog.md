# ประวัติการเปลี่ยนแปลง: หลักสูตร MCP สำหรับผู้เริ่มต้น

เอกสารนี้ใช้เป็นบันทึกของการเปลี่ยนแปลงที่สำคัญทั้งหมดที่ทำกับหลักสูตร Model Context Protocol (MCP) สำหรับผู้เริ่มต้น การเปลี่ยนแปลงถูกบันทึกในลำดับย้อนกลับ (การเปลี่ยนแปลงล่าสุดก่อน)

## 16 มิถุนายน 2026

### การปรับสเปค MCP และการตรวจสอบตัวอย่าง

ตรวจสอบหลักสูตรเทียบกับ **สเปค MCP 2025-11-25** ปัจจุบันและ SDK อย่างเป็นทางการล่าสุด จากนั้นแก้ไขอ้างอิงสเปคเก่าที่เหลือและยืนยันว่าตัวอย่างหลักยังคงคอมไพล์และใช้งานได้

#### การแก้ไขเวอร์ชันสเปค (2025-06-18 / 2025-03-26 → 2025-11-25)

ปรับปรุงเนื้อหาอังกฤษที่ยังกล่าวว่าเวอร์ชันสเปคเก่าเป็นมาตรฐาน *ปัจจุบัน/ล่าสุด* และเปลี่ยนลิงก์ไปยังเส้นทางสเปค canonical `modelcontextprotocol.io`:
- **05-AdvancedTopics/mcp-security/README.md**: ปรับปรุงแบนเนอร์ "มาตรฐานปัจจุบัน", บทนำ, หัวข้อหลักการความปลอดภัย, หัวข้อข้อกำหนดบังคับ, ส่วน Microsoft Entra ID, ลิงก์อ้างอิงและทรัพยากร รวมถึงหมายเหตุปิดความปลอดภัย (8 ลิงก์) เป็น 2025-11-25
- **05-AdvancedTopics/mcp-transport/README.md**: ปรับปรุงลิงก์สเปคในส่วนทรัพยากรเพิ่มเติมและแบนเนอร์ "มาตรฐานปัจจุบัน" เป็น 2025-11-25
- **05-AdvancedTopics/mcp-realtimesearch/README.md**: แทนที่ลิงก์ความปลอดภัยและความน่าเชื่อถือ `2025-03-26` ที่ล้าสมัยด้วยหน้าการปฏิบัติที่ดีที่สุดด้านความปลอดภัย `2025-11-25`
- **03-GettingStarted/14-sampling/README.md**: ปรับปรุงลิงก์เอกสารการสุ่มอย่างเป็นทางการเป็น 2025-11-25
- **03-GettingStarted/05-stdio-server/README.md**: ปรับปรุงอ้างอิง "สเปค MCP ปัจจุบัน" ในปัจจุบันและลิงก์ทรัพยากรเพิ่มเติมเป็น 2025-11-25 (บันทึกการเลิกใช้ SSE ทางประวัติศาสตร์คงไว้เพื่อความถูกต้อง)

#### การตรวจสอบตัวอย่างเทียบกับ SDK ปัจจุบัน

- **TypeScript (03-GettingStarted/01-first-server/solution/typescript)**: `npm install` ติดตั้ง `@modelcontextprotocol/sdk@1.29.0` ได้สำเร็จ; `tsc --noEmit` ผ่านโดยไม่มีข้อผิดพลาดชนิด — API `McpServer`/`StdioServerTransport` ที่มีอยู่ยังถูกต้อง
- **Python (03-GettingStarted/01-first-server/solution/python)**: ตรวจสอบใน `.venv` แยกด้วย `mcp[cli]` (1.27.2); `py_compile` ผ่านและ `FastMCP.list_tools()` ส่งคืนเครื่องมือ `add` และ `subtract` อย่างถูกต้อง
- ยืนยันว่าช่วงเวอร์ชันตัวอย่าง `@modelcontextprotocol/sdk` ทั้งหมด (`>=1.26.0` / `^1.26.0` / `^1.27.0`) สามารถแก้ไขเป็นเวอร์ชันปัจจุบัน `1.29.0` โดยไม่มีการเปลี่ยนแปลง API ที่ทำให้ใช้งานไม่ได้

#### การจัดตำแหน่งพินการพึ่งพา (เติมช่องว่างเวอร์ชัน)

ปรับพิน SDK ที่ล้าสมัยเพื่อให้ตัวอย่างทุกตัวติดตามการเผยแพร่ MCP ปัจจุบัน สอดคล้องกับแนวปฏิบัติทั่วทั้งรีโป:
- **03-GettingStarted/05-stdio-server/solution/typescript/package.json**: ปรับพิน `@modelcontextprotocol/sdk` จาก `^1.8.0` → `>=1.26.0` และปรับคำอธิบายแพ็กเกจที่ล้าสมัย `"updated for MCP 2025-06-18"` เป็น `"aligned with MCP Specification 2025-11-25"`
- **10-StreamliningAIWorkflows.../lab3/code/weather_mcp/pyproject.toml** และ **lab4/code/github_mcp_server/pyproject.toml**: ปรับพิน `mcp==1.23.0` → `mcp>=1.26.0`; สร้างไฟล์ `uv.lock` ใหม่ทั้งสอง (`uv lock`) เพื่อให้ไฟล์ล็อกแก้ไขเป็น `mcp 1.27.2` ปัจจุบันและสอดคล้องกับไฟล์รายการ

#### การวิเคราะห์ช่องว่างหลักสูตร — การครอบคลุมฟีเจอร์สเปคล่าสุด

### การเพิ่มความแข็งแกร่งด้านความปลอดภัย & การแก้ไขช่องโหว่พึ่งพา

ดำเนินการตรวจสอบความปลอดภัยเต็มรูปแบบในทุกไฟล์รายการพึ่งพาและซอร์สโค้ดตัวอย่าง จากนั้นแก้ไขคำเตือน npm ทั้งหมดและพบช่องโหว่ในโค้ดหนึ่งรายการ หลังการแก้ไข, `npm audit` แสดง **0 ช่องโหว่** ทุกไดเรกทอรีที่ตรวจสอบ

#### ช่องโหว่พึ่งพา npm (ผ่านทาง) — แก้ไขแล้ว

ตรวจสอบไฟล์ `package-lock.json` จำนวน 15 ไฟล์ที่คอมมิต ช่องโหว่จำกัดอยู่ที่พึ่งพาผ่านทางที่ดึงมาโดยเครื่องมือพัฒนา MCP Inspector, ลูกค้า OpenAI และ MCP SDK; ทั้งหมดแก้ไขได้โดยไม่ทำให้ตัวอย่างเสียหาย:
- **10-StreamliningAIWorkflows.../lab4/code/github_mcp_server/inspector** และ **lab3/code/weather_mcp/inspector**: ปรับรุ่น `@modelcontextprotocol/inspector` (`0.16.6` / `0.14.1` → `0.22.0`) ซึ่งแก้ไขคำเตือนของแพ็กเกจที่บันเดิลมาเช่น `ajv`, `brace-expansion`, `diff`, `path-to-regexp` และ `ws` เพิ่มบรรทัด npm `overrides` กำหนด `shell-quote@1.8.4` ที่แก้ไขแล้วเพื่อขจัดคำเตือนวิกฤตที่เหลือจาก `concurrently`; สร้างไฟล์ล็อกใหม่ทั้งสอง (ตอนนี้ 0 ช่องโหว่)
- **03-GettingStarted/samples/typescript**: ใช้คำสั่ง `npm audit fix` อัปเดตแพ็กเกจผ่านทาง `qs` (ระดับกลาง) เป็นเวอร์ชันแก้ไข
- **03-GettingStarted/samples/javascript**: ใช้คำสั่ง `npm audit fix` อัปเดตแพ็กเกจผ่านทาง `hono` (ระดับกลาง) เป็นเวอร์ชันแก้ไข
- **03-GettingStarted/03-llm-client/solution/typescript**: ใช้คำสั่ง `npm audit fix` อัปเดตแพ็กเกจผ่านทาง `form-data` (ระดับสูง) เป็นเวอร์ชันแก้ไข
- **03-GettingStarted/11-simple-auth/solution/typescript**: สร้างไฟล์ `package-lock.json` ที่ขาดหายไปเพื่อให้โครงการสร้างซ้ำและตรวจสอบได้ (0 ช่องโหว่)

#### การแก้ไขความปลอดภัยระดับโค้ด (OWASP A03: Injection)

- **10-StreamliningAIWorkflows.../lab4/code/github_mcp_server/src/server.py**: ลบ `shell=True` ออกจากเครื่องมือ `open_in_vscode` คำสั่งก่อนหน้านี้คือ `subprocess.run(["start", "", vscode_path, folder_path], shell=True)` อนุญาตให้เครื่องหมายพิเศษของเชลล์ในเส้นทางโฟลเดอร์ถูกตีความโดย `cmd.exe` (ช่องโหว่การฉีดคำสั่ง) ตอนนี้จะเรียกไฟล์ `Code.exe` ที่แก้ไขเส้นทางแล้วโดยตรงพร้อมโฟลเดอร์เป็นอาร์กิวเมนต์ — ไม่มีเชลล์ — ซึ่งเทียบเท่าฟังก์ชันและปลอดภัย

- ตรวจสอบทุกชุดความต้องการ Python ด้วย `pip-audit` พบว่า `05-AdvancedTopics` และ `03-GettingStarted/samples/python` รายงานว่า **ไม่มีช่องโหว่ที่รู้จัก** (ช่วงเวอร์ชัน `mcp` / `httpx` / `pydantic` / `python-dotenv` ของพวกเขาแก้ไขเป็นเวอร์ชันที่แก้ไขแล้ว)

#### MCP Security Best Practices (02-Security/mcp-best-practices.md) - อัปเดตเวอร์ชันสเปค
- **อัปเดตเวอร์ชันโปรโตคอล**: อัปเดตให้ใช้อ้างอิง MCP Specification ล่าสุด 2025-11-25 (ปล่อย 25 พฤศจิกายน 2025)
  - อัปเดตการอ้างอิงเวอร์ชันสเปคทั้งหมดจาก 2025-06-18 เป็น 2025-11-25
  - อัปเดตการอ้างอิงวันที่ในเอกสารจาก 18 สิงหาคม 2025 เป็น 18 ธันวาคม 2025
  - ตรวจสอบให้แน่ใจว่าลิงก์สเปคทั้งหมดชี้ไปที่เอกสารปัจจุบัน
- **การตรวจสอบเนื้อหา**: ตรวจสอบอย่างละเอียดแนวปฏิบัติที่ดีที่สุดด้านความปลอดภัยตามมาตรฐานล่าสุด
  - **โซลูชันความปลอดภัยของ Microsoft**: ตรวจสอบคำศัพท์และลิงก์ใน Prompt Shields (ก่อนหน้านี้ "การตรวจจับความเสี่ยง Jailbreak"), Azure Content Safety, Microsoft Entra ID, และ Azure Key Vault
  - **ความปลอดภัย OAuth 2.1**: ยืนยันความสอดคล้องกับแนวปฏิบัติที่ดีที่สุดด้านความปลอดภัย OAuth ล่าสุด
  - **มาตรฐาน OWASP**: ตรวจสอบให้แน่ใจว่าอ้างอิง OWASP Top 10 สำหรับ LLMs ยังใช้งานได้
  - **บริการ Azure**: ตรวจสอบลิงก์เอกสารและแนวปฏิบัติที่ดีที่สุดทั้งหมดของ Microsoft Azure
- **ความสอดคล้องกับมาตรฐาน**: ยืนยันว่ามาตรฐานความปลอดภัยทั้งหมดที่อ้างอิงยังเป็นปัจจุบัน
  - กรอบการจัดการความเสี่ยง AI ของ NIST
  - ISO 27001:2022
  - แนวปฏิบัติที่ดีที่สุดด้านความปลอดภัย OAuth 2.1
  - กรอบการรักษาความปลอดภัยและการปฏิบัติตามของ Azure
- **แหล่งข้อมูลการใช้งานจริง**: ตรวจสอบลิงก์คู่มือและทรัพยากรสำหรับการใช้งานจริงทั้งหมด
  - รูปแบบการตรวจสอบสิทธิ์ Azure API Management
  - คู่มือการรวม Microsoft Entra ID
  - การบริหารจัดการความลับใน Azure Key Vault
  - ท่อส่ง DevSecOps และโซลูชันการตรวจสอบ

### การประกันคุณภาพเอกสาร
- **ความสอดคล้องกับสเปค**: ตรวจสอบให้แน่ใจว่าข้อกำหนดความปลอดภัยที่บังคับใช้ทั้งหมดใน MCP (MUST/MUST NOT) สอดคล้องกับสเปคล่าสุด
- **ความทันสมัยของแหล่งข้อมูล**: ตรวจสอบลิงก์ภายนอกทั้งหมดไปยังเอกสาร Microsoft, มาตรฐานความปลอดภัย และคู่มือการใช้งาน
- **ความครอบคลุมแนวปฏิบัติที่ดีที่สุด**: ยืนยันความครอบคลุมครบถ้วนของการรับรอง, การอนุญาต, ภัยคุกคามเฉพาะ AI, ความปลอดภัยซัพพลายเชน, และรูปแบบองค์กร

  - **เซิร์ฟเวอร์แบบปกติกับแบบระดับต่ำ**: เปรียบเทียบรายละเอียดและตัวอย่างโค้ด Python และ TypeScript สำหรับทั้งสองแนวทาง
  - **การออกแบบแบบใช้ผู้จัดการ**: อธิบายการจัดการเครื่องมือ/ทรัพยากร/พรอมต์โดยใช้ตัวจัดการสำหรับการใช้งานเซิร์ฟเวอร์ที่ยืดหยุ่นและปรับขนาดได้
  - **รูปแบบการใช้งานจริง**: กรณีศึกษาจริงที่รูปแบบเซิร์ฟเวอร์ระดับต่ำมีประโยชน์สำหรับฟีเจอร์ขั้นสูงและสถาปัตยกรรม

#### การรับรองความถูกต้องง่าย ๆ (03-GettingStarted/11-simple-auth)
- **เพิ่มบทใหม่**: คู่มือทีละขั้นตอนในการใช้งานการรับรองความถูกต้องง่าย ๆ ในเซิร์ฟเวอร์ MCP
  - **แนวคิดการรับรองความถูกต้อง**: อธิบายความแตกต่างระหว่างการรับรองความถูกต้องกับการอนุญาต และการจัดการข้อมูลรับรอง
  - **การใช้งานการรับรองความถูกต้องพื้นฐาน**: รูปแบบการรับรองความถูกต้องแบบมิดเดิลแวร์ใน Python (Starlette) และ TypeScript (Express) พร้อมตัวอย่างโค้ด
  - **ก้าวสู่ความปลอดภัยขั้นสูง**: คำแนะนำในการเริ่มต้นด้วยการรับรองความถูกต้องง่าย ๆ และก้าวสู่ OAuth 2.1 และ RBAC พร้อมการอ้างอิงโมดูลความปลอดภัยขั้นสูง

การเพิ่มเหล่านี้ให้คำแนะนำเชิงปฏิบัติสำหรับการสร้างการใช้งานเซิร์ฟเวอร์ MCP ที่มั่นคง ปลอดภัย และยืดหยุ่นมากขึ้น เชื่อมโยงแนวคิดพื้นฐานกับรูปแบบการใช้งานขั้นสูงในสภาพแวดล้อมการผลิต

    - ความสามารถสตรีมมิ่งเรียลไทม์สำหรับสถานการณ์ต้องการความหน่วงต่ำ

    - การประมวลผล MCP แบบกระจายผ่านหลาย Azure Functions
    - สถาปัตยกรรมขนส่งไฮบริดที่รวมประเภทขนส่งหลายแบบ
    - กลยุทธ์ความทนทานของข้อความ, ความน่าเชื่อถือ และการจัดการข้อผิดพลาด

    - การตรวจสอบสิทธิ์ด้วย managed identity และการเข้าถึงแบบน้อยที่สุด
    - เทเลเมทรี Application Insights และการตรวจสอบประสิทธิภาพ

    - ความพิจารณาการทดสอบประสิทธิภาพและโหลด

#### แนวคิดหลัก (01-CoreConcepts/) - ปรับปรุงใหม่ทั้งหมด  
- **อัปเดตเวอร์ชันโปรโตคอล**: ปรับปรุงให้อ้างอิงกับข้อกำหนด MCP ปัจจุบัน 2025-06-18 พร้อมเวอร์ชันตามวันที่ (รูปแบบ YYYY-MM-DD)  
- **ปรับปรุงสถาปัตยกรรม**: ขยายคำอธิบายของ Hosts, Clients และ Servers ให้สะท้อนรูปแบบสถาปัตยกรรม MCP ปัจจุบัน  
  - Hosts กำหนดชัดเจนว่าเป็นแอปพลิเคชัน AI ที่ประสานการเชื่อมต่อ MCP client หลายตัว  
  - Clients อธิบายว่าเป็นตัวเชื่อมโปรโตคอลที่รักษาความสัมพันธ์หนึ่งต่อหนึ่งกับเซิร์ฟเวอร์  
  - Servers ปรับปรุงเพื่อครอบคลุมสถานการณ์การปรับใช้ในพื้นที่และระยะไกล  
- **ปรับโครงสร้าง Primitive**: ปรับปรุงใหม่ทั้งหมดสำหรับ server และ client primitives  
  - Server Primitives: ทรัพยากร (แหล่งข้อมูล), Prompts (แม่แบบ), เครื่องมือ (ฟังก์ชันที่ดำเนินการได้) พร้อมคำอธิบายและตัวอย่างละเอียด  
  - Client Primitives: การสุ่มตัวอย่าง (LLM completions), การดึงข้อมูล (user input), การจัดเก็บบันทึก (debugging/monitoring)  
  - อัปเดตด้วยรูปแบบวิธีการค้นหา (`*/list`), ดึงข้อมูล (`*/get`), และการเรียกใช้งาน (`*/call`) ปัจจุบัน  
- **สถาปัตยกรรมโปรโตคอล**: แนะนำแบบจำลองสถาปัตยกรรมสองชั้น  
  - ชั้นข้อมูล: พื้นฐาน JSON-RPC 2.0 พร้อมการจัดการวงจรชีวิตและ primitives  
  - ชั้นขนส่ง: STDIO (ท้องถิ่น) และ Streamable HTTP พร้อม SSE (ระยะไกล)  
- **กรอบความปลอดภัย**: หลักการความปลอดภัยครบถ้วนรวมถึงความยินยอมของผู้ใช้ที่ชัดเจน, การปกป้องข้อมูลส่วนบุคคล, ความปลอดภัยการทำงานของเครื่องมือ และความปลอดภัยชั้นการขนส่ง  
- **รูปแบบการสื่อสาร**: ปรับปรุงข้อความโปรโตคอลเพื่อแสดงกระบวนการเริ่มต้น, ค้นหา, ดำเนินงาน และแจ้งเตือน  
- **ตัวอย่างโค้ด**: อัปเดตตัวอย่างหลายภาษา (.NET, Java, Python, JavaScript) ให้สะท้อนแบบแผน SDK MCP ปัจจุบัน  

#### ความปลอดภัย (02-Security/) - ปรับปรุงระบบความปลอดภัยอย่างครบถ้วน  
- **สอดคล้องมาตรฐาน**: ปรับให้สอดคล้องเต็มรูปแบบกับข้อกำหนดความปลอดภัย MCP 2025-06-18  
- **วิวัฒนาการการตรวจสอบสิทธิ์**: จัดทำเอกสารวิวัฒนาการจากเซิร์ฟเวอร์ OAuth แบบกำหนดเองไปสู่การมอบหมายผู้ให้บริการข้อมูลประจำตัวภายนอก (Microsoft Entra ID)  
- **การวิเคราะห์ภัยคุกคามเฉพาะ AI**: ขยายความครอบคลุมการโจมตี AI สมัยใหม่  
  - สถานการณ์การโจมตีการฉีด prompt อย่างละเอียดพร้อมตัวอย่างจริง  
  - กลไกการวางยาพิษเครื่องมือและรูปแบบการโจมตี "rug pull"  
  - การวางยาพิษบริบทหน้าต่างและการโจมตีความสับสนแบบโมเดล  
- **โซลูชันความปลอดภัย AI ของ Microsoft**: ครอบคลุมระบบนิเวศความปลอดภัย Microsoft อย่างครบถ้วน  
  - AI Prompt Shields พร้อมเทคนิคการตรวจจับขั้นสูง, การเน้นจุด และการใช้ตัวคั่น  
  - รูปแบบการผสานรวม Azure Content Safety  
  - GitHub Advanced Security สำหรับการปกป้องห่วงโซ่อุปทาน  
- **การบรรเทาภัยคุกคามขั้นสูง**: การควบคุมความปลอดภัยอย่างละเอียดสำหรับ  
  - การแฮ็กเซสชันด้วยสถานการณ์การโจมตีเฉพาะ MCP และข้อกำหนด session ID แบบเข้ารหัส  
  - ปัญหา Confused deputy ในสถานการณ์พร็อกซี MCP พร้อมข้อกำหนดความยินยอมชัดเจน  
  - ช่องโหว่การส่งผ่านโทเค็นพร้อมการควบคุมการตรวจสอบบังคับ  
- **ความปลอดภัยห่วงโซ่อุปทาน**: ขยายความครอบคลุมห่วงโซ่อุปทาน AI รวมถึงโมเดลพื้นฐาน, บริการฝัง (embeddings), ผู้ให้บริการบริบท และ API จากบุคคลที่สาม  
- **ความปลอดภัยพื้นฐาน**: ปรับปรุงการผสานรวมกับรูปแบบความปลอดภัยองค์กร รวมถึงสถาปัตยกรรม zero trust และระบบนิเวศความปลอดภัย Microsoft  
- **การจัดระเบียบทรัพยากร**: แบ่งหมวดหมู่ลิงก์ทรัพยากรอย่างครบถ้วนตามประเภท (เอกสารทางการ, มาตรฐาน, งานวิจัย, โซลูชัน Microsoft, คู่มือการใช้งาน)  

### การปรับปรุงคุณภาพเอกสาร  
- **วัตถุประสงค์การเรียนรู้ที่มีโครงสร้าง**: ปรับปรุงวัตถุประสงค์การเรียนรู้ให้มีผลลัพธ์ชัดเจนและปฏิบัติได้  
- **การอ้างอิงข้าม**: เพิ่มลิงก์เชื่อมโยงระหว่างหัวข้อความปลอดภัยและแนวคิดหลักที่เกี่ยวข้อง  
- **ข้อมูลปัจจุบัน**: ปรับปรุงการอ้างอิงวันที่และลิงก์ข้อกำหนดสู่มาตรฐานปัจจุบันทั้งหมด  
- **คำแนะนำการใช้งาน**: เพิ่มแนวทางการใช้งานเฉพาะที่ปฏิบัติได้ตลอดทั้งสองส่วน

- ดำเนินการตัวอย่างเซิร์ฟเวอร์ Server-Sent Events (SSE)  

### แนวคิดหลัก (01-CoreConcepts/)  
- เพิ่มคำอธิบายรายละเอียดสถาปัตยกรรม client-server  
- สร้างเอกสารเกี่ยวกับส่วนประกอบหลักของโปรโตคอล  
- จัดทำเอกสารรูปแบบการส่งข้อความใน MCP
