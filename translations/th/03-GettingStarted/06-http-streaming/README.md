# การสตรีม HTTPS ด้วยโปรโตคอล Model Context (MCP)

บทนี้เป็นคู่มือครบวงจรสำหรับการใช้งานสตรีมมิ่งแบบปลอดภัย, ขยายระบบได้ และสตรีมแบบเรียลไทม์โดยใช้โปรโตคอล Model Context (MCP) ผ่าน HTTPS ครอบคลุมแรงจูงใจในการสตรีมกลไกการขนส่งที่มีพร้อม วิธีการใช้งาน HTTP ที่สตรีมได้ใน MCP แนวทางปฏิบัติที่ดีที่สุดด้านความปลอดภัย การย้ายจาก SSE และคำแนะนำเชิงปฏิบัติสำหรับการสร้างแอปพลิเคชัน MCP สตรีมมิ่งของคุณเอง 

> **มองไปข้างหน้า:** บทเรียนนี้อธิบาย Streamable HTTP ภายใต้ **MCP Specification 2025-11-25** ที่ซึ่งเซสชันถูกสร้างขึ้นในระหว่าง `initialize` และผูกกับหัวข้อ `Mcp-Session-Id` การปล่อยตัวอย่าง  `2026-07-28` จะตัดการจับมือและ ID เซสชันออกทั้งหมด ทำให้แต่ละคำขอเป็นอิสระและสามารถกำหนดเส้นทางไปยังเซิร์ฟเวอร์ตัวใดก็ได้โดยไม่ต้องมี sticky sessions ดูเพิ่มเติมที่ [What’s Changing in MCP: The 2026-07-28 Release Candidate](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) เพื่อรายละเอียด

## กลไกการขนส่งและการสตรีมใน MCP

ส่วนนี้จะสำรวจกลไกการขนส่งต่าง ๆ ที่มีใน MCP และบทบาทของพวกเขาในการเปิดใช้งานความสามารถในการสตรีมสำหรับการสื่อสารแบบเรียลไทม์ระหว่างไคลเอนต์และเซิร์ฟเวอร์

### กลไกการขนส่งคืออะไร?

กลไกการขนส่งคือวิธีการแลกเปลี่ยนข้อมูลระหว่างไคลเอนต์และเซิร์ฟเวอร์ MCP รองรับประเภทการขนส่งหลายประเภทเพื่อให้เหมาะสมกับสภาพแวดล้อมและความต้องการที่แตกต่างกัน:

- **stdio**: อินพุต/เอาต์พุตมาตรฐาน เหมาะสำหรับเครื่องมือในเครื่องและ CLI แบบง่าย แต่ไม่เหมาะสำหรับเว็บหรือคลาวด์
- **SSE (Server-Sent Events)**: ช่วยให้เซิร์ฟเวอร์ส่งข้อมูลอัปเดตแบบเรียลไทม์ไปยังไคลเอนต์ผ่าน HTTP เหมาะสำหรับ UI เว็บ แต่จมีข้อจำกัดด้านความสามารถขยายและความยืดหยุ่น ตั้งแต่ MCP Specification 2025-06-18 การขนส่ง SSE (Server-Sent Events) แบบแยกเดี่ยวถูกเลิกใช้และถูกแทนที่ด้วยการขนส่ง "Streamable HTTP"
- **Streamable HTTP**: กลไกการขนส่งสตรีมมิ่งสมัยใหม่บน HTTP รองรับการแจ้งเตือนและความสามารถขยายที่ดีขึ้น แนะนำให้ใช้สำหรับสถานการณ์การใช้งานในผลิตและคลาวด์ส่วนใหญ่

### ตารางเปรียบเทียบ

ดูตารางเปรียบเทียบด้านล่างเพื่อเข้าใจความแตกต่างระหว่างกลไกการขนส่งเหล่านี้:

| การขนส่ง          | อัปเดตแบบเรียลไทม์ | สตรีมมิ่ง | ขยายระบบได้ | กรณีการใช้งาน           |
|-------------------|------------------|-----------|-------------|-------------------------|
| stdio             | ไม่               | ไม่        | ต่ำ          | เครื่องมือ CLI ในเครื่อง  |
| SSE               | ใช่               | ใช่        | ปานกลาง      | เว็บ, อัปเดตแบบเรียลไทม์ |
| Streamable HTTP   | ใช่               | ใช่        | สูง          | คลาวด์, หลายไคลเอนต์    |

> **คำแนะนำ:** การเลือกกลไกการขนส่งที่เหมาะสมมีผลต่อประสิทธิภาพ การขยายระบบ และประสบการณ์ของผู้ใช้ **Streamable HTTP** แนะนำให้ใช้สำหรับแอปพลิเคชันสมัยใหม่ที่ต้องการขยายตัวและพร้อมสำหรับคลาวด์

สังเกตกลไกการขนส่ง stdio และ SSE ที่คุณได้เห็นในบทก่อนหน้าและกลไกการขนส่ง Streamable HTTP ซึ่งเป็นหัวข้อในบทนี้

## การสตรีม: แนวคิดและแรงจูงใจ

การเข้าใจแนวคิดพื้นฐานและแรงจูงใจเบื้องหลังการสตรีมเป็นสิ่งสำคัญสำหรับการสร้างระบบสื่อสารแบบเรียลไทม์ที่มีประสิทธิภาพ

**การสตรีม** เป็นเทคนิคในโปรแกรมเครือข่ายที่อนุญาตให้ส่งและรับข้อมูลเป็นส่วนย่อยเล็ก ๆ ที่จัดการได้ หรือเป็นลำดับของเหตุการณ์ แทนที่จะรอให้ตอบกลับทั้งหมดพร้อมใช้งาน เทคนิคนี้เป็นประโยชน์อย่างยิ่งสำหรับ:

- ไฟล์หรือชุดข้อมูลขนาดใหญ่
- การอัปเดตแบบเรียลไทม์ (เช่น แชท, แถบความคืบหน้า)
- การคำนวณที่ใช้เวลานานที่ต้องการแจ้งข้อมูลแก่ผู้ใช้

นี่คือสิ่งที่คุณควรรู้เกี่ยวกับการสตรีมในภาพรวม:

- ข้อมูลถูกส่งอย่างต่อเนื่อง ไม่ใช่ทั้งหมดในครั้งเดียว
- ไคลเอนต์สามารถประมวลผลข้อมูลได้เมื่อมาถึง
- ลดความล่าช้าที่รับรู้และปรับปรุงประสบการณ์ผู้ใช้

### ทำไมต้องใช้การสตรีม?

เหตุผลในการใช้การสตรีมมีดังนี้:

- ผู้ใช้ได้รับฟีดแบ็คทันที ไม่ใช่เฉพาะตอนจบ
- เปิดใช้งานแอปพลิเคชันแบบเรียลไทม์และ UI ที่ตอบสนอง
- การใช้ทรัพยากรเครือข่ายและการประมวลผลมีประสิทธิภาพมากขึ้น

### ตัวอย่างง่าย: เซิร์ฟเวอร์ & ไคลเอนต์สตรีม HTTP

นี่คือตัวอย่างง่าย ๆ ว่าการสตรีมสามารถใช้งานได้อย่างไร:

#### Python

**เซิร์ฟเวอร์ (Python, ใช้ FastAPI และ StreamingResponse):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**ไคลเอนต์ (Python, ใช้ requests):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

ตัวอย่างนี้แสดงให้เห็นเซิร์ฟเวอร์ที่ส่งชุดข้อความไปยังไคลเอนต์ในขณะที่ข้อความเหล่านั้นพร้อมใช้งาน แทนที่จะรอให้ข้อความทั้งหมดพร้อม

**ทำงานอย่างไร:**

- เซิร์ฟเวอร์ส่งข้อความแต่ละข้อความทันทีที่พร้อม
- ไคลเอนต์รับและแสดงผลแต่ละส่วนเมื่อมาถึง

**ข้อกำหนด:**

- เซิร์ฟเวอร์ต้องใช้การตอบกลับแบบสตรีมมิ่ง (เช่น `StreamingResponse` ใน FastAPI)
- ไคลเอนต์ต้องประมวลผลการตอบกลับแบบสตรีม (`stream=True` ใน requests)
- Content-Type โดยปกติจะเป็น `text/event-stream` หรือ `application/octet-stream`

#### Java

**เซิร์ฟเวอร์ (Java, ใช้ Spring Boot และ Server-Sent Events):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**ไคลเอนต์ (Java, ใช้ Spring WebFlux WebClient):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**บันทึกการใช้งาน Java:**

- ใช้สแตก reactive ของ Spring Boot กับ `Flux` สำหรับสตรีมมิ่ง
- `ServerSentEvent` ให้การสตรีมแบบเหตุการณ์ที่มีโครงสร้างพร้อมชนิดของเหตุการณ์
- `WebClient` กับ `bodyToFlux()` ทำให้การบริโภคสตรีมแบบ reactive
- `delayElements()` จำลองเวลาประมวลผลระหว่างเหตุการณ์
- เหตุการณ์สามารถมีชนิด (`info`, `result`) เพื่อการจัดการที่ดีขึ้นบนไคลเอนต์

### เปรียบเทียบ: การสตรีมแบบคลาสสิก vs สตรีม MCP

ความแตกต่างระหว่างการสตรีมแบบ "คลาสสิค" และการสตรีมใน MCP สามารถแสดงได้ดังนี้:

| คุณสมบัติ              | การสตรีม HTTP แบบคลาสสิค     | การสตรีม MCP (การแจ้งเตือน)     |
|------------------------|-------------------------------|-------------------------------------|
| การตอบกลับหลัก         | แบ่งเป็นชิ้น                   | ครั้งเดียว ตอนจบ                  |
| อัปเดตความคืบหน้า    | ส่งเป็นข้อมูลชิ้น              | ส่งเป็นข้อความแจ้งเตือน          |
| ข้อกำหนดไคลเอนต์      | ต้องประมวลผลสตรีม              | ต้องใช้ message handler          |
| กรณีการใช้งาน         | ไฟล์ขนาดใหญ่, สตรีมโทเค็น AI  | ความคืบหน้า, บันทึก, ฟีดแบ็คแบบเรียลไทม์ |

### ความแตกต่างสำคัญที่สังเกตได้

นอกจากนี้ นี่คือความแตกต่างที่สำคัญ:

- **รูปแบบการสื่อสาร:**
  - การสตรีม HTTP แบบคลาสสิค: ใช้การเข้ารหัสแบบ chunked transfer ง่าย ๆ เพื่อส่งข้อมูลเป็นชิ้น
  - การสตรีม MCP: ใช้ระบบแจ้งเตือนที่มีโครงสร้างผ่านโปรโตคอล JSON-RPC

- **รูปแบบข้อความ:**
  - HTTP แบบคลาสสิค: ชิ้นข้อความแบบข้อความธรรมดาพร้อมขึ้นบรรทัดใหม่
  - MCP: อ็อบเจ็กต์ LoggingMessageNotification ที่มีเมตาดาต้า

- **การใช้งานบนไคลเอนต์:**
  - HTTP แบบคลาสสิค: ไคลเอนต์ง่าย ๆ ที่ประมวลผลสตรีมได้
  - MCP: ไคลเอนต์ที่ซับซ้อนขึ้นพร้อม message handler เพื่อประมวลผลข้อความประเภทต่าง ๆ

- **การอัปเดตความคืบหน้า:**
  - HTTP แบบคลาสสิค: ความคืบหน้าเป็นส่วนหนึ่งของสตรีมตอบกลับหลัก
  - MCP: ความคืบหน้าถูกส่งผ่านข้อความแจ้งเตือนแยกต่างหากในขณะที่ผลลัพธ์หลักส่งตอนท้าย

### คำแนะนำ

มีบางสิ่งที่เราแนะนำเมื่อพิจารณาระหว่างการใช้งานสตรีมมิ่งแบบคลาสสิค (เป็น endpoint ที่เรานำเสนอข้างต้นผ่าน `/stream`) กับการเลือกสตรีมผ่าน MCP

- **สำหรับความต้องการสตรีมมิ่งแบบง่าย:** การสตรีม HTTP แบบคลาสสิคง่ายต่อการใช้งานและเพียงพอสำหรับความต้องการพื้นฐาน

- **สำหรับแอปพลิเคชันที่ซับซ้อนและมีปฏิสัมพันธ์สูง:** การสตรีม MCP ให้แนวทางที่มีโครงสร้างมากขึ้นพร้อมเมตาดาต้าที่ละเอียดและแยกระหว่างการแจ้งเตือนและผลลัพธ์สุดท้าย

- **สำหรับแอปพลิเคชัน AI:** ระบบการแจ้งเตือนของ MCP มีประโยชน์อย่างยิ่งสำหรับงาน AI ที่ใช้เวลานานที่ต้องการแจ้งความคืบหน้าให้ผู้ใช้ทราบ

## การสตรีมใน MCP

ตกลงแล้ว ตอนนี้คุณได้เห็นคำแนะนำและการเปรียบเทียบเกี่ยวกับความแตกต่างระหว่างการสตรีมแบบคลาสสิคและการสตรีมใน MCP อย่างชัดเจนแล้ว เรามาลงรายละเอียดว่าอย่างไรคุณจะใช้การสตรีมใน MCP ได้

การเข้าใจว่าการสตรีมทำงานอย่างไรในโครงสร้าง MCP เป็นสิ่งจำเป็นสำหรับการสร้างแอปพลิเคชันที่ตอบสนองและให้ฟีดแบ็คแบบเรียลไทม์แก่ผู้ใช้ในระหว่างการทำงานที่ใช้เวลานาน

ใน MCP การสตรีมไม่ได้หมายถึงการส่งผลลัพธ์หลักเป็นชิ้น ๆ แต่หมายถึงการส่ง **ข้อความแจ้งเตือน** ไปยังไคลเอนต์ในขณะที่ตัวเครื่องมือกำลังดำเนินการคำขอ การแจ้งเตือนเหล่านี้อาจรวมถึงการอัปเดตความคืบหน้า, บันทึก, หรือเหตุการณ์อื่น ๆ

### มันทำงานอย่างไร

ผลลัพธ์หลักยังคงถูกส่งเป็นการตอบกลับครั้งเดียว อย่างไรก็ตาม การแจ้งเตือนสามารถส่งเป็นข้อความแยกระหว่างการดำเนินงานเพื่ออัปเดตไคลเอนต์ในแบบเรียลไทม์ ไคลเอนต์ต้องสามารถจัดการและแสดงข้อความแจ้งเตือนได้

## การแจ้งเตือนคืออะไร?

เราพูดถึง "การแจ้งเตือน" หมายความว่าอย่างไรในบริบทของ MCP?

การแจ้งเตือนคือข้อความที่ส่งจากเซิร์ฟเวอร์ไปยังไคลเอนต์เพื่อแจ้งความคืบหน้า, สถานะ, หรือเหตุการณ์อื่น ๆ ระหว่างการดำเนินการที่ใช้เวลานาน การแจ้งเตือนช่วยเพิ่มความโปร่งใสและประสบการณ์ของผู้ใช้

ตัวอย่างเช่น ไคลเอนต์ควรส่งการแจ้งเตือนเมื่อการจับมือเริ่มต้นกับเซิร์ฟเวอร์เสร็จสิ้นแล้ว

การแจ้งเตือนจะมีลักษณะเป็นข้อความ JSON ดังนี้:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

การแจ้งเตือนเป็นส่วนหนึ่งของหัวข้อใน MCP ที่เรียกว่า ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging)

เพื่อให้ระบบ logging ทำงานได้ เซิร์ฟเวอร์ต้องเปิดใช้งานฟีเจอร์/ความสามารถดังนี้:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> ขึ้นอยู่กับ SDK ที่ใช้ การ logging อาจถูกเปิดโดยดีฟอลต์ หรือคุณอาจต้องเปิดใช้งานอย่างชัดเจนในคอนฟิกเซิร์ฟเวอร์ของคุณ

มีประเภทของการแจ้งเตือนดังนี้:

| ระดับ    | คำอธิบาย                       | ตัวอย่างการใช้งาน             |
|-----------|-------------------------------|---------------------------------|
| debug     | ข้อมูลดีเทลสำหรับการดีบัก      | จุดเข้า/ออกของฟังก์ชัน       |
| info      | ข้อความข้อมูลทั่วไป             | อัปเดตความคืบหน้า            |
| notice    | เหตุการณ์ปกติแต่สำคัญ          | การเปลี่ยนแปลงการตั้งค่า     |
| warning   | สถานการณ์เตือน                 | การใช้ฟีเจอร์ที่เลิกใช้        |
| error     | สถานการณ์ผิดพลาด               | ความล้มเหลวในการดำเนินการ    |
| critical  | สถานการณ์วิกฤต                 | ความล้มเหลวของส่วนประกอบระบบ |
| alert     | ต้องดำเนินการทันที             | ตรวจพบการเสียหายของข้อมูล   |
| emergency | ระบบใช้งานไม่ได้ทั้งหมด         | ความล้มเหลวของระบบทั้งระบบ   |

## การใช้งานการแจ้งเตือนใน MCP

ในการใช้งานการแจ้งเตือนใน MCP คุณต้องตั้งค่าเซิร์ฟเวอร์และไคลเอนต์ให้สามารถจัดการอัปเดตแบบเรียลไทม์ได้ ซึ่งช่วยให้แอปพลิเคชันของคุณมอบฟีดแบ็คทันทีให้กับผู้ใช้ในระหว่างการทำงานที่ใช้เวลานาน

### ฝั่งเซิร์ฟเวอร์: การส่งการแจ้งเตือน

เริ่มต้นที่ฝั่งเซิร์ฟเวอร์ ใน MCP คุณกำหนดเครื่องมือที่สามารถส่งการแจ้งเตือนในขณะประมวลผลคำขอ เซิร์ฟเวอร์ใช้วัตถุ context (มักเรียก `ctx`) เพื่อส่งข้อความไปยังไคลเอนต์

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

ในตัวอย่างก่อนหน้า เครื่องมือ `process_files` ส่งการแจ้งเตือนสามครั้งไปยังไคลเอนต์ในขณะที่ประมวลผลแต่ละไฟล์ เมธอด `ctx.info()` ถูกใช้เพื่อส่งข้อความข้อมูล

นอกจากนี้ เพื่อเปิดใช้งานการแจ้งเตือน ให้ตรวจสอบว่าเซิร์ฟเวอร์ของคุณใช้กลไกการขนส่งแบบสตรีมมิ่ง (เช่น `streamable-http`) และไคลเอนต์ของคุณใช้งาน message handler เพื่อประมวลผลการแจ้งเตือน นี่คือตัวอย่างการตั้งค่าเซิร์ฟเวอร์ให้ใช้การขนส่ง `streamable-http`:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

ในตัวอย่าง .NET นี้ เครื่องมือ `ProcessFiles` ถูกตกแต่งด้วยแอตทริบิวต์ `Tool` และส่งการแจ้งเตือนสามครั้งไปยังไคลเอนต์ในขณะที่ประมวลผลแต่ละไฟล์ เมธอด `ctx.Info()` ใช้ส่งข้อความข้อมูล

เพื่อเปิดใช้งานการแจ้งเตือนในเซิร์ฟเวอร์ MCP ของคุณบน .NET ให้แน่ใจว่าคุณใช้กลไกการขนส่งแบบสตรีมมิ่ง:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### ฝั่งไคลเอนต์: การรับการแจ้งเตือน

ไคลเอนต์ต้องใช้งาน message handler เพื่อประมวลผลและแสดงการแจ้งเตือนเมื่อได้รับมา

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

ในโค้ดก่อนหน้า ฟังก์ชัน `message_handler` ตรวจสอบว่าข้อความที่ได้รับเป็นการแจ้งเตือนหรือไม่ หากใช่มันจะแสดงการแจ้งเตือน มิฉะนั้นจะประมวลผลข้อความเซิร์ฟเวอร์ปกติ นอกจากนี้ สังเกตว่า `ClientSession` ถูกสร้างขึ้นพร้อมกับ `message_handler` เพื่อจัดการการแจ้งเตือนที่เข้ามา

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

ในตัวอย่าง .NET นี้ ฟังก์ชัน `MessageHandler` ตรวจสอบว่าข้อความเป็นการแจ้งเตือนหรือไม่ หากใช่จะพิมพ์การแจ้งเตือน มิฉะนั้นจะประมวลผลเป็นข้อความเซิร์ฟเวอร์ปกติ `ClientSession` ถูกสร้างขึ้นพร้อม handler ผ่าน `ClientSessionOptions`

เพื่อเปิดใช้งานการแจ้งเตือน ให้แน่ใจว่าเซิร์ฟเวอร์ของคุณใช้กลไกการขนส่งแบบสตรีมมิ่ง (เช่น `streamable-http`) และไคลเอนต์ของคุณใช้งาน message handler เพื่อประมวลผลการแจ้งเตือน

## การแจ้งเตือนความคืบหน้า & สถานการณ์การใช้งาน

ส่วนนี้อธิบายแนวคิดของการแจ้งเตือนความคืบหน้าใน MCP ว่าทำไมจึงสำคัญ และวิธีการใช้งานโดยใช้ Streamable HTTP คุณจะพบแบบฝึกหัดปฏิบัติเพื่อเสริมความเข้าใจของคุณด้วย

การแจ้งเตือนความคืบหน้าเป็นข้อความแบบเรียลไทม์ที่ส่งจากเซิร์ฟเวอร์ไปยังไคลเอนต์ในระหว่างการทำงานที่ใช้เวลานาน แทนที่จะรอให้กระบวนการทั้งหมดเสร็จ เซิร์ฟเวอร์จะคอยอัปเดตสถานะปัจจุบันให้ไคลเอนต์ทราบ สิ่งนี้ช่วยเพิ่มความโปร่งใส, ประสบการณ์ผู้ใช้ และทำให้การดีบักง่ายขึ้น

**ตัวอย่าง:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### ทำไมต้องใช้การแจ้งเตือนความคืบหน้า?

การแจ้งเตือนความคืบหน้ามีความสำคัญด้วยเหตุผลหลายประการ:

- **ประสบการณ์ผู้ใช้ที่ดีขึ้น:** ผู้ใช้เห็นการอัปเดตขณะทำงาน แทนที่จะเห็นแค่ตอนจบ
- **ฟีดแบ็คแบบเรียลไทม์:** ไคลเอนต์สามารถแสดงแถบความคืบหน้าหรือบันทึก เพิ่มความรู้สึกตอบสนองของแอป
- **ดีบักและตรวจสอบง่ายขึ้น:** นักพัฒนาและผู้ใช้สามารถเห็นว่ากระบวนการใดช้า หรือติดขัด

### วิธีใช้งานการแจ้งเตือนความคืบหน้า

นี่คือวิธีใช้การแจ้งเตือนความคืบหน้าใน MCP:

- **ฝั่งเซิร์ฟเวอร์:** ใช้ `ctx.info()` หรือ `ctx.log()` เพื่อส่งแจ้งเตือนในขณะที่ประมวลผลแต่ละรายการ ข้อความจะถูกส่งก่อนที่ผลลัพธ์หลักจะพร้อม
- **ฝั่งไคลเอนต์:** ใช้งาน message handler ที่ฟังและแสดงการแจ้งเตือนที่มาถึง handler จะสามารถแยกความแตกต่างระหว่างการแจ้งเตือนและผลลัพธ์สุดท้ายได้

**ตัวอย่างฝั่งเซิร์ฟเวอร์:**

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**ตัวอย่างฝั่งไคลเอนต์:**

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## ข้อควรระวังด้านความปลอดภัย

เมื่อติดตั้งเซิร์ฟเวอร์ MCP ที่ใช้การขนส่งผ่าน HTTP ความปลอดภัยเป็นข้อกังวลหลักที่ต้องระมัดระวังในหลายมิติเกี่ยวกับช่องโหว่และกลไกป้องกัน

### ภาพรวม

ความปลอดภัยมีความสำคัญเมื่อเปิดเผยเซิร์ฟเวอร์ MCP บน HTTP Streamable HTTP แนะนำช่องโหว่ใหม่และต้องการการตั้งค่าที่รอบคอบ

### จุดสำคัญ


- **การตรวจสอบ Origin Header**: ตรวจสอบ `Origin` header เสมอเพื่อป้องกันการโจมตีแบบ DNS rebinding
- **การผูกกับ Localhost**: สำหรับการพัฒนาในเครื่อง ให้ผูกเซิร์ฟเวอร์กับ `localhost` เพื่อหลีกเลี่ยงการเปิดเผยไปยังอินเทอร์เน็ตสาธารณะ
- **การพิสูจน์ตัวตน**: นำการพิสูจน์ตัวตนมาใช้ (เช่น API keys, OAuth) สำหรับการนำไปใช้ในสภาพแวดล้อมจริง
- **CORS**: กำหนดนโยบาย Cross-Origin Resource Sharing (CORS) เพื่อจำกัดการเข้าถึง
- **HTTPS**: ใช้ HTTPS ในสภาพแวดล้อมจริงเพื่อเข้ารหัสข้อมูลที่รับส่ง

### แนวทางปฏิบัติที่ดีที่สุด

- อย่าไว้วางใจคำขอเข้าที่ไม่มีการตรวจสอบ
- บันทึกและติดตามการเข้าถึงและข้อผิดพลาดทั้งหมด
- อัปเดตไลบรารีที่ใช้เป็นประจำเพื่อแก้ไขช่องโหว่ด้านความปลอดภัย

### ความท้าทาย

- การสร้างสมดุลระหว่างความปลอดภัยกับความสะดวกในการพัฒนา
- การทำให้เข้ากันได้กับสภาพแวดล้อมของไคลเอนต์หลายแบบ

## การอัปเกรดจาก SSE เป็น Streamable HTTP

สำหรับแอปพลิเคชันที่ใช้งาน Server-Sent Events (SSE) ในปัจจุบัน การย้ายไปใช้ Streamable HTTP จะช่วยเพิ่มความสามารถและความยั่งยืนในระยะยาวสำหรับการใช้งาน MCP ของคุณ

### ทำไมต้องอัปเกรด?

มีเหตุผลสำคัญสองประการในการอัปเกรดจาก SSE เป็น Streamable HTTP:

- Streamable HTTP ให้ความสามารถในการขยายตัว ความเข้ากันได้ และรองรับการแจ้งเตือนที่มากกว่าการใช้ SSE
- เป็นการขนส่งที่แนะนำสำหรับแอป MCP ใหม่ๆ

### ขั้นตอนการย้าย

นี่คือวิธีการย้ายจาก SSE เป็น Streamable HTTP ในแอป MCP ของคุณ:

- **อัปเดตโค้ดฝั่งเซิร์ฟเวอร์** ให้ใช้ `transport="streamable-http"` ใน `mcp.run()`
- **อัปเดตโค้ดฝั่งไคลเอนต์** ให้ใช้ `streamablehttp_client` แทน SSE client
- **นำตัวจัดการข้อความมาใช้** ในไคลเอนต์เพื่อประมวลผลการแจ้งเตือน
- **ทดสอบความเข้ากันได้** กับเครื่องมือและเวิร์กโฟลว์ที่มีอยู่

### การรักษาความเข้ากันได้

แนะนำให้รักษาความเข้ากันได้กับไคลเอนต์ SSE ที่มีอยู่ในระหว่างกระบวนการย้าย นี่คือบางกลยุทธ์:

- คุณสามารถรองรับทั้ง SSE และ Streamable HTTP โดยการรันทั้งสองรูปแบบขนส่งบน endpoints ที่แตกต่างกัน
- ย้ายไคลเอนต์ไปยังขนส่งรูปแบบใหม่อย่างค่อยเป็นค่อยไป

### ความท้าทาย

ให้ตระหนักถึงความท้าทายต่อไปนี้ในระหว่างการย้าย:

- การทำให้แน่ใจว่าไคลเอนต์ทั้งหมดได้รับการอัปเดต
- การจัดการความแตกต่างในการส่งมอบการแจ้งเตือน

## ข้อควรพิจารณาด้านความปลอดภัย

ความปลอดภัยควรเป็นเรื่องสำคัญอันดับแรกเมื่อดำเนินการใช้งานเซิร์ฟเวอร์ใดๆ โดยเฉพาะเมื่อใช้การขนส่งแบบ HTTP เช่น Streamable HTTP ใน MCP

เมื่อดำเนินการเซิร์ฟเวอร์ MCP ด้วยการขนส่งแบบ HTTP ความปลอดภัยจะกลายเป็นข้อกังวลหลักที่ต้องให้ความสนใจอย่างรอบคอบต่อหลายช่องทางการโจมตีและกลไกการป้องกัน

### ภาพรวม

ความปลอดภัยเป็นสิ่งสำคัญเมื่อเปิดเผยเซิร์ฟเวอร์ MCP ผ่าน HTTP Streamable HTTP นำเสนอพื้นผิวการโจมตีใหม่และต้องการการตั้งค่าที่รอบคอบ

นี่คือข้อพิจารณาด้านความปลอดภัยที่สำคัญ:

- **การตรวจสอบ Origin Header**: ตรวจสอบ `Origin` header เสมอเพื่อป้องกันการโจมตีแบบ DNS rebinding
- **การผูกกับ Localhost**: สำหรับการพัฒนาในเครื่อง ให้ผูกเซิร์ฟเวอร์กับ `localhost` เพื่อหลีกเลี่ยงการเปิดเผยไปยังอินเทอร์เน็ตสาธารณะ
- **การพิสูจน์ตัวตน**: นำการพิสูจน์ตัวตนมาใช้ (เช่น API keys, OAuth) สำหรับการนำไปใช้ในสภาพแวดล้อมจริง
- **CORS**: กำหนดนโยบาย Cross-Origin Resource Sharing (CORS) เพื่อจำกัดการเข้าถึง
- **HTTPS**: ใช้ HTTPS ในสภาพแวดล้อมจริงเพื่อเข้ารหัสข้อมูลที่รับส่ง

### แนวทางปฏิบัติที่ดีที่สุด

นอกจากนี้ นี่คือแนวทางปฏิบัติที่ดีที่สุดเมื่อทำการรักษาความปลอดภัยในเซิร์ฟเวอร์ MCP streaming ของคุณ:

- อย่าไว้วางใจคำขอเข้าที่ไม่มีการตรวจสอบ
- บันทึกและติดตามการเข้าถึงและข้อผิดพลาดทั้งหมด
- อัปเดตไลบรารีที่ใช้เป็นประจำเพื่อแก้ไขช่องโหว่ด้านความปลอดภัย

### ความท้าทาย

คุณจะเผชิญกับความท้าทายบางอย่างเมื่อดำเนินการเรื่องความปลอดภัยในเซิร์ฟเวอร์ MCP streaming:

- การสร้างสมดุลระหว่างความปลอดภัยกับความสะดวกในการพัฒนา
- การทำให้เข้ากันได้กับสภาพแวดล้อมของไคลเอนต์หลายแบบ

### การบ้าน: สร้างแอป MCP สตรีมมิ่งของคุณเอง

**สถานการณ์:**
สร้างเซิร์ฟเวอร์และไคลเอนต์ MCP ที่เซิร์ฟเวอร์ประมวลผลรายการไอเท็ม (เช่น ไฟล์ หรือ เอกสาร) และส่งการแจ้งเตือนสำหรับแต่ละไอเท็มที่ประมวลผล ไคลเอนต์จะแสดงการแจ้งเตือนแต่ละรายการเมื่อได้รับ

**ขั้นตอน:**

1. สร้างเครื่องมือเซิร์ฟเวอร์ที่ประมวลผลรายการและส่งการแจ้งเตือนสำหรับแต่ละไอเท็ม
2. สร้างไคลเอนต์ที่มีตัวจัดการข้อความเพื่อแสดงการแจ้งเตือนแบบเรียลไทม์
3. ทดสอบการทำงานโดยรันทั้งเซิร์ฟเวอร์และไคลเอนต์ พร้อมสังเกตการแจ้งเตือน
