# ความปลอดภัย MCP: การปกป้องครบวงจรสำหรับระบบ AI

[![แนวทางปฏิบัติที่ดีที่สุดด้านความปลอดภัย MCP](../../../translated_images/th/03.175aed6dedae133f.webp)](https://youtu.be/88No8pw706o)

_(คลิกที่ภาพด้านบนเพื่อดูวิดีโอบทเรียนนี้)_

ความปลอดภัยเป็นพื้นฐานสำคัญของการออกแบบระบบ AI ซึ่งเป็นเหตุผลที่เราให้ความสำคัญกับมันในส่วนที่สองของเรา นี่สอดคล้องกับหลักการ **Secure by Design** ของ Microsoft จาก [โครงการ Secure Future Initiative](https://www.microsoft.com/security/blog/2025/04/17/microsofts-secure-by-design-journey-one-year-of-success/)

โปรโตคอล Model Context (MCP) นำเสนอความสามารถใหม่ที่ทรงพลังสู่แอปพลิเคชันที่ขับเคลื่อนด้วย AI ในขณะเดียวกันก็แนะนำความท้าทายด้านความปลอดภัยเฉพาะที่เกินกว่าความเสี่ยงซอฟต์แวร์แบบดั้งเดิม ระบบ MCP เผชิญทั้งปัญหาด้านความปลอดภัยที่มีอยู่แล้ว (รหัสที่ปลอดภัย, สิทธิ์น้อยที่สุด, ความปลอดภัยของห่วงโซ่อุปทาน) รวมถึงภัยคุกคามเฉพาะ AI เช่น การฉีดพรวดพราด, การทำพิษเครื่องมือ, การยึดเซสชัน, การโจมตี confused deputy, ช่องโหว่ token passthrough และการแก้ไขความสามารถแบบไดนามิก

บทเรียนนี้สำรวจความเสี่ยงด้านความปลอดภัยที่สำคัญที่สุดในการใช้งาน MCP — ครอบคลุมการพิสูจน์ตัวตน การอนุญาต สิทธิ์เกินความจำเป็น การฉีดพรวดพราดทางอ้อม ความปลอดภัยในเซสชัน ปัญหา confused deputy การจัดการโทเค็น และช่องโหว่ในห่วงโซ่อุปทาน คุณจะได้เรียนรู้การควบคุมที่ใช้งานได้จริงและแนวทางปฏิบัติที่ดีที่สุดเพื่อลดความเสี่ยงเหล่านี้ในขณะที่ใช้โซลูชันของ Microsoft เช่น Prompt Shields, Azure Content Safety และ GitHub Advanced Security เพื่อเสริมสร้างการปรับใช้ MCP ของคุณ

## วัตถุประสงค์การเรียนรู้

เมื่อจบบทเรียนนี้ คุณจะสามารถ:

- **ระบุภัยคุกคามเฉพาะ MCP**: รับรู้ความเสี่ยงด้านความปลอดภัยเฉพาะในระบบ MCP รวมถึงการฉีดพรวดพราด, การทำพิษเครื่องมือ, สิทธิ์เกินจำเป็น, การยึดเซสชัน, ปัญหา confused deputy, ช่องโหว่ token passthrough และความเสี่ยงห่วงโซ่อุปทาน
- **ใช้การควบคุมความปลอดภัย**: นำมาตรการบรรเทาที่มีประสิทธิภาพมาใช้ รวมถึงการพิสูจน์ตัวตนที่แข็งแกร่ง, สิทธิ์น้อยที่สุด, การจัดการโทเค็นที่ปลอดภัย, การควบคุมความปลอดภัยในเซสชัน และการตรวจสอบห่วงโซ่อุปทาน
- **ใช้ประโยชน์จากโซลูชันความปลอดภัยของ Microsoft**: เข้าใจและปรับใช้ Microsoft Prompt Shields, Azure Content Safety และ GitHub Advanced Security เพื่อปกป้องภาระงาน MCP
- **ตรวจสอบความปลอดภัยของเครื่องมือ**: ตระหนักถึงความสำคัญของการตรวจสอบข้อมูลเมตาของเครื่องมือ, การเฝ้าระวังการเปลี่ยนแปลงไดนามิก และการป้องกันการโจมตีด้วยการฉีดพรวดพราดทางอ้อม
- **บูรณาการแนวทางปฏิบัติที่ดีที่สุด**: ผสมผสานพื้นฐานความปลอดภัยที่มีอยู่แล้ว (การเขียนโค้ดที่ปลอดภัย, การเสริมความแข็งแกร่งของเซิร์ฟเวอร์, zero trust) กับการควบคุมเฉพาะ MCP เพื่อการปกป้องที่ครอบคลุม

# สถาปัตยกรรมและการควบคุมความปลอดภัย MCP

การใช้งาน MCP สมัยใหม่ต้องการแนวทางความปลอดภัยแบบหลายชั้นที่จัดการทั้งความปลอดภัยซอฟต์แวร์แบบดั้งเดิมและภัยคุกคามเฉพาะ AI ข้อกำหนด MCP ที่พัฒนาอย่างรวดเร็วยังคงปรับปรุงการควบคุมความปลอดภัยให้ดีขึ้น ทำให้สามารถผสานรวมกับสถาปัตยกรรมความปลอดภัยองค์กรและแนวทางปฏิบัติที่ดีที่สุดได้ดียิ่งขึ้น

งานวิจัยจาก [รายงาน Microsoft Digital Defense](https://aka.ms/mddr) แสดงให้เห็นว่า **98% ของการละเมิดที่รายงานจะถูกป้องกันได้ด้วยการรักษาความสะอาดด้านความปลอดภัยที่เข้มแข็ง** กลยุทธ์การป้องกันที่มีประสิทธิภาพที่สุดคือการผสมผสานแนวทางปฏิบัติด้านความปลอดภัยพื้นฐานกับการควบคุมเฉพาะ MCP — มาตรการความปลอดภัยพื้นฐานที่พิสูจน์แล้วยังคงมีผลกระทบมากที่สุดในการลดความเสี่ยงโดยรวม

## ภูมิทัศน์ความปลอดภัยปัจจุบัน

> **หมายเหตุ:** ข้อมูลนี้สะท้อนมาตรฐานความปลอดภัย MCP ณ วันที่ **5 กุมภาพันธ์ 2026** ซึ่งสอดคล้องกับ **ข้อกำหนด MCP 2025-11-25** โปรโตคอล MCP ยังคงพัฒนาอย่างรวดเร็ว และการใช้งานในอนาคตอาจแนะนำรูปแบบการพิสูจน์ตัวตนใหม่และการควบคุมที่ได้รับการปรับปรุง โปรดอ้างอิง [ข้อกำหนด MCP รุ่นปัจจุบัน](https://spec.modelcontextprotocol.io/), [ที่เก็บ MCP บน GitHub](https://github.com/modelcontextprotocol) และ [เอกสารแนวทางความปลอดภัยที่ดีที่สุด](https://modelcontextprotocol.io/specification/2025-11-25/basic/security_best_practices) เพื่อคำแนะนำล่าสุด

> **มองไปข้างหน้า:** ตัวอย่างรุ่น `2026-07-28` จะเพิ่มความเข้มงวดในด้านการอนุญาตมากขึ้น — ลูกค้าต้องตรวจสอบพารามิเตอร์ `iss` ในการตอบกลับการอนุญาต (RFC 9207), ประกาศ `application_type` ของ OpenID Connect ระหว่างการลงทะเบียนไคลเอนต์แบบไดนามิก และผูกข้อมูลรับรองที่ลงทะเบียนกับเซิร์ฟเวอร์อนุญาตที่ออกให้ ดูที่ [อะไรเปลี่ยนแปลงใน MCP: ตัวอย่างรุ่น 2026-07-28](../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) สำหรับรายการ SEP การอนุญาตทั้งหมด

## 🏔️ การประชุมเชิงปฏิบัติการ MCP Security Summit (Sherpa)

สำหรับ **การฝึกอบรมเชิงปฏิบัติการด้านความปลอดภัย**, เราขอแนะนำอย่างยิ่งให้เข้าร่วม **MCP Security Summit Workshop** (Sherpa) — การเดินทางที่มีไกด์อย่างครบวงจรเพื่อการรักษาความปลอดภัยเซิร์ฟเวอร์ MCP ใน Microsoft Azure

### ภาพรวมเวิร์กช็อป

[MCP Security Summit Workshop](https://azure-samples.github.io/sherpa/) นำเสนอการฝึกอบรมด้านความปลอดภัยที่เป็นรูปธรรมผ่านกระบวนการ "ช่องโหว่ → เจาะระบบ → แก้ไข → ตรวจสอบ" ที่พิสูจน์แล้ว คุณจะ:

- **เรียนรู้จากการทำลายสิ่งต่าง ๆ**: สัมผัสกับช่องโหว่โดยการเจาะเซิร์ฟเวอร์ที่ตั้งใจไม่ปลอดภัย
- **ใช้ความปลอดภัยเนทีฟของ Azure**: ใช้ Azure Entra ID, Key Vault, การจัดการ API และ AI Content Safety
- **ปฏิบัติตาม Defense-in-Depth**: ก้าวผ่านค่ายต่าง ๆ สร้างชั้นความปลอดภัยที่ครอบคลุม
- **นำมาตรฐาน OWASP มาใช้**: ทุกเทคนิคเชื่อมโยงกับ [OWASP MCP Azure Security Guide](https://microsoft.github.io/mcp-azure-security-guide/)
- **รับโค้ดสำหรับใช้งานจริง**: ได้รับการใช้งานจริงและผ่านการทดสอบ

### วิวัฒนาการของการพิสูจน์ตัวตน MCP

ข้อกำหนด MCP ได้พัฒนาอย่างมากในแนวทางการพิสูจน์ตัวตนและอนุญาต:

- **แนวทางเดิม**: ข้อกำหนดในช่วงต้นต้องให้นักพัฒนาสร้างเซิร์ฟเวอร์พิสูจน์ตัวตนแบบกำหนดเอง โดยเซิร์ฟเวอร์ MCP ทำหน้าที่เป็น OAuth 2.0 Authorization Server ที่จัดการการพิสูจน์ตัวตนผู้ใช้โดยตรง
- **มาตรฐานปัจจุบัน (2025-11-25)**: ข้อกำหนดที่อัปเดตอนุญาตให้เซิร์ฟเวอร์ MCP มอบหมายการพิสูจน์ตัวตนให้ผู้ให้บริการระบุตัวตนภายนอก (เช่น Microsoft Entra ID) เพื่อเพิ่มความปลอดภัยและลดความซับซ้อนของการใช้งาน
- **Transport Layer Security**: สนับสนุนกลไกการขนส่งที่ปลอดภัยอย่างเต็มที่ด้วยรูปแบบการพิสูจน์ตัวตนที่เหมาะสมสำหรับการเชื่อมต่อทั้งท้องถิ่น (STDIO) และระยะไกล (Streamable HTTP)

## ความปลอดภัยในการพิสูจน์ตัวตน & อนุญาต

### ความท้าทายความปลอดภัยปัจจุบัน

การใช้งาน MCP สมัยใหม่เผชิญกับความท้าทายหลายอย่างในด้านการพิสูจน์ตัวตนและการอนุญาต:

### ความเสี่ยง & เวกเตอร์ภัยคุกคาม

- **ตรรกะการอนุญาตที่กำหนดค่าผิดพลาด**: การใช้งานการอนุญาตที่ผิดพลาดในเซิร์ฟเวอร์ MCP อาจเปิดเผยข้อมูลที่ละเอียดอ่อนและใช้งานการควบคุมการเข้าถึงผิดพลาด
- **การโจมตี OAuth Token**: การขโมยโทเค็นจากเซิร์ฟเวอร์ MCP ท้องถิ่นทำให้ผู้โจมตีสามารถแอบอ้างเป็นเซิร์ฟเวอร์และเข้าถึงบริการปลายทางได้
- **ช่องโหว่ Token Passthrough**: การจัดการโทเค็นที่ไม่เหมาะสมสร้างช่องโหว่การข้ามการควบคุมความปลอดภัยและช่องว่างความรับผิดชอบ
- **สิทธิ์เกินความจำเป็น**: เซิร์ฟเวอร์ MCP ที่มีสิทธิ์เกินควบคุมละเมิดหลักการสิทธิ์น้อยที่สุดและขยายพื้นผิวการโจมตี

#### Token Passthrough: รูปแบบปฏิเสธที่สำคัญ

**Token passthrough ถูกห้ามอย่างชัดเจน** ในข้อกำหนดการอนุญาต MCP ปัจจุบันเนื่องจากมีผลกระทบร้ายแรงด้านความปลอดภัย:

##### การหลีกเลี่ยงการควบคุมความปลอดภัย
- เซิร์ฟเวอร์ MCP และ API ปลายทางใช้การควบคุมความปลอดภัยสำคัญ (การจำกัดอัตรา, การตรวจสอบคำขอ, การเฝ้าระวังการรับส่งข้อมูล) ที่ขึ้นอยู่กับการตรวจสอบโทเค็นอย่างถูกต้อง
- การใช้โทเค็นโดยตรงจากไคลเอนต์ถึง API ข้ามการป้องกันที่จำเป็นเหล่านี้ ทำลายสถาปัตยกรรมความปลอดภัย

##### ความรับผิดชอบ & ความท้าทายในการตรวจสอบ
- เซิร์ฟเวอร์ MCP ไม่สามารถแยกแยะระหว่างไคลเอนต์ที่ใช้โทเค็นที่ออกโดยต้นทางได้ ทำให้เกิดช่องว่างในการตรวจสอบ
- บันทึกของเซิร์ฟเวอร์ทรัพยากรปลายทางแสดงต้นทางคำขอที่ผิดพลาดแทนผู้ทำหน้าที่เซิร์ฟเวอร์ MCP จริง
- การสอบสวนเหตุการณ์และการตรวจสอบเพื่อความเป็นไปตามข้อกำหนดยากขึ้นอย่างมาก

##### ความเสี่ยงในการขโมยข้อมูล
- การยืนยันโทเค็นที่ไม่มีผลตรวจสอบทำให้ผู้ไม่ประสงค์ดีที่ได้โทเค็นขโมยมาใช้เซิร์ฟเวอร์ MCP เป็นพร็อกซีสำหรับขโมยข้อมูล
- การละเมิดขอบเขตความน่าเชื่อถืออนุญาตให้เข้าถึงโดยไม่ผ่านการควบคุมความปลอดภัยที่ตั้งใจไว้

##### เวกเตอร์การโจมตีหลายบริการ
- โทเค็นที่ถูกโจมตีและยอมรับโดยหลายบริการเปิดใช้การเคลื่อนที่ในแนวข้างผ่านระบบที่เชื่อมต่อ
- สมมติฐานความน่าเชื่อถือระหว่างบริการอาจถูกละเมิดเมื่อไม่สามารถตรวจสอบต้นทางของโทเค็นได้

### การควบคุมความปลอดภัย & มาตรการลดความเสี่ยง

**ข้อกำหนดความปลอดภัยที่สำคัญ:**

> **บังคับใช้:** เซิร์ฟเวอร์ MCP **ต้องไม่** ยอมรับโทเค็นใด ๆ ที่ไม่ได้ออกโดยชัดเจนสำหรับเซิร์ฟเวอร์ MCP นั้น

#### การควบคุมการพิสูจน์ตัวตน & อนุญาต

- **ตรวจสอบการอนุญาตอย่างเข้มงวด**: ดำเนินการตรวจสอบตรรกะอนุญาตของเซิร์ฟเวอร์ MCP อย่างครอบคลุมเพื่อให้แน่ใจว่าเฉพาะผู้ใช้และไคลเอนต์ที่ตั้งใจเท่านั้นที่สามารถเข้าถึงทรัพยากรที่ละเอียดอ่อนได้
  - **คู่มือการใช้งาน**: [Azure API Management เป็นเกตเวย์พิสูจน์ตัวตนสำหรับเซิร์ฟเวอร์ MCP](https://techcommunity.microsoft.com/blog/integrationsonazureblog/azure-api-management-your-auth-gateway-for-mcp-servers/4402690)
  - **การผสานรวมตัวตน**: [การใช้ Microsoft Entra ID สำหรับการพิสูจน์ตัวตนเซิร์ฟเวอร์ MCP](https://den.dev/blog/mcp-server-auth-entra-id-session/)

- **การจัดการโทเค็นอย่างปลอดภัย**: นำแนวทางปฏิบัติที่ดีที่สุดของ [Microsoft สำหรับการตรวจสอบโทเค็นและวงจรชีวิต](https://learn.microsoft.com/en-us/entra/identity-platform/access-tokens) มาใช้
  - ตรวจสอบว่า claims กลุ่มเป้าหมายโทเค็นตรงกับตัวตนเซิร์ฟเวอร์ MCP
  - ดำเนินนโยบายการหมุนเวียนและหมดอายุของโทเค็นอย่างเหมาะสม
  - ป้องกันการโจมตีซ้ำด้วยโทเค็นและการใช้งานโดยไม่ได้รับอนุญาต

- **การจัดเก็บโทเค็นที่ปลอดภัย**: การเก็บโทเค็นอย่างปลอดภัยโดยเข้ารหัสทั้งในระหว่างเก็บและระหว่างส่ง
  - **แนวทางปฏิบัติที่ดีที่สุด**: [แนวทางการจัดเก็บโทเค็นและการเข้ารหัสที่ปลอดภัย](https://youtu.be/uRdX37EcCwg?si=6fSChs1G4glwXRy2)

#### การใช้งานการควบคุมการเข้าถึง

- **หลักการสิทธิ์น้อยที่สุด**: ให้สิทธิ์กับเซิร์ฟเวอร์ MCP เฉพาะที่จำเป็นสำหรับการทำงานที่ตั้งใจเท่านั้น
  - ทบทวนและอัปเดตสิทธิ์อย่างสม่ำเสมอเพื่อป้องกันการเพิ่มสิทธิ์เกินควบคุม
  - **เอกสาร Microsoft**: [การเข้าถึงที่ปลอดภัยสำหรับสิทธิ์น้อยที่สุด](https://learn.microsoft.com/entra/identity-platform/secure-least-privileged-access)

- **การควบคุมการเข้าถึงตามบทบาท (RBAC)**: นำการมอบหมายบทบาทที่ละเอียดถี่ถ้วนมาใช้
  - กำหนดขอบเขตบทบาทอย่างเข้มงวดกับทรัพยากรและการกระทำเฉพาะ
  - หลีกเลี่ยงสิทธิ์กว้างขวางหรือไม่จำเป็นที่ขยายพื้นผิวการโจมตี

- **การตรวจสอบสิทธิ์อย่างต่อเนื่อง**: ดำเนินการตรวจสอบการเข้าถึงและเฝ้าระวังอย่างต่อเนื่อง
  - เฝ้าระวังรูปแบบการใช้สิทธิ์เพื่อหาความผิดปกติ
  - แก้ไขสิทธิ์เกินจำเป็นหรือไม่ได้ใช้ทันที

## ภัยคุกคามความปลอดภัยเฉพาะ AI

### การโจมตีด้วยการฉีดพรวดพราด & การจัดการเครื่องมือ

การใช้งาน MCP สมัยใหม่เผชิญกับเวกเตอร์การโจมตีเฉพาะ AI ที่ซับซ้อนซึ่งมาตรการความปลอดภัยแบบดั้งเดิมไม่สามารถจัดการได้เต็มที่:

#### **การฉีดพรวดพราดทางอ้อม (Cross-Domain Prompt Injection)**

**การฉีดพรวดพราดทางอ้อม** เป็นหนึ่งในช่องโหว่ที่สำคัญที่สุดในระบบ AI ที่เปิดใช้ MCP ผู้โจมตีฝังคำสั่งที่เป็นอันตรายไว้ในเนื้อหาภายนอก — เอกสาร หน้าเว็บ อีเมล หรือแหล่งข้อมูล — ซึ่งระบบ AI จะประมวลผลเป็นคำสั่งที่ถูกต้องตามกฎหมาย

**สถานการณ์การโจมตี:**
- **การฉีดพรวดพราดในเอกสาร**: คำสั่งที่เป็นอันตรายซ่อนไว้ในเอกสารที่ประมวลผล ซึ่งเรียกการทำงาน AI ที่ไม่ตั้งใจ
- **การใช้ประโยชน์จากเนื้อหาเว็บ**: หน้าเว็บที่ถูกโจมตีมีพรวดพราดฝังตัวที่บิดเบือนพฤติกรรม AI เมื่อถูกดึงข้อมูล
- **การโจมตีผ่านอีเมล**: พรวดพราดในอีเมลที่ทำให้ผู้ช่วย AI รั่วไหลข้อมูลหรือทำงานที่ไม่ได้รับอนุญาต
- **การปนเปื้อนแหล่งข้อมูล**: ฐานข้อมูลหรือ API ที่ถูกโจมตีนำเสนอเนื้อหาปนเปื้อนแก่ระบบ AI

**ผลกระทบในโลกจริง**: การโจมตีเหล่านี้อาจทำให้เกิดการขโมยข้อมูล รั่วไหลข้อมูลส่วนบุคคล การสร้างเนื้อหาที่เป็นอันตราย และการบิดเบือนปฏิสัมพันธ์กับผู้ใช้ รายละเอียดเพิ่มเติมดูได้ที่ [การฉีดพรวดพราดใน MCP (Simon Willison)](https://simonwillison.net/2025/Apr/9/mcp-prompt-injection/)

![แผนภาพการโจมตีด้วยการฉีดพรวดพราด](../../../translated_images/th/prompt-injection.ed9fbfde297ca877.webp)

#### **การโจมตีทำพิษเครื่องมือ**

**การทำพิษเครื่องมือ** มุ่งเป้าไปที่ข้อมูลเมต้าที่กำหนดเครื่องมือ MCP โดยใช้ประโยชน์จากวิธีที่ LLMs แปลความหมายคำอธิบายเครื่องมือและพารามิเตอร์เพื่อทำการตัดสินใจรันคำสั่ง

**กลไกการโจมตี:**
- **การปรับแต่งข้อมูลเมตา**: ผู้โจมตีฝังคำสั่งอันตรายในคำอธิบายเครื่องมือ, การกำหนดพารามิเตอร์ หรือในตัวอย่างการใช้งาน
- **คำสั่งที่มองไม่เห็น**: พรวดพราดที่ซ่อนในข้อมูลเมตาเครื่องมือซึ่งประมวลผลโดยโมเดล AI แต่ไม่ปรากฏต่อผู้ใช้มนุษย์
- **การแก้ไขเครื่องมือแบบไดนามิก ("Rug Pulls")**: เครื่องมือที่ผู้ใช้อนุมัติถูกแก้ไขภายหลังเพื่อทำงานที่เป็นอันตรายโดยที่ผู้ใช้ไม่รู้ตัว
- **การฉีดพารามิเตอร์**: เนื้อหาอันตรายฝังในโครงสร้างพารามิเตอร์เครื่องมือซึ่งมีผลต่อพฤติกรรมโมเดล


**ความเสี่ยงของเซิร์ฟเวอร์ที่โฮสต์**: เซิร์ฟเวอร์ MCP ระยะไกลนำมาซึ่งความเสี่ยงที่เพิ่มขึ้นเนื่องจากคำจำกัดความของเครื่องมือสามารถอัปเดตได้หลังจากที่ผู้ใช้อนุมัติในครั้งแรก ซึ่งสร้างสถานการณ์ที่เครื่องมือที่เคยปลอดภัยกลายเป็นอันตราย สำหรับการวิเคราะห์อย่างละเอียด โปรดดูที่ [Tool Poisoning Attacks (Invariant Labs)](https://invariantlabs.ai/blog/mcp-security-notification-tool-poisoning-attacks)

![Tool Injection Attack Diagram](../../../translated_images/th/tool-injection.3b0b4a6b24de6bef.webp)

#### **เวกเตอร์การโจมตี AI เพิ่มเติม**

- **การฉีดข้อความข้ามโดเมน (XPIA)**: การโจมตีขั้นสูงที่ใช้เนื้อหาจากหลายโดเมนเพื่อเลี่ยงการควบคุมความปลอดภัย
- **การปรับเปลี่ยนความสามารถแบบไดนามิก**: การเปลี่ยนแปลงความสามารถของเครื่องมือแบบเรียลไทม์ที่หลีกเลี่ยงการประเมินความปลอดภัยในขั้นต้น
- **การวางยาพิษหน้าต่างบริบท**: การโจมตีโดยการจัดการหน้าต่างบริบทขนาดใหญ่เพื่อซ่อนคำสั่งที่เป็นอันตราย
- **การโจมตีทำให้โมเดลสับสน**: การใช้ข้อจำกัดของโมเดลเพื่อสร้างพฤติกรรมที่ไม่คาดคิดหรือไม่ปลอดภัย


### ผลกระทบของความเสี่ยงด้านความปลอดภัย AI

**ผลลัพธ์ที่กระทบสูง:**
- **การขโมยข้อมูล**: การเข้าถึงและขโมยข้อมูลสำคัญขององค์กรหรือข้อมูลส่วนบุคคลโดยไม่ได้รับอนุญาต
- **การละเมิดความเป็นส่วนตัว**: การเปิดเผยข้อมูลที่ระบุตัวบุคคล (PII) และข้อมูลธุรกิจที่เป็นความลับ  
- **การจัดการระบบโดยไม่ได้ตั้งใจ**: การเปลี่ยนแปลงระบบและกระบวนงานที่สำคัญโดยไม่ได้ตั้งใจ
- **การขโมยข้อมูลรับรอง**: การบุกรุกโทเค็นการตรวจสอบสิทธิ์และข้อมูลรับรองบริการ
- **การเคลื่อนที่แนวนอน**: การใช้ระบบ AI ที่ถูกแทรกแซงเป็นจุดหมุนสำหรับการโจมตีเครือข่ายที่กว้างขึ้น

### โซลูชันความปลอดภัย AI ของ Microsoft

#### **เกราะป้องกันคำสั่ง AI: การป้องกันขั้นสูงจากการโจมตีแบบฉีดคำสั่ง**

Microsoft **เกราะป้องกันคำสั่ง AI** มอบการป้องกันอย่างครอบคลุมทั้งจากการโจมตีแบบฉีดคำสั่งโดยตรงและโดยอ้อมผ่านหลายชั้นความปลอดภัย:

##### **กลไกการป้องกันหลัก:**

1. **การตรวจจับและกรองขั้นสูง**
   - อัลกอริทึมการเรียนรู้ของเครื่องและเทคนิค NLP ตรวจจับคำสั่งที่เป็นอันตรายในเนื้อหาภายนอก
   - การวิเคราะห์เอกสาร เว็บเพจ อีเมล และแหล่งข้อมูลแบบเรียลไทม์เพื่อหาภัยคุกคามฝังตัว
   - การเข้าใจบริบทของรูปแบบคำสั่งที่ถูกต้องเทียบกับคำสั่งที่เป็นอันตราย

2. **เทคนิคการส่องสว่าง**  
   - แยกแยะระหว่างคำสั่งระบบที่น่าเชื่อถือกับอินพุตภายนอกที่อาจถูกเจาะระบบ
   - วิธีการแปลงข้อความที่เพิ่มความเกี่ยวข้องของโมเดลในขณะที่แยกเนื้อหาที่เป็นอันตรายออก
   - ช่วยให้ระบบ AI รักษาลำดับชั้นคำสั่งที่ถูกต้องและละเลยคำสั่งที่ถูกฉีดเข้ามา

3. **ระบบกำหนดขอบเขตและทำเครื่องหมายข้อมูล**
   - กำหนดขอบเขตอย่างชัดเจนระหว่างข้อความระบบที่น่าเชื่อถือกับข้อความอินพุตภายนอก
   - เครื่องหมายพิเศษเน้นขอบเขตระหว่างแหล่งข้อมูลที่น่าเชื่อถือและที่ไม่น่าเชื่อถือ
   - การแยกชัดเจนป้องกันความสับสนของคำสั่งและการดำเนินการคำสั่งโดยไม่ได้รับอนุญาต

4. **ข่าวกรองภัยคุกคามอย่างต่อเนื่อง**
   - Microsoft ตรวจสอบลักษณะการโจมตีที่เกิดขึ้นใหม่และอัปเดตกองกำลังป้องกันอย่างต่อเนื่อง
   - การล่าภัยคุกคามเชิงรุกสำหรับเทคนิคการฉีดใหม่และเวกเตอร์การโจมตี
   - การอัปเดตโมเดลความปลอดภัยเป็นประจำเพื่อรักษาประสิทธิภาพต่อภัยคุกคามที่พัฒนา

5. **การบูรณาการ Azure Content Safety**
   - เป็นส่วนหนึ่งของชุด Azure AI Content Safety ที่ครอบคลุม
   - การตรวจจับเพิ่มเติมสำหรับการพยายามปลดล็อก jailbreak เนื้อหาที่เป็นอันตราย และการละเมิดนโยบายความปลอดภัย
   - การควบคุมความปลอดภัยแบบรวมศูนย์ข้ามองค์ประกอบแอปพลิเคชัน AI

**ทรัพยากรการใช้งาน**: [Microsoft Prompt Shields Documentation](https://learn.microsoft.com/azure/ai-services/content-safety/concepts/jailbreak-detection)

## ภัยคุกคามความปลอดภัย MCP ขั้นสูง

### ช่องโหว่การแย่งชิงเซสชัน

**การแย่งชิงเซสชัน** เป็นเวกเตอร์การโจมตีที่สำคัญใน MCP ที่มีสถานะซึ่งบุคคลที่ไม่ได้รับอนุญาตได้มาซึ่งและใช้งานตัวระบุเซสชันที่ถูกต้องเพื่อปลอมตัวเป็นลูกค้าและดำเนินการที่ไม่ได้รับอนุญาต

#### **สถานการณ์การโจมตีและความเสี่ยง**

- **การฉีดคำสั่งแย่งชิงเซสชัน**: ผู้โจมตีที่มีรหัสเซสชันถูกขโมยฉีดเหตุการณ์ที่เป็นอันตรายเข้าไปในเซิร์ฟเวอร์ที่แชร์สถานะเซสชัน อาจทำให้เกิดการดำเนินการที่เป็นอันตรายหรือเข้าถึงข้อมูลที่ละเอียดอ่อน
- **การปลอมตัวโดยตรง**: รหัสเซสชันที่ถูกขโมยช่วยให้เรียกเซิร์ฟเวอร์ MCP โดยตรงที่เลี่ยงการตรวจสอบสิทธิ์ ทำให้ผู้โจมตีเหมือนเป็นผู้ใช้ที่ถูกต้อง
- **สตรีมที่สามารถทำงานต่อได้ถูกแทรกแซง**: ผู้โจมตีสามารถยุติคำขออย่างกะทันหัน ทำให้ลูกค้าที่ถูกต้องสามารถทำงานต่อด้วยเนื้อหาที่อาจเป็นอันตราย

#### **การควบคุมความปลอดภัยสำหรับการจัดการเซสชัน**

**ข้อกำหนดที่สำคัญ:**
- **การตรวจสอบการอนุญาต**: เซิร์ฟเวอร์ MCP ที่ดำเนินการตรวจสอบการอนุญาต **ต้อง** ตรวจสอบคำขอที่เข้ามาทั้งหมด และ **ต้องไม่** พึ่งพาเซสชันสำหรับการตรวจสอบสิทธิ์
- **การสร้างเซสชันที่ปลอดภัย**: ใช้รหัสเซสชันที่ไม่สามารถทำนายได้และเข้ารหัสด้วยตัวสร้างหมายเลขสุ่มที่ปลอดภัยทางคริปโต
- **การผูกพันกับผู้ใช้เฉพาะ**: ผูกรหัสเซสชันกับข้อมูลผู้ใช้เฉพาะโดยใช้รูปแบบ เช่น `<user_id>:<session_id>` เพื่อป้องกันการใช้งานข้ามผู้ใช้
- **การจัดการวงจรชีวิตเซสชัน**: ดำเนินการหมดอายุ การหมุนเวียน และการยกเลิกอย่างเหมาะสมเพื่อลดช่วงเวลาที่เปราะบาง
- **ความปลอดภัยในการส่งข้อมูล**: ใช้ HTTPS สำหรับการสื่อสารทั้งหมดเพื่อป้องกันการดักจับรหัสเซสชัน

### ปัญหาเจ้าหน้าที่สับสน (Confused Deputy Problem)

ปัญหา **เจ้าหน้าที่สับสน** เกิดขึ้นเมื่อเซิร์ฟเวอร์ MCP ทำหน้าที่เป็นพร็อกซีตรวจสอบสิทธิ์ระหว่างลูกค้าและบริการบุคคลที่สาม ซึ่งสร้างโอกาสให้หลีกเลี่ยงการอนุญาตด้วยการใช้ประโยชน์จากไอดีลูกค้าคงที่

#### **กลไกการโจมตีและความเสี่ยง**

- **การเลี่ยงความยินยอมด้วยคุกกี้**: การตรวจสอบสิทธิ์ของผู้ใช้ก่อนหน้าสร้างคุกกี้ยินยอมที่ผู้โจมตีใช้ประโยชน์ผ่านคำขออนุญาตที่เป็นอันตรายพร้อม URI การเปลี่ยนเส้นทางที่สร้างขึ้น
- **การขโมยรหัสอนุญาต**: คุกกี้ยินยอมที่มีอยู่อาจทำให้เซิร์ฟเวอร์อนุญาตข้ามหน้าจอการยินยอม ส่งรหัสไปยังจุดสิ้นสุดที่ผู้โจมตีควบคุม  
- **การเข้าถึง API โดยไม่ได้รับอนุญาต**: รหัสอนุญาตที่ถูกขโมยจะช่วยแลกเปลี่ยนโทเค็นและปลอมตัวผู้ใช้โดยไม่ต้องได้รับการอนุมัติอย่างชัดเจน

#### **กลยุทธ์การลดความเสี่ยง**

**การควบคุมที่จำเป็น:**
- **ข้อกำหนดความยินยอมอย่างชัดเจน**: เซิร์ฟเวอร์พร็อกซี MCP ที่ใช้ไอดีลูกค้าคงที่ **ต้อง** ได้รับความยินยอมจากผู้ใช้สำหรับลูกค้าที่ลงทะเบียนแบบไดนามิกแต่ละราย
- **การใช้งานความปลอดภัย OAuth 2.1**: ปฏิบัติตามแนวทางปฏิบัติที่ดีที่สุดของ OAuth ปัจจุบัน รวมถึง PKCE (Proof Key for Code Exchange) สำหรับคำขออนุญาตทั้งหมด
- **การตรวจสอบลูกค้าอย่างเข้มงวด**: ดำเนินการตรวจสอบ URI การเปลี่ยนเส้นทางและไอดีลูกค้าอย่างเข้มงวดเพื่อป้องกันการใช้ประโยชน์

### ช่องโหว่การส่งโทเค็นต่อ  

**การส่งโทเค็นต่อ** เป็นแบบแผนที่ไม่พึงประสงค์อย่างชัดเจนที่เซิร์ฟเวอร์ MCP รับโทเค็นของลูกค้าโดยไม่ได้รับการตรวจสอบที่เหมาะสมแล้วส่งต่อไปยัง API ลงสาย ซึ่งละเมิดข้อกำหนดการอนุญาตของ MCP

#### **ผลกระทบด้านความปลอดภัย**

- **การเลี่ยงการควบคุม**: การใช้งานโทเค็นลูกค้าโดยตรงถึง API เลี่ยงการจำกัดอัตรา การตรวจสอบ และการควบคุมการเฝ้าระวังที่สำคัญ
- **การบิดเบือนบันทึกตรวจสอบ**: โทเค็นที่ออกโดยต้นทางทำให้ไม่สามารถระบุตัวลูกค้าได้ ทำลายความสามารถในการตรวจสอบเหตุการณ์
- **การขโมยข้อมูลผ่านพร็อกซี**: โทเค็นที่ไม่ได้ตรวจสอบช่วยให้ผู้ประสงค์ร้ายใช้เซิร์ฟเวอร์เป็นพร็อกซีสำหรับการเข้าถึงข้อมูลโดยไม่ได้รับอนุญาต
- **การละเมิดขอบเขตความน่าเชื่อถือ**: บริการลงสายอาจละเมิดสมมติฐานความน่าเชื่อถือเมื่อไม่สามารถตรวจสอบต้นทางของโทเค็นได้
- **การขยายการโจมตีหลายบริการ**: โทเค็นที่ถูกแทรกแซงและยอมรับในหลายบริการช่วยให้เคลื่อนที่แนวนอน

#### **การควบคุมความปลอดภัยที่จำเป็น**

**ข้อกำหนดที่ไม่สามารถเจรจาต่อรองได้:**
- **การตรวจสอบโทเค็น**: เซิร์ฟเวอร์ MCP **ต้องไม่** รับโทเค็นที่ไม่ได้ออกโดยชัดเจนสำหรับเซิร์ฟเวอร์ MCP
- **การตรวจสอบผู้รับโทเค็น**: ตรวจสอบเสมอว่าคำเรียกร้องผู้รับโทเค็นตรงกับตัวตนของเซิร์ฟเวอร์ MCP
- **วงจรชีวิตโทเค็นที่เหมาะสม**: ใช้โทเค็นเข้าถึงที่มีอายุสั้นพร้อมแนวปฏิบัติการหมุนเวียนอย่างปลอดภัย


## ความปลอดภัยซัพพลายเชนสำหรับระบบ AI

ความปลอดภัยซัพพลายเชนได้พัฒนามากกว่าการพึ่งพาซอฟต์แวร์แบบดั้งเดิมเพื่อครอบคลุมระบบนิเวศ AI ทั้งหมด การใช้งาน MCP สมัยใหม่ต้องตรวจสอบและติดตามส่วนประกอบที่เกี่ยวข้องกับ AI ทั้งหมดอย่างเข้มงวด เนื่องจากแต่ละส่วนประกอบนำเสนอช่องโหว่ที่อาจส่งผลต่อความสมบูรณ์ของระบบ

### ส่วนประกอบที่ขยายของซัพพลายเชน AI

**การพึ่งพาซอฟต์แวร์แบบดั้งเดิม:**
- ไลบรารีและเฟรมเวิร์กโอเพ่นซอร์ส
- อิมเมจคอนเทนเนอร์และระบบฐาน  
- เครื่องมือพัฒนาและสายการสร้าง
- ส่วนประกอบและบริการโครงสร้างพื้นฐาน

**องค์ประกอบซัพพลายเชนเฉพาะ AI:**
- **โมเดลพื้นฐาน**: โมเดลที่ผ่านการฝึกฝนล่วงหน้าจากผู้ให้บริการหลายรายที่ต้องการการตรวจสอบแหล่งที่มา
- **บริการฝังตัว**: บริการเวกเตอร์ไรซ์และการค้นหาความหมายภายนอก
- **ผู้ให้บริบท**: แหล่งข้อมูล ฐานความรู้ และคลังเอกสาร  
- **API บุคคลที่สาม**: บริการ AI ภายนอก, สายงาน ML, และจุดสิ้นสุดการประมวลผลข้อมูล
- **วัตถุโมเดล**: น้ำหนัก การกำหนดค่า และตัวแปรโมเดลที่ปรับแต่งแล้ว
- **แหล่งข้อมูลการฝึกอบรม**: ชุดข้อมูลที่ใช้สำหรับการฝึกอบรมและการปรับแต่งโมเดล

### กลยุทธ์ความปลอดภัยซัพพลายเชนที่ครอบคลุม

#### **การตรวจสอบส่วนประกอบและความน่าเชื่อถือ**
- **การตรวจสอบแหล่งที่มา**: ตรวจสอบต้นทาง ใบอนุญาต และความสมบูรณ์ของส่วนประกอบ AI ทั้งหมดก่อนการรวมเข้า
- **การประเมินความปลอดภัย**: ดำเนินการสแกนช่องโหว่และตรวจสอบความปลอดภัยสำหรับโมเดล แหล่งข้อมูล และบริการ AI
- **การวิเคราะห์ชื่อเสียง**: ประเมินประวัติและแนวปฏิบัติด้านความปลอดภัยของผู้ให้บริการ AI
- **การตรวจสอบความสอดคล้อง**: ตรวจสอบให้แน่ใจว่าส่วนประกอบทั้งหมดสอดคล้องกับข้อกำหนดด้านความปลอดภัยและกฎระเบียบขององค์กร

#### **สายการปรับใช้ที่ปลอดภัย**  
- **การสแกนความปลอดภัยแบบอัตโนมัติ CI/CD**: ผสานรวมการสแกนความปลอดภัยตลอดสายการปรับใช้แบบอัตโนมัติ
- **ความสมบูรณ์ของวัตถุ**: ใช้วิธีการตรวจสอบด้วยคริปโตสำหรับวัตถุที่ปรับใช้ทั้งหมด (โค้ด, โมเดล, การกำหนดค่า)
- **การปรับใช้อย่างเป็นขั้นตอน**: ใช้แนวทางการปรับใช้อย่างก้าวหน้าโดยมีการตรวจสอบความปลอดภัยในแต่ละขั้นตอน
- **ที่เก็บวัตถุที่น่าเชื่อถือ**: ปรับใช้เฉพาะจากที่เก็บวัตถุที่ตรวจสอบและปลอดภัยเท่านั้น

#### **การตรวจสอบและตอบสนองอย่างต่อเนื่อง**
- **การสแกนการพึ่งพิง**: เฝ้าระวังช่องโหว่อย่างต่อเนื่องสำหรับซอฟต์แวร์และส่วนประกอบ AI ที่พึ่งพิงทั้งหมด
- **การติดตามโมเดล**: ประเมินพฤติกรรมโมเดล การเปลี่ยนแปลงประสิทธิภาพ และความผิดปกติด้านความปลอดภัยอย่างต่อเนื่อง
- **การติดตามสุขภาพบริการ**: ตรวจสอบบริการ AI ภายนอกสำหรับความพร้อมใช้งาน เหตุการณ์ความปลอดภัย และการเปลี่ยนนโยบาย
- **การบูรณาการข่าวกรองภัยคุกคาม**: รวมฟีดภัยคุกคามเฉพาะสำหรับความเสี่ยงความปลอดภัย AI และ ML

#### **การควบคุมการเข้าถึงและสิทธิ์ขั้นต่ำสุด**
- **สิทธิ์ในระดับส่วนประกอบ**: จำกัดการเข้าถึงโมเดล ข้อมูล และบริการตามความจำเป็นทางธุรกิจ
- **การจัดการบัญชีบริการ**: ใช้บัญชีบริการเฉพาะพร้อมสิทธิ์ขั้นต่ำที่จำเป็น
- **การแยกเครือข่าย**: แยกส่วนประกอบ AI และจำกัดการเข้าถึงเครือข่ายระหว่างบริการ
- **การควบคุม API Gateway**: ใช้เกตเวย์ API แบบรวมศูนย์เพื่อควบคุมและตรวจสอบการเข้าถึงบริการ AI ภายนอก

#### **การตอบสนองและกู้คืนเหตุการณ์**
- **กระบวนการตอบสนองอย่างรวดเร็ว**: กระบวนการที่จัดตั้งขึ้นสำหรับการติดตั้งแพตช์หรือเปลี่ยนส่วนประกอบ AI ที่ถูกแทรกแซง
- **การหมุนเวียนข้อมูลรับรอง**: ระบบอัตโนมัติสำหรับการหมุนเวียนความลับ คีย์ API และข้อมูลรับรองบริการ
- **ความสามารถในการย้อนกลับ**: ความสามารถในการย้อนกลับไปยังเวอร์ชันที่น่าเชื่อถือก่อนหน้าของส่วนประกอบ AI อย่างรวดเร็ว
- **การกู้คืนจากการละเมิดซัพพลายเชน**: กระบวนการเฉพาะสำหรับการตอบสนองต่อการละเมิดบริการ AI ทางต้นน้ำ

- **การสแกนความลับ**: การตรวจจับอัตโนมัติสำหรับข้อมูลรับรอง คีย์ API และโทเค็นในที่เก็บ
- **การสแกนการพึ่งพิง**: การประเมินช่องโหว่สำหรับการพึ่งพิงและไลบรารีโอเพ่นซอร์ส
- **การวิเคราะห์ CodeQL**: การวิเคราะห์โค้ดแบบสแตติกสำหรับช่องโหว่ด้านความปลอดภัยและปัญหาการเขียนโค้ด
- **ข้อมูลเชิงลึกซัพพลายเชน**: การมองเห็นสถานะสุขภาพและความปลอดภัยของการพึ่งพิง

**การบูรณาการ Azure DevOps & Azure Repos:**
- การผสานรวมการสแกนความปลอดภัยที่ราบรื่นข้ามแพลตฟอร์มพัฒนา Microsoft
- การตรวจสอบความปลอดภัยอัตโนมัติใน Azure Pipelines สำหรับงาน AI
- การบังคับใช้กฎนโยบายสำหรับการปรับใช้ส่วนประกอบ AI ที่ปลอดภัย

**แนวทางปฏิบัติภายใน Microsoft:**
Microsoft ดำเนินการแนวปฏิบัติความปลอดภัยซัพพลายเชนที่ครอบคลุมในผลิตภัณฑ์ทั้งหมด เรียนรู้เกี่ยวกับแนวทางที่พิสูจน์แล้วใน [The Journey to Secure the Software Supply Chain at Microsoft](https://devblogs.microsoft.com/engineering-at-microsoft/the-journey-to-secure-the-software-supply-chain-at-microsoft/)


## แนวทางปฏิบัติความปลอดภัยพื้นฐานสำหรับ Foundation

การใช้งาน MCP สืบทอดและสร้างบนพื้นฐานของท่าทางความปลอดภัยที่มีอยู่ขององค์กรของคุณ การเสริมความเข้มแข็งในแนวทางปฏิบัติความปลอดภัยพื้นฐานจะช่วยเพิ่มความปลอดภัยโดยรวมของระบบ AI และการปรับใช้ MCP อย่างมาก

### หลักความปลอดภัยพื้นฐาน

#### **แนวทางปฏิบัติการพัฒนาที่ปลอดภัย**
- **ความสอดคล้องกับ OWASP**: ปกป้องจากช่องโหว่เว็บแอปพลิเคชัน [OWASP Top 10](https://owasp.org/www-project-top-ten/)
- **การป้องกันเฉพาะ AI**: ใช้มาตรการควบคุมสำหรับ [OWASP Top 10 for LLMs](https://genai.owasp.org/download/43299/?tmstv=1731900559)
- **การจัดการความลับที่ปลอดภัย**: ใช้บรรจุภัณฑ์สำหรับโทเค็น คีย์ API และข้อมูลการกำหนดค่าที่ละเอียดอ่อนโดยเฉพาะ
- **การเข้ารหัสครบวงจร**: ใช้การสื่อสารที่ปลอดภัยผ่านส่วนประกอบแอปพลิเคชันและการไหลของข้อมูลทั้งหมด
- **การตรวจสอบความถูกต้องของข้อมูลเข้า**: การตรวจสอบอย่างเข้มงวดสำหรับข้อมูลที่ผู้ใช้ป้อน พารามิเตอร์ API และแหล่งข้อมูล

#### **การเสริมความปลอดภัยโครงสร้างพื้นฐาน**
- **การพิสูจน์ตัวตนหลายปัจจัย**: MFA ที่บังคับใช้สำหรับบัญชีผู้ดูแลระบบและบัญชีบริการทั้งหมด
- **การจัดการแพตช์**: การติดตั้งแพตช์อัตโนมัติและทันท่วงทีสำหรับระบบปฏิบัติการ เฟรมเวิร์ก และการพึ่งพิง  
- **การบูรณาการผู้ให้บริการตัวตน**: การจัดการตัวตนแบบศูนย์กลางผ่านผู้ให้บริการตัวตนองค์กร (Microsoft Entra ID, Active Directory)
- **การแยกเครือข่าย**: การแยกส่วนตรรกะของส่วนประกอบ MCP เพื่อลดโอกาสเคลื่อนที่ด้านข้าง
- **หลักการสิทธิ์ขั้นต่ำสุด**: ให้สิทธิ์ที่จำเป็นน้อยที่สุดสำหรับส่วนประกอบและบัญชีระบบทั้งหมด

#### **การตรวจสอบและตรวจจับความปลอดภัย**
- **การบันทึกที่ครอบคลุม**: การบันทึกรายละเอียดกิจกรรมแอปพลิเคชัน AI รวมถึงการตรวจสอบการสื่อสารระหว่างลูกค้าและเซิร์ฟเวอร์ MCP
- **การบูรณาการ SIEM**: การจัดการข้อมูลและเหตุการณ์ความปลอดภัยแบบรวมศูนย์สำหรับการตรวจจับพฤติกรรมผิดปกติ
- **การวิเคราะห์พฤติกรรม**: การตรวจสอบโดยใช้ AI เพื่อค้นหาลักษณะผิดปกติในพฤติกรรมระบบและผู้ใช้
- **ข่าวกรองภัยคุกคาม**: การบูรณาการฟีดภัยคุกคามภายนอกและตัวชี้วัดการละเมิด (IOCs)
- **การตอบสนองเหตุการณ์**: กระบวนการกำหนดไว้อย่างดีสำหรับการตรวจจับ ตอบสนอง และกู้คืนเหตุการณ์ความปลอดภัย

#### **สถาปัตยกรรมความไว้วางใจเป็นศูนย์**
- **อย่าเชื่อใจเลย ตรวจสอบตลอดเวลา**: การตรวจสอบต่อเนื่องของผู้ใช้ อุปกรณ์ และการเชื่อมต่อเครือข่าย
- **การแบ่งส่วนเครือข่ายอย่างละเอียด**: การควบคุมเครือข่ายที่แยกส่วนงานและบริการแต่ละรายการ
- **ความปลอดภัยที่เน้นตัวตน**: นโยบายความปลอดภัยที่อิงตามตัวตนที่ตรวจสอบได้แทนที่จะเป็นตำแหน่งเครือข่าย
- **การประเมินความเสี่ยงแบบต่อเนื่อง**: การประเมินท่าทางความปลอดภัยแบบไดนามิกตามบริบทและพฤติกรรมปัจจุบัน
- **การเข้าถึงแบบมีเงื่อนไข**: การควบคุมการเข้าถึงที่ปรับตามปัจจัยความเสี่ยง ตำแหน่ง และความไว้วางใจของอุปกรณ์

### รูปแบบการบูรณาการองค์กร

#### **การบูรณาการระบบนิเวศความปลอดภัยของ Microsoft**
- **Microsoft Defender for Cloud**: การจัดการท่าทางความปลอดภัยบนคลาวด์อย่างครอบคลุม
- **Azure Sentinel**: SIEM และ SOAR แบบเนทีฟบนคลาวด์สำหรับการปกป้องงาน AI
- **Microsoft Entra ID**: การจัดการตัวตนและการเข้าถึงองค์กรพร้อมนโยบายการเข้าถึงแบบมีเงื่อนไข
- **Azure Key Vault**: การจัดการความลับแบบศูนย์กลางพร้อมการสำรองข้อมูลฮาร์ดแวร์ (HSM)
- **Microsoft Purview**: การกำกับดูแลข้อมูลและการปฏิบัติตามข้อกำหนดสำหรับแหล่งข้อมูลและกระบวนงาน AI

#### **การปฏิบัติตามกฎระเบียบและการกำกับดูแล**
- **การสอดคล้องกับกฎระเบียบ**: ตรวจสอบให้แน่ใจว่าการใช้งาน MCP สอดคล้องกับข้อกำหนดการปฏิบัติตามเฉพาะอุตสาหกรรม (GDPR, HIPAA, SOC 2)

- **การจัดประเภทข้อมูล**: การจัดหมวดหมู่และจัดการข้อมูลที่ละเอียดอ่อนที่ประมวลผลโดยระบบ AI อย่างเหมาะสม  
- **เส้นทางการตรวจสอบ**: การบันทึกอย่างครบถ้วนเพื่อให้เป็นไปตามกฎระเบียบและการสอบสวนทางนิติเวช  
- **การควบคุมความเป็นส่วนตัว**: การใช้หลักการความเป็นส่วนตัวโดยออกแบบในสถาปัตยกรรมระบบ AI  
- **การจัดการการเปลี่ยนแปลง**: กระบวนการที่เป็นทางการสำหรับการตรวจสอบความปลอดภัยของการเปลี่ยนแปลงในระบบ AI  

แนวปฏิบัติพื้นฐานเหล่านี้สร้างพื้นฐานความปลอดภัยที่แข็งแกร่งซึ่งช่วยเพิ่มประสิทธิภาพของการควบคุมด้านความปลอดภัยเฉพาะของ MCP และให้การปกป้องที่ครอบคลุมสำหรับแอปพลิเคชันที่ขับเคลื่อนด้วย AI  

## ข้อสรุปที่สำคัญเกี่ยวกับความปลอดภัย  

- **แนวทางความปลอดภัยแบบหลายชั้น**: รวมแนวปฏิบัติด้านความปลอดภัยพื้นฐาน (การเขียนโค้ดที่ปลอดภัย, การเข้าถึงน้อยที่สุด, การตรวจสอบซัพพลายเชน, การตรวจสอบอย่างต่อเนื่อง) กับการควบคุมเฉพาะ AI เพื่อการปกป้องที่ครอบคลุม  

- **ภูมิทัศน์ภัยคุกคามเฉพาะ AI**: ระบบ MCP เผชิญกับความเสี่ยงเฉพาะ เช่น การฉีดคำสั่ง, การวางยาพิษเครื่องมือ, การแย่งชิงเซสชัน, ปัญหาตัวแทนสับสน, ช่องโหว่การส่งผ่านโทเค็น และการให้สิทธิ์เกินควรที่ต้องการมาตรการป้องกันเฉพาะทาง  

- **การพิสูจน์ตัวตนและการอนุญาตที่ยอดเยี่ยม**: ใช้การพิสูจน์ตัวตนที่แข็งแกร่งโดยใช้ผู้ให้บริการระบุตัวตนอิสระ (Microsoft Entra ID), บังคับใช้การตรวจสอบโทเค็นอย่างถูกต้อง และไม่ยอมรับโทเค็นที่ไม่ได้ออกให้สำหรับเซิร์ฟเวอร์ MCP ของคุณโดยเฉพาะ  

- **การป้องกันการโจมตี AI**: ใช้ Microsoft Prompt Shields และ Azure Content Safety เพื่อป้องกันการโจมตีแบบฉีดคำสั่งทางอ้อมและการวางยาพิษเครื่องมือ พร้อมทั้งตรวจสอบเมตาดาต้าของเครื่องมือและตรวจสอบการเปลี่ยนแปลงแบบไดนามิก  

- **ความปลอดภัยของเซสชันและการขนส่ง**: ใช้ IDs เซสชันที่ปลอดภัยด้วยการเข้ารหัสและไม่กำหนดได้ที่ผูกกับตัวผู้ใช้, ใช้การจัดการวงจรชีวิตเซสชันอย่างถูกต้อง และไม่ใช้เซสชันในการพิสูจน์ตัวตน  

- **แนวทางปฏิบัติที่ดีที่สุดด้านความปลอดภัย OAuth**: ป้องกันการโจมตีตัวแทนสับสนผ่านความยินยอมของผู้ใช้ที่ชัดเจนสำหรับลูกค้าลงทะเบียนแบบไดนามิก, ดำเนินการ OAuth 2.1 พร้อม PKCE อย่างถูกต้อง และตรวจสอบ URI การเปลี่ยนเส้นทางอย่างเข้มงวด  

- **หลักการความปลอดภัยโทเค็น**: หลีกเลี่ยงรูปแบบการส่งผ่านโทเค็นที่ไม่เหมาะสม, ตรวจสอบการอ้างสิทธิ์ของผู้รับโทเค็น, ใช้โทเค็นอายุสั้นพร้อมการหมุนเวียนอย่างปลอดภัย และรักษาขอบเขตความน่าเชื่อถืออย่างชัดเจน  

- **ความปลอดภัยของซัพพลายเชนอย่างครบถ้วน**: ปฏิบัติต่อส่วนประกอบของระบบนิเวศ AI ทั้งหมด (โมเดล, แฝงข้อมูล, ผู้ให้บริการบริบท, API ภายนอก) ด้วยความเข้มงวดด้านความปลอดภัยเท่ากับซอฟต์แวร์แบบดั้งเดิม  

- **วิวัฒนาการอย่างต่อเนื่อง**: ติดตามมาตรฐาน MCP ที่เปลี่ยนแปลงอย่างรวดเร็ว, มีส่วนร่วมกับมาตรฐานชุมชนความปลอดภัย และรักษาท่าทางความปลอดภัยที่ปรับตัวได้ตามพัฒนาการของโปรโตคอล  

- **การบูรณาการความปลอดภัยของ Microsoft**: ใช้ประโยชน์จากระบบนิเวศความปลอดภัยของ Microsoft อย่างครบถ้วน (Prompt Shields, Azure Content Safety, GitHub Advanced Security, Entra ID) เพื่อเพิ่มการปกป้องแอปพลิเคชัน MCP

## **เอกสารความปลอดภัยเพิ่มเติม**  

สำหรับคำแนะนำด้านความปลอดภัยอย่างละเอียด โปรดอ้างอิงเอกสารเฉพาะทางในส่วนนี้:  

- **[MCP Security Best Practices 2025](./mcp-security-best-practices-2025.md)** - แนวปฏิบัติด้านความปลอดภัยที่ครบถ้วนสำหรับการนำ MCP ไปใช้  
- **[Azure Content Safety Implementation](./azure-content-safety-implementation.md)** - ตัวอย่างการใช้งานจริงสำหรับการรวม Azure Content Safety  
- **[MCP Security Controls 2025](./mcp-security-controls-2025.md)** - การควบคุมและเทคนิคด้านความปลอดภัยล่าสุดสำหรับการติดตั้ง MCP  
- **[MCP Best Practices Quick Reference](./mcp-best-practices.md)** - คู่มืออ้างอิงด่วนสำหรับแนวปฏิบัติด้านความปลอดภัยหลักของ MCP  
- **[BlueHat 2026: Securing the future of AI: Securing MCP with defense in depth patterns](https://www.youtube.com/watch?v=cVWB58kEt-Y)** - รูปแบบการป้องกันเชิงลึกจากศูนย์ตอบสนองความปลอดภัยของ Microsoft (MSRC)
