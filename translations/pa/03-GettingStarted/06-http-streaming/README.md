# ਮਾਡਲ ਸੰਦਰਭ ਪ੍ਰੋਟੋਕੋਲ (MCP) ਨਾਲ HTTPS ਸਟ੍ਰੀਮਿੰਗ

ਇਹ ਅਧਿਆਇ Model Context Protocol (MCP) ਦੀ ਵਰਤੋਂ ਕਰਕੇ HTTPS ਨਾਲ ਸੁਰੱਖਿਅਤ, ਸਕੇਲਯੋਗ ਅਤੇ ਰੀਅਲ-ਟਾਈਮ ਸਟ੍ਰੀਮਿੰਗ ਲਾਗੂ ਕਰਨ ਲਈ ਇੱਕ ਵਿਸਤ੍ਰਿਤ ਮਾਰਗਦਰਸ਼ਨ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ। ਇਸ ਵਿੱਚ ਸਟ੍ਰੀਮਿੰਗ ਲਈ ਪ੍ਰੇਰਣਾ, ਉਪਲਬਧ ਟ੍ਰਾਂਸਪੋਰਟ ਮਕੈਨਜ਼ਮ, MCP ਵਿੱਚ ਸਟ੍ਰੀਮੇਬਲ HTTP ਲਾਗੂ ਕਰਨ ਦਾ ਢੰਗ, ਸੁਰੱਖਿਆ ਦੀਆਂ بہترین ਪ੍ਰਥਾਵਾਂ, SSE ਤੋਂ ਮਾਈਗ੍ਰੇਸ਼ਨ ਅਤੇ ਆਪਣੇ ਸਟ੍ਰੀਮਿੰਗ MCP ਐਪਲੀਕੇਸ਼ਨਾਂ ਨੂੰ ਬਣਾਉਣ ਲਈ ਪ੍ਰਯੋਗਿਕ ਦਿਸ਼ਾ-ਨਿਰਦੇਸ਼ ਸ਼ਾਮਲ ਹਨ।

> **ਅਗਲਾ ਵੇਖੋ:** ਇਹ ਪਾਠ MCP ਵਿਸ਼ੇਸ਼ਤਾ 2025-11-25 ਦੇ ਅਧੀਨ ਸਟ੍ਰੀਮੇਬਲ HTTP ਦਾ ਵੇਰਵਾ ਦਿੰਦਾ ਹੈ, ਜਿੱਥੇ ਇੱਕ ਸੈਸ਼ਨ `initialize` ਦੌਰਾਨ ਸਥਾਪਿਤ ਕੀਤਾ ਜਾਂਦਾ ਹੈ ਅਤੇ `Mcp-Session-Id` ਅਤੇਰ ਨਾਲ ਪਿੰਨ ਕੀਤਾ ਜਾਂਦਾ ਹੈ। `2026-07-28` ਰਿਲੀਜ਼ ਉਮੀਦਵਾਰ ਪੂਰੀ ਤਰ੍ਹਾਂ ਹੈਂਡਸ਼ੇਕ ਅਤੇ ਸੈਸ਼ਨ ID ਨੂੰ ਹਟਾ ਦਿੰਦਾ ਹੈ, ਜਿਸ ਨਾਲ ਹਰ ਬੇਨਤੀ ਖੁਦ-ਮੁਤਾਬਿਕ ਅਤੇ ਕਿਸੇ ਵੀ ਸਰਵਰ ਉਦਾਹਰਨ ਨੂੰ ਰਾਊਟ ਕਰਨ ਯੋਗ ਹੋ ਜਾਂਦੀ ਹੈ ਬਿਨਾਂ ਸਟਿੱਕੀ ਸੈਸ਼ਨਾਂ ਦੇ। ਵੇਰਵਾ ਲਈ [MCP ਵਿੱਚ ਕੀ ਬਦਲ ਰਿਹਾ ਹੈ: 2026-07-28 ਰਿਲੀਜ਼ ਉਮੀਦਵਾਰ](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) ਵੇਖੋ।

## MCP ਵਿੱਚ ਟ੍ਰਾਂਸਪੋਰਟ ਮਕੈਨਜ਼ਮ ਅਤੇ ਸਟ੍ਰੀਮਿੰਗ

ਇਹ ਭਾਗ MCP ਵਿੱਚ ਉਪਲਬਧ ਵੱਖ-ਵੱਖ ਟ੍ਰਾਂਸਪੋਰਟ ਮਕੈਨਜ਼ਮਾਂ ਨੂੰ ਅਤੇ ਕਲਾਇੰਟ ਅਤੇ ਸਰਵਰਾਂ ਵਿਚਕਾਰ ਰੀਅਲ ਟਾਈਮ ਸੰਚਾਰ ਲਈ ਸਟ੍ਰੀਮਿੰਗ ਸਮਰੱਥਾਵਾਂ ਦੇ ਯੋਗਦਾਨ ਨੂੰ ਪੜ੍ਹਦਾ ਹੈ।

### ਟ੍ਰਾਂਸਪੋਰਟ ਮਕੈਨਜ਼ਮ ਕੀ ਹੈ?

ਇੱਕ ਟ੍ਰਾਂਸਪੋਰਟ ਮਕੈਨਜ਼ਮ ਇਹ ਪਰਿਭਾਸ਼ਿਤ ਕਰਦਾ ਹੈ ਕਿ ਡਾਟਾ ਕਿਵੇਂ ਕਲਾਇੰਟ ਅਤੇ ਸਰਵਰ ਵਿਚਕਾਰ ਦੀਨਦ-ਬਰਦ ਕਰਨ ਵਾਲਾ ਹੈ। MCP ਵੱਖ-ਵੱਖ ਮਾਹੌਲ ਅਤੇ ਲੋੜਾਂ ਲਈ ਕਈ ਟ੍ਰਾਂਸਪੋਰਟ ਕਿਸਮਾਂ ਦਾ ਸਮਰਥਨ ਕਰਦਾ ਹੈ:

- **stdio**: ਮਿਆਰੀ ਇਨਪੁੱਟ/ਆਉਟਪੁੱਟ, ਸਥਾਨਕ ਅਤੇ CLI-ਅਧਾਰਤ ਟੂਲਾਂ ਲਈ ਉਚਿਤ। ਸਾਦਾ ਹੈ ਪਰ ਵੈੱਬ ਜਾਂ ਕਲਾਉਡ ਲਈ ਉਚਿਤ ਨਹੀਂ।
- **SSE (ਸਰਵਰ-ਭੇਜੇ ਇਵੈਂਟ)**: ਸਰਵਰਾਂ ਨੂੰ HTTP 'ਤੇ ਕਲਾਇੰਟਾਂ ਨੂੰ ਰੀਅਲ-ਟਾਈਮ ਅੱਪਡੇਟ ਭੇਜਣ ਦਿੰਦਾ ਹੈ। ਵੈੱਬ UIs ਲਈ ਠੀਕ ਹੈ, ਪਰ ਸਕੇਲਯੋਗਤਾ ਅਤੇ ਲਚੀਲਤਾ ਵਿੱਚ ਸੀਮਿਤ। MCP Specification 2025-06-18 ਅਨੁਸਾਰ, ਸਵਤੰਤਰ SSE ਟ੍ਰਾਂਸਪੋਰਟ ਨੂੰ ਬੰਦ ਕਰ ਦਿੱਤਾ ਗਿਆ ਹੈ ਅਤੇ ਇਸ ਦੀ ਜਗ੍ਹਾ "ਸਟ੍ਰੀਮੇਬਲ HTTP" ਟ੍ਰਾਂਸਪੋਰਟ ਲਈ ਗਈ ਹੈ।
- **ਸਟ੍ਰੀਮੇਬਲ HTTP**: ਆਧੁਨਿਕ HTTP ਅਧਾਰਿਤ ਸਟ੍ਰੀਮਿੰਗ ਟ੍ਰਾਂਸਪੋਰਟ, ਸੂਚਨਾਵਾਂ ਅਤੇ ਪ੍ਰਮਾਣਿਤ ਸਕੇਲਯੋਗਤਾ ਲਈ ਸਮਰਥ ਕ੍ਰਮ। ਵੱਧਤਰ ਉਤਪਾਦਨ ਅਤੇ ਕਲਾਉਡ ਪਰਿਸ਼ਰਾਂ ਲਈ ਸਿਫਾਰਸ਼ੀ ਹੈ।

### ਤੁਲਨਾਤਮਕ ਸਾਰਣੀ

ਹੇਠਾਂ ਦਿੱਤੀ ਤੁਲਨਾਤਮਕ ਸਾਰਣੀ ਦੇਖੋ ਤਾਂ ਜੋ ਇਹ ਸਮਝ ਸਕੋ ਕਿ ਇਹ ਟ੍ਰਾਂਸਪੋਰਟ ਮਕੈਨਜ਼ਮ ਕੀਵੇਂ ਵੱਖ-ਵੱਖ ਹਨ:

| ਟ੍ਰਾਂਸਪੋਰਟ           | ਰੀਅਲ-ਟਾਈਮ ਅੱਪਡੇਟ | ਸਟ੍ਰੀਮਿੰਗ | ਸਕੇਲਯੋਗਤਾ | ਵਰਤੋਂ ਮਾਮਲਾ              |
|-------------------|------------------|-----------|-------------|-------------------------|
| stdio             | ਨਹੀਂ             | ਨਹੀਂ       | ਘੱਟ         | ਸਥਾਨਕ CLI ਟੂਲ           |
| SSE               | ਹਾਂ              | ਹਾਂ        | ਦਰਮਿਆਨਾ   | ਵੈੱਬ, ਰੀਅਲ-ਟਾਈਮ ਅੱਪਡੇਟ  |
| Streamable HTTP   | ਹਾਂ              | ਹਾਂ        | ਉੱਚ         | ਕਲਾਉਡ, ਬਹੁ-ਕਲਾਇੰਟ       |

> **ਸੁਝਾਅ:** ਠੀਕ ਟ੍ਰਾਂਸਪੋਰਟ ਦੀ ਚੋਣ ਪਰਦਰਸ਼ਨ, ਸਕੇਲਯੋਗਤਾ ਅਤੇ ਉਪਭੋਗਤਾ ਅਨੁਭਵ ਨੂੰ ਪ੍ਰਭਾਵਿਤ ਕਰਦੀ ਹੈ। ਆਧੁਨਿਕ, ਸਕੇਲਯੋਗ ਅਤੇ ਕਲਾਉਡ-ਤਿਆਰ ਐਪਲੀਕੇਸ਼ਨਾਂ ਲਈ **ਸਟ੍ਰੀਮੇਬਲ HTTP** ਦੀ ਸਿਫਾਰਸ਼ ਕੀਤੀ ਜਾਂਦੀ ਹੈ।

ਪਿਛਲੇ ਅਧਿਆਇਆਂ ਵਿੱਚ ਤੁਹਾਨੂੰ ਜੋ stdio ਅਤੇ SSE ਟ੍ਰਾਂਸਪੋਰਟ ਦਿਖਾਏ ਗਏ ਸਨ ਅਤੇ ਇਸ ਅਧਿਆਇ ਵਿੱਚ ਕਵਰੇਜ ਹੋਣ ਵਾਲਾ ਸਟ੍ਰੀਮੇਬਲ HTTP ਟ੍ਰਾਂਸਪੋਰਟ ਨੂੰ ਧਿਆਨ ਵਿੱਚ ਰੱਖੋ।

## ਸਟ੍ਰੀਮਿੰਗ: ਸੰਕਲਪ ਅਤੇ ਪ੍ਰੇਰਣਾ

ਸਟ੍ਰੀਮਿੰਗ ਦੇ ਬੁਨਿਆਦੀ ਸੰਕਲਪਾਂ ਅਤੇ ਪ੍ਰੇਰਣਾ ਨੂੰ ਸਮਝਣਾ ਪ੍ਰਭਾਵਸ਼ਾਲੀ ਰੀਅਲ-ਟਾਈਮ ਕਮਿunikੇਸ਼ਨ ਸਿਸਟਮਾਂ ਨੂੰ ਲਾਗੂ ਕਰਨ ਲਈ ਜਰੂਰੀ ਹੈ।

**ਸਟ੍ਰੀਮਿੰਗ** ਇੱਕ ਨੈੱਟਵਰਕ ਪ੍ਰੋਗਰਾਮਿੰਗ ਤਕਨੀਕ ਹੈ ਜੋ ਡਾਟਾ ਨੂੰ ਛੋਟੇ, ਪ੍ਰਬੰਧਨਯੋਗ ਹਿੱਸਿਆਂ ਵਿੱਚ ਜਾਂ ਘਟਨਾਵਾਂ ਦੀ ਲੜੀ ਵਜੋਂ ਭੇਜਣ ਅਤੇ ਪ੍ਰਾਪਤ ਕਰਨ ਦੀ ਸਹੂਲਤ ਦਿੰਦੀ ਹੈ, ਬਜਾਏ ਪੂਰੇ ਜਵਾਬ ਦੀ ਉਡੀਕ ਕਰਨ ਦੇ। ਇਹ ਖਾਸ ਤੌਰ 'ਤੇ ਉਪਯੋਗੀ ਹੈ:

- ਵੱਡੇ ਫ਼ਾਇਲਾਂ ਜਾਂ ਡੇਟਾਸੇਟਾਂ ਲਈ।
- ਰੀਅਲ-ਟਾਈਮ ਅੱਪਡੇਟ ਲਈ (ਉਦਾਹਰਨ ਵਜੋਂ, ਚੈਟ, ਪ੍ਰਗਤੀ ਬਾਰ)।
- ਲੰਬੇ ਸਮੇਂ ਚੱਲਣ ਵਾਲੇ ਹਿਸਾਬ-ਕਿਤਾਬਾਂ ਲਈ ਜਿੱਥੇ ਤੁਸੀਂ ਉਪਭੋਗਤਾ ਨੂੰ ਜਾਣਕਾਰੀ ਦਿੰਦਿਆਂ ਰਹਿਣਾ ਚਾਹੁੰਦੇ ਹੋ।

ਸਟ੍ਰੀਮਿੰਗ ਬਾਰੇ ਕੁਝ ਉੱਚ-ਸਤਰ ਦੇ ਬਿੰਦੂ ਇਨ੍ਹਾਂ ਹਨ:

- ਡਾਟਾ ਹੌਲੀ-ਹੌਲੀ ਭੇਜਿਆ ਜਾਂਦਾ ਹੈ, ਸਾਰਾ ਇੱਕ ਵਾਰ ਨਹੀਂ।
- ਕਲਾਇੰਟ ਜਿਵੇਂ-ਜਿਵੇਂ ਡਾਟਾ ਪ੍ਰਾਪਤ ਹੁੰਦਾ ਹੈ ਪ੍ਰਕਿਰਿਆ ਕਰ ਸਕਦਾ ਹੈ।
- ਪ੍ਰਤੀਤ ਲੈਟੈਂਸੀ ਘਟਾਉਂਦਾ ਹੈ ਅਤੇ ਉਪਭੋਗਤਾ ਅਨੁਭਵ ਨੂੰ ਬਿਹਤਰ ਬਣਾਉਂਦਾ ਹੈ।

### ਸਟ੍ਰੀਮਿੰਗ ਦੀ ਲੋੜ ਕਿਉਂ ਹੈ?

ਸਟ੍ਰੀਮਿੰਗ ਵਰਤਣ ਦੇ ਕਾਰਨ ਇਹ ਹਨ:

- ਉਪਯੋਗਤਾ ਨੂੰ ਤੁਰੰਤ ਪ੍ਰਤੀਕਿਰਿਆ ਮਿਲਦੀ ਹੈ, ਕੇਵਲ ਅੰਤ 'ਤੇ ਨਹੀਂ।
- ਰੀਅਲ-ਟਾਈਮ ਐਪਲੀਕੇਸ਼ਨਾਂ ਅਤੇ ਪ੍ਰਤੀਕਿਰਿਆਸ਼ੀਲ UI ਨੂੰ ਯੋਗ ਬਣਾਉਂਦਾ ਹੈ।
- ਨੈੱਟਵਰਕ ਅਤੇ ਕੰਪਿਊਟਿੰਗ ਸਰੋਤਾਂ ਦੀ ਬਿਹਤਰ ਵਰਤੋਂ।

### ਸਧਾਰਣ ਉਦਾਹਰਨ: HTTP ਸਟ੍ਰੀਮਿੰਗ ਸਰਵਰ ਅਤੇ ਕਲਾਇੰਟ

ਇਹ ਇੱਕ ਸਧਾਰਣ ਉਦਾਹਰਨ ਹੈ ਕਿ ਸਟ੍ਰੀਮਿੰਗ ਕਿਵੇਂ ਲਾਗੂ ਕੀਤੀ ਜਾ ਸਕਦੀ ਹੈ:

#### ਪਾਈਥਨ

**ਸਰਵਰ (Python, FastAPI ਅਤੇ StreamingResponse ਦੀ ਵਰਤੋਂ ਨਾਲ):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**ਕਲਾਇੰਟ (Python, requests ਦੀ ਵਰਤੋਂ ਨਾਲ):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

ਇਹ ਉਦਾਹਰਨ ਦਿਖਾਉਂਦੀ ਹੈ ਕਿ ਸਰਵਰ ਕਿਵੇਂ ਉਪਲਬਧ ਹੋਣ ਵਾਲੇ ਸੁਨੇਹਿਆਂ ਦੀ ਲੜੀ ਕਲਾਇੰਟ ਨੂੰ ਭੇਜਦਾ ਹੈ, ਨਾ ਕਿ ਸਾਰੇ ਸੁਨੇਹਿਆਂ ਦੀ ਉਡੀਕ ਕਰਦਾ ਹੈ।

**ਇਹ ਕਿਵੇਂ ਕਾਰਜ ਕਰਦਾ ਹੈ:**

- ਸਰਵਰ ਹਰ ਸੁਨੇਹਾ ਜਿਵੇਂ-ਜਿਵੇਂ ਤਿਆਰ ਹੁੰਦਾ ਹੈ ਐਨੂੰ ਨਿਕਾਲਦਾ ਹੈ।
- ਕਲਾਇੰਟ ਹਰ ਹਿੱਸਾ ਪ੍ਰਾਪਤ ਕਰਦਾ ਅਤੇ ਪ੍ਰਿੰਟ ਕਰਦਾ ਹੈ ਜਿਵੇਂ ਉਹ ਆਉਂਦਾ ਹੈ।

**ਲੋੜੀਂਦੇ ਗੱਲਾਂ:**

- ਸਰਵਰ ਨੂੰ ਸਟ੍ਰੀਮਿੰਗ ਜਵਾਬ ਵਰਤਣਾ ਚਾਹੀਦਾ ਹੈ (ਜਿਵੇਂ FastAPI ਵਿੱਚ `StreamingResponse`)।
- ਕਲਾਇੰਟ ਨੂੰ ਜਵਾਬ ਨੂੰ ਸਟ੍ਰੀਮ ਵਜੋਂ ਪ੍ਰਕਿਰਿਆ ਕਰਨਾ ਚਾਹੀਦਾ ਹੈ (`stream=True` requests ਵਿੱਚ)।
- ਸਮੱਗਰੀ ਕਿਸਮ ਆਮ ਤੌਰ 'ਤੇ `text/event-stream` ਜਾਂ `application/octet-stream` ਹੁੰਦੀ ਹੈ।

#### ਜਾਵਾ

**ਸਰਵਰ (Java, Spring Boot ਅਤੇ Server-Sent Events ਦੀ ਵਰਤੋਂ ਨਾਲ):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**ਕਲਾਇੰਟ (Java, Spring WebFlux WebClient ਦੀ ਵਰਤੋਂ ਨਾਲ):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**Java ਲਾਗੂ ਕਰਨ ਦੇ ਨੋਟਸ:**

- Spring Boot ਦੇ ਪ੍ਰਤੀਕਿਰਿਆਸ਼ੀਲ ਢਾਂਚੇ ਅਤੇ `Flux` ਨਾਲ ਸਟ੍ਰੀਮਿੰਗ ਦੀ ਵਰਤੋਂ ਕਰਦਾ ਹੈ
- `ServerSentEvent` ਘਟਨਾਵਾਂ ਦੇ ਟਾਈਪਾਂ ਨਾਲ ਢਾਂਚਾਕ੍ਰਿਤ ਸਟ੍ਰੀਮਿੰਗ ਮੁਹੱਈਆ ਕਰਦਾ ਹੈ
- `WebClient` ਨੂੰ `bodyToFlux()` ਨਾਲ ਪ੍ਰਤੀਕਿਰਿਆਸ਼ੀਲ ਸਟ੍ਰੀਮਿੰਗ ਪ੍ਰਾਪਤੀ ਯੋਗ ਬਣਾਉਂਦਾ ਹੈ
- `delayElements()` ਢਿੱਚਿਆਂ ਵਿਚਕਾਰ ਪ੍ਰਕਿਰਿਆ ਸਮਾਂ ਮਾਡਲ ਕਰਦਾ ਹੈ
- ਘਟਨਾਵਾਂ ਵਿਚ ਟਾਈਪ ਹੋ ਸਕਦੇ ਹਨ (`info`, `result`) ਕਲਾਇੰਟ ਸੰਭਾਲ ਲਈ ਬਿਹਤਰ

### ਤੁਲਨਾ: ਪਰੰਪਰਾ ਸਟ੍ਰੀਮਿੰਗ ਵర్సਸ MCP ਸਟ੍ਰੀਮਿੰਗ

ਇਹ ਦਿਖਾਉਂਦਾ ਹੈ ਕਿ "ਪੁਰਾਣੇ" ਢੰਗ ਨਾਲ ਅਤੇ MCP ਵਿੱਚ ਸਟ੍ਰੀਮਿੰਗ ਕੰਮ ਕਰਨ ਵਿੱਚ ਕੀ ਅੰਤਰ ਹਨ:

| ਵਿਸ਼ੇਸ਼ਤਾ          | ਪਰੰਪਰਾਗਤ HTTP ਸਟ੍ਰੀਮਿੰਗ  | MCP ਸਟ੍ਰੀਮਿੰਗ (ਸੂਚਨਾਵਾਂ)      |
|---------------------|----------------------------|---------------------------------|
| ਮੁੱਖ ਜਵਾਬ          | ਟੁਕੜਿਆਂ ਵਿੱਚ              | ਇੱਕੋ ਜਵਾਬ, ਅੰਤ 'ਤੇ            |
| ਪ੍ਰਗਤੀ ਅੱਪਡੇਟ         | ਡਾਟਾ ਟੁਕੜੇ ਵਜੋਂ ਭੇਜੇ ਜਾਂਦੇ ਹਨ | ਸੂਚਨਾਵਾਂ ਵਜੋਂ ਭੇਜੇ ਜਾਂਦੇ ਹਨ   |
| ਕਲਾਇੰਟ ਦੀ ਲੋੜਾਂ       | ਸਟ੍ਰੀਮ ਪ੍ਰਕਿਰਿਆ ਕਰਨੀ ਪੈਂਦੀ ਹੈ | ਸੁਨੇਹਾ ਹਨਡਲਰ ਲਾਗੂ ਕਰਨਾ ਪੈਂਦਾ ਹੈ|
| ਵਰਤੋਂ ਮਾਮਲਾ          | ਵੱਡੇ ਫ਼ਾਇਲ, AI ਟੋਕਨ ਸਟ੍ਰੀਮ | ਪ੍ਰਗਤੀ, ਲੌਗ, ਰੀਅਲ-ਟਾਈਮ ਫੀਡਬੈਕ|

### ਮੁੱਖ ਭਿੰਨਤਾ

ਤੇਹਾਂ ਸਾਰੀਆਂ ਮੁੱਖ ਭਿੰਨਤਾਵਾਂ ਹਨ:

- **ਸੰਚਾਰ ਪੈਟਰਨ:**
  - ਪਰੰਪਰਾਗਤ HTTP ਸਟ੍ਰੀਮਿੰਗ: ਸਾਦਾ Chunked ਟ੍ਰਾਂਸਫਰ ਇन्कੋਡਿੰਗ ਨਾਲ ਡਾਟਾ ਟੁਕੜਿਆਂ ਵਿੱਚ ਭੇਜਦਾ ਹੈ
  - MCP ਸਟ੍ਰੀਮਿੰਗ: JSON-RPC ਪ੍ਰੋਟੋਕੋਲ ਵਾਲੀ ਢਾਂਚਾਕ੍ਰਿਤ ਸੂਚਨਾ ਪ੍ਰਣਾਲੀ ਵਰਤਦਾ ਹੈ

- **ਸੁਨੇਹੇ ਦਾ ਫਾਰਮੈਟ:**
  - ਪਰੰਪਰਾਗਤ HTTP: ਸਧਾਰਣ ਟੈਕਸਟ ਟੁਕੜੇ ਨਾਲ ਨਿਊਲਾਈਨ
  - MCP: ਢਾਂਚਾਕ੍ਰਿਤ LoggingMessageNotification ਆਬਜੇਕਟ ਮੈਟਾਡੇਟਾ ਸਮੇਤ

- **ਕਲਾਇੰਟ ਲਾਗੂ ਕਰਨ:**
  - ਪਰੰਪਰਾਗਤ HTTP: ਸਧਾਰਣ ਕਲਾਇੰਟ ਜੋ ਸਟ੍ਰੀਮਿੰਗ ਜਵਾਬਾਂ ਨੂੰ ਪ੍ਰਕਿਰਿਆ ਕਰਦਾ ਹੈ
  - MCP: ਵਧੀਆ ਅਦੱਦਾ ਵਾਲਾ ਕਲਾਇੰਟ ਜਿਸ ਵਿੱਚ ਵੱਖ-ਵੱਖ ਕਿਸਮਾਂ ਦੇ ਸੁਨੇਹਿਆਂ ਨੂੰ ਸੰਭਾਲਣ ਵਾਲਾ ਸੁਨੇਹਾ ਹਨਡਲਰ ਹੁੰਦਾ ਹੈ

- **ਪ੍ਰਗਤੀ ਅੱਪਡੇਟ:**
  - ਪਰੰਪਰਾਗਤ HTTP: ਮੁੱਖ ਜਵਾਬ ਸਟ੍ਰੀਮ ਦਾ ਹਿੱਸਾ ਹੁੰਦਾ ਹੈ
  - MCP: ਮੁੱਖ ਜਵਾਬ ਨੂੰ ਅੰਤ 'ਤੇ ਭੇਜਿਆ ਜਾਂਦਾ ਹੈ, ਪ੍ਰਗਤੀ ਵੱਖ-ਵੱਖ ਸੂਚਨਾ ਸੁਨੇਹਿਆਂ ਰਾਹੀਂ ਜਾਂਦੀ ਹੈ

### ਸਿਫਾਰਸ਼ਾਂ

ਕੁਝ ਗੱਲਾਂ ਜਿਹੜੀਆਂ ਅਸੀਂ ਸਿਫਾਰਸ਼ ਕਰਦੇ ਹਾਂ ਜਦੋਂ ਤੁਸੀਂ ਪਰੰਪਰਾਗਤ ਸਟ੍ਰੀਮਿੰਗ `/stream` ਰਾਹੀਂ ਲਾਗੂ ਕਰਨ ਤੇ MCP ਸਟ੍ਰੀਮਿੰਗ ਚੋਣ ਕਰਨ ਵਿਚਕਾਰ ਵਿਚਾਰ ਕਰ ਰਹੇ ਹੋ:

- **ਸਧਾਰਣ ਸਟ੍ਰੀਮਿੰਗ ਲਈ:** ਪਰੰਪਰਾਗਤ HTTP ਸਟ੍ਰੀਮਿੰਗ ਲਾਗੂ ਕਰਨ ਲਈ ਸਧਾਰਣ ਹੈ ਅਤੇ ਬੁਨਿਆਦੀ ਲੋੜਾਂ ਲਈ ਕਾਫੀ ਹੈ।

- **ਕਠਿਨ, ਇੰਟਰਐਕਟਿਵ ਐਪਲੀਕੇਸ਼ਨਾਂ ਲਈ:** MCP ਸਟ੍ਰੀਮਿੰਗ ਇੱਕ ਵਧੀਆ ਢਾਂਚਾਕ੍ਰਿਤ ਤਰੀਕਾ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ ਜਿਸ ਵਿੱਚ ਧਨਾਤਮਕ ਮੈਟਾਡੇਟਾ ਅਤੇ ਸੂਚਨਾਵਾਂ ਅਤੇ ਅੰਤਿਮ ਨਤੀਜੇ ਵਿਚਕਾਰ ਵੰਡ ਹੋਂਦੀ ਹੈ।

- **AI ਐਪਲੀਕੇਸ਼ਨਾਂ ਲਈ:** MCP ਦੀ ਸੂਚਨਾ ਪ੍ਰਣਾਲੀ ਲੰਬੇ ਸਮੇਂ ਚੱਲਣ ਵਾਲੇ AI ਕਾਰਜਾਂ ਲਈ ਖਾਸ ਤੌਰ 'ਤੇ ਉਪਯੋਗੀ ਹੈ ਜਿੱਥੇ ਤੁਸੀਂ ਉਪਭੋਗਤਿਆਂ ਨੂੰ ਪ੍ਰਗਤੀ ਦੀ ਜਾਣਕਾਰੀ ਦੇਣੀ ਚਾਹੁੰਦੇ ਹੋ।

## MCP ਵਿੱਚ ਸਟ੍ਰੀਮਿੰਗ

ਠੀਕ ਹੈ, ਤਾਂ ਤੁਹਾਨੂੰ ਹੁਣ ਤੱਕ ਕੁਝ ਸਿਫਾਰਸ਼ਾਂ ਅਤੇ ਤੁਲਨੇਕ ਦਿੱਸ ਚੁੱਕੇ ਹੋ MCP ਅਤੇ ਪਰੰਪਰਾਗਤ ਸਟ੍ਰੀਮਿੰਗ ਵਿੱਚ ਫਰਕਾਂ 'ਤੇ। ਆਓ ਵਿਸਤਾਰ ਵਿੱਚ ਦੇਖੀਏ ਕਿ ਤੁਸੀਂ MCP ਵਿੱਚ ਸਟ੍ਰੀਮਿੰਗ ਕਿਵੇਂ ਵਰਤ ਸਕਦੇ ਹੋ।

MCP ਫਰੇਮਵਰਕ ਅੰਦਰ ਸਟ੍ਰੀਮਿੰਗ ਕਿਵੇਂ ਕੰਮ ਕਰਦੀ ਹੈ ਇਹ ਸਮਝਣਾ ਜਰੂਰੀ ਹੈ ਤਾਂ ਜੋ ਤੁਸੀਂ ਆਪਣੇ ਐਪਲੀਕੇਸ਼ਨਾਂ ਨੂੰ ਪ੍ਰਤੀਕਿਰਿਆਸ਼ੀਲ ਬਣਾ ਸਕੋ ਜੋ ਲੰਬੇ ਸਮੇਂ ਚੱਲਣ ਵਾਲੇ ਕਾਰਜ ਦੌਰਾਨ ਉਪਭੋਗਤਿਆਂ ਨੂੰ ਤੁਰੰਤ ਫੀਡਬੈਕ ਦਿੰਦੇ ਹਨ।

MCP ਵਿੱਚ ਸਟ੍ਰੀਮਿੰਗ ਦੀ ਮਾਹਰਤਾ ਇਹ ਨਹੀਂ ਹੈ ਕਿ ਮੁੱਖ ਜਵਾਬ ਟੁਕੜਿਆਂ ਵਿੱਚ ਭੇਜਿਆ ਜਾਵੇ, ਸਗੋਂ ਇਹ ਹੈ ਕਿ ਜਦੋਂ ਕੋਈ ਟੂਲ ਬੇਨਤੀ ਪ੍ਰਕਿਰਿਆ ਕਰ ਰਿਹਾ ਹੋਵੇ ਤਾਂ ਉਪਭੋਗਤਾ ਨੂੰ **ਸੂਚਨਾਵਾਂ** ਭੇਜੀਆਂ ਜਾਣ। ਇਹ ਸੂਚਨਾਵਾਂ ਪ੍ਰਗਤੀ ਅੱਪਡੇਟਸ, ਲੌਗ ਜਾਂ ਹੋਰ ਘਟਨਾਵਾਂ ਸ਼ਾਮਲ ਕਰ ਸਕਦੀਆਂ ਹਨ।

### ਇਹ ਕਿਵੇਂ ਕਾਰਜ ਕਰਦਾ ਹੈ

ਮੁੱਖ ਨਤੀਜਾ ਫਿਰ ਵੀ ਇੱਕਲ ਜਵਾਬ ਵਜੋਂ ਭੇਜਿਆ ਜਾਂਦਾ ਹੈ। ਹਾਲਾਂਕਿ, ਪ੍ਰਕਿਰਿਆ ਦੌਰਾਨ ਵੱਖਰੇ ਸੁਨੇਹਿਆ ਰੂਪ ਵਿੱਚ ਸੂਚਨਾਵਾਂ ਭੇਜੀਆਂ ਜਾਂਦੀਆਂ ਹਨ ਜਿਨ੍ਹਾਂ ਨਾਲ ਕਲਾਇੰਟ ਅਸਲ ਸਮੇਂ ਵਿੱਚ ਅੱਪਡੇਟ ਰਹਿੰਦਾ ਹੈ। ਕਲਾਇੰਟ ਇਹ ਸੂਚਨਾਵਾਂ ਪ੍ਰਕਿਰਿਆ ਅਤੇ ਦਿਖਾਉਣ ਯੋਗ ਹੋਣਾ ਚਾਹੀਦਾ ਹੈ।

## ਸੂਚਨਾ ਕੀ ਹੈ?

ਅਸੀਂ "ਸੂਚਨਾ" ਬਾਰੇ ਗੱਲ ਕੀਤੀ, MCP ਦੇ ਸੰਦਰਭ ਵਿੱਚ ਇਸ ਦਾ ਕੀ ਮਤਲਬ ਹੈ?

ਇੱਕ ਸੂਚਨਾ ਉਸ ਮੈਸੇਜ ਨੂੰ ਕਿਹਾ ਜਾਂਦਾ ਹੈ ਜੋ ਸਰਵਰ ਤੋਂ ਕਲਾਇੰਟ ਨੂੰ ਭੇਜਿਆ ਜਾਂਦਾ ਹੈ ਜਿਸ ਵਿੱਚ ਵਰਤੀ ਜਾ ਰਹੀ ਲੰਬੀ ਕਾਰਵਾਈ ਦੌਰਾਨ ਪ੍ਰਗਤੀ, ਸਥਿਤੀ ਜਾਂ ਹੋਰ ਘਟਨਾਵਾਂ ਬਾਰੇ ਜਾਣਕਾਰੀ ਹੁੰਦੀ ਹੈ। ਇਹ ਸੂਚਨਾਵਾਂ ਪਾਰਦਰਸ਼ੀਤਾ ਅਤੇ ਉਪਭੋਗਤਾ ਅਨੁਭਵ ਨੂੰ ਬਿਹਤਰ ਕਰਦੀਆਂ ਹਨ।

ਉਦਾਹਰਨ ਵਜੋਂ, ਇੱਕ ਕਲਾਇੰਟ ਨੂੰ ਸੇਵਾ ਸਰਵਰ ਨਾਲ ਪ੍ਰਾਰੰਭਿਕ ਹੈਂਡਸ਼ੇਕ ਹੋਣ ਤੋਂ ਬਾਅਦ ਇੱਕ ਸੂਚਨਾ ਭੇਜਣੀ ਚਾਹੀਦੀ ਹੈ।

ਸੂਚਨਾ ਇਸ ਤਰ੍ਹਾਂ ਦੇ JSON ਸੁਨੇਹੇ ਦੇ ਰੂਪ ਵਿੱਚ ਦਿੱਖਦੀ ਹੈ:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

ਸੂਚਨਾਵਾਂ MCP ਵਿੱਚ ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) ਟਾਪਿਕ ਨਾਲ ਸੰਬੰਧਿਤ ਹੁੰਦੀਆਂ ਹਨ।

ਲੌਗਿੰਗ ਨੂੰ ਚਲਾਉਣ ਲਈ, ਸਰਵਰ ਨੂੰ ਇਹ ਫੀਚਰ/ਸਮਰੱਥਾ ਸੌਂਪਣੀ ਪੈਂਦੀ ਹੈ, ਇਸ ਤਰ੍ਹਾਂ:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> ਵਰਤੇ ਗਏ SDK ਤੇ ਨਿਰਭਰ ਕਰਦਿਆਂ, ਲੌਗਿੰਗ ਆਮ ਤੌਰ 'ਤੇ ਸੁਚਾਲੂ ਹੋ ਸਕਦੀ ਹੈ ਜਾਂ ਤੁਹਾਨੂੰ ਆਪਣੇ ਸਰਵਰ কੰਫਿਗਰੇਸ਼ਨ ਵਿੱਚ ਇਸਨੂੰ ਖਾਸ ਤੌਰ 'ਤੇ ਚਾਲੂ ਕਰਨਾ ਪੈ ਸਕਦਾ ਹੈ।

ਵੱਖ-ਵੱਖ ਕਿਸਮਾਂ ਦੀਆਂ ਸੂਚਨਾਵਾਂ ਹੁੰਦੀਆਂ ਹਨ:

| ਪੱਧਰ      | ਵਰਣਨ                         | ਉਦਾਹਰਨ ਵਰਤੋਂ ਦਾ ਮਾਮਲਾ      |
|-----------|-------------------------------|------------------------------|
| ਡੀਬੱਗ     | ਵਿਸਥਾਰਪੂਰਕ ਡੀਬੱਗਿੰਗ ਜਾਣਕਾਰੀ  | ਫੰਕਸ਼ਨ ਦਾਖਲ/ਬਾਹਰ ਨਿਕਲਣਾ     |
| ਜਾਣਕਾਰੀ  | ਆਮ ਜਾਣਕਾਰੀ ਸੁਨੇਹੇ             | ਕਾਰਜ ਪ੍ਰਗਤੀ ਅੱਪਡੇਟਸ          |
| ਸੂਚਨਾ     | ਆਮ ਪਰ ਮਹੱਤਵਪੂਰਨ ਕਾਰਜ        | ਸੰਰਚਨਾ ਤਬਦੀਲੀਆਂ            |
| ਚੇਤਾਵਨੀ  | ਚੇਤਾਵਨੀ ਸਥਿਤੀਆਂ             | ਪੁਰਾਣਾ ਹੋ ਚੁੱਕਾ ਫੀਚਰ          |
| ਗਲਤੀ      | ਗਲਤੀ ਸਥਿਤੀਆਂ                | ਕਾਰਜ ਦੇ ਅਸਫਲ ਹੋਣ             |
| ਆਲੋਚਨਾਤਮਕ| ਗੰਭੀਰ ਸਥਿਤੀਆਂ               | ਸਿਸਟਮ ਕੰਪੋਨੈਂਟ ਨਾਕਾਮੀ         |
| ਚੇਤਾਵਨੀ   | ਤੁਰੰਤ ਕਾਰਵਾਈ ਲੋੜੀਂਦੀ         | ਡੇਟਾ ਕਰਪਸ਼ਨ ਦਾ ਪਤਾ ਚੱਲਣਾ     |
| ਐਮਰਜੈਂਸੀ | ਸਿਸਟਮ ਅਣਇਸਤਮਾਲਯੋਗ            | ਸਿਸਟਮ ਦੀ ਪੂਰੀ ਤਰ੍ਹਾਂ ਫੇਲ     |

## MCP ਵਿੱਚ ਸੂਚਨਾਵਾਂ ਲਾਗੂ ਕਰਨਾ

MCP ਵਿੱਚ ਸੂਚਨਾਵਾਂ ਲਾਗੂ ਕਰਨ ਲਈ, ਤੁਹਾਨੂੰ ਦੋਹਾਂ ਸਰਵਰ ਅਤੇ ਕਲਾਇੰਟ ਪਾਸਿਆਂ ਨੂੰ ਪਹਿਲਾਂ ਹੀ ਸੈੱਟ ਕਰਨਾ ਪੈਦਾ ਹੈ ਤਾਂ ਜੋ ਵਾਸਤਵਿਕ-ਸਮੇਂ ਅੱਪਡੇਟਸ ਨੂੰ ਹੈਂਡਲ ਕੀਤਾ ਜਾ ਸਕੇ। ਇਹ ਤੁਹਾਡੇ ਐਪ ਨੂੰ ਲੰਬੇ ਸਮੇਂ ਚੱਲ ਰਹੀਆਂ ਕਾਰਵਾਈਆਂ ਦੌਰਾਨ ਉਪਭੋਗਤਿਆਂ ਨੂੰ ਤੁਰੰਤ ਪ੍ਰਤੀਕਿਰਿਆ ਦੇਣ ਯੋਗ ਬਣਾਉਂਦਾ ਹੈ।

### ਸਰਵਰ-ਸਾਇਡ: ਸੂਚਨਾਵਾਂ ਭੇਜਣਾ

ਆਓ ਪਹਿਲਾਂ ਸਰਵਰ ਪਾਸੇ ਬਾਰੇ ਅੱਗੇ ਵਧੀਏ। MCP ਵਿੱਚ, ਤੁਸੀਂ ਟੂਲ ਪਰਿਭਾਸ਼ਿਤ ਕਰਦੇ ਹੋ ਜੋ ਬੇਨਤੀਆਂ ਨੂੰ ਪ੍ਰਕਿਰਿਆ ਕਰਦਿਆਂ ਸੂਚਨਾਵਾਂ ਭੇਜ ਸਕਦੇ ਹਨ। ਸਰਵਰ ਸਮਰੱਥਾ ਆਬਜੇਕਟ (ਆਮ ਤੌਰ 'ਤੇ `ctx`) ਦੀ ਵਰਤੋਂ ਕਰਕੇ ਕਲਾਇੰਟ ਨੂੰ ਸੁਨੇਹੇ ਭੇਜਦਾ ਹੈ।

#### ਪਾਈਥਨ

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

ਪਿਛਲੇ ਉਦਾਹਰਨ ਵਿੱਚ, `process_files` ਟੂਲ ਹਰ ਫ਼ਾਇਲ ਪ੍ਰਕਿਰਿਆ ਕਰਦੇ ਸਮੇਂ ਕਲਾਇੰਟ ਨੂੰ ਤਿੰਨ ਸੂਚਨਾਵਾਂ ਭੇਜਦਾ ਹੈ। `ctx.info()` ਵਿਧੀ ਜਾਣਕਾਰੀ ਸੁਨੇਹੇ ਭੇਜਣ ਲਈ ਵਰਤੀ ਜਾਂਦੀ ਹੈ।

ਇਸ ਤੋਂ ਅਲਾਵਾ, ਸੂਚਨਾਵਾਂ ਨੂੰ ਯਕੀਨੀ ਬਣਾਉਣ ਲਈ ਇਹ ਦੇਖੋ ਕਿ ਤੁਹਾਡਾ ਸਰਵਰ ਸਟ੍ਰੀਮਿੰਗ ਟ੍ਰਾਂਸਪੋਰਟ (ਜਿਵੇਂ `streamable-http`) ਵਰਤ ਰਿਹਾ ਹੈ ਅਤੇ ਤੁਹਾਡਾ ਕਲਾਇੰਟ ਸੂਚਨਾਵਾਂ ਨੂੰ ਪ੍ਰਕਿਰਿਆ ਕਰਨ ਵਾਲਾ ਸੁਨੇਹਾ ਹੈਂਡਲਰ ਲਾਗੂ ਕਰਦਾ ਹੈ। ਇੱਥੇ ਦਿੱਖ ਰਿਹਾ ਹੈ ਕਿ ਤੁਸੀਂ ਸਰਵਰ ਵਿੱਚ `streamable-http` ਟ੍ਰਾਂਸਪੋਰਟ ਕਿਵੇਂ ਵਰਤ ਸਕਦੇ ਹੋ:

```python
mcp.run(transport="streamable-http")
```

### ਕਲਾਇੰਟ-ਸਾਇਡ: ਸੂਚਨਾਵਾਂ ਪ੍ਰਾਪਤ ਕਰਣਾ

ਕਲਾਇੰਟ ਨੂੰ ਇੱਕ ਸੁਨੇਹਾ ਹੈਂਡਲਰ ਲਾਗੂ ਕਰਨਾ ਚਾਹੀਦਾ ਹੈ ਜੋ ਆਉਂਦੀਆਂ ਸੂਚਨਾਵਾਂ ਨੂੰ ਪ੍ਰਕਿਰਿਆ ਅਤੇ ਪ੍ਰਦਰਸ਼ਿਤ ਕਰੇ।

#### ਪਾਈਥਨ

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

ਪਿਛਲੇ ਕੋਡ ਵਿੱਚ, `message_handler` ਫੰਕਸ਼ਨ ਜਾਂਚਦਾ ਹੈ ਕਿ ਆਉਂਦਾ ਸੁਨੇਹਾ ਸੂਚਨਾ ਹੈ ਜਾਂ ਨਹੀਂ। ਜੇ ਹੈ, ਤਾਂ ਇਹ ਸੂਚਨਾ ਪ੍ਰਿੰਟ ਕਰਦਾ ਹੈ; ਨਹੀਂ ਤਾਂ ਇਸ ਨੂੰ ਸਧਾਰਣ ਸਰਵਰ ਸੁਨੇਹੇ ਵਾਂਗ ਪ੍ਰਕਿਰਿਆ ਕਰਦਾ ਹੈ। ਇਸਦੇ ਨਾਲ ਇਹ ਵੀ ਧਿਆਨ ਦਿਓ ਕਿ `ClientSession` ਨੂੰ `message_handler` ਨਾਲ ਆਰੰਭ ਕੀਤਾ ਗਿਆ ਹੈ ਤਾਂ ਜੋ ਆਉਂਦੀਆਂ ਸੂਚਨਾਵਾਂ ਹੈਂਡਲ ਹੋ ਸਕਣ।

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");

    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

ਇਸ .NET ਉਦਾਹਰਨ ਵਿੱਚ, `MessageHandler` ਫੰਕਸ਼ਨ ਜਾਂਚਦਾ ਹੈ ਕਿ ਆਉਂਦਾ ਸੁਨੇਹਾ ਸੂਚਨਾ ਹੈ ਜਾਂ ਨਹੀਂ। ਜੇ ਹੈ, ਤਾਂ ਇਹ ਸੂਚਨਾ ਪ੍ਰਿੰਟ ਕਰਦਾ ਹੈ; ਨਹੀਂ ਤਾਂ ਇਸ ਨੂੰ ਸਧਾਰਣ ਸਰਵਰ ਸੁਨੇਹੇ ਵਾਂਗ ਪ੍ਰਕਿਰਿਆ ਕਰਦਾ ਹੈ। `ClientSession` ਨੂੰ ਸੁਨੇਹਾ ਹੈਂਡਲਰ ਦੇ ਨਾਲ `ClientSessionOptions` ਰਾਹੀਂ ਆਰੰਭ ਕੀਤਾ ਗਿਆ ਹੈ।

ਸੂਚਨਾਵਾਂ ਚਲਾਉਣ ਲਈ, ਯਕੀਨੀ ਬਣਾਓ ਕਿ ਤੁਹਾਡੇ ਸਰਵਰ ਵਿੱਚ ਸਟ੍ਰੀਮਿੰਗ ਟ੍ਰਾਂਸਪੋਰਟ (ਜਿਵੇਂ `streamable-http`) ਵਰਤਿਆ ਜਾ ਰਿਹਾ ਹੈ ਅਤੇ ਤੁਹਾਡਾ ਕਲਾਇੰਟ ਸੂਚਨਾਵਾਂ ਪ੍ਰਕਿਰਿਆ ਕਰਨ ਵਾਲਾ ਸੁਨੇਹਾ ਹੈਂਡਲਰ ਲਾਗੂ ਕਰਦਾ ਹੈ।

## ਪ੍ਰਗਤੀ ਸੂਚਨਾਵਾਂ ਅਤੇ ਸੰਦਰਭ

ਇਹ ਭਾਗ MCP ਵਿੱਚ ਪ੍ਰਗਤੀ ਸੂਚਨਾਵਾਂ ਦੇ ਸੰਕਲਪ ਅਤੇ ਮਤਲਬ ਨੂੰ ਵਿਆਖਿਆ ਕਰਦਾ ਹੈ, ਉਹ ਕਿਉਂ ਮਹੱਤਵਪੂਰਣ ਹਨ, ਅਤੇ ਸਟ੍ਰੀਮੇਬਲ HTTP ਦੀ ਵਰਤੋਂ ਨਾਲ ਉਨ੍ਹਾਂ ਨੂੰ ਕਿਵੇਂ ਲਾਗੂ ਕੀਤਾ ਜਾ ਸਕਦਾ ਹੈ। ਤੁਹਾਨੂੰ ਆਪਣੀ ਸਮਝ ਨੂੰ ਮਜ਼ਬੂਤ ਕਰਨ ਲਈ ਇੱਕ ਪ੍ਰਯੋਗਿਕ ਅਸਾਈਨਮੈਂਟ ਵੀ ਮਿਲੇਗਾ।

ਪ੍ਰਗਤੀ ਸੂਚਨਾਵਾਂ ਉਨ੍ਹਾਂ ਸੁਨੇਹਿਆਂ ਹਨ ਜੋ сервер ਤੋਂ ਕਲਾਇੰਟ ਨੂੰ ਲੰਬੇ ਸਮੇਂ ਚੱਲ ਰਹੀਆਂ ਕਾਰਜਾਂ ਦੌਰਾਨ ਭੇਜੇ ਜਾਂਦੇ ਹਨ। ਪੂਰੇ ਪ੍ਰਕਿਰਿਆ ਦੇ ਖ਼ਤਮ ਹੋਣ ਦੀ ਉਡੀਕ ਕਰਨ ਦੀ ਬਜਾਏ, ਸਰਵਰ ਕਲਾਇੰਟ ਨੂੰ ਮੌਜੂਦਾ ਸਥਿਤੀ ਬਾਰੇ ਜਾਣੂ ਕਰਵਾਉਂਦਾ ਹੈ। ਇਹ ਪਾਰਦਰਸ਼ੀਤਾ ਨੂੰ ਬਿਹਤਰ ਕਰਦਾ ਹੈ, ਉਪਭੋਗਤਾ ਅਨੁਭਵ ਸੁਧਾਰਦਾ ਹੈ, ਅਤੇ ਡਿਬੱਗਿੰਗ ਨੂੰ ਆਸਾਨ ਬਣਾਉਂਦਾ ਹੈ।

**ਉਦਾਹਰਨ:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### ਪ੍ਰਗਤੀ ਸੂਚਨਾਵਾਂ ਦੀ ਲੋੜ ਕਿਉਂ?

ਪ੍ਰਗਤੀ ਸੂਚਨਾਵਾਂ ਕਈ ਕਾਰਨਾਂ ਕਰਕੇ ਜ਼ਰੂਰੀ ਹਨ:

- **ਵਧੀਆ ਉਪਭੋਗਤਾ ਅਨੁਭਵ:** ਉਪਭੋਗਤਾ ਕੰਮ ਦੇ ਚੱਲਣ ਸਮੇਂ ਅੱਪਡੇਟ ਵੇਖਦੇ ਹਨ, ਕੇਵਲ ਅੰਤ 'ਤੇ ਨਹੀਂ।
- **ਰੀਅਲ-ਟਾਈਮ ਪ੍ਰਤੀਕਿਰਿਆ:** ਕਲਾਇੰਟ ਪ੍ਰਗਤੀ ਬਾਰ ਜਾਂ ਲੌਗਾਂ ਦਿਖਾ ਸਕਦੇ ਹਨ, ਜਿਸ ਨਾਲ ਐਪ ਜ਼ਿਆਦਾ ਜਵਾਬਦੇਹ ਮਹਿਸੂਸ ਹੁੰਦੀ ਹੈ।
- **ਡਿਬੱਗਿੰਗ ਅਤੇ ਨਿਗਰਾਨੀ ਆਸਾਨ:** ਵਿਕਾਸਕਾਰ ਅਤੇ ਉਪਭੋਗਤਾ ਵੇਖ ਸਕਦੇ ਹਨ ਕਿ ਕਾਰਜ ਕਿੱਥੇ ਸੁਸਤ ਜਾਂ ਰੁਕਿਆ ਹੋਇਆ ਹੈ।

### ਪ੍ਰਗਤੀ ਸੂਚਨਾਵਾਂ ਨੂੰ ਕਿਵੇਂ ਲਾਗੂ ਕਰੀਏ

ਇੱਥੇ ਵੇਖੋ ਕਿ MCP ਵਿੱਚ ਪ੍ਰਗਤੀ ਸੂਚਨਾਵਾਂ ਕਿਵੇਂ ਲਾਗੂ ਕੀਤੀਆਂ ਜਾ ਸਕਦੀਆਂ ਹਨ:

- **ਸਰਵਰ 'ਤੇ:** ਜਦੋਂ ਹਰ ਆਈਟਮ ਪ੍ਰਕਿਰਿਆ ਕਰਦਾ ਹੈ ਤਾਂ `ctx.info()` ਜਾਂ `ctx.log()` ਨਾਲ ਸੂਚਨਾਵਾਂ ਭੇਜੋ। ਇਹ ਮੁੱਖ ਨਤੀਜੇ ਦੇ ਤਿਆਰ ਹੋਣ ਤੋਂ ਪਹਿਲਾਂ ਕਲਾਇੰਟ ਨੂੰ ਸੁਨੇਹੇ ਭੇਜਦਾ ਹੈ।
- **ਕਲਾਇੰਟ 'ਤੇ:** ਇੱਕ ਸੁਨੇਹਾ ਹੈਂਡਲਰ ਲਾਗੂ ਕਰੋ ਜੋ ਆ ਰਹੀਆਂ ਸੂਚਨਾਵਾਂ ਨੂੰ ਸੁਣਦਾ ਅਤੇ ਦਿਖਾਉਂਦਾ ਹੈ। ਇਹ ਹੈਂਡਲਰ ਸੂਚਨਾਵਾਂ ਅਤੇ ਅੰਤਿਮ ਨਤੀਜੇ ਵਿਚ ਸੰਭਾਲ ਕਰਦਾ ਹੈ।

**ਸਰਵਰ ਉਦਾਹਰਨ:**

#### ਪਾਈਥਨ

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

## ਸੁਰੱਖਿਆ ਨਾਲ ਸੰਬੰਧਿਤ ਗੱਲਾਂ

ਜਦੋਂ MCP ਸਰਵਰ HTTP-ਅਧਾਰਿਤ ਟ੍ਰਾਂਸਪੋਰਟਾਂ ਨਾਲ ਲਾਗੂ ਕੀਤੇ ਜਾਂਦੇ ਹਨ, ਤਦ ਸੁਰੱਖਿਆ ਇੱਕ ਮੁੱਖ ਚਿੰਤਾ ਬਣ ਜਾਂਦੀ ਹੈ ਜਿਸ ਵਿੱਚ ਕਈ ਹਮਲੇ ਦੇ ਰਾਹਾਂ ਅਤੇ ਸੁਰੱਖਿਆ ਪ੍ਰਣਾਲੀ ਦਾ ਧਿਆਨ ਰੱਖਣਾ ਜਰੂਰੀ ਹੈ।

### ਸਾਰ

HTTP ਰਾਹੀਂ MCP ਸਰਵਰ ਨੂੰ ਖੋਲ੍ਹਦੇ ਸਮੇਂ ਸੁਰੱਖਿਆ ਮਹੱਤਵਪੂਰਨ ਹੁੰਦੀ ਹੈ। ਸਟ੍ਰੀਮੇਬਲ HTTP ਨਵੇਂ ਹਮਲਿਆਂ ਦੇ ਮੌਕੇ ਲਿਆਉਂਦਾ ਹੈ ਅਤੇ ਇਸ ਲਈ ਸੰਭਾਲ ਨਾਲ ਸੰਰਚਨਾ ਦੀ ਲੋੜ ਹੁੰਦੀ ਹੈ।

### ਮੁੱਖ ਬਿੰਦੂ


- **ਓਰੀਜਿਨ ਹੈਡਰ ਵੈਲੀਡੇਸ਼ਨ**: DNS ਰੀਬਾਈਂਡਿੰਗ ਹਮਲਿਆਂ ਨੂੰ ਰੋਕਣ ਲਈ ਹਮੇਸ਼ਾਂ `Origin` ਹੈਡਰ ਦੀ ਜਾਂਚ ਕਰੋ।
- **ਲੋਕਲਹੋਸਟ ਬਾਈਂਡਿੰਗ**: ਸਥਾਨਕ ਵਿਕਾਸ ਲਈ ਸਰਵਰਾਂ ਨੂੰ `localhost` ਨਾਲ ਬਾਈਂਡ ਕਰੋ ਤਾਂ ਜੋ ਉਹ ਪਬਲਿਕ ਇੰਟਰਨੈਟ ਲਈ ਖੁੱਲ੍ਹੇ ਨਾ ਹੋਣ।
- **ਪ੍ਰਮਾਣਿਕਤਾ**: ਪ੍ਰੋਡਕਸ਼ਨ ਤਾਇਨਾਤੀ ਲਈ ਪ੍ਰਮਾਣਿਕਤਾ (ਜਾਂ API ਕੀਜ਼, OAuth) ਲਾਗੂ ਕਰੋ।
- **CORS**: ਪਹੁੰਚ ਨੂੰ ਸੀਮਿਤ ਕਰਨ ਲਈ Cross-Origin Resource Sharing (CORS) ਨੀਤੀਆਂ ਸੰਰਚਿਤ ਕਰੋ।
- **HTTPS**: ਟ੍ਰੈਫਿਕ ਨੂੰ ਏਨਕ੍ਰਿਪਟ ਕਰਨ ਲਈ ਪ੍ਰੋਡਕਸ਼ਨ ਵਿੱਚ HTTPS ਦੀ ਵਰਤੋਂ ਕਰੋ।

### ਚੰਗੀਆਂ ਪ੍ਰਥਾਵਾਂ

- ਆਉਣ ਵਾਲੀਆਂ ਬੇਨਤੀਆਂ ਬਿਨਾਂ ਵੈਲੀਡੇਸ਼ਨ ਦੇ ਕਦੇ ਭਰੋਸਾ ਨਾ ਕਰੋ।
- ਸਾਰੇ ਐਕਸੈਸ ਅਤੇ ਗਲਤੀਆਂ ਦੀ ਲਾਗਿੰਗ ਅਤੇ ਨਿਗਰਾਨੀ ਕਰੋ।
- ਸੁਰੱਖਿਆ ਖਾਮੀਆਂ ਨੂੰ ਮਿਟਾਉਣ ਲਈ ਨਿਯਮਤ ਤੌਰ 'ਤੇ ਡਿਪੈਂਡੈਂਸੀਜ਼ ਨੂੰ ਅੱਪਡੇਟ ਕਰੋ।

### ਚੁਣੌਤੀਆਂ

- ਸੁਰੱਖਿਆ ਅਤੇ ਵਿਕਾਸ ਦੀ ਸੁਵਿਧਾ ਦਾ ਸੰਤੁਲਨ ਬਣਾਇਆ ਰੱਖਣਾ
- ਵੱਖ-ਵੱਖ ਕਲਾਇੰਟ ਵਾਤਾਵਰਣਾਂ ਨਾਲ ਅਨੁਕੂਲਤਾ ਯਕੀਨੀ ਬਣਾਉਣਾ

## SSE ਤੋਂ Streamable HTTP ਵਿੱਚ ਅੱਪਗਰੇਡ ਕਰਨਾ

ਜਿਹੜੇ ਐਪਲੀਕੇਸ਼ਨਾਂ ਵਿਚ ਸਾਰਵਰ-ਸੈਂਟ ਇਵੈਂਟਸ (SSE) ਦੀ ਵਰਤੋਂ ਹੋ ਰਹੀ ਹੈ, Streamable HTTP ਵਿੱਚ ਮਾਈਗ੍ਰੇਸ਼ਨ ਤੁਹਾਡੇ MCP ਇਮਪਲੀਮੈਂਟੇਸ਼ਨਾਂ ਲਈ ਵਧੀਆ ਸਮਰੱਥਾ ਅਤੇ ਲੰਬੇ ਸਮੇਂ ਦੀ ਸਥਿਰਤਾ ਪੇਸ਼ ਕਰਦਾ ਹੈ।

### ਅੱਪਗਰੇਡ ਕਰਨ ਦਾ ਕਾਰਨ

SSE ਤੋਂ Streamable HTTP ਵਿੱਚ ਅੱਪਗਰੇਡ ਕਰਨ ਦੇ ਦੋ ਮੋਹਕ ਕਾਰਣ ਹਨ:

- Streamable HTTP SSE ਨਾਲੋਂ ਬਿਹਤਰ ਸਕੇਲਬਿਲਟੀ, ਅਨੁਕੂਲਤਾ ਅਤੇ ਵਿਸਤ੍ਰਿਤ ਸੂਚਨਾ ਸਹਾਇਤਾ ਦਿੰਦਾ ਹੈ।
- ਇਹ ਨਵੇਂ MCP ਐਪਸ ਲਈ ਸਿਫਾਰਸ਼ੀ ਟ੍ਰਾਂਸਪੋਰਟ ਹੈ।

### ਮਾਈਗ੍ਰੇਸ਼ਨ ਦੇ ਕਦਮ

ਤੁਹਾਡੇ MCP ਐਪਲੀਕੇਸ਼ਨਾਂ ਵਿੱਚ SSE ਤੋਂ Streamable HTTP ਵਿੱਚ ਮਾਈਗ੍ਰੇਟ ਕਰਨ ਦਾ ਤਰੀਕਾ ਇੱਥੇ ਹੈ:

- **ਸਰਵਰ ਕੋਡ ਅੱਪਡੇਟ ਕਰੋ** `mcp.run()` ਵਿੱਚ `transport="streamable-http"` ਦੀ ਵਰਤੋਂ ਕਰਨ ਲਈ।
- **ਕਲਾਇੰਟ ਕੋਡ ਅੱਪਡੇਟ ਕਰੋ** SSE ਕਲਾਇੰਟ ਦੀ ਥਾਂ `streamablehttp_client` ਦੀ ਵਰਤੋਂ ਕਰੋ।
- **ਕਲਾਇੰਟ ਵਿੱਚ ਇਕ ਸੂਚਨਾ ਪ੍ਰੋਸੈਸ ਕਰਨ ਵਾਲਾ ਮੈਸਜ ਹੈਂਡਲਰ ਲਾਗੂ ਕਰੋ**।
- ਮੌਜੂਦਾ ਸੰਦਾਂ ਅਤੇ ਵਰਕਫਲੋਜ਼ ਨਾਲ **ਅਨੁਕੂਲਤਾ ਲਈ ਟੈਸਟ ਕਰੋ**।

### ਅਨੁਕੂਲਤਾ ਬਰਕਰਾਰ ਰੱਖਣਾ

ਮਾਈਗ੍ਰੇਸ਼ਨ ਪ੍ਰਕਿਰਿਆ ਦੌਰਾਨ ਮੌਜੂਦਾ SSE ਕਲਾਇੰਟਾਂ ਨਾਲ ਅਨੁਕੂਲਤਾ ਬਰਕਰਾਰ ਰੱਖਣ ਦੀ ਸਿਫਾਰਸ਼ ਕੀਤੀ ਜਾਂਦੀ ਹੈ। ਕੁਝ ਤਰੀਕੇ ਇੱਥੇ ਹਨ:

- ਤੁਸੀਂ ਦੋਵਾਂ ਟ੍ਰਾਂਸਪੋਰਟ ਨੂੰ ਵੱਖ-ਵੱਖ ਐਂਡਪੋਇੰਟਸ 'ਤੇ ਚਲਾ ਕੇ SSE ਅਤੇ Streamable HTTP ਦੋਵਾਂ ਦਾ ਸਮਰਥਨ ਕਰ ਸਕਦੇ ਹੋ।
- ਕਲਾਇੰਟਾਂ ਨੂੰ ਹੌਲੇ-ਹੌਲੇ ਨਵੇਂ ਟ੍ਰਾਂਸਪੋਰਟ 'ਤੇ ਮਾਈਗ੍ਰੇਟ ਕਰੋ।

### ਚੁਣੌਤੀਆਂ

ਮਾਈਗ੍ਰੇਸ਼ਨ ਦੌਰਾਨ ਹੇਠਾਂ ਦਿੱਤੀਆਂ ਚੁਣੌਤੀਆਂ ਨੂੰ ਧਿਆਨ ਵਿੱਚ ਰੱਖੋ:

- ਸਾਰੇ ਕਲਾਇੰਟਾਂ ਦਾ ਅਪਡੇਟ ਹੋਣਾ ਯਕੀਨੀ ਬਣਾਉਣਾ
- ਸੂਚਨਾ ਡਿਲਿਵਰੀ ਵਿੱਚ ਫਰਕਾਂ ਨੂੰ ਸੰਭਾਲਣਾ

## ਸੁਰੱਖਿਆ ਨਾਲ ਸੰਬੰਧਿਤ ਵਿਚਾਰ

ਕਿਸੇ ਵੀ ਸਰਵਰ ਨੂੰ ਲਾਗੂ ਕਰਨ ਸਮੇਂ ਸੁਰੱਖਿਆ ਸਭ ਤੋਂ ਮੁੱਖ ਤਰਜੀਹ ਹੋਣੀ ਚਾਹੀਦੀ ਹੈ, ਖਾਸ ਕਰਕੇ HTTP-ਆਧਾਰਿਤ ਟ੍ਰਾਂਸਪੋਰਟਾਂ ਜਿਵੇਂ Streamable HTTP ਦੀ ਵਰਤੋਂ ਕਰਦੇ ਸਮੇਂ MCP ਵਿੱਚ।

HTTP-ਆਧਾਰਿਤ ਟ੍ਰਾਂਸਪੋਰਟਾਂ ਨਾਲ MCP ਸਰਵਰਾਂ ਨੂੰ ਲਾਗੂ ਕਰਦੇ ਹੋਏ ਸੁਰੱਖਿਆ ਇੱਕ ਮਹੱਤਵਪੂਰਨ ਚਿੰਤਾ ਬਣ ਜਾਂਦੀ ਹੈ ਜਿਸ ਵਿੱਚ ਕਈ ਹਮਲਿਆਂ ਅਤੇ ਸੁਰੱਖਿਆ ਮਕੈਨਿਜ਼ਮਾਂ ਦਾ ਧਿਆਨ ਰੱਖਣਾ ਲਾਜ਼ਮੀ ਹੈ।

### ਝਲਕ

MCP ਸਰਵਰਾਂ ਨੂੰ HTTP 'ਤੇ ਖੋਲ੍ਹਦੇ ਸਮੇਂ ਸੁਰੱਖਿਆ ਬਹੁਤ ਜਰੂਰੀ ਹੈ। Streamable HTTP ਕੁਝ ਨਵੇਂ ਹਮਲੇ ਦੇ ਖਤਰਿਆਂ ਨਾਲ ਆਉਂਦਾ ਹੈ ਅਤੇ ਇਸ ਲਈ ਸੰਭਾਲ ਕਾਂਫਿਗਰੇਸ਼ਨ ਦੀ ਲੋੜ ਹੁੰਦੀ ਹੈ।

ਕੁਝ ਮੁੱਖ ਸੁਰੱਖਿਆ ਵਿਚਾਰ ਇੱਥੇ ਦਿੱਤੇ ਗਏ ਹਨ:

- **ਓਰੀਜਿਨ ਹੈਡਰ ਵੈਲੀਡੇਸ਼ਨ**: DNS ਰੀਬਾਈਂਡਿੰਗ ਹਮਲਿਆਂ ਨੂੰ ਰੋਕਣ ਲਈ ਹਮੇਸ਼ਾਂ `Origin` ਹੈਡਰ ਦੀ ਜਾਂਚ ਕਰੋ।
- **ਲੋਕਲਹੋਸਟ ਬਾਈਂਡਿੰਗ**: ਸਥਾਨਕ ਵਿਕਾਸ ਲਈ ਸਰਵਰਾਂ ਨੂੰ `localhost` ਨਾਲ ਬਾਈਂਡ ਕਰੋ ਤਾਂ ਜੋ ਉਹ ਪਬਲਿਕ ਇੰਟਰਨੈਟ ਲਈ ਖੁੱਲ੍ਹੇ ਨਾ ਹੋਣ।
- **ਪ੍ਰਮਾਣਿਕਤਾ**: ਪ੍ਰੋਡਕਸ਼ਨ ਤਾਇਨਾਤੀ ਲਈ ਪ੍ਰਮਾਣਿਕਤਾ (ਜਾਂ API ਕੀਜ਼, OAuth) ਲਾਗੂ ਕਰੋ।
- **CORS**: ਪਹੁੰਚ ਨੂੰ ਸੀਮਿਤ ਕਰਨ ਲਈ Cross-Origin Resource Sharing (CORS) ਨੀਤੀਆਂ ਸੰਰਚਿਤ ਕਰੋ।
- **HTTPS**: ਟ੍ਰੈਫਿਕ ਨੂੰ ਏਨਕ੍ਰਿਪਟ ਕਰਨ ਲਈ ਪ੍ਰੋਡਕਸ਼ਨ ਵਿੱਚ HTTPS ਦੀ ਵਰਤੋਂ ਕਰੋ।

### ਚੰਗੀਆਂ ਪ੍ਰਥਾਵਾਂ

ਇੱਥੇ ਕੁਝ ਚੰਗੀਆਂ ਪ੍ਰਥਾਵਾਂ ਹਨ ਜੋ MCP ਸਟਰੀਮਿੰਗ ਸਰਵਰ ਵਿੱਚ ਸੁਰੱਖਿਆ ਲਾਗੂ ਕਰਦੇ ਸਮੇਂ ਮੰਨਣੀਆਂ ਚਾਹੀਦੀਆਂ ਹਨ:

- ਆਉਣ ਵਾਲੀਆਂ ਬੇਨਤੀਆਂ ਬਿਨਾਂ ਵੈਲੀਡੇਸ਼ਨ ਦੇ ਕਦੇ ਭਰੋਸਾ ਨਾ ਕਰੋ।
- ਸਾਰੇ ਐਕਸੈਸ ਅਤੇ ਗਲਤੀਆਂ ਦੀ ਲਾਗਿੰਗ ਅਤੇ ਨਿਗਰਾਨੀ ਕਰੋ।
- ਸੁਰੱਖਿਆ ਖਾਮੀਆਂ ਨੂੰ ਮਿਟਾਉਣ ਲਈ ਨਿਯਮਤ ਤੌਰ 'ਤੇ ਡਿਪੈਂਡੈਂਸੀਜ਼ ਨੂੰ ਅੱਪਡੇਟ ਕਰੋ।

### ਚੁਣੌਤੀਆਂ

MCP ਸਟਰੀਮਿੰਗ ਸਰਵਰਾਂ ਵਿੱਚ ਸੁਰੱਖਿਆ ਲਾਗੂ ਕਰਨ ਵੇਲੇ ਤੁਸੀਂ ਕੁਝ ਚੁਣੌਤੀਆਂ ਦਾ ਸਾਹਮਣਾ ਕਰੋਗے:

- ਸੁਰੱਖਿਆ ਅਤੇ ਵਿਕਾਸ ਦੀ ਸੁਵਿਧਾ ਦਾ ਸੰਤੁਲਨ ਬਣਾਇਆ ਰੱਖਣਾ
- ਵੱਖ-ਵੱਖ ਕਲਾਇੰਟ ਵਾਤਾਵਰਣਾਂ ਨਾਲ ਅਨੁਕੂਲਤਾ ਯਕੀਨੀ ਬਣਾਉਣਾ

### ਅਸਾਇਨਮੈਂਟ: ਆਪਣਾ ਮੀਲਮਿਲਾਪੀ MCP ਐਪ ਬਣਾਓ

**ਦ੍ਰਿਸ਼ਟੀਕੋਣ:**
ਐਕ MCP ਸਰਵਰ ਅਤੇ ਕਲਾਇੰਟ ਬਣਾਓ ਜਿੱਥੇ ਸਰਵਰ ਆਈਟਮਾਂ ਦੀ ਸੂਚੀ (ਜਿਵੇਂ ਫਾਈਲਾਂ ਜਾਂ ਦਸਤਾਵੇਜ਼) ਨੂੰ ਪ੍ਰੋਸੈਸ ਕਰਦਾ ਹੈ ਅਤੇ ਹਰ ਪ੍ਰੋਸੈਸ ਕੀਤੇ ਆਈਟਮ ਲਈ ਸੂਚਨਾ ਭੇਜਦਾ ਹੈ। ਕਲਾਇੰਟ ਨੂੰ ਹਰ ਸੂਚਨਾ ਨੂੰ ਭਾਗ ਸਕਦੇ ਸਮੇਂ ਵੇਖਣਾ ਚਾਹੀਦਾ ਹੈ।

**ਕਦਮ:**

1. ਇਕ ਸਰਵਰ ਟੂਲ ਲਾਗੂ ਕਰੋ ਜੋ ਸੂਚੀ ਨੂੰ ਪ੍ਰੋਸੈਸ ਕਰਦਾ ਹੋਵੇ ਅਤੇ ਹਰ ਆਈਟਮ ਲਈ ਸੂਚਨਾਵਾਂ ਭੇਜੇ।
2. ਇਕ ਕਲਾਇੰਟ ਲਾਗੂ ਕਰੋ ਜਿਸ ਵਿੱਚ ਇੱਕ ਮੈਸਜ ਹੈਂਡਲਰ ਹੋਵੇ ਜੋ ਜੀਵੰਤ ਸਮੇਂ ਸੂਚਨਾਵਾਂ ਦਿਖਾਏ।
3. ਆਪਣੇ ਇਮਪਲੀਮੈਂਟੇਸ਼ਨ ਨੂੰ ਚਲਾ ਕੇ ਅਤੇ ਦੋਹਾਂ ਸਰਵਰ ਅਤੇ ਕਲਾਇੰਟ ਨੂੰ ਟੈਸਟ ਕਰੋ ਅਤੇ ਸੂਚਨਾਵਾਂ ਦਾ ਅਧਿਐਨ ਕਰੋ।
