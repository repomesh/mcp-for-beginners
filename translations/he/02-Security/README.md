# אבטחת MCP: הגנה מקיפה למערכות בינה מלאכותית

[![הנחיות אבטחה של MCP](../../../translated_images/he/03.175aed6dedae133f.webp)](https://youtu.be/88No8pw706o)

_(לחצו על התמונה למעלה לצפייה בסרטון של השיעור)_

אבטחה היא יסודית לתכנון מערכות בינה מלאכותית, ולכן אנו מקדישים לה את החלק השני שלנו. זאת בהתאמה לעקרון **Secure by Design** של מיקרוסופט מתוך [יוזמת Secure Future](https://www.microsoft.com/security/blog/2025/04/17/microsofts-secure-by-design-journey-one-year-of-success/).

פרוטוקול הקשר לדגם (MCP) מביא יכולות חדשות וחזקות ליישומים המונעים בינה מלאכותית, תוך שהצגת אתגרים אבטחתיים ייחודיים החורגים מסיכוני תוכנה מסורתיים. מערכות MCP מתמודדות עם חששות אבטחה מבוססים (קידוד מאובטח, זכות מינוייה מינימלית, אבטחת שרשרת אספקה) וכן עם איומים חדשים ייחודיים לבינה מלאכותית הכוללים הזרקת פקודות, הרעלת כלים, השתלטות על מושבים, התקפות confused deputy, פרצות העברת אסימונים, ושינוי דינמי של יכולות.

שיעור זה חוקר את סיכוני האבטחה הקריטיים ביותר ביישומי MCP—כולל אימות, הרשאות, הרשאות מופרזות, הזרקת פקודות עקיפה, אבטחת מושבים, בעיות confused deputy, ניהול אסימונים, ופרצות שרשרת אספקה. תלמדו אמצעי בקרה מעשיים והנחיות מיטביות להפחתת סיכונים אלה תוך שימוש בפתרונות מיקרוסופט כגון Prompt Shields, Azure Content Safety ו-GitHub Advanced Security לחיזוק פריסת MCP שלכם.

## יעדי הלמידה

בסיום שיעור זה תוכלו:

- **זיהוי איומים ייחודיים ל-MCP**: להכיר סיכוני אבטחה מיוחדים במערכות MCP כולל הזרקת פקודות, הרעלת כלים, הרשאות מופרזות, השתלטות על מושבים, בעיות confused deputy, פרצות העברת אסימונים וסיכוני שרשרת אספקה
- **הטמעת אמצעי אבטחה**: ליישם פעולות הפחתה יעילות כולל אימות חזק, גישת זכות מינוייה מינימלית, ניהול אסימונים מאובטח, בקרה על אבטחת מושבים ואימות שרשרת אספקה
- **ניצול פתרונות אבטחה של מיקרוסופט**: להבין ולפרוס את Microsoft Prompt Shields, Azure Content Safety, ו-GitHub Advanced Security להגנת עומסי MCP
- **אימות אבטחת כלים**: לזהות את חשיבות אימות מטא-דאטה של כלים, ניטור שינויים דינמיים והגנה מפני התקפות הזרקת פקודה עקיפה
- **שילוב מיטב הנוהגים**: לשלב עקרונות אבטחה מבוססי יסוד (קידוד מאובטח, חיזוק שרתים, אפס אמון) עם אמצעי MCP ייחודיים להגנה מקיפה

# ארכיטקטורת ואמצעי אבטחה של MCP

יישומי MCP מודרניים דורשים גישות אבטחה רב-שכבתיות המטפלות הן באבטחת תוכנה מסורתית והן באיומים ייחודיים לבינה מלאכותית. מפרט MCP המתפתח במהירות ממשיך לשפר את אמצעי האבטחה שלו ומאפשר אינטגרציה טובה יותר עם ארכיטקטורות אבטחה ארגוניות ונוהגים מבוססים.

מחקר מ[דוח ההגנה הדיגיטלית של מיקרוסופט](https://aka.ms/mddr) מראה ש-**98% מהפריצות המדווחות נמנעות על ידי היגיינת אבטחה חזקה**. אסטרטגיית ההגנה היעילה משלבת פרקטיקות אבטחה יסודיות עם אמצעי MCP ייחודיים—אמצעי אבטחה בסיסיים ונחקרים נותרו בעלי ההשפעה הגדולה ביותר בהפחתת סיכוני אבטחה כוללים.

## נוף אבטחה נוכחי

> **הערה:** מידע זה משקף את תקני האבטחה של MCP מתאריך **5 בפברואר 2026**, בהתאמה למפרט **MCP Specification 2025-11-25**. פרוטוקול MCP ממשיך להתפתח במהירות ועלולות להתווסף דפוסי אימות חדשים ואמצעי בקרה משופרים בהמשך. תמיד יש להתייעץ עם [מפרט MCP הנוכחי](https://spec.modelcontextprotocol.io/), [מאגר GitHub של MCP](https://github.com/modelcontextprotocol), ו[מסמכי מיטב הנוהגים לאבטחה](https://modelcontextprotocol.io/specification/2025-11-25/basic/security_best_practices) לקבלת ההנחיות המעודכנות.

> **מבט קדימה:** מועמד השחרור של `2026-07-28` מחזק עוד יותר את ההרשאה — לקוחות חייבים לאמת את הפרמטר `iss` בתגובות הרשאה (RFC 9207), להצהיר על `application_type` ב- OpenID Connect בעת רישום דינמי של לקוחות, ולחבר אישורים רשומים לשרת ההרשאה המנפיק. ראו [שינויים במפרט MCP: מועמד שחרור 2026-07-28](../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) לרשימה מלאה של SEPs.

### התפתחות אימות MCP

מפרט MCP התפתח מהותית בגישתו לאימות והרשאות:

- **גישה מקורית**: מפרטים מוקדמים דרשו לפתח שרתי אימות מותאמים אישית, כאשר שרתי MCP שימשו כשרתי הרשאה OAuth 2.0 שמנהלים אימות משתמש ישיר
- **התקן נוכחי (2025-11-25)**: פרט חדש מאפשר לשרתי MCP להאציל אימות לספקי זהות חיצוניים (כגון Microsoft Entra ID), משפר את עמידות האבטחה ומפשט את היישום
- **אבטחת שכבת תחבורה**: תמיכה משופרת במנגנוני העברה מאובטחים עם דפוסי אימות נאותים zarówno בחיבורים מקומיים (STDIO) וגם מרוחקים (HTTP זרימתי)

## אבטחת אימות והרשאה

### אתגרי אבטחה נוכחיים

יישומים מודרניים של MCP מתמודדים עם אתגרים מרובים בתחום האימות וההרשאות:

### סיכונים ונתיבי התקפה

- **לוגיקת הרשאה מוטעית**: יישום שגוי של הרשאות בשרתי MCP עלול לחשוף מידע רגיש וליישם בקרות גישה שגויות
- **פגיעה באסימוני OAuth**: גניבת אסימוני שרת MCP מקומית מאפשרת לתוקפים להציג עצמם כשרתים ולגשת לשירותים מצד לקוחות
- **פרצות בהעברת אסימונים**: ניהול שגוי של אסימונים יוצר עקיפות של בקרות אבטחה ופערי אחריות
- **הרשאות מופרזות**: שרתי MCP עם הרשאות יתר מפרים את עקרון הזכות המינימלית ומגדילים את שטח ההתקפה

#### העברת אסימונים: תבנית אנטי קריטית

**העברת אסימונים אסורה במפורש** במפרט ההרשאה הנוכחי של MCP עקב השלכות אבטחה חמורות:

##### עקיפת בקרה אבטחתית  
- שרתי MCP ו-API בצד קליינט מטמיעים בקרות אבטחה קריטיות (הגבלת קצב, אימות בקשות, ניטור תעבורה) התלויות באימות אסימונים תקין  
- שימוש ישיר של לקוח באסימוני API עוקף הגנות חיוניות אלה ומערער את ארכיטקטורת האבטחה

##### אתגרי אחריות ובקרה  
- שרתי MCP אינם יכולים להבחין בין לקוחות המשתמשים באסימונים מנפיקי מקור, מה שמפר לפסי בקרת רישום
- יומני שרת המשאבים מראים מקורות בקשות מטעות במקום שרת MCP אמצעי
- חקירת אירועים ובקרות ציות נהיים קשים משמעותית

##### סיכוני חשיפת מידע
- טענות אסימונים לא מאומתות מאפשרות לתוקפים עם אסימונים גנובים להשתמש בשרתי MCP כפרוקסי להוצאת נתונים
- הפרות גבולות אמון מאפשרות דפוסי גישה בלתי מורשים העוברים על בקרות אבטחה

##### וקטורי התקפה מולטית-שירותיים
- אסימונים פגומים המתקבלים על ידי כמה שירותים מאפשרים תנועה צידית במערכות מחוברות
- הנחות אמון בין שירותים מופרות כאשר מקורות אסימונים אינם ניתנים לאימות

### אמצעי בקרה ואמצעי הפחתה

**דרישות אבטחה קריטיות:**

> **חובה**: שרתי MCP **אסור להם** לקבל אסימונים שלא הונפקו במפורש עבור שרת MCP

#### בקרה על אימות והרשאה

- **סקירת הרשאה קפדנית**: עריכת ביקורות מקיפות על לוגיקת הרשאות שרת MCP להבטיח שרק משתמשים ולקוחות מיועדים יקבלו גישה למשאבים רגישים
  - **מדריך ליישום**: [ניהול Azure API כאפיק אימות לשרתי MCP](https://techcommunity.microsoft.com/blog/integrationsonazureblog/azure-api-management-your-auth-gateway-for-mcp-servers/4402690)
  - **אינטגרציית זהות**: [שימוש ב-Microsoft Entra ID לאימות שרת MCP](https://den.dev/blog/mcp-server-auth-entra-id-session/)

- **ניהול אסימונים מאובטח**: יישום [הנחיות ולאימות מחזור חיים של אסימונים של מיקרוסופט](https://learn.microsoft.com/en-us/entra/identity-platform/access-tokens)
  - אימות טענות קהל האסימון התואמות לזהות שרת MCP
  - הפעלת מדיניות סיבוב ופג תוקף תקינה לאסימונים
  - מניעת התקפות חידוש אסימונים ושימוש בלתי מורשה

- **אחסון אסימונים מוגן**: אחסון אסימונים מוצפן במנוחה ובהעברה
  - **מיטב נוהגים**: [אחסון ואבטחת אסימונים](https://youtu.be/uRdX37EcCwg?si=6fSChs1G4glwXRy2)

#### יישום בקרה על גישה

- **עקרון זכות המינוייה מינימלית**: להעניק לשרתי MCP רק את ההרשאות המינימליות הנדרשות לתפקוד המיועד
  - סקירות ועדכונים תקופתיים למניעת הרחבת הרשאות
  - **תיעוד מיקרוסופט**: [גישה מאובטחת בזכות מינוייה מינימלית](https://learn.microsoft.com/entra/identity-platform/secure-least-privileged-access)

- **בקרת גישה מבוססת תפקידים (RBAC)**: ליישם הקצאות תפקיד מדויקות
  - מגבלת תחום תפקידים למשאבים ופעולות ספציפיות
  - הימנעות מהרשאות נרחבות או מיותרות המרחיבות את שטח ההתקפה

- **ניטור מתמשך של הרשאות**: לבצע ביקורות וניטור גישת מתמשכים
  - מעקב אחר דפוסי שימוש חריגים בהרשאות
  - תיקון מיידי של הרשאות מופרזות או בלתי מנוצלות

## איומי אבטחה ייחודיים לבינה מלאכותית

### הזרקת פקודות והתקפות מניפולציית כלים

יישומי MCP מודרניים מתמודדים עם וקטורי התקפה מתוחכמים ייחודיים ל-AI, שהגנות מסורתיות אינן מכסות באופן מלא:

#### **הזרקת פקודות עקיפה (Cross-Domain Prompt Injection)**

**הזרקת פקודות עקיפה** היא אחת מהפגיעויות הקריטיות ביותר במערכות AI הפועלות עם MCP. תוקפים משבצים הוראות זדוניות בתוך תוכן חיצוני—מסמכים, דפי אינטרנט, אימיילים או מקורות נתונים—שהמערכות מעבדות כפקודות לגיטימיות.

**תסריטי התקפה:**
- **הזרקה מבוססת מסמכים**: הוראות זדוניות מוסתרות במסמכים המעובדים הגורמות לפעולות לא מכוונות של הבינה
- **ניצול תוכן אינטרנט**: דפי אינטרנט פרוצים שמכילים פקודות משובצות המניפולציה את התנהגות הבינה בעת סקרייפינג
- **התקפות אימייל**: פקודות זדוניות במיילים שגורמים לעוזרי בינה לדלוף מידע או לבצע פעולות לא מורשות
- **זיהום מקורות נתונים**: מסדי נתונים או APIs שהותקפו ומספקים תכנים נגועים למערכות בינה

**השפעה בעולם האמיתי**: התקפות אלו עלולות לגרום לחשיפת נתונים, הפרת פרטיות, יצירת תוכן מזיק ומניפולציה באינטראקציות משתמש. לניתוח מפורט, ראו [הזרקת פקודות ב-MCP (סיימון ויליסון)](https://simonwillison.net/2025/Apr/9/mcp-prompt-injection/).

![תרשים התקפת הזרקת פקודות](../../../translated_images/he/prompt-injection.ed9fbfde297ca877.webp)

#### **התקפות הרעלת כלים**

**הרעלת כלים** מכוונת למטא-דאטה המגדירה כלים ב-MCP, וניצלת את האופן שבו מודלים גדולים מפרשים תיאורי כלים ופרמטרים לקבלת החלטות ביצוע.

**מנגנוני התקפה:**
- **מניפולציה במטא-דאטה**: תוקפים מזריקים הוראות זדוניות לתיאורי כלים, הגדרות פרמטרים או דוגמאות שימוש
- **הוראות בלתי נראות**: פקודות מוסתרות במטא-דאטה של כלים המעובדים ע"י מודלים אך נעלמות ממשתמשים אנושיים
- **שינוי דינמי של כלים ("רוג פול")**: כלים שאושרו ע"י המשתמש אחר כך משתנים לבצע פעולות זדוניות ללא ידיעת המשתמש
- **הזרקת פרמטרים**: תוכן זדוני המשובץ בסכימות פרמטרים של כלים שמשפיע על התנהגות המודל
**סיכוני שרת אירוח**: שרתי MCP מרוחקים מציגים סיכונים מוגברים כיוון שהגדרות הכלים יכולות להתעדכן לאחר אישור המשתמש הראשוני, ויוצרות מצבים בהם כלים שהיו בטוחים בעבר הופכים למזיקים. לניתוח מקיף, ראו [התקפות הרעלת כלים (Invariant Labs)](https://invariantlabs.ai/blog/mcp-security-notification-tool-poisoning-attacks).

![תרשים התקפת הזרקת כלים](../../../translated_images/he/tool-injection.3b0b4a6b24de6bef.webp)

#### **וקטורי התקפה נוספים ב-AI**

- **הזרקת הוראה בין-תחומית (XPIA)**: התקפות מתוחכמות המנצלות תוכן ממספר תחומים לעקיפת בקרות אבטחה  
- **שינוי יכולות דינאמי**: שינויים בזמן אמת ביכולות הכלים שמתחמקים מהערכות אבטחה ראשוניות  
- **הרעלת חלון הקשר**: התקפות המנצלות חלונות הקשר גדולים להסתיר הוראות מזיקות  
- **התקפות בלבול מודל**: ניצול מגבלות מודלים ליצירת התנהגויות בלתי צפויות או לא בטוחות  


### השפעת סיכוני אבטחת AI

**השלכות בעלות השפעה גבוהה:**
- **חשיפת נתונים**: גישה לא מורשית וגניבת נתונים רגישים של הארגון או מידע אישי  
- **הפרות פרטיות**: חשיפת מידע מזהה אישי (PII) ונתוני עסקים חסויים  
- **מניפולציה של מערכות**: שינויים לא מתוכננים במערכות קריטיות וזרימות עבודה  
- **גניבת אישורים**: פגיעה בטוקני אימות ואישורי שירות  
- **תנועה צידית**: שימוש במערכות AI פגומות כצירים להתקפות רחבות יותר ברשת

## איומי אבטחה מתקדמים ב-MCP

### פגיעות בשיבוש מושב (Session Hijacking)

**שיבוש מושב** מייצג וקטור התקפה קריטי ביישומים stateful של MCP, בהם צדדים לא מורשים משיגים ומנצלים מזהי מושבים לגיטימיים כדי להתחזות ללקוחות ולבצע פעולות לא מורשות.

#### **תרחישי התקפה וסיכונים**

- **הזרקת הוראות בשיבוש מושב**: תוקפים עם מזהי מושב גנובים מוזרקים אירועים מזיקים לשרתים שמשתפים מצב מושב, ועלולים לגרום לפעולות מזיקות או גישה לנתונים רגישים  
- **התחזות ישירה**: מזהי מושב גנובים מאפשרים קריאות ישירות לשרת MCP שמדלגות על אימות, וטיפול בתוקפים כמשתמשים לגיטימיים  
- **זרמי המשך פגומים**: תוקפים יכולים לקטוע בקשות לפני זמנן, ולגרום ללקוחות לגיטימיים להתחיל מחדש עם תוכן שעלול להיות מזיק  

#### **בקרות אבטחה לניהול מושבים**

**דרישות קריטיות:**
- **אימות הרשאות**: שרתי MCP המיישמים הרשאה **חייבים** לאמת את כל הבקשות הנכנסות ו**אסור** להסתמך על מושבים לאימות  
- **יצירת מושבים מאובטחת**: שימוש במזהי מושב קריפטוגרפיים, לא דטרמיניסטיים שהונפקו באמצעות מחוללי מספרים אקראיים מאובטחים  
- **קשירת מזהי מושבים למשתמש**: קשירת מזהי מושבים למידע ספציפי למשתמש באמצעות פורמטים כמו `<user_id>:<session_id>` למניעת ניצול מושבים בין משתמשים  
- **ניהול מחזור חיים של מושבים**: יישום של פקיעת תוקף, סיבוב והתבטלות כדי להגביל חלונות פגיעות  
- **אבטחת תעבורה**: HTTPS חובה לכל התקשורת למניעת יירוט מזהי מושב  

### בעיית סגן מבולבל (Confused Deputy)

בעיית הסגן המבולבל מתרחשת כאשר שרתי MCP פועלים כסוכני אימות בין לקוחות לשירותים צד שלישי, ויוצרים אפשרויות לעקיפת הרשאות באמצעות ניצול מזהה לקוח סטטי.

#### **מנגנוני התקפה וסיכונים**

- **עקיפת הסכמה מבוססת עוגיות**: אימות משתמש קודם יוצר עוגיות הסכמה שהן מנוצלות על ידי תוקפים באמצעות בקשות הרשאה זדוניות עם URI הצלבה מיועדת  
- **גניבת קוד הרשאה**: עוגיות הסכמה קיימות עשויות לגרום לשרתים להחמיץ מסכי הסכמה, ולנתב קודים לנקודות קצה בשליטת התוקף  
- **גישה לא מורשית ל-API**: קודי הרשאה גנובים מאפשרים החלפת טוקנים והתחזות למשתמשים ללא אישור מפורש  

#### **אסטרטגיות הקלה**

**בקרות חובה:**
- **דרישות הסכמה מפורשות**: שרתי פרוקסי MCP המשתמשים במזהי לקוח סטטיים **חייבים** לקבל הסכמה עבור כל לקוח שנרשם דינמית  
- **יישום אבטחת OAuth 2.1**: עקוב אחר שיטות האבטחה הטובות ביותר של OAuth לרבות PKCE לכל בקשות ההרשאה  
- **אימות מחמיר של הלקוח**: יישום אימות קפדני של URI Redirect ומזהי לקוח למניעת ניצול  

### פגיעות בהעברת טוקנים (Token Passthrough)

העברת טוקנים היא דפוס אנטי שבו שרתי MCP מקבלים טוקני לקוח ללא אימות נאות ומעבירים אותם ל-APIs מידריים, בניגוד למפרטי ההרשאה של MCP.

#### **השלכות אבטחה**

- **עקיפת בקרות**: שימוש ישיר בטוקנים מלקוח ל-API עוקף בקרות קריטיות של הגבלת קצב, אימות ומעקב  
- **שיבוש שרשראות ביקורת**: טוקנים שהונפקו במקום העליון מקשים על זיהוי הלקוח ושבירת יכולות חקירת אירועים  
- **גניבת נתונים פרוקסי**: טוקנים לא מאומתים מאפשרים לתוקפים להשתמש בשרתים כפרוקסי לגישה לא מורשית לנתונים  
- **הפרת גבולות אמון**: שירותים מידריים עלולים לסמוך בטעות על טוקנים כשמקורות הטוקנים אינם ניתנים לאימות  
- **התרחבות התקפה רב-שירותית**: טוקנים פגומים המקבלים שירותים מרובי מאפשרים תנועה צידית  

#### **בקרות אבטחה דרושות**

**דרישות בלתי מתפשרות:**
- **אימות טוקנים**: שרתי MCP **אסור** לקבל טוקנים שלא ניתנו במפורש עבור שרת ה-MCP  
- **אימות קהל (Audience)**: תמיד יש לוודא שטענות הקהל של הטוקן תואמות לזהות שרת ה-MCP  
- **מחזור חיים תקין לטוקן**: יישום טוקנים בעלי תוקף קצר עם נוהלי סיבוב מאובטחים  


## אבטחת שרשרת אספקה למערכות AI

אבטחת שרשרת האספקה התפתחה מעבר לתלותות תוכנה מסורתיות כדי לכלול את כל מערך ה-AI. יישומי MCP מודרניים חייבים לאמת ולנטר ברציפות את כל רכיבי ה-AI, משום שכל אחד מהם מציג פוטנציאל פגיעות שיוכל לפגוע בשלמות המערכת.

### רכיבי שרשרת אספקה מורחבים ל-AI

**תלותות תוכנה מסורתיות:**  
- ספריות וקונספטים בקוד פתוח  
- דימויי מכולות ומערכות בסיס  
- כלים לפיתוח וצנרת בנייה  
- רכיבי תשתית ושירותים  

**רכיבי שרשרת אספקה ייחודיים ל-AI:**  
- **מודלים בסיסיים**: מודלים מאומנים מראש מספקים שונים שיש לאמת את מקורם  
- **שירותי הטענה**: שירותי וקטוריזציה וחיפוש סמנטי חיצוניים  
- **ספקי הקשר**: מקורות נתונים, בסיסי ידע, ומאגרי מסמכים  
- **APIs צד שלישי**: שירותי AI חיצוניים, צנרות ML, ונקודות קצה לעיבוד נתונים  
- **חפצי מודל**: משקלים, תצורות וגרסאות מודל מותאמות  
- **מקורות נתוני אימון**: מערכי נתונים המשמשים לאימון ולדקונטינג מודלים  

### אסטרטגיית אבטחה מקיפה לשרשרת אספקה

#### **אימות רכיבים ואמון**  
- **אימות מקור**: ודא את מקור, רישוי ושלמות כל רכיבי ה-AI לפני אינטגרציה  
- **הערכת אבטחה**: ערוך סריקות פגיעות ובדיקות אבטחה למודלים, מקורות נתונים ושירותי AI  
- **ניתוח מוניטין**: הערך את היסטוריית האבטחה ושיטות העבודה של ספקי שירותי AI  
- **אימות תאימות**: ודא שכל הרכיבים עומדים בדרישות האבטחה והרגולציה של הארגון  

#### **צנרת פריסה מאובטחת**  
- **אבטחת CI/CD אוטומטית**: אינטגרציה של סריקות אבטחה לאורך כל צנרת הפריסה האוטומטית  
- **שלמות חפצים**: יישום אימות קריפטוגרפי לכל החפצים שמופצים (קוד, מודלים, תצורות)  
- **פריסה הדרגתית**: שימוש באסטרטגיות פריסה הדרגתית עם אימות אבטחה בכל שלב  
- **מאגרי חפצים מהימנים**: פריסה רק ממאגרים מאומתים ובטוחים של חפצי תוכנה  

#### **ניטור מתמשך ותגובה**  
- **סריקת תלותיות**: ניטור מתמשך לפגיעויות בכל תלות תוכנה ורכיבי AI  
- **ניטור מודלים**: הערכה מתמשכת של התנהגות המודל, סטייה בביצועים וחריגות אבטחה  
- **מעקב בריאות שירותים**: ניטור שירותי AI חיצוניים זמינות, תקריות אבטחה ושינויים במדיניות  
- **אינטגרציה של מודיעין איומים**: שילוב פידים ספציפיים לסיכוני אבטחת AI ו-ML  

#### **בקרת גישה וזכויות מינימליות**  
- **הרשאות ברמת רכיב**: הגבלת גישה למודלים, נתונים ושירותים לפי צורך עסקי  
- **ניהול חשבונות שירות**: יישום חשבונות שירות ייעודיים עם הרשאות מזעריות  
- **סגמנטציה רשתית**: בידוד רכיבי AI והגבלת גישה בין השירותים ברשת  
- **בקרות שער API**: שימוש בשערי API מרכזיים לשליטה ומעקב על גישה לשירותי AI חיצוניים  

#### **תגובה לאירועים ושחזור**  
- **נהלי תגובה מהירים**: תהליכים מוסדרים לתיקון או החלפת רכיבי AI שנפגעו  
- **סיבוב אישורים**: מערכות אוטומטיות לסיבוב סודות, מפתחות API ואישורי שירות  
- **יכולת שחזור**: אפשרות להחזיר במהירות לגרסאות קודמות תקינות של רכיבי AI  
- **שחזור מפריצות בשרשרת אספקה**: נהלים ספציפיים להתמודדות עם פגיעות בשירותי AI מדרגה עליונה  

### כלי וממשקי אבטחה של מייקרוסופט

**אינטגרציה עם Azure DevOps ו-Azure Repos:**  
- אינטגרציה חלקה של סריקות אבטחה בפלטפורמות הפיתוח של מייקרוסופט

- אכיפת מדיניות לפריסת רכיבי AI מאובטחים  

**שיטות עבודה פנימיות של מייקרוסופט:**  
מייקרוסופט מיישמת נוהלי אבטחה מקיפים לשרשרת אספקה על כל מוצריה. למדו על שיטות מוכחות ב-[המסע לאבטחת שרשרת האספקה בתוכנה במייקרוסופט](https://devblogs.microsoft.com/engineering-at-microsoft/the-journey-to-secure-the-software-supply-chain-at-microsoft/).


## נהלי אבטחה בסיסיים

יישומי MCP יורשים ומרחיבים את עמדת האבטחה הקיימת בארגון שלכם. חיזוק נהלי אבטחה בסיסיים משפר משמעותית את האבטחה הכוללת של מערכות AI ופריסות MCP.

### יסודות אבטחה מרכזיים

#### **נהלי פיתוח מאובטחים**  
- **ציות ל-OWASP**: הגנה מפני פגיעויות יישומי ווב מובילות לפי [OWASP Top 10](https://owasp.org/www-project-top-ten/)  
- **הגנות ייחודיות ל-AI**: יישום בקרים עבור [OWASP Top 10 ל-LLMs](https://genai.owasp.org/download/43299/?tmstv=1731900559)  
- **ניהול סודות מאובטח**: שימוש בכספות ייעודיות לטוקנים, מפתחות API ונתוני תצורה רגישים  
- **הצפנה מקצה לקצה**: יישום תקשורת מאובטחת בכל רכיבי היישום וזרמי הנתונים  
- **אימות קלט**: אימות קפדני של כל קלטי משתמש, פרמטרי API ומקורות נתונים  

#### **הקשחת תשתית**  
- **אימות רב-שלבי (MFA)**: חובה על כל חשבונות הנהלה ושירות  
- **ניהול עדכונים**: הטמעת תיקוני אבטחה בזמן לכל מערכות ההפעלה, מסגרות התוכנה ותלותות  
- **אינטגרציה עם ספקי זהות**: ניהול זהויות מרכזי דרך ספקי זהות ארגוניים (Microsoft Entra ID, Active Directory)  
- **סגמנטציה רשתית**: בידוד לוגי של רכיבי MCP להגבלת תנועה צידית  
- **עקרון ההרשאה המינימלית**: הרשאות מזעריות נדרשות לכל רכיבי המערכת והחשבונות  

#### **ניטור ואיתור אבטחה**  
- **רישום מקיף**: רישום מפורט של פעילויות יישום AI כולל אינטראקציות לקוח-שרת MCP  
- **אינטגרציה עם SIEM**: קונסול אבטחה מרכזי לאיתור חריגות  
- **אנליטיקה להתנהגות**: ניטור מבוסס AI לזיהוי דפוסים חריגים במערכת ובמשתמשים  
- **מודיעין איומים**: שילוב פידים חיצוניים וסימני פריצה (IOCs)  
- **ניהול מענה לאירועים**: נהלים מוגדרים לאיתור, תגובה ושחזור מאירועי אבטחה  

#### **ארכיטקטורת Zero Trust**  
- **לעולם לא לסמוך, תמיד לאמת**: אימות רציף של משתמשים, התקנים וחיבורים  
- **מיקרוסגמנטציה**: בקרות רשת מדויקות המבודדות משימות ושירותים פרטניים  
- **אבטחה מבוססת זהות**: מדיניות אבטחה על בסיס זהויות מאומתות במקום מיקום רשת  
- **הערכת סיכונים דינמית**: הערכת עמדה אבטחתית רציפה על פי ההקשר וההתנהגות הנוכחית  
- **גישה מותנית**: בקרות גישה המסתגלות לפי גורמי סיכון, מיקום ואמון התקנים  

### דפוסי אינטגרציה ארגונית

#### **אינטגרציית אקוסיסטם אבטחה של מייקרוסופט**  
- **Microsoft Defender for Cloud**: ניהול מצבי אבטחה בענן מקיף  
- **Azure Sentinel**: SIEM ו-SOAR מולד בענן להגנת עומסי עבודה של AI  
- **Microsoft Entra ID**: ניהול זהויות וגישה ארגוני עם מדיניות גישה מותנית  
- **Azure Key Vault**: ניהול סודות מרכזי עם גיבוי של מודול אבטחה חומרתי (HSM)  
- **Microsoft Purview**: ממשל וציות לנתונים עבור מקורות נתונים וזרימות עבודה של AI  

#### **ציות וממשל**  
- **התאמה לרגולציה**: ודא שיישומי MCP עומדים בדרישות ציות ענפיות (GDPR, HIPAA, SOC 2)
- **סיווג נתונים**: סיווג וטיפול מתאים בנתונים רגישים המעובדים על ידי מערכות AI  
- **רשומות ביקורת**: רישום מקיף לצורך תאימות רגולטורית וחקירות פורנזיות  
- **בקרות פרטיות**: יישום עקרונות פרטיות-by-design בארכיטקטורת מערכות AI  
- **ניהול שינויים**: תהליכים פורמליים לסקירות אבטחה של שינויים במערכות AI  

פרקטיקות יסוד אלו יוצרות בסיס אבטחה חזק שמגביר את יעילות בקרות האבטחה הספציפיות ל-MCP ומספק הגנה מקיפה ליישומים המונעים על ידי AI.  

## עיקרי אבטחה מרכזיים  

- **גישה רב-שכבתית לאבטחה**: שילוב פרקטיקות אבטחה יסודיות (קידוד מאובטח, הרשאות מזעריות, אימות שרשרת אספקה, ניטור מתמשך) עם בקרות ספציפיות ל-AI להגנה מקיפה  

- **נוף איומי AI ספציפיים**: מערכות MCP מתמודדות עם סיכונים ייחודיים כגון הזרקת פקודות מזויפות, הרעלת כלים, חטיפת סשנים, בעיות סגן מבולבל, פרצות בהעברת טוקנים, והרשאות מופרזות הדורשות הקלות מתמחות  

- **מצוינות באימות והרשאה**: יישום אימות חזק באמצעות ספקי זהות חיצוניים (Microsoft Entra ID), כפיית אימות טוקנים נכונים ולעולם לא לקבל טוקנים שלא הונפקו במפורש עבור שרת MCP שלך  

- **מניעת התקפות AI**: פריסת Microsoft Prompt Shields ו-Azure Content Safety להגנה מפני הזרקות פקודות עקיפות והרעלות כלים, תוך אימות מטה-נתונים של כלים וניטור שינויים דינמיים  

- **אבטחת סשנים וטרנספורט**: שימוש במזהי סשן קריפטוגרפיים, לא-דטרמיניסטיים, הקשורים לזהויות משתמש; ניהול מחזור חיים נכון של סשנים; ולעולם לא להשתמש בסשנים לאימות  

- **פרקטיקות מיטביות לאבטחת OAuth**: מניעת התקפות סגן מבולבל באמצעות הסכמה מפורשת של המשתמש ללקוחות רשומים דינמית, יישום OAuth 2.1 עם PKCE, ואימות מחמיר של URI להפניה מחדש  

- **עקרונות אבטחת טוקנים**: הימנעות מתבניות אנטי-העברת טוקן, אימות תובנות קהל היעד של הטוקן, שימוש בטוקנים קצרים עם סיבוב בטוח, ושמירת גבולות אמון ברורים  

- **אבטחת שרשרת אספקה מקיפה**: התייחסות לכל רכיבי האקוסיסטם של AI (מודלים, אמבדינגים, ספקי הקשר, APIs חיצוניים) עם אותו קפדן אבטחה כמו תלותיות תוכנה מסורתיות  

- **התפתחות מתמדת**: הישארות מעודכנת עם מפרטי MCP המשתנים במהירות, תרומה לסטנדרטים של קהילת האבטחה, ושמירה על עמדות אבטחה אדפטיביות ככל שהפרוטוקול מתפתח  

- **שילוב אבטחה של מיקרוסופט**: ניצול אקוסיסטם האבטחה המקיף של מיקרוסופט (Prompt Shields, Azure Content Safety, GitHub Advanced Security, Entra ID) להגנה משופרת בפריסת MCP
