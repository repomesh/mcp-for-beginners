# Безопасность MCP: комплексная защита систем ИИ

[![Лучшие практики безопасности MCP](../../../translated_images/ru/03.175aed6dedae133f.webp)](https://youtu.be/88No8pw706o)

_(Нажмите на изображение выше, чтобы посмотреть видео этого урока)_

Безопасность является фундаментальной частью проектирования систем ИИ, поэтому мы уделяем ей приоритет как второй секции. Это соответствует принципу Microsoft **Secure by Design** из [Инициативы безопасного будущего](https://www.microsoft.com/security/blog/2025/04/17/microsofts-secure-by-design-journey-one-year-of-success/).

Протокол контекста модели (MCP) открывает мощные новые возможности для приложений на базе ИИ, одновременно вводя уникальные проблемы безопасности, выходящие за рамки традиционных рисков программного обеспечения. Системы MCP сталкиваются с как устоявшимися вопросами безопасности (безопасное кодирование, минимальные привилегии, безопасность цепочки поставок), так и с новыми специфичными для ИИ угрозами, включая инъекции подсказок, отравление инструментов, захват сессий, атаки «запутавшегося заместителя», уязвимости обхода токенов и динамическую модификацию возможностей.

В этом уроке рассматриваются самые критичные риски безопасности при реализации MCP — включая аутентификацию, авторизацию, чрезмерные полномочия, косвенные инъекции подсказок, безопасность сессий, проблемы «запутавшегося заместителя», управление токенами и уязвимости в цепочке поставок. Вы узнаете действенные меры контроля и лучшие практики для снижения этих рисков, используя решения Microsoft, такие как Prompt Shields, Azure Content Safety и GitHub Advanced Security для усиления защиты MCP.

## Цели обучения

К концу этого урока вы сможете:

- **Определять специфичные для MCP угрозы**: распознавать уникальные риски безопасности в системах MCP, включая инъекции подсказок, отравление инструментов, чрезмерные полномочия, захват сессий, проблемы запутавшегося заместителя, уязвимости обхода токенов и риски цепочки поставок
- **Применять меры безопасности**: внедрять эффективные методы смягчения, включая надёжную аутентификацию, доступ с минимальными привилегиями, безопасное управление токенами, контроль безопасности сессий и проверку цепочки поставок
- **Использовать решения безопасности Microsoft**: понимать и применять Microsoft Prompt Shields, Azure Content Safety и GitHub Advanced Security для защиты рабочих нагрузок MCP
- **Проверять безопасность инструментов**: понимать важность проверки метаданных инструментов, мониторинга динамических изменений и защиты от косвенных инъекций подсказок
- **Интегрировать лучшие практики**: сочетать устоявшиеся основы безопасности (безопасное кодирование, усиление серверов, модель нулевого доверия) с контрольными механизмами MCP для комплексной защиты

# Архитектура и меры безопасности MCP

Современные реализации MCP требуют многоуровневого подхода к безопасности, учитывая как традиционные угрозы в ПО, так и специфичные угрозы ИИ. Быстро развивающаяся спецификация MCP продолжает совершенствовать меры безопасности, обеспечивая лучшую интеграцию с архитектурами корпоративной безопасности и проверенными лучшими практиками.

Исследования из [Microsoft Digital Defense Report](https://aka.ms/mddr) показывают, что **98% зарегистрированных нарушений могли быть предотвращены при строгом соблюдении гигиены безопасности**. Наиболее эффективная стратегия защиты сочетает фундаментальные меры безопасности с контролями MCP — проверенные базовые меры остаются самым важным фактором снижения общего риска.

### Эволюция аутентификации MCP

Спецификация MCP значительно изменилась в подходах к аутентификации и авторизации:

- **Исходный подход**: ранние спецификации требовали от разработчиков создавать собственные серверы аутентификации, где MCP серверы выступали как OAuth 2.0 Authorization Servers, управляя аутентификацией пользователей напрямую
- **Текущий стандарт (2025-11-25)**: обновленная спецификация позволяет MCP серверам делегировать аутентификацию внешним провайдерам идентичности (например, Microsoft Entra ID), повышая уровень безопасности и снижая сложность реализации
- **Защита транспортного уровня**: расширенная поддержка надежных механизмов транспортировки с корректными паттернами аутентификации как для локальных (STDIO), так и удаленных (Streamable HTTP) подключений

## Безопасность аутентификации и авторизации

### Современные вызовы безопасности

Современные реализации MCP сталкиваются с рядом проблем в аутентификации и авторизации:

### Риски и векторы атак

- **Ошибки в логике авторизации**: неправильная реализация авторизации в MCP серверах может привести к раскрытию чувствительных данных и некорректному применению контроля доступа
- **Компрометация OAuth токенов**: кража токенов локального MCP сервера позволяет злоумышленникам выдать себя за сервер и получить доступ к последующим сервисам
- **Уязвимости обхода токенов**: неправильная обработка токенов создает возможности обхода мер безопасности и пробелов в учёте
- **Чрезмерные полномочия**: избыточные привилегии MCP серверов нарушают принцип минимальных прав и расширяют поверхность атаки

#### Обход токенов: критический анти-паттерн

**Обход токенов строго запрещён** согласно текущей спецификации авторизации MCP из-за серьёзных последствий для безопасности:

##### Обход мер безопасности
- MCP серверы и downstream API реализуют важные меры безопасности (ограничение частоты запросов, валидация запросов, мониторинг трафика), которые зависят от правильной проверки токенов
- Прямое использование токенов клиентом к API обходит эти ключевые защиты, подрывая архитектуру безопасности

##### Проблемы ответственности и аудита  
- MCP серверы не могут отличить клиентов, использующих токены, выданные upstream, что нарушает цепочки аудита
- Логи downstream ресурсных серверов отображают ошибочное происхождение запросов вместо реальных промежуточных MCP серверов
- Расследования инцидентов и проверки соответствия становятся значительно сложнее

##### Риски утечки данных
- Непроверенные утверждения токенов позволяют злоумышленникам с украденными токенами использовать MCP серверы как прокси для утечки данных
- Нарушения границ доверия создают несанкционированные схемы доступа, обходящие задуманные меры безопасности

##### Векторы атак на несколько сервисов
- Компрометированные токены, принимаемые несколькими сервисами, позволяют перемещаться по связанным системам
- Нарушается доверие между сервисами, когда происхождение токенов не может быть проверено

### Контроль и смягчение угроз безопасности

**Критические требования безопасности:**

> **ОБЯЗАТЕЛЬНО:** MCP серверы **НЕ ДОЛЖНЫ** принимать токены, которые явно не были выданы для данного MCP сервера

#### Средства аутентификации и авторизации

- **Тщательный аудит авторизации**: проводите всесторонние проверки логики авторизации MCP серверов, чтобы обеспечить доступ только предназначенным пользователям и клиентам к чувствительным ресурсам
  - **Руководство по реализации**: [Azure API Management как шлюз аутентификации для MCP серверов](https://techcommunity.microsoft.com/blog/integrationsonazureblog/azure-api-management-your-auth-gateway-for-mcp-servers/4402690)
  - **Интеграция идентичности**: [Использование Microsoft Entra ID для аутентификации MCP серверов](https://den.dev/blog/mcp-server-auth-entra-id-session/)

- **Безопасное управление токенами**: применяйте [лучшие практики валидации и жизненного цикла токенов Microsoft](https://learn.microsoft.com/en-us/entra/identity-platform/access-tokens)
  - Валидируйте совпадение аудитории токена с идентичностью MCP сервера
  - Внедряйте правильное обновление и срок действия токенов
  - Предотвращайте повторное использование токенов и несанкционированное применение

- **Защищённое хранение токенов**: обеспечьте шифрование хранения токенов как в покое, так и при передаче
  - **Лучшие практики**: [Руководство по безопасному хранению и шифрованию токенов](https://youtu.be/uRdX37EcCwg?si=6fSChs1G4glwXRy2)

#### Реализация контроля доступа

- **Принцип наименьших привилегий**: предоставляйте MCP серверам только минимальные права, необходимые для требуемой функциональности
  - Регулярно проверяйте и обновляйте полномочия, чтобы избежать их роста
  - **Документация Microsoft**: [Обеспечение безопасного доступа с минимальными правами](https://learn.microsoft.com/entra/identity-platform/secure-least-privileged-access)

- **Ролевая модель контроля доступа (RBAC)**: внедряйте детализированные назначения ролей
  - Жёстко ограничивайте роли конкретными ресурсами и действиями
  - Избегайте широких или лишних полномочий, расширяющих поверхность атаки

- **Постоянный мониторинг разрешений**: внедряйте непрерывный аудит и контроль доступа
  - Следите за аномалиями в использовании полномочий
  - Быстро устраняйте избыточные или неиспользуемые права

## Специфичные для ИИ угрозы безопасности

### Инъекции подсказок и атаки с манипуляцией инструментами

Современные реализации MCP сталкиваются со сложными специфичными для ИИ векторами атак, которые традиционные меры безопасности не способны полностью нейтрализовать:

#### **Косвенная инъекция подсказок (перекрестные инъекции)**

**Косвенная инъекция подсказок** — одна из самых критичных уязвимостей в системах ИИ на основе MCP. Злоумышленники внедряют вредоносные инструкции в внешний контент — документы, веб-страницы, электронные письма или источники данных, которые системы ИИ затем воспринимают как легитимные команды.

**Сценарии атак:**
- **Документные инъекции**: вредоносные инструкции, скрытые в обрабатываемых документах, вызывающие непредвиденные действия ИИ
- **Эксплуатация веб-контента**: скомпрометированные веб-страницы с встроенными подсказками, манипулирующими поведением ИИ при сканировании
- **Почтовые атаки**: вредоносные подсказки в письмах, заставляющие ИИ-ассистентов раскрывать информацию или выполнять неавторизованные действия
- **Загрязнение источников данных**: скомпрометированные базы данных или API, предоставляющие заражённый контент системам ИИ

**Реальные последствия**: такие атаки могут привести к утечке данных, нарушениям конфиденциальности, генерации вредоносного контента и манипулированию взаимодействиями с пользователем. Для подробного анализа см. [Инъекция подсказок в MCP (Simon Willison)](https://simonwillison.net/2025/Apr/9/mcp-prompt-injection/).

![Диаграмма атаки инъекции подсказок](../../../translated_images/ru/prompt-injection.ed9fbfde297ca877.webp)

#### **Атаки с отравлением инструментов**

**Отравление инструментов** нацелено на метаданные, определяющие инструменты MCP, эксплуатируя интерпретацию описаний и параметров инструментов LLM для принятия решений о выполнении.

**Механизмы атаки:**
- **Манипуляция метаданными**: злоумышленники внедряют вредоносные инструкции в описания инструментов, определения параметров или примеры использования
- **Невидимые инструкции**: скрытые подсказки в метаданных инструментов, которые обрабатываются моделями ИИ, но невидимы для пользователей
- **Динамическая модификация инструментов («Rug Pulls»)**: инструменты, одобренные пользователями, позднее изменяются для выполнения вредоносных действий без ведома пользователя
- **Инъекции параметров**: вредоносный контент, встроенный в схемы параметров инструментов, влияющий на поведение модели


**Риски хостинговых серверов**: Удалённые MCP-серверы представляют повышенные риски, так как определения инструментов могут обновляться после первоначального одобрения пользователем, создавая сценарии, при которых ранее безопасные инструменты становятся вредоносными. Для всестороннего анализа см. [Атаки отравления инструментов (Invariant Labs)](https://invariantlabs.ai/blog/mcp-security-notification-tool-poisoning-attacks).

![Диаграмма атаки внедрения инструмента](../../../translated_images/ru/tool-injection.3b0b4a6b24de6bef.webp)

#### **Дополнительные векторы атак ИИ**

- **Внедрение подсказок через перекрёстные домены (XPIA)**: Сложные атаки, использующие контент из нескольких доменов для обхода средств защиты
- **Динамическое изменение возможностей**: Изменения возможностей инструментов в реальном времени, которые ускользают от первоначальных оценок безопасности
- **Отравление окна контекста**: Атаки, манипулирующие большими окнами контекста для скрытия вредоносных инструкций
- **Атаки путаницы модели**: Использование ограничений модели для создания непредсказуемого или небезопасного поведения


### Влияние рисков ИИ на безопасность

**Последствия с высоким уровнем воздействия:**
- **Экфильтрация данных**: Несанкционированный доступ и кража конфиденциальных данных предприятия или личных данных
- **Нарушения конфиденциальности**: Разглашение персональных идентифицируемых данных (PII) и конфиденциальной бизнес-информации  
- **Манипуляции с системами**: Непреднамеренные изменения в критических системах и рабочих процессах
- **Кража учётных данных**: Компрометация токенов аутентификации и учётных данных сервисов
- **Латеральное перемещение**: Использование скомпрометированных систем ИИ в качестве точек опоры для более широких сетевых атак

### Решения Microsoft по безопасности ИИ

## Продвинутые угрозы безопасности MCP

### Уязвимости перехвата сессии

**Перехват сессии** представляет собой критический вектор атаки в реализациях MCP с поддержкой состояния, когда несанкционированные лица получают и используют действительные идентификаторы сессий для выдачи себя за клиентов и выполнения несанкционированных действий.

#### **Сценарии атак и риски**

- **Внедрение подсказок через перехват сессии**: Злоумышленники с похищенными идентификаторами сессий внедряют вредоносные события в серверы с общим состоянием сессии, что может запускать вредоносные действия или доступ к конфиденциальным данным
- **Прямое выдавание себя за другого**: Похищенные идентификаторы сессий позволяют делать прямые вызовы MCP сервера, обходя аутентификацию, воспринимая злоумышленников как легитимных пользователей
- **Скомпрометированные возобновляемые потоки**: Злоумышленники могут преждевременно завершать запросы, заставляя легитимных клиентов возобновлять с потенциально вредоносным содержимым

#### **Контрмеры для управления сессиями**

**Критические требования:**
- **Проверка авторизации**: MCP-серверы с реализацией авторизации **ДОЛЖНЫ** проверять ВСЕ входящие запросы и **НЕ ДОЛЖНЫ** полагаться на сессии для аутентификации
- **Безопасная генерация сессий**: Использование криптографически надёжных, недетерминированных идентификаторов сессий, генерируемых безопасными генераторами случайных чисел
- **Привязка к пользователю**: Привязка идентификаторов сессий к информации пользователя с использованием форматов, таких как `<user_id>:<session_id>`, чтобы предотвратить злоупотребление сессиями между пользователями
- **Управление жизненным циклом сессии**: Реализация корректного истечения срока, ротации и аннулирования для ограничения окон уязвимости
- **Безопасность транспорта**: Обязательное использование HTTPS для всей коммуникации для предотвращения перехвата идентификаторов сессий

### Проблема "озадаченного заместителя"

Проблема **озадаченного заместителя** возникает, когда MCP-серверы выступают в роли прокси аутентификации между клиентами и сторонними сервисами, создавая возможности обхода авторизации путём эксплуатации статичных идентификаторов клиента.

#### **Механизмы атак и риски**

- **Обход согласия на основе cookies**: Предыдущая аутентификация пользователя создает cookies согласия, которые злоумышленники используют через вредоносные запросы авторизации с подделанными URI перенаправления
- **Кража кода авторизации**: Существующие cookies согласия могут привести к тому, что серверы авторизации пропускают экраны согласия, перенаправляя коды на контролируемые злоумышленниками конечные точки  
- **Несанкционированный доступ к API**: Похищенные коды авторизации позволяют обмен токенов и выдачу себя за пользователя без явного одобрения

#### **Стратегии смягчения**

**Обязательные меры:**
- **Требования к явному согласию**: MCP-прокси-серверы с использованием статичных идентификаторов клиента **ДОЛЖНЫ** получать согласие пользователя для каждого динамически зарегистрированного клиента
- **Реализация безопасности OAuth 2.1**: Следовать актуальным лучшим практикам безопасности OAuth, включая PKCE (Proof Key for Code Exchange) для всех запросов авторизации
- **Строгая валидация клиента**: Реализовать строгую проверку URI перенаправления и идентификаторов клиентов для предотвращения эксплуатации

### Уязвимости при передаче токенов  

**Передача токенов** представляет собой явное антишаблон, когда MCP-серверы принимают клиентские токены без надлежащей проверки и пересылают их в нижестоящие API, нарушая спецификации авторизации MCP.

#### **Последствия для безопасности**

- **Обход контроля**: Прямое использование клиентских токенов в API обходят критически важные ограничения по скорости, проверки и мониторинга
- **Повреждение аудита**: Токены, выданные выше по цепочке, делают невозможной идентификацию клиента, нарушая возможности расследования инцидентов
- **Экфильтрация данных через прокси**: Непроверенные токены позволяют злоумышленникам использовать серверы как прокси для несанкционированного доступа к данным
- **Нарушения доверительных границ**: Доверительные предположения услуг могут быть нарушены, когда происхождение токенов невозможно проверить
- **Расширение атак на несколько сервисов**: Скомпрометированные токены, принятые несколькими сервисами, позволяют осуществлять латеральное перемещение

#### **Требуемые меры безопасности**

**Обязательные требования:**
- **Проверка токенов**: MCP-серверы **НЕ ДОЛЖНЫ** принимать токены, не явно выданные для MCP-сервера
- **Проверка аудитории (audience)**: Всегда проверять, что аудитория токена соответствует идентичности MCP-сервера
- **Правильный жизненный цикл токена**: Реализовать краткоживущие токены доступа с безопасной ротацией


## Безопасность цепочки поставок AI систем

Безопасность цепочки поставок вышла за рамки традиционных зависимостей ПО и охватывает всю экосистему ИИ. Современные реализации MCP должны строго проверять и контролировать все AI-компоненты, поскольку каждый из них вносит потенциальные уязвимости, которые могут скомпрометировать целостность системы.

### Расширенные компоненты цепочки поставок AI

**Традиционные зависимости ПО:**
- Библиотеки и фреймворки с открытым исходным кодом
- Образы контейнеров и базовые системы  
- Инструменты разработки и конвейеры сборки
- Компоненты и сервисы инфраструктуры

**Специфичные для AI элементы цепочки поставок:**
- **Основные модели**: Предобученные модели от различных поставщиков, требующие проверки происхождения
- **Сервисы векторизации**: Внешние сервисы векторизации и семантического поиска
- **Поставщики контекста**: Источники данных, базы знаний и хранилища документов  
- **Сторонние API**: Внешние AI-сервисы, ML-конвейеры и конечные точки обработки данных
- **Артефакты моделей**: Весовые коэффициенты, конфигурации и дообученные варианты моделей
- **Источники данных для обучения**: Наборы данных, используемые для тренировки и дообучения моделей

### Комплексная стратегия безопасности цепочки поставок

#### **Проверка компонентов и доверие**
- **Проверка происхождения**: Подтверждение источника, лицензирования и целостности всех AI-компонентов перед интеграцией
- **Оценка безопасности**: Проведение сканирования уязвимостей и обзоров безопасности моделей, источников данных и AI-сервисов
- **Анализ репутации**: Оценка истории безопасности и практик поставщиков AI-сервисов
- **Проверка соответствия**: Обеспечение соответствия всех компонентов организационным требованиям безопасности и регулятивным нормам

#### **Безопасные конвейеры развертывания**  
- **Автоматическое сканирование безопасности CI/CD**: Интеграция сканирования безопасности во все этапы автоматизированных конвейеров развертывания
- **Целостность артефактов**: Реализация криптографической проверки всех развертываемых артефактов (кода, моделей, конфигураций)
- **Пошаговое развертывание**: Использование прогрессивных стратегий с проверкой безопасности на каждом этапе
- **Доверенные хранилища артефактов**: Развёртывание только из проверенных, защищённых реестров и репозиториев артефактов

#### **Постоянный мониторинг и реагирование**
- **Сканирование зависимостей**: Постоянный мониторинг уязвимостей для всех программных и AI-компонент
- **Мониторинг моделей**: Непрерывная оценка поведения моделей, дрейфа производительности и аномалий безопасности
- **Отслеживание состояния сервисов**: Мониторинг внешних AI-сервисов на предмет доступности, инцидентов безопасности и изменений политики
- **Интеграция разведданных об угрозах**: Включение лент угроз, специфичных для рисков ИИ и ML

#### **Контроль доступа и принцип наименьших привилегий**
- **Разрешения на уровне компонентов**: Ограничение доступа к моделям, данным и сервисам на основе деловой необходимости
- **Управление служебными аккаунтами**: Внедрение выделенных служебных аккаунтов с минимально необходимыми правами
- **Сегментация сети**: Изоляция компонентов ИИ и ограничение сетевого доступа между сервисами
- **Контроль API-шлюзов**: Использование централизованных API-шлюзов для контроля и мониторинга доступа к внешним AI-сервисам

#### **Реагирование на инциденты и восстановление**
- **Процедуры быстрого реагирования**: Установленные процессы для исправления уязвимых или замены скомпрометированных AI-компонентов
- **Ротация учётных данных**: Автоматизированные системы ротирования секретов, API-ключей и служебных учётных данных
- **Возможности отката**: Способность быстро вернуть предыдущие проверенные версии AI-компонентов
- **Восстановление после взлома цепочки поставок**: Специфичные процедуры реагирования на компрометацию ведущих AI-сервисов

## Лучшие практики для обеспечения безопасности базового уровня

Реализации MCP наследуют и развивают существующую в вашей организации позицию безопасности. Укрепление базовых практик значительно повышает общую безопасность AI-систем и внедрений MCP.

### Основы безопасности

#### **Практики безопасной разработки**
- **Соответствие OWASP**: Защита от [OWASP Топ-10](https://owasp.org/www-project-top-ten/) уязвимостей веб-приложений
- **Специфичные меры для ИИ**: Реализация контролей для [OWASP Топ-10 для LLM](https://genai.owasp.org/download/43299/?tmstv=1731900559)
- **Безопасное управление секретами**: Использование выделенных хранилищ для токенов, API-ключей и конфиденциальных данных конфигурации
- **Сквозное шифрование**: Реализация защищённых коммуникаций во всех компонентах приложения и потоках данных
- **Валидация входных данных**: Тщательная проверка всех пользовательских вводов, параметров API и источников данных

#### **Укрепление инфраструктуры**
- **Многофакторная аутентификация**: Обязательное MFA для всех административных и служебных аккаунтов
- **Управление патчами**: Автоматизированное своевременное устранение уязвимостей операционных систем, фреймворков и зависимостей  
- **Интеграция провайдеров идентичности**: Централизованное управление идентичностью через корпоративные провайдеры (Microsoft Entra ID, Active Directory)
- **Сегментация сети**: Логическая изоляция компонентов MCP для ограничения латерального перемещения
- **Принцип наименьших привилегий**: Минимально необходимые разрешения для всех системных компонентов и аккаунтов

#### **Мониторинг и обнаружение безопасности**
- **Всестороннее логирование**: Детализированное логирование активности AI-приложений, включая взаимодействия клиента и сервера MCP
- **Интеграция SIEM**: Централизованное управление информацией и событиями безопасности для обнаружения аномалий
- **Аналитика поведения**: Мониторинг с применением ИИ для выявления необычных моделей в поведении системы и пользователей
- **Разведка угроз**: Интеграция внешних информационных потоков о угрозах и индикаторов компрометации (IOC)
- **Реагирование на инциденты**: Чётко определённые процедуры обнаружения, реагирования и восстановления после инцидентов безопасности

#### **Архитектура с нулевым доверием (Zero Trust)**
- **Никогда не доверять, всегда проверять**: Постоянная проверка пользователей, устройств и сетевых соединений
- **Микросегментация**: Точные сетевые контролы, изолирующие отдельные рабочие нагрузки и сервисы
- **Идентично-центричная безопасность**: Политики безопасности на основе проверенных идентичностей, а не местоположения в сети
- **Постоянная оценка рисков**: Динамическая оценка позиции безопасности на основе текущего контекста и поведения
- **Условный доступ**: Контроли доступа, адаптирующиеся на основе факторов риска, местоположения и доверия к устройству

### Шаблоны интеграции в предприятия

#### **Интеграция в экосистему безопасности Microsoft**
- **Microsoft Defender for Cloud**: Комплексное управление позицией безопасности облака
- **Azure Sentinel**: Облачные возможности SIEM и SOAR для защиты AI-нагрузок
- **Microsoft Entra ID**: Корпоративное управление идентичностью и доступом с политиками условного доступа
- **Azure Key Vault**: Централизованное управление секретами с поддержкой аппаратных модулей безопасности (HSM)
- **Microsoft Purview**: Управление данными и соответствием для источников и рабочих процессов AI

#### **Соответствие и управление**
- **Соответствие нормативным требованиям**: Обеспечение реализации MCP в соответствии с отраслевыми требованиями (GDPR, HIPAA, SOC 2)

- **Классификация данных**: Правильная категоризация и обработка конфиденциальных данных, обрабатываемых системами ИИ
- **Аудиторские следы**: Всестороннее ведение журналов для соблюдения нормативных требований и судебной экспертизы
- **Контроль конфиденциальности**: Внедрение принципов защиты конфиденциальности по умолчанию в архитектуру систем ИИ
- **Управление изменениями**: Формальные процессы для обзорa безопасности модификаций систем ИИ

Эти базовые практики создают надежную базу безопасности, которая повышает эффективность специфических для MCP мер безопасности и обеспечивает всестороннюю защиту приложений на базе ИИ.

## Основные выводы по безопасности

- **Слойный подход к безопасности**: Сочетайте базовые практики безопасности (безопасное кодирование, принцип наименьших привилегий, проверка цепочки поставок, непрерывный мониторинг) с контролями, специфичными для ИИ, для всесторонней защиты

- **Специфические угрозы для ИИ**: Системы MCP сталкиваются с уникальными рисками, включая инъекции запросов, отравление инструментов, похищение сессий, проблемы с запутавшимся помощником, уязвимости передачи токенов и чрезмерные полномочия, которые требуют специализированных мер противодействия

- **Отличная аутентификация и авторизация**: Внедрять надежную аутентификацию с использованием внешних поставщиков идентификации (Microsoft Entra ID), обеспечивать правильную проверку токенов и никогда не принимать токены, не выданные явно для вашего MCP-сервера

- **Предотвращение атак на ИИ**: Используйте Microsoft Prompt Shields и Azure Content Safety для защиты от косвенных инъекций запросов и атак отравления инструментов, одновременно проверяя метаданные инструментов и контролируя динамические изменения

- **Безопасность сессий и транспорта**: Используйте криптографически защищённые, недетерминированные идентификаторы сессий, привязанные к идентичности пользователя, реализуйте правильное управление жизненным циклом сессии и никогда не используйте сессии для аутентификации

- **Лучшие практики безопасности OAuth**: Предотвращайте атаки с запутавшимся помощником посредством явного согласия пользователя для динамически зарегистрированных клиентов, корректной реализации OAuth 2.1 с PKCE и строгой проверкой redirect URI  

- **Принципы безопасности токенов**: Избегайте антипаттернов передачи токенов, проверяйте claims аудитории токена, внедряйте краткоживущие токены с безопасной ротацией и поддерживайте четкие границы доверия

- **Всесторонняя безопасность цепочки поставок**: Относитесь к всем компонентам экосистемы ИИ (модели, embedding'и, поставщики контекста, внешние API) с той же строгостью безопасности, что и к традиционным программным зависимостям

- **Постоянная эволюция**: Следите за быстрым развитием спецификаций MCP, вносите вклад в стандарты сообщества по безопасности и поддерживайте адаптивные положения безопасности по мере совершенствования протокола

- **Интеграция с безопасностью Microsoft**: Используйте всестороннюю экосистему безопасности Microsoft (Prompt Shields, Azure Content Safety, GitHub Advanced Security, Entra ID) для усиления защиты развертываний MCP
