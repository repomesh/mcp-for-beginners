# HTTPS ಸ್ಟ್ರೀಮಿಂಗ್‌ ವಿತ್ ಮಾಡೆಲ್ ಕಾಂಟೆಕ್ಸ್ಟ್ ಪ್ರೋಟೋಕಾಲ್ (MCP)

ಈ ಅಧ್ಯಾಯವು ಮಾಡೆಲ್ ಕಾಂಟೆಕ್ಸ್ಟ್ ಪ್ರೋಟೋಕಾಲ್ (MCP) ಬಳಸಿಕೊಂಡು HTTPS ಮೂಲಕ ಸುರಕ್ಷಿತ, ಅಗಲಿಸುವ ಮತ್ತು实时 ಸ್ಟ್ರೀಮಿಂಗ್ ಅನ್ನು ಜಾರಿಗೊಳಿಸುವ ಕುರಿತು ಸಂಪೂರ್ಣ ಮಾರ್ಗಸೂಚಿಯನ್ನು ನೀಡುತ್ತದೆ. ಇದರಲ್ಲಿ ಸ್ಟ್ರೀಮಿಂಗ್‌ನ ಪ್ರೇರಣೆ, ಲಭ್ಯವಿರುವ ಸಾರಣಾ ಯಂತ್ರಗಳು, MCP ನಲ್ಲಿ ಸ್ಟ್ರೀಮಿಂಗ್ HTTP ಅನ್ನು ಹೇಗೆ ಜಾರಿಗೊಳಿಸುವುದು, ಸುರಕ್ಷತಾ ಉತ್ತಮ ಅಭ್ಯಾಸಗಳು, SSE ರಿಂದ ಸ್ಥಳಾಂತರಿಸುವ ಪ್ರಕ್ರಿಯೆ, ಮತ್ತು ನಿಮ್ಮದೇ ಸ್ಟ್ರೀಮಿಂಗ್ MCP ಅಪ್ಲಿಕೇಶನ್ ಗಳನ್ನು ನಿರ್ಮಿಸಲು ಪ್ರಾಯೋಗಿಕ ಮಾರ್ಗದರ್ಶನಗಳನ್ನು ಒಳಗೊಂಡಿದೆ.

> **ಮುಂದಿನ ಅವಲೋಕನ:** ಈ ಪಾಠವು **MCP ಸ್ಪೆಸಿಫಿಕೇಶನ್ 2025-11-25** ಅಡಿಯಲ್ಲಿ ಸ್ಟ್ರೀಮಿಂಗ್ HTTP ಅನ್ನು ವಿವರಿಸುತ್ತದೆ, ಅದರಲ್ಲಿ ಒಂದು ಸೆಷನ್ `initialize` ಸಮಯದಲ್ಲಿ ಸ್ಥಾಪಿತಗೊಂಡು `Mcp-Session-Id` ಹೆಡರ್‌ನೊಂದಿಗೆ ಪಿನ್ಮುದುರುತ್ತದೆ. `2026-07-28` ಬಿಡುಗಡೆ ಅಭ್ಯರ್ಥಿ ಹ್ಯಾಂಡ್‌ಶೇಕ್ ಮತ್ತು ಸೆಷನ್ ID ಅನ್ನು ಸಂಪೂರ್ಣವಾಗಿ ತೆಗೆದುಹಾಕಿ, ಪ್ರತಿ ವಿನಂತಿ ಸ್ವತಂತ್ರವಾಗಿದ್ದು ಯಾವುದೆ ಸರ್ವರ್ ಉದ್ಘಟನೆಗೂ ಮಾರ್ಗಸೂಚಿ ಮಾಡಬಹುದು, ಸ್ಟಿಕ್ಕಿ ಸೆಷನ್ ಗಳಿಲ್ಲದೆ. ವಿವರಗಳಿಗೆ [MCP ನಲ್ಲಿ ಏನಾಗುತ್ತಿದೆ: 2026-07-28 ಬಿಡುಗಡೆ ಅಭ್ಯರ್ಥಿ](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) ನೋಡಿರಿ.

## MCP ಯಲ್ಲಿ ಸಾರಣಾ ಯಂತ್ರಗಳು ಮತ್ತು ಸ್ಟ್ರೀಮಿಂಗ್

ಈ ವಿಭಾಗವು MCP ನಲ್ಲಿ ಲಭ್ಯವಿರುವ ವಿಭಿನ್ನ ಸಾರಣಾ ಯಂತ್ರಗಳನ್ನು ಮತ್ತು ಅವುಗಳು ಕ್ಲೈಂಟ್ ಮತ್ತು ಸರ್ವರ್‌ಗಳ ನಡುವَى实时 ಸಂವಹನಕ್ಕೆ ಸ್ಟ್ರೀಮಿಂಗ್ ಸಾಮರ್ಥ್ಯಗಳನ್ನು ಹೇಗೆಂಟುಮಾಡುತ್ತವೆ ಎಂಬುದನ್ನು ಪರಿಶೀಲಿಸುತ್ತದೆ.

### ಸಾರಣಾ ಯಂತ್ರ ಎಂದರೆ ಏನು?

ಸಾರಣಾ ಯಂತ್ರವು ಕ್ಲೈಂಟ್ ಮತ್ತು ಸರ್ವರ್ ನಡುವೆ ಡೇಟಾ ವಿನಿಮಯವನ್ನು ಹೇಗೆ ನಡೆಸಲಾಗುತ್ತದೆ ಎಂಬುದನ್ನು ನಿರ್ಧರಿಸುತ್ತದೆ. MCP ವಿವಿಧ ಪರಿಸ್ಥಿತಿಗಳ ಹಾಗೂ ಅಗತ್ಯಗಳಿಗಾಗಿ ಹಲವು ಸಾರಣಾ ರೀತಿಗಳನ್ನು ಬೆಂಬಲಿಸುತ್ತದೆ:

- **stdio**: ಸ್ಟ್ಯಾಂಡರ್ಡ್ ಇನ್ಪುಟ್/ಔಟ್‌ಪುಟ್, ಸ್ಥಳೀಯ ಮತ್ತು CLI ಆಧಾರಿತ ಟೂಲ್ಗಳಿಗೆ ಸೂಕ್ತ. ಸರಳ ಆದರೆ ವೆಬ್ ಅಥವಾ ಕ್ಲೌಡ್‌ಗೆ ಅನ್ವಯಿಸದು.
- **SSE (ಸರ್ವರ್-ಸೆಂಟ್ ಇವೆಂಟ್ಸ್)**: ಸರ್ವರ್‌ಗಳು ಐಚ್ಛಿಕ HTTP ಮೂಲಕ实时 ನವೀಕರಣಗಳನ್ನು ಕ್ಲೈಂಟ್‌ಗಳಿಗೆ ಒಪ್ಪಿಸಲು ಸಹಾಯ ಮಾಡುತ್ತದೆ. ವೆಬ್ UI ಗಳಿಗೆ ಉತ್ತಮ, ಆದರೆ ಅಗಲುವಿಕೆ ಮತ್ತು ಸ್ವಾತಂತ್ರ್ಯದಲ್ಲಿ ಮಿತಿಯಿದೆ. MCP ಸ್ಪೆಸಿಫಿಕೇಶನ್ 2025-06-18 ರಿಂದ, ಸ್ಟ್ಯಾಂಡ್ಅಲೋನ್ SSE ಸಾರಣಾ ಯಂತ್ರವನ್ನು ನಿಷಿದ್ಧಗೊಳಿಸಿ "ಸ್ಟ್ರೀಮಬಲ್ HTTP" ಸಾರಣಾ ಮೂಲಕ ಬದಲಿಸಲಾಗಿದೆ.
- **ಸ್ಟ್ರೀಮಬಲ್ HTTP**: ಆಧುನಿಕ HTTP ಆಧಾರಿತ ಸ್ಟ್ರೀಮಿಂಗ್ ಸಾರಣಾ ವ್ಯವಸ್ಥೆ, ಸೂಚನೆಗಳು ಮತ್ತು ಉತ್ತಮ ಅಗಲುವಿಕೆಯನ್ನು ಬೆಂಬಲಿಸುತ್ತದೆ. ಬಹುಮಟ್ಟಿಗೆ ಉತ್ಪಾದನೆ ಮತ್ತು ಕ್ಲೌಡ್ ಪರಿಸ್ಥಿತಿಗಳಿಗೆ ಶಿಫಾರಸು ಮಾಡಲಾಗಿದೆ.

### ಹೋಲಿಕೆ ಟೇಬಲ್

ಈ ಹೋಲಿಕೆ ಟೇಬಲ್‌ಗೆ ಒಂದು ನೋಟ ಮಾಡಿ ಈ ಸಾರಣಾ ಯಂತ್ರಗಳ ನಡುವಿನ ವ್ಯತ್ಯಾಸಗಳನ್ನು ಅರ್ಥಮಾಡಿಕೊಳ್ಳಿ:

| ಸಾರಣೆ |实时 ನವೀಕರಣಗಳು | ಸ್ಟ್ರೀಮಿಂಗ್ | ಅಗಲುವಿಕೆ | ಬಳಕೆ ಪ್ರಕರಣ |
|-------------------|------------------|-----------|-------------|-------------------------|
| stdio | ಇಲ್ಲ | ಇಲ್ಲ | ಕಡಿಮೆ | ಸ್ಥಳೀಯ CLI ಟೂಲ್ಗಳು |
| SSE | ಹೌದು | ಹೌದು | ಮಧ್ಯಮ | ವೆಬ್,实时 ನವೀಕರಣಗಳು |
| ಸ್ಟ್ರೀಮಬಲ್ HTTP | ಹೌದು | ಹೌದು | ಹೆಚ್ಚು | ಕ್ಲೌಡ್, ಬಹು-ಕ್ಲೈಂಟ್ |

> **ಟಿಪ್:** ಸರಿಯಾದ ಸಾರಣೆ ಆಯ್ಕೆ ಮಾಡುವುದರಿಂದ ಕಾರ್ಯಕ್ಷಮತೆ, ಅಗಲುವಿಕೆ ಮತ್ತು ಬಳಕೆದಾರ ಅನುಭವ ಪ್ರಭಾವಿತವಾಗುತ್ತದೆ. **ಸ್ಟ್ರೀಮಬಲ್ HTTP** ಆಧುನಿಕ, ಅಗಲುವಿಕೆ ಪ್ರಧಾನ ಮತ್ತು ಕ್ಲೌಡ್-ಸಿದ್ಧ ಅಪ್ಲಿಕೇಶನ್ ಗಾಗಿ ಶಿಫಾರಸು.

ನಿಮಗೆ ಹಿಂದಿನ ಅಧ್ಯಾಯಗಳಲ್ಲಿ ತೋರಿಸಿದ್ದ stdio ಮತ್ತು SSE ಸಾರಣೆಗಳು ಮತ್ತು ಈ ಅಧ್ಯಾಯದಲ್ಲಿ ಚರ್ಚಿಸಲಾದ ಸ್ಟ್ರೀಮಬಲ್ HTTP ಸಾರಣೆಯನ್ನು ಗಮನಿಸಿ.

## ಸ್ಟ್ರೀಮಿಂಗ್: ಸಂದರ್ಭಗಳು ಮತ್ತು ಪ್ರೇರಣೆ

ಸ್ಟ್ರೀಮಿಂಗ್ ಹಿಂದಿನ ಪ್ರಮುಖ ತತ್ವಗಳು ಮತ್ತು ಪ್ರೇರಣೆಗಳನ್ನು ಅರ್ಥಮಾಡಿಕೊಳ್ಳುವುದು ಪರಿಣಾಮಕಾರಿ实时 ಸಂವಹನ ವ್ಯವಸ್ಥೆಗಳನ್ನು ಜಾರಿಗೊಳಿಸಲು ಮುಖ್ಯ.

**ಸ್ಟ್ರೀಮಿಂಗ್** ಎಂಬುದು ನೆಟ್ವರ್ಕ್ ಪ್ರೋಗ್ರಾಮಿಂಗ್‌ನಲ್ಲಿ ಒಂದು ತಂತ್ರಜ್ಞಾನ, ಇದರಿಂದ ಡೇಟಾವು ಸಣ್ಣ ಮತ್ತು manageable ತುಣುಕುಗಳಾಗಿ ಅಥವಾ ಘಟನೆಗಳ ಸರಣಿಯಾಗಿ ಕಳುಹಿಸಲಾಗುತ್ತದೆ, ಸಂಪೂರ್ಣ ಪ್ರತಿಕ್ರಿಯೆಗೆ ಕಾಯದೆ. ಇದು ವಿಶೇಷವಾಗಿ ಉಪಯುಕ್ತ:

- ದೊಡ್ಡ ಫೈಲ್‌ಗಳು ಅಥವಾ ದೊಡ್ಡ ಡೇಟಾಸೆಟ್‌ಗಳು.
-实时 ನವೀಕರಣಗಳು (ಉದಾ: ಚಾಟ್, ಪ್ರಗತಿಯ ಬಾಲುಗಳು).
- ದೀರ್ಘಕಾಲ ತಾಳುವ ಗಣನೆಗಳು, ಬಳಕೆದಾರರಿಗೆ ಮಾಹಿತಿ ನೀಡಬೇಕಾದ ಸಂದರ್ಭಗಳಲ್ಲಿ.

ಸ್ಟ್ರೀಮಿಂಗ್ ಕುರಿತು ನೀವು ತಿಳಿದುಕೊಳ್ಳಬೇಕಾದ ಮುಖ್ಯ ಅಂಶಗಳು ಹೀಗಿವೆ:

- ಡೇಟಾ ಹಂತೇ ಹಂತವಾಗಿ ತಲುಪುತ್ತದೆ, ಒಟ್ಟಾರೆ ಅಲ್ಲ.
- ಕ್ಲೈಂಟ್ ಡೇಟಾವನ್ನು ತಲುಪಿದಂತೆ ಪ್ರಕ್ರಿಯೆ ಮಾಡಬಹುದು.
- ತೋರುತ್ತಿರುವ ವಿಳಂಬವನ್ನು ಕಡಿಮೆ ಮಾಡಿ ಬಳಕೆದಾರ ಅನುಭವವನ್ನು ಸುಧಾರಿಸುತ್ತದೆ.

### ಸ್ಟ್ರೀಮಿಂಗ್ ಬಳಸುವ ಕಾರಣವೇನು?

ಸ್ಟ್ರೀಮಿಂಗ್ ಬಳಸುವುದಕ್ಕೆ ಪ್ರಮುಖ ಕಾರಣಗಳು ಇಲ್ಲಿವೆ:

- ಬಳಕೆದಾರರಿಗೆ ತಕ್ಷಣ ಪ್ರತಿಕ್ರಿಯೆ ಸಿಗುತ್ತದೆ, ಸುದೀರ್ಘ ಪ್ರಕ್ರಿಯೆಯ ಕೊನೆಯಲ್ಲಿ ಮಾತ್ರವಲ್ಲ.
-实时 ಅಪ್ಲಿಕೇಶನ್ ಗಳು ಮತ್ತು ಸ್ಪಂದನಶೀಲ UI ಗಳಿಗೆ ಅವಕಾಶ ನೀಡುತ್ತದೆ.
- ನೆಟ್ವರ್ಕ್ ಮತ್ತು ಗಣನಾ ಸ منابع ಗಳಿಗಾಗುವ ಕಡಿಮೆ ವ್ಯಯ.

### ಸರಳ ಉದಾಹರಣೆ: HTTP ಸ್ಟ್ರೀಮಿಂಗ್ ಸರ್ವರ್ ಮತ್ತು ಕ್ಲೈಂಟ್

ಸ್ಟ್ರೀಮಿಂಗ್ ಅನ್ನು ಹೇಗೆ ಜಾರಿಗೊಳಿಸುವುದೆಂದರೆ ಈ ಖಂಡಿತ ಉದಾಹರಣೆ:

#### ಪೈಥಾನ್

**ಸರ್ವರ್ (ಪೈಥಾನ್, FastAPI ಮತ್ತು StreamingResponse ಬಳಸಿ):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**ಕ್ಲೈಂಟ್ (ಪೈಥಾನ್, requests ಬಳಸಿ):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

ಈ ಉದಾಹರಣೆ ಸರ್ವರ್ ಒಂದೇ ಸಮಯದಲ್ಲಿ ಲಭ್ಯವಾಗುತ್ತಿರುವ ಸಂದೇಶಗಳ ಸರಣಿಯನ್ನು ಕ್ಲೈಂಟ್‌ಗೆ ಕಳುಹಿಸುತ್ತದೆ, ಎಲ್ಲಾ ಸಂದೇಶಗಳಿಗಾಗಿ ಕಾಯದೇ.

**ಈ ರೀತಿಯಲ್ಲಿ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ:**

- ಸರ್ವರ್ ಪ್ರತಿಯೊಂದು ಸಂದೇಶ ಯೋಗ್ಯ ಹೊತ್ತಿಗೆ ಹಂಚಿಕೊಳ್ಳುತ್ತದೆ.
- ಕ್ಲೈಂಟ್ ತಲುಪಿದಂತೆ ಪ್ರತಿ ತುಣುಕನ್ನು ಸ್ವೀಕರಿಸಿ ಮುದ್ರಿಸುತ್ತದೆ.

**ಅಗತ್ಯಗಳು:**

- ಸರ್ವರ್ ಸ್ಟ್ರೀಮ್ ಪ್ರತಿಕ್ರಿಯೆಯನ್ನು ಬಳಸಬೇಕು (ಉದಾ: FastAPI ನಲ್ಲಿ `StreamingResponse`).
- ಕ್ಲೈಂಟ್ ಪ್ರತಿಕ್ರಿಯೆಯನ್ನು ಸ್ಟ್ರೀಮ್ ಆಗಿ ಪ್ರಕ್ರಿಯೆ ಮಾಡಬೇಕು (`requests` ನಲ್ಲಿ `stream=True`).
- ವಿಷಯ-ಪ್ರಕಾರ ಸಾಮಾನ್ಯವಾಗಿ `text/event-stream` ಅಥವಾ `application/octet-stream` ಆಗಿರುತ್ತದೆ.

#### ಜಾವಾ

**ಸರ್ವರ್ (ಜಾವಾ, Spring Boot ಮತ್ತು Server-Sent Events ಬಳಸಿ):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**ಕ್ಲೈಂಟ್ (ಜಾವಾ, Spring WebFlux WebClient ಬಳಸಿ):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**ಜಾವಾ ಜಾರಿ ಟಿಪ್ಪಣಿಗಳು:**

- ಸ್ಟ್ರೀಮಿಂಗ್ಗಾಗಿ Spring Boot ರಿಯಾಕ್ಟಿವ್ ಸ್ಟ್ಯಾಂಕ್ `Flux` ಬಳಕೆ
- `ServerSentEvent` ಘಟನೆಯ ಪ್ರಕಾರಗಳೊಂದಿಗೆ ರಚನಾತ್ಮಕ ಘಟನೆ ಸ್ಟ್ರೀಮಿಂಗ್ ನೀಡುತ್ತದೆ
- `WebClient` ನಲ್ಲಿ `bodyToFlux()` ರಿಯಾಕ್ಟಿವ್ ಸ್ಟ್ರೀಮಿಂಗ್ ಸಹಾಯ
- `delayElements()` ಘಟನೆಗಳ ನಡುವೆ ಪ್ರಕ್ರಿಯೆ ಸಮಯವನ್ನು ಅನೂಕರಿಸುತ್ತದೆ
- ಕ್ಲೈಂಟ್ ಹ್ಯಾಂಡ್ಲಿಗ್ ಗೆ ಉತ್ತಮ ಜ್ಞಾನಕೊಡುವ ಘಟನಾ ಪ್ರಕಾರಗಳು (`info`, `result`)

### ಹೋಲಿಕೆ: ಕ್ಲಾಸಿಕ್ ಸ್ಟ್ರೀಮಿಂಗ್ ವಿ MCP ಸ್ಟ್ರೀಮಿಂಗ್

"ಕ್ಲಾಸಿಕ್" ರೀತಿಯ ಸ್ಟ್ರೀಮಿಂಗ್ ಮತ್ತು MCP ನಲ್ಲಿ ಸ್ಟ್ರೀಮಿಂಗ್ ಹೇಗೆ ವ್ಯತ್ಯಾಸವಿದೆ ಎಂಬುದನ್ನು ಇವು ಹೀಗೆ ವಿವರಿಸುತ್ತದೆ:

| ವೈಶಿಷ್ಟ್ಯ | ಕ್ಲಾಸಿಕ್ HTTP ಸ್ಟ್ರೀಮಿಂಗ್ | MCP ಸ್ಟ್ರೀಮಿಂಗ್ (ಸೂಚನೆಗಳು) |
|------------------------|-------------------------------|-------------------------------------|
| ಮುಖ್ಯ ಪ್ರತಿಕ್ರಿಯೆ | ತುಣುಕುಗಳಾಗಿ | ಒಟ್ಟয়ে, ಕೊನೆಯಲ್ಲಿ |
| ಪ್ರಗತಿ ನವೀಕರಣಗಳು | ಡೇಟಾ ತುಣುಕುಗಳಾಗಿ ಕಳುಹಿಸಲಾಗುತ್ತದೆ | ಸೂಚನೆಗಳಾಗಿ ಕಳುಹಿಸಲಾಗುತ್ತದೆ |
| ಕ್ಲೈಂಟ್ ಅಗತ್ಯಗಳು | ಸ್ಟ್ರೀಮ್ ಪ್ರಕ್ರಿಯೆ ಮಾಡಬೇಕು | ಸಂದೇಶ ಹ್ಯಾಂಡ್ಲರ್ ಸಹಿತ ಜಾರಿ ಮಾಡಬೇಕು |
| ಬಳಸುವ ಪ್ರಕರಣ | ದೊಡ್ಡ ಫೈಲ್‌ಗಳು, AI ಟೋಕನ್ ಸ್ಟ್ರೀಮ್ಸ್ | ಪ್ರಗತಿ, ಲಾಗ್,实时 ಪ್ರತಿಕ್ರಿಯೆ |

### ಪ್ರಮುಖ ಹೋಲಿಕೆಗಳ ವೀಕ್ಷಣೆ

ಇವು ಕೆಲ ಪ್ರಮುಖ ವ್ಯತ್ಯಾಸಗಳು:

- **ಸಂವಹನ ಮಾದರಿ:**
  - ಕ್ಲಾಸಿಕ್ HTTP ಸ್ಟ್ರೀಮಿಂಗ್: ಡೇಟಾವನ್ನು ತುಣುಕಾಗಿ ಕಳುಹಿಸಲು ಸರಳ ತುಣುಕು ಎನ್‌ಕೋಡಿಂಗ್ ಬಳಕೆ
  - MCP ಸ್ಟ್ರೀಮಿಂಗ್: JSON-RPC ಪ್ರೋಟೋಕಾಲ್ ಹೊಂದಿರುವ ರಚನಾತ್ಮಕ ಸೂಚನಾ ವ್ಯವಸ್ಥೆ

- **ಸಂದೇಶ ಫಾರ್ಮಾಟ್:**
  - ಕ್ಲಾಸಿಕ್ HTTP: ಹೊಸ ಸಾಲುಗಳೊಂದಿಗೆ ಸಾಗುವ ಸರಳ ಪಠ್ಯ ತುಣುಕುಗಳು
  - MCP: ಮೆಟಾಡೇಟಾ ಹೊಂದಿರುವ ರಚನಾತ್ಮಕ LoggingMessageNotification ವಸ್ತುಗಳು

- **ಕ್ಲೈಂಟ್ ಜಾರಿ ನಿರ್ವಹಣೆ:**
  - ಕ್ಲಾಸಿಕ್ HTTP: ಸರಳ ಸ್ಟ್ರೀಮ್ ಪ್ರಕ್ರಿಯೆ ಹೊಂದಿರುವ ಕ್ಲೈಂಟ್
  - MCP: ವಿಭಿನ್ನ ಸಂದೇಶ ಪ್ರಕಾರಗಳನ್ನು ಪ್ರಕ್ರಿಯೆ ಮಾಡುವ ಸಂದೇಶ ಹ್ಯಾಂಡ್ಲರ್ ಹೊಂದಿರುವ ಜಟಿಲ ಕ್ಲೈಂಟ್

- **ಪ್ರಗತಿ ನವೀಕರಣಗಳು:**
  - ಕ್ಲಾಸಿಕ್ HTTP: ಪ್ರಗತಿ ಮುಖ್ಯ ಪ್ರತಿಕ್ರಿಯೆ ಸ್ಟ್ರೀಮ್ ಭಾಗವಾಗಿದೆ
  - MCP: ಪ್ರಗತಿ ಪ್ರತ್ಯೇಕ ಸೂಚನಾ ಸಂದೇಶಗಳ ಮೂಲಕ ಕಳುಹಿಸಲಾಗುತ್ತದೆ, ಮುಖ್ಯ ಪ್ರತಿಕ್ರಿಯೆ ಕೊನೆಯಲ್ಲಿ ಬರುತ್ತದೆ

### ಶಿಫಾರಸುಗಳು

ಕ್ಲಾಸಿಕ್ ಸ್ಟ್ರೀಮಿಂಗ್ (`/stream` ಬಳಸಿ) ಮತ್ತು MCP ಮೂಲಕ ಸ್ಟ್ರೀಮಿಂಗ್ ಆಯ್ಕೆ ಮಾಡುವಾಗ ನಮ್ಮ ಶಿಫಾರಸುಗಳು ಇವು:

- **ಸರಳ ಸ್ಟ್ರೀಮಿಂಗ್ ಅಗತ್ಯಗಳಿಗೆ:** ಕ್ಲಾಸಿಕ್ HTTP ಸ್ಟ್ರೀಮಿಂಗ್ ಸರಳ ಮತ್ತು ಮೂಲ ಸ್ಟ್ರೀಮಿಂಗ್ ಅಗತ್ಯಗಳಿಗೆ ಸಾಕಾಗುತ್ತದೆ.

- **ಸಂಕೀರ್ಣ, ಇಂಟರ್ಯಾಕ್ಟಿವ್ ಅಪ್ಲಿಕೇಶನ್ ಗಾಗಿ:** MCP ಸ್ಟ್ರೀಮಿಂಗ್ ಹೆಚ್ಚಿನ ಮೆಟಾಡೇಟಾ ಮತ್ತು ಸೂಚನೆಗಳ ಮತ್ತು ಅಂತಿಮ ಫಲಿತಾಂಶಗಳ ವಿಭಜನೆಳ್ಳಿಯೊಂದಿಗೆ ರಚನಾತ್ಮಕ ವಿಧಾನ ಒದಗಿಸುತ್ತದೆ.

- **AI ಅಪ್ಲಿಕೇಶನ್ ಗಾಗಿ:** MCP ಸೂಚನಾ ವ್ಯವಸ್ಥೆ ದೀರ್ಘಕಾಲ ಉದ್ಯೋಗದಲ್ಲಿರುವ AI ಕಾರ್ಯಗಳ ಪ್ರಗತಿಯನ್ನು ಬಳಕೆದಾರರಿಗೆ ತಿಳಿಸಲು ಅತ್ಯಂತ ಉಪಯುಕ್ತ.

## MCP ನಲ್ಲಿ ಸ್ಟ್ರೀಮಿಂಗ್

ಹೌದು, ನೀವು ಈಗಾಗಲೇ ಕ್ಲಾಸಿಕ್ ಸ್ಟ್ರೀಮಿಂಗ್ ಮತ್ತು MCP ಸ್ಟ್ರೀಮಿಂಗ್ ನಡುವಿನ ತೌಲನೆಯನ್ನು ನೋಡಿ, ಶಿಫಾರಸುಗಳನ್ನು ಗಮನಿಸಿದಿರುತ್ತೀರಿ. ಈಗ MCP ನಲ್ಲಿ ಸ್ಟ್ರೀಮಿಂಗ್ ಮೆಲುಕುಹಾಕುವ ಪರಿಣಾಮಕಾರಿ ಮಾರ್ಗದ ಬಗ್ಗೆ ನಿರ್ದಿಷ್ಟವಾಗಿ ತಿಳಿಯೋಣ.

MCP ಫ್ರೇಮ್ವರ್ಕ್ ಒಳಗ ತಮಗೆ实时 ಪ್ರತಿಕ್ರಿಯೆ ನೀಡುವ ಸ್ಪಂದನಶೀಲ ಅಪ್ಲಿಕೇಶನ್ ನಿರ್ಮಾಣಕ್ಕೆ ಸ್ಟ್ರೀಮಿಂಗ್ ಹೇಗೆ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ ಎಂಬುದನ್ನು ಅರ್ಥಮಾಡಿಕೊಳ್ಳುವುದು ಮುಖ್ಯ.

MCP ನಲ್ಲಿ ಸ್ಟ್ರೀಮಿಂಗ್ ಮುಖ್ಯ ಪ್ರತಿಕ್ರಿಯೆಯನ್ನು ತುಣುಕುಗಳಲ್ಲಿ ಕಳುಹಿಸುವುದಲ್ಲ, ಬದಲಾಗಿ ಒಂದು ಸಾಧನವು ವಿನಂತಿಯನ್ನು ಪ್ರಕ್ರಿಯೆ ಮಾಡುತ್ತಿರುವಾಗ ಕ್ಲೈಂಟ್‌ಗೆ **ಸೂಚನೆಗಳನ್ನು** ಕಳುಹಿಸುವ ಬಗ್ಗೆ. ಈ ಸೂಚನೆಗಳು ಪ್ರಗತಿ ನವೀಕರಣಗಳು, ಲಾಗ್ ಅಥವಾ ಇತರ ಘಟನೆಗಳನ್ನು ಒಳಗೊಂಡಿರಬಹುದು.

### ಕಾರ್ಯವಿಧಾನ

ಮುಖ್ಯ ಫಲಿತಾಂಶವನ್ನು ಇನ್ನೂ ಒಂದೇ ತಿರುವಿನಲ್ಲಿ ಕಳುಹಿಸಲಾಗುತ್ತದೆ. ಆದಾಗ್ಯೂ, ಪ್ರಕ್ರಿಯೆಯಲ್ಲಿರುವಾಗ ಪ್ರತ್ಯೇಕ ಸಂದೇಶಗಳಾಗಿ ಸೂಚನೆಗಳು ಕಳುಹಿಸಬಹುದು ಮತ್ತು dadurch实时 ಕ್ಲೈಂಟ್ ಮಾಹಿತಿಯನ್ನು ನವೀಕರಿಸುತ್ತವೆ. ಕ್ಲೈಂಟ್ ಈ ಸೂಚನೆಗಳನ್ನು ಹ್ಯಾಂಡ್ಲ್ ಮಾಡಿ ಪ್ರದರ್ಶಿಸಲು ಆಗಬೇಕು.

## ಸೂಚನೆ ಎಂದರೆ ಏನು?

"ಸೂಚನೆ" ಎಂದರೆ MCP ಅವಕಾಶದಲ್ಲಿ ಏನು ಅರ್ಥ?

ಸೂಚನೆ ಎಂದರೆ ಸರ್ವರ್‌ನಿಂದ ಕ್ಲೈಂಟ್‌ಗೆ ಕಳುಹಿಸಲಾದ ಸಂದೇಶ, ದೀರ್ಘಕಾಲದ ವ್ಯಾಪಕ ಕಾರ್ಯಾಚರಣೆಯ ವೇಳೆ ಪ್ರಗತಿ, ಸ್ಥಿತಿ ಅಥವಾ ಇತರ ಘಟನೆಗಳ ಬಗ್ಗೆ ಮಾಹಿತಿ ನೀಡುತ್ತದೆ. ಸೂಚನೆಗಳು ಸುವಿವರಣ ಮತ್ತು ಬಳಕೆದಾರ ಅನುಭವವನ್ನು ಸುಧಾರಿಸುತ್ತವೆ.

ಉದಾಹರಣೆಗೆ, ಸರ್ವರ್‌ಗಾಗಿ ಶುರುವಾಗುವ ಹ್ಯಾಂಡ್‌ಶೇಕ್ ಮಾಡಿದ ಮೇಲೆ ಕ್ಲೈಂಟ್ ಒಂದು ಸೂಚನೆಯನ್ನು ಕಳುಹಿಸಬೇಕಾಗುತ್ತದೆ.

ಸೂಚನೆ JSON ಸಂದೇಶವಾಗಿ ಹೀಗಿರಬಹುದು:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

ಸೂಚನೆಗಳು MCP ಯಲ್ಲಿ ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) ಎಂದು ಹೇಳುವ ವಿಷಯಕ್ಕೆ ಸೇರಿವೆ.

ಲಾಗಿಂಗ್ ಅನ್ನು ಕಾರ್ಯಗತಗೊಳಿಸಲು, ಸರ್ವರ್ ಅದನ್ನು ಈ ರೀತಿ ವೈಶಿಷ್ಟ್ಯ/ ಸಾಮರ್ಥ್ಯವಾಗಿ ಸಕ್ರಿಯಗೊಳಿಸಬೇಕು:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> ಬಳಸುವ SDK ನಿರ್ಣಯದ ಮೇಲೆ, ಲಾಗಿಂಗ್ ಡೀಫಾಲ್ಟ್ ಆಗಿಯೇ ಸಕ್ರಿಯವಾಗಿರಬಹುದು, ಇಲ್ಲವಾದರೆ ನಿಮ್ಮ ಸರ್ವರ್ ಸಂರಚನೆಯಲ್ಲಿ ಸ್ಪಷ್ಟವಾಗಿ ಸಕ್ರಿಯಗೊಳಿಸಬೇಕಾಗಬಹುದು.

ವಿಭಿನ್ನ ವಿಧದ ಸೂಚನೆಗಳಿವೆ:

| ಮಟ್ಟ | ವಿವರಣೆ | ಉದಾಹರಣೆಯ ಬಳಕೆ ಪ್ರಕರಣ |
|-----------|-------------------------------|---------------------------------|
| ಡಿಬಗ್ | ಸಂಗ್ರಹಣೀಯ ಡೀಟೈಲ್ ಮಾಹಿತಿ | ಫಂಕ್ಷನ್ ಪ್ರವೇಶ/ನಿರ್ಗಮನ ಹ೦ದರಿಕೆಗಳು |
| ಮಾಹಿತಿ | ಸಾಮಾನ್ಯ ಮಾಹಿತಿ ಸಂದೇಶಗಳು | ಕಾರ್ಯಾಚರಣೆ ಪ್ರಗತಿ ನವೀಕರಣಗಳು |
| ಸೂಚನೆ | ಸಾಮಾನ್ಯವೂ ಆದ ಪುಸ್ತಕಪೂರ್ಣ ಘಟನೆಗಳು | ಸಂರಚನೆ ಬದಲಾವಣೆಗಳು |
| ಎಚ್ಚರಿಕೆ | ಎಚ್ಚರಿಕೆ ಸ್ಥಿತಿಗಳು | ನಿಷೇಧಿತ ವೈಶಿಷ್ಟ್ಯ ಬಳಕೆ |
| ದೋಷ | ದೋಷ ಸ್ಥಿತಿಗಳು | ಕಾರ್ಯಾಚರಣೆ ವೈಫಲ್ಯಗಳು |
| ಸಂಕೀರ್ಣ | ಗಂಭೀರ ಸ್ಥಿತಿಗಳು | ವ್ಯವಸ್ಥಾ ಘಟಕ ವೈಫಲ್ಯಗಳು |
|ಜಾಗೃತಿ|ತಕ್ಷಣ ಕ್ರಮ ತೆಗೆದುಕೊಳ್ಳಬೇಕು|ಡೇಟಾ ಕಾಲಿತಾಪನೆ ಕಂಡುಬಂದಿದೆ|
| ತುರ್ತು | ವ್ಯವಸ್ಥೆ ಉಪಯೋಗಕ್ಕೆ ಬಾರದಂತಾಗಿದೆ | ಸಂಪೂರ್ಣ ವ್ಯವಸ್ಥೆ ವೈಫಲ್ಯ |

## MCP ನಲ್ಲಿ ಸೂಚನೆಗಳನ್ನು ಜಾರಿಗೊಳಿಸುವುದು

MCP ನಲ್ಲಿ ಸೂಚನೆಗಳನ್ನು ಜಾರಿಗೊಳಿಸಲು, ನೀವು ಸರ್ವರ್ ಮತ್ತು ಕ್ಲೈಂಟ್ ಎರಡರನ್ನೂ实时 ನವೀಕರಣಗಳನ್ನು ಹ್ಯಾಂಡ್ಲ್ ಮಾಡಲು ಹೊಂದಿಸಬೇಕು. ಇದು ನಿಮ್ಮ ಅಪ್ಲಿಕೇಶನ್ ಅನ್ನು ದೀರ್ಘಕಾಲದ ಕಾರ್ಯಗಳ ವೇಳೆ ತಕ್ಷಣದ ಪ್ರತಿಕ್ರಿಯೆ ನೀಡುವಂತೆ ಮಾಡುತ್ತದೆ.

### ಸರ್ವರ್ ಬದಿಯ: ಸೂಚನೆಗಳನ್ನು ಕಳುಹಿಸುವುದು

ಸರ್ವರ್ ಬದಿಯನ್ನಿಂದ ಪ್ರಾರಂಭಿಸೋಣ. MCP ಯಲ್ಲಿ, ನೀವು ವಿನಂತಿಗಳನ್ನು ಪ್ರಕ್ರಿಯೆ ಮಾಡುವಾಗ ಸೂಚನೆಗಳನ್ನು ಕಳುಹಿಸುವ ಸಾಧನಗಳನ್ನು ನಿಶ್ಚಯಿಸುತ್ತೀರಿ. ಸರ್ವರ್ ಸಾಮಾನ್ಯವಾಗಿ `ctx` ಎಂದು ಕರೆಯಲ್ಪಡುವ ಕಾಂಟೆಕ್ಸ್ಟ್ ವಸ್ತುವನ್ನು ಬಳಸಿ ಕ್ಲೈಂಟ್‌ಗೆ ಸಂದೇಶಗಳನ್ನು ಕಳುಹಿಸುತ್ತದೆ.

#### ಪೈಥಾನ್

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

ಮೊದಲು ನೀಡಲಾದ ಉದಾಹರಣೆಯಲ್ಲಿ, `process_files` ಸಾಧನವು ಪ್ರತಿಯೊಂದು ಫೈಲ್ ಅನ್ನು ಪ್ರಕ್ರಿಯೆಮಾಡುವಾಗ ಮೂರು ಸೂಚನೆಗಳನ್ನು ಕ್ಲೈಂಟ್‌ಗೆ ಕಳುಹಿಸುತ್ತದೆ. `ctx.info()` ಪದ್ಧತಿಯನ್ನು ಮಾಹಿತಿ ಸಂದೇಶಗಳನ್ನು ಕಳುಹಿಸಲು ಬಳಸಲಾಗಿದೆ.

ಜೊತೆಗೆ, ಸೂಚನೆಗಳನ್ನು ಸಕ್ರಿಯಗೊಳಿಸಲು, ನಿಮ್ಮ ಸರ್ವರ್ Streaming HTTP (ಉದಾ: `streamable-http`) ಸಾರಣೆಯನ್ನಾಡಬೇಕು ಮತ್ತು ನಿಮ್ಮ ಕ್ಲೈಂಟ್ ಸೂಚನೆಗಳನ್ನು ಪ್ರಕ್ರಿಯೆ ಮಾಡಲು ಸಂದೇಶ ಹ್ಯಾಂಡ್ಲರ್ ಅನ್ನು ಜಾರಿ ಮಾಡಿರಬೇಕು. ಸರ್ವರ್ ಅವರನ್ನು `streamable-http` ಸಾರಣೆಗಾಗಿ ಹೇಗೆ ಸಿದ್ಧಪಡಿಸುವುದು ಹೀಗಾಗಿರುತ್ತದೆ:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

ಈ .NET ಉದಾಹರಣೆಯಲ್ಲಿ, `ProcessFiles` ಸಾಧನವು `Tool` ಅಟ್ರಿಬ್ಯೂಟ್‌ನೊಂದಿಗೆ ಅಲಂಕರಿಸಲಾಗಿದೆ ಮತ್ತು ಪ್ರತಿ ಫೈಲ್ ಪ್ರಕ್ರಿಯೆ ಮಾಡುವಾಗ ಮೂರು ಸೂಚನೆಗಳನ್ನು ಕ್ಲೈಂಟ್‌ಗೆ ಕಳುಹಿಸುತ್ತದೆ. `ctx.Info()` ವಿಧಾನ ಮಾಹಿತಿ ಸಂದೇಶಗಳನ್ನು ಕಳುಹಿಸಲು ಉಪಯೋಗವಾಗಿದೆ.

ನಿಮ್ಮ .NET MCP ಸರ್ವರ್‌ನಲ್ಲಿ ಸೂಚನೆಗಳನ್ನು ಸಕ್ರಿಯಗೊಳಿಸಲು, Streaming HTTP ಸಾರಣೆಯನ್ನು ಬಳಸುವುದು ಖಚಿತಪಡಿಸಿಕೊಳ್ಳಿ:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### ಕ್ಲೈಂಟ್ ಬದಿಯ: ಸೂಚನೆಗಳನ್ನು ಸ್ವೀಕರಿಸುವುದು

ಕ್ಲೈಂಟ್ ಬದಿ ಸೂಚನೆಗಳನ್ನು ಸ್ವೀಕರಿಸಿ ಪ್ರದರ್ಶಿಸಲು ಸಂದೇಶ ಹ್ಯಾಂಡ್ಲರ್ ಅನ್ನು ಜಾರಿಗೆ ತರುವ ಅಗತ್ಯವಿದೆ.

#### ಪೈಥಾನ್

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

ಮೇಲಿನ ಕೋಡ್‌ನಲ್ಲಿ, `message_handler` ಕಾರ್ಯ incoming message ಸೂಚನೆ ಆಗಿದೆಯೇ ಎಂದು ಪರಿಶೀಲಿಸುತ್ತದೆ. ಆಗಿದ್ದರೆ, ಅದು ಸೂಚನೆಯನ್ನು ಮುದ್ರಿಸುತ್ತದೆ; ಇಲ್ಲವೆ, ಅದು ಸಾಮಾನ್ಯ ಸರ್ವರ್ ಸಂದೇಶ ಎಂದು ಪ್ರಕ್ರಿಯೆಮಾಡುತ್ತದೆ. ಜೊತೆಯಾಗಿ, `ClientSession` ಆರಂಭಿಕಗೊಂಡಿದ್ದು `message_handler` ಮೂಲಕ ಸೂಚನೆಗಳನ್ನು ಹ್ಯಾಂಡ್ಲ್ ಮಾಡುತ್ತದೆ.

#### .NET


ಈ .NET ಉದಾಹರಣೆಯಲ್ಲಿ, `MessageHandler` incoming message ಸೂಚನೆ ಆಗಿದೆಯೇ ಎಂದು ಪರಿಶೀಲಿಸುತ್ತದೆ. ಆಗಿದ್ದರೆ, ಸೂಚನೆಯನ್ನು ಮುದ್ರಿಸುತ್ತದೆ; ಇಲ್ಲವೆ, ಸಾಮಾನ್ಯ ಸರ್ವರ್ ಸಂದೇಶ ಎಂದು ಪ್ರಕ್ರಿಯೆಮಾಡುತ್ತದೆ. `ClientSession` ಅನ್ನು `ClientSessionOptions` ಮೂಲಕ ಸಂದೇಶ ಹ್ಯಾಂಡ್ಲರ್ ಜೊತೆಗೆ ಆರಂಭಿಸಲಾಗುತ್ತದೆ.

ಸೂಚನೆಗಳನ್ನು ಸಕ್ರಿಯಗೊಳಿಸಲು, ನಿಮ್ಮ ಸರ್ವರ್ Streaming HTTP (ಉದಾ: `streamable-http`) ಸಾರಣೆಯನ್ನು ಬಳಸುತ್ತಿರುವುದು ಖಚಿತಪಡಿಸಿ ಮತ್ತು ಕ್ಲೈಂಟ್ ಸಂದೇಶ ಹ್ಯಾಂಡ್ಲರ್ ಅನ್ನು ಜಾರಿಗೆ ತರಬೇಕು.

## ಪ್ರಗತಿ ಸೂಚನೆಗಳು ಹಾಗೂ ಸಂದರ್ಭಗಳು

ಈ ವಿಭಾಗವು MCP ನಲ್ಲಿ ಪ್ರಗತಿ ಸೂಚನೆ ಎಂಬ ತತ್ತ್ವ, ಅವು ಪ್ರಾಮುಖ್ಯತೆ ಮತ್ತು Streamable HTTP ಬಳಸಿಕೊಂಡು ಅವುಗಳನ್ನು ಹೇಗೆ ಜಾರಿಗೊಳಿಸುವುದು ಎಂಬುದನ್ನು ವಿವರಿಸುತ್ತದೆ. ಜೊತೆಯಾಗಿ, ನಿಮ್ಮ ತಿಳುವಳಿಕೆಯನ್ನು ಬಲಪಡಿಸಲು ಒಂದು ಪ್ರಾಯೋಗಿಕ ಕಾರ್ಯವನ್ನು ಒದಗಿಸುತ್ತದೆ.

ಪ್ರಗತಿ ಸೂಚನೆಗಳು ದೀರ್ಘಕಾಲದ ಕಾರ್ಯಾಚರಣೆಗಳ ವೇಳೆ ಸರ್ವರ್‌ನಿಂದ ಕ್ಲೈಂಟ್‌ಗೆ实时 ಕಳುಹಿಸಲಾದ ಸಂದೇಶಗಳಾಗಿವೆ. ಸಂಪೂರ್ಣ ಕೆಲಸ ಮುಗಿಯುವ ತನಕ ಕಾಯದೆ, ಸರ್ವರ್ ಪ್ರಸ್ತುತ ಸ್ಥಿತಿಯನ್ನು ಕ್ಲೈಂಟ್‌ಗೆ ನವೀಕರಿಸುತ್ತಿರುತ್ತದೆ. ಇದು ಉತ್ಪಾದಕತೆ, ಬಳಕೆದಾರ ಅನುಭವ ಮತ್ತು ಡೀಬಗಿಂಗ್ ಸುಲಭಗೊಳಿಸುತ್ತದೆ.

**ಉದಾಹರಣೆ:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### ಏಕೆ ಪ್ರಗತಿ ಸೂಚನೆಗಳನ್ನು ಬಳಸಿ?

ಪ್ರಗತಿ ಸೂಚನೆಗಳು ಕೆಲ ಮುಖ್ಯ ಕಾರಣಗಳಿಂದ ಅವಶ್ಯಕ:

- **ಉತ್ತಮ ಬಳಕೆದಾರ ಅನುಭವ:** ಕೆಲಸ ಮುಂದುವರೇನು ಎಂದು ಬಳಕೆದಾರರು ನವೀಕರಣಗಳನ್ನು ಕಾಣುತ್ತಾರೆ, ಕೊನೆಯಲ್ಲಿ ಮಾತ್ರವಲ್ಲ.
- **实时 ಪ್ರತಿಕ್ರಿಯೆ:** ಕ್ಲೈಂಟ್‌ಗಳು ಪ್ರಗತಿ ಬಾರ್ ಅಥವಾ ಲಾಗ್‌ಗಳನ್ನು ಪ್ರದರ್ಶಿಸಬಹುದು, ಅಪ್ಲಿಕೇಶನ್ ಸ್ಪಂದನಾತ್ಮಕವಾಗುತ್ತದೆ.
- **ಸರಳ ಡೀಬಗಿಂಗ್ ಮತ್ತು ಮೇಲ್ವಿಚಾರಣಾ:** ಡೆವಲಪರ್‌ಗಳು ಮತ್ತು ಬಳಕೆದಾರರು ಪ್ರಕ್ರಿಯೆ ಎಲ್ಲಿಗೆ ನಿಧಾನವಾಗಿದೆ ಅಥವಾ ಅಡಚಣೆ ಇದೆ ಎಂದು ನೋಡಬಹುದು.

### ಪ್ರಗತಿ ಸೂಚನೆಗಳನ್ನು ಹೇಗೆ ಜಾರಿಗೊಳಿಸುವುದು

ಇವು MCP ನಲ್ಲಿ ಪ್ರಗತಿ ಸೂಚನೆಗಳನ್ನು ಜಾರಿಗೊಳಿಸುವ ವಿಧಾನ:

- **ಸರ್ವರ್ ಬದಿ:** ಪ್ರತಿ ಐಟಂ ಪ್ರಕ್ರಿಯೆ ಆಗುತ್ತಿದ್ದಂತೆ `ctx.info()` ಅಥವಾ `ctx.log()` ಬಳಸಿ ಸೂಚನೆಗಳನ್ನು ಕಳುಹಿಸು. ಇದು ಪ್ರಮುಖ ಫಲಿತಾಂಶ ಸಿದ್ಧವಾಗುವ ಮುಂಚೆ ಕ್ಲೈಂಟ್‌ಗೆ ಸಂದೇಶ ನೀಡುತ್ತದೆ.
- **ಕ್ಲೈಂಟ್ ಬದಿ:** ಸೂಚನೆಗಳನ್ನು ಕೆಯಲ್ಲಿ ಇಟ್ಟು, ಅವುಗಳನ್ನು ಪ್ರದರ್ಶಿಸುವ ಸಂದೇಶ ಹ್ಯಾಂಡ್ಲರ್ ಅನ್ನು ಜಾರಿಗೆ ತರು. ಈ ಹ್ಯಾಂಡ್ಲರ್ ಸೂಚನೆಗಳನ್ನು ಮತ್ತು ಅಂತಿಮ ಫಲಿತಾಂಶವನ್ನು ವಿಭಜಿಸುತ್ತದೆ.

**ಸರ್ವರ್ ಉದಾಹರಣೆ:**

#### ಪೈಥಾನ್

```python
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**ಕ್ಲೈಂಟ್ ಉದಾಹರಣೆ:**

#### ಪೈಥಾನ್

```python

        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## ಸುರಕ್ಷತೆ ಪರಿಗಣನೆಗಳು

HTTP ಆಧಾರಿತ ಸಾರಣೆಗಳೊಂದಿಗೆ MCP ಸರ್ವರ್‌ಗಳನ್ನು ಜಾರಿಗೊಳಿಸುವಾಗ, ಸುರಕ್ಷತೆ ಬಹುಮುಖ್ಯವಾದ ವಿಚಾರವಾಗುತ್ತದೆ, ಇದಕ್ಕೆ ಬಹುಮುಖ ಹಲ್ಲುಗಳ ಮಾರ್ಗ ಮತ್ತು ರಕ್ಷಣೆ ಮೆಕ್ಯಾನಿಸಮ್‌ಗಳ ಮೇಲೆ ಜಾಗೃತಿ ಅಗತ್ಯ.

### ಅವಲೋಕನ

MCP ಸರ್ವರ್‌ಗಳನ್ನು HTTP ಮೂಲಕ ಹೊರತರಲು ಸುರಕ್ಷತೆ ಅತಿ ಮುಖ್ಯ. ಸ್ಟ್ರೀಮಬಲ್ HTTP ಹೊಸ ಹಲ್ಲು ಹಿನ್ನೆಲೆಗಳನ್ನು ಪರಿಚಯಿಸುತ್ತದೆ ಮತ್ತು ನಿರ್ದಿಷ್ಟ ಸಂರಚನೆ ಅಗತ್ಯ.

### ಪ್ರಮುಖ ಅಂಶಗಳು


- **Origin Header Validation**: DNS ರಿಬೈಂಡಿಂಗ್ ದಾಳಿಗಳನ್ನು ತಡೆಯಲು `Origin` ಹೆಡರ್ ಅನ್ನು תמיד सत्यापित ಮಾಡಿ.
- **Localhost Binding**: ಸ್ಥಳೀಯ ಅಭಿವೃದ್ಧಿಗಾಗಿ, ಸರ್ವರ್‌ಗಳನ್ನು `localhost` ಗೆ ಬಾಂಧಿಸಿ, ಅವುಗಳನ್ನು ಸಾರ್ವಜನಿಕ ಇಂಟರ್ನೆಟ್‌ಗೆ ಅನಾವರಣಗೊಳಿಸುವುದನ್ನು ತಪ್ಪಿಸಲು.
- **Authentication**: ಉತ್ಪಾದನಾ ವಿನ್ಯಾಸಗಳಿಗೆ ಪ್ರমাণೀಕರಣವನ್ನು (ಉದಾ., API ಕೀಗಳು, OAuth) ಅನುಷ್ಠಾನಗೊಳಿಸಿ.
- **CORS**: ಪ್ರವೇಶವನ್ನು ನಿಯಂತ್ರಿಸಲು ಕ್ರಾಸ್-ಓರಿಜಿನ್ ರೆಸೋರ್ಸ್ ಶೇಯರಿಂಗ್ (CORS) ನೀತಿಗಳನ್ನು ಸಂರಚಿಸಿ.
- **HTTPS**: ಸಂಚಾರವನ್ನು ಗುಪ್ತಗೊಳಿಸಲು ಉತ್ಪಾದನೆಯಲ್ಲಿ HTTPS ಅನ್ನು ಬಳಸಿರಿ.

### ಉತ್ತಮ ಅಭ್ಯಾಸಗಳು

- ತಪ್ಪದೆ ಮಾನ್ಯತೆ ಇಲ್ಲದೆ ಬರುವ ವಿನಂತಿಗಳನ್ನು ನಂಬಬೇಡಿ.
- ಎಲ್ಲಾ ಪ್ರವೇಶ ಮತ್ತು ದೋಷಗಳನ್ನು ನವದೇಶಿಸಿ ಮತ್ತು ಮೇಲ್ವಿಚಾರಣೆ ಮಾಡಿ.
- ಭದ್ರತೆ ರಂಧ್ರಗಳನ್ನು ತುಂಬಿಸಲು ಆಧಾರಗಳನ್ನು ನಿಯಮಿತವಾಗಿ ನವೀಕರಿಸಿ.

### ಸವಾಲುಗಳು

- ಅಭಿವೃದ್ಧಿಗೆ ಸುಲಭತೆಯೊಂದಿಗೆ ಭದ್ರತೆಯ ಸಮತೋಲನ.
- ವಿವಿಧ ಕ್ಲೈಂಟ್ ಪರಿಸರಗಳೊಂದಿಗೆ ಹೊಂದಿಕೆಯನ್ನು ಖಚಿತಪಡಿಸುವುದು.

## SSE ಇಂದ Streamable HTTP ಗೆ ಅಪ್ಗ್ರೇಡ್ ಮಾಡುವುದು

ಪ್ರಸ್ತುತ Server-Sent Events (SSE) ಬಳಸುತ್ತಿರುವ ಅಪ್ಲಿಕೇಶನ್‌ಗಳಿಗೆ, Streamable HTTP ಗೆ ಸ್ಥಳಾಂತರಿಸುವುದು ನಿಮ್ಮ MCP ಅನುಷ್ಠಾನಗಳಿಗೆ ಉತ್ತಮ ಸಾಮರ್ಥ್ಯ ಮತ್ತು ದೀರ್ಘಕಾಲಿಕ ಸ್ಥಿರತೆಯನ್ನು ಒದಗಿಸುತ್ತದೆ.

### ಏಕೆ ಅಪ್ಗ್ರೇಡ್ ಮಾಡಬೇಕು?

SSE ನಿಂದ Streamable HTTP ಗೆ ಅಪ್ಗ್ರೇಡ್ ಮಾಡುವ ಎರಡು ಪ್ರಮುಖ ಕಾರಣಗಳಿವೆ:

- Streamable HTTP SSE ಗಿಂತ ಉತ್ತಮ ಸ್ಕೇಲಬಿಲಿಟಿ, ಹೊಂದಿಕೆ ಮತ್ತು ಶ್ರೀಮಂತ ಸೂಚನೆ ಬೆಂಬಲವನ್ನು ನೀಡುತ್ತದೆ.
- ಇದು ಹೊಸ MCP ಅಪ್ಲಿಕೇಶನ್‌ಗಳಿಗೆ ಶಿಫಾರಸು ಮಾಡಿದ ಸಂಚಾರವಾಗಿದೆ.

### ಸ್ಥಳಾಂತರಿಸುವ ಹಂತಗಳು

ನಿಮ್ಮ MCP ಅಪ್ಲಿಕೇಶನ್‌ಗಳಲ್ಲಿ SSE ನಿಂದ Streamable HTTP ಗೆ ಹೇಗೆ ಸ್ಥಳಾಂತರಿಸುವದು ಇಲ್ಲಿದೆ:

- **ಸರ್ವರ್ ಕೋಡ್ ಅನ್ನು ನವೀಕರಿಸಿ** `mcp.run()` ನಲ್ಲಿ `transport="streamable-http"` ಬಳಸಲು.
- **ಕ್ಲೈಂಟ್ ಕೋಡ್ ಅನ್ನು ನವೀಕರಿಸಿ** SSE ಕ್ಲೈಂಟ್ ಬದಲಿಗೆ `streamablehttp_client` ಬಳಸಲು.
- **ಕ್ಲೈಂಟ್‌ನಲ್ಲಿ ಸಂದೇಶ ಹ್ಯಾಂಡ್ಲರ್ ಅನ್ನು ಅನುಷ್ಠಾನಗೊಳಿಸಿ** ಸೂಚನೆಗಳನ್ನು ಪ್ರಕ್ರಿಯೆ ಮಾಡಲು.
- **ಹೊಂದಿಕೆಯನ್ನು ಪರೀಕ್ಷಿಸಿ** ಇತ್ತೀಚಿನ ಸಾಧನಗಳು ಮತ್ತು ಕಾರ್ಯಸರಣಿಗಳೊಂದಿಗೆ.

### ಹೊಂದಿಕೆಯನ್ನು ಕಾಪಾಡಿಕೊಳ್ಳುವದು

ಸ್ಥಳಾಂತರಿಸುವಾಗ ಪ್ರಸ್ತುತ SSE 클라이ಂಟ್‌ಗಳೊಂದಿಗೆ ಹೊಂದಿಕೆಯನ್ನು ಕಾಪಾಡಿಕೊಳ್ಳುವುದು ಶಿಫಾರಸು ಮಾಡಲ್ಪಟ್ಟಿದೆ. ಕೆಲವು ತಂತ್ರಗಳನ್ನು ಇಲ್ಲಿ ನೀಡಲಾಗಿದೆ:

- ನೀವು ವಿಭಿನ್ನ Endpoints ನಲ್ಲಿ SSE ಮತ್ತು Streamable HTTP ಎರಡನ್ನೂ ಚಾಲನೆ ಮಾಡುವ ಮೂಲಕ ಎರಡನ್ನೂ ಬೆಂಬಲಿಸಬಹುದು.
- ಗ್ರಾಜುವಲ್ ಆಗಿ ಕ್ಲೈಂಟ್‌ಗಳನ್ನು ಹೊಸ ಸಂಚಾರಕ್ಕೆ ಸ್ಥಳಾಂತರಿಸಿ.

### ಸವಾಲುಗಳು

ಸ್ಥಳಾಂತರಿಸುವಾಗ ಈ ಸವಾಲುಗಳನ್ನು ನೀವು ಎದುರಿಸಬೇಕು:

- ಎಲ್ಲಾ ಕ್ಲೈಂಟ್‌ಗಳು ನವೀಕರಿಸಲ್ಪಟ್ಟಿದ್ದಾರೆ ಎಂದು ಖಚಿತಪಡಿಸುವುದು
- ಸೂಚನೆಗಳ ವಿತರಣೆಯಲ್ಲಿ ವ್ಯತ್ಯಾಸಗಳನ್ನು ನಿರ್ವಹಿಸುವುದು

## ಭದ್ರತಾ ವಿಚಾರಣೆಗಳು

ಯಾವುದೇ ಸರ್ವರ್ ಅನ್ನು ಅನುಷ್ಠಾನಗೊಳಿಸುವಾಗ ಭದ್ರತೆ ಮಹತ್ವದ ಆದ್ಯತೆ ಆಗಿರುತ್ತದೆ, ವಿಶೇಷವಾಗಿ MCP ನಲ್ಲಿ Streamable HTTP പോലുള്ള HTTP ಆಧಾರಿತ ಸಂಚಾರಗಳನ್ನು ಬಳಸುವಾಗ.

MCP ಸರ್ವರ್‌ಗಳನ್ನು HTTP ಆಧಾರಿತ ಸಂಚಾರಗಳೊಂದಿಗೆ ಅನುಷ್ಠಾನಗೊಳಿಸುವಾಗ, ಹಲವು ದಾಳಿಯ ಮಾರ್ಗಗಳು ಮತ್ತು ರಕ್ಷಣಾ ತಂತ್ರಗಳನ್ನು ಜಾಗ್ರತೆ ಮತ್ತು ನಿಯಮಿತ ಗಮನ ನೀಡಬೇಕಾಗುತ್ತದೆ.

### ಅವಲೋಕನ

MCP ಸರ್ವರ್‌ಗಳನ್ನು HTTP ಮೂಲಕ ಅನಾವರಣ ಮಾಡುವಾಗ ಭದ್ರತೆ ಪ್ರಮುಖವಾಗುತ್ತದೆ. Streamable HTTP ಹೊಸ ದಾಳಿ ಪರಿಧಿಗಳನ್ನು ಪರಿಚಯಿಸುತ್ತದೆ ಮತ್ತು ಸೂಕ್ತ ಸಂರಚನೆ ಅಗತ್ಯವಿದೆ.

ಇಲ್ಲಿ ಕೆಲವು ಪ್ರಮುಖ ಭದ್ರತಾ ವಿಚಾರಣೆಗಳಿವೆ:

- **Origin Header Validation**: DNS ರಿಬೈಂಡಿಂಗ್ ದಾಳಿಗಳನ್ನು ತಡೆಯಲು `Origin` ಹೆಡರ್ ಅನ್ನು ಸದಾ ಪರಿಶೀಲಿಸಿ.
- **Localhost Binding**: ಸ್ಥಳೀಯ ಅಭಿವೃದ್ಧಿಗಾಗಿ, ಸರ್ವರ್‌ಗಳನ್ನು ಸಾರ್ವಜನಿಕ ಇಂಟರ್ನೆಟ್ ಗೆ ಅನಾವರಣಗೊಳಿಸದಂತೆ `localhost` ಗೆ ಬಾಂಧಿಸಿ.
- **Authentication**: ಉತ್ಪಾದನಾ ವಿನ್ಯಾಸಗಳಿಗೆ ಪ್ರমাণೀಕರಣವನ್ನು (ಆದರ್ಶ: API ಕೀಸ್, OAuth) ಅನುಷ್ಠಾನಗೊಳಿಸಿ.
- **CORS**: ಕ್ರಾಸ್-ಓರಿಜಿನ್ ರೆಸೋರ್ಸ್ ಶೇಯರಿಂಗ್ (CORS) ನೀತಿಗಳನ್ನು ಪ್ರವೇಶ ನಿಯಂತ್ರಣಕ್ಕಾಗಿ ಸಂರಚಿಸಿ.
- **HTTPS**: ಸಂಚಾರದ ಗುಪ್ತೀಕರಣಕ್ಕಾಗಿ ಉತ್ಪಾದನೆಯಲ್ಲಿ HTTPS ಬಳಸಿ.

### ಉತ್ತಮ ಅಭ್ಯಾಸಗಳು

ನಿಮ್ಮ MCP ಸ್ಟ್ರೀಮಿಂಗ್ ಸರ್ವರ್‌ನಲ್ಲಿ ಭದ್ರತೆಯನ್ನು ಅನುಷ್ಠಾನಗೊಳಿಸುವಾಗ ಈ ಉತ್ತಮ ಅಭ್ಯಾಸಗಳನ್ನು ಪಾಲಿಸುವುದು ಸಹಾಯಕ:

- ಪರಿಶೀಲನೆ ಇಲ್ಲದೆ ಬರುವ ವಿನಂತಿಗಳನ್ನು ಎಂದಿಗೂ ನಂಬಬೇಡಿ.
- ಎಲ್ಲಾ ಪ್ರವೇಶ ಮತ್ತು ದೋಷಗಳನ್ನು ದಾಖಲೆ ಮಾಡಿ ಮತ್ತು ಮೇಲ್ವಿಚಾರಣೆ ಮಾಡಿ.
- ಭದ್ರತಾ ದೋಷಗಳನ್ನು ಸರಿಪಡಿಸಲು ಆಧಾರಗಳನ್ನು ನಿಯಮಿತವಾಗಿ ನವೀಕರಿಸಿ.

### ಸವಾಲುಗಳು

MCP ಸ್ಟ್ರೀಮಿಂಗ್ ಸರ್ವರ್‌ಗಳಲ್ಲಿ ಭದ್ರತೆಯನ್ನು ಅನುಷ್ಠಾನಗೊಳಿಸುವಾಗ ನೀವು ಕೆಲವು ಸವಾಲುಗಳನ್ನು ಎದುರಿಸುತ್ತೀರಿ:

- ಅಭಿವೃದ್ಧಿಗೆ ಸುಲಭತೆಯೊಂದಿಗೆ ಭದ್ರತೆಯ ಸಮತೋಲನ
- ವಿವಿಧ ಕ್ಲೈಂಟ್ ಪರಿಸರಗಳೊಂದಿಗೆ ಹೊಂದಿಕೆಯನ್ನು ಖಚಿತಪಡಿಸುವುದು

### ಕಲಿಕೆ: ನಿಮ್ಮ ಸ್ವಂತ ಸ್ಟ್ರೀಮಿಂಗ್ MCP ಅಪ್ಲಿಕೇಶನ್ ನಿರ್ಮಿಸಿ

**ಘಟನೆ:**
ಒಂದು MCP ಸರ್ವರ್ ಮತ್ತು ಕ್ಲೈಂಟ್ ನಿರ್ಮಿಸಿ, ಸರ್ವರ್ ಐಟಮ್‌ಗಳ (ಉದಾ. ಫೈಲ್‌ಗಳು ಅಥವಾ ದಾಖಲೆಗಳು) ಪಟ್ಟಿಯನ್ನು ಪ್ರಕ್ರಿಯೆ ಮಾಡಿ ಪ್ರತಿ ಪ್ರಕ್ರಿಯೆಗೊಳಿಸಿದ ಐಟಮ್‌ಗೆ ಸೂಚನೆಗಳನ್ನು ಕಳುಹಿಸುತ್ತದೆ. ಕ್ಲೈಂಟ್ ಪ್ರತಿ ಸೂಚನೆಯನ್ನು ಹಾಗೆ ಬಂದಂತೆ ಪ್ರದರ್ಶಿಸಬೇಕು.

**ಹಂತಗಳು:**

1. ಐಟಂಗಳ ಪಟ್ಟಿಯನ್ನು ಪ್ರಕ್ರಿಯೆ ಮಾಡಿ ಪ್ರತಿ ಐಟಮ್‌ಗೆ ಸೂಚನೆಗಳನ್ನು ಕಳುಹಿಸುವ ಸರ್ವರ್ ಸಾಧನವನ್ನು ಅನುಷ್ಠಾನಗೊಳಿಸಿ.
2. ಸಂದೇಶ ಹ್ಯಾಂಡ್ಲರ್ ಹೊಂದಿದ ಕ್ಲೈಂಟ್ ಅನ್ನು ಅನುಷ್ಠಾನಗೊಳ್ಳಿಸಿ, ಅದು ಏಕೆಂದರೆ ಸೂಚನೆಗಳನ್ನು实时 ತೋರಿಸುತ್ತದೆ.
3. ಸರ್ವರ್ ಮತ್ತು ಕ್ಲೈಂಟ್ ಎರಡನ್ನೂ ಚಾಲನೆ ಮಾಡಿ ನಿಮ್ಮ ಅನುಷ್ಠಾನವನ್ನು ಪರೀಕ್ಷಿಸಿ ಮತ್ತು ಸೂಚನೆಗಳನ್ನು ವೀಕ್ಷಿಸಿ.
