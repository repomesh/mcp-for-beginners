# HTTPS ஸ்ட்ரீமிங் உடன் மொdelayல் Context Protocol (MCP)

இந்த அத்தியாயம் HTTPS பயன்படுத்தி Model Context Protocol (MCP) மூலம் பாதுகாப்பான, அளவைக் குறைக்கும் மற்றும் நேரடியாக ஸ்ட்ரீமிங் செயல்முறைகளை நடைமுறைப்படுத்துவதற்கான விரிவான வழிகாட்டலை வழங்குகிறது. இது ஸ்ட்ரீமிங் விழிப்புணர்வு, கிடைக்கும் டிரான்ஸ்போர்ட் முறைகள், MCP இல் ஸ்ட்ரீமிங் HTTP ஐ எப்படி நடைமுறைப்படுத்துவது, பாதுகாப்பு சிறந்த நடைமுறைகள், SSE இருந்து மைக்ரேஷன் மற்றும் உங்கள் சொந்த MCP ஸ்ட்ரீமிங் செயலிகள் உருவாக்குவதற்கான நடைமுறை வழிகாட்டுதலை உள்ளடக்கியது.

> **முன்னோக்கி பாருங்கள்:** இந்த பாடம் **MCP குறிப்பிடல் 2025-11-25** உடன் Streamable HTTP பற்றி விவரிக்கிறது, இதில் `initialize` இன் போது ஒரு அமர்வு நிறுவப்பட்டு `Mcp-Session-Id` தலைப்புடன் குறிக்கப்பட்டுள்ளது. `2026-07-28` வெளியீட்டு محாயிரு கைபேசி முன்மாதிரியானது கைமுறை மற்றும் அமர்வு ஐடி முழுமையான முறையில் அகற்றப்படுகிறது, ஒவ்வொரு கோரிக்கையும் தனிப்பட்டவையாக இருக்கக்கூடிய மற்றும் எந்த சர்வர் செயலிக்கு வேண்டுமானாலும் வழிமாற்றப்படுகிறது, ஒட்டிக்கடு அமர்வுகள் இல்லாமல். விவரங்களுக்கு [MCP இல் மாற்றங்கள்: 2026-07-28 வெளியீட்டு முன்மாதிரி](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) பார்க்கவும்.

## MCP இல் டிரான்ஸ்போர்ட் முறைகள் மற்றும் ஸ்ட்ரீமிங்

இந்த பகுதி MCP இல் கிடைக்கும் வெவ்வேறு டிரான்ஸ்போர்ட் முறைகள் மற்றும் அவை கிளையன்ட் மற்றும் சர்வர் இடையேயான நேரடி தொடர்புக்கான ஸ்ட்ரீமிங் திறன்களை இயல்பாக்கும் பாத்திரத்தை ஆராய்கிறது.

### டிரான்ஸ்போர்ட் மெக்கானிசம் என்றால் என்ன?

டிரான்ஸ்போர்ட் மெக்கானிசம் என்பது கிளையன்ட் மற்றும் சர்வர் இடையேயான தரவு பரிமாற்றம் எப்படி நடைபெறும் என்பதைக் குறிப்பிடுகிறது. MCP பல வெவ்வேறு சூழல் மற்றும் தேவைகளுக்கு ஏற்ப பல டிரான்ஸ்போர்ட் வகைகளை ஆதரிக்கிறது:

- **stdio**: தரநிலை உள்ளீடு/வெளியீடு, உள்ளக மற்றும் கட்டளை வரித் துணைக்கருவிகளுக்கு பொருத்தமானது. எளிமையானது ஆனால் வலை அல்லது கிளவோடு பொருத்தமல்ல.
- **SSE (Server-Sent Events)**: HTTP மூலம் சர்வர்கள் நேரடி புதுப்பிப்புகளை கிளையன்ட்களுக்கு அனுப்பும். வலை UI களுக்கு நல்லது, ஆனால் அளவுலாபனம் மற்றும் விளைவூட்டலில் வரம்பு உள்ளது. MCP குறிப்பிடல் 2025-06-18 இல் தனித்துவமான SSE டிரான்ஸ்போர்ட் இயக்களிப்பு ஒதுக்கப்பட்டது மற்றும் "Streamable HTTP" டிரான்ஸ்போர்ட் மூலம் மாற்றப்பட்டது.
- **Streamable HTTP**: நவீன HTTP அடிப்படையிலான ஸ்ட்ரீமிங் டிரான்ஸ்போர்ட், அறிவிப்புகள் மற்றும் சிறந்த அளவுலாபனத்துடன். பெரும்பாலான உற்பத்தி மற்றும் கிளவோடு சூழல்களுக்கு பரிந்துரைக்கப்படுகிறது.

### ஒப்பீட்டு அட்டவணை

இந்த டிரான்ஸ்போர்ட் முறைகளுக்கு இடையேயான வேறுபாடுகளை அறிய கீழ்காணும் ஒப்பீட்டு அட்டவணையை கவனிக்கவும்:

| டிரான்ஸ்போர்ட்       | நேரடி புதுப்பிப்புகள் | ஸ்ட்ரீமிங் | அளவுலாபனம் | பயன்பாட்டு நிலை       |
|---------------------|-------------------|------------|-------------|-----------------------|
| stdio               | இல்லை             | இல்லை      | குறைந்தது   | உள்ளக CLI கருவிகள்    |
| SSE                 | ஆம்               | ஆம்        | நடுத்தர      | வலை, நேரடி புதுப்பிப்புகள் |
| Streamable HTTP     | ஆம்               | ஆம்        | அதிகம்       | கிளவோடு, பற்பல கிளையன்டுகள் |

> **குறிப்பு:** சரியான டிரான்ஸ்போர்ட் தேர்வு செயல்திறன், அளவுலாபனம் மற்றும் பயனர் அனுபவத்துடன் பாதிக்கிறது. **Streamable HTTP** நவீன, அளவுலாபனமான, கிளவோடு தயாரான செயலிகளுக்கு பரிந்துரைக்கப்படுகிறது.

முந்தைய அத்தியாயங்களில் stdio மற்றும் SSE டிரான்ஸ்போர்ட்கள் பற்றி நீங்கள் பார்த்தீர்கள் மற்றும் இந்த அத்தியாயத்தில் ஸ்ட்ரீமிங் HTTP என்பது பற்றி எடுத்துக்காட்டப்பட்டுள்ளது என்பதை கவனிக்கவும்.

## ஸ்ட்ரீமிங்: கருத்துக்கள் மற்றும் ஊக்கங்கள்

ஸ்ட்ரீமிங் அடிப்படையான கருத்துக்களையும் அதன் பின்னேயுள்ள ஊக்கங்களையும் புரிந்துகொள்ளுதல், செயல்திறன் மிகுந்த நேரடி தொடர்பு அமைப்புகளை உருவாக்குவதற்கு அவசியம்.

**ஸ்ட்ரீமிங்** என்பது ஒரு நெட்வொர்க் நிரலாக்கத்தில், முழு பதிலுக்காக காத்திருக்காமல் சிறிய, கையாளக்கூடிய துண்டாகவும் அல்லது நிகழ்வுக்கட்டங்களாகவும் தரவை அனுப்பவும் பெறவும் அனுமதிக்கும் தொழில்நுட்பமாகும். இது குறிப்பாக பயனுள்ளது:

- பெரிய கோப்புகள் அல்லது தரவுத்தொகுப்புகள்.
- நேரடி புதுப்பிப்புகள் (எ.கா., உரையாடல், முன்னேற்ற பட்டைகள்).
- நீண்டகால கணக்குரூபீகைகள், இவை பயனருக்கு அறிய வைக்க விரும்புகையில்.

மேல்தொகுப்பாக ஸ்ட்ரீமிங் பற்றி நீங்கள் தெரிந்து கொள்ள வேண்டியது:

- தரவு படி படி வழங்கப்படுகிறது, ஒரே முறையில் அல்ல.
- கிளையன்ட் வரவிருக்கும் தரவை உடனுக்குடன் செயல்படுத்த முடியும்.
- உணரப்படும் தாமதத்தை குறைத்து பயனர் அனுபவத்தை மேம்படுத்துகிறது.

### ஏன் ஸ்ட்ரீமிங் பயன்படுத்த வேண்டும்?

ஸ்ட்ரீமிங் பயன்படுத்தப்படுவதற்கான காரணங்கள் பின்வருமாறு:

- பயனர்கள் உடனுக்குடன் பின்னூட்டம் பெறுகிறார்கள், முடிவில் மட்டும் அல்ல
- நேரடி செயலிகளுக்கு மற்றும் பதிலளிக்கும் UI க்களுக்கு உதவும்
- நெட்வொர்க் மற்றும் கணக்கும் வளங்களை அதிகரிக்க உதவும்

### எளிய எடுத்துக்காட்டு: HTTP ஸ்ட்ரீமிங் சர்வர் & கிளையன்ட்

ஸ்ட்ரீமிங் எவ்வாறு செயல்படுவதை எடுத்துக்காட்டும் எளிய உதாரணம் இங்கே:

#### Python

**சர்வர் (Python, FastAPI மற்றும் StreamingResponse பயன்படுத்துகிறது):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**கிளையன்ட் (Python, requests பயன்படுத்துகிறது):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

இந்த எடுத்துக்காட்டு சர்வர் அனைத்து செய்திகள் தயாராக காத்திருப்பதற்குப் பதிலாக செய்திகளை நிரப்புகின்றவுடன் கிளையன்டுக்கு அனுப்புகிறது என்பதைக் காட்டுகிறது.

**இது எப்படி செயல்படுகிறது:**

- சர்வர் ஒவ்வொரு செய்தியையும் தயாராக உள்ளவுடன் அளிக்கிறது.
- கிளையன்ட் வரும் ஒவ்வொரு துண்டையும் பெறுகிறது மற்றும் அச்சிடுகிறது.

**தேவைகள்:**

- சர்வர் ஸ்ட்ரீமிங் பதிலை பயன்படுத்த வேண்டும் (எ.கா., FastAPI இல் `StreamingResponse`).
- கிளையன்ட் பதிலை ஸ்ட்ரீம் (`stream=True` in requests) ஆக செயல்படுத்து வேண்டும்.
- பொதுவாக Content-Type `text/event-stream` அல்லது `application/octet-stream`.

#### Java

**சர்வர் (Java, Spring Boot மற்றும் Server-Sent Events பயன்படுத்துகிறது):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**கிளையன்ட் (Java, Spring WebFlux WebClient):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**Java செயல்படுத்தல் குறிப்புகள்:**

- Spring Boot இன் 반응்தன்மை ஸ்டாக் `Flux` உடன் ஸ்ட்ரீமிங் பயன்பாடு
- `ServerSentEvent` நிகழ்வு வகைகளுடன் கட்டமைக்கப்பட்ட நிகழ்வு ஸ்ட்ரீமிங் வழங்கும்
- `WebClient` உடன் `bodyToFlux()` 반응்தன்மை ஸ்ட்ரீமிங் பெறுதலை செய்கிறது
- `delayElements()` நிகழ்வுகளுக்கு இடையேயான செயலாக்க நேரத்தை மாதிரி செய்கிறது
- நிகழ்வுகள் (`info`, `result` போன்ற) வகைகள் கொண்டிருக்கலாம், சிறந்த கிளையன்ட் கையாள்வுக்கு

### ஒப்பீடு: பாரம்பரிய ஸ்ட்ரீமிங் மற்றும் MCP ஸ்ட்ரீமிங்

பாரம்பரியமான முறையிலும் MCP இல் ஸ்ட்ரீமிங் எப்படி வேறுபடுகிறது என்பதைக் காட்டும் ஒப்பீடு:

| அம்சம்                  | பாரம்பரிய HTTP ஸ்ட்ரீமிங்       | MCP ஸ்ட்ரீமிங் (அறிவிப்புகள்)     |
|-------------------------|--------------------------------|----------------------------------|
| பிரதான பதில்           | துண்டுகளாக                     | ஒரே, முடிவில் வழங்கப்படும்       |
| முன்னேற்ற புதுப்பிப்புகள்| தரவு துண்டுகளாக அனுப்பப்படுகிறது | அறிவிப்புகளாக அனுப்பப்படுகிறது     |
| கிளையன்ட் தேவைகள்      | ஸ்ட்ரீம் செயலாக்க வேண்டும்       | செய்தி கையாள்வதை நடைமுறைப்படுத்த வேண்டும் |
| பயன்பாடு               | பெரிய கோப்புகள், AI டோக்கன் ஸ்ட்ரீம்கள் | முன்னேற்றம், பதிவு, நேரடி பின்னூட்டம் |

### முக்காடு வித்தியாசங்கள்

மேலும் கீழ்கண்ட முக்கிய வித்தியாசங்களைக் காணலாம்:

- **தொடர்பு முறைகள்:**
  - பாரம்பரிய HTTP ஸ்ட்ரீமிங்: எளிய துண்டு மாற்றக் குறி கொடுப்பை (chunked transfer encoding) பயன்படுத்துகிறது
  - MCP ஸ்ட்ரீமிங்: JSON-RPC ப்ரொடோக்கோல் கொண்ட கட்டமைக்கப்பட்ட அறிவிப்பு முறை

- **செய்தி வடிவம்:**
  - பாரம்பரிய HTTP: புதிய வரிசைகளுடன் எளிய சொற்பொழிகள் துண்டுகள்
  - MCP: கட்டமைக்கப்பட்ட LoggingMessageNotification பொருட்கள் வகை மற்றும் விவரங்கள் உடன்

- **கிளையன்ட் செயல்படுத்தல்:**
  - பாரம்பரிய HTTP: ஸ்ட்ரீமிங் பதில்களை செயல்படுத்தும் எளிய கிளையன்ட்
  - MCP: பலவிதமான வகை செய்திகளை நிர்வகிக்கும் அறிவிப்பு கையாள்பவருடன் கூடிய மேம்பட்ட கிளையன்ட்

- **பயனாளர் முன்னேற்ற புதுப்பிப்புகள்:**
  - பாரம்பரிய HTTP: பிரதான பதில் ஸ்ட்ரீமில் இருக்கிறது
  - MCP: பிரதான பதில் முடிவில் வரும் பொழுது தனி அறிவிப்புகள் மூலம் முன்னேற்றம் அனுப்பப்படும்

### பரிந்துரைகள்

பாரம்பரிய ஸ்ட்ரீமிங் (`/stream` என நாம் முன்பு எடுத்துக்காட்டியது) மற்றும் MCP ஸ்ட்ரீமிங் என்ற இரண்டில் தேர்ந்தெடுப்பதில் கீழ்காணும் பரிந்துரைகள் உள்ளன.

- **எளிய ஸ்ட்ரீமிங் தேவைகளுக்கு:** பாரம்பரிய HTTP ஸ்ட்ரீமிங் எளிதாக நடைமுறைப்படுத்தக்கூடியது மற்றும் அடிப்படையான தேவைகளுக்கு போதும்.

- **சிக்கலான, தொடர்புடைய செயலிகளுக்கு:** MCP ஸ்ட்ரீமிங் ஸ்ட்ரக்சர்ட் அணுகுதலோடு விவரமான மெட்டாடேட்டாவுடன் மற்றும் அறிவிப்புகள் மற்றும் இறுதி முடிவுகளுக்கு மாறுபடுத்தப்பட்ட அமைப்பைக் கொண்டுள்ளது.

- **AI செயலிகளுக்கு:** MCP அறிவிப்பு முறை நீண்டகால AI பணி முன்னேற்றத்தைக் தெரிவிக்க பயனுள்ளதாகும்.

## MCP இல் ஸ்ட்ரீமிங்

இப்போதுவரை பாரம்பரிய மற்றும் MCP ஸ்ட்ரீமைகளுக்கு இடையேயான பரிந்துரைகள் மற்றும் ஒப்பீடுகளை பார்த்திருக்கிறீர்கள். இப்போது MCP இல் ஸ்ட்ரீமிங் எப்படி செயல்படுகிறது என்பதை விரிவாகப் பார்ப்போம்.

MCP கட்டமைப்பில் ஸ்ட்ரீமிங் என்பது பிரதான பதிலை துண்டுகளாக அனுப்பும் பதிலில்லை, ஆனால் ஒரு கருவி கோரிக்கையை செயலாக்கும் போது கிளையன்டுக்கு **அறிவிப்புகள்** அனுப்புவதாகும். இந்த அறிவிப்புகள் முன்னேற்ற புதுப்பிப்புகள், பதிவுகள் அல்லது வேறு நிகழ்வுகளை உள்ளடக்கலாம்.

### இது எப்படி செயல்படுகிறது

முக்கிய முடிவுகள் இன்னும் ஒரே பதிலாக அனுப்பப்படுகின்றன. இருப்பினும், செயலாக்கத்தின் போது தனி செய்திகளாக அறிவிப்புகள் அனுப்பப்படோமாம் மற்றும் அவை நேரடியாக கிளையன்டை புதுப்பிக்கின்றன. கிளையன்ட் இந்த அறிவிப்புகளை கையாளி காட்சிப்படுத்தும் திறன் வேண்டும்.

## அறிவிப்பு என்பதின் பொருள் என்ன?

"அறிவிப்பு" என்று கூறினோம், MCP உடன் அது என்ன அர்த்தம்?

அறிவிப்பு என்பது ஒரு நீண்டகால பணியின் போது முன்னேற்றம், நிலை அல்லது பிற நிகழ்வுகளை பற்றிய தகவலை சர்வர் இருந்து கிளையன்டுக்கு அனுப்பும் செய்தி ஆகும். அறிவிப்புகள் தெளிவுத்தன்மையை மற்றும் பயனர் அனுபவத்தை மேம்படுத்துகின்றன.

உதாரணமாக, சர்வருடன் ஆரம்ப கைபிடிப்பு முடிந்ததும் கிளையன்ட் ஒரு அறிவிப்பை அனுப்ப வேண்டும்.

அறிவிப்பு கீழ்காணும் JSON செய்தி போல தோற்றமளிக்கிறது:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

அறிவிப்புகள் MCP இல் ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) என்ற தலைப்பிற்கு சிறந்தவை.

பதிவை செயல்படுத்த சர்வர் கீழ்காணும் விதமாக ஒரு பண்பு/திறன் enabling செய்ய வேண்டும்:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> பயன்படுத்தப்படும் SDK பொருந்தி, பதிவு இயங்குவதற்கான அமைப்பு இயல்பாக இயங்கு செய்யலாம், அல்லது நீங்கள் தெளிவாக உங்கள் சர்வர் அமைப்பில் செயல்படுத்த வேண்டும்.

அறிவிப்புகளுக்கு வெவ்வேறு வகைகள் உள்ளன:

| மட்டம்       | விளக்கம்                        | பயன்பாட்டு எடுத்துக்காட்டு          |
|-------------|-------------------------------|----------------------------------|
| debug       | விரிவான பிழைத்துடுப்பு தகவல் | செயல்பாட்டு நுழைவு/வெளி புள்ளிகள்  |
| info        | பொதுவான தகவல் செய்திகள்       | செயலாக்க முன்னேற்ற புதுப்பிப்புகள்  |
| notice      | சாதாரண ஆனால் முக்கிய நிகழ்வுகள்| கட்டமைப்பு மாற்றங்கள்              |
| warning     | எச்சரிக்கை நிலைகள்            | பழைய பண்பாடு பயன்பாடு           |
| error       | பிழை நிலைகள்                 | செயல்பாட்டு தோல்விகள்             |
| critical    | மிக முக்கிய நிலைகள்           | அமைப்பு கூறு தோல்விகள்           |
| alert       | உடனடி நடவடிக்கை தேவை        | தரவு அழிவு கண்டறிதல்             |
| emergency   | அமைப்பு பயன்படுத்த முடியாது    | முழுமையான அமைப்பு தோல்வி        |

## MCP இல் அறிவிப்புகளை நடைமுறைப்படுத்தல்

அறிவிப்புகளை MCP இல் நடைமுறைப்படுத்த, சர்வர் மற்றும் கிளையன்ட் இரண்டிலும் நேரடி புதுப்பிப்புகளை கையாளும் அமைப்பை உருவாக்க வேண்டும். இதனால் நீண்டகால செயல்களில் பயனர்களுக்கு உடனடி பின்னூட்டம் வழங்க முடியும்.

### சர்வர் பக்கம்: அறிவிப்புகளை அனுப்புதல்

சர்வர் பக்கம் தொடங்குவோம். MCP இல், கோரிக்கைகள் செயலாக்கப்படும் போது அறிவிப்புகளை அனுப்பும் கருவிகளை நீங்கள் வரையறுக்க வேண்டும். சர்வர் பொதுவாக உள்ளூர் சூழல் பொருளான `ctx` ஐ பயன்படுத்தி கிளையன்டுக்கு செய்திகளை அனுப்புகிறது.

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

மேலுள்ள எடுத்துக்காட்டில், `process_files` கருவி ஒவ்வொரு கோப்பையும் செயலாக்கும்போது கிளையன்டுக்கு மூன்று அறிவிப்புகளை அனுப்புகிறது. `ctx.info()` முறை தகவல் குறிப்பு செய்திகளை அனுப்ப பயன்படுத்தப்படுகிறது.

மேலும், அறிவிப்புகளை இயக்க, உங்கள் சர்வர் ஸ்ட்ரீமிங் டிரான்ஸ்போர்ட் (எ.கா., `streamable-http`) பயன்படுத்தி, உங்கள் கிளையன்ட் அறிவிப்புகளை கையாளும் செய்தி கையாள்பவரை செயல்படுத்த வேண்டும். கீழே சர்வரை `streamable-http` டிரான்ஸ்போர்டுடன் எப்படி அமைப்பது என்பதை காணலாம்:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

இந்த .NET எடுத்துக்காட்டில், `ProcessFiles` கருவி `Tool` அடையாளத்துடன் அலங்கரிக்கப்பட்டு, ஒவ்வொரு கோப்பையும் செயலாக்கும் போது கிளையன்டுக்கு மூன்று அறிவிப்புகளை அனுப்புகிறது. `ctx.Info()` முறை தகவல் குறிப்பு செய்திகளை அனுப்ப பயன்படுத்தப்படுகிறது.

உங்கள் .NET MCP சர்வரில் அறிவிப்புகளை இயக்கு, ஸ்ட்ரீமிங் டிரான்ஸ்போர்ட் பயன்படுத்துவதை உறுதி செய்யவும்:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### கிளையன் பக்கம்: அறிவிப்புகளை பெறுதல்

கிளையன்ட் அறிவிப்புகளை பெறவும் காட்சிப்படுத்தவும் செய்தி கையாள்பவரை செயல்படுத்த வேண்டும்.

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

மேலுள்ள கோடில், `message_handler` செயல்பாடு வரும் செய்தி அறிவிப்பு பொருட்டா என்பதை சரிபார்க்கிறது. அது அறிவிப்பு என்றால் அதனை அச்சிடுகிறது, இல்லையெனில் வழக்கமான சர்வர் செய்தியாக செயல்படுத்துகிறது. மேலும், `ClientSession` ஆரம்பிக்கும்போது `message_handler` அறிவிப்புகளை கையாள பயன்படுகிறது.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

இந்த .NET எடுத்துக்காட்டில், `MessageHandler` செயல்பாடு வரும் செய்தி அறிவிப்பா என்பதை ஆய்வு செய்கிறது. அதில் தெரிவிக்கப்படும் உதவிக்கு அச்சிடுகிறது, இல்லையாக இருந்தால் வழக்கான சர்வர் செய்தியாக செயல்படுத்துகிறது. `ClientSession` செய்தி கையாள்பவர் மூலம் `ClientSessionOptions` உதவியுடன் ஆரம்பிக்கப்படுகிறது.

அறிவிப்புகளை இயக்கு, உங்கள் சர்வர் ஸ்ட்ரீமிங் டிரான்ஸ்போர்ட் (எ.கா., `streamable-http`) பயன்படுத்துவதை உறுதி செய்யவும் மற்றும் உங்கள் கிளையன்ட் அறிவிப்புகளை கையாளும் செய்தி கையாள்பவரை செயல்படுத்த வேண்டும்.

## முன்னேற்ற அறிவிப்புகள் & சூழல்கள்

இந்தப் பகுதி MCP இல் முன்னேற்ற அறிவிப்புகளின் கருத்து என்ன, அவை ஏன் அவசியம், மற்றும் Streamable HTTP பயன்படுத்தி அவற்றை எப்படி நடைமுறைப்படுத்துவது என்பதை விளக்குகிறது. இதோடு உங்கள் புரிதலை வலுப்படுத்த ஒரு நடைமுறைப் பணியுமுண்டு.

முன்னேற்ற அறிவிப்புகள் நீண்டகால செயல்களில் சர்வர் இருந்து கிளையன்டுக்கு நேரடி செய்திகளாக அனுப்பப்படுகின்றன. முழு செயல்திறனுக்காக காத்திருப்பதைவிட, சர்வர் தற்போதைய நிலையை கிளையன்டிடம் தொடர்ந்து தெரிவிக்கிறது. இது தெளிவுத்தன்மையை, பயனர் அனுபவத்தை மேம்படுத்துகிறது மற்றும் பிழைத்திருத்தத்தை எளிதாக்குகிறது.

**எடுத்துக்காட்டு:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### ஏன் முன்னேற்ற அறிவிப்புகள் பயன்படுத்த வேண்டும்?

முன்னேற்ற அறிவிப்புகள் பல காரணங்களுக்காக அவசியம்:

- **மிக சிறந்த பயனர் அனுபவம்:** பணிகள் முன்னேறும்போது பயனர்கள் புதுப்பிப்புகளை காண்கின்றனர், முடிவில் மட்டும் அல்ல.
- **நேரடி பின்னூட்டம்:** கிளையன்டுகள் முன்னேற்ற பட்டைகள் அல்லது பதிவு காட்சிகள் காட்ட முடியும், செயலி பதிலளிப்பாக தோற்றமளிக்கிறது.
- **எளிதான பிழைத்திருத்தம் மற்றும் கண்காணிப்பு:** டெவலப்பர்கள் மற்றும் பயனர்கள் ஒரு செயலி எந்த பகுதியில் மெதுவாக அல்லது சிலந்தையாக இருக்கின்றது என்று தெரிந்துகொள்ளலாம்.

### முன்னேற்ற அறிவிப்புகள் எப்படி நடைமுறைப்படுத்தலாம்

MCP இல் முன்னேற்ற அறிவிப்புகள் செயல்படபடுத்தும் விதம்:

- **சர்வர் பக்கம்:** ஒவ்வொரு உருப்படியும் செயல்படும்போது `ctx.info()` அல்லது `ctx.log()` பயன்படுத்தி அறிவிப்புகளை அனுப்பவும். இதனால் பிரதான முடிவுக்கு முன்னரே கிளையன்டுக்கு செய்தி அனுப்பப்படுகிறது.
- **கிளையன்ட் பக்கம்:** வரவிருக்கும் அறிவிப்புகளை கையாளும் மற்றும் காட்சிப்படுத்தும் செய்தி கையாள்பவர் செயல்படுத்த வேண்டும். இது அறிவிப்புகள் மற்றும் இறுதி முடிவுகளை வேறுபடுத்தும்.

**சர்வர் எடுத்துக்காட்டு:**

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**கிளையன்ட் எடுத்துக்காட்டு:**

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## பாதுகாப்பு பரிசீலனைகள்

HTTP அடிப்படையிலான டிரான்ஸ்போர்ட்களுடன் MCP சர்வர்களை செயல்படுத்தும்போது, பாதுகாப்பு மிக முக்கிய பிரச்சினையாக மாறுகிறது மற்றும் பல வகையான தாக்குதல்கள் மற்றும் பாதுகாப்பு முறைகளை கவனிக்க வேண்டும்.

### கண்ணோட்டம்

HTTP வழியாக MCP சர்வர்களை வெளிப்படுத்தும் போது பாதுகாப்பு அவசியம். Streamable HTTP புதிய தாக்குதல் முகப்புகளை அறிமுகப்படுத்துகிறது மற்றும் கவனமாக கட்டமைக்கப்பட வேண்டும்.

### முக்கிய குறிப்புகள்
- **மூலம் தலைப்பு சரிபார்த்தல்**: DNS மறுசுழற்சி தாக்குதல்களைத் தடுப்பதற்காக எப்போதும் `Origin` தலைப்பைப் பரிசோதிக்கவும்.
- **உள்ளூர் ஹோஸ்ட் தொடர்பமைத்தல்**: உள்ளூர் மேம்பாட்டிற்கான, சேவையகங்களை பொது இணையத்தில் வெளிப்படுத்தாமல் `localhost` இல் பிணைக்கவும்.
- **அங்கீகாரம்**: உற்பத்தி உட்படையானவைகளுக்காக அங்கீகாரத்தை (எ.கா., API விசைகள், OAuth) அமல்படுத்தவும்.
- **CORS**: அணுகலை கட்டுப்படுத்த சர்வதேச மூல பகிர்வு கொள்கைகளை அமைக்கவும்.
- **HTTPS**: போக்குவரத்தை குறியாக்குவதற்காக உற்பத்தியில் HTTPS ஐப் பயன்படுத்தவும்.

### சிறந்த நடைமுறைகள்

- சரிபார்ப்புத் துறை இல்லாமல் வரும் கோரிக்கைகளுக்கு ஒருபோதும் நம்பிக்கை வைக்கக்கூடாது.
- அனைத்து அணுகல் மற்றும் பிழைகளை பதிவுசெய்து கண்காணிக்கவும்.
- பாதுகாப்பு குறைபாடுகளை சரிசெய்ய வெளிப்படையாக ஏற்புடைய நூலகங்களை தவறாமல் புதுப்பிக்கவும்.

### சவால்கள்

- பாதுகாப்பையும் மேம்பாட்டின் எளிமையையும் சமநிலை காட்டுதல்
- பலவிதமான கிளையண்ட் சூழல்களின் ஒத்துழைவை உறுதிசெய்தல்

## SSE இலிருந்து Streamable HTTP க்கு மேம்படுத்தல்

தற்போது Server-Sent Events (SSE) பயன்படுத்தும் செயலியில், Streamable HTTP க்கு மாற்றம் செய்வது உங்கள் MCP செயலாக்கங்களுக்கு மேம்பட்ட திறன் மற்றும் நீண்டகால நிலைத்தன்மையை வழங்குகிறது.

### ஏன் மேம்படுத்த வேண்டும்?

SSE இலிருந்து Streamable HTTP க்கு மாற்றம் செய்ய இரண்டு முக்கிய காரணங்கள் உள்ளன:

- Streamable HTTP, SSE விட சிறந்த அளவிடல் திறனை, ஒத்துழைவை மற்றும் விரிவான அறிவித்தல் ஆதரவுகளை வழங்குகிறது.
- இது புதிய MCP செயலிகளுக்கான பரிந்துரைக்கபட்ட போக்குவரத்து முறையாகும்.

### மாற்று படிகள்

உங்கள் MCP செயலிகளுக்கு SSE இலிருந்து Streamable HTTP க்கு மாற்ற உதவ வரும் படிகள்:

- **சேவையகக் குறியீட்டை** `mcp.run()` இல் `transport="streamable-http"` என புதுப்பிக்கவும்.
- **வாடிக்கையாளர் குறியீட்டை** SSE வாடிக்கையாளர் பதிலாக `streamablehttp_client` பயன்படுத்த도록 மாற்றவும்.
- **அறிவித்தல்கள் செயலாக்க குறியாளர்** வாடிக்கையாளரில் அமல்படுத்தவும்.
- **தொடர்புடைய கருவிகள் மற்றும் பணிகளுடன் ஒத்துழைவை** சோதிக்கவும்.

### ஒத்துழைவை பராமரித்தல்

மாற்று செயல்பாட்டின்போது தற்போதைய SSE வாடிக்கையாளர்களுடன் ஒத்துழைப்பை பராமரிப்பது பரிந்துரைக்கப்படுகிறது. சில முறைகள்:

- இரண்டு பரிமாற்றங்களையும் வேறு இடங்களில் இயக்குவதன் மூலம் SSE மற்றும் Streamable HTTP இரண்டையும் ஆதரிக்கலாம்.
- மெதுவாக வாடிக்கையாளர்களை புதிய போக்குவரத்துக்கு மாற்றவும்.

### சவால்கள்

மாற்றின் போது கீழ்க்காணும் சவால்களை எதிர்கொள்ள வேண்டும்:

- அனைத்து வாடிக்கையாளர்களும் புதுப்பிக்கப்படுவதை உறுதி செய்தல்
- அறிவித்தல் விநியோகத்தில் வேறுபாடுகளை கையாள்தல்

## பாதுகாப்பு பரிசீலனைகள்

எந்தவொரு சேவையகத்தாலும் பாதுகாப்பு மிக முக்கியம், குறிப்பாக MCP இல் Streamable HTTP போன்ற HTTP அடிப்படையிலான போக்குவரத்தை பயன்படுத்தும் போது.

### மேலோட்டம்

MCP சேவையகங்களை HTTP இல் வெளிப்படுத்தும் போது பாதுகாப்பு அவசியம். Streamable HTTP புதிய தாக்குதல் பரப்புகளை உருவாக்குகிறது மற்றும் சீரான கட்டமைப்பைக் கேட்கிறது.

முகாமை பின்பற்ற வேண்டிய சில முக்கிய பாதுகாப்பு அம்சங்கள்:

- **மூலம் தலைப்பு சரிபார்த்தல்**: DNS மறுசுழற்சி தாக்குதல்களைத் தடுப்பதற்காக எப்போதும் `Origin` தலைப்பைப் பரிசோதிக்கவும்.
- **உள்ளூர் ஹோஸ்ட் தொடர்பமைத்தல்**: உள்ளூர் மேம்பாட்டிற்கான, சேவையகங்களை பொது இணையத்தில் வெளிப்படுத்தாமல் `localhost` இல் பிணைக்கவும்.
- **அங்கீகாரம்**: உற்பத்தி உட்படையானவைகளுக்காக அங்கீகாரத்தை (எ.கா., API விசைகள், OAuth) அமல்படுத்தவும்.
- **CORS**: அணுகலை கட்டுப்படுத்த சர்வதேச மூல பகிர்வு கொள்கைகளை அமைக்கவும்.
- **HTTPS**: போக்குவரத்தை குறியாக்குவதற்காக உற்பத்தியில் HTTPS ஐப் பயன்படுத்தவும்.

### சிறந்த நடைமுறைகள்

மேலும், உங்கள் MCP ஸ்ட்ரீமிங் சேவையகத்தில் பாதுகாப்பை நடைமுறைப்படுத்தும் போது பின்பற்ற வேண்டிய சிறந்த நடைமுறைகள்:

- சரிபார்ப்புத் துறை இல்லாமல் வரும் கோரிக்கைகளுக்கு ஒருபோதும் நம்பிக்கை வைக்கக்கூடாது.
- அனைத்து அணுகல் மற்றும் பிழைகளையும் பதிவுசெய்து கண்காணிக்கவும்.
- பாதுகாப்பு குறைபாடுகளை சரிசெய்ய வெளிப்படையாக ஏற்புடைய நூலகங்களை தவறாமல் புதுப்பிக்கவும்.

### சவால்கள்

MCP ஸ்ட்ரீமிங் சேவையகங்களில் பாதுகாப்பு அமல்படுத்தும் போது எதிர்கொள்ளும் சவால்கள்:

- பாதுகாப்பையும் மேம்பாட்டின் எளிமையையும் சமநிலை காட்டுதல்
- பலவிதமான கிளையண்ட் சூழல்களின் ஒத்துழைவை உறுதிசெய்தல்

### பணிகள்: உங்கள் சொந்த ஸ்ட்ரீமிங் MCP செயலியை உருவாக்கவும்

**சூழல்:**
ஒரு MCP சேவையகமும் வாடிக்கையாளருமானது, சேவையகமும் ஒரு பட்டியலை (எ.கா., கோப்புகள் அல்லது ஆவணங்கள்) செயல்படுத்து, ஒவ்வொரு செயலாக்கப்பட்ட உருப்படியுக்கும் அறிவித்தல்களை அனுப்பும். வாடிக்கையாளர் வரும் அறிவித்தல்களை உடனுக்குடன் காட்சி படுத்த வேண்டும்.

**படிகள்:**

1. பட்டியலை செயல்படுத்தி ஒவ்வொரு உருப்படியுக்கும் அறிவித்தல் அனுப்பும் சேவையக கருவியை உருவாக்கவும்.
2. அறிவித்தல்களை நேரடியாக காட்சிப்படுத்த வாடிக்கையாளரைச் செய்தி செயலி உடன் உருவாக்கவும்.
3. சேவையகம் மற்றும் வாடிக்கையாளரை இயக்கி உங்கள் அமல்பாட்டைப் பரிசோதிக்கவும், அறிவித்தல்களை கவனிக்கவும்.

## மேலதிக வாசிப்பு மற்றும் அடுத்தது என்ன?

MCP ஸ்ட்ரீமிங்கில் உங்கள் பயணத்தை தொடரவும், மேலும் உயர் செயலிகளை உருவாக்க விரிவுபடுத்த இந்த பகுதி கூடுதல் வளங்களையும் பரிந்துரைகளையும் வழங்குகிறது.
