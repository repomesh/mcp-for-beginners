# মডেল কন্টেক্সট প্রোটোকল (MCP)-এর সাথে HTTPS স্ট্রিমিং

এই অধ্যায়টি HTTPS ব্যবহার করে মডেল কন্টেক্সট প্রোটোকল (MCP) সহ নিরাপদ, স্কেলযোগ্য এবং রিয়েল-টাইম স্ট্রিমিং বাস্তবায়নের জন্য একটি ব্যাপক গাইড প্রদান করে। এটি স্ট্রিমিং-এর প্রেরণা, উপলব্ধ ট্রান্সপোর্ট মেকানিজম, MCP-তে স্ট্রিমেবল HTTP কীভাবে বাস্তবায়ন করবেন, নিরাপত্তার অভিজ্ঞতা, SSE থেকে মাইগ্রেশন এবং নিজের স্ট্রিমিং MCP অ্যাপ্লিকেশন তৈরি করার জন্য প্রায়োগিক নির্দেশনা কভার করে।

> **আগাম দেখুন:** এই পাঠটি **MCP স্পেসিফিকেশন 2025-11-25** এর অধীনে স্ট্রিমেবল HTTP বর্ণনা করে, যেখানে `initialize` সময় একটি সেশন প্রতিষ্ঠিত হয় এবং একটি `Mcp-Session-Id` হেডারের মাধ্যমে পিন করা হয়। `2026-07-28` রিলিজ ক্যান্ডিডেট সম্পূর্ণভাবে হ্যান্ডশেক এবং সেশন আইডি সরিয়ে দেয়, ফলে প্রতিটি অনুরোধ স্বয়ংসম্পূর্ণ হয় এবং যেকোনো সার্ভার ইনস্ট্যান্সে রাউটেবল হয় স্টিকি সেশন ছাড়াই। বিস্তারিত জানতে দেখুন [MCP-তে কী পরিবর্তন হচ্ছে: 2026-07-28 রিলিজ ক্যান্ডিডেট](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md)।

## MCP-তে ট্রান্সপোর্ট মেকানিজম এবং স্ট্রিমিং

এই অংশে MCP-তে উপলব্ধ বিভিন্ন ট্রান্সপোর্ট মেকানিজম এবং ক্লায়েন্ট ও সার্ভারের মধ্যে রিয়েল-টাইম যোগাযোগের জন্য স্ট্রিমিং সক্ষম করার ভূমিকাটি আলোচনা করা হয়েছে।

### ট্রান্সপোর্ট মেকানিজম কি?

একটি ট্রান্সপোর্ট মেকানিজম নির্ধারণ করে ক্লায়েন্ট এবং সার্ভারের মধ্যে ডেটা কিভাবে বিনিময় হবে। MCP পরিবেশ এবং প্রয়োজনীয়তা অনুযায়ী একাধিক ট্রান্সপোর্ট টাইপ সমর্থন করে:

- **stdio**: স্ট্যান্ডার্ড ইনপুট/আউটপুট, স্থানীয় এবং CLI-ভিত্তিক টুলের জন্য উপযুক্ত। সহজ কিন্তু ওয়েব বা ক্লাউডের জন্য উপযুক্ত নয়।
- **SSE (সার্ভার-সেন্ট ইভেন্টস)**: সার্ভারগুলি HTTP মাধ্যমে ক্লায়েন্টকে রিয়েল-টাইম আপডেট পাঠানোর অনুমতি দেয়। ওয়েব UI-এর জন্য ভালো কিন্তু স্কেলেবিলিটি এবং নমনীয়তায় সীমাবদ্ধ। MCP স্পেসিফিকেশন 2025-06-18 থেকে, স্বতন্ত্র SSE (সার্ভার-সেন্ট ইভেন্টস) ট্রান্সপোর্ট বাতিল হয়ে "স্ট্রিমেবল HTTP" ট্রান্সপোর্ট দ্বারা প্রতিস্থাপিত হয়েছে।
- **স্ট্রিমেবল HTTP**: আধুনিক HTTP-ভিত্তিক স্ট্রিমিং ট্রান্সপোর্ট, বিজ্ঞপ্তি এবং উন্নত স্কেলেবিলিটি সমর্থন করে। অধিকাংশ উৎপাদন এবং ক্লাউড দৃশ্যে সুপারিশকৃত।

### তুলনা টেবিল

নিচের তুলনা টেবিলে এই ট্রান্সপোর্ট মেকানিজমগুলির মধ্যে পার্থক্য বোঝার জন্য দেখুন:

| ট্রান্সপোর্ট       | রিয়েল-টাইম আপডেট | স্ট্রিমিং | স্কেলেবিলিটি | ব্যবহারের ক্ষেত্র          |
|-------------------|--------------------|-----------|--------------|----------------------------|
| stdio             | না                 | না        | কম          | স্থানীয় CLI টুলস          |
| SSE               | হ্যাঁ               | হ্যাঁ     | মাঝারী       | ওয়েব, রিয়েল-টাইম আপডেট  |
| স্ট্রিমেবল HTTP  | হ্যাঁ               | হ্যাঁ     | উচ্চ          | ক্লাউড, মাল্টি-ক্লায়েন্ট  |

> **টিপ:** সঠিক ট্রান্সপোর্ট নির্বাচন পারফরম্যান্স, স্কেলেবিলিটি এবং ব্যবহারকারীর অভিজ্ঞতাকে প্রভাবিত করে। আধুনিক, স্কেলযোগ্য এবং ক্লাউড-প্রস্তুত অ্যাপ্লিকেশনের জন্য **স্ট্রিমেবল HTTP** সুপারিশকৃত।

আগের অধ্যায়ে stdio এবং SSE ট্রান্সপোর্ট দেখানো হয়েছে এবং এই অধ্যায়ে স্ট্রিমেবল HTTP-এর ট্রান্সপোর্ট অন্তর্ভুক্ত।

## স্ট্রিমিং: ধারণা এবং প্রেরণা

স্ট্রিমিং-এর মৌলিক ধারণা এবং প্রেরণা বোঝা কার্যকর রিয়েল-টাইম যোগাযোগ সিস্টেম বাস্তবায়নের জন্য অপরিহার্য।

**স্ট্রিমিং** হল একটি নেটওয়ার্ক প্রোগ্রামিং কৌশল যা ডেটাকে ছোট, পরিচালনাযোগ্য টুকরো বা ইভেন্টের ধারাবাহিক আকারে পাঠানো ও গ্রহণের অনুমতি দেয়, পুরো উত্তর প্রস্তুত না হওয়া পর্যন্ত অপেক্ষা না করে। এটি বিশেষত উপযোগী:

- বড় ফাইল বা ডেটাসেটের জন্য।
- রিয়েল-টাইম আপডেটের জন্য (যেমন, চ্যাট, প্রগ্রেস বার)।
- দীর্ঘ চলমান গণনায় যেখানে ব্যবহারকারীকে অবহিত রাখা জরুরি।

উচ্চ স্তরে স্ট্রিমিং সম্পর্কে যা জানার দরকার:

- ডেটা পর্যায়ক্রমে পৌঁছে দেওয়া হয়, একবারে নয়।
- ক্লায়েন্ট ডেটা আসার সাথে সাথে প্রক্রিয়া করতে পারে।
- উপলক্ষিত দেরি কমায় এবং ব্যবহারকারীর অভিজ্ঞতা উন্নত করে।

### কেন স্ট্রিমিং ব্যবহার করবেন?

স্ট্রিমিং ব্যবহারের কারণগুলি হলো:

- ব্যবহারকারী সাথে সাথে প্রতিক্রিয়া পান, শুধুমাত্র শেষে নয়।
- রিয়েল-টাইম অ্যাপ্লিকেশন এবং দ্রুত প্রতিক্রিয়াশীল UI তৈরি করে।
- নেটওয়ার্ক এবং কম্পিউট রিসোর্স আরো দক্ষ ব্যবহার।

### সহজ উদাহরণ: HTTP স্ট্রিমিং সার্ভার ও ক্লায়েন্ট

এখানে স্ট্রিমিং কিভাবে বাস্তবায়িত হতে পারে তার একটি সহজ উদাহরণ:

#### পাইথন

**সার্ভার (পাইথন, FastAPI এবং StreamingResponse ব্যবহার করে):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**ক্লায়েন্ট (পাইথন, requests ব্যবহার করে):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

এই উদাহরণে দেখানো হয়েছে কিভাবে সার্ভার মেসেজগুলিকে সময়মতো পাঠায়, পুরো মেসেজ অপেক্ষা না করেই।

**কিভাবে কাজ করে:**

- সার্ভার প্রতিটি মেসেজ প্রস্তুত হওয়ার সাথে সাথে সরবরাহ করে।
- ক্লায়েন্ট প্রতিটি অংশ আসার সাথে সাথে গ্রহণ এবং মুদ্রণ করে।

**চাহিদাসমূহ:**

- সার্ভার স্ট্রিমিং রেস্পন্স ব্যবহার করতে হবে (যেমন, FastAPI-এর `StreamingResponse`)।
- ক্লায়েন্ট রেস্পন্স স্ট্রিম (`stream=True` requests-এ) হিসেবে প্রক্রিয়া করতে হবে।
- Content-Type সাধারণত `text/event-stream` অথবা `application/octet-stream`।

#### জাভা

**সার্ভার (জাভা, Spring Boot এবং Server-Sent Events ব্যবহার করে):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**ক্লায়েন্ট (জাভা, Spring WebFlux WebClient ব্যবহার করে):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**জাভা বাস্তবায়ন নোট:**

- স্ট্রিমিংয়ের জন্য Spring Boot-এর রিয়েক্টিভ স্ট্যাক `Flux` ব্যবহার করে
- `ServerSentEvent` ইভেন্ট টাইপ সহ কাঠামোবদ্ধ ইভেন্ট স্ট্রিমিং সরবরাহ করে
- `WebClient`-এর `bodyToFlux()` রিয়েক্টিভ স্ট্রিমিং গ্রহণ সক্ষম করে
- `delayElements()` ইভেন্টের মাঝে প্রক্রিয়াকরণের সময় অনুকরণ করে
- ইভেন্টগুলির ধরনের (`info`, `result`) ক্লায়েন্ট-হ্যান্ডলিং উন্নত করে

### তুলনা: ক্লাসিক স্ট্রিমিং বনাম MCP স্ট্রিমিং

"ক্লাসিক্যাল" পদ্ধতিতে স্ট্রিমিং কীভাবে কাজ করে এবং MCP-তে কিভাবে কাজ করে তার মধ্যে পার্থক্য নিম্নরূপ:

| বৈশিষ্ট্য                | ক্লাসিক HTTP স্ট্রিমিং        | MCP স্ট্রিমিং (বিজ্ঞপ্তি)        |
|------------------------|------------------------------|---------------------------------|
| প্রধান রেসপন্স          | টুকরো টুকরো (Chunked)        | একটি, শেষে                      |
| প্রগ্রেস আপডেট         | ডেটা টুকরোগুলোর মাধ্যমে পাঠানো | বিজ্ঞপ্তি হিসেবে পাঠানো          |
| ক্লায়েন্টের প্রয়োজন  | স্ট্রিম প্রক্রিয়া করা আবশ্যক | মেসেজ হ্যান্ডলার বাস্তবায়ন প্রয়োজন |
| ব্যবহারের ক্ষেত্র       | বড় ফাইল, AI টোকেন স্ট্রিম   | প্রগ্রেস, লগ, রিয়েল-টাইম প্রতিক্রিয়া  |

### লক্ষ্য হওয়া মূল পার্থক্যসমূহ

এছাড়াও, কিছু মূল পার্থক্য হলো:

- **যোগাযোগের ধরন:**
  - ক্লাসিক HTTP স্ট্রিমিং: সরল chunked ট্রান্সফার এনকোডিং ব্যবহার করে ডেটা পাঠায়
  - MCP স্ট্রিমিং: JSON-RPC প্রোটোকল সহ কাঠামোবদ্ধ বিজ্ঞপ্তি সিস্টেম ব্যবহার করে

- **মেসেজ ফর্ম্যাট:**
  - ক্লাসিক HTTP: নতুন লাইনের সাথেই প্লেইন টেক্সট টুকরো
  - MCP: মেটাডেটাসহ কাঠামোবদ্ধ LoggingMessageNotification অবজেক্ট

- **ক্লায়েন্ট বাস্তবায়ন:**
  - ক্লাসিক HTTP: সহজ ক্লায়েন্ট যা স্ট্রিমিং রেস্পন্স প্রক্রিয়া করে
  - MCP: আরও উন্নত ক্লায়েন্ট, মেসেজ হ্যান্ডলার সহ যেটি বিভিন্ন ধরণের মেসেজ প্রক্রিয়া করে

- **প্রগ্রেস আপডেট:**
  - ক্লাসিক HTTP: প্রগ্রেস প্রধান রেস্পন্স স্ট্রিমের অংশ
  - MCP: প্রধান রেস্পন্স শেষে আসলেও প্রগ্রেস আলাদা বিজ্ঞপ্তি মেসেজ হিসেবে পাঠানো হয়

### সুপারিশসমূহ

ক্লাসিক্যাল স্ট্রিমিং (যেমন `/stream` এন্ডপয়েন্ট) এবং MCP স্ট্রিমিং এর মধ্যে বেছে নেয়ার সময় কিছু জিনিস আমরা সুপারিশ করি:

- **সরল স্ট্রিমিং প্রয়োজনের জন্য:** ক্লাসিক HTTP স্ট্রিমিং বাস্তবায়নে সহজ এবং মৌলিক স্ট্রিমিং প্রয়োজনের জন্য যথেষ্ট।

- **জটিল, ইন্টারেক্টিভ অ্যাপ্লিকেশনের জন্য:** MCP স্ট্রিমিং আরও কাঠামোবদ্ধ পন্থা দেয়, আরও সমৃদ্ধ মেটাডেটা এবং বিজ্ঞপ্তি ও চূড়ান্ত ফলাফলের পৃথকীকরণ সহ।

- **AI অ্যাপ্লিকেশনের জন্য:** দীর্ঘমেয়াদী AI টাস্কের ক্ষেত্রে MCP-এর বিজ্ঞপ্তি সিস্টেম বিশেষভাবে উপযোগী, যেখানে ব্যবহারকারীকে প্রগ্রেস সম্পর্কে অবহিত রাখতে হয়।

## MCP-তে স্ট্রিমিং

এতদূর আমরা কিছু সুপারিশ এবং তুলনা দেখিয়েছি ক্লাসিক্যাল স্ট্রিমিং ও MCP স্ট্রিমিং এর মধ্যে পার্থক্যের। এবার বিস্তারিত দেখব MCP-তে স্ট্রিমিং কিভাবে কাজে লাগানো যায়।

MCP ফ্রেমওয়ার্কে স্ট্রিমিং কীভাবে কাজ করে বুঝতে পারা গুরুত্বপূর্ণ, যাতে অ্যাপ্লিকেশনগুলো দীর্ঘ সময়ের অপারেশনের সময় রিয়েল-টাইম ফিডব্যাক দিতে পারে।

MCP-তে স্ট্রিমিং মানে প্রধান রেস্পন্স টুকরো করে পাঠানো নয়, বরং একটি টুল একটি অনুরোধ প্রক্রিয়া করার সময় ক্লায়েন্টকে **বিজ্ঞপ্তি** পাঠানো। এই বিজ্ঞপ্তিগুলোতে প্রগ্রেস আপডেট, লগ বা অন্যান্য ইভেন্ট থাকতে পারে।

### কিভাবে কাজ করে

প্রধান ফলাফল এখনও একটি একক রেস্পন্স হিসাবে পাঠানো হয়। তবে, প্রক্রিয়াকরণের সময় বিজ্ঞপ্তিগুলি আলাদা মেসেজ হিসেবে পাঠানো যায় এবং ক্লায়েন্টকে রিয়েল টাইমে আপডেট করে। ক্লায়েন্টকে এসব বিজ্ঞপ্তি গ্রহণ এবং প্রদর্শন করতে সক্ষম হতে হবে।

## বিজ্ঞপ্তি কী?

আমরা বলেছি "বিজ্ঞপ্তি", MCP-এর প্রেক্ষিতে এর অর্থ কী?

একটি বিজ্ঞপ্তি হল সার্ভার থেকে ক্লায়েন্টকে পাঠানো একটি মেসেজ যা দীর্ঘমেয়াদী অপারেশনের সময় প্রগ্রেস, অবস্থা বা অন্যান্য ইভেন্ট সম্পর্কে অবহিত করে। বিজ্ঞপ্তি স্বচ্ছতা বাড়ায় এবং ব্যবহারকারীর অভিজ্ঞতা উন্নত করে।

উদাহরণস্বরূপ, ক্লায়েন্টের উচিত সার্ভারের সাথে প্রাথমিক হ্যান্ডশেক সম্পন্ন হওয়ার পর একটি বিজ্ঞপ্তি পাঠানো।

একটি বিজ্ঞপ্তি JSON মেসেজের মতো দেখতে পারে:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

বিজ্ঞপ্তিগুলো MCP-তে ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging) নামে একটি টপিকের অন্তর্গত।

লগিং কাজ করার জন্য, সার্ভারকে এটি একটি ফিচার/ক্ষমতা হিসেবে সক্ষম করতে হয় এভাবে:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> ব্যবহৃত SDK অনুসারে, লগিং ডিফল্টরূপে সক্ষম থাকতে পারে অথবা আপনাকে স্পষ্টভাবে সার্ভার কনফিগারেশনে এটি সক্ষম করতে হতে পারে।

বিভিন্ন ধরনের বিজ্ঞপ্তি রয়েছে:

| স্তর       | বর্ণনা                       | উদাহরণ ব্যবহারের ক্ষেত্র       |
|-----------|-------------------------------|-------------------------------|
| debug     | বিস্তারিত ডিবাগিং তথ্য          | ফাংশন এন্ট্রি/এক্সিট পয়েন্ট  |
| info      | সাধারণ তথ্যবহুল মেসেজ          | অপারেশনের প্রগ্রেস আপডেট      |
| notice    | স্বাভাবিক কিন্তু গুরুত্বপূর্ণ ইভেন্ট | কনফিগারেশন পরিবর্তন          |
| warning   | সতর্কতা শর্ত                     | বাতিল ফিচার ব্যবহারের অবস্থা  |
| error     | ত্রুটি শর্ত                      | অপারেশন ব্যর্থতা              |
| critical  | গুরুতর শর্ত                    | সিস্টেম কম্পোনেন্ট ব্যর্থতা    |
| alert     | অবিলম্বে পদক্ষেপ গ্রহণ আবশ্যক    | ডেটা ভাঙন সনাক্তকরণ          |
| emergency | সিস্টেম ব্যবহার অযোগ্য         | সম্পূর্ণ সিস্টেম ব্যর্থতা      |

## MCP-তে বিজ্ঞপ্তি বাস্তবায়ন

MCP-তে বিজ্ঞপ্তি বাস্তবায়নের জন্য আপনাকে সার্ভার এবং ক্লায়েন্ট উভয় সারিতে রিয়েল-টাইম আপডেট গ্রহণের ব্যবস্থা করতে হবে। এটি দীর্ঘমেয়াদী অপারেশনের সময় ব্যবহারকারীকে দ্রুত প্রতিক্রিয়া দেওয়ার সুযোগ তৈরি করে।

### সার্ভার পাশ: বিজ্ঞপ্তি প্রেরণ

চলুন সার্ভার পাশ থেকে শুরু করি। MCP-তে আপনি এমন টুল সংজ্ঞায়িত করেন যা অনুরোধ প্রক্রিয়াকরণের সময় বিজ্ঞপ্তি পাঠাতে পারে। সার্ভার `ctx` (সাধারণত) কনটেক্সট অবজেক্ট ব্যবহার করে ক্লায়েন্টকে মেসেজ পাঠায়।

#### পাইথন

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

আগের উদাহরণে, `process_files` টুলটি প্রতিটি ফাইল প্রক্রিয়া করার সময় ক্লায়েন্টকে তিনটি বিজ্ঞপ্তি পাঠায়। `ctx.info()` মেথড তথ্যবহুল মেসেজ পাঠানোর জন্য ব্যবহৃত হয়।

তদুপরি, বিজ্ঞপ্তি সক্ষম করতে নিশ্চিত করুন যে আপনার সার্ভার স্ট্রিমিং ট্রান্সপোর্ট (যেমন `streamable-http`) ব্যবহার করছে এবং ক্লায়েন্ট মেসেজ হ্যান্ডলার বাস্তবায়ন করেছে। সার্ভারে `streamable-http` ট্রান্সপোর্ট ব্যবহার করার উদাহরণ:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

এই .NET উদাহরণে, `ProcessFiles` টুলটি `Tool` অ্যাট্রিবিউট দিয়ে চিহ্নিত এবং প্রতিটি ফাইল প্রক্রিয়াকরণের সময় তিনটি বিজ্ঞপ্তি ক্লায়েন্টকে পাঠায়। `ctx.Info()` মেথড তথ্যবহুল মেসেজ পাঠানোর জন্য ব্যবহৃত হয়।

.NET MCP সার্ভারে বিজ্ঞপ্তি সক্ষম করতে নিশ্চিত করুন স্ট্রিমিং ট্রান্সপোর্ট ব্যবহার করা হচ্ছে:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### ক্লায়েন্ট পাশ: বিজ্ঞপ্তি গ্রহণ

ক্লায়েন্টকে অবশ্যই একটি মেসেজ হ্যান্ডলার বাস্তবায়ন করতে হবে যা আসা বিজ্ঞপ্তিগুলো প্রক্রিয়া এবং প্রদর্শন করবে।

#### পাইথন

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

আগের কোডে, `message_handler` ফাংশন চেক করে আসা মেসেজ বিজ্ঞপ্তি কিনা। যদি হয়, তা মুদ্রণ করে; নয়তো এটি সাধারণ সার্ভার মেসেজ হিসেবে প্রক্রিয়া করে। লক্ষ্য করুন কিভাবে `ClientSession` `message_handler` দিয়ে ইনিশিয়ালাইজ করা হয়েছে বিজ্ঞপ্তি হ্যান্ডলিংয়ের জন্য।

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

এই .NET উদাহরণে, `MessageHandler` ফাংশন চেক করে আসা মেসেজ বিজ্ঞপ্তি কিনা। যদি হয়, তা মুদ্রণ করে; নয়তো এটি সাধারণ সার্ভার মেসেজ হিসেবে প্রক্রিয়া করে। `ClientSession` `ClientSessionOptions` এর মাধ্যমে মেসেজ হ্যান্ডলার দিয়ে ইনিশিয়ালাইজ করা হয়।

বিজ্ঞপ্তি সক্ষম করতে, নিশ্চিত করুন আপনার সার্ভার স্ট্রিমিং ট্রান্সপোর্ট (যেমন `streamable-http`) ব্যবহার করছে এবং ক্লায়েন্ট মেসেজ হ্যান্ডলার বাস্তবায়ন করেছে।

## প্রগ্রেস বিজ্ঞপ্তি ও পরিস্থিতি

এই অংশে MCP-তে প্রগ্রেস বিজ্ঞপ্তির ধারণা, এর প্রয়োজনীয়তা এবং স্ট্রিমেবল HTTP ব্যবহার করে এটি বাস্তবায়ন করা শেখানো হয়েছে। বোঝা দৃঢ় করার জন্য একটি প্রায়োগিক অ্যাসাইনমেন্টও আছে।

প্রগ্রেস বিজ্ঞপ্তি হলো সার্ভার থেকে ক্লায়েন্টকে দীর্ঘ চলমান অপারেশন চলাকালীন রিয়েল-টাইমে পাঠানো মেসেজ। পুরো প্রক্রিয়া শেষ হওয়ার আগে সার্ভার ক্লায়েন্টকে বর্তমান অবস্থা সম্পর্কে আপডেট রাখে। এটি স্বচ্ছতা বাড়ায়, ব্যবহারকারীর অভিজ্ঞতা উন্নত করে এবং ডিবাগিং সহজ করে।

**উদাহরণ:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### কেন প্রগ্রেস বিজ্ঞপ্তি ব্যবহার করবেন?

প্রগ্রেস বিজ্ঞপ্তি কয়েকটি কারণে অপরিহার্য:

- **ভালো ব্যবহারকারীর অভিজ্ঞতা:** কাজ যত এগোয়, ব্যবহারকারী আপডেট দেখতে পায়, শুধু শেষে নয়।
- **রিয়েল-টাইম ফিডব্যাক:** ক্লায়েন্ট প্রগ্রেস বার বা লগ দেখাতে পারে, অ্যাপ যাতে সাড়া দেয় তাই নিশ্চিত করে।
- **সহজ ডিবাগিং ও মনিটরিং:** ডেভেলপার এবং ব্যবহারকারী দেখতে পারে প্রক্রিয়া কোথায় ধীর বা আটকে আছে।

### কিভাবে প্রগ্রেস বিজ্ঞপ্তি বাস্তবায়ন করবেন

MCP-তে প্রগ্রেস বিজ্ঞপ্তি বাস্তবায়ন করার পদ্ধতি:

- **সার্ভারে:** `ctx.info()` অথবা `ctx.log()` ব্যবহার করে প্রতিটি আইটেম প্রক্রিয়াকরণের সময় বিজ্ঞপ্তি পাঠান। এটি মূল ফলাফল প্রস্তুত হওয়ার আগে ক্লায়েন্টকে মেসেজ পাঠায়।
- **ক্লায়েন্টে:** একটি মেসেজ হ্যান্ডলার বাস্তবায়ন করুন যা আসা বিজ্ঞপ্তি শুনে এবং প্রদর্শন করবে। এটি বিজ্ঞপ্তি এবং চূড়ান্ত ফলাফল আলাদাভাবে চিহ্নিত করে।

**সার্ভার উদাহরণ:**

#### পাইথন

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**ক্লায়েন্ট উদাহরণ:**

#### পাইথন

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## নিরাপত্তার বিবেচনা

HTTP-ভিত্তিক ট্রান্সপোর্ট সহ MCP সার্ভার বাস্তবায়নের সময় নিরাপত্তা একটি গুরুত্বপূর্ণ বিষয় যা বিভিন্ন আক্রমণ ভেক্টর এবং সুরক্ষা কৌশল নিয়ে সতর্ক হওয়া প্রয়োজন।

### পর্যালোচনা

HTTP-এর মাধ্যমে MCP সার্ভার প্রকাশ করার সময় নিরাপত্তা অপরিহার্য। স্ট্রিমেবল HTTP নতুন আক্রমণের পথ নিয়ে আসে এবং সতর্ক কনফিগারেশন প্রয়োজন।

### মূল পয়েন্টসমূহ


- **অরিজিন হেডার যাচাইকরণ**: DNS রিবাইন্ডিং আক্রমণ এড়াতে `Origin` হেডার সর্বদা যাচাই করুন।
- **লোকালহোস্ট বাইন্ডিং**: লোকাল ডেভেলপমেন্টের জন্য, সার্ভারগুলোকে `localhost` এ বাইন্ড করুন যাতে সেগুলো পাবলিক ইন্টারনেটে এক্সপোজ না হয়।
- **প্রমাণীকরণ**: প্রোডাকশনে স্থাপনার জন্য প্রমাণীকরণ (যেমন, API কী, OAuth) প্রয়োগ করুন।
- **CORS**: অ্যাক্সেস সীমাবদ্ধ করার জন্য Cross-Origin Resource Sharing (CORS) নীতিমালা কনফিগার করুন।
- **HTTPS**: ট্রাফিক এনক্রিপ্ট করার জন্য প্রোডাকশনে HTTPS ব্যবহার করুন।

### সেরা অনুশীলনসমূহ

- যাচাইকরণ ছাড়া আগত রিকোয়েস্ট কখনো বিশ্বাস করবেন না।
- সব অ্যাক্সেস এবং ত্রুটি লগ এবং মনিটর করুন।
- নিরাপত্তা দুর্বলতা প্যাচ করার জন্য নিয়মিত ডিপেন্ডেন্সি আপডেট করুন।

### চ্যালেঞ্জসমূহ

- সুবিধাজনক উন্নয়নের সঙ্গে নিরাপত্তার সমন্বয় সাধন
- বিভিন্ন ক্লায়েন্ট পরিবেশের সঙ্গে সামঞ্জস্য নিশ্চিত করা

## SSE থেকে Streamable HTTP তে আপগ্রেড

বর্তমানে Server-Sent Events (SSE) ব্যবহারকারী অ্যাপ্লিকেশনগুলোর জন্য, Streamable HTTP তে মাইগ্রেশন আপনার MCP বাস্তবায়নের জন্য উন্নত সক্ষমতা এবং দীর্ঘমেয়াদী টিকে থাকার সুবিধা প্রদান করে।

### কেন আপগ্রেড করবেন?

SSE থেকে Streamable HTTP তে আপগ্রেড করার দুটি প্রধান কারণ রয়েছে:

- Streamable HTTP SSE এর তুলনায় ভালো স্কেলেবিলিটি, সামঞ্জস্য এবং অধিক সমৃদ্ধ বিজ্ঞপ্তি সমর্থন প্রদান করে।
- এটি নতুন MCP অ্যাপ্লিকেশনগুলোর জন্য সুপারিশকৃত ট্রান্সপোর্ট।

### মাইগ্রেশন ধাপসমূহ

আপনার MCP অ্যাপ্লিকেশনগুলোতে SSE থেকে Streamable HTTP তে মাইগ্রেট করার পদ্ধতি নিচে দেওয়া হলো:

- **সার্ভার কোড আপডেট করুন** `mcp.run()` এ `transport="streamable-http"` ব্যবহার করে।
- **ক্লায়েন্ট কোড আপডেট করুন** SSE ক্লায়েন্টের পরিবর্তে `streamablehttp_client` ব্যবহার করুন।
- **ক্লায়েন্টে একটি মেসেজ হ্যান্ডলার বাস্তবায়ন করুন** যা বিজ্ঞপ্তিগুলো প্রক্রিয়া করবে।
- **বিদ্যমান টুল এবং ওয়ার্কফ্লোগুলোর সঙ্গে সামঞ্জস্য পরীক্ষা করুন**।

### সামঞ্জস্য রক্ষা

মাইগ্রেশনের সময় বিদ্যমান SSE ক্লায়েন্টদের সাথে সামঞ্জস্য রক্ষা করার পরামর্শ দেওয়া হয়। কিছু কৌশল হলো:

- আলাদা এন্ডপয়েন্টে SSE এবং Streamable HTTP উভয় ট্রান্সপোর্ট চালিয়ে সমর্থন দিতে পারেন।
- ধীরে ধীরে ক্লায়েন্টদের নতুন ট্রান্সপোর্টে স্থানান্তর করুন।

### চ্যালেঞ্জসমূহ

মাইগ্রেশন সময় নিম্নলিখিত চ্যালেঞ্জগুলোর সমাধান নিশ্চিত করুন:

- নিশ্চিত করুন সব ক্লায়েন্ট আপডেট হয়েছে
- বিজ্ঞপ্তি বিতরণে পার্থক্যগুলি মোকাবেলা করা

## নিরাপত্তা বিবেচনা

কোনও সার্ভার বাস্তবায়নের সময়, বিশেষ করে MCP তে Streamable HTTP এর মতো HTTP-ভিত্তিক ট্রান্সপোর্ট ব্যবহারের সময় নিরাপত্তা সর্বোচ্চ অগ্রাধিকার হওয়া উচিত।

HTTP-ভিত্তিক ট্রান্সপোর্টসহ MCP সার্ভার বাস্তবায়নের সময় নিরাপত্তা একটি প্রধান উদ্বেগ যা বিভিন্ন আক্রমণ ও সুরক্ষা প্রক্রিয়াগুলোর প্রতি মনোযোগ প্রয়োজন।

### সাধারণ বিবরণ

MCP সার্ভারগুলো HTTP এর মাধ্যমে এক্সপোজ করার সময় নিরাপত্তা অত্যন্ত গুরুত্বপূর্ণ। Streamable HTTP নতুন আক্রমণের সুযোগ তৈরি করে এবং সুনির্দিষ্ট কনফিগারেশন দাবি করে।

এখানে কিছু মূল নিরাপত্তা বিবেচনা দেওয়া হলো:

- **অরিজিন হেডার যাচাইকরণ**: DNS রিবাইন্ডিং আক্রমণ এড়াতে `Origin` হেডার সর্বদা যাচাই করুন।
- **লোকালহোস্ট বাইন্ডিং**: লোকাল ডেভেলপমেন্টের জন্য, সার্ভারগুলোকে `localhost` এ বাইন্ড করুন যাতে সেগুলো পাবলিক ইন্টারনেটে এক্সপোজ না হয়।
- **প্রমাণীকরণ**: প্রোডাকশনে স্থাপনার জন্য প্রমাণীকরণ (যেমন, API কী, OAuth) প্রয়োগ করুন।
- **CORS**: অ্যাক্সেস সীমাবদ্ধ করার জন্য Cross-Origin Resource Sharing (CORS) নীতিমালা কনফিগার করুন।
- **HTTPS**: ট্রাফিক এনক্রিপ্ট করার জন্য প্রোডাকশনে HTTPS ব্যবহার করুন।

### সেরা অনুশীলনসমূহ

এছাড়াও, আপনার MCP স্ট্রিমিং সার্ভারে নিরাপত্তা বাস্তবায়নের সময় এখানে কিছু সেরা অনুশীলন দেওয়া হলো:

- যাচাইকরণ ছাড়া আগত রিকোয়েস্ট কখনো বিশ্বাস করবেন না।
- সব অ্যাক্সেস এবং ত্রুটি লগ এবং মনিটর করুন।
- নিরাপত্তা দুর্বলতা প্যাচ করার জন্য নিয়মিত ডিপেন্ডেন্সি আপডেট করুন।

### চ্যালেঞ্জসমূহ

MCP স্ট্রিমিং সার্ভারে নিরাপত্তা বাস্তবায়নের সময় আপনি কিছু চ্যালেঞ্জের সম্মুখীন হবেন:

- সুবিধাজনক উন্নয়নের সঙ্গে নিরাপত্তার সমন্বয় সাধন
- বিভিন্ন ক্লায়েন্ট পরিবেশের সঙ্গে সামঞ্জস্য নিশ্চিত করা

### নির্দিষ্টকরণ: নিজের স্ট্রিমিং MCP অ্যাপ তৈরি করুন

**পরিস্থিতিঃ**
এমন একটি MCP সার্ভার এবং ক্লায়েন্ট তৈরি করুন যেখানে সার্ভার একটি আইটেমের তালিকা (যেমন, ফাইল বা ডকুমেন্ট) প্রক্রিয়া করে এবং প্রতিটি প্রক্রিয়াজাত আইটেমের জন্য একটি বিজ্ঞপ্তি প্রদান করে। ক্লায়েন্ট প্রতিটি বিজ্ঞপ্তি আসার সঙ্গে সঙ্গে প্রদর্শন করবে।

**ধাপসমূহ:**

1. এমন একটি সার্ভার টুল বাস্তবায়ন করুন যা একটি তালিকা প্রক্রিয়া করে এবং প্রতিটি আইটেমের জন্য বিজ্ঞপ্তি পাঠায়।
2. রিয়েল টাইমে বিজ্ঞপ্তি প্রদর্শনের জন্য একটি মেসেজ হ্যান্ডলারসহ একটি ক্লায়েন্ট বাস্তবায়ন করুন।
3. আপনার বাস্তবায়ন পরীক্ষা করতে উভয় সার্ভার ও ক্লায়েন্ট চালান এবং বিজ্ঞপ্তিগুলো পর্যবেক্ষণ করুন।
