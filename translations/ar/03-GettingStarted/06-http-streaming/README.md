# البث عبر HTTPS باستخدام بروتوكول سياق النموذج (MCP)

يقدم هذا الفصل دليلًا شاملاً لتنفيذ البث الآمن والقابل للتوسع والزمني الحقيقي باستخدام بروتوكول سياق النموذج (MCP) عبر HTTPS. يغطي الدافع للبث، وآليات النقل المتاحة، وكيفية تنفيذ HTTP قابل للبث في MCP، أفضل ممارسات الأمان، الهجرة من SSE، والإرشادات العملية لبناء تطبيقات MCP للبث خاصتك.

> **نظرة مستقبلية:** يصف هذا الدرس HTTP القابل للبث ضمن **مواصفة MCP 2025-11-25**، حيث يتم إنشاء جلسة أثناء `initialize` ويتم تثبيتها باستخدام رأس `Mcp-Session-Id`. تزيل النسخة التجريبية `2026-07-28` المصافحة ومعرف الجلسة بالكامل، مما يجعل كل طلب ذاتي المحتوى وقابلًا للتوجيه إلى أي مثيل خادم دون الحاجة إلى جلسات لزجة. انظر [ما الذي يتغير في MCP: النسخة التجريبية 2026-07-28](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md) للتفاصيل.

## آليات النقل والبث في MCP

يستكشف هذا القسم مختلف آليات النقل المتاحة في MCP ودورها في تمكين قدرات البث للاتصال الزمني الحقيقي بين العملاء والخوادم.

### ما هي آلية النقل؟

تحدد آلية النقل كيفية تبادل البيانات بين العميل والخادم. يدعم MCP أنواع نقل متعددة لتناسب بيئات ومتطلبات مختلفة:

- **stdio**: الإدخال/الإخراج القياسي، مناسب للأدوات المحلية والأوامر القائمة على CLI. بسيط لكنه غير مناسب للويب أو السحابة.
- **SSE (الأحداث المرسلة من الخادم)**: يسمح للخوادم بدفع تحديثات فورية للعملاء عبر HTTP. جيد لواجهات الويب، لكنه محدود في القابلية للتوسع والمرونة. اعتبارًا من مواصفة MCP 2025-06-18، تم إهمال نقل SSE المستقل واستبداله بنقل "HTTP القابل للبث".
- **HTTP القابل للبث**: نقل بث حديث معتمد على HTTP، يدعم الإشعارات وقابلية التوسع الأفضل. يُوصى به لمعظم سيناريوهات الإنتاج والسحابة.

### جدول المقارنة

ألقِ نظرة على جدول المقارنة أدناه لفهم الاختلافات بين هذه الآليات:

| النقل             | التحديثات الزمنية الحقيقية | البث       | القابلية للتوسع | حالة الاستخدام          |
|-------------------|----------------------------|------------|-----------------|--------------------------|
| stdio             | لا                         | لا         | منخفض           | أدوات سطر الأوامر المحلية|
| SSE               | نعم                        | نعم        | متوسط           | الويب، التحديثات الزمنية  |
| HTTP القابل للبث  | نعم                        | نعم        | عالي            | السحابة، تعدد العملاء     |

> **نصيحة:** اختيار آلية النقل المناسبة يؤثر على الأداء والقابلية للتوسع وتجربة المستخدم. يُوصى باستخدام **HTTP القابل للبث** للتطبيقات الحديثة والقابلة للتوسع والمهيأة للسحابة.

لاحظ آليات النقل stdio وSSE التي تم عرضها في الفصول السابقة وكيف أن HTTP القابل للبث هو آلية النقل المغطاة في هذا الفصل.

## البث: المفاهيم والدوافع

فهم المفاهيم الأساسية والدوافع وراء البث ضروري لتنفيذ أنظمة اتصال زمني حقيقي فعالة.

**البث** هو تقنية في برمجة الشبكات تتيح إرسال واستقبال البيانات في أجزاء صغيرة ممكن إدارتها أو كتتابع من الأحداث، بدلاً من انتظار كامل الاستجابة لتكون جاهزة. هذا مفيد بشكل خاص لـ:

- الملفات أو مجموعات البيانات الكبيرة.
- التحديثات الزمنية الحقيقية (مثل الدردشة، أشرطة التقدم).
- العمليات الحسابية طويلة الأمد حيث تريد إبقاء المستخدم مطلعًا.

إليك ما تحتاج معرفته عن البث على مستوى عالٍ:

- يتم تسليم البيانات تدريجيًا، وليس دفعة واحدة.
- يمكن للعميل معالجة البيانات فور وصولها.
- يقلل من زمن الاستجابة المدرك ويحسن تجربة المستخدم.

### لماذا نستخدم البث؟

الأسباب لاستخدام البث هي كما يلي:

- يحصل المستخدمون على تغذية راجعة فورًا، وليس فقط في النهاية
- يمكن تمكين التطبيقات الزمنية الحقيقية وواجهات المستخدم التفاعلية
- استخدام أكثر كفاءة لموارد الشبكة والحوسبة

### مثال بسيط: خادم وعميل بث HTTP

إليك مثال بسيط على كيفية تنفيذ البث:

#### بايثون

**الخادم (بايثون، باستخدام FastAPI وStreamingResponse):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**العميل (بايثون، باستخدام requests):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

يوضح هذا المثال خادمًا يرسل سلسلة من الرسائل إلى العميل حالما تصبح متاحة، بدلاً من الانتظار حتى تكون كل الرسائل جاهزة.

**كيف يعمل:**

- يقوم الخادم بإرسال كل رسالة بمجرد جاهزيتها.
- يستقبل العميل كل قطعة ويطبعها فور وصولها.

**المتطلبات:**

- يجب على الخادم استخدام استجابة بث (مثل `StreamingResponse` في FastAPI).
- يجب على العميل معالجة الاستجابة كتيار (`stream=True` في requests).
- نوع المحتوى غالبًا ما يكون `text/event-stream` أو `application/octet-stream`.

#### جافا

**الخادم (جافا، باستخدام Spring Boot وServer-Sent Events):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**العميل (جافا، باستخدام Spring WebFlux WebClient):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**ملاحظات تنفيذ جافا:**

- يستخدم الطبقة التفاعلية في Spring Boot مع `Flux` للبث
- `ServerSentEvent` يوفر بث أحداث منظم بأنواع أحداث
- `WebClient` مع `bodyToFlux()` يمكّن استهلاك بث تفاعلي
- `delayElements()` يحاكي وقت المعالجة بين الأحداث
- يمكن أن يكون للأحداث أنواع (`info`، `result`) لتحسين معالجة العميل

### مقارنة: البث الكلاسيكي مقابل بث MCP

يمكن تصوير الفروقات بين كيفية عمل البث بالطريقة "الكلاسيكية" مقارنة بكيفية عمله في MCP كما يلي:

| الميزة                | البث الكلاسيكي عبر HTTP      | بث MCP (الإشعارات)                |
|-----------------------|-------------------------------|-----------------------------------|
| الاستجابة الأساسية     | متجزئة                       | واحدة، في النهاية                |
| تحديثات التقدم         | تُرسل كقطع بيانات            | تُرسل كإشعارات                   |
| متطلبات العميل         | يجب معالجة التيار             | يجب تنفيذ معالج رسائل            |
| حالة الاستخدام         | ملفات كبيرة، تدفقات رموز AI  | التقدم، السجلات، التغذية الراجعة الزمنية |

### الاختلافات الرئيسية الملحوظة

بالإضافة إلى ذلك، إليك بعض الاختلافات الرئيسية:

- **نمط الاتصال:**
  - البث الكلاسيكي عبر HTTP: يستخدم ترميز نقل مجزأ بسيط لإرسال البيانات على شكل قطع
  - بث MCP: يستخدم نظام إشعارات منظم باستخدام بروتوكول JSON-RPC

- **تنسيق الرسالة:**
  - HTTP الكلاسيكي: قطع نصية عادية مع أسطر جديدة
  - MCP: كائنات LoggingMessageNotification منظمة مع بيانات وصفية

- **تنفيذ العميل:**
  - HTTP الكلاسيكي: عميل بسيط يعالج ردود البث
  - MCP: عميل أكثر تطورًا مع معالج رسائل لمعالجة أنواع مختلفة من الرسائل

- **تحديثات التقدم:**
  - HTTP الكلاسيكي: التقدم جزء من تيار الاستجابة الرئيسي
  - MCP: يتم إرسال التقدم عبر رسائل إشعارات منفصلة بينما تأتي الاستجابة النهائية في النهاية

### التوصيات

هناك بعض الأمور التي نوصي بها عند اختيار تنفيذ البث الكلاسيكي (كنقطة نهاية أظهرناها أعلاه باستخدام `/stream`) مقابل اختيار البث عبر MCP.

- **للحاجات البسيطة للبث:** البث الكلاسيكي عبر HTTP أبسط في التنفيذ وكافٍ للحاجات الأساسية.

- **للتطبيقات المعقدة والتفاعلية:** يوفر بث MCP نهجًا أكثر تنظيمًا مع بيانات وصفية أغنى وفصل بين الإشعارات والنتائج النهائية.

- **لتطبيقات الذكاء الاصطناعي:** نظام إشعارات MCP مفيد بشكل خاص للمهام الطويلة الأمد حيث تريد إبقاء المستخدمين مطلعين على التقدم.

## البث في MCP

حسنًا، بعد أن رأيت بعض التوصيات والمقارنات حول الفرق بين البث الكلاسيكي والبث في MCP، دعنا نغوص في التفاصيل حول كيفية الاستفادة من البث في MCP.

فهم كيفية عمل البث ضمن إطار MCP ضروري لبناء تطبيقات تفاعلية تقدم تغذية راجعة زمنية حقيقية للمستخدمين أثناء العمليات الطويلة.

في MCP، لا يتعلق البث بإرسال الاستجابة الرئيسية على شكل قطع، بل بإرسال **إشعارات** إلى العميل أثناء معالجة أداة للطلب. يمكن أن تتضمن هذه الإشعارات تحديثات التقدم، السجلات، أو أحداث أخرى.

### كيف يعمل

لا تزال النتيجة الرئيسية تُرسل كرد واحد. مع ذلك، يمكن إرسال الإشعارات كرسائل منفصلة أثناء المعالجة مما يحدث العميل في الزمن الحقيقي. يجب أن يكون العميل قادرًا على التعامل مع هذه الإشعارات وعرضها.

## ما هي الإشعار؟

قلنا "إشعار"، ماذا يعني ذلك في سياق MCP؟

الإشعار هو رسالة ترسل من الخادم إلى العميل لإبلاغه بالتقدم، الحالة، أو أحداث أخرى أثناء عملية طويلة المدى. تحسن الإشعارات الشفافية وتجربة المستخدم.

على سبيل المثال، يفترض أن يرسل العميل إشعارًا بمجرد الانتهاء من المصافحة الأولية مع الخادم.

يبدو الإشعار كالرسالة JSON التالية:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

تنتمي الإشعارات إلى موضوع في MCP يُشار إليه باسم ["التسجيل"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging).

لكي يعمل التسجيل، يحتاج الخادم إلى تمكينه كميزة/قدرة كما يلي:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> اعتمادًا على الـ SDK المستخدم، قد يتم تمكين التسجيل افتراضيًا، أو قد تحتاج إلى تمكينه صراحة في إعدادات الخادم.

هناك أنواع مختلفة من الإشعارات:

| المستوى    | الوصف                         | مثال حالة الاستخدام          |
|------------|-------------------------------|------------------------------|
| debug      | معلومات تفصيلية لتصحيح الأخطاء | نقاط دخول/خروج الدوال       |
| info       | رسائل معلومات عامة           | تحديثات تقدم العملية         |
| notice     | أحداث طبيعية لكنها هامة      | تغييرات التكوين              |
| warning    | حالات تحذير                  | استخدام ميزة مهملة            |
| error      | حالات خطأ                    | فشل العملية                  |
| critical   | حالات حرجة                   | فشل مكونات النظام             |
| alert      | يجب اتخاذ إجراء فوري          | اكتشاف تلف بيانات            |
| emergency  | النظام غير قابل للاستخدام   | فشل كامل للنظام               |

## تنفيذ الإشعارات في MCP

لتنفيذ الإشعارات في MCP، تحتاج إلى إعداد جانبي الخادم والعميل للتعامل مع التحديثات الزمنية الحقيقية. هذا يسمح لتطبيقك بتوفير تغذية راجعة فورية للمستخدمين أثناء العمليات الطويلة.

### جانب الخادم: إرسال الإشعارات

لنبدأ مع جانب الخادم. في MCP، تعرف الأدوات التي يمكنها إرسال إشعارات أثناء معالجة الطلبات. يستخدم الخادم كائن السياق (عادة `ctx`) لإرسال الرسائل إلى العميل.

#### بايثون

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

في المثال السابق، ترسل أداة `process_files` ثلاث إشعارات إلى العميل أثناء معالجة كل ملف. تُستخدم طريقة `ctx.info()` لإرسال رسائل معلوماتية.

بالإضافة إلى ذلك، لتمكين الإشعارات، تأكد أن خادمك يستخدم نقل بث (مثل `streamable-http`) وأن عميلك ينفذ معالج رسائل لمعالجة الإشعارات. إليك كيفية إعداد الخادم لاستخدام نقل `streamable-http`:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

في هذا المثال على .NET، تُزين أداة `ProcessFiles` بوسم `Tool` وترسل ثلاث إشعارات إلى العميل أثناء معالجة كل ملف. تُستخدم طريقة `ctx.Info()` لإرسال رسائل معلوماتية.

لتمكين الإشعارات في خادم MCP الخاص بك على .NET، تأكد من استخدام نقل بث:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### جانب العميل: استقبال الإشعارات

يجب على العميل تنفيذ معالج رسائل لمعالجة وعرض الإشعارات فور وصولها.

#### بايثون

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

في الشيفرة السابقة، يتحقق دالة `message_handler` مما إذا كانت الرسالة الواردة إشعارًا. إذا كانت كذلك، يقوم بطباعة الإشعار؛ وإلا يعالجه كرسالة خادم عادية. لاحظ أيضًا كيف يتم تهيئة `ClientSession` مع `message_handler` لمعالجة الإشعارات الواردة.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

في هذا المثال على .NET، يتحقق دالة `MessageHandler` مما إذا كانت الرسالة الواردة إشعارًا. إذا كانت كذلك، يطبع الإشعار؛ وإلا يعالجه كرسالة خادم عادية. يتم تهيئة `ClientSession` مع معالج الرسائل عبر `ClientSessionOptions`.

لتمكين الإشعارات، تأكد من استخدام خادمك نقل بث (مثل `streamable-http`) وأن عميلك ينفذ معالج رسائل لمعالجة الإشعارات.

## إشعارات التقدم والسيناريوهات

يشرح هذا القسم مفهوم إشعارات التقدم في MCP، لماذا هي مهمة، وكيفية تنفيذها باستخدام HTTP القابل للبث. ستجد أيضًا مهمة عملية لتعزيز فهمك.

إشعارات التقدم هي رسائل زمنية حقيقية تُرسل من الخادم إلى العميل أثناء العمليات الطويلة. بدلاً من انتظار اكتمال العملية بأكملها، يبقي الخادم العميل على اطلاع بالحالة الحالية. هذا يحسن الشفافية، تجربة المستخدم، ويسهل التصحيح.

**مثال:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### لماذا نستخدم إشعارات التقدم؟

إشعارات التقدم ضرورية لعدة أسباب:

- **تجربة مستخدم أفضل:** يرى المستخدمون التحديثات أثناء تقدم العمل، وليس فقط في النهاية.
- **تغذية راجعة زمنية حقيقية:** يمكن للعملاء عرض أشرطة التقدم أو السجلات، مما يجعل التطبيق يبدو تفاعليًا.
- **تصحيح ومراقبة أسهل:** يمكن للمطورين والمستخدمين رؤية أين قد يكون الإجراء بطيئًا أو عالقًا.

### كيفية تنفيذ إشعارات التقدم

إليك كيفية تنفيذ إشعارات التقدم في MCP:

- **على الخادم:** استخدم `ctx.info()` أو `ctx.log()` لإرسال الإشعارات مع معالجة كل عنصر. يرسل هذا رسالة إلى العميل قبل أن تكون النتيجة الرئيسية جاهزة.
- **على العميل:** نفذ معالج رسائل يستمع ويعرض الإشعارات فور وصولها. يميز هذا المعالج بين الإشعارات والنتيجة النهائية.

**مثال الخادم:**

#### بايثون

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**مثال العميل:**

#### بايثون

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## اعتبارات الأمان

عند تنفيذ خوادم MCP باستخدام نقل HTTP، يصبح الأمان مسألة أساسية تتطلب اهتمامًا دقيقًا بعدة نواحي هجوم وآليات حماية.

### نظرة عامة

الأمان أمر حيوي عند تعريض خوادم MCP عبر HTTP. يقدم HTTP القابل للبث أسطح هجوم جديدة ويتطلب تكوينًا دقيقًا.

### نقاط رئيسية


- **التحقق من رأس الأصل**: تحقق دائمًا من رأس `Origin` لمنع هجمات إعادة ربط DNS.
- **الربط بالمضيف المحلي**: للتطوير المحلي، اربط الخوادم بـ `localhost` لتجنب عرضها على الإنترنت العام.
- **المصادقة**: نفّذ المصادقة (مثل مفاتيح API، OAuth) للنشر في بيئات الإنتاج.
- **CORS**: قم بتكوين سياسات مشاركة الموارد عبر الأصول (CORS) لتقييد الوصول.
- **HTTPS**: استخدم HTTPS في بيئة الإنتاج لتشفير حركة المرور.

### أفضل الممارسات

- لا تثق أبدًا في الطلبات الواردة دون تحقق.
- سجّل وراقب كافة محاولات الوصول والأخطاء.
- حدّث التبعيات بانتظام لسد الثغرات الأمنية.

### التحديات

- التوازن بين الأمان وسهولة التطوير
- ضمان التوافق مع بيئات العميل المختلفة

## الترقية من SSE إلى Streamable HTTP

بالنسبة للتطبيقات التي تستخدم حاليًا Server-Sent Events (SSE)، فإن الترحيل إلى Streamable HTTP يوفر قدرات محسّنة واستدامة أفضل على المدى الطويل لتطبيقات MCP الخاصة بك.

### لماذا الترقية؟

هناك سببين قويين للترقية من SSE إلى Streamable HTTP:

- Streamable HTTP يقدم قابلية توسع أفضل، وتوافق أعلى، ودعم إعلام أكثر ثراءً مقارنة بـ SSE.
- وهو النقل الموصى به لتطبيقات MCP الجديدة.

### خطوات الترحيل

إليك كيفية الترحيل من SSE إلى Streamable HTTP في تطبيقات MCP الخاصة بك:

- **قم بتحديث كود الخادم** لاستخدام `transport="streamable-http"` في `mcp.run()`.
- **قم بتحديث كود العميل** لاستخدام `streamablehttp_client` بدلًا من عميل SSE.
- **نفّذ معالج رسائل** في العميل لمعالجة الإشعارات.
- **اختبر التوافق** مع الأدوات وسير العمل الموجودة.

### الحفاظ على التوافق

من المستحسن الحفاظ على التوافق مع عملاء SSE الحاليين أثناء عملية الترحيل. إليك بعض الاستراتيجيات:

- يمكنك دعم كل من SSE و Streamable HTTP عن طريق تشغيل كلا النقلين على نقاط نهاية مختلفة.
- قم بترحيل العملاء تدريجيًا إلى النقل الجديد.

### التحديات

تأكد من معالجة التحديات التالية أثناء الترحيل:

- ضمان تحديث جميع العملاء
- التعامل مع الاختلافات في توصيل الإشعارات

## اعتبارات الأمان

يجب أن يكون الأمان أولوية قصوى عند تنفيذ أي خادم، وخصوصًا عند استخدام نقلات HTTP مثل Streamable HTTP في MCP.

عند تنفيذ خوادم MCP باستخدام نقلات معتمدة على HTTP، يصبح الأمان مسألة بالغة الأهمية تتطلب اهتمامًا دقيقًا بعدة نواحي للهجوم وآليات الحماية.

### نظرة عامة

الأمان أمر حيوي عند كشف خوادم MCP عبر HTTP. يضيف Streamable HTTP أسطح هجوم جديدة ويتطلب تهيئة دقيقة.

فيما يلي بعض اعتبارات الأمان الرئيسية:

- **التحقق من رأس الأصل**: تحقق دائمًا من رأس `Origin` لمنع هجمات إعادة ربط DNS.
- **الربط بالمضيف المحلي**: للتطوير المحلي، اربط الخوادم بـ `localhost` لتجنب عرضها على الإنترنت العام.
- **المصادقة**: نفّذ المصادقة (مثل مفاتيح API، OAuth) للنشر في بيئات الإنتاج.
- **CORS**: قم بتكوين سياسات مشاركة الموارد عبر الأصول (CORS) لتقييد الوصول.
- **HTTPS**: استخدم HTTPS في بيئة الإنتاج لتشفير حركة المرور.

### أفضل الممارسات

بالإضافة إلى ذلك، إليك بعض أفضل الممارسات التي يجب اتباعها عند تنفيذ الأمان في خادم بث MCP الخاص بك:

- لا تثق أبدًا في الطلبات الواردة دون تحقق.
- سجّل وراقب كافة محاولات الوصول والأخطاء.
- حدّث التبعيات بانتظام لسد الثغرات الأمنية.

### التحديات

ستواجه بعض التحديات عند تنفيذ الأمان في خوادم بث MCP:

- التوازن بين الأمان وسهولة التطوير
- ضمان التوافق مع بيئات العميل المختلفة

### المهمة: بناء تطبيق MCP للبث الخاص بك

**السيناريو:**
بناء خادم MCP وعميل حيث يعالج الخادم قائمة عناصر (مثل ملفات أو مستندات) ويرسل إشعارًا لكل عنصر تمت معالجته. يجب أن يعرض العميل كل إشعار فور وصوله.

**الخطوات:**

1. نفّذ أداة خادم تعالج القائمة وترسل إشعارات لكل عنصر.
2. نفّذ عميلًا بمعالج رسائل لعرض الإشعارات في الوقت الفعلي.
3. اختبر تنفيذك بتشغيل كل من الخادم والعميل ومراقبة الإشعارات.
