# Урок: Създаване на MCP сървър за уеб търсене

Тази глава демонстрира как да изградите реален AI агент, който се интегрира с външни API-та, обработва различни типове данни, управлява грешки и оркестрира няколко инструмента — всичко това в продукционно готов формат. Ще видите:

- **Интеграция с външни API-та, изискващи удостоверяване**
- **Обработка на разнообразни типове данни от множество крайни точки**

В края на урока ще имате практическо изживяване с шаблони и най-добри практики, които са съществени за напреднали AI и приложения, захранвани от LLM.

В този урок ще научите как да изградите напреднал MCP сървър и клиент, който разширява възможностите на LLM с данни от уеб в реално време чрез SerpAPI. Това е критично умение за разработване на динамични AI агенти, които могат да достъпват актуална информация от уеб.

### Python

```python
# Пример за използване на инструмента general_search
from mcp import ClientSession, StdioServerParameters
from mcp.client.stdio import stdio_client

async def run_search():
    server_params = StdioServerParameters(
        command="python",
        args=["server.py"],
    )
    async with stdio_client(server_params) as (reader, writer):
        async with ClientSession(reader, writer) as session:
            await session.initialize()
            result = await session.call_tool("general_search", arguments={"query": "open source LLMs"})
            print(result)
```

- **Интерактивен режим**: Стартира меню-базиран интерфейс, където може ръчно да избирате и извиквате инструменти, въвеждате собствени заявки и виждате резултати в реално време. Това е идеално за изследване на възможностите на сървъра и експериментиране с различни входове.

## Напреднали концепции в този урок

Преди да започнете да изграждате, ето някои важни напреднали концепции, които ще се появят през тази глава. Разбирането им ще ви помогне да се ориентирате, дори ако сте нови за тях:

- **Оркестрация на множество инструменти**: Това означава стартиране на няколко различни инструмента (като уеб търсене, новини, търсене на продукти и Q&A) в един MCP сървър. Позволява на сървъра да обработва разнообразни задачи, а не само една.
- **Управление на ограничения на честотата на API**: Много външни API-та (като SerpAPI) ограничават колко заявки можете да правите за определен период. Добър код проверява тези ограничения и ги управлява грациозно, така че приложението ви да не спира, ако удари граница.
- **Парсиране на структурирани данни**: Отговорите от API често са сложни и вложени. Тази концепция означава преобразуване на тези отговори в чисти, лесни за използване формати, приятелски към LLM или други програми.
- **Възстановяване при грешки**: Понякога нещата се объркват — може да има мрежова грешка или API да не върне очакваното. Възстановяването при грешки означава вашият код да се справя с тези проблеми и все пак да дава полезна обратна връзка, без да спира.
- **Валидация на параметри**: Това е свързано с проверка, че всички входове към вашите инструменти са правилни и безопасни за ползване. Включва задаване на подразбиращи се стойности и гарантиране, че типовете са верни, което помага да се избегнат бъгове и обърквания.

Тази секция ще ви помогне да диагностицирате и решите често срещани проблеми, които може да срещнете при работа с MCP сървъра за уеб търсене. Ако се натъкнете на грешки или неочаквано поведение, тези съвети често помагат бързо да разрешите повечето проблеми, преди да потърсите допълнителна помощ.

## Отстраняване на проблеми

При работа с MCP сървъра за уеб търсене понякога може да възникнат проблеми — това е нормално при разработка с външни API-та и нови инструменти. Тази секция предоставя практични решения на най-често срещаните проблеми, за да се върнете бързо в работен ритъм. Ако срещнете грешка, започнете от тук: долните съвети адресират проблемите, с които повечето потребители се сблъскват, и често могат да разрешат проблема без допълнителна помощ.

### Често срещани проблеми

По-долу са някои от най-често срещаните проблеми с ясни обяснения и стъпки за разрешаване:

1. **Липсва KEY SERPAPI в .env файла**
   - Ако видите грешка `SERPAPI_KEY environment variable not found`, това означава, че приложението ви не може да намери API ключа за достъп до SerpAPI. За да го оправите, създайте файл `.env` в основната директория на проекта (ако не съществува) и добавете ред като `SERPAPI_KEY=your_serpapi_key_here`. Заменете `your_serpapi_key_here` с вашия реален ключ от сайта на SerpAPI.

2. **Грешки като „Module not found“**
   - Грешки като `ModuleNotFoundError: No module named 'httpx'` показват, че липсва необходим Python пакет. Това обикновено става, ако не сте инсталирали всичките зависимости. За да разрешите това, изпълнете `pip install -r requirements.txt` в терминала, за да инсталирате всичко необходимо.

3. **Проблеми с връзката**
   - Ако получите грешка като `Error during client execution`, това често означава, че клиентът не може да се свърже със сървъра, или сървърът не работи както се очаква. Проверете внимателно дали клиентът и сървърът са съвместими версии и дали `server.py` съществува и работи в правилната директория. Рестартирането и на двата компонента също помага.

4. **Грешки от SerpAPI**
   - Ако видите `Search API returned error status: 401`, това означава, че вашият SerpAPI ключ липсва, е неправилен или е изтекъл. Отидете в таблото на SerpAPI, потвърдете ключа си и актуализирайте `.env` файла, ако е необходимо. Ако ключът е верен, но грешката продължава, проверете дали безплатният ви план не е изчерпал квотата.

Забележете как DEBUG режим включва допълнителни редове за HTTP заявки, отговори и други вътрешни детайли. Това е много полезно за отстраняване на проблеми.

За да активирате DEBUG режим, задайте ниво на логване DEBUG в горната част на вашия `client.py` или `server.py`:

# [Python](#tab/python-debug)


```python
# В горната част на вашия client.py или server.py
import logging
logging.basicConfig(
    level=logging.DEBUG,  # Променете от INFO на DEBUG
    format="%(asctime)s - %(name)s - %(levelname)s - %(message)s"
)
```
