## Транспортни механизми и поточно предаване в MCP

Този раздел разглежда различните транспортни механизми, налични в MCP, и тяхната роля за осигуряване на възможности за поточно предаване за комуникация в реално време между клиенти и сървъри.

### Какво е транспортен механизъм?

Транспортният механизъм дефинира как се обменят данните между клиента и сървъра. MCP поддържа няколко типа транспорт, за да паснат на различни среди и изисквания:

- **stdio**: Стандартен вход/изход, подходящ за локални и CLI-базирани инструменти. Проста, но неподходяща за уеб или облак.
- **SSE (Server-Sent Events)**: Позволява на сървърите да изпращат ъпдейти в реално време към клиентите по HTTP. Подходящ за уеб интерфейси, но ограничен по мащабируемост и гъвкавост. От MCP Спецификация 2025-06-18 отделният SSE транспорт е остарял и заменен с "Поточно HTTP" транспорт.
- **Поточно HTTP**: Модерен HTTP-базиран транспорт за поточно предаване, поддържащ известия и по-добра мащабируемост. Препоръчва се за повечето продукционни и облачни сценарии.

### Таблица за сравнение

Вижте таблицата за сравнение по-долу, за да разберете разликите между тези транспортни механизми:

| Транспорт       | Обновления в реално време | Поточно | Мащабируемост | Приложение               |
|-----------------|----------------------------|---------|---------------|-------------------------|
| stdio           | Не                         | Не      | Ниска         | Локални CLI инструменти |
| SSE             | Да                         | Да      | Средна        | Уеб, обновления в реално време |
| Поточно HTTP   | Да                         | Да      | Висока        | Облак, многоклиентски   |

> **Подсказка:** Изборът на правилен транспорт влияе на производителността, мащабируемостта и потребителското изживяване. **Поточно HTTP** се препоръчва за модерни, мащабируеми и готови за облак приложения.

Обърнете внимание на транспортите stdio и SSE, които видяхте в предишните глави, и как поточното HTTP е транспортът, разгледан в тази глава.

## Поточно предаване: Концепции и мотивация

Разбирането на основните концепции и мотиви зад поточното предаване е от съществено значение за реализиране на ефективни системи за комуникация в реално време.

**Поточното предаване** е техника в мрежовото програмиране, която позволява данните да се изпращат и получават на малки, управляеми парчета или като последователност от събития, вместо да се чака цял отговор да е готов. Това е особено полезно за:

- Големи файлове или набори от данни.
- Ъпдейти в реално време (например чат, ленти за прогрес).
- Дълги изчисления, при които искате да държите потребителя информиран.

Ето какво трябва да знаете за поточното предаване на високо ниво:

- Данните се доставят постепенено, не наведнъж.
- Клиентът може да обработва данните веднага щом пристигнат.
- Намалява възприеманата латентност и подобрява потребителското изживяване.

### Защо да използваме поточно предаване?

Причините да използвате поточно предаване са следните:

- Потребителите получават обратна връзка веднага, не само в края.
- Позволява реално време приложения и отзивчиви интерфейси.
- По-ефикасно използване на мрежовите и изчислителни ресурси.

### Пример: Поточно HTTP сървър и клиент

Ето един прост пример как може да се реализира поточно предаване:

#### Python

**Сървър (Python, използвайки FastAPI и StreamingResponse):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**Клиент (Python, използвайки requests):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

Този пример демонстрира как сървърът изпраща серия от съобщения към клиента, когато станат налични, вместо да чака всички съобщения да са готови.

**Как работи:**

- Сървърът връща всяко съобщение, веднага щом е готово.
- Клиентът получава и отпечатва всяка част веднага щом пристигне.

**Изисквания:**

- Сървърът трябва да използва поточен отговор (напр. `StreamingResponse` във FastAPI).
- Клиентът трябва да обработва отговора като поток (`stream=True` в requests).
- Content-Type обикновено е `text/event-stream` или `application/octet-stream`.

#### Java

**Сървър (Java, използвайки Spring Boot и Server-Sent Events):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**Клиент (Java, използвайки Spring WebFlux WebClient):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**Бележки за Java имплементация:**

- Използва реактивния стек на Spring Boot с `Flux` за поточно предаване
- `ServerSentEvent` предоставя структурирано поточно събитие с типове събития
- `WebClient` с `bodyToFlux()` позволява реактивна консумация на потока
- `delayElements()` симулира време за обработка между събитията
- Събитията могат да имат типове (`info`, `result`) за по-добра обработка от клиента

### Сравнение: Класическо поточно предаване срещу MCP поточно предаване

Разликите между класическото поточно предаване и това в MCP могат да се представят така:

| Функция                | Класическо HTTP Поточно предаване | MCP Поточно предаване (Известия)   |
|------------------------|---------------------------------|------------------------------------|
| Основен отговор        | Частичен (chunked)               | Един-единствен, в края             |
| Ъпдейти за напредък    | Изпращат се като части данни    | Изпращат се като известия          |
| Изисквания към клиента | Трябва да обработва потока      | Трябва да имплементира обработчик на съобщения |
| Приложение             | Големи файлове, AI токен потоци | Напредък, логове, обратна връзка в реално време |

### Наблюдавани ключови разлики

Освен това, ето някои ключови различия:

- **Комуникационен модел:**
  - Класическо HTTP: Използва прост chunked трансфер, за да изпраща данни на части
  - MCP: Използва структурирана система за известия с JSON-RPC протокол

- **Формат на съобщенията:**
  - Класическо HTTP: Текстови части с нови редове
  - MCP: Структурирани LoggingMessageNotification обекти с метаданни

- **Имплементация на клиента:**
  - Класическо HTTP: Прост клиент, който обработва поточни отговори
  - MCP: По-сложен клиент с обработчик на съобщения за обработка на различни типове съобщения

- **Ъпдейти за напредък:**
  - Класическо HTTP: Напредъкът е част от основния поточен отговор
  - MCP: Напредъкът се изпраща чрез отделни известия, докато основният отговор идва в края

### Препоръки

Има някои препоръки при избора между класическо поточно предаване (като крайна точка, показана по-горе с `/stream`) и поточно предаване чрез MCP.

- **За прости поточни нужди:** Класическото HTTP поточно предаване е по-лесно за реализиране и достатъчно за основни нужди.

- **За сложни, интерактивни приложения:** MCP поточното предаване предоставя по-структуриран подход с по-богати метаданни и разделяне на известия и крайни резултати.

- **За AI приложения:** Системата за известия в MCP е особено полезна при дългосрочни AI задачи, където искате да държите потребителите информирани за напредъка.

## Поточно предаване в MCP

Добре, вече видяхте някои препоръки и сравнения за разликите между класическото поточно предаване и това в MCP. Нека разгледаме в детайли как точно можете да използвате поточно предаване в MCP.

Разбирането как работи поточното предаване в рамките на MCP е от съществено значение за изграждане на отзивчиви приложения, които предоставят обратна връзка на потребителите в реално време по време на дългосрочни операции.

В MCP поточното предаване не е просто изпращане на основния отговор на части, а изпращане на **известия** към клиента, докато инструмент обработва заявка. Тези известия могат да включват ъпдейти за напредък, логове или други събития.

### Как работи

Основният резултат все още се изпраща като единичен отговор. Въпреки това, известията могат да се изпращат като отделни съобщения по време на обработката, като по този начин актуализират клиента в реално време. Клиентът трябва да може да обработва и показва тези известия.

## Какво е известие?

Казахме "известие", какво означава това в контекста на MCP?

Известието е съобщение, изпратено от сървъра към клиента, за да информира за напредък, статус или други събития по време на дългосрочна операция. Известията повишават прозрачността и подобряват потребителското изживяване.

Например, клиентът трябва да изпрати известие, след като първоначалното ръкостискане със сървъра е направено.

Известието изглежда така като JSON съобщение:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

Известието принадлежи към тема в MCP, наречена ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging).

За да работи логването, сървърът трябва да го активира като функция/възможност така:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> В зависимост от използвания SDK, логването може да е активирано по подразбиране, или може да се наложи да го активирате изрично в конфигурацията на сървъра.

Има различни типове известия:

| Ниво     | Описание                       | Примерно приложение          |
|-----------|-------------------------------|-----------------------------|
| debug     | Подробна отстраняваща грешки информация | Точки на влизане/излизане от функция |
| info      | Общи информационни съобщения  | Ъпдейти за напредъка на операцията  |
| notice    | Нормални, но значими събития | Промени в конфигурацията    |
| warning   | Условия за предупреждение     | Използване на остаряла функция |
| error     | Състояния на грешка           | Провали при операция         |
| critical  | Критични състояния            | Провали на системни компоненти |
| alert     | Трябва незабавно действие      | Открита корупция на данни   |
| emergency | Системата е неизползваема     | Пълна системна повреда      |

## Имплементиране на известия в MCP

За да имплементирате известия в MCP, трябва да подготвите както сървърната, така и клиентската страна да обработват ъпдейти в реално време. Това позволява на вашето приложение да предоставя незабавна обратна връзка на потребителите по време на дългосрочни операции.

### От страна на сървъра: Изпращане на известия

Нека започнем със сървърната страна. В MCP дефинирате инструменти, които могат да изпращат известия по време на обработка на заявки. Сървърът използва контекстния обект (обикновено `ctx`), за да изпраща съобщения към клиента.

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

В предишния пример инструментът `process_files` изпраща три известия към клиента, докато обработва всеки файл. Методът `ctx.info()` се използва за изпращане на информационни съобщения.

Освен това, за да активирате известията, уверете се, че сървърът използва потоков транспорт (като `streamable-http`) и клиентът имплементира обработчик на съобщения за обработка на известия. Ето как да настроите сървъра да използва транспорт `streamable-http`:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

В този пример с .NET инструментът `ProcessFiles` е маркиран с атрибут `Tool` и изпраща три известия към клиента, докато обработва всеки файл. Методът `ctx.Info()` се използва за изпращане на информационни съобщения.

За да активирате известия в .NET MCP сървър, уверете се, че използвате потоков транспорт:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### От страна на клиента: Получаване на известия

Клиентът трябва да имплементира обработчик на съобщения, който да обработва и показва известията веднага щом пристигнат.

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

В кода по-горе функцията `message_handler` проверява дали входящото съобщение е известие. Ако е, то го отпечатва; в противен случай го обработва като обикновено сървърно съобщение. Обърнете внимание и как `ClientSession` се инициализира с `message_handler` за обработка на входящи известия.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

В този пример с .NET функцията `MessageHandler` проверява дали входящото съобщение е известие. Ако е, отпечатва го; иначе го обработва като обикновено сървърно съобщение. `ClientSession` се инициализира с обработчика на съобщения чрез `ClientSessionOptions`.

За да активирате известия, уверете се, че сървърът използва потоков транспорт (като `streamable-http`), а клиентът имплементира обработчик за известия.

## Известия за напредък и сценарии

Този раздел обяснява концепцията за известия за напредък в MCP, защо са важни и как да ги реализирате чрез Поточно HTTP. Ще намерите и практически упражнения за затвърждаване на знанията.

Известията за напредък са съобщения в реално време, изпратени от сървъра към клиента по време на дълги операции. Вместо да чака цялата операция да завърши, сървърът държи клиента актуализиран за текущото състояние. Това повишава прозрачността, потребителското изживяване и улеснява отстраняването на проблеми.

**Пример:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### Защо да използваме известия за напредък?

Известията за напредък са важни по няколко причини:

- **По-добро потребителско изживяване:** Потребителите виждат ъпдейти във времето на работа, а не само накрая.
- **Обратна връзка в реално време:** Клиентите могат да показват ленти за прогрес или логове, правейки приложението отзивчиво.
- **По-лесно отстраняване на грешки и мониторинг:** Разработчиците и потребителите могат да видят къде процесът се забавя или засяда.

### Как да имплементираме известия за напредък

Ето как да имплементирате известия за напредък в MCP:

- **На сървъра:** Използвайте `ctx.info()` или `ctx.log()` за изпращане на известия, когато всеки елемент се обработва. Това изпраща съобщение към клиента преди основния резултат да е готов.
- **На клиента:** Имплементирайте обработчик на съобщения, който слуша и показва известия веднага щом пристигнат. Този обработчик разграничава известията от крайния резултат.

**Пример сървър:**

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**Пример клиент:**

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## Надграждане от SSE към Streamable HTTP

За приложения, които в момента използват Server-Sent Events (SSE), мигрирането към Streamable HTTP осигурява разширени възможности и по-добра дългосрочна устойчивост за вашите MCP реализации.

### Защо да надграждате?

Има две убедителни причини да надградите от SSE към Streamable HTTP:

- Streamable HTTP предлага по-добра мащабируемост, съвместимост и по-богата поддръжка на известия от SSE.
- Препоръчва се като транспорт за нови MCP приложения.

### Стъпки за миграция

Ето как можете да мигрирате от SSE към Streamable HTTP в MCP приложенията си:

- **Актуализирайте сървърния код** да използва `transport="streamable-http"` в `mcp.run()`.
- **Актуализирайте клиентския код** да използва `streamablehttp_client` вместо SSE клиент.
- **Имплементирайте обработчик на съобщения** в клиента за обработка на известията.
- **Тествайте за съвместимост** с наличните инструменти и работни потоци.

### Поддържане на съвместимост

Препоръчва се да поддържате съвместимост с наличните SSE клиенти по време на процеса на миграция. Ето някои стратегии:

- Можете да поддържате както SSE, така и Streamable HTTP като пускате и двата транспорта на различни крайни точки.
- Постепенно мигрирайте клиентите към новия транспорт.

### Предизвикателства

Уверете се, че решавате следните предизвикателства по време на миграцията:

- Осигуряване на обновления на всички клиенти
- Обработка на разлики в доставката на известия
