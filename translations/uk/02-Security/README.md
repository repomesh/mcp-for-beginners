# Безпека MCP: Всеохоплюючий захист систем ШІ

[![MCP Security Best Practices](../../../translated_images/uk/03.175aed6dedae133f.webp)](https://youtu.be/88No8pw706o)

_(Натисніть на зображення вище, щоб переглянути відео цього уроку)_

Безпека є фундаментальною для проєктування систем ШІ, саме тому ми приділяємо їй пріоритет у другому розділі. Це відповідає принципу Microsoft **Secure by Design** із [Ініціативи Secure Future](https://www.microsoft.com/security/blog/2025/04/17/microsofts-secure-by-design-journey-one-year-of-success/).

Протокол контексту моделі (MCP) надає потужні нові можливості для AI-застосунків і водночас створює унікальні виклики безпеки, що виходять за межі традиційних ризиків програмного забезпечення. Системи MCP стикаються як із загальновідомими проблемами безпеки (безпечне кодування, найменші привілеї, безпека ланцюга постачання), так і з новими загрозами специфічними для ШІ, такими як ін’єкція підказок, отруєння інструментів, захоплення сесій, атаки confused deputy, вразливості токенів passthrough і динамічна модифікація можливостей.

Цей урок досліджує найкритичніші ризики безпеки реалізацій MCP — охоплюючи автентифікацію, авторизацію, надмірні дозволи, непряму ін’єкцію підказок, безпеку сесій, проблеми confused deputy, керування токенами та вразливості ланцюга постачання. Ви дізнаєтесь практичні заходи контролю та найкращі практики для пом'якшення цих ризиків, використовуючи рішення Microsoft, такі як Prompt Shields, Azure Content Safety та GitHub Advanced Security для зміцнення вашого впровадження MCP.

## Навчальні цілі

До кінця цього уроку ви зможете:

- **Визначити специфічні загрози MCP**: Розпізнавати унікальні ризики безпеки в системах MCP, включно з ін’єкцією підказок, отруєнням інструментів, надмірними дозволами, захопленням сесій, проблемами confused deputy, вразливістю passthrough токенів і ризиками ланцюга постачання
- **Застосувати заходи безпеки**: Впроваджувати ефективні пом'якшувальні заходи, включаючи надійну автентифікацію, доступ за принципом найменших привілеїв, безпечне керування токенами, контроль безпеки сесій і верифікацію ланцюга постачання
- **Використовувати рішення Microsoft для безпеки**: Розуміти та розгортати Microsoft Prompt Shields, Azure Content Safety і GitHub Advanced Security для захисту навантажень MCP
- **Перевіряти безпеку інструментів**: Усвідомлювати важливість валідації метаданих інструментів, моніторингу динамічних змін і захисту від непрямих атак ін’єкції підказок
- **Інтегрувати найкращі практики**: Поєднувати встановлені основи безпеки (безпечне кодування, зміцнення серверів, zero trust) з контролями, специфічними для MCP, для всебічного захисту

# Архітектура безпеки MCP та контролі

Сучасні реалізації MCP потребують багаторівневих підходів до безпеки, які враховують як традиційні загрози програмного забезпечення, так і специфічні для ШІ загрози. Швидко що розвивається стандарт MCP постійно вдосконалює свої засоби безпеки, забезпечуючи кращу інтеграцію з корпоративними архітектурами безпеки та встановленими найкращими практиками.

Дослідження з [Звіту про цифровий захист Microsoft](https://aka.ms/mddr) показують, що **98% зареєстрованих порушень можна було б запобігти завдяки надійній гігієні безпеки**. Найефективнішою стратегією захисту є поєднання фундаментальних заходів безпеки з контролями, специфічними для MCP — перевірені базові заходи безпеки залишаються найважливішими для зниження загального ризику безпеки.

### Еволюція аутентифікації MCP

Специфікація MCP суттєво розвинулась у підходах до аутентифікації та авторизації:

- **Початковий підхід**: Ранні специфікації вимагали від розробників створювати власні сервери аутентифікації, і MCP сервери виконували роль OAuth 2.0 серверів авторизації для прямого керування аутентифікацією користувачів
- **Поточний стандарт (2025-11-25)**: Оновлена специфікація дозволяє серверам MCP делегувати аутентифікацію зовнішнім постачальникам ідентичності (наприклад, Microsoft Entra ID), покращуючи безпеку та зменшуючи складність впровадження
- **Безпека транспортного рівня**: Покращена підтримка безпечних протоколів передачі із належними схемами аутентифікації для локальних (STDIO) та віддалених (Streamable HTTP) з’єднань

## Безпека автентифікації та авторизації

### Поточні виклики безпеки

Сучасні реалізації MCP стикаються з низкою викликів у сфері автентифікації та авторизації:

### Ризики та загрози

- **Неправильно налаштована логіка авторизації**: Помилки в реалізації авторизації на серверах MCP можуть призвести до викриття конфіденційних даних або неправильного застосування контролю доступу
- **Компрометація OAuth токенів**: Крадіжка токенів з локального сервера MCP дозволяє зловмисникам імітувати сервери та отримувати доступ до подальших сервісів
- **Вразливості passthrough токенів**: Неправильне оброблення токенів створює обхідні шляхи для контролю безпеки та прогалини відповідальності
- **Надмірні дозволи**: Перевищення привілеїв серверів MCP порушує принцип найменших привілеїв і розширює поверхню атак

#### Token Passthrough: Критичний контрпринцип

**Пасстрог токенів суворо заборонений** у поточній специфікації авторизації MCP через серйозні наслідки для безпеки:

##### Обхід контролю безпеки
- Сервери MCP та подальші API реалізують критичні заходи безпеки (обмеження частоти запитів, валідація запитів, моніторинг трафіку), які залежать від коректної валідації токенів
- Використання токенів безпосередньо клієнтом до API обходить ці важливі захисти, підриваючи архітектуру безпеки

##### Проблеми відповідальності та аудиту  
- Сервери MCP не можуть розрізнити клієнтів, які використовують токени, видані вище, що руйнує сліди аудиту
- Логи ресурсних серверів показують хибне походження запитів замість реальних проміжних серверів MCP
- Розслідування інцидентів і аудит відповідності істотно ускладнюються

##### Ризики витоку даних
- Невалідовані твердження токенів дозволяють зловмисникам зі вкраденими токенами використовувати сервери MCP як проксі для витоку даних
- Порушення меж довіри дозволяють несанкціоновані схеми доступу, що обходять передбачені заходи безпеки

##### Вектори атак мультисервісів
- Компрометовані токени, прийняті кількома сервісами, дозволяють бокове пересування по зв’язаних системах
- Припущення довіри між сервісами можуть бути порушені, якщо не можна підтвердити походження токенів

### Заходи контролю безпеки та пом'якшення

**Критичні вимоги безпеки:**

> **ОБОВ’ЯЗКОВО:** Сервери MCP **НЕ ДОЛЖНІ** приймати будь-які токени, які не були явно видані для сервера MCP

#### Контролі автентифікації та авторизації

- **Ретельний огляд авторизації**: Проводити комплексні аудити логіки авторизації сервера MCP, щоб гарантувати доступ лише для передбачених користувачів і клієнтів до чутливих ресурсів
  - **Посібник з впровадження**: [Azure API Management як шлюз автентифікації для серверів MCP](https://techcommunity.microsoft.com/blog/integrationsonazureblog/azure-api-management-your-auth-gateway-for-mcp-servers/4402690)
  - **Інтеграція ідентичності**: [Використання Microsoft Entra ID для автентифікації MCP серверів](https://den.dev/blog/mcp-server-auth-entra-id-session/)

- **Безпечне керування токенами**: Впроваджуйте [найкращі практики Microsoft щодо валідації токенів і життєвого циклу](https://learn.microsoft.com/en-us/entra/identity-platform/access-tokens)
  - Перевіряйте, щоб аудиторія токена відповідала ідентичності сервера MCP
  - Впровадьте належну політику ротації та терміну дії токенів
  - Запобігайте атакам повторного використання та несанкціонованому використанню токенів

- **Захищене зберігання токенів**: Захищене зберігання токенів із шифруванням у стані спокою та при передачі
  - **Найкращі практики**: [Рекомендації щодо безпечного зберігання та шифрування токенів](https://youtu.be/uRdX37EcCwg?si=6fSChs1G4glwXRy2)

#### Впровадження контролю доступу

- **Принцип найменших привілеїв**: Надавайте серверам MCP лише мінімально необхідні дозволи для передбаченої функціональності
  - Регулярно переглядайте та оновлюйте дозволи, щоб запобігти розширенню привілеїв
  - **Документація Microsoft**: [Безпечний доступ з найменшими привілеями](https://learn.microsoft.com/entra/identity-platform/secure-least-privileged-access)

- **Рольовий контроль доступу (RBAC)**: Впроваджуйте детальне призначення ролей
  - Чітко обмежуйте ролі конкретними ресурсами та діями
  - Уникайте широких або непотрібних дозволів, які розширюють поверхню атаки

- **Постійний моніторинг дозволів**: Впроваджуйте безперервний аудит і моніторинг доступу
  - Відстежуйте аномалії у використанні дозволів
  - Швидко усувайте надмірні або неактивні привілеї

## Специфічні загрози безпеці ШІ

### Атаки з ін’єкцією підказок і маніпуляцією інструментів

Сучасні реалізації MCP стикаються з витонченими векторами атак, специфічними для ШІ, які традиційні заходи безпеки не можуть повністю нейтралізувати:

#### **Непряма ін’єкція підказок (ін’єкція міждоменних підказок)**

**Непряма ін’єкція підказок** є однією з найкритичніших вразливостей у системах ШІ на базі MCP. Зловмисники вставляють шкідливі інструкції у зовнішній контент — документи, веб-сторінки, емейли чи джерела даних, які системи ШІ обробляють як легітимні команди.

**Сценарії атак:**
- **Ін’єкція в документи**: Шкідливі інструкції, приховані в оброблюваних документах, що викликають небажані дії ШІ
- **Використання веб-контенту**: Скомпрометовані веб-сторінки з вбудованими підказками, що маніпулюють поведінкою ШІ при скрейпінгу
- **Атаки через емейли**: Шкідливі підказки в листах, що змушують ШІ-асистентів розкривати інформацію або виконувати несанкціоновані дії
- **Зараження джерел даних**: Скомпрометовані БД або API, що постачають заражений контент системам ШІ

**Реальний вплив**: Ці атаки можуть призвести до витоку даних, порушень конфіденційності, генерації шкідливого контенту та маніпуляцій користувацькими взаємодіями. Детальний аналіз див. у [Prompt Injection in MCP (Simon Willison)](https://simonwillison.net/2025/Apr/9/mcp-prompt-injection/).

![Prompt Injection Attack Diagram](../../../translated_images/uk/prompt-injection.ed9fbfde297ca877.webp)

#### **Атаки отруєння інструментів**

**Отруєння інструментів** спрямоване на метадані, що описують інструменти MCP, експлуатуючи спосіб, яким великі мовні моделі інтерпретують описи та параметри інструментів для прийняття рішень щодо виконання.

**Механізми атаки:**
- **Маніпуляція метаданими**: Зловмисники впроваджують шкідливі інструкції в описи інструментів, визначення параметрів або приклади використання
- **Невидимі інструкції**: Приховані підказки в метаданих інструментів, які обробляються моделями ШІ, але не видимі користувачам
- **Динамічні зміни інструментів ("Rug Pulls")**: Інструменти, схвалені користувачами, пізніше змінюються для шкідливих дій без відома користувачів
- **Ін’єкція параметрів**: Шкідливий контент, вбудований у схеми параметрів інструментів, що впливає на поведінку моделей


**Ризики розміщених серверів**: Віддалені MCP-сервери містять підвищені ризики, оскільки визначення інструментів можуть бути оновлені після початкового погодження користувача, створюючи ситуації, в яких раніше безпечні інструменти стають шкідливими. Для всебічного аналізу див. [Атаки з підміною інструментів (Invariant Labs)](https://invariantlabs.ai/blog/mcp-security-notification-tool-poisoning-attacks).

![Діаграма атаки з ін’єкцією інструментів](../../../translated_images/uk/tool-injection.3b0b4a6b24de6bef.webp)

#### **Додаткові вектори атак ШІ**

- **Міждоменна ін’єкція підказок (XPIA)**: Складні атаки, які використовують контент з кількох доменів для обходу контролю безпеки
- **Динамічна модифікація можливостей**: Зміни можливостей інструментів у реальному часі, що уникають початкових оцінок безпеки
- **Отруєння вікна контексту**: Атаки, які маніпулюють великими вікнами контексту, щоб приховати шкідливі інструкції
- **Атаки зі сплутуванням моделей**: Використання обмежень моделі для створення непередбачуваної або небезпечної поведінки


### Вплив ризиків безпеки ШІ

**Наслідки з високим впливом:**
- **Витік даних**: Несанкціонований доступ та крадіжка чутливої корпоративної або персональної інформації
- **Порушення конфіденційності**: Виток персональних даних (PII) та конфіденційної бізнес-інформації  
- **Маніпуляції системою**: Непередбачені зміни критичних систем і робочих процесів
- **Крадіжка облікових даних**: Компрометація маркерів автентифікації та облікових записів сервісів
- **Бічний рух**: Використання скомпрометованих систем ШІ як точок опори для ширших атак у мережі

## Складні загрози безпеки MCP

### Уразливості викрадення сесій

**Викрадення сесій** є критичним вектором атаки в станоутримуваних реалізаціях MCP, коли неавторизовані сторони отримують і зловживають легітимними ідентифікаторами сесій для імітації клієнтів і виконання несанкціонованих дій.

#### **Сценарії атак і ризики**

- **Ін’єкція підказок при викраденні сесії**: Зловмисники із викраденими ID сесій впроваджують шкідливі події на серверах з спільним станом сесії, що може призвести до шкідливих дій або доступу до конфіденційних даних
- **Пряма імітація**: Викрадені ID сесій дають змогу здійснювати безпосередні виклики MCP сервера, минаючи аутентифікацію, сприймаючи атакувальників як легітимних користувачів
- **Скомпрометовані відновлювальні потоки**: Зловмисники можуть достроково завершувати запити, змушуючи легітимних клієнтів відновлюватися з потенційно шкідливим контентом

#### **Контролі безпеки для керування сесіями**

**Критичні вимоги:**
- **Перевірка авторизації**: MCP сервери, що реалізують авторизацію, **МАЮТЬ** перевіряти ВСІ вхідні запити і **НЕ МАЮТЬ** покладатися на сесії для автентифікації
- **Безпечне створення сесій**: Використання криптографічно безпечних, недетермінованих ID сесій, згенерованих за допомогою надійних генераторів випадкових чисел
- **Прив'язка до користувача**: Прив’язка ID сесії до інформації конкретного користувача з використанням форматів на кшталт `<user_id>:<session_id>`, щоб уникнути зловживання сесіями між користувачами
- **Керування життєвим циклом сесії**: Впровадження коректного терміну дії, обертання та анулювання сесій для обмеження вікна вразливостей
- **Безпека транспорту**: Обов’язкове використання HTTPS для всіх комунікацій задля запобігання перехопленню ID сесій

### Проблема сплутаного помічника (Confused Deputy)

**Проблема сплутаного помічника** трапляється, коли MCP сервери виступають як проксі для автентифікації між клієнтами та сторонніми сервісами, створюючи можливості обходу авторизації через експлуатацію статичних ID клієнтів.

#### **Механіка атак і ризики**

- **Обхід згоди на основі cookie**: Попередня автентифікація користувача створює cookie згоди, які зловмисники експлуатують через шкідливі авторизаційні запити з підробленими URI перенаправлення
- **Крадіжка коду авторизації**: Існуючі cookie згоди можуть змусити сервери авторизації пропускати екрани згоди, перенаправляючи коди на контрольовані зловмисниками кінцеві точки  
- **Несанкціонований доступ до API**: Викрадені коди авторизації дозволяють обмінювати їх на токени та імітувати користувачів без явної згоди

#### **Стратегії пом’якшення**

**Обов’язкові контролі:**
- **Явні вимоги до згоди**: MCP проксі-сервери, що використовують статичні ID клієнтів, **МАЮТЬ** отримувати згоду користувача для кожного динамічно зареєстрованого клієнта
- **Впровадження безпеки OAuth 2.1**: Дотримання актуальних передових практик безпеки OAuth, включно з PKCE (Proof Key for Code Exchange) для всіх авторизаційних запитів
- **Сувора валідація клієнтів**: Впровадження ретельної перевірки URI перенаправлення та ідентифікаторів клієнтів для запобігання експлуатації

### Уразливості передачі токенів  

**Передача токенів** є явним антишаблоном, коли MCP сервери приймають токени клієнтів без належної валідації й пересилають їх до нижченаведених API, порушуючи специфікації авторизації MCP.

#### **Наслідки для безпеки**

- **Обхід контролю**: Пряме використання токенів клієнтом на API оминає критично важливі обмеження частоти, валідацію і моніторинг
- **Пошкодження аудиту**: Токени, видані зверху, унеможливлюють ідентифікацію клієнтів, порушуючи можливості розслідування інцидентів
- **Витік даних через проксі**: Невалідавані токени дають змогу зловмисникам використовувати сервери як проксі для несанкціонованого доступу
- **Порушення меж довіри**: Довіра нижченаведених сервісів порушується, якщо походження токенів не можна перевірити
- **Розширення атак на кілька сервісів**: Скомпрометовані токени, що приймаються на кількох сервісах, дозволяють виконувати бічний рух

#### **Обов’язкові заходи безпеки**

**Без компромісів:**
- **Валідація токенів**: MCP сервери **НЕ МАЮТЬ** приймати токени, які не були явно видані для MCP сервера
- **Перевірка аудиторії**: Завжди перевіряти відповідність вимог аудиторії токенів ідентичності MCP сервера
- **Коректний життєвий цикл токена**: Впровадження короткочасних токенів доступу з безпечними практиками ротації


## Безпека ланцюга поставок для ШІ-систем

Безпека ланцюга поставок вийшла за межі традиційних залежностей ПО і охоплює всю екосистему ШІ. Сучасні реалізації MCP повинні ретельно перевіряти та контролювати всі компоненти, пов’язані з ШІ, оскільки кожен з них вносить потенційні вразливості, які можуть підірвати цілісність системи.

### Розширені компоненти ланцюга поставок ШІ

**Традиційні залежності програмного забезпечення:**
- Бібліотеки та фреймворки з відкритим кодом
- Образи контейнерів та базові системи  
- Засоби розробки та конвеєри збірки
- Компоненти та сервіси інфраструктури

**Специфічні для ШІ елементи ланцюга поставок:**
- **Базові моделі**: Попередньо навчені моделі від різних постачальників, що потребують перевірки походження
- **Сервіси векторизації**: Зовнішні сервіси векторизації та семантичного пошуку
- **Постачальники контексту**: Джерела даних, бази знань та сховища документів  
- **API третіх сторін**: Зовнішні сервіси ШІ, ML-конвеєри та кінцеві точки обробки даних
- **Артефакти моделей**: Ваги, конфігурації та варіанти тонкого налаштування моделей
- **Джерела даних для навчання**: Набори даних, використані для навчання та тонкого налаштування моделей

### Всебічна стратегія безпеки ланцюга поставок

#### **Перевірка компонентів і довіра**
- **Перевірка походження**: Перевірка джерела, ліцензії та цілісності всіх компонентів ШІ перед інтеграцією
- **Оцінка безпеки**: Проведення сканування вразливостей та оглядів безпеки моделей, джерел даних і сервісів ШІ
- **Аналіз репутації**: Оцінка безпекової історії та практик постачальників сервісів ШІ
- **Перевірка відповідності**: Забезпечення відповідності усіх компонентів організаційним вимогам безпеки та нормативним стандартам

#### **Безпечні конвеєри розгортання**  
- **Автоматизоване сканування CI/CD безпеки**: Інтеграція сканування безпеки на всіх етапах автоматичних конвеєрів розгортання
- **Цілісність артефактів**: Впровадження криптографічної перевірки всіх розгортаних артефактів (код, моделі, конфігурації)
- **Поступове розгортання**: Використання поетапних стратегій з перевіркою безпеки на кожному етапі
- **Довірені сховища артефактів**: Розгортання лише з перевірених, безпечних реєстрів та сховищ артефактів

#### **Безперервний моніторинг і реагування**
- **Сканування залежностей**: Постійний моніторинг вразливостей усіх залежностей програмного забезпечення та компонентів ШІ
- **Моніторинг моделей**: Безперервна оцінка поведінки моделей, зсуву продуктивності та аномалій безпеки
- **Відстеження здоров’я сервісів**: Моніторинг зовнішніх сервісів ШІ щодо доступності, інцидентів безпеки та змін політик
- **Інтеграція розвідки про загрози**: Включення потоків розвідки про загрози, специфічних для безпеки ШІ та машинного навчання

#### **Контроль доступу та мінімальні привілеї**
- **Дозволи на рівні компонентів**: Обмеження доступу до моделей, даних і сервісів лише за необхідністю бізнесу
- **Управління сервісними обліковими записами**: Впровадження виділених сервісних акаунтів з мінімальними необхідними правами
- **Сегментація мережі**: Ізоляція компонентів ШІ та обмеження мережевого доступу між сервісами
- **Контроль API Gateway**: Використання централізованих шлюзів API для контролю і моніторингу доступу до зовнішніх сервісів ШІ

#### **Реагування на інциденти і відновлення**
- **Процедури швидкого реагування**: Встановлені процеси для виправлення або заміни скомпрометованих компонентів ШІ
- **Ротація облікових даних**: Автоматизовані системи для ротації секретів, API ключів та сервісних облікових даних

- **Відновлення після порушення ланцюга поставок**: Специфічні процедури реагування на компрометацію вищого рівня AI-сервісів

Реалізації MCP успадковують і розвивають існуючу безпекову позицію вашої організації. Посилення основних практик безпеки суттєво підвищує загальний рівень безпеки систем ШІ та впроваджень MCP.

- **Безпечне управління секретами**: Використання окремих сховищ для токенів, API-ключів і конфіденційних конфігураційних даних
- **Кінцеве шифрування**: Реалізація захищених комунікацій у всіх компонентах додатку та потоках даних
- **Валідація вхідних даних**: Ретельна перевірка всіх користувацьких вводів, параметрів API та джерел даних

#### **Зміцнення інфраструктури**
- **Багатофакторна автентифікація**: Обов’язкове MFA для всіх адміністративних та сервісних облікових записів
- **Управління оновленнями**: Автоматизоване, своєчасне патч-менеджмент для ОС, фреймворків і залежностей  
- **Інтеграція провайдерів ідентичності**: Централізоване управління ідентичністю через корпоративні провайдери (Microsoft Entra ID, Active Directory)
- **Сегментація мережі**: Логічна ізоляція компонентів MCP для обмеження потенціалу бічного руху
- **Принцип мінімальних привілеїв**: Мінімально необхідні дозволи для всіх системних компонентів і облікових записів

#### **Моніторинг та виявлення**
- **Розгорнуте логування**: Детальне ведення журналу діяльності AI-застосунків, включно з взаємодіями клієнт-сервер MCP
- **Інтеграція з SIEM**: Централізоване управління інформацією та подіями безпеки для виявлення аномалій
- **Аналітика поведінки**: ШІ-інструменти моніторингу для виявлення незвичайних патернів у поведінці системи та користувачів
- **Розвідка про загрози**: Інтеграція зовнішніх потоків загроз і індикаторів компрометації (IOC)
- **Реагування на інциденти**: Чітко визначені процедури виявлення, реагування та відновлення після інцидентів безпеки

#### **Архітектура Zero Trust**
- **Ніколи не довіряти, завжди перевіряти**: Безперервна перевірка користувачів, пристроїв та мережевих з’єднань
- **Мікросегментація**: Гранулярні мережеві контролі, що ізолюють окремі робочі навантаження та сервіси
- **Безпека, орієнтована на ідентичність**: Політики безпеки, що базуються на підтверджених ідентичностях замість мережевого розташування
- **Безперервна оцінка ризиків**: Динамічна оцінка безпеки на основі поточного контексту та поведінки
- **Умовний доступ**: Контроль доступу, який адаптується залежно від факторів ризику, місця розташування та довіри до пристрою

### Шаблони інтеграції в корпоративному середовищі

#### **Інтеграція в екосистему безпеки Microsoft**
- **Microsoft Defender for Cloud**: Комплексне управління безпекою хмарної інфраструктури
- **Azure Sentinel**: Хмарний SIEM і SOAR для захисту навантажень ШІ
- **Microsoft Entra ID**: Управління ідентичністю і доступом у підприємствах з політиками умовного доступу
- **Azure Key Vault**: Централізоване управління секретами з підтримкою апаратних модулів безпеки (HSM)
- **Microsoft Purview**: Управління даними й відповідність нормативам для джерел даних і робочих процесів ШІ

#### **Відповідність і управління**
- **Відповідність нормативам**: Забезпечення відповідності реалізацій MCP галузевим вимогам (GDPR, HIPAA, SOC 2)

- **Класифікація даних**: Правильне категоризування та обробка конфіденційних даних, що опрацьовуються системами ШІ
- **Аудиторські журнали**: Комплексне ведення журналів для дотримання нормативних вимог і криміналістичного розслідування
- **Контролі приватності**: Впровадження принципів privacy-by-design в архітектурі систем ШІ
- **Управління змінами**: Формальні процеси для безпекових оглядів змін у системах ШІ

Ці базові практики створюють надійну основу безпеки, яка підвищує ефективність специфічних для MCP засобів контролю безпеки та забезпечує комплексний захист AI-додатків.

## Основні висновки щодо безпеки

- **Багаторівневий підхід до безпеки**: Поєднуйте базові практики безпеки (безпечне кодування, найменші привілеї, перевірка ланцюжка постачання, безперервний моніторинг) із специфічними для ШІ контролями для всебічного захисту

- **Специфічні загрози для ШІ**: Системи MCP стикаються з унікальними ризиками, включаючи ін’єкції запитів, отруєння інструментів, викрадення сесій, проблему плутаного уповноваженого, вразливості з передачею токенів та надмірні дозволи, що вимагають спеціалізованих заходів захисту

- **Відмінна аутентифікація та авторизація**: Запроваджуйте надійну аутентифікацію з використанням зовнішніх постачальників ідентичності (Microsoft Entra ID), забезпечуйте правильну перевірку токенів і ніколи не приймайте токени, які не були явно видані для вашого MCP-сервера

- **Запобігання атакам на ШІ**: Використовуйте Microsoft Prompt Shields та Azure Content Safety для захисту від непрямих ін’єкцій запитів та отруєння інструментів, водночас перевіряючи метадані інструментів та моніторячи динамічні зміни

- **Безпека сесій та транспорту**: Використовуйте криптографічно захищені, недетерміновані ідентифікатори сесій, прив’язані до ідентичностей користувачів, впроваджуйте належне управління життєвим циклом сесій і ніколи не використовуйте сесії для аутентифікації

- **Кращі практики безпеки OAuth**: Запобігайте атакам «плутаного уповноваженого» через явну згоду користувача для динамічно зареєстрованих клієнтів, правильну реалізацію OAuth 2.1 з PKCE та сувору валідацію URI перенаправлення  

- **Принципи безпеки токенів**: Уникайте антипатернів з передачею токенів, перевіряйте заяви аудиторії токенів, впроваджуйте короткострокові токени з безпечним ротаційним механізмом та підтримуйте чіткі межі довіри

- **Комплексна безпека ланцюжка постачання**: Обробляйте всі компоненти екосистеми ШІ (моделі, embeddings, провайдери контексту, зовнішні API) з таким же рівнем безпеки, як традиційні програмні залежності

- **Безперервна еволюція**: Будьте в курсі швидкоплинних специфікацій MCP, вносьте свій внесок у стандарти безпеки спільноти та підтримуйте адаптивну позицію безпеки в міру розвитку протоколу

- **Інтеграція безпеки Microsoft**: Використовуйте всебічну екосистему безпеки Microsoft (Prompt Shields, Azure Content Safety, GitHub Advanced Security, Entra ID) для посиленого захисту впроваджень MCP
