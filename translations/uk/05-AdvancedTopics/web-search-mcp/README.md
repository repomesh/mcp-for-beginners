## MCP сервер веб-пошуку

У цьому розділі представлено архітектуру і можливості MCP сервера веб-пошуку. Ви побачите, як FastMCP та SerpAPI використовуються разом для розширення можливостей LLM даними в реальному часі з вебу.

### Огляд

У цій реалізації використовуються чотири інструменти, які демонструють здатність MCP безпечно й ефективно обробляти різноманітні завдання, керовані зовнішніми API:

- **general_search**: Для широких результатів веб-пошуку
- **news_search**: Для останніх новинних заголовків
- **product_search**: Для даних електронної комерції
- **qna**: Для фрагментів запитань і відповідей

### Особливості
- **Приклади коду**: Включає мовно-специфічні блоки коду для Python (і легко розширювані на інші мови) з використанням кодових точок для ясності

### Python

```python
# Приклад використання інструмента general_search
from mcp import ClientSession, StdioServerParameters
from mcp.client.stdio import stdio_client

async def run_search():
    server_params = StdioServerParameters(
        command="python",
        args=["server.py"],
    )
    async with stdio_client(server_params) as (reader, writer):
        async with ClientSession(reader, writer) as session:
            await session.initialize()
            result = await session.call_tool("general_search", arguments={"query": "open source LLMs"})
            print(result)
```

Перед запуском клієнта корисно розуміти, що саме робить сервер. Він обробляє вхідні запити, керує викликами API, парсить відповіді і повертає клієнту структуровані результати.

Ви можете переглянути повну реалізацію у файлі [`server.py`](../../../../05-AdvancedTopics/web-search-mcp/server.py).

## Використання

MCP сервер веб-пошуку є основним компонентом, який надає інструменти для веб-, новинного, продуктової пошуку та Q&A через інтеграцію з SerpAPI. Він приймає запити, керує викликами API, парсить відповіді і повертає структуровані результати клієнту.

Повну реалізацію можна переглянути у файлі [`server.py`](../../../../05-AdvancedTopics/web-search-mcp/server.py).

### Запуск сервера

Щоб запустити MCP сервер, використайте таку команду:

```bash
python server.py
```

Сервер працюватиме як MCP сервер на основі stdio, до якого клієнт може підключитися безпосередньо.

### Режими клієнта

Клієнт (`client.py`) підтримує два режими взаємодії з MCP сервером:

- **Звичайний режим**: Виконує автоматизовані тести, які перевіряють всі інструменти і їх відповіді. Це корисно для швидкої перевірки коректної роботи сервера і інструментів.
- **Інтерактивний режим**: Запускає інтерфейс із меню, де можна вручну обирати інструменти, вводити власні запити і бачити результати в реальному часі. Ідеально для дослідження можливостей сервера й експериментів з різними вхідними даними.

Повну реалізацію можна переглянути у файлі [`client.py`](../../../../05-AdvancedTopics/web-search-mcp/client.py).

### Запуск клієнта

Щоб запустити автоматизовані тести (це автоматично запустить сервер):

```bash
python client.py
```

Або увімкніть інтерактивний режим:

```bash
python client.py --interactive
```

## Просунуті концепції цього уроку

Перед тим, як почати створювати, варто ознайомитися з деякими важливими просунутими концепціями, які зустрічатимуться протягом цього розділу. Розуміння їх допоможе вам краще орієнтуватися, навіть якщо ви з цими поняттями не знайомі:

- **Оркестрування кількох інструментів**: Це означає запуск кількох різних інструментів (веб-пошук, новини, продукти, Q&A) на одному MCP сервері. Це дозволяє серверу обробляти різні завдання, а не тільки одне.
- **Обробка ліміту запитів API**: Багато зовнішніх API (наприклад, SerpAPI) обмежують кількість запитів за певний час. Хороший код перевіряє ці обмеження і коректно їх обробляє, щоб програма не зламалася при досягненні ліміту.
- **Парсинг структурованих даних**: Відповіді API часто є складними та вкладеними. Ця концепція полягає у перетворенні їх у чисті, зручні формати, дружні для LLM або інших програм.
- **Відновлення після помилок**: Іноді трапляються проблеми — наприклад, збій мережі або API не повертає очікувані дані. Відновлення означає, що ваш код може впоратися з цими проблемами і надати корисний зворотний зв’язок, замість аварійного завершення.
- **Перевірка параметрів**: Це перевірка, що всі вхідні дані для інструментів правильні та безпечні для використання. Включає встановлення значень за замовчуванням і перевірку типів, що допомагає уникнути помилок і плутанини.

Цей розділ допоможе вам діагностувати та усувати поширені проблеми, з якими можна зіткнутися при роботі з MCP сервером веб-пошуку. Якщо ви натрапите на помилки або несподівану поведінку, цей розділ містить рішення найпоширеніших проблем. Ознайомтеся з цими порадами перед тим, як звертатися за додатковою допомогою — вони часто швидко розв’язують проблему.

## Усунення несправностей

Під час роботи з MCP сервером веб-пошуку час від часу можуть виникати проблеми — це нормально при розробці з зовнішніми API і новими інструментами. Цей розділ пропонує практичні рішення для найпоширеніших проблем, щоб ви могли швидко повернутися до роботи. Якщо з'являється помилка, починайте тут: наведені поради розглядають проблеми, які найчастіше виникають у користувачів, і часто можуть розв’язати проблему без додаткової допомоги.

### Поширені проблеми

Нижче наведено деякі з найпоширеніших проблем, з якими стикаються користувачі, разом з чіткими поясненнями і кроками для їх усунення:

1. **Відсутній SERPAPI_KEY у файлі .env**
   - Якщо ви бачите помилку `SERPAPI_KEY environment variable not found`, це означає, що ваш додаток не може знайти API-ключ для доступу до SerpAPI. Для виправлення створіть файл з назвою `.env` у корені вашого проекту (якщо його ще немає) і додайте рядок `SERPAPI_KEY=your_serpapi_key_here`. Обов’язково замініть `your_serpapi_key_here` на ваш реальний ключ із сайту SerpAPI.

2. **Помилки “Module not found”**
   - Помилки типу `ModuleNotFoundError: No module named 'httpx'` означають, що відсутній потрібний пакет Python. Зазвичай це трапляється, якщо ви не встановили всі залежності. Для усунення проблеми виконайте у терміналі `pip install -r requirements.txt`, щоб інсталювати все необхідне для вашого проєкту.

3. **Проблеми з підключенням**
   - Якщо ви отримуєте помилку типу `Error during client execution`, це часто означає, що клієнт не може підключитися до сервера або сервер не запускається як очікувалося. Перевірте, що клієнт і сервер сумісних версій і що файл `server.py` присутній і запускається в правильній директорії. Перезапуск сервера та клієнта також може допомогти.

4. **Помилки SerpAPI**
   - Повідомлення `Search API returned error status: 401` означає, що ваш ключ SerpAPI відсутній, неправильний або минув термін його дії. Перейдіть у вашу панель SerpAPI, перевірте ключ і оновіть файл `.env` за потреби. Якщо ключ правильний, але помилка все одно з’являється, перевірте, чи не вичерпано у вас безкоштовний ліміт.

### Режим відлагодження

За замовчуванням додаток логує лише важливу інформацію. Якщо ви хочете бачити деталі про що відбувається (наприклад, щоб діагностувати складні проблеми), можна увімкнути режим DEBUG. Він показує значно більше про кожен крок, який виконує додаток.

**Приклад: звичайний вивід**
```plaintext
2025-06-01 10:15:23,456 - __main__ - INFO - Calling general_search with params: {'query': 'open source LLMs'}
2025-06-01 10:15:24,123 - __main__ - INFO - Successfully called general_search

GENERAL_SEARCH RESULTS:
... (search results here) ...
```

**Приклад: вивід DEBUG**
```plaintext
2025-06-01 10:15:23,456 - __main__ - INFO - Calling general_search with params: {'query': 'open source LLMs'}
2025-06-01 10:15:23,457 - httpx - DEBUG - HTTP Request: GET https://serpapi.com/search ...
2025-06-01 10:15:23,458 - httpx - DEBUG - HTTP Response: 200 OK ...
2025-06-01 10:15:24,123 - __main__ - INFO - Successfully called general_search

GENERAL_SEARCH RESULTS:
... (search results here) ...
```

Зверніть увагу, що режим DEBUG включає додаткові рядки про HTTP запити, відповіді та інші внутрішні деталі. Це дуже корисно для усунення неполадок.

Щоб увімкнути режим DEBUG, встановіть рівень логування у DEBUG на початку вашого `client.py` або `server.py`:

# [Python](#tab/python-debug)


```python
# На початку вашого client.py або server.py
import logging
logging.basicConfig(
    level=logging.DEBUG,  # Змініть з INFO на DEBUG
    format="%(asctime)s - %(name)s - %(levelname)s - %(message)s"
)
```
