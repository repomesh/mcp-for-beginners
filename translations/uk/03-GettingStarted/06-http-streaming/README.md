# HTTPS Стрімінг з Протоколом Контексту Моделі (MCP)

У цій главі представлено всебічний посібник з реалізації безпечного, масштабованого та реального часу стрімінгу за допомогою Протоколу Контексту Моделі (MCP) з використанням HTTPS. Розглядається мотивація стрімінгу, доступні транспортні механізми, як реалізувати стрімінг HTTP у MCP, найкращі практики безпеки, міграція зі SSE, та практичні рекомендації для побудови власних потокових MCP-застосунків.

> **Дивлячись уперед:** цей урок описує Streamable HTTP за **MCP Specification 2025-11-25**, де сесія встановлюється під час `initialize` та прив’язується за допомогою заголовка `Mcp-Session-Id`. Реліз-кандидат `2026-07-28` повністю прибирає рукопотискання і ID сесії, роблячи кожен запит самодостатнім і маршрутизованим на будь-який сервер без прив’язки до сесії. Деталі дивіться у [Що змінюється в MCP: реліз-кандидат 2026-07-28](../../01-CoreConcepts/mcp-2026-07-28-release-candidate.md).

## Транспортні механізми та стрімінг у MCP

У цьому розділі розглядаються різні транспортні механізми, доступні в MCP, та їх роль у забезпеченні стрімінгових можливостей для реального часу взаємодії між клієнтами та серверами.

### Що таке транспортний механізм?

Транспортний механізм визначає, як дані обмінюються між клієнтом і сервером. MCP підтримує кілька типів транспорту, щоб задовольнити різні умови та вимоги:

- **stdio**: Стандартний ввід/вивід, підходить для локальних інструментів та командного рядка. Простий, але не підходить для веб чи хмари.
- **SSE (Server-Sent Events)**: Дозволяє серверам відправляти оновлення в реальному часі клієнтам через HTTP. Добре для веб-інтерфейсів, але обмежено в масштабованості і гнучкості. Відповідно до MCP Specification 2025-06-18, окремий транспорт SSE застарів і замінений транспортом "Streamable HTTP".
- **Streamable HTTP**: Сучасний стрімінговий транспорт на базі HTTP, який підтримує сповіщення і кращу масштабованість. Рекомендований для більшості виробничих та хмарних сценаріїв.

### Таблиця порівняння

Ознайомтеся з таблицею порівнянь нижче, щоб зрозуміти відмінності між цими транспортними механізмами:

| Транспорт        | Оновлення в реальному часі | Потоковий | Масштабованість | Сценарії використання    |
|------------------|----------------------------|-----------|-----------------|-------------------------|
| stdio            | Ні                         | Ні        | Низька          | Локальні CLI інструменти |
| SSE              | Так                        | Так       | Середня         | Веб, оновлення в реальному часі |
| Streamable HTTP  | Так                        | Так       | Висока          | Хмара, мультіклієнтські |

> **Порада:** Вибір правильного транспорту впливає на продуктивність, масштабованість і користувацький досвід. **Streamable HTTP** рекомендований для сучасних, масштабованих та готових до хмари застосунків.

Зауважте транспорти stdio і SSE, що були показані вам в попередніх главах, і як у цій главі розглядається транспорт Streamable HTTP.

## Стрімінг: концепції та мотивація

Розуміння основних концепцій і мотивацій стрімінгу необхідне для впровадження ефективних систем комунікації в реальному часі.

**Стрімінг** — це техніка у мережевому програмуванні, яка дозволяє відправляти і отримувати дані маленькими, керованими фрагментами або як послідовність подій, замість очікування готовності всієї відповіді. Це особливо корисно для:

- Великих файлів або наборів даних.
- Оновлень у реальному часі (наприклад, чат, прогрес-бари).
- Довготривалих обчислень, де хочеться інформувати користувача.

Ось що потрібно знати про стрімінг на високому рівні:

- Дані доставляються поступово, а не всі одразу.
- Клієнт може обробляти дані по мірі їх надходження.
- Зменшує відчутну затримку і покращує досвід користувача.

### Чому використовувати стрімінг?

Причини використовувати стрімінг такі:

- Користувачі отримують миттєвий зворотній зв’язок, а не лише в кінці
- Дозволяє створювати реальні додатки і чуйні UI
- Ефективніше використовує мережеві та обчислювальні ресурси

### Простий приклад: HTTP сервер та клієнт із стрімінгом

Ось простий приклад, як можна реалізувати стрімінг:

#### Python

**Сервер (Python, використовуючи FastAPI та StreamingResponse):**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import time

app = FastAPI()

async def event_stream():
    for i in range(1, 6):
        yield f"data: Message {i}\n\n"
        time.sleep(1)

@app.get("/stream")
def stream():
    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

**Клієнт (Python, використовуючи requests):**

```python
import requests

with requests.get("http://localhost:8000/stream", stream=True) as r:
    for line in r.iter_lines():
        if line:
            print(line.decode())
```

Цей приклад демонструє сервер, який відправляє послідовність повідомлень клієнту у міру їх готовності, а не чекає на всі повідомлення одночасно.

**Як це працює:**

- Сервер видає кожне повідомлення, як тільки воно готове.
- Клієнт отримує й виводить кожен фрагмент по мірі надходження.

**Вимоги:**

- Сервер має використовувати стрімінгову відповідь (наприклад, `StreamingResponse` у FastAPI).
- Клієнт повинен обробляти відповідь як потік (`stream=True` у requests).
- Content-Type зазвичай `text/event-stream` або `application/octet-stream`.

#### Java

**Сервер (Java, використовуючи Spring Boot та Server-Sent Events):**

```java
@RestController
public class CalculatorController {

    @GetMapping(value = "/calculate", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> calculate(@RequestParam double a,
                                                   @RequestParam double b,
                                                   @RequestParam String op) {
        
        double result;
        switch (op) {
            case "add": result = a + b; break;
            case "sub": result = a - b; break;
            case "mul": result = a * b; break;
            case "div": result = b != 0 ? a / b : Double.NaN; break;
            default: result = Double.NaN;
        }

        return Flux.<ServerSentEvent<String>>just(
                    ServerSentEvent.<String>builder()
                        .event("info")
                        .data("Calculating: " + a + " " + op + " " + b)
                        .build(),
                    ServerSentEvent.<String>builder()
                        .event("result")
                        .data(String.valueOf(result))
                        .build()
                )
                .delayElements(Duration.ofSeconds(1));
    }
}
```

**Клієнт (Java, використовуючи Spring WebFlux WebClient):**

```java
@SpringBootApplication
public class CalculatorClientApplication implements CommandLineRunner {

    private final WebClient client = WebClient.builder()
            .baseUrl("http://localhost:8080")
            .build();

    @Override
    public void run(String... args) {
        client.get()
                .uri(uriBuilder -> uriBuilder
                        .path("/calculate")
                        .queryParam("a", 7)
                        .queryParam("b", 5)
                        .queryParam("op", "mul")
                        .build())
                .accept(MediaType.TEXT_EVENT_STREAM)
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(System.out::println)
                .blockLast();
    }
}
```

**Примітки до реалізації Java:**

- Використовує реактивний стек Spring Boot з `Flux` для стрімінгу
- `ServerSentEvent` забезпечує структурований стрімінг подій з типами подій
- `WebClient` з `bodyToFlux()` дозволяє реактивне споживання стрімів
- `delayElements()` імітує час обробки між подіями
- Події можуть мати типи (`info`, `result`) для кращої обробки клієнтом

### Порівняння: класичний стрімінг проти MCP-стрімінгу

Відмінності у тому, як стрімінг працює у "класичному" розумінні та як він реалізований у MCP, можна показати так:

| Функція                | Класичний HTTP Стрімінг      | MCP Стрімінг (сповіщення)      |
|-----------------------|------------------------------|-------------------------------|
| Основна відповідь      | Фрагментована                 | Одна, наприкінці               |
| Оновлення прогресу     | Відправляються як фрагменти   | Відправляються як сповіщення  |
| Вимоги до клієнта      | Має обробляти стрім           | Має реалізувати обробник повідомлень |
| Сценарії використання  | Великі файли, потоки токенів AI | Прогрес, логи, зворотній зв’язок у реальному часі |

### Ключові відмінності

Додатково наведені деякі ключові відмінності:

- **Схема комунікації:**
  - Класичний HTTP стрімінг: використовує просту передачу фрагментованих даних (chunked transfer encoding)
  - MCP стрімінг: використовує структуровану систему сповіщень за протоколом JSON-RPC

- **Формат повідомлень:**
  - Класичний HTTP: простий текст з новими рядками
  - MCP: структуровані об’єкти LoggingMessageNotification з метаданими

- **Реалізація клієнта:**
  - Класичний HTTP: простий клієнт, що обробляє потік відповідей
  - MCP: складніший клієнт з обробником повідомлень для різних типів повідомлень

- **Оновлення прогресу:**
  - Класичний HTTP: прогрес є частиною основного потоку відповіді
  - MCP: прогрес відправляється окремими сповіщеннями, тоді як основний результат приходить наприкінці

### Рекомендації

Ось деякі рекомендації при виборі між класичною реалізацією стрімінгу (як ми показали раніше з `/stream`) та стрімінгом через MCP.

- **Для простих потреб стрімінгу:** Класичний HTTP стрімінг легший у реалізації і достатній для базових випадків.

- **Для складних інтерактивних додатків:** MCP стрімінг пропонує структурований підхід із багатшими метаданими та розділенням сповіщень і фінального результату.

- **Для AI-застосунків:** Система сповіщень MCP особливо корисна для тривалих AI-завдань, коли потрібно інформувати користувачів про прогрес.

## Стрімінг у MCP

Добре, ви вже бачили рекомендації і порівняння між класичним стрімінгом і MCP. Тепер розглянемо докладно, як саме можна використовувати стрімінг у MCP.

Розуміння, як працює стрімінг у рамках MCP, є необхідним для побудови чуйних додатків, що надають зворотній зв’язок у реальному часі під час тривалих операцій.

У MCP стрімінг — це не надсилання основної відповіді шматками, а відправлення **сповіщень** клієнту під час обробки запиту. Ці сповіщення можуть містити оновлення прогресу, логи або інші події.

### Як це працює

Основний результат надсилається як єдина відповідь. Однак сповіщення можуть надсилатися окремими повідомленнями під час обробки, таким чином оновлюючи клієнта в реальному часі. Клієнт має вміти обробляти та відображати ці сповіщення.

## Що таке Сповіщення?

Ми сказали "Сповіщення", що це означає у контексті MCP?

Сповіщення — це повідомлення, відправлене сервером клієнту для інформування про прогрес, статус або інші події під час тривалої операції. Сповіщення покращують прозорість і користувацький досвід.

Наприклад, клієнт має надіслати сповіщення після встановлення початкового зв’язку з сервером.

Сповіщення виглядає ось так у вигляді JSON-повідомлення:

```json
{
  jsonrpc: "2.0";
  method: string;
  params?: {
    [key: string]: unknown;
  };
}
```

Сповіщення належать до теми в MCP, котра називається ["Logging"](https://modelcontextprotocol.io/specification/draft/server/utilities/logging).

Щоб увімкнути логування, сервер має активувати відповідну можливість/фічу таким чином:

```json
{
  "capabilities": {
    "logging": {}
  }
}
```

> [!NOTE]
> Залежно від SDK, логування може бути увімкнене за замовчуванням, або вам потрібно буде явно активувати його у конфігурації сервера.

Існують різні типи сповіщень:

| Рівень   | Опис                            | Приклади використання           |
|----------|--------------------------------|--------------------------------|
| debug    | Детальна інформація для налагодження | Точки входу/виходу функцій      |
| info     | Загальні інформаційні повідомлення | Оновлення прогресу              |
| notice   | Звичайні, але значущі події     | Зміни у конфігурації            |
| warning  | Потенційні проблеми             | Використання застарілих функцій |
| error    | Помилки                      | Збої операцій                  |
| critical | Критичні проблеми             | Відмови компонентів системи     |
| alert    | Необхідні негайні дії         | Виявлено пошкодження даних      |
| emergency| Система непридатна до роботи   | Повний збій системи             |

## Реалізація Сповіщень у MCP

Щоб реалізувати сповіщення в MCP, потрібно налаштувати як сервер, так і клієнт для обробки оновлень у реальному часі. Це дозволить вашому додатку надавати миттєвий зворотній зв’язок користувачам під час тривалих операцій.

### Серверна частина: надсилання сповіщень

Почнемо з серверної частини. У MCP ви визначаєте інструменти, які можуть відправляти сповіщення під час обробки запитів. Сервер використовує об’єкт контексту (зазвичай `ctx`) для відправлення повідомлень клієнту.

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    await ctx.info("Processing file 1/3...")
    await ctx.info("Processing file 2/3...")
    await ctx.info("Processing file 3/3...")
    return TextContent(type="text", text=f"Done: {message}")
```

У наведеному прикладі інструмент `process_files` відправляє три сповіщення клієнту під час обробки кожного файлу. Метод `ctx.info()` використовується для надсилання інформаційних повідомлень.

Крім того, для увімкнення сповіщень слід переконатися, що сервер використовує транспорт для стрімінгу (наприклад, `streamable-http`), а клієнт реалізує обробник повідомлень для їх обробки. Ось як налаштувати сервер на використання транспорту `streamable-http`:

```python
mcp.run(transport="streamable-http")
```

#### .NET

```csharp
[Tool("A tool that sends progress notifications")]
public async Task<TextContent> ProcessFiles(string message, ToolContext ctx)
{
    await ctx.Info("Processing file 1/3...");
    await ctx.Info("Processing file 2/3...");
    await ctx.Info("Processing file 3/3...");
    return new TextContent
    {
        Type = "text",
        Text = $"Done: {message}"
    };
}
```

У цьому .NET прикладі інструмент `ProcessFiles` позначений атрибутом `Tool` і відправляє три сповіщення клієнту під час обробки кожного файлу. Метод `ctx.Info()` використовується для надсилання інформаційних повідомлень.

Щоб увімкнути сповіщення на сервері MCP .NET, переконайтеся, що використовується стрімінговий транспорт:

```csharp
var builder = McpBuilder.Create();
await builder
    .UseStreamableHttp() // Enable streamable HTTP transport
    .Build()
    .RunAsync();
```

### Клієнтська частина: отримання сповіщень

Клієнт має реалізувати обробник повідомлень для обробки і відображення сповіщень у міру їх надходження.

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)

async with ClientSession(
   read_stream, 
   write_stream,
   logging_callback=logging_collector,
   message_handler=message_handler,
) as session:
```

У наведеному коді функція `message_handler` перевіряє, чи є вхідне повідомлення сповіщенням. Якщо так, її виводить; інакше обробляє як звичайне серверне повідомлення. Також зверніть увагу, що `ClientSession` ініціалізується з `message_handler` для обробки вхідних сповіщень.

#### .NET

```csharp
// Define a message handler
void MessageHandler(IJsonRpcMessage message)
{
    if (message is ServerNotification notification)
    {
        Console.WriteLine($"NOTIFICATION: {notification}");
    }
    else
    {
        Console.WriteLine($"SERVER MESSAGE: {message}");
    }
}

// Create and use a client session with the message handler
var clientOptions = new ClientSessionOptions
{
    MessageHandler = MessageHandler,
    LoggingCallback = (level, message) => Console.WriteLine($"[{level}] {message}")
};

using var client = new ClientSession(readStream, writeStream, clientOptions);
await client.InitializeAsync();

// Now the client will process notifications through the MessageHandler
```

У цьому .NET прикладі функція `MessageHandler` перевіряє, чи є вхідне повідомлення сповіщенням. Якщо так, її виводить; інакше обробляє як звичайне серверне повідомлення. `ClientSession` ініціалізується обробником повідомлень через `ClientSessionOptions`.

Щоб увімкнути сповіщення, переконайтеся, що сервер використовує стрімінговий транспорт (наприклад, `streamable-http`), а клієнт реалізує обробник повідомлень для обробки сповіщень.

## Прогрес-сповіщення та сценарії використання

У цьому розділі пояснюється концепція прогрес-сповіщень у MCP, чому вони важливі і як їх реалізувати за допомогою Streamable HTTP. Також є практичне завдання для закріплення.

Прогрес-сповіщення — це повідомлення в реальному часі, що надсилаються сервером клієнту під час тривалих операцій. Замість того, щоб чекати завершення процесу, сервер постійно інформує клієнта про поточний статус. Це покращує прозорість, користувацький досвід і спрощує відлагодження.

**Приклад:**

```text

"Processing document 1/10"
"Processing document 2/10"
...
"Processing complete!"

```

### Чому використовувати прогрес-сповіщення?

Прогрес-сповіщення важливі з кількох причин:

- **Кращий користувацький досвід:** Користувачі бачать оновлення у міру виконання, а не лише в кінці.
- **Зворотній зв’язок у реальному часі:** Клієнти можуть відображати прогрес-бари або логи, роблячи додаток більш інтерактивним.
- **Простіше відлагодження і моніторинг:** Розробники і користувачі бачать, де процес може сповільнюватися або зависати.

### Як реалізувати прогрес-сповіщення

Ось як можна реалізувати прогрес-сповіщення у MCP:

- **На сервері:** Використовуйте `ctx.info()` або `ctx.log()` для надсилання сповіщень при обробці кожного елемента. Це надсилає повідомлення клієнту до того, як основний результат буде готовий.
- **На клієнті:** Реалізуйте обробник повідомлень, який слухає і відображає сповіщення по мірі їх надходження. Цей обробник розрізняє сповіщення і фінальний результат.

**Приклад сервера:**

#### Python

```python
@mcp.tool(description="A tool that sends progress notifications")
async def process_files(message: str, ctx: Context) -> TextContent:
    for i in range(1, 11):
        await ctx.info(f"Processing document {i}/10")
    await ctx.info("Processing complete!")
    return TextContent(type="text", text=f"Done: {message}")
```

**Приклад клієнта:**

#### Python

```python
async def message_handler(message):
    if isinstance(message, types.ServerNotification):
        print("NOTIFICATION:", message)
    else:
        print("SERVER MESSAGE:", message)
```

## Розгляди безпеки

При реалізації MCP серверів із HTTP-транспортами безпека стає першочерговим питанням, яке вимагає уваги до багатьох векторів атак та механізмів захисту.

### Огляд

Безпека критично важлива при експонуванні MCP серверів через HTTP. Streamable HTTP вводить нові поверхні атаки і потребує ретельного налаштування.

### Ключові моменти


- **Перевірка заголовка Origin**: Завжди перевіряйте заголовок `Origin`, щоб запобігти атакам DNS rebinding.
- **Прив’язка до localhost**: Для локальної розробки прив’язуйте сервери до `localhost`, щоб уникнути їх відкриття для загального інтернету.
- **Аутентифікація**: Впроваджуйте аутентифікацію (наприклад, API ключі, OAuth) для виробничих розгортань.
- **CORS**: Налаштуйте політики Cross-Origin Resource Sharing (CORS) для обмеження доступу.
- **HTTPS**: Використовуйте HTTPS у виробництві для шифрування трафіку.

### Найкращі практики

- Ніколи не довіряйте вхідним запитам без перевірки.
- Логуйте та моніторьте всі доступи та помилки.
- Регулярно оновлюйте залежності, щоб усунути вразливості безпеки.

### Виклики

- Балансування безпеки з простотою розробки
- Забезпечення сумісності з різними клієнтськими середовищами

## Перехід з SSE на Streamable HTTP

Для застосунків, які зараз використовують Server-Sent Events (SSE), перехід на Streamable HTTP надає розширені можливості та кращу довгострокову підтримку ваших MCP реалізацій.

### Чому варто оновитися?

Існує дві вагомі причини для переходу з SSE на Streamable HTTP:

- Streamable HTTP пропонує кращу масштабованість, сумісність та ширшу підтримку сповіщень ніж SSE.
- Це рекомендований транспорт для нових MCP застосунків.

### Кроки міграції

Ось як ви можете перейти з SSE на Streamable HTTP у ваших MCP застосунках:

- **Оновіть серверний код**, використовуючи `transport="streamable-http"` у `mcp.run()`.
- **Оновіть клієнтський код**, використовуючи `streamablehttp_client` замість SSE клієнта.
- **Реалізуйте обробник повідомлень** у клієнта для обробки сповіщень.
- **Перевірте сумісність** з існуючими інструментами і робочими процесами.

### Підтримка сумісності

Рекомендується підтримувати сумісність із існуючими SSE клієнтами під час міграції. Ось деякі стратегії:

- Ви можете підтримувати SSE і Streamable HTTP одночасно, запустивши обидва транспорти на різних кінцевих точках.
- Поступово мігруйте клієнтів на новий транспорт.

### Виклики

Переконайтеся, що ви вирішуєте наступні виклики під час міграції:

- Усунення оновлення всіх клієнтів
- Врахування різниці в доставці сповіщень

Безпека має бути пріоритетом номер один при реалізації будь-якого сервера, особливо при використанні HTTP-транспортів, таких як Streamable HTTP у MCP.

При реалізації MCP серверів із HTTP-транспортами безпека стає ключовою проблемою, що вимагає уважності до багатьох векторів атак та захисних механізмів.

### Огляд

Ось кілька ключових питань безпеки:

- **Перевірка заголовка Origin**: Завжди перевіряйте заголовок `Origin`, щоб запобігти атакам DNS rebinding.
- **Прив’язка до localhost**: Для локальної розробки прив’язуйте сервери до `localhost`, щоб уникнути їх відкриття для загального інтернету.
- **Аутентифікація**: Впроваджуйте аутентифікацію (наприклад, API ключі, OAuth) для виробничих розгортань.
- **CORS**: Налаштуйте політики Cross-Origin Resource Sharing (CORS) для обмеження доступу.
- **HTTPS**: Використовуйте HTTPS у виробництві для шифрування трафіку.

### Найкращі практики

Також, ось кілька найкращих практик для забезпечення безпеки вашого MCP стрімінгового сервера:

- Ніколи не довіряйте вхідним запитам без перевірки.
- Логуйте та моніторьте всі доступи та помилки.

- Забезпечення сумісності з різними клієнтськими середовищами

### Завдання: Створіть власний стрімінговий MCP додаток

**Сценарій:**
Створіть MCP сервер і клієнт, де сервер обробляє список елементів (наприклад, файли чи документи) і надсилає сповіщення для кожного обробленого елемента. Клієнт має відображати кожне сповіщення в момент його надходження.

**Кроки:**

1. Реалізуйте серверний інструмент, що обробляє список і надсилає сповіщення для кожного елемента.
2. Реалізуйте клієнт з обробником повідомлень для відображення сповіщень у реальному часі.
3. Перевірте вашу реалізацію, запустивши сервер і клієнта одночасно, та спостерігайте за сповіщеннями.
